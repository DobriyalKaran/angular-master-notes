S-01

Variables, Scope & Hoisting

Covers: var · let · const · Function/Block/Module Scope · Hoisting · TDZ · Lexical Environment · Scope Chain

1. What is it? (Kya hai ye?)

Variable ek named memory reference hai — ek label jo kisi value ko point karta hai. Scope decide karta hai ki wo label code mein kahan se visible hai. Aur Hoisting JavaScript engine ka wo behavior hai jisme declarations ko code run hone se pehle register kar liya jata hai.

2. Why do we need it? (Zaroorat kyun hai?)

Socho agar har variable globally visible hota — 50 files ke Angular project mein count naam ka variable 20 jagah hota, sab ek dusre ko overwrite karte. Scope hume isolation deta hai: har function, har block apni chhoti duniya mein kaam karta hai. Ye encapsulation ka foundation hai — wahi cheez jo aage jaake Angular components ko independent banati hai.

3. What problem does it solve?

  • Naming collisions — do developers same naam use kar sakte hain bina takraye.
  • Accidental mutation — const reassignment rok deta hai, bugs compile... sorry, run hone se pehle hi pakde jaate hain (TypeScript mein toh compile time pe).
  • Predictability — block scope (let/const) se loop counters aur temporary values leak nahi karte.

4. Internal Working (Engine ke andar kya hota hai?)

Jab JS engine (V8) koi function ya script run karta hai, wo ek Execution Context banata hai. Har execution context ke paas ek Lexical Environment hota hai — ek internal object jisme do cheezein hoti hain:

  • Environment Record — is scope ke saare variables/functions ki actual storage.
  • Outer reference — parent lexical environment ka pointer (isi se scope chain banti hai).

Execution do phases mein hota hai:

  • Creation Phase — engine code ko scan karta hai. var declarations ko undefined se initialize karta hai, function declarations ko poora hoist karta hai, aur let/const ko register toh karta hai par initialize nahi karta — ye "uninitialized" state hi Temporal Dead Zone (TDZ) hai.
  • Execution Phase — line by line code chalta hai, assignments ab hoti hain.

7. Real Project Example (Angular context)

Angular component ki har .ts file ek ES module hai — isliye tumhare component ke bahar likhe constants globally leak nahi karte. Production mein common pattern:

// payment.component.ts
const RETRY_LIMIT = 3;              // module-scoped, poori app mein safe
const GATEWAY_TIMEOUT_MS = 30_000;   // const = intent clear: ye config hai, state nahi

@Component({...})
export class PaymentComponent {
  private attempts = 0;  // class field = instance scope, har component apna
}

Razorpay-clone jaise project mein config values const module-level pe, mutable state class fields mein — ye separation hi clean architecture ki shuruaat hai.

9. Advanced Example — Loop + Closure trap

// Interview classic: output kya hoga?
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log('var:', i), 0);
}
// var: 3, 3, 3  — kyunki teeno callbacks EK hi 'i' share karte hain

for (let j = 0; j < 3; j++) {
  setTimeout(() => console.log('let:', j), 0);
}
// let: 0, 1, 2  — har iteration mein NAYA block-scoped 'j' banta hai

10. Code Explanation — Line by Line

  • for (var i = 0; ...) — var function/global scoped hai, toh poore loop mein sirf ek i exist karta hai.
  • setTimeout(() => ..., 0) — callback abhi run nahi hota; wo macrotask queue mein jaata hai (Section 6 mein detail). Tab tak loop khatam, i = 3 ho chuka.
  • Callback jab chalta hai, wo closure se live i padhta hai — value ka snapshot nahi, reference. Isliye teeno baar 3.
  • let j — spec kehta hai ki for loop ka let har iteration pe fresh binding banata hai, aur previous iteration ki value copy hoti hai. Har callback apne apne j ko capture karta hai.

11. Common Mistakes

❌ Galtiyan jo interviews aur PRs mein pakdi jaati hain
  • var use karna 2026 mein — legacy codebases ke alawa koi excuse nahi.
  • Ye maanna ki const object ko freeze kar deta hai. const obj = {} ke baad obj.x = 1 bilkul valid hai — sirf reassignment (obj = {}) blocked hai.
  • TDZ ko "hoisting nahi hota" samajhna. let/const hoist hote hain — bas initialize nahi hote. Isi liye error ReferenceError: Cannot access 'b' before initialization aata hai, na ki b is not defined.
  • Undeclared assignment: x = 5 (bina keyword) non-strict mode mein global variable bana deta hai. ES modules by-default strict hain, isliye Angular mein ye error dega — good!

12. Best Practices

✅ Production rules
  • Default const. Reassignment chahiye toh hi let. var kabhi nahi.
  • Variables ko use se theek pehle declare karo, top pe dump mat karo — readability + TDZ dono ka fayda.
  • Module-level constants ko SCREAMING_SNAKE_CASE mein rakho — intent turant dikhta hai.

13. Performance

let/const vs var ka runtime performance difference practically zero hai — V8 dono ko optimize kar leta hai. Asli performance angle: block scoping se variables jaldi unreachable hote hain, toh GC unhe jaldi collect kar sakta hai (Section 8). Aur tight loops mein har iteration pe fresh let binding banane ki cost negligible hai — micro-optimize mat karo.

14. Edge Cases

  • typeof undeclaredVar → 'undefined' (no error), lekin TDZ mein typeof b (jahan let b neeche hai) → ReferenceError. Ye ek famous gotcha hai.
  • const declaration bina initializer ke → SyntaxError. let allowed hai (value undefined).
  • Function declarations block ke andar (if block mein) — browsers mein weird legacy semantics. Kabhi mat karo; function expression use karo.
  • Global var → window ki property banta hai; global let/const → nahi banta. window.x check karne wale purane code isi pe toot te hain.

15. Interview Questions

EASY var, let aur const mein kya difference hai?

Scope: var function-scoped, let/const block-scoped. Hoisting: teeno hoist hote hain, par var undefined se initialize hota hai jabki let/const TDZ mein rehte hain. Reassignment: const mein nahi ho sakti. Redeclaration: var allow karta hai, baaki nahi. Global pe: var window property banata hai, baaki nahi.

EASY Hoisting kya hai — ek line mein define karo, phir example do.

Hoisting = creation phase mein declarations ka memory mein pehle se register hona. console.log(x); var x = 5; → undefined print hoga (error nahi), kyunki x already undefined se initialize ho chuka tha.

MEDIUM TDZ exactly kab start aur end hoti hai?

TDZ scope ke start se shuru hoti hai (declaration line se nahi!) aur declaration statement execute hone pe khatam. Matlab agar block ki line 1 pe variable access karo aur line 10 pe let ho, toh line 1–9 tak TDZ hai. Isi liye ise "temporal" (time-based) kehte hain, "positional" nahi — agar function declaration ke baad call ho jo variable use karta hai, toh error nahi aayega.

{
  const fn = () => console.log(x); // define ho raha, run nahi
  let x = 5;
  fn(); // ✅ 5 — call ke waqt TDZ khatam ho chuki thi
}
MEDIUM Function declaration aur var — dono hoist hote hain, toh conflict ho toh kaun jeet ta hai?

Creation phase mein function declaration jeetta hai — variable ko undefined nahi, poora function milta hai. Lekin execution phase mein assignment overwrite kar degi. console.log(typeof a); var a = 1; function a(){} → 'function' print hoga.

HARD Lexical Environment aur Variable Environment mein kya farak hai spec level pe?

ES spec mein execution context ke paas dono hote hain: VariableEnvironment var aur function declarations hold karta hai (function-level), jabki LexicalEnvironment let/const bindings ke liye hai aur har block pe naya ban sakta hai. Block enter karne pe LexicalEnvironment change hota hai, VariableEnvironment wahi rehta hai — isi se var block se "leak" karta hai.

SCENARIO Legacy Angular codebase mein ek utility file hai jisme sab kuch var se likha hai aur ek loop ke andar event listeners register hote hain jo galat index print karte hain. Fix strategy kya hogi?

Root cause: shared var binding + closures. Fixes (priority order): (1) var → let in loop headers — har iteration ki fresh binding milegi; (2) agar ES5 target ho toh IIFE se value capture karo; (3) better: listener ko data-attribute ya event.currentTarget se index nikalne do taaki closure dependency hi khatam ho. Sath mein ESLint rule no-var enable karo taaki regression na ho.

16. Revision Notes

⚡ 30-second recall
  • Execution = Creation phase (hoisting) + Execution phase (assignments).
  • var → hoisted + undefined; let/const → hoisted + TDZ; function declaration → fully hoisted.
  • Scope chain = Lexical Environment ka outer pointer chain; lookup andar→bahar.
  • const = reassignment lock, mutation lock nahi.
  • Loop + async callback: var = shared binding (3,3,3), let = per-iteration binding (0,1,2).

S-02

Functions, Arrow Functions & this

Covers: Function declarations/expressions · First-class functions · Arrow functions · this binding rules · call/apply/bind

1. What is it?

Function ek callable object hai — haan, object! Isi liye functions ke paas properties ho sakti hain (fn.name, fn.length) aur inhe variables mein store, arguments mein pass, aur return kiya ja sakta hai — ise first-class functions kehte hain. this ek implicit parameter hai jo har normal function call ke waqt decide hota hai — kaise call kiya, usse, na ki kahan likha, usse.

2. Why do we need it?

First-class functions ke bina RxJS impossible hai — map(x => x * 2) mein tum ek function ko data ki tarah pass kar rahe ho. Angular ka poora reactive ecosystem (operators, callbacks, effects) isi capability pe khada hai. Aur this chahiye taaki ek hi method alag-alag objects pe reuse ho sake bina object ka naam hardcode kiye.

3. What problem does it solve?

  • Code reuse without duplication — ek method, kai instances. this runtime pe sahi instance choose karta hai.
  • Behavior as data — strategies, callbacks, higher-order functions. Bina iske event-driven programming nahi ho sakti.
  • Arrow functions ne this-loss problem solve ki — ES5 era ka var self = this hack khatam.

Yahi "lexical this" ka asli matlab hai.

9. Advanced Example — apna bind polyfill (interview favourite)

Function.prototype.myBind = function(ctx, ...preArgs) {
  const originalFn = this;                    // L1
  return function boundFn(...laterArgs) {     // L2
    if (new.target) {                          // L3: new ke saath call hua?
      return new originalFn(...preArgs, ...laterArgs);  // L4
    }
    return originalFn.apply(ctx, [...preArgs, ...laterArgs]); // L5
  };
};

10. Code Explanation — Line by Line

  • L1: myBind ko fn.myBind(obj) se call kiya jayega, toh implicit binding se this = original function. Use capture kar liya.
  • L2: Ek naya function return — closure se originalFn, ctx, preArgs yaad rakhega. ...preArgs partial application enable karta hai (currying ka base).
  • L3: new.target tabhi defined hota hai jab function new ke saath call ho. Real bind ka rule: new binding, bind ko bhi override kar deta hai — hum wahi replicate kar rahe hain.
  • L5: Normal call pe apply se explicit binding — pre-filled + baad wale args merge karke.

11. Common Mistakes

❌ Galtiyan
  • Class methods ko event handlers mein directly pass karna: btn.addEventListener('click', this.save) — this kho jayega. Fix: this.save.bind(this) ya arrow wrapper, ya class field syntax save = () => {...}.
  • Object literal ke methods ko arrow banana: { greet: () => this.name } — yahan this object nahi, bahar ka scope hai. Methods ke liye shorthand syntax use karo.
  • Arrow function ko new se call karne ki koshish → TypeError, kyunki [[Construct]] hai hi nahi.
  • Arrow ke andar arguments expect karna — nahi milta; rest params (...args) use karo.

12. Best Practices

✅ Production rules
  • Callbacks = arrow functions (lexical this). Object/class methods = normal method syntax.
  • RxJS subscribe/operators mein hamesha arrows — component ka this preserve hota hai.
  • bind render-path mein baar-baar mat karo (har call naya function banata hai) — constructor mein ek baar, ya class field arrow.

13. Performance

Arrow vs normal function — V8 mein call performance same league mein hai. Asli cost: har render/call pe naya function object banana (e.g., template ke andar inline arrows) — ye GC pressure badhata hai aur OnPush comparisons tod deta hai (naya reference har baar). bind bhi har call pe naya bound function banata hai — hot paths mein cache karo.

14. Edge Cases

  • bind ke upar dobara bind → pehla hi jeetta hai; dusra context ignore hota hai.
  • Arrow ko call/apply/bind se context dena → context silently ignore, args pass ho jaate hain.
  • Class bodies hamesha strict mode mein — method ko detach karke call karo toh this = undefined (window nahi), isliye error jaldi milta hai.
  • setTimeout(obj.method, 100) — implicit binding call-site pe hoti hai, pass karne se nahi. Reference pass hua, dot ka context nahi.

15. Interview Questions

EASY Function declaration vs function expression — hoisting difference?

Declaration poori hoist hoti hai (define hone se pehle call kar sakte ho). Expression variable ki tarah treat hoti hai — var se ho toh undefined milega (call pe TypeError), const se ho toh TDZ ReferenceError.

EASY call, apply, bind mein difference?

call(ctx, a, b) — turant call, args alag-alag. apply(ctx, [a,b]) — turant call, args array. bind(ctx, a) — call NAHI karta, naya permanently-bound function return karta hai. Yaad rakhne ka trick: Apply = Array.

MEDIUM Arrow functions mein this "bind" hota hai ya kuch aur? Precisely batao.

Precisely: arrow mein this bind hota hi nahi — arrow ke environment record mein this ki entry nahi hoti. Lookup scope chain se resolve hota hai, jaise koi normal variable. Isliye "arrow captures this" kehna bhi loose hai — wo capture nahi karta, bas resolve outward karta hai. Consequence: call-site kuch bhi ho, arrow ka this kabhi nahi badalta.

MEDIUM Output kya hoga? const obj = { x: 1, getX: () => this.x, getX2() { return (() => this.x)(); } }

obj.getX() → undefined (arrow ka this = module/global scope, object nahi). obj.getX2() → 1 — outer normal method ka this implicit binding se obj hai, andar ka arrow usi ko resolve karta hai. Ye pattern (method + inner arrow) hi Angular components mein hero hai.

HARD new binding, bind ko kaise override karta hai — aur kyun design kiya gaya aisa?

Spec: bound function pe new lagao toh [[Construct]] original target function pe chala jaata hai aur bound this ignore hota hai (partial args phir bhi lagte hain). Design reason: new ka contract hai "fresh object banao" — agar bind ka stale context jeet jaata toh constructor semantics hi toot jaate. Isse partial application + construction dono coexist karte hain: const Btn = El.bind(null, 'button') jaisa factory pattern.

SCENARIO Ek Angular component mein third-party JS library (e.g., chart lib) callback deti hai jisme wo apna context set karti hai this pe (chart instance). Tumhe callback ke andar chart instance BHI chahiye aur component BHI. Kaise karoge?

Normal function use karo taaki library ka this (chart) mile, aur component ko closure variable se pakdo: const self = this; chart.on('click', function(e){ /* this = chart, self = component */ }). Ya cleaner: arrow use karo (component this ke liye) aur chart instance callback params ya bahar ki reference se lo. Interview mein dono batao + trade-off: library APIs jo this pe depend karti hain wahan arrow use karne se wo context permanently inaccessible ho jaata hai.

S-03

Closures

Covers: [[Environment]] slot · Lexical scoping · Module pattern · Private state · Closure memory leaks

1. What is it?

Closure = function + uska birth-place ka lexical environment. Jab function apne define hone ki jagah se bahar jaakar bhi wahan ke variables access kar pata hai — wahi closure hai. Technically: har JS function closure hai, kyunki har function apne [[Environment]] internal slot mein apna janm-sthal ka scope reference rakhta hai.

2. Why do we need it?

JS mein (classes ke #private se pehle) data privacy ka ek hi tareeka tha — closure. Aur async programming ka toh foundation hi hai: jab setTimeout/subscribe ka callback 5 second baad chalta hai, tab tak outer function stack se hat chuka hota hai — callback ko variables closure ki wajah se milte hain. Bina closure ke async JS kaam hi nahi karta.

3. What problem does it solve?

  • State without globals — counter, cache, config kisi function ke andar band, bahar se sirf controlled access.
  • Callbacks ko context dena — async callback ko pata hai kis request/user/component ke liye chal raha hai.
  • Function factories — multiplyBy(3) jaise specialized functions banana.

4. Internal Working

Section 1 se yaad karo: har scope ka ek Lexical Environment hota hai. Jab function create hota hai (call nahi — create!), engine us function object ke hidden [[Environment]] slot mein current lexical environment ka reference daal deta hai. Baad mein jab function call hota hai, uska naya environment isi saved reference ko apna outer banata hai.

Key insight: normally function return hone pe uska environment garbage collect ho jaata hai. Lekin agar koi inner function us environment ko [[Environment]] se pakde hue hai aur wo inner function khud reachable hai — toh environment zinda rehta hai. GC use collect nahi kar sakta. Yahi closure ka "memory" hai — aur yahi memory leak ka source bhi (Section 8).

9. Advanced Example — Debounce (interview + production dono ka king)

function debounce(fn, delay) {                    // L1
  let timerId = null;                             // L2
  return function(...args) {                      // L3
    clearTimeout(timerId);                        // L4
    timerId = setTimeout(() => {                  // L5
      fn.apply(this, args);                       // L6
    }, delay);
  };
}
// Usage: search input pe API spam rokna
searchBox.addEventListener('input', debounce(e => callSearchApi(e.target.value), 300));

10. Code Explanation — Line by Line

  • L1–L2: debounce call hote hi timerId closure environment mein ban gaya. Ye saari future calls mein shared rahega — yahi shared state debouncing possible banati hai.
  • L3: Returned function normal hai, arrow nahi — jaan-boojh ke, taaki caller ka this mile (event listener mein this = element).
  • L4: Har nayi call pichla pending timer cancel karti hai. Closure ki wajah se timerId calls ke beech survive karta hai.
  • L5–L6: Naya timer set. Andar arrow use kiya taaki L3 wale function ka this aur args lexically milen — Section 2 ka concept yahan closure se mila. Do concepts, ek pattern.

11. Common Mistakes

❌ Galtiyan

  • Closure ke andar bade objects (poora component, DOM tree) capture karna jab sirf ek field chahiye thi — GC us poore object ko nahi chhod paata.

S-04

Objects, Arrays, Destructuring, Spread & Rest

Covers: References vs values · Property descriptors · Shallow/deep copy · Destructuring · Spread/Rest · Immutability (OnPush ka foundation!)

1. What is it?

Object = key-value pairs ka dynamic collection (heap pe). Array = special object jiska length auto-manage hota hai aur keys numeric hain. Destructuring = object/array se values nikaalne ka syntax. Spread (... RHS pe) = values ko failana. Rest (... LHS pe) = bachi values ko samet-na. Same symbol, opposite kaam — position decide karti hai.

2. Why do we need it?

Angular ka OnPush change detection aur Signals dono ek hi cheez pe chalte hain: reference equality (===). Naya data = naya reference, warna Angular maanta hai "kuch nahi badla". Spread se immutable updates isi liye Angular dev ki daily bread hai. Ye section skip kiya toh OnPush kabhi samajh nahi aayega.

3. What problem does it solve?

  • Mutation bugs — do jagah same object reference, ek ne badla, dusra bhi "badal gaya" bina bataye. Immutable updates ye class of bugs khatam karte hain.
  • Cheap change detection — deep comparison (har field check) O(n) hai; reference check O(1). Immutability O(1) wala option unlock karti hai.
  • Verbose extraction — const name = user.name; const age = user.age; → const {name, age} = user;

4. Internal Working

Stack vs Heap: primitives (number, string, boolean...) value ke roop mein copy hote hain. Objects/arrays heap pe bante hain, variable mein sirf reference (address) hota hai. const b = a object ke liye = address copy, data copy nahi.

V8 Hidden Classes (Shapes): V8 same structure wale objects ke liye ek shared "hidden class" banata hai — property access phir direct memory offset se hota hai (C++ struct jitna fast). Property add/delete karne se object nayi hidden class pe migrate hota hai; dynamic shapes engine ko "megamorphic" slow path pe dhakel deti hain.

Spread internals: {...obj} own enumerable properties ko iterate karke naye object pe copy karta hai — shallow! Nested objects ke sirf references copy hote hain. [...arr] iterator protocol use karta hai (isi liye Set/Map/string bhi spread hote hain).

11. Common Mistakes

❌ Galtiyan
  • Spread ko deep copy samajhna — #1 production bug source. Nested update = nested spread.
  • OnPush component mein @Input object ko parent mein mutate karna, phir "UI update nahi ho raha" ka ticket banana.
  • arr.sort()/reverse()/splice() ko non-mutating samajhna. ES2023 ke toSorted/toReversed/toSpliced/with non-mutating versions hain.
  • Destructuring defaults null pe lagne ki umeed: const {x = 5} = {x: null} → x = null! Default sirf undefined pe.
  • Bade arrays pe har chhoti update pe deep clone — O(n) × har keystroke = janky UI.

12. Best Practices

✅ Production rules
  • State updates: sirf changed path pe naye objects (structural sharing) — baaki branches ka reference reuse. Perf + correctness dono.
  • Function options ke liye object param + destructuring + defaults — positional args 3 se zyada hue toh refactor.
  • Readonly types lagao: ReadonlyArray<T>, Readonly<T> — TypeScript compile pe hi mutation pakad lega.
  • Deep clone chahiye toh structuredClone; JSON trick sirf JSON-safe data pe.

13. Performance

  • Spread/copy O(n) hai — 10k items ki list pe har update full copy tolerable hai, per-frame animation loop mein nahi.
  • Structural sharing yaad rakho: list.map(...) mein unchanged items ka same reference return karo — @for + track tab DOM reuse kar payega.
  • Hidden classes: objects ko same shape mein initialize karo (constructor mein saari properties), delete avoid karo — V8 fast path pe rahega.
  • Array holes ([1,,3]) V8 ko slow "holey" mode mein daal dete hain — avoid.

14. Edge Cases

  • {...null} aur {...undefined} → {} (no error). [...null] → TypeError (iterator chahiye).
  • Spread sirf own enumerable properties copy karta hai — getters evaluate ho jaate hain (value ban jaate hain), prototype chain ignore.
  • const {a, b} = arr array pe bhi chalta hai (index properties), aur const {length} = 'hello' → 5.
  • Rest element destructuring mein last hi ho sakta hai: const [a, ...rest, b] → SyntaxError.
  • Already-declared variables pe object destructuring ko parens chahiye: ({x} = obj); warna { block samjha jaata hai.

Covers: [[Prototype]] · __proto__ vs prototype · Prototype chain · new operator · class syntax internals · super · #private fields

1. What is it?

Har JS object ke paas ek hidden link hota hai — [[Prototype]] — jo kisi dusre object ko point karta hai. Property nahi mili toh engine is link pe chala jaata hai, phir uske link pe... jab tak null na mile. Isi chain-walk ko prototypal inheritance kehte hain. Class isi machinery ke upar ek saaf syntax hai — naya inheritance model nahi hai.

2. Why do we need it?

Memory sharing. 1000 user objects banao — agar har object ke paas apna greet function ho toh 1000 function copies. Prototype pe rakho toh ek copy, sab share karte hain. Java se aane wale devs ke liye: prototype ≈ class ka method table, bas ye runtime pe live object hai jise modify kiya ja sakta hai.

3. What problem does it solve?

  • Method duplication — shared behavior ek jagah, instances lightweight.
  • Extensibility — built-ins ko polyfill karna (Array.prototype.flat purane browsers mein) isi se possible hua.
  • Classes ne developer-experience problem solve ki — constructor functions + manual prototype wiring error-prone tha; class/extends/super ne use declarative bana diya.

  • L3: ??= — nullish assignment; pehli call pe hi banata hai. (Note: DI wale frameworks mein singleton container se lo, manual nahi — pattern samajhne ke liye hai.)
  • L4–L5: Accessor properties prototype pe defined hoti hain (data nahi, get/set functions). Validation setter mein — invariant object ke andar enforce hota hai.

11. Common Mistakes

❌ Galtiyan
  • prototype aur [[Prototype]] ko same samajhna. Dog.prototype ≠ Dog ka apna prototype. Dog khud function hai, uska [[Prototype]] = Function.prototype (ya extends mein Parent).
  • Arrow function ko prototype method banana — this lexical ho jayega, instance nahi milega.
  • Derived class constructor mein this ko super() se pehle touch karna → ReferenceError. Kyun: this base constructor banata hai, derived ko wo super() se milta hai.
  • Built-in prototypes ko production mein modify karna (monkey-patching) — collisions + future spec breakage (smooshgate yaad hai?).
  • Class fields (x = 1) instance pe bante hain, methods prototype pe — save = () => {} field hai, har instance pe nayi copy. 1000 instances = 1000 functions. Jaan ke use karo (event handlers), anjaane mein nahi.

12. Best Practices

✅ Production rules
  • Composition > inheritance — Angular mein khaaskar. 1 level base class theek, 3-level hierarchy = design smell.
  • Privacy: JS runtime privacy chahiye toh #field; team-convention kaafi hai toh TS private.
  • Prototype check: instanceof for classes, Object.getPrototypeOf for introspection, in vs hasOwnProperty jaan ke choose karo (chain include karna hai ya nahi).
  • Object.create(null) for pure dictionaries — no inherited keys, no prototype-pollution surface.

13. Performance

Prototype chain lookup V8 inline caches se practically free hai — agar shapes stable hain (Section 4 cross-ref). Lambi chains (5+) IC misses pe hi cost karti hain. Object.setPrototypeOf existing object pe = performance killer (saare ICs invalidate) — prototype creation time pe set karo (Object.create/new), baad mein kabhi nahi. Methods prototype pe rakhna memory win hai; per-instance arrows sirf jahan this-binding chahiye.

14. Edge Cases

  • instanceof prototype chain check karta hai, constructor nahi: Object.setPrototypeOf(x, Dog.prototype) ke baad x instanceof Dog → true, bina Dog constructor chale.
  • class methods mein super ka [[HomeObject]] static bind hota hai definition pe — method ko copy karke dusre object pe rakho, super phir bhi purani chain use karega.
  • Object.create(null) wale object pe toString/hasOwnProperty nahi hote — Object.hasOwn(obj, key) (ES2022) use karo.
  • Derived class mein constructor omit karo toh implicit constructor(...args){ super(...args) } milta hai.
  • Getters/setters spread mein values ban jaate hain — accessor copy nahi hota (Section 4 cross-ref).

Trade-off batao: inheritance one-shot reuse mein fast tha, par axis-of-change multiple hone pe composition jeet-ti hai.

S-06

Event Loop — Call Stack, Web APIs, Microtasks & Macrotasks

Covers: Call Stack · Web APIs · Task (Macrotask) Queue · Microtask Queue · Rendering pipeline · Zone.js connection

🎯 Angular ke liye sabse important section

Zone.js exactly in queues ko patch karke Angular ko batata hai "async kaam khatam hua, ab change detection chalao". Event loop samjhe bina Zone.js, runOutsideAngular, zoneless Angular, ya "ExpressionChangedAfterItHasBeenChecked" — kuch bhi first-principles se samajh nahi aayega.

1. What is it?

JavaScript single-threaded hai — ek time pe ek hi kaam. Event Loop wo orchestrator hai jo decide karta hai: stack khali hua, ab kaunsa pending kaam chale. Components: Call Stack (abhi kya chal raha hai), Web APIs (browser ke background workers — timer, network, DOM events), Task/Macrotask Queue, aur Microtask Queue (VIP line).

2. Why do we need it?

Single thread pe agar fetch ka wait bhi thread block karta, toh 2-second API call = 2 second frozen UI — no clicks, no scroll, no render. Event loop model se JS delegate karta hai: "browser, tu network sambhal, result aaye toh callback queue mein daal dena, main tab tak aur kaam karta hoon." Concurrency without threads (aur without locks/race conditions ka dard jo Java devs jaante hain).

3. What problem does it solve?

  • Non-blocking I/O — network/timers/disk UI freeze nahi karte.
  • Ordering guarantees — microtask vs macrotask priority se predictable execution order milta hai (Promise chains hamesha timers se pehle).
  • Frameworks ko hook point — Zone.js/React scheduler isi model ke gaps mein apna kaam ghusate hain.

4. Internal Working — Poora model

Loop ka ek cycle (simplified spec):

  1. Macrotask queue se ek task uthao, run karo (stack khali hone tak).
  2. Microtask checkpoint: microtask queue ko poora drain karo — aur jo microtasks is dauraan naye bane, unhe bhi! (Isi se infinite microtask loop page hang kar sakta hai.)
  3. Agar render ka time hai (~16.6ms pe): style → layout → paint. requestAnimationFrame callbacks render se theek pehle.
  4. Repeat.

Golden rule: Har ek macrotask ke baad saare microtasks. Ratio 1 : all.

11. Common Mistakes

❌ Galtiyan

  • Promise callback ko macrotask bolna interview mein — .then microtask hai. Ye galti screening round mein hi filter kar deti hai.
  • "Async code parallel chalta hai" — JS mein concurrent hota hai (interleaved), parallel nahi (Web Workers ko chhod ke). Web APIs ka kaam (network, timer) browser threads pe hota hai, tumhara callback hamesha main thread pe.
  • Heavy sync loop ko async function mein daal ke sochna ki non-blocking ho gaya — async keyword thread nahi banata; andar ka sync code utna hi blocking hai.

12. Best Practices

✅ Production rules
  • Long tasks (>50ms) todo: chunking + macrotask yield, ya requestIdleCallback, ya Web Worker mein bhejo (serialization cost tol ke).
  • DOM read/write batching: visual updates requestAnimationFrame mein — render se theek pehle, layout thrashing se bachaav.
  • Angular: high-frequency listeners runOutsideAngular; state settle hone pe zone.run se wapas.
  • Ordering pe depend karna hai toh explicit karo (await chains) — "ye timer usse pehle chalega" wali implicit assumptions PR mein flag karo.

S-07

Promise & Async/Await

Covers: Promise states · then chaining internals · Error propagation · Combinators (all/allSettled/race/any) · async/await desugaring · Angular APP_INITIALIZER/resolvers

1. What is it?

Promise = future value ka placeholder object. Ek aisa dabba jisme value abhi nahi hai, par aane wali hai (ya error aayega). Teen states: pending → fulfilled (value ke saath) ya rejected (reason ke saath). Fulfilled/rejected dono ko settled kehte hain — aur settled state immutable hai, ek baar set toh hamesha wahi. async/await isi Promise machinery ke upar syntax hai jo async code ko sync jaisa dikhata hai.

2. Why do we need it?

Promises se pehle callbacks the — aur unke saath callback hell: nested pyramids, error handling har level pe manually, aur inversion of control ka trust issue (callback kisi third-party fn ko diya — wo use 0 baar bhi call kar sakta hai, 5 baar bhi). Promise ne contract diya: exactly once settle hoga, state immutable hogi, aur error chain se neeche automatically flow karegi.

3. What problem does it solve?

  • Pyramid of doom → flat .then chains → aur bhi flat async/await.
  • Error handling ka duplication → ek .catch/ek try/catch poori chain cover karta hai.
  • Trust problem → resolve/reject sirf ek baar effective; baad ke calls silently ignore.
  • Composition → parallel (all), race, partial failure (allSettled) — callbacks se ye likhna nightmare tha.

4. Internal Working

Promise object ke andar (spec ke internal slots):

  • [[PromiseState]] — pending / fulfilled / rejected
  • [[PromiseResult]] — value ya rejection reason
  • [[PromiseFulfillReactions]] / [[PromiseRejectReactions]] — pending state mein registered .then callbacks ki lists

Settle hone pe kya hota hai: reactions ko microtask queue mein daala jaata hai (Section 6 direct connection). Agar promise pehle se settled hai aur tum .then lagao — callback phir bhi sync nahi chalta, microtask mein hi jaata hai. Ye guarantee ("always async") race conditions ki poori category khatam karti hai.

.then ka asli magic: har .then ek naya promise return karta hai. Us naye promise ki fate tumhare callback ke return pe depend karti hai:

5. Execution Flow — async/await desugared

async/await generator + promise state machine ka sugar hai. Engine tumhare function ko suspend/resume points pe todta hai:

// Tumne likha:
async function getUser(id) {
  const res  = await fetch(`/api/users/${id}`);
  const user = await res.json();
  return user.name;
}

// Engine ne conceptually ye samjha:
function getUser(id) {
  return fetch(`/api/users/${id}`)      // await #1 = .then boundary
    .then(res => res.json())             // await #2 = agla .then
    .then(user => user.name);           // return = final fulfillment value
}
  getUser(7) call
    │ sync chalega pehle await tak (fetch fire hua)
    │ AWAIT → function SUSPEND, caller ko turant pending promise wapas
    ▼
  ...event loop apna kaam karta rahega (UI zinda!)...
    ▼
  fetch settle → resume-callback MICROTASK mein
    │ function wahi se resume, res mila
    │ agla await → phir suspend...
    ▼
  return user.name → getUser ka promise fulfilled

Rules jo isse nikalte hain: (1) async function hamesha promise return karta hai — return 5 bhi Promise.resolve(5) ban jaata hai; (2) throw = rejected promise; (3) await sirf current function suspend karta hai, thread nahi — caller aage badh jaata hai.

9. Advanced Example — retry with exponential backoff + timeout

function withTimeout(promise, ms) {                              // L1
  return Promise.race([
    promise,
    new Promise((_, reject) =>
      setTimeout(() => reject(new Error('Timeout')), ms))    // L2
  ]);
}

async function retryFetch(url, { retries = 3, baseDelay = 300 } = {}) {
  for (let attempt = 0; attempt <= retries; attempt++) {       // L3
    try {
      const res = await withTimeout(fetch(url), 5000);         // L4
      if (!res.ok) throw new Error(`HTTP ${res.status}`);      // L5
      return await res.json();                                  // L6
    } catch (err) {
      if (attempt === retries) throw err;                       // L7
      const delay = baseDelay * 2 ** attempt;                   // L8
      await new Promise(r => setTimeout(r, delay));             // L9
    }
  }
}

10. Code Explanation — Line by Line

  • L1–L2: race — jo pehle settle ho, wahi jeete. Timeout promise reject karta hai ms pe. Note: original fetch abort nahi hota — race sirf result ignore karta hai (Section 6 ka lesson). Production mein AbortController + fetch(url, {signal}) se actual cancellation add karte hain.
  • L3: Loop-based retry — recursion se cleaner, stack safe.
  • L4: await = yahan reject hua toh seedha catch mein — try/catch async errors ko sync jaisa handle karta hai.
  • L5: Fetch ka famous gotcha: 404/500 pe reject nahi hota (sirf network failure pe hota hai). HTTP errors manually throw karne padte hain. Interview mein ye zaroor bolo.
  • L6: return await — normally redundant, lekin try block ke andar zaroori hai: bina await ke res.json() ka rejection is catch mein nahi aayega (function return ho chuka hoga). Subtle, senior-level detail.
  • L7–L9: Last attempt fail = error upar propagate. Warna exponential delay (300, 600, 1200ms) — "promisified setTimeout" pattern se await-able sleep.

11. Common Mistakes

❌ Galtiyan
  • Sequential await jab parallel possible tha: const a = await f1(); const b = await f2(); — independent calls, phir bhi total = t1 + t2. Fix: pehle fire karo, phir await: const [a, b] = await Promise.all([f1(), f2()]).
  • Floating promises: this.save() (async) call kiya, na await, na catch — error silently unhandledrejection mein gaya. ESLint no-floating-promises lagao.
  • forEach + async: items.forEach(async x => await save(x)) — forEach promises ka wait nahi karta; loop "khatam" pehle, saves baad mein. Sequential chahiye toh for...of, parallel toh map + Promise.all.
  • .then(fn).catch(fn) vs .then(fn, fn) confuse karna — dusre form mein usi then ke fn ka error catch nahi hota.
  • Constructor executor mein async function dena: new Promise(async (res, rej) => ...) — executor ke throw silently kho jaate hain (rejected promise return hota hai jise koi nahi sunta).

Modules, Memory Management & Garbage Collection

Covers: ESM vs CommonJS · Live bindings · Tree shaking · Stack vs Heap · Mark-and-Sweep · Generational GC · SPA memory leaks · WeakMap/WeakRef

1. What is it?

Module = apne scope wali file jo explicitly batati hai kya export karti hai aur kya import karti hai. Memory management = allocation (values banana) + retention (kaun kise zinda rakh raha hai) + release. JS mein release automatic hai — Garbage Collector wo values free karta hai jo unreachable ho gayi hain. Key word: unreachable, unused nahi. GC ko nahi pata tum kya use karoge — use sirf ye pata hai kis tak pahuncha ja sakta hai.

2. Why do we need it?

Modules ke bina: script tags ka order-dependent jenga, global namespace ki dharmshala, aur "kaun kis pe depend karta hai" ka koi record nahi. Memory management ke bina: C/C++ wala manual free() — use-after-free, double-free, leaks. JS ne dono automate kiye — lekin SPA era mein automatic GC ka matlab "leak-proof" nahi hai. Angular app ghanton open rehti hai; ek chhota retention bug × hours = crashed tab. Isliye ye section sirf theory nahi, production survival hai.

3. What problem does it solve?

  • ESM: dependency graph static hai (imports top pe, strings compile-time known) → bundler poora graph bana ke tree shaking kar sakta hai — unused exports final bundle se bahar. Angular ka providedIn: 'root' isi liye design hua: service kahin inject nahi hui toh shake out ho jaati hai.
  • GC: memory safety by default — dangling pointers ki poori category nonexistent.
  • Weak collections (WeakMap/WeakSet/WeakRef): "cache rakho par zinda mat rakho" problem solve karte hain.

11. Common Mistakes

❌ Galtiyan
