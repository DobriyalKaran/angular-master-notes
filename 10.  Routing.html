S-01

The Router — Core Engine

Covers: provideRouter vs RouterModule · Routes config · router-outlet · redirects · pathMatch full/prefix · wildcard · title

1. What is it? (Kya hai ye?)

Angular Router ek client-side navigation engine hai. Kaam: browser ke URL ko dekhkar decide karna ki screen pe kaunsa component render hoga — bina server se naya HTML page maange. Link click → URL badla (History API se) → Router URL ko ek Route config se match karta hai → sirf <router-outlet> ke andar wala component swap. Baaki page (header/sidebar) as-is. Isliye navigation instant feel hoti hai.

2. Why do we need it? (Zaroorat kyun hai?)

Ek SPA by default single HTML file hai jiske andar 50 "screens" ho sakte hain. Bina Router ke tumhe khud *ngIf ki gigantic ladder likhni padegi — "agar state === 'cart' toh CartComponent". Ye maintainable nahi, aur browser ko navigation ka pata hi nahi chalta.

3. What problem does it solve?

Bina Router — 4 problems
  • No shareable URLs: user cart pe hai par URL abhi bhi / — link bheja toh dost home pe.
  • Back/Forward toota: browser ko navigation ka pata nahi, back button app se bahar phenkता hai.
  • No deep-link / refresh: refresh pe in-memory state gayab, wapas home.
  • No structure: lazy loading, guards, nested layouts — sab manually.
Router deta hai

URL ↔ component ka single source of truth. History, deep-linking, guards, code-splitting — sab declaratively, free.

4. Internal Working (Engine ke andar kya hota hai?)

Har navigation ek fixed pipeline chalati hai (ratne layak — interview favourite):

  1. URL → UrlTree: UrlSerializer raw URL string ko structured tree mein parse karta hai (segments, params, query, fragment).
  2. Recognize (matching): Router Routes array ko top-to-bottom scan karke pehla match dhoondta hai. Yahin CanMatch guards aur lazy loadChildren chalte hain.
  3. Build RouterState: matched routes se ActivatedRoute ka tree banta hai (params, data ke saath).
  4. Guards check: CanDeactivate → CanActivate/CanActivateChild. Koi false/UrlTree de toh cancel/redirect.
  5. Resolve: resolvers async data pre-fetch karte hain; sab settle hone tak navigation wait.
  6. Activate: purane components destroy, naye ko outlet mein render, Location URL sync.
Diagram — Router pipeline
  Browser URL          UrlSerializer        Recognizer          RouterState
  /user/42?tab=x  ─▶  parse → UrlTree  ─▶  match Routes[]  ─▶  ActivatedRoute tree
                                              │
                                              ▼
                    Guards  ─▶  Resolvers  ─▶  Activate
              (CanMatch · CanDeactivate · CanActivate · Resolve)
                                              │
                                              ▼
                    <router-outlet> renders component  +  Location syncs URL

5. Execution Flow

Yeh exact sequence har router.navigate() pe fire hoti hai — Router events se observe ki ja sakti hai (S-10):

NavigationStart
  └─ RouteConfigLoadStart / RouteConfigLoadEnd   (agar route lazy hai)
  └─ RoutesRecognized                            (URL match, RouterState ready)
  └─ GuardsCheckStart / GuardsCheckEnd           (saare guards pass ya cancel)
  └─ ResolveStart / ResolveEnd                   (resolvers ne data diya)
  └─ ActivationStart / ChildActivationStart
NavigationEnd                                    ✅ success — outlet updated
  (ya NavigationCancel / NavigationError agar beech mein ruka)

6. Architecture (Moving parts)

RouterCentral service — navigate(), navigateByUrl(), events, current state.
RoutesConfig array — URL patterns ka component/child/guard mapping. Single source of truth.
RouterOutletPlaceholder directive — yahin matched component mount hota hai.
ActivatedRouteActive route ki info — params, queryParams, data — observables + snapshot.
UrlSerializerURL string ↔ UrlTree. Custom serializer possible.
LocationBrowser History API ka wrapper — back/forward, pushState.
RouterLink / RouterLinkActiveDeclarative navigation + active-link styling directives.

7. Real Project Example (Angular context)

Modern standalone Angular (v17+) — har naye project ka boilerplate:

app.routes.ts
import { Routes } from '@angular/router';
import { Home } from './home/home';

export const routes: Routes = [
  { path: '', component: Home, title: 'Dashboard' },
  { path: 'products', loadComponent: () => import('./products/list').then(m => m.ProductList) },
  { path: 'products/:id', loadComponent: () => import('./products/detail').then(m => m.ProductDetail) },
  { path: 'login', component: Login },
  { path: '**', component: NotFound },   // wildcard — hamesha LAST
];
app.config.ts
import { ApplicationConfig } from '@angular/core';
import { provideRouter, withComponentInputBinding, withViewTransitions } from '@angular/router';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(
      routes,
      withComponentInputBinding(),   // route params ko @Input se bind karo
      withViewTransitions(),         // v17+ smooth cross-fade
    ),
  ],
};
app.html
<nav>
  <a routerLink="/products" routerLinkActive="active">Products</a>
  <a routerLink="/login" routerLinkActive="active">Login</a>
</nav>
<router-outlet />   <!-- yahan matched component render hoga -->
Module vs Standalone

Purana: RouterModule.forRoot(routes) ek AppRoutingModule mein. Modern: provideRouter(routes) in app.config.ts — tree-shakable, less boilerplate. Child ke liye forChild ki zaroorat nahi; standalone lazy routes seedhe array export karte hain.

9. Advanced Example — pathMatch & redirects

typescript
const routes: Routes = [
  { path: '', redirectTo: '/dashboard', pathMatch: 'full' },
  { path: 'legacy-users', redirectTo: 'users', pathMatch: 'prefix' },
  { path: 'dashboard', component: Dashboard },
  { path: 'users', component: Users },
];
pathMatch: full vs prefix

'prefix' (default): match ho jaata hai agar URL shuru hota hai is path se. 'full': pura URL exactly path ke barabar. Empty-path redirect ke saath hamesha pathMatch: 'full' — kyunki '' har URL ka prefix hai, prefix use kiya toh har navigation redirect loop mein phans jaati hai. Classic trap.

10. Code Explanation — Line by Line

  1. path: '' → empty path = default route. URL / pe render.
  2. path: 'about' → /about. Config mein leading slash kabhi nahi.
  3. path: '**' → wildcard, koi bhi unmatched URL yahan. Order matters — hamesha last, warna sab isi pe match.
  4. redirectTo + pathMatch:'full' → empty URL ko dashboard pe safely bhejta hai bina loop ke.

11. Common Mistakes

Router basics traps
  • Wildcard ** config ke beech mein → neeche wale routes dead.
  • Config mein path: '/about' (leading slash) → runtime error. Paths relative.
  • Empty-path redirect pe pathMatch:'full' bhoolna → infinite redirect loop.
  • <router-outlet> template mein daalna bhool jaana → match hota hai, render nahi.
  • Standalone component ke imports mein RouterLink/RouterOutlet na add karna → silently dead.

12. Best Practices

Do this
  • Har route pe title — document title auto-update (a11y + SEO).
  • Feature-wise route files (products.routes.ts), ek giant array nahi.
  • Path strings ko constants/enum mein — magic strings avoid.
  • withComponentInputBinding() enable karo — subscription boilerplate hatta.

13. Performance

Router config ka sabse bada perf lever lazy loading hai (S-06) — initial bundle chhota = fast LCP/TTI. Eager routes minimal rakho; sirf core (home, shell) main bundle mein. withViewTransitions() perceived speed badhata hai bina real cost ke.

14. Edge Cases

Trailing slash / caseDefault matcher case-sensitive, trailing slash matters — normalize ya custom UrlMatcher.
Named (aux) outlets<router-outlet name="side"/> + outlet:'side' route. URL: /main(side:chat).
SSR + routingServer pe bhi guards/resolvers chalte hain — browser-only code ko isPlatformBrowser se guard karo.

15. Interview Questions

EASY SPA mein Router ki zaroorat kya hai?

SPA ek hi HTML page hai jismein kai screens. Router URL ko component se map karta hai bina full reload ke — shareable URLs, back/forward, deep-linking, refresh-safety. Bina Router ke sab manually *ngIf ladder se karna padta aur toot jaata.

EASY routerLink vs href?

href full page reload (server round-trip). routerLink client-side navigation — no reload, state preserved, History API se URL update. Accessibility ke liye real <a> render hota hai isliye new-tab bhi kaam karta hai.

MEDIUM Empty-path redirect pe pathMatch:'full' kyun zaroori?

'' har URL ka prefix hai. Default 'prefix' use kiya toh har navigation empty-path route pe match hoke redirect kar degi → infinite loop. 'full' ensure karta hai ki match tabhi ho jab pura URL exactly khaali ho.

HARD Navigation lifecycle ke internal steps batao.

URL → UrlSerializer (UrlTree) → Recognize (CanMatch + lazy load, Routes match) → RouterState build → Guards (CanDeactivate → CanActivate/Child) → Resolve → Activate (outlet render + Location sync). Events isi flow ko mirror karte hain: NavigationStart … RoutesRecognized … GuardsCheck … Resolve … NavigationEnd.

SCENARIO Routes match ho rahe hain (console mein event dikhta hai) par screen blank hai. Kyun?

Template mein <router-outlet> missing hai (ya standalone component ke imports mein RouterOutlet add nahi kiya). Match hota hai par render karne ki jagah hi nahi. Outlet add karo.

16. Revision Notes

⚡ 30-second recall
  • Pipeline: URL → UrlSerializer → Recognize (CanMatch, lazy) → RouterState → Guards → Resolve → Activate.
  • Setup: provideRouter(routes, ...features) + <router-outlet/>. Legacy: RouterModule.forRoot.
  • Wildcard ** last; empty-redirect pathMatch:'full'; config paths relative (no leading slash).
  • Har route title; feature-wise route files; lazy for scale.

S-02

Navigation

Covers: routerLink (array form) · routerLinkActive · router.navigate vs navigateByUrl · NavigationExtras · relativeTo · state

1. What is it? (Kya hai ye?)

Navigation = "user ko doosri route pe le jaana". Router ise History API (pushState) se karta hai — page reload nahi hota, URL badalta hai, outlet swap. Do tarike: declarative (template links) aur programmatic (TS code).

2. Why do we need it? (Zaroorat kyun hai?)

Har navigation user click nahi hoti. Kabhi navigation ek logic ka result hai — form submit success, login ke baad, guard ke andar, timer. Iske liye code se navigate karna padta hai. Aur clicks ke liye accessible links chahiye. Dono cases ke liye alag tools.

3. What problem does it solve?

Raw <a href> full reload karta hai — SPA ka poora fayda khatam. Aur window.location.href = ... se bhi reload + koi Router awareness nahi (guards/events skip). Router ki navigation in dono ko fix karti hai: no reload, guards/events intact, History managed.

4. Internal Working

routerLink ek directive hai jo click intercept karke router.navigateByUrl(urlTree) call karta hai (default left-click, no modifier). router.navigate([...]) segments array ko UrlTree mein serialize karke same pipeline (S-01) chalata hai. Dono ek Promise<boolean> return karte hain — true success, false = guard/error ne roka.

5. Execution Flow

click / navigate()  ─▶  build UrlTree  ─▶  [ same navigation pipeline as S-01 ]
                                              │
                       Promise<boolean>  ◀────┘   true = success · false = blocked

6. Architecture — declarative vs programmatic

navigate([...])Segments array · relative navigation (relativeTo) · auto-encode · dynamic params ke liye behtar.
navigateByUrl('...')Complete URL string · hamesha absolute (root se) · jab pura URL already string ho.

7. Real Project Example (Angular context)

login-form.ts
import { Router } from '@angular/router';
import { inject } from '@angular/core';

export class LoginForm {
  private router = inject(Router);

  async onLogin() {
    const ok = await this.auth.login(this.form.value);
    if (ok) this.router.navigate(['/dashboard'], { queryParams: { welcome: 1 } });
  }
}

9. Advanced Example — NavigationExtras & state

typescript
this.router.navigate(['../sibling'], {
  relativeTo: this.route,        // current route ke relative
  queryParams: { page: 2 },
  queryParamsHandling: 'merge',  // existing query params ke saath merge
  fragment: 'section-3',
  state: { from: 'checkout' },   // URL mein dikhta NAHI — history.state mein
  replaceUrl: false,             // true → history replace (back skip)
  skipLocationChange: false,     // true → URL bar update nahi
});
state — hidden data

URL mein na dikhne wala data (object, "kahan se aaye") pass karne ke liye state. Destination mein padho: this.router.getCurrentNavigation()?.extras.state (turant, navigation ke dauraan) ya history.state (baad mein, refresh-safe).

10. Code Explanation — Line by Line

  1. ['../sibling'] + relativeTo → current route se ek level upar jaake sibling. Bina relativeTo ke Angular root se resolve karega.
  2. queryParamsHandling:'merge' → naye query params purane ke saath merge (na dena → purane ud jaate hain).
  3. state → History API mein store, URL clean rehta hai.
  4. replaceUrl:true → current history entry replace, back button is step ko skip karega.

11. Common Mistakes

Traps
  • [routerLink] vs routerLink: brackets = expression (array/variable), bina brackets = literal string. routerLink="[product, 5]" literally galat URL.
  • Ek <a> pe routerLink + href dono → href full reload trigger.
  • routerLinkActive bina {exact:true} root pe → / har jagah active dikhega.
  • state ko sole source banana → user direct URL khole toh undefined. Fallback rakho.

12. Best Practices

Do this
  • Template navigation → routerLink (real <a href>, right-click new-tab kaam karta hai). router.navigate sirf logic-driven cases.
  • Relative navigation ke liye hamesha relativeTo: this.route.
  • Dynamic params array form se — manual string concat se encoding bugs.

13. Performance

Navigation itself lightweight hai. Watch out: rapid double-clicks concurrent navigations trigger karte hain (nayi purani ko cancel karti hai) — buttons ko debounce/disable karo. skipLocationChange sirf special cases mein (modals-as-routes), warna back button toota.

14. Edge Cases

Empty navigate([])Same route, sirf query/fragment update — filters ke liye common.
Guard returns UrlTree mid-navOriginal navigate() ka Promise false resolve karta hai; redirect alag navigation.
External URLRouter internal routes ke liye hai — external links ke liye normal <a href> ya window.open.

15. Interview Questions

EASY navigate vs navigateByUrl?

navigate([...]) segments array leta hai, relative navigation + auto-encoding support karta hai. navigateByUrl('...') ek complete absolute URL string leta hai. Dynamic params ke liye navigate, pura URL already string ho toh navigateByUrl. Dono Promise<boolean> return karte hain.

MEDIUM URL mein na dikhने wala data destination ko kaise bhejein?

NavigationExtras.state use karo — navigate(['/x'], { state: {...} }). Destination mein getCurrentNavigation()?.extras.state (turant) ya history.state (baad mein). Refresh pe survive karta hai (History API) par direct URL open pe absent — fallback rakho.

SCENARIO Ek "delete" button click pe router.navigate se list pe wapas jaate ho, par back button user ko deleted item ke detail page pe le jaata hai (jo ab error deta hai). Fix?

Navigation pe replaceUrl: true do — detail page ki history entry replace ho jaayegi, back button us dead page pe wapas nahi jaayega.

16. Revision Notes

⚡ 30-second recall
  • Declarative routerLink (array form for dynamic) · programmatic navigate/navigateByUrl.
  • NavigationExtras: relativeTo, queryParams, queryParamsHandling, fragment, state, replaceUrl, skipLocationChange.
  • routerLinkActive + {exact:true} for root. state = hidden data (history.state).

S-03

Route Parameters

Covers: :id syntax · paramMap observable vs snapshot · component reuse trap · withComponentInputBinding · matrix params

1. What is it? (Kya hai ye?)

Route params URL path ke required, positional parts hain jo ek resource identify karte hain. { path: 'user/:id' } mein :id placeholder. /user/42, /user/99 — same component, alag data.

2. Why do we need it? (Zaroorat kyun hai?)

1000 users ke liye 1000 routes nahi likhne — ek pattern, infinite values. URL mein resource id hone se page deep-linkable + shareable ho jaata hai.

3. What problem does it solve?

Bina route params, tum resource id ko service/state mein rakhte, par refresh pe woh gayab aur URL share karne se kaam nahi chalta. Route param ise URL mein bake karta hai — refresh-safe, shareable, bookmarkable.

4. Internal Working (Component reuse ka raaz)

Ye samajhna zaroori hai

Jab tum /user/42 se /user/43 jaate ho, route config same hai — sirf param badla. Angular performance ke liye component ko reuse karta hai (destroy+recreate NAHI). Iska matlab constructor/ngOnInit dobara nahi chalte. Isliye snapshot purana reh jaata hai, par paramMap observable naya value emit karta hai.

Diagram — snapshot staleness on reuse
 /user/42 ─┐
           ├─▶  UserDetail  (reused, NOT re-created)
 /user/43 ─┘         │
                     ├─▶  snapshot.paramMap  →  stuck at 42   ✕
                     └─▶  paramMap$ (observable)  →  emits 42 → 43   ✓

5. Execution Flow

navigate /user/43  ─▶  same route config matched  ─▶  reuse component instance
                                                          │
                              ngOnInit does NOT run again ─┘
                              paramMap$ emits '43'  ─▶  reload data

6. Architecture — reading params

route.snapshot.paramMap.get('id')Ek baar read, phir kabhi update nahi. Fresh-component cases ke liye.
route.paramMap (Observable)Param badalne pe re-emit. Same-route param change ke liye zaroori.
input() + withComponentInputBinding()Modern — param seedhe input mein bind, reactive, no subscription.

7. Real Project Example (Angular context)

user-detail.ts
import { ActivatedRoute } from '@angular/router';
import { inject } from '@angular/core';

export class UserDetail {
  private route = inject(ActivatedRoute);

  ngOnInit() {
    this.route.paramMap.subscribe(params => {
      const id = params.get('id');   // har naye id pe dobara chalega
      this.loadUser(id);
    });
  }
}

9. Advanced Example — Component Input Binding (v16+)

user-detail.ts (signals)
import { input, computed } from '@angular/core';

export class UserDetail {
  id = input.required<string>();                 // route ':id' auto-binds here
  user = computed(() => this.userService.byId(this.id()));  // reactive
}
Kya-kya bind hota hai

Input name se match: path params (:id), query params, matrix params, aur route data/resolve keys. Priority: data → path → matrix → query. Signal input() ke saath snapshot-vs-observable ki poori bahas khatam.

10. Code Explanation — Line by Line

  1. input.required<string>() — ek required signal input; Router isme :id ka value inject karta hai (binding feature on hone par).
  2. Jab route param badalta hai, signal update hota hai — bina ngOnInit/subscription.
  3. computed() automatically re-run hota hai jab id() change ho — reactive data derivation.

11. Common Mistakes

Traps
  • Same-route param change pe snapshot pe bharosa → stale data (#1 interview trap).
  • Params hamesha strings — get('id') se '42', number nahi. Number(id) karo.
  • Manual subscribe() ka unsubscribe bhoolna → leak. takeUntilDestroyed()/toSignal()/input binding.
  • Optional cheez ke liye route param — route params required hain; optional ko query param banao.

12. Best Practices

Do this
  • withComponentInputBinding() + signal input() — cleanest, reactive, leak-free.
  • Agar same route pe param change possible hai → observable (snapshot nahi).
  • Params ko explicitly parse/validate karo (string → number/enum).

13. Performance

Component reuse ek perf feature hai — usse ladо mat, uske saath kaam karo (observable/input binding). Manual subscriptions leak karke memory badhati hain; toSignal()/takeUntilDestroyed() se automatic cleanup.

14. Edge Cases

Matrix params/user;role=admin/42 — route.snapshot.paramMap se milte hain, query se alag (route-scoped).
Parent param in childChild paramMap mein parent ka param nahi — route.parent.paramMap ya paramsInheritanceStrategy:'always'.
Missing paramget() null deta hai — required input ke saath binding error, snapshot ke saath silent null.

15. Interview Questions

EASY Route param kis type ka hota hai?

Hamesha string. paramMap.get('id') se '42' milta hai. Number chahiye toh Number(id) se convert karo — warna comparison/arithmetic bugs.

MEDIUM snapshot vs paramMap observable — kab kya?

Snapshot: component har navigation pe fresh banta hai aur param baad mein change nahi hoga. Observable: same route pe param badalta hai (/user/42→/user/43) — component reuse hota hai, ngOnInit dobara nahi chalta, snapshot stale reh jaata; observable emit karta hai. Modern: withComponentInputBinding() + input().

HARD Component reuse ka mechanism aur usse jude bugs?

Default RouteReuseStrategy same route config pe (sirf params badle) component instance reuse karta hai (perf). Bug: constructor/ngOnInit dobara nahi, snapshot reads aur one-time fetch stale ho jaate. Fix: paramMap observable / input binding / toSignal(). Custom reuse strategy se tab-caching bhi, par memory carefully.

SCENARIO /user/5 se /user/9 jaane pe page abhi bhi 5 ka data dikha raha hai. Fix?

Component reuse (same config) ke kaaran ngOnInit-based snapshot fetch dobara nahi chala. Fix: route.paramMap observable subscribe karke har emit pe reload, ya withComponentInputBinding() + input() + computed(). Snapshot yahan galat choice.

16. Revision Notes

⚡ 30-second recall
  • Route params = required, positional, always string. :id pattern.
  • Same-route change → component reused → ngOnInit skip → snapshot stale. Use paramMap observable / input binding.
  • Modern: withComponentInputBinding() + input.required<string>() + computed().

S-04

Query Parameters

Covers: queryParamMap · queryParams navigation · queryParamsHandling merge/preserve · route param vs query param · URL-as-state

1. What is it? (Kya hai ye?)

Query params URL ke ? ke baad wale optional, named key-value pairs hain — /search?q=phone&sort=price. Route matching se independent; view ko modify karte hain (filter, sort, page, tab).

2. Why do we need it? (Zaroorat kyun hai?)

Filters/sort/pagination ki state ko URL mein daalne se woh shareable ho jaati hai — link bhejo, wahi filtered view khulti hai; refresh pe restore; back button restore. Ye "URL as state" pattern hai.

3. What problem does it solve?

Agar filter state sirf component memory mein ho, refresh pe gayab aur share karne se kaam nahi. Query params ise persist + portable banate hain, bina naye route banaye (kyunki ye optional hain).

4. Internal Working

Query params UrlTree ke ek alag part mein rehte hain (path se separate). Inke badalne pe route match nahi badalta — isliye component reuse hota hai aur queryParamMap observable naya value emit karta hai (paramMap jaisa).

5. Execution Flow

navigate([], { queryParams:{sort}, queryParamsHandling:'merge' })
        │
        ▼
same route (path unchanged) ─▶ component reused ─▶ queryParamMap$ emits new value

6. Architecture — route param vs query param

PositionPath ke andar /user/42? ke baad ?tab=info
Required?Haan — matching ka partOptional — matching se independent

S-06

Lazy Loading

Covers: loadComponent · loadChildren · dynamic import() · code-splitting · route files · canMatch gating

1. What is it? (Kya hai ye?)

Features ko alag JS chunks mein todna aur sirf tab download karna jab user us route pe navigate kare. Mechanism: route config mein dynamic import(), jise build tool alag chunk mein split karta hai.

2. Why do we need it? (Zaroorat kyun hai?)

By default saara code ek bade main.js mein — 40-screen app mein user ne sirf login khola par poora app download kiya. First paint slow, bandwidth waste. Lazy loading initial bundle chhota karta hai.

3. What problem does it solve?

Result

Chhota initial bundle → tez first load (better LCP/TTI). Ye seedha Core Web Vitals aur "how do you optimize a large Angular app" ka jawab hai. Feature chunk tabhi fetch jab pehli baar visit, phir cache.

4. Internal Working

import('./x') ek dynamic import hai — esbuild/webpack ise dekhkar us code ko alag chunk mein split kar deta hai. Router matching ke dauraan (Recognize phase) chunk fetch hota hai, phir uske routes/component ke saath continue.

Diagram — bundle splitting
 main.js (eager, loads first)          reports.chunk.js   ← on /reports
 ┌──────────────────────────┐          admin.chunk.js     ← on /admin
 │ Home · shell · core       │              ▲
 │ ~small ✓                   │   fetched only when route visited (then cached)
 └──────────────────────────┘

5. Execution Flow

navigate /admin ─▶ Recognize ─▶ RouteConfigLoadStart ─▶ download admin.chunk.js
                                   RouteConfigLoadEnd  ─▶ continue guards/resolve/activate

6. Architecture — loadComponent vs loadChildren

loadComponentSingle standalone component lazy (v14+). Ek component return karta hai.
loadChildrenPoore feature area (multiple routes) lazy. Ek Routes array (ya legacy NgModule) return.

S-07

Guards

Covers: functional guards · 5 types · UrlTree/RedirectCommand redirect · CanMatch vs CanActivate · CanDeactivate · execution order

1. What is it? (Kya hai ye?)

Guard ek function hai jo navigation ke beech chalta hai aur decide karta hai: allow (true), block (false), ya redirect (UrlTree/RedirectCommand). Async bhi (Observable/Promise) — jaise server auth check.

2. Why do we need it? (Zaroorat kyun hai?)

Bina guards, har protected component ke ngOnInit mein "logged in? nahi? login pe bhejo" repeat karna padta. Guards ise route config mein centralize karte hain — declarative, reusable, testable.

3. What problem does it solve?

Auth/authorization, feature-flag gating, unsaved-changes protection, conditional lazy-loading — ye sab cross-cutting concerns hain jo guards route-level pe solve karte hain, component logic ko clean rakhkar.

4. Internal Working — 5 types

CanActivateRoute enter se pehle · auth/authorization
CanActivateChildKisi child enter se pehle · section-wide protection
CanDeactivateRoute chhodne se pehle · unsaved changes
CanMatchRoute match hote waqt (load se pehle) · lazy gating, feature flags, A/B
ResolveActivation se pehle (data pre-fetch) · S-08, technically ek guard

5. Execution Flow — order

Diagram — guard execution order
 CanMatch  ─▶  CanDeactivate  ─▶  CanActivate / CanActivateChild  ─▶  Resolve  ─▶  Activate
(match time)   (leaving old)      (top → down, parent to child)     (pre-fetch)   (render)

 Ek array [g1, g2] ke andar guards declared order mein; pehla false/UrlTree baaki ko roke.

6. Architecture — functional guards (modern)

Angular 15+ mein class-based guards deprecated. Aaj functional guards — chhote, tree-shakable, inject() se DI.

7. Real Project Example (Angular context)

auth.guard.ts
import { CanActivateFn, Router, UrlTree } from '@angular/router';
import { inject } from '@angular/core';

export const authGuard: CanActivateFn = (route, state): boolean | UrlTree => {
  const auth = inject(AuthService);
  const router = inject(Router);
  if (auth.isLoggedIn()) return true;

  // ✅ redirect — false return karke navigate MAT karo
  return router.createUrlTree(['/login'], { queryParams: { returnUrl: state.url } });
};

11. Common Mistakes

Traps
  • Redirect ke liye false + manual navigate → broken URL, race. Hamesha UrlTree/RedirectCommand.
  • Functional guard mein bina inject() service access — constructor nahi hota.
  • Lazy route pe canActivate gate (chunk phir bhi download) — canMatch use karo.
  • Guard mein heavy sync work → navigation block. Async ke liye Observable/Promise return.
  • Auth ko sirf guard pe bharosa — guard client-side hai, security nahi. Backend pe bhi validate.

12. Best Practices

Do this
  • Functional guards; purane class guards ko inject()-based migrate.
  • Stateless rakho — decision store/service se derive (testable).
  • Ek guard = ek responsibility; array mein compose.
  • Redirect pe returnUrl capture — login ke baad wapas (great UX).

13. Performance

Guards navigation ke critical path pe hain — heavy sync work UI ko block karta hai. Async guards ke Observables ko take(1)/settle karao. Multiple async guards sequential slow: prefer single guard jo state se derive kare. Lazy gating canMatch se bandwidth bachao.

S-08

Resolvers

Covers: ResolveFn · pre-fetch before activation · data access · blocking trade-off · error handling

1. What is it? (Kya hai ye?)

Resolver ek function hai jo route activate hone se pehle chalta hai aur data (usually Observable/Promise) return karta hai. Router settle hone tak wait karke component ko data-ready render karta hai — data ActivatedRoute.data mein already maujood.

2. Why do we need it? (Zaroorat kyun hai?)

Bina resolver, component ngOnInit mein fetch karta hai aur tab tak template mein khaali/undefined dikhta hai (flicker, layout shift, spinner-then-content jump). Resolver "component pehle se data ke saath aata hai" bana deta hai.

3. What problem does it solve?

No-data flash + data-fetch logic ka scatter. Resolver flicker khatam karta hai aur fetch ko route-level pe centralize karta hai.

4. Internal Working

Guards pass hone ke baad, activate se pehle, Router saare resolve functions run karta hai. Har ek Observable/Promise ke first value tak wait karta hai. Sab settle → data route.data mein daalkar component activate.

5. Execution Flow

guards pass ─▶ ResolveStart ─▶ run resolvers (await first value) ─▶ ResolveEnd
                                                                       │
                              component renders with route.data ready ─┘

6. Architecture

user.resolver.ts
import { ResolveFn } from '@angular/router';
import { inject } from '@angular/core';

export const userResolver: ResolveFn<User> = (route) => {
  const id = route.paramMap.get('id')!;
  return inject(UserService).getUser(id);   // Observable<User>
};

9. Advanced Example — error handling

typescript
export const postResolver: ResolveFn<Post> = (route) => {
  const router = inject(Router);
  return inject(PostService).bySlug(route.paramMap.get('slug')!).pipe(
    take(1),                                   // stream complete karao
    catchError(() => {
      router.navigate(['/not-found']);
      return EMPTY;                            // navigation cancel
    }),
  );
};

10. Code Explanation — Line by Line

  1. ResolveFn<Post> — return type type-safe.
  2. take(1) — resolver ka Observable complete hona chahiye, warna navigation hamesha ke liye ruk jaayegi.
  3. catchError — error pe /not-found redirect + EMPTY se navigation cancel; user stuck nahi hota.

11. Common Mistakes

Traps
  • Slow API resolver mein → poori navigation freeze, koi feedback. Loading indicator (events) do.
  • Error handle na karna → NavigationError, user stuck. catchError se fallback/redirect.
  • Infinite/non-completing stream (jaise interval) → navigation hamesha ruki. take(1)/first().
  • Har cheez resolver mein → app slow feel. Sirf must-have data.

12. Best Practices

Do this

Sirf critical, fast data resolver mein. Non-critical/bada data component mein skeleton loader ke saath async. Modern: signals + toSignal()/Resource API se in-component loading zyada flexible.

13. Performance

Resolver blocking hai — navigation uske settle hone tak rukti hai. Slow API = hang feel. Isliye resolver ko chhota + fast rakho; parallel resolvers OK (Router saath chalata hai) par slowest hi bottleneck. Non-critical data ko defer karo.

14. Edge Cases

Same-route param changeResolver dobara chalta hai (naye param ke liye); route.data observable emit karता hai — snapshot stale ho sakta.
Multiple resolversEk route pe kai keys — sab parallel, sab settle hone tak wait.
Resolver + guardGuard pehle; guard fail toh resolver chalta hi nahi.

15. Interview Questions

EASY Resolver kya karta hai?

Route activate hone se pehle data pre-fetch karta hai; Router settle hone tak wait karke component ko data-ready render karta hai — no flicker. Data route.data['key'] se milta hai.

MEDIUM Resolver vs component mein fetch — trade-offs?

Resolver: no flicker + centralized fetch, par navigation blocking (slow API → hang). In-component: flexible + skeleton loader, par flicker/logic component mein. Rule: critical fast data → resolver; bada/optional → component with skeleton. Modern signals/Resource API in-component ko favor karte hain.

SCENARIO Resolver-based page slow API pe "hang" karta hai bina kisi feedback ke. Do fixes?

(1) router.events se global loading bar (Resolve/Navigation start-end). (2) Non-critical data ko resolver se hatao, component mein skeleton ke saath load. Aur take(1)+catchError se hang/stuck avoid karo.

16. Revision Notes

⚡ 30-second recall
  • ResolveFn pre-fetch; component route.data['key'] se padhta hai. Blocking.
  • Keep minimal + fast. take(1) (complete) + catchError (fallback/redirect).
  • Critical → resolver; optional/bada → in-component skeleton.

S-09

Preloading

Covers: NoPreloading · PreloadAllModules · custom PreloadingStrategy · data-driven preload

1. What is it? (Kya hai ye?)

Lazy loading ka best-of-both: initial bundle chhota, par app render ho jaane ke baad background mein lazy chunks chupke se download. Result: fast initial load + snappy navigation.

2. Why do we need it? (Zaroorat kyun hai?)

Pure lazy mein trade-off: pehli visit pe chunk-download ka lag. Preloading is lag ko hataता hai — user navigate kare toh chunk already cache mein.

3. What problem does it solve?

Lazy loading ka "first visit slow" problem, bina initial bundle bloat kiye.

4. Internal Working

Initial navigation settle hone ke baad, Router PreloadingStrategy.preload(route, load) har lazy route ke liye call karta hai. Strategy decide karti hai load() chalayen (chunk fetch) ya nahi (of(null)).

5. Execution Flow

initial navigation done ─▶ browser idle ─▶ strategy.preload(route, load) per lazy route
                                                │
                       load() → fetch chunk in background  |  of(null) → skip

6. Architecture — strategies

NoPreloadingDefault — lazy sirf navigation pe.
PreloadAllModulesSaare lazy routes initial ke baad background preload.
Custom strategyTumhaari logic — sirf selected/likely routes.

7. Real Project Example (Angular context)

app.config.ts
import { provideRouter, withPreloading, PreloadAllModules } from '@angular/router';

provideRouter(routes, withPreloading(PreloadAllModules));

8. Beginner Example

typescript
// default (kuch preload nahi):
provideRouter(routes, withPreloading(NoPreloading));

9. Advanced Example — custom, data-driven

selective-preload.ts
import { PreloadingStrategy, Route } from '@angular/router';
import { Observable, of } from 'rxjs';
import { Injectable } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class SelectivePreload implements PreloadingStrategy {
  preload(route: Route, load: () => Observable<unknown>): Observable<unknown> {
    return route.data?.['preload'] ? load() : of(null);
  }
}

// route: { path: 'reports', data: { preload: true },
//          loadComponent: () => import('./reports').then(m => m.Reports) }

10. Code Explanation — Line by Line

  1. preload(route, load) har lazy route ke liye call. load() → chunk fetch.
  2. route.data['preload'] true → load() return, chunk background mein.
  3. False → of(null), preload skip. Config mein withPreloading(SelectivePreload).

11. Common Mistakes

Traps
  • Slow/metered connections pe PreloadAllModules data waste — mobile users hurt. Selective behtar.
  • Preloading initial nav settle hone tak shuru nahi — turant ready expect karna galat.
  • canMatch-blocked routes preload nahi hote — logic align.

12. Best Practices

Do this

High-traffic routes (jaha 80% chance) ko data.preload + selective strategy. Rare admin/reports pure lazy. navigator.connection se slow connection pe preloading skip — SDE-3 polish.

13. Performance

Preloading initial load ko touch nahi karta (idle time use), par indiscriminate PreloadAllModules bandwidth kha sakta hai. Selective + connection-aware = optimal: fast initial + instant likely-routes.

14. Edge Cases

Nested lazy childrenPreload strategy har lazy boundary pe call hoti hai — nested chunks bhi cover.
Preload failureBackground fetch fail silently — user navigate kare toh normal load retry.
Very large appPreloadAllModules 100+ chunks fetch kar sakta — selective mandatory.

15. Interview Questions

EASY Preloading kya hai?

App initial render ke baad background mein lazy chunks pehle se download karna, taaki user navigate kare toh instant ho. Best-of-both: chhota initial bundle + snappy nav.

MEDIUM PreloadAllModules vs custom strategy?

PreloadAllModules saare lazy routes preload karta hai — chhote apps ke liye theek, bade/mobile pe bandwidth waste. Custom PreloadingStrategy (data.preload flag) se sirf likely/high-traffic routes preload — controlled aur connection-aware.

SCENARIO Lazy route pe navigate karne pe slow network pe UI freeze feel hoti hai. Long-term fix?

Immediate: router.events se loading indicator. Long-term: frequently-visited lazy routes ko selective preloading se background mein pre-fetch karo — chunk already ready hoga jab user jaaye.

16. Revision Notes

⚡ 30-second recall
  • NoPreloading (default) / PreloadAllModules / custom PreloadingStrategy.
  • Custom: preload(route, load) → route.data.preload ? load() : of(null).
  • Selective + connection-aware for large/mobile apps.

S-10

Router Events

Covers: router.events stream · event types · global loader · analytics · NavigationCancel code · scroll restoration

1. What is it? (Kya hai ye?)

router.events ek Observable<Event> hai jo navigation lifecycle ke har step pe ek typed event emit karta hai. Isse side-effects trigger karte hain: loading spinner, analytics, error handling, scroll-restore.

2. Why do we need it? (Zaroorat kyun hai?)

Navigation ke around global behaviours chahiye — har lazy chunk load pe loader, har page-view pe analytics, har error pe fallback. Events ye centralized observability dete hain.

3. What problem does it solve?

Bina events, tumhe har component mein manually loading/analytics wire karna padta. Events ek jagah se poori app ki navigation observe karte hain.

4. Internal Working — event types

NavigationStartNavigation shuru
RouteConfigLoadStart/EndLazy chunk download shuru/khatam
RoutesRecognizedURL match, RouterState ban gaya
GuardsCheckStart/End · ResolveStart/EndGuards / resolvers pipeline
NavigationEnd✅ Success — outlet updated
NavigationCancel · NavigationErrorGuard ne roka / lazy-load ya resolver fail
ScrollScroll position restore

5. Execution Flow

NavigationStart ─▶ (RouteConfigLoad*) ─▶ RoutesRecognized ─▶ GuardsCheck* ─▶ Resolve*
        ─▶ ActivationStart ─▶ NavigationEnd | NavigationCancel | NavigationError

6. Architecture

router.events ek long-lived stream hai (app lifetime). Har event ek object — instanceof ya type-guard filter se identify. Always unsubscribe (takeUntilDestroyed()).

7. Real Project Example — global loading bar

app.ts
import { Router, NavigationStart, NavigationEnd, NavigationCancel,
  NavigationError, RouteConfigLoadStart, RouteConfigLoadEnd } from '@angular/router';
import { signal } from '@angular/core';

export class App {
  loading = signal(false);
  constructor() {
    inject(Router).events.pipe(takeUntilDestroyed()).subscribe(e => {
      if (e instanceof NavigationStart || e instanceof RouteConfigLoadStart)
        this.loading.set(true);
      if (e instanceof NavigationEnd || e instanceof NavigationCancel ||
          e instanceof NavigationError || e instanceof RouteConfigLoadEnd)
        this.loading.set(false);
    });
  }
}

8. Beginner Example

typescript
router.events.subscribe(e => console.log(e.constructor.name));  // debug: dekho kya fire hota

9. Advanced Example — analytics + cancel reason

typescript
router.events.pipe(
  filter((e): e is NavigationEnd => e instanceof NavigationEnd)
).subscribe(e => analytics.pageView(e.urlAfterRedirects));   // NOT e.url

router.events.pipe(
  filter((e): e is NavigationCancel => e instanceof NavigationCancel)
).subscribe(e => console.log('cancelled:', e.code));   // NavigationCancellationCode

10. Code Explanation — Line by Line

  1. Type-guard filter ((e): e is NavigationEnd) — downstream mein typed event.
  2. urlAfterRedirects — redirect ke baad ka final URL (analytics ke liye sahi).
  3. NavigationCancel.code — NavigationCancellationCode enum se cancel ki wajah.

11. Common Mistakes

Traps
  • Sirf NavigationEnd pe loader band → Cancel/Error pe spinner atka.
  • Subscribe karke unsubscribe na karna → leak (root mein bhi habit).
  • Redirect ke baad galat URL log — urlAfterRedirects use karo, url nahi.
  • Har event pe heavy work — stream high-frequency, filter karke minimal.

12. Best Practices

Do this
  • Type-guard filter for typed events.
  • Scroll restoration built-in se: withInMemoryScrolling({ scrollPositionRestoration:'enabled' }) — manual events nahi.
  • Debug: withDebugTracing() (dev only, production mein hata do).

13. Performance

Stream har navigation pe kai events emit karta hai — subscribe mein filter karke sirf zaroori events pe kaam karo. Heavy synchronous work event handler mein change detection storm bana sakta hai (zoneless mein bhi careful). Analytics ko debounce/batch karo.

14. Edge Cases

Redirect chainsEk navigation kai RoutesRecognized emit kar sakti — final urlAfterRedirects pe bharosa.
Cancelled by guardNavigationCancel.code batata hai kyun (guard rejected / redirect / newer nav).
Error recoveryNavigationError ke baad app pichle URL pe rehti hai — global error UI dikhao.

15. Interview Questions

EASY Global loading bar kaise banaoge?

MEDIUM Analytics page-view ke liye kaunsa event/property?

SCENARIO Guard redirect ke baad loading spinner kabhi band hi nahi hota. Kyun?
