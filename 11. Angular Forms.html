S-01

Template-Driven Forms

Covers: FormsModule · ngModel · [(ngModel)] · template reference vars · ngForm · ngModelGroup · when to use vs reactive

1. What is it? (Kya hai ye?)

Template-driven form ek approach hai jahan form ka logic HTML template mein rehta hai, TS class mein nahi. Tum ngModel directives lagate ho, aur Angular background mein automatically FormControl/FormGroup bana deta hai. Model, validation, state — sab template directives se derive.

2. Why do we need it? (Zaroorat kyun hai?)

Chhote, simple, mostly-static forms (login, contact, ek-do field settings) ke liye reactive ka poora TS setup overkill hai. Template-driven mein 2 lines mein two-way binding + validation mil jaati hai — fast to write, kam boilerplate.

3. What problem does it solve?

Simple forms ke liye "form model banao, control map karo, subscribe karo" wala ceremony hata deta hai. AngularJS-style familiar ngModel two-way binding — quick prototypes aur chhote forms ke liye ideal.

4. Internal Working (Engine ke andar kya hota hai?)

Asal mein hota kya hai

ngModel directive har input ke liye ek FormControl banata hai aur ise parent NgForm (jo <form> pe auto-apply hota hai) mein register kar deta hai. Yani reactive jaisa hi model banta hai — bas tum banate nahi, Angular banata hai, template ke through. Isiliye ye "template-driven" hai.

Diagram — template-driven binding
 <input ngModel name="email">
        │
        ▼
 NgModel directive ──creates──▶ FormControl  ──registers──▶ NgForm (on <form>)
        ▲                                                        │
        └──────────── two-way [(ngModel)] sync ─────────────────┘
              (view ⇄ component property, async via ngModelChange)

5. Execution Flow

user types ─▶ input event ─▶ ControlValueAccessor ─▶ FormControl.setValue
      ─▶ NgForm state update (valid/dirty/touched) ─▶ [(ngModel)] writes to component prop
      ─▶ change detection ─▶ view (error messages) update
Async note

Template-driven mein control creation asynchronous hai (Angular ise ek microtask mein register karta hai). Isliye ngOnInit mein turant form.controls.x access karne pe undefined mil sakta — reactive mein aisa nahi.

6. Architecture (Building blocks)

FormsModuleImport zaroori (standalone component ke imports mein). Ye ngModel, ngForm laata hai.
ngModelEk input ko control banata hai. name attribute zaroori (registration key).
NgForm<form> pe auto-directive — poore form ka aggregate (value, valid, dirty).
ngModelGroupRelated controls ko ek nested group mein bundle karta hai.
Template ref #f="ngForm"Form/control ki state template mein expose karta hai.

7. Real Project Example (Angular context)

contact.ts
import { Component } from '@angular/core';
import { FormsModule } from '@angular/forms';

@Component({
  selector: 'app-contact',
  standalone: true,
  imports: [FormsModule],
  templateUrl: './contact.html',
})
export class Contact {
  model = { email: '', message: '' };
  submit(form: NgForm) {
    if (form.valid) this.api.send(this.model);
  }
}
contact.html
<form #f="ngForm" (ngSubmit)="submit(f)">
  <input name="email" [(ngModel)]="model.email" required email #email="ngModel">
  <small *ngIf="email.invalid && email.touched">Valid email daalo</small>

  <textarea name="message" [(ngModel)]="model.message" required></textarea>

  <button [disabled]="f.invalid">Send</button>
</form>

10. Code Explanation — Line by Line

  1. #f="ngForm" — template ref jo NgForm instance deta hai (f.valid, f.value).
  2. name="email" — control ka registration key; iske bina ngModel error dega.
  3. [(ngModel)]="model.email" — two-way: view↔component property.

  5. required email — built-in validator directives (S-04 details).

11. Common Mistakes

Traps
  • name attribute bhoolna → runtime error, control register nahi hota.
  • FormsModule import na karna → ngModel silently ignore.
  • ngOnInit mein turant control access → async creation ke kaaran undefined.
  • Bade/dynamic forms template-driven mein banana → template bloat, testing hard. Reactive use karo.
  • Error dikhane ke liye sirf invalid check → user ke type karne se pehle hi red. touched/dirty bhi lagao.

12. Best Practices

Do this
  • Sirf chhote/simple forms ke liye — 5-6 se zyada fields ya dynamic ho toh reactive.
  • Error *ngIf="ctrl.invalid && (ctrl.dirty || ctrl.touched)" — premature red avoid.
  • Submit pe form.valid check + button [disabled]="f.invalid".
  • Template ref vars se granular state; magic string field names consistent rakho.

13. Performance

Template-driven forms har CD cycle mein template directives evaluate karte hain — bade forms mein ye mehnga (aur debug-hard) ho jaata. Chhote forms mein negligible. Bade forms mein reactive zyada efficient + predictable, kyunki model TS mein explicit hai aur RxJS se fine-grained control milta hai.

14. Edge Cases

Standalone ngModel<form> ke bina bhi kaam karta hai — ek isolated two-way binding.
Dynamic name[name]="'item'+i" — *ngFor mein unique names zaroori warna collide.
Typed value?Template-driven forms typed nahi hote — type safety sirf reactive mein (v14+).

15. Interview Questions

EASY Template-driven vs reactive — ek line farak?

Template-driven: form logic template mein, Angular controls implicitly banata hai (ngModel), chhote/simple forms ke liye. Reactive: form model TS class mein explicit, programmatic, typed, dynamic + complex forms + testing ke liye. Template-driven = async control creation; reactive = synchronous.

EASY ngModel ke liye name kyun zaroori?

name control ki registration key hai — NgForm isi se control ko form model mein register karta hai. Bina name ke Angular error deta hai kyunki wo control ko address nahi kar paata.

MEDIUM Template-driven mein control creation async kyun, aur iska impact?

Angular ngModel controls ko ek microtask mein register karta hai (template render ke baad). Isliye ngOnInit/constructor mein form.controls.x turant access karo toh undefined mil sakta. Fix: ngAfterViewInit ya setTimeout, ya bas reactive use karo jahan model synchronous hai.

SCENARIO Ek form mein user ne kuch type nahi kiya par saare fields already red error dikha rahe hain. Fix?

Error condition sirf invalid pe hai. Add touched/dirty: *ngIf="ctrl.invalid && (ctrl.touched || ctrl.dirty)" — ab error tabhi dikhega jab user ne field ko touch/edit kiya ho.

16. Revision Notes

⚡ 30-second recall
  • Import FormsModule. ngModel + name → Angular implicitly control banata hai.
  • #f="ngForm", #c="ngModel" for state; ngModelGroup for nesting.
  • Async control creation; NOT typed. Small/simple forms only. Error on invalid && (touched||dirty).

S-02

Reactive Forms

Covers: ReactiveFormsModule · FormControl · FormGroup · typed forms (v14+) · valueChanges/statusChanges · setValue vs patchValue · state flags

1. What is it? (Kya hai ye?)

Reactive forms mein form model TS class mein explicitly define hota hai — FormControl, FormGroup, FormArray objects ka tree. Template sirf is model ko bind karta hai ([formGroup], formControlName). Model source-of-truth hai; template uska view.

2. Why do we need it? (Zaroorat kyun hai?)

Complex forms — multi-step, dynamic fields, cross-field validation, conditional logic, heavy testing — template-driven mein manage karna nightmare. Reactive model explicit + synchronous + typed + RxJS-powered hai, isliye ye sab clean ho jaata hai.

3. What problem does it solve?

Reactive deta hai
  • Type safety (v14+ default) — galat field access compile-time error.
  • Synchronous access — control turant available, testing easy.
  • RxJS streams — valueChanges/statusChanges se reactive logic (auto-save, dependent fields).
  • Dynamic structure — runtime pe controls add/remove (S-06).

4. Internal Working (Engine ke andar kya hota hai?)

Har FormControl ek object hai jismein value, validators, aur status hota hai, plus do RxJS Subjects: valueChanges aur statusChanges. Template ka formControlName ek ControlValueAccessor (CVA) ke through DOM input ko is control se jodta hai — dono direction sync. Value badalne pe control validators re-run karta hai aur parent tak status cascade karta hai.

Diagram — reactive model ⇄ view
 Component (source of truth)          Template (view)
 ┌───────────────────────────┐        ┌──────────────────────────┐
 │ FormGroup                 │        │ <form [formGroup]="form"> │
 │  ├─ email:  FormControl   │◀──CVA──▶│   <input formControlName │
 │  └─ pass:   FormControl   │  sync  │          ="email">        │
 │  valueChanges$ / status$  │        └──────────────────────────┘
 └───────────────────────────┘

5. Execution Flow

user types ─▶ input event ─▶ CVA writes FormControl.value
   ─▶ validators run ─▶ control status (VALID/INVALID) ─▶ cascade up to FormGroup
   ─▶ valueChanges$ / statusChanges$ emit ─▶ your subscriptions react (auto-save, etc.)

6. Architecture — the three building blocks

FormControlSingle field — value + validators + status. Atom.
FormGroupNamed collection of controls (object shape). Ek form / ek nested section.
FormArrayIndexed list of controls (dynamic rows). S-06.
Base: AbstractControlTeeno ka parent — value, valid, errors, valueChanges, setValue yahin se.

7. Real Project Example (Angular context)

login.ts
import { Component, inject } from '@angular/core';
import { FormControl, FormGroup, Validators, ReactiveFormsModule } from '@angular/forms';

@Component({ standalone: true, imports: [ReactiveFormsModule], templateUrl: './login.html' })
export class Login {
  form = new FormGroup({
    email: new FormControl('', { nonNullable: true, validators: [Validators.required, Validators.email] }),
    password: new FormControl('', { nonNullable: true, validators: [Validators.required, Validators.minLength(8)] }),
  });

  ngOnInit() {
    // reactive dependent logic — RxJS ka fayda
    this.form.controls.email.valueChanges.subscribe(v => console.log('email:', v));
  }

  submit() {
    if (this.form.invalid) return;
    const { email, password } = this.form.getRawValue();   // typed!
    this.auth.login(email, password);
  }
}
login.html
<form [formGroup]="form" (ngSubmit)="submit()">
  <input formControlName="email" placeholder="Email">
  <small *ngIf="form.controls.email.touched && form.controls.email.invalid">
    Valid email chahiye
  </small>
  <input formControlName="password" type="password">
  <button [disabled]="form.invalid">Login</button>
</form>

9. Advanced Example — setValue vs patchValue + typed forms

typescript
// TYPED (v14+): email/password auto-inferred as string (nonNullable)
form = new FormGroup({
  email: new FormControl('', { nonNullable: true }),
  age:   new FormControl<number | null>(null),
});

// setValue → SAARE controls dena zaroori (strict, typo pakadta hai)
this.form.setValue({ email: 'a@b.com', age: 25 });

// patchValue → subset OK (partial update)
this.form.patchValue({ email: 'x@y.com' });

// suppress valueChanges emit (auto-save loop avoid)
this.form.patchValue(draft, { emitEvent: false });
nonNullable kyun

Default FormControl('') ka type string | null hota hai (reset pe null aata). { nonNullable: true } se reset bhi initial value pe jaata aur type sirf string — cleaner. Ya NonNullableFormBuilder (S-03).

10. Code Explanation — Line by Line

  1. new FormGroup({...}) — form model, keys = control names.
  2. { nonNullable: true, validators: [...] } — control ka options object.
  3. valueChanges.subscribe — har value change pe react (dependent fields, auto-save).
  4. getRawValue() — disabled controls sameth pura typed value (.value disabled ko skip karta hai).
  5. Template [formGroup]="form" + formControlName="email" — model ko view se bind.

11. Common Mistakes

Traps
  • ReactiveFormsModule import bhoolna (standalone imports mein) → formControlName dead.
  • valueChanges.subscribe ka unsubscribe na karna → memory leak. takeUntilDestroyed() / AsyncPipe.
  • setValue ko partial object dena → error (saare controls chahiye). Partial ke liye patchValue.
  • Disabled control ka value .value se expect karna → wo skip hota hai; getRawValue() use karo.
  • Auto-save loop: valueChanges mein patchValue → infinite. emitEvent: false.

12. Best Practices

Do this
  • Type inference pe bharosa karo — extra annotations mat likho; nonNullable se null hatao.
  • Subscriptions ko takeUntilDestroyed() ya toSignal() se manage karo.
  • Template mein controls ko getter (get email() { return this.form.controls.email }) se access — cleaner.
  • Submit pe invalid guard + markAllAsTouched() se saare errors dikhao.

13. Performance

Reactive forms model-driven hain, isliye CD template parsing pe depend nahi karta — bade forms mein template-driven se fast + predictable. valueChanges pe heavy work ho toh debounceTime/distinctUntilChanged lagao. OnPush components ke saath acche baithte hain kyunki updates observable-driven hain.

14. Edge Cases

updateOnnew FormControl('', { updateOn: 'blur' }) — har keystroke ki jagah blur pe validate (perf/UX).
markAsTouched/PristineProgrammatic state control — submit pe saare errors reveal karne ke liye.
Signal Forms (v21+)Naya experimental API — form() + [formField], signals pe based. Abhi @experimental, production ke liye Reactive stable choice.

15. Interview Questions

EASY setValue vs patchValue?

setValue ko form ke saare controls dena zaroori — strict, missing/extra key pe error (typos pakadta hai). patchValue subset accept karta hai — partial update. Backend se pura object aaye toh setValue, kuch fields update karne ho toh patchValue.

MEDIUM Reactive forms typed kab se, aur nonNullable ka role?

Angular v14 (preview) se, v15 mein stable — default typed. FormControl('') ka type string | null (reset pe null). { nonNullable: true } null hatata hai aur reset ko initial value pe le jaata — type sirf string. Template-driven forms typed nahi hote.

MEDIUM .value vs getRawValue()?

.value disabled controls ko skip karta hai. getRawValue() disabled sameth pura value deta hai. Backend ko disabled fields bhi bhejne ho (jaise read-only computed) toh getRawValue().

HARD Auto-save valueChanges se banaya aur infinite loop aa gaya. Kyun aur fix?

valueChanges subscription ke andar patchValue/setValue call kiya → wo phir valueChanges emit karta hai → loop. Fix: programmatic writes pe { emitEvent: false } pass karo, aur debounceTime + distinctUntilChanged se redundant saves rokо.

SCENARIO Submit pe form invalid hai par user ko koi error nahi dikh raha (kyunki fields untouched hain). Fix?

Errors touched pe gated hain aur user ne submit direct dabaya. Submit handler mein this.form.markAllAsTouched() call karo — saare controls touched ho jaayenge aur errors reveal ho jaayenge.

16. Revision Notes

⚡ 30-second recall
  • Import ReactiveFormsModule. Model = FormControl/FormGroup/FormArray in TS.
  • Typed since v14; nonNullable removes null. setValue (full) vs patchValue (partial).
  • .value skips disabled; getRawValue() includes. emitEvent:false to avoid loops. markAllAsTouched() to reveal errors.

S-03

FormBuilder

Covers: inject(FormBuilder) · fb.group/control/array · shorthand syntax · NonNullableFormBuilder · when it helps

1. What is it? (Kya hai ye?)

FormBuilder ek injectable service hai jo reactive form banane ka shorthand deta hai. new FormControl(...) / new FormGroup(...) baar-baar likhne ki jagah fb.group({...}), fb.control(...), fb.array([...]).

2. Why do we need it? (Zaroorat kyun hai?)

Bade forms mein new FormControl ka repetition verbose aur noisy ho jaata. FormBuilder same model ko concise, readable syntax mein likhne deta hai — kam boilerplate, same power.

3. What problem does it solve?

Verbosity. Ye koi nayi capability nahi deta — bas ergonomics. Type safety, validators, structure — sab wahi, bas syntax chhota.

4. Internal Working

FormBuilder ek factory hai: fb.group(config) internally new FormGroup banaता hai, har key ke value ko FormControl mein wrap karta hai (agar array shorthand diya toh [initialValue, validators, asyncValidators] unpack karta hai). Output bilkul wahi typed model hai jo tum manually banate.

5. Execution Flow

fb.group({ email: ['', [Validators.required]] })
        │
        ▼
FormBuilder.group ─▶ for each key ─▶ new FormControl(init, validators) ─▶ new FormGroup
        │
        ▼
   typed FormGroup (same as manual)

6. Architecture — the shorthands

fb.control(init, validators)= new FormControl(...)
fb.group({ k: [init, validators] })= new FormGroup({ k: new FormControl(...) })
fb.array([...])= new FormArray([...])
fb.nonNullable (NonNullableFormBuilder)Saare controls nonNullable — no null in types.

7. Real Project Example (Angular context)

signup.ts
import { Component, inject } from '@angular/core';
import { FormBuilder, Validators, ReactiveFormsModule } from '@angular/forms';

@Component({ standalone: true, imports: [ReactiveFormsModule], templateUrl: './signup.html' })
export class Signup {
  private fb = inject(FormBuilder);

  form = this.fb.nonNullable.group({
    name:  ['', [Validators.required]],
    email: ['', [Validators.required, Validators.email]],
    address: this.fb.group({          // nested group
      city: ['', Validators.required],
      zip:  ['', Validators.required],
    }),
    hobbies: this.fb.array([]),        // dynamic (S-06)
  });
}

10. Code Explanation — Line by Line

  1. inject(FormBuilder) — modern DI (constructor injection ki jagah).
  2. this.fb.nonNullable.group — null-free typed controls.
  3. name: ['', [Validators.required]] — tuple: initial value + sync validators.

11. Common Mistakes

Traps
  • Tuple positions confuse: ['', asyncValidator] — async ko sync position mein daalna. Sync = 2nd, async = 3rd.
  • fb.group ke bina nonNullable — types mein null ghus jaata; reset behaviour unexpected.
  • FormBuilder ko naya capability samajhna — ye sirf shorthand hai; power same.
  • Legacy constructor injection habit — modern inject(FormBuilder) use karo.

12. Best Practices

Do this
  • Bade forms → FormBuilder; ek-do control → plain new FormControl bhi theek.
  • fb.nonNullable.group default banao — cleaner types + predictable reset.
  • Consistency: poore codebase mein ek style (FormBuilder ya plain) rakho.

13. Performance

FormBuilder ka koi runtime overhead nahi — ye sirf construction-time factory hai, output identical FormGroup. Perf reactive forms jaisi hi. Fayda purely developer-ergonomics + readability.

14. Edge Cases

Mixed nullable/nonNullableEk nonNullable group ke andar bhi specific control ko nullable rakh sakte ho manually.

SCENARIO Reset ke baad form controls null ho ja rahe hain aur types mein string | null aa raha hai. Clean fix?

S-04

Validators (built-in)

Covers: Validators.required/min/max/minLength/pattern/email · errors object · hasError · showing messages · status VALID/INVALID · group-level

1. What is it? (Kya hai ye?)

Validators wo functions hain jo ek control ki value check karke batate hain valid hai ya nahi. Angular kai built-in validators deta hai (Validators.required, minLength, email, pattern...). Invalid pe ye control ke errors object mein ek key daalte hain.

2. Why do we need it? (Zaroorat kyun hai?)

Garbage/invalid data ko submit hone se rokna, aur user ko turant, specific feedback dena ("email galat hai", "password chhota hai"). Validation UX + data integrity dono ka core hai.

3. What problem does it solve?

Manual if (value.length < 8) checks scatter ho jaate. Validators declarative, reusable, aur control ke status system se integrated hain — form.valid automatically sab controls se derive hota hai.

4. Internal Working (Engine ke andar kya hota hai?)

errors object ka contract

Har validator ek function hai: (control) => ValidationErrors | null. Valid → null. Invalid → ek object jaise { required: true } ya { minlength: { requiredLength: 8, actualLength: 3 } }. Angular saare validators ke errors merge karke control ke errors mein daal deta hai, aur status INVALID kar deta hai. Ek bhi control invalid → parent group bhi invalid.

Diagram — validation status cascade
 value change ─▶ run validators ─▶ errors: {minlength:{...}} | null
                                        │
              control.status = INVALID ◀┘  ─▶ cascade up ─▶ FormGroup.status = INVALID
                                                              ─▶ form.valid = false

5. Execution Flow

control.setValue('ab') ─▶ Validators.minLength(8) runs ─▶ returns {minlength:{requiredLength:8,actualLength:2}}
        ─▶ control.errors = {minlength:...} ─▶ control.invalid = true
        ─▶ template: *ngIf ctrl.hasError('minlength') ─▶ show message

6. Architecture — built-in validators

Validators.requiredNon-empty. Error key: required
Validators.requiredTrueCheckbox must be checked (terms accept).
Validators.min / maxNumeric bounds. Keys: min/max
Validators.minLength / maxLengthString/array length. Keys: minlength/maxlength
Validators.emailEmail format. Key: email
Validators.pattern(regex)Regex match. Key: pattern

S-05

Async Validators

Covers: AsyncValidatorFn · Observable/Promise return · PENDING status · debounce/switchMap · server uniqueness · updateOn:'blur'

1. What is it? (Kya hai ye?)

Async validator wo validator hai jo decision ke liye async operation (usually HTTP) karta hai — jaise "kya ye username already taken hai?". Ye ek Observable/Promise return karta hai jo ValidationErrors | null emit karta hai.

2. Why do we need it? (Zaroorat kyun hai?)

Kuch rules client pe check nahi ho sakte — email/username uniqueness, coupon validity, server-side business rules. Inke liye backend se poochna padta hai, jo inherently async hai.

3. What problem does it solve?

Server-dependent validation ko form ke status system mein integrate karta hai — control automatically PENDING hota hai jab tak server jawab na de, aur invalid ho toh form.valid false. Manual flags manage karne ki zaroorat nahi.

4. Internal Working (Engine ke andar kya hota hai?)

PENDING status

Async validators sync validators pass hone ke baad chalte hain (taaki bekaar API calls na ho). Jab async chal raha ho, control ka status PENDING hota hai. Emit hone pe VALID/INVALID. Har async validator ka Observable complete hona chahiye (ya first-emit) warna status hamesha PENDING atka rehta.

Diagram — async validation flow
 value change ─▶ SYNC validators ─┬─ fail ─▶ INVALID (async skip)
                                  └─ pass ─▶ status = PENDING ─▶ run ASYNC validator
                                                    │
                              HTTP ──▶ emit {taken:true} | null ─▶ INVALID | VALID

5. Execution Flow

type "karan" ─▶ sync ok ─▶ PENDING ─▶ debounce ─▶ switchMap(http /check?u=karan)
        ─▶ server: taken ─▶ emit {usernameTaken:true} ─▶ control INVALID
        (agla keystroke → previous HTTP switchMap se cancel)

6. Architecture — 3rd position

Async validators control ke teesre argument mein jaate hain: new FormControl(value, [syncValidators], [asyncValidators]) ya FormBuilder tuple ka third element.

7. Real Project Example (Angular context)

unique-username.validator.ts
import { AsyncValidatorFn, AbstractControl, ValidationErrors } from '@angular/forms';
import { inject } from '@angular/core';
import { of, timer } from 'rxjs';
import { switchMap, map, catchError, first } from 'rxjs';

export function uniqueUsername(): AsyncValidatorFn {
  return (control: AbstractControl): Observable<ValidationErrors | null> => {
    const api = inject(UserService);
    if (!control.value) return of(null);
    return timer(400).pipe(                       // debounce
      switchMap(() => api.checkUsername(control.value)),
      map(taken => (taken ? { usernameTaken: true } : null)),
      catchError(() => of(null)),                 // server error → don't block
      first(),                                    // COMPLETE the stream
    );
  };
}
usage
username: ['', [Validators.required], [uniqueUsername()]],   // 3rd position!

10. Code Explanation — Line by Line

  1. timer(400) — debounce; rapid typing pe har keystroke API na ho.
  2. switchMap — naya value aaye toh pichhli HTTP cancel (stale response avoid).
  3. map(taken => taken ? {...} : null) — server bool ko errors object mein badalna.
  4. catchError(() => of(null)) — server down pe form block na karo.
  5. first() — stream complete karo, warna control PENDING mein atka rahega.

11. Common Mistakes

Traps
  • Stream complete na karna (first()/take(1) bhoolna) → control hamesha PENDING.
  • switchMap ki jagah mergeMap → stale/out-of-order responses.
  • Debounce na lagana → har keystroke pe API hammering.
  • Async validator ko 2nd (sync) position mein daalna → error/silent.
  • form.valid ko submit pe turant check jabki PENDING chal raha ho → PENDING valid nahi hai, handle karo.

12. Best Practices

Do this
  • Hamesha debounce (timer) + switchMap + catchError + first().
  • updateOn: 'blur' se API calls drastically kam.
  • PENDING state UI mein dikhao ("checking…"); submit button ko PENDING pe bhi disable.
  • Server error pe block mat karo (of(null)) — user stuck na ho.

13. Performance

Async validators = network calls, isliye sabse mehnge. Debounce + switchMap + updateOn:'blur' ke bina ye API ko flood kar dete hain. switchMap in-flight requests cancel karke bandwidth bachata. Server pe rate-limit/cache endpoint rakho.

14. Edge Cases

Sync failSync validator fail ho toh async chalta hi nahi (API waste avoid) — by design.
Race with submitSubmit jab PENDING → status abhi decide nahi. statusChanges pe wait ya PENDING pe submit block.
DI in validatorFunctional validator mein inject() injection context mein chahiye — factory se banao.

15. Interview Questions

EASY Async validator kab use hota hai aur kya return karta hai?

Jab validation ke liye server/async operation chahiye (username/email uniqueness). Ye ek Observable/Promise return karta hai jo ValidationErrors | null emit karta hai. Chalne ke dauraan control PENDING status mein rehta hai.

MEDIUM Async validators sync ke baad kyun chalte hain?

Efficiency — agar sync validation (jaise "required"/"minLength") already fail hai toh server ko poochne ka koi matlab nahi (bekaar API call). Isliye Angular async validators ko tabhi run karta hai jab saare sync validators pass ho jaayein.

HARD Username check ke liye switchMap vs mergeMap — kyun switchMap?

switchMap naya emission (agla keystroke) aane pe pichhli in-flight HTTP ko cancel kar deta hai — sirf latest value ka response matter karta hai, stale/out-of-order responses avoid. mergeMap saari parallel chalata → purani (galat) response naye value pe apply ho sakti, aur zyada API calls.

SCENARIO Async validator laga par control hamesha "checking…" (PENDING) mein atka rehta hai. Kyun?

Validator ka Observable complete nahi ho raha. HTTP call complete hoti hai par agar tumne switchMap(timer...) jaisa non-completing source use kiya bina first()/take(1) ke, stream open rehta hai. Fix: pipe ke end mein first() ya take(1) add karo.

16. Revision Notes

⚡ 30-second recall
  • AsyncValidatorFn → Observable<ValidationErrors|null>. 3rd position. Status PENDING while running.
  • Runs only after sync validators pass. Recipe: timer (debounce) → switchMap → map → catchError(of(null)) → first().
  • Must complete the stream (first()). updateOn:'blur' cuts API calls. Disable submit on PENDING.

Covers: FormArray · push/removeAt/insert/clear · iterate in template · add/remove rows · typed arrays · FormRecord

1. What is it? (Kya hai ye?)

Dynamic form = jiski structure runtime pe badalti hai — user "Add another phone" click kare toh naya field aaye. Iska tool hai FormArray: controls ki ek indexed list jise programmatically add/remove kar sakte ho.

2. Why do we need it? (Zaroorat kyun hai?)

Bahut sare forms mein fields ki sankhya fixed nahi — multiple phone numbers, order line items, skill tags, quiz questions. Fixed FormGroup se ye handle nahi hota; FormArray variable-length collections deta hai.

3. What problem does it solve?

Variable-length data ka model + validation + binding. Har row apna control/group hota hai apni validation ke saath, aur array-level pe bhi rules (min 1 item) lag sakte.

4. Internal Working

FormArray internally controls: AbstractControl[] rakhta hai. push()/removeAt() is array ko mutate karke status recompute karte hain. Value ek array hoti hai (object nahi). Template mein controls pe *ngFor karke har item ko [formGroupName]="i"/[formControlName]="i" se bind karte ho.

Diagram — FormArray structure
 FormArray "phones"
 ├─ [0] FormControl "9111111111"
 ├─ [1] FormControl "9222222222"   ◀── push() adds here
 └─ [2] FormControl "9333333333"
       value → ["9111111111","9222222222","9333333333"]   (an array!)

5. Execution Flow

"Add phone" click ─▶ phones.push(fb.control('', Validators.required))
        ─▶ FormArray recomputes status ─▶ template *ngFor re-renders ─▶ new input appears
"Remove" click ─▶ phones.removeAt(i) ─▶ re-render (row gone)

6. Architecture — FormArray API

push(ctrl)End mein control add.
removeAt(i)Index i pe control hatao.
insert(i, ctrl) / at(i)Specific position insert / access.
clear()Saare controls hatao (reset list).
controlsArray — template *ngFor ke liye.
FormRecordDynamic keyed collection (object with dynamic keys) — array ka named cousin.

Custom Validators

Covers: ValidatorFn · factory (parameterized) validators · cross-field (group) validators · reusability · ControlValueAccessor peek

1. What is it? (Kya hai ye?)

Jab built-in validators kaafi na ho, tum apna validator likhte ho — ek function (control) => ValidationErrors | null. Isse business-specific rules (password strength, PAN format, cross-field match) enforce karte ho.

2. Why do we need it? (Zaroorat kyun hai?)

Har app ke apne rules hote hain jo Angular ship nahi karta — "password aur confirm-password same ho", "start-date < end-date", "IFSC valid ho". Custom validators inhe reusable, testable functions mein daal dete hain.

3. What problem does it solve?

Business logic ko template/component se hatakar declarative, reusable validator mein rakhta hai. Cross-field rules (jo single control ke bas ki baat nahi) group-level validators se solve hote hain.

4. Internal Working (Engine ke andar kya hota hai?)

Control-level vs Group-level

Control validator ek control ki value dekhta hai. Cross-field rule (do fields compare) ke liye validator ko parent FormGroup pe lagana padta hai — kyunki tabhi dono controls ek jagah accessible hote hain. Group validator group.get('password') aur group.get('confirm') dono padh sakta hai. Error group pe set hota hai (ya specific control pe setErrors).

any control changes ─▶ group re-validates ─▶ passwordMatch(group) runs

6. Architecture — 3 flavours

Simple ValidatorFn(control) => errors|null — ek field ka rule (e.g. no-whitespace).
Factory (parameterized)Function jo config leke ValidatorFn return kare (e.g. forbiddenValue('admin')).
Group (cross-field)Group pe lagta hai; multiple controls compare karta.

7. Real Project Example — cross-field

password-match.validator.ts
import { AbstractControl, ValidationErrors, ValidatorFn } from '@angular/forms';

export const passwordMatch: ValidatorFn = (group: AbstractControl): ValidationErrors | null => {
  const pass = group.get('password')?.value;
  const confirm = group.get('confirm')?.value;
  return pass === confirm ? null : { passwordMismatch: true };
};

// usage — validator GROUP pe (3rd arg / options)
form = this.fb.group({
  password: ['', Validators.required],
  confirm:  ['', Validators.required],
}, { validators: passwordMatch });
template
<small *ngIf="form.hasError('passwordMismatch') && form.get('confirm')?.touched">
  Passwords match nahi karte
</small>

9. Advanced Example — factory (parameterized)

forbidden-value.validator.ts
export function forbiddenValue(...banned: string[]): ValidatorFn {
  return (control: AbstractControl): ValidationErrors | null => {
    const v = (control.value || '').toLowerCase();
    return banned.includes(v) ? { forbidden: { value: control.value } } : null;
  };
}
// usage — reusable + configurable
username: ['', [Validators.required, forbiddenValue('admin', 'root', 'null')]],
Factory pattern kyun

Validator ko parameter chahiye (banned list, min-age) toh seedhe ValidatorFn nahi likh sakte — ek factory function likho jo config leke ValidatorFn return kare. Yahi Validators.minLength(8) bhi karta hai andar se.

10. Code Explanation — Line by Line

  1. Simple: (control) => ValidationErrors | null — value check, object ya null.
  2. Group: argument poora group hai; group.get('x') se multiple controls.
  3. Factory: outer function config leta, inner ValidatorFn return karta (closure).
  4. Error object ki key template mein hasError('key') se match karo.

11. Common Mistakes

Traps
  • Cross-field validator ko control pe lagana (chahiye group pe) → doosra field accessible nahi.
  • Factory ki jagah seedhe parameterized function pass karna → ValidatorFn signature mismatch.
  • Error object return karne ki jagah true/false return → Angular object expect karta hai (ya null).
  • Empty value pe error dena jab field optional hai → required ka kaam mat karo; empty pe null.
  • Group error ko control-level pe dikhane ki koshish bina setErrors — group error group pe hota hai.

12. Best Practices

Do this
  • Validators ko standalone pure functions rakho — easily unit-testable.
  • Parameterized rules → factory pattern (Validators.minLength jaisa).
  • Optional field: empty value pe null (valid) return karo; required alag validator.
  • Descriptive error keys + error object mein useful details (dynamic messages).

13. Performance

Custom sync validators har value change pe run — mahenga computation (heavy regex, large loops) avoid ya memoize karo. Cross-field group validator poore group ke har change pe chalta; keep it cheap. Server-dependent rules ko sync mein mat ghusaao — async validator (S-05) use karo.

15. Interview Questions

EASY Custom validator ka signature aur return?

(control: AbstractControl) => ValidationErrors | null. Valid pe null, invalid pe ek error object (jaise { forbidden: true }). Template mein hasError('forbidden') se check.

MEDIUM Password aur confirm-password match validate kaise? Validator kahan lagega?

Cross-field rule hai, isliye validator FormGroup pe lagega (control pe nahi) — fb.group({...}, { validators: passwordMatch }). Validator group.get('password') aur group.get('confirm') dono padhkar compare karta; mismatch pe group pe { passwordMismatch: true }.

MEDIUM Parameterized validator (jaise min-age 18) kaise banate ho?

Factory pattern: ek function jo config (minAge) leta hai aur ek ValidatorFn return karta hai (closure). function minAge(age){ return (c) => c.value < age ? {minAge:true} : null }. Yahi Angular ke apne Validators.minLength(n) ka pattern hai.

HARD Apna custom input component (jaise star-rating) ko reactive forms se kaise integrate karoge?

ControlValueAccessor interface implement karo: writeValue (model→view), registerOnChange (view→model), registerOnTouched, setDisabledState. Component ko NG_VALUE_ACCESSOR provider se register karo. Tab formControlName="rating" tumhare component pe waise hi kaam karega jaise native input pe — validators bhi apply honge.

SCENARIO Cross-field validator laga par confirm field pe error highlight nahi ho raha (error form-level pe hai). Kaise dikhao/clear karo?

Group validator ka error group pe rehta hai, control pe nahi. Do options: (1) template mein form.hasError('passwordMismatch') check karke confirm ke paas message dikhao; (2) validator ke andar group.get('confirm').setErrors({ mismatch: true }) set karo — par phir valid hone pe manually clear (setErrors(null)) bhi karna hoga warna atka rahega.
