S-01

HttpClient — Setup & Basics

Covers: provideHttpClient · withFetch · inject(HttpClient) · Observable-based (lazy/cold) · typed GET · subscribe vs AsyncPipe · JSON auto-parse

1. What is it? (Kya hai ye?)

HttpClient Angular ki injectable service hai jo browser se HTTP requests bhejती hai (REST APIs se baat karne ke liye). Har method (get, post...) ek Observable return karta hai — response asynchronously stream ke through aata hai, RxJS operators ke saath.

2. Why do we need it? (Zaroorat kyun hai?)

Native fetch/XMLHttpRequest se bhi request bhej sakte, par HttpClient Angular-integrated hai: automatic JSON parsing, typed responses, interceptors (auth/logging), RxJS operators (retry/cancel/switchMap), testability, aur error normalization (HttpErrorResponse). Ye sab manually likhna painful.

3. What problem does it solve?

HttpClient deta hai
  • Observable-based — retry, cancel, debounce, switchMap "free" mein.
  • Typed responses — http.get<User>(), compile-time safety.
  • Interceptors — cross-cutting concerns (auth token, logging) ek jagah (S-08).
  • Auto JSON parse + error normalization + testability.

4. Internal Working (Engine ke andar kya hota hai?)

Cold Observable + lazy

http.get() turant request nahi bhejता — ye ek cold Observable return karta hai. Request tabhi jaati hai jab koi subscribe() kare (ya AsyncPipe subscribe kare). Har subscribe ek nayi request fire karega. Andar: request interceptor chain se guzarti hai → backend (fetch by default) → response → back through interceptors → JSON parse → tumhare subscriber ko emit → phir Observable complete ho jaata (single value + complete).

Diagram — request lifecycle
 http.get<User>(url)  ─▶ returns COLD Observable (nothing sent yet)
        │
   .subscribe() ──▶ interceptor chain (out) ─▶ fetch backend ─▶ SERVER
        ▲                                                          │
        └── value + complete ◀─ JSON parse ◀─ interceptor chain (in) ◀┘

5. Execution Flow

inject(HttpClient) ─▶ http.get<User>('/api/user/1')
   ─▶ subscribe (or AsyncPipe) ─▶ request fired ─▶ interceptors ─▶ backend
   ─▶ response ─▶ auto JSON.parse ─▶ typed User ─▶ next(user) ─▶ complete()

6. Architecture (Building blocks)

provideHttpClient()Modern setup (app.config). Supersedes HttpClientModule.
withFetch()Fetch backend (v20 default). SSR-safe. XHR (withXhr) legacy.
withInterceptors([...])Functional interceptors (S-08).
HttpClientThe service — inject(HttpClient), methods return Observables.
HttpResponse / HttpErrorResponseFull response / error wrappers.

7. Real Project Example (Angular context)

app.config.ts
import { provideHttpClient, withFetch, withInterceptors } from '@angular/common/http';

export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(
      withFetch(),                        // fetch backend (default v20, SSR-safe)
      withInterceptors([authInterceptor]) // S-08
    ),
  ],
};
user.service.ts
import { Injectable, inject } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class UserService {
  private http = inject(HttpClient);
  private base = '/api/users';

  getUser(id: number): Observable<User> {
    return this.http.get<User>(`${this.base}/${id}`);   // typed, lazy
  }
}
component (AsyncPipe — no manual subscribe)
user$ = inject(UserService).getUser(1);
// template: <div *ngIf="user$ | async as user">{{ user.name }}</div>

9. Advanced Example — full response & observe

typescript
// sirf body (default): observe: 'body'
this.http.get<User>(url);

// poora response chahiye (status, headers)
this.http.get<User>(url, { observe: 'response' })
  .subscribe(resp => {
    console.log(resp.status);              // 200
    console.log(resp.headers.get('X-Total-Count'));
    console.log(resp.body);                // User
  });

// non-JSON (file download)
this.http.get('/report.pdf', { responseType: 'blob' });

10. Code Explanation — Line by Line

  1. inject(HttpClient) — modern DI; service ko HttpClient milta hai.
  2. http.get<User>(url) — generic <User> se response typed; ye ek cold Observable.
  3. observe: 'response' — sirf body ki jagah pura HttpResponse (status + headers).
  4. responseType: 'blob' — JSON ke alawa (files, text) ke liye.
  5. | async — AsyncPipe auto subscribe + unsubscribe (leak-free).

11. Common Mistakes

Traps
  • http.get() call karke subscribe na karna → request kabhi jaati hi nahi (cold/lazy).
  • Manual subscribe ka unsubscribe bhoolna → memory leak. AsyncPipe ya takeUntilDestroyed().
  • Ek hi Observable ko multiple jagah subscribe → multiple identical requests. shareReplay(1).
  • HttpClientModule import karna (deprecated) — provideHttpClient() use karo.
  • Response ko component se seedhe fetch karna — service layer mein rakho (separation).

12. Best Practices

Do this
  • HTTP calls ko services mein encapsulate karo; components sirf consume karein.
  • Templates mein AsyncPipe use karo — auto subscribe/unsubscribe.
  • Hamesha typed: http.get<T>(). any avoid.
  • provideHttpClient(withFetch()) — modern, SSR-safe default.

13. Performance

Duplicate requests sabse common perf issue — same data ko multiple subscribers ke liye shareReplay(1) se cache karo. Templates mein bina AsyncPipe ke multiple | async par same Observable → multiple requests. withFetch() modern backend, better SSR + streaming. Interceptors mein heavy work poore app ko slow kar sakta (S-08).

14. Edge Cases

Empty/204 responseBody null aa sakta — typing mein handle karo.
Non-2xxObservable error channel pe jaata (HttpErrorResponse) — S-05.
SSRFetch backend server pe safe; XHR SSR pe deprecated (v23 mein removal).
httpResource()v20 experimental — signal-based reactive HTTP; abhi Observable-based stable choice.

15. Interview Questions

EASY http.get() call kiya par request nahi gayi. Kyun?

HttpClient methods cold Observable return karte hain — lazy. Request tabhi fire hoti hai jab koi subscribe() kare (ya AsyncPipe subscribe kare). Bina subscribe ke kuch nahi hota. Aur har subscribe ek nayi request bhejता hai.

EASY Modern HttpClient setup kaise karte ho?

provideHttpClient(withFetch()) app.config ke providers mein. HttpClientModule deprecated hai. Interceptors withInterceptors([...]) se. Service mein inject(HttpClient).

MEDIUM Same API data 3 jagah chahiye bina 3 requests ke. Kaise?

Observable pe shareReplay(1) lagao — pehla subscriber request fire karega, response cache hoga, baaki subscribers cached value paayenge (multicast). Bina iske har subscriber alag request bhejta (cold Observable ki wajah se).

SCENARIO Response ke status code aur ek custom header (X-Total-Count) chahiye pagination ke liye, sirf body nahi. Kaise?

http.get<T>(url, { observe: 'response' }) pass karo — ab Observable pura HttpResponse emit karega. resp.status, resp.headers.get('X-Total-Count'), aur resp.body sab available.

16. Revision Notes

⚡ 30-second recall
  • Setup: provideHttpClient(withFetch()). inject(HttpClient). Methods → cold Observable (lazy).
  • Typed: http.get<T>(). observe:'response' for full response; responseType:'blob' for files.
  • AsyncPipe for auto sub/unsub. shareReplay(1) to dedupe. Services encapsulate calls.

S-02

CRUD Operations

Covers: GET · POST · PUT · PATCH · DELETE · request body · typed responses · PUT vs PATCH · idempotency · full CRUD service

1. What is it? (Kya hai ye?)

CRUD = Create, Read, Update, Delete — data ke 4 basic operations, jo HTTP methods pe map hote hain: POST (create), GET (read), PUT/PATCH (update), DELETE (delete). HttpClient har ek ke liye ek method deta hai.

2. Why do we need it? (Zaroorat kyun hai?)

Har data-driven app resources ko manage karta hai — user banao, padho, edit karo, delete karo. REST conventions in operations ko standard HTTP verbs se express karte hain, jisse APIs predictable + cacheable + tooling-friendly ho.

3. What problem does it solve?

Server pe data manipulate karne ka standardized, semantic tarika. Sahi verb use karne se caching, retries (idempotency), aur error semantics correct rehti hain.

4. Internal Working

Har method same pipeline (S-01) chalata hai, bas HTTP verb aur body alag. post/put/patch ek body argument lete hain (auto JSON-serialized). get/delete usually body nahi. Sab typed Observable return karte hain jo single value emit karke complete hoti hai.

5. Execution Flow

create ─▶ http.post<User>(url, body)   ─▶ 201 Created ─▶ new User
read   ─▶ http.get<User[]>(url)        ─▶ 200 OK      ─▶ User[]
update ─▶ http.put/patch<User>(url,b)  ─▶ 200 OK      ─▶ updated User
delete ─▶ http.delete<void>(url)       ─▶ 204 No Content

6. Architecture — verb map

VerbCRUDBody?Idempotent?
GETReadNoYes (safe)
POSTCreateYesNo
PUTUpdate (replace)Yes (full)Yes
PATCHUpdate (partial)Yes (partial)Usually
DELETEDeleteNoYes

7. Real Project Example (Angular context)

user.service.ts — full CRUD
@Injectable({ providedIn: 'root' })
export class UserService {
  private http = inject(HttpClient);
  private base = '/api/users';

  list():            Observable<User[]> { return this.http.get<User[]>(this.base); }
  get(id: number):   Observable<User>   { return this.http.get<User>(`${this.base}/${id}`); }
  create(u: NewUser):Observable<User>   { return this.http.post<User>(this.base, u); }
  update(id: number, u: User): Observable<User> { return this.http.put<User>(`${this.base}/${id}`, u); }
  patch(id: number, partial: Partial<User>): Observable<User> {
    return this.http.patch<User>(`${this.base}/${id}`, partial);
  }
  remove(id: number): Observable<void> { return this.http.delete<void>(`${this.base}/${id}`); }
}

9. Advanced Example — optimistic update pattern

typescript
toggleDone(todo: Todo) {
  const prev = todo.done;
  todo.done = !prev;                                  // UI turant update (optimistic)
  this.http.patch<Todo>(`/api/todos/${todo.id}`, { done: todo.done })
    .subscribe({
      error: () => { todo.done = prev; },              // fail → rollback
    });
}

10. Code Explanation — Line by Line

  1. post<User>(url, body) — body object auto JSON-serialize, response typed as User.
  2. put pura resource replace; patch sirf diye fields update (Partial<User>).
  3. delete<void> — usually 204, koi body nahi.
  4. Optimistic: UI pehle update, error pe rollback — snappy UX.

11. Common Mistakes

Traps
  • PUT ko partial body ke saath use karna → server missing fields ko wipe kar sakta. Partial = PATCH.
  • Response ko subscribe kiye bina "fire" samajhna — POST bhi cold hai, subscribe zaroori.
  • POST ko retry karna (non-idempotent) → duplicate records. Sirf idempotent verbs retry.
  • Error case handle na karna — CRUD fail ho sakti (validation, conflict). catchError (S-05).
  • Har mutation ke baad poori list re-fetch karna — optimistic update ya local cache update karo.

12. Best Practices

Do this
  • Ek resource = ek service; CRUD methods typed + consistent naming.
  • PUT = full replace, PATCH = partial — semantics respect karo.
  • Mutations pe optimistic update + rollback for snappy UX.
  • Base URL ko config/environment se; magic strings avoid.

13. Performance

Har mutation pe poora list re-fetch (waterfall) avoid karo — response se local state update karo ya optimistic. Bulk operations ke liye batch endpoints. GET responses ko shareReplay/cache karo jahan data slow-changing ho. PATCH (chhota payload) PUT se light jab sirf ek-do fields badle.

14. Edge Cases

201 with LocationPOST created resource ka URL Location header mein — observe:'response' se padho.
409 ConflictConcurrent update — optimistic-locking (version/etag) se handle.
DELETE returns bodyKuch APIs deleted resource return karte — <T> adjust.

15. Interview Questions

EASY PUT vs PATCH?

PUT poora resource replace karta hai (full representation bhejni chahiye — missing fields wipe ho sakti). PATCH partial update — sirf badle hue fields. Ek field toggle karna ho toh PATCH (chhota payload, safe), pura object replace karna ho toh PUT.

MEDIUM Kaunse HTTP verbs idempotent hain aur ye retry ke liye kyun important?

GET, PUT, DELETE idempotent (same request kai baar = same result). POST idempotent nahi (har baar naya resource). Isliye failed request retry karte waqt sirf idempotent verbs safe hain — POST retry se duplicate records ban sakte. Retry logic (S-06) mein verb check karo.

SCENARIO Todo ka checkbox toggle karte hi UI turant update karna hai, par server slow hai. Best pattern?

Optimistic update: UI ko turant naye state pe set karo, phir PATCH bhejo. Error case mein purani value pe rollback karo (error callback). Isse UX snappy rehti hai; server confirm karne ka wait nahi karna padta.

16. Revision Notes

⚡ 30-second recall
  • POST=create, GET=read, PUT=replace, PATCH=partial, DELETE=delete. All cold Observables.
  • Idempotent: GET/PUT/DELETE (retry-safe); POST not (no retry → dupes).
  • Body auto JSON-serialized. Optimistic update + rollback for snappy mutations.

S-03

Headers

Covers: HttpHeaders · immutability · set vs append · Content-Type · Authorization · reading response headers · headers via interceptor

1. What is it? (Kya hai ye?)

HTTP headers request/response ke saath jaane wali metadata key-value pairs hain — Content-Type, Authorization, Accept, custom (X-Request-Id). Angular inhe HttpHeaders class se represent karta hai.

2. Why do we need it? (Zaroorat kyun hai?)

Server ko batana padta hai kaun request kar raha (auth token), body kis format mein hai (content-type), kaun sa response format chahiye (accept), plus tracing/versioning custom headers. Ye sab headers ke through hota hai.

3. What problem does it solve?

Request/response ka metadata channel — data (body) se alag. Auth, content negotiation, caching directives, tracing — sab headers se.

4. Internal Working (Engine ke andar kya hota hai?)

HttpHeaders IMMUTABLE hai

Ye critical hai: HttpHeaders immutable hai. headers.set(...)/headers.append(...) original ko modify nahi karte — ek naya HttpHeaders instance return karte hain. Isliye return value ko capture karna zaroori. Ye immutability interceptors ko safe banati hai (ek request doosri ko affect nahi karti).

Diagram — immutability
 let h = new HttpHeaders();
 h.set('A', '1');          ✗ result discarded — h still empty!
 h = h.set('A', '1');      ✓ capture the NEW instance
        │
        └─▶ each set/append returns a brand-new HttpHeaders (original untouched)

5. Execution Flow

build HttpHeaders (chained set/append) ─▶ pass in options { headers }
        ─▶ request carries headers ─▶ server reads (auth/content-type)
        ─▶ response headers ─▶ read via resp.headers.get('X-...') (observe:'response')

6. Architecture — set vs append

set(name, value)Us header ki saari values replace karta hai (ek value).
append(name, value)Existing ke saath ek aur value add (multi-value headers).
delete(name) / has() / get()Remove / check / read (get single, getAll for multi).
Constructor { ... }new HttpHeaders({ Authorization: 'Bearer x' }) — object se init.

7. Real Project Example (Angular context)

typescript
import { HttpHeaders } from '@angular/common/http';

const headers = new HttpHeaders()
  .set('Content-Type', 'application/json')
  .set('Authorization', `Bearer ${token}`)
  .set('X-Request-Id', crypto.randomUUID());

  .subscribe(order => { /* ... */ });

10. Code Explanation — Line by Line

  1. new HttpHeaders() — khaali immutable headers.
  2. .set(...).set(...) — chaining; har call naya instance, final captured.
  3. { headers } — request options mein pass.
  4. Response headers ke liye observe:'response' zaroori (body-only mode mein headers nahi).

11. Common Mistakes

Traps
  • headers.set(...) ka return capture na karna (immutability bhoolna) → header laga hi nahi.
  • Content-Type: application/json manually set karna jab body object hai — HttpClient auto set karta hai; manual set FormData/blob ke saath todega.
  • Response headers padhne ke liye observe:'response' bhoolna → headers accessible nahi.
  • CORS: custom response headers server ko Access-Control-Expose-Headers se expose karne padte, warna browser hide karta.
  • Auth header har method mein manually — interceptor use karo.

12. Best Practices

Do this
  • Cross-cutting headers (auth, tracing) → interceptor (S-08), na ki har call.
  • Immutability yaad rakho — hamesha return value use karo.
  • JSON body pe Content-Type ko auto rehne do; FormData/blob pe manually mat set karo.
  • Sensitive tokens ko interceptor se centralize — leak/duplication avoid.

13. Performance

Headers khud lightweight. Immutability ka matlab har set ek naya object banata — hot paths mein hundreds of headers na banao, par typical use mein negligible. Interceptor mein header logic centralize karke redundant work avoid.

14. Edge Cases

Multi-value headerappend se multiple; padhne ke liye getAll(name).
Case-insensitivityHeader names case-insensitive (content-type == Content-Type).
FormData uploadContent-Type khud mat set karo — browser boundary ke saath set karega.

15. Interview Questions

EASY HttpHeaders mutable hai ya immutable? Impact?

Immutable. set/append/delete original modify nahi karte — naya HttpHeaders instance return karte hain. Isliye return value capture karna zaroori (h = h.set(...)). Ye interceptors ko safe banata hai (requests ek doosre ko affect nahi karti).

MEDIUM set vs append?

set us header ki saari existing values replace karta hai (single value). append existing ke saath ek aur value add karta hai (multi-value headers). Single overwrite → set, multiple values → append.

SCENARIO Auth token har request pe chahiye. Best jagah kahan?

Ek functional interceptor (S-08) mein — req.clone({ headers: req.headers.set('Authorization', ...) }). Har service call mein manually header lagana repetitive + error-prone hai; interceptor ise centralize karta hai aur token refresh bhi ek jagah handle hota.

16. Revision Notes

⚡ 30-second recall
  • HttpHeaders IMMUTABLE — set/append return new instance; capture it.
  • set = replace, append = add. Read response headers via observe:'response'.
  • JSON Content-Type auto; don't set for FormData. Auth/tracing → interceptor.

S-04

Params (Query)

Covers: HttpParams · immutability · set vs append · fromObject shorthand · array values · auto-encoding · pagination/filter/sort

1. What is it? (Kya hai ye?)

HttpParams URL ke query string (?page=2&sort=name) ko represent karta hai. Ye request ke saath jaane wale optional key-value filters hain — pagination, search, sort, filters ke liye.

2. Why do we need it? (Zaroorat kyun hai?)

Query params se ek hi endpoint ko customize karte hain: "users, page 2, sorted by name, active only". Manual string concat (url + '?page=' + page) encoding bugs deta hai (space, &, special chars). HttpParams ye safely + immutably handle karta hai.

3. What problem does it solve?

Safe query-string construction with auto URL-encoding, aur immutable/predictable param building (interceptors mein bhi safe).

4. Internal Working (Engine ke andar kya hota hai?)

HttpParams bhi IMMUTABLE

HttpHeaders ki tarah, HttpParams bhi immutable hai — params.set(...) naya instance return karta hai, original unchanged. Values automatically URL-encoded hoti hain (space → %20, etc). HttpClient params ko URL ke saath append karke request banata.

Diagram — params to URL
 new HttpParams()
   .set('page', '2')          each set() → NEW HttpParams (immutable)
   .set('sort', 'name')
   .append('tag', 'a')
   .append('tag', 'b')
        │
        ▼   auto-encoded
 GET /api/users?page=2&sort=name&tag=a&tag=b

5. Execution Flow

build HttpParams ─▶ options { params } ─▶ HttpClient appends encoded query to URL
        ─▶ GET /api/users?page=2&sort=name ─▶ server filters ─▶ response

6. Architecture — two ways to build

Chained set/appendnew HttpParams().set('page','2') — explicit, dynamic.
fromObject shorthandnew HttpParams({ fromObject: { page: 2, sort: 'name' } }) — concise.
set vs appendset replace; append multi-value (?tag=a&tag=b).
fromStringExisting query string se parse.

7. Real Project Example (Angular context)

user.service.ts
import { HttpParams } from '@angular/common/http';

listUsers(page: number, sort: string, filters: Record<string, string>) {
  let params = new HttpParams()
    .set('page', page)               // number auto-stringify (v15+)
    .set('sort', sort);

  for (const [k, v] of Object.entries(filters)) {
    if (v) params = params.set(k, v);  // ← reassign! immutable
  }

  return this.http.get<User[]>('/api/users', { params });
}

typescript

9. Advanced Example — array values + special chars

typescript
let params = new HttpParams()
  .append('tag', 'angular')
  .append('tag', 'rxjs')                 // ?tag=angular&tag=rxjs
  .set('q', 'hello world & more');       // auto-encoded → hello%20world%20%26%20more

10. Code Explanation — Line by Line

  1. new HttpParams() — empty immutable params.
  2. .set('page', page) — number bhi allowed (auto-stringify, v15+).
  3. params = params.set(k, v) — reassign kyunki immutable (loop mein critical).
  4. append — same key multiple values (array filters).
  5. Special chars auto-encoded — manual encoding ki zaroorat nahi.

11. Common Mistakes

Traps
  • Loop mein params.set() ka return na capture karna → sirf aakhri (ya koi nahi) param lagta. Reassign zaroori.
  • Manual string concat + encodeURIComponent bhoolna → broken URLs (space/&).
  • set use karke multi-value expect karna — multi ke liye append.
  • Empty/undefined values bhi bhejना → ?filter=undefined. Conditionally add.
  • Bahut sare params (long URL) → server/browser length limits; POST-with-body consider.

12. Best Practices

Do this
  • Immutability yaad rakho — loop mein hamesha reassign.
  • Simple cases → fromObject; dynamic/conditional → chained with reassign.
  • Empty values conditionally skip — clean URLs.
  • Encoding HttpClient pe chhodo; manual encodeURIComponent mat karo.

13. Performance

Params khud lightweight. Immutability se har set naya object — loop mein hundreds pe minor, typical use negligible. Query-based caching: same params = same URL = HTTP cache/shareReplay friendly. Debounce search params before firing (typeahead — S-07).

14. Edge Cases

S-05

Error Handling

Covers: catchError · HttpErrorResponse · status codes · client (status 0) vs server errors · throwError · fallback value · global vs local · error channel

1. What is it? (Kya hai ye?)

Error handling = jab request fail ho (network down, 404, 500, validation) toh gracefully handle karna — crash ke bajaye user-friendly message, fallback, ya retry. HttpClient failures ko Observable ke error channel pe bhejता hai, jise catchError se pakadte hain.

2. Why do we need it? (Zaroorat kyun hai?)

Network reliable nahi — timeouts, server errors, offline. Bina handling ke app crash ya blank screen. Proper handling se user ko clear feedback, retry option, aur app stable rehti hai.

3. What problem does it solve?

Failures ko catch karke UX (messages, fallback data) aur robustness (retry, logging) deta hai. Client-side errors (network, status 0) aur server errors (4xx/5xx) ko distinguish karke sahi action leta hai.

4. Internal Working (Engine ke andar kya hota hai?)

HttpErrorResponse anatomy

Non-2xx ya network failure pe HttpClient ek HttpErrorResponse ko error channel pe emit karta hai. Key fields: status (HTTP code, ya 0 for network/client-side/CORS failure), error (server ka error body), message, url. status === 0 ka matlab request server tak pahunchi hi nahi (offline/CORS/DNS).

Diagram — success vs error channels
 http.get() ─▶ subscribe({
                 next:  (data)  ─▶  2xx response (success channel)
                 error: (err: HttpErrorResponse) ─▶ non-2xx OR network fail
               })
   status === 0     ─▶ client-side / network / CORS (server tak nahi pahuncha)
   status 4xx/5xx   ─▶ server responded with error

5. Execution Flow

request ─▶ 500 error ─▶ error channel ─▶ catchError(err => ...)
   ├─ recover: return of(fallback)      ─▶ stream continues with fallback (VALID)
   └─ propagate: return throwError(()=>err) ─▶ error reaches subscriber's error cb

6. Architecture — catchError choices

Recoverreturn of(fallbackValue) — stream continue with default (e.g. empty list).
Rethrowreturn throwError(() => userFacingError) — propagate (transformed) to subscriber.
LocalService/component level catchError — specific handling.
GlobalError interceptor (S-08) — 401→logout, 500→toast, centralized.

import { catchError, throwError, of } from 'rxjs';
import { HttpErrorResponse } from '@angular/common/http';

getUser(id: number): Observable<User> {
  return this.http.get<User>(`/api/users/${id}`).pipe(
    catchError((err: HttpErrorResponse) => {
      if (err.status === 0)   return throwError(() => new Error('Network down — check connection'));
      if (err.status === 404) return throwError(() => new Error('User not found'));
      if (err.status === 401) { this.auth.logout(); }
      return throwError(() => new Error(err.error?.message ?? 'Something went wrong'));
    })
  );
}

S-06

Retry

Covers: retry(count) · retry({count,delay}) · exponential backoff · retryWhen (deprecated) · idempotency · retry only transient · retry + catchError order

1. What is it? (Kya hai ye?)

Retry = failed request ko automatically dobara try karna. Transient failures (temporary network glitch, 503, timeout) aksar retry pe pass ho jaate. RxJS ka retry operator failed Observable ko re-subscribe (yani nayi request) karta hai.

2. Why do we need it? (Zaroorat kyun hai?)

Networks flaky hote hain — ek momentary drop se user ko error dena bura UX hai. Automatic retry (with backoff) transient failures ko silently recover kar deta, robustness badhata.

3. What problem does it solve?

Transient failures ko user ke saamne aane se pehle recover karna — resilience. Par sirf transient errors pe (permanent 404 pe retry bekaar/harmful).

4. Internal Working (Engine ke andar kya hota hai?)

retry = re-subscribe

retry(n) jab source error kare, to use Observable ko dobara subscribe karta hai — aur cold HTTP Observable pe re-subscribe ka matlab ek nayi request. n attempts tak. RxJS 7.3+ mein retry({ count, delay }) config object — delay number (fixed) ya function (backoff) ho sakta. Purana retryWhen deprecated hai.

Diagram — retry with backoff
 request ─▶ ✗ error ─▶ wait 1s  ─▶ retry #1 ─▶ ✗
                       wait 2s  ─▶ retry #2 ─▶ ✗
                       wait 4s  ─▶ retry #3 ─▶ ✓ success  (or give up → error)
        (exponential backoff: delay doubles each attempt)

5. Execution Flow

http.get() ─▶ error ─▶ retry({count:3, delay}) ─┬─ attempts left ─▶ delay ─▶ re-request
                                                └─ exhausted ─▶ catchError / error channel

6. Architecture — retry forms

retry(3)Simple — 3 immediate retries (no delay). Basic.
retry({ count: 3, delay: 1000 })Fixed 1s delay between attempts (RxJS 7.3+).
retry({ count, delay: (err, i) => ... })Function delay → exponential backoff, conditional retry.
retryWhenDeprecated — use config delay function instead.

7. Real Project Example (Angular context)

exponential backoff + conditional
import { retry, timer, throwError } from 'rxjs';

getData() {
  return this.http.get<Data>('/api/data').pipe(
    retry({
      count: 3,
      delay: (error: HttpErrorResponse, retryCount) => {
        // sirf transient errors pe retry (5xx / network)
        if (error.status && error.status < 500 && error.status !== 0)
          return throwError(() => error);          // 4xx → don't retry
        return timer(Math.pow(2, retryCount) * 1000); // 2s, 4s, 8s
      },
    }),
    catchError(err => throwError(() => new Error('Failed after retries')))
  );
}

10. Code Explanation — Line by Line

  1. retry({ count: 3, delay: fn }) — 3 attempts, delay function se controlled.
  2. error.status < 500 && !== 0 → 4xx (client error) — retry pointless, throwError to stop.
  3. timer(2^retryCount * 1000) — exponential backoff (2s, 4s, 8s).
  4. catchError retry ke baad — jab saare attempts exhaust ho jaayein.

11. Common Mistakes

Traps
  • Non-idempotent requests (POST) ko blindly retry → duplicate records. Sirf GET/PUT/DELETE.
  • Permanent errors (400/404/401) pe retry → bekaar load, delay. Conditional retry (transient only).
  • Backoff ke bina immediate retries → server ko hammer (thundering herd).
  • retryWhen use karna — deprecated; retry({ delay }) function form.
  • catchError ko retry se pehle rakhna → retry kabhi trigger nahi hoga (error already caught).

12. Best Practices

Do this
  • Sirf idempotent + transient (5xx, network, timeout) pe retry.
  • Exponential backoff (+jitter) — server ko flood na karo.
  • Retry count cap (2-3); phir graceful failure.
  • Order: retry pehle, catchError baad. Global retry policy → interceptor (S-08).

13. Performance

Poorly-tuned retry sabse khatarnak — no backoff pe "retry storm" server ko down kar sakta (thundering herd). Exponential backoff + jitter (random delay add) se load spread hota. Retry cap zaroori warna slow endpoint pe attempts pile up. Idempotency check duplicate side-effects rokta.

14. Edge Cases

Timeout + retrytimeout(5000) phir retry — slow requests ko fail-fast + retry.
429 Too Many RequestsRetry-After header respect karo — us duration ka delay.

Permanent errors (400/404/401) retry pe kabhi pass nahi honge — bekaar load + delay. Aur POST (non-idempotent) retry se duplicates. Fix: conditional retry — delay function mein status check karke sirf transient (5xx/network/timeout) + idempotent verbs pe retry, baaki pe throwError.

Implement: retry({ count: 3, delay: (err, i) => timer(Math.pow(2, i) * 1000) }). Production mein jitter (random offset) bhi add karo taaki retries synchronize na ho.

⚡ 30-second recall

S-07

Cancellation

Covers: unsubscribe · switchMap auto-cancel · takeUntilDestroyed · typeahead search · AbortSignal (fetch) · why cancel matters

1. What is it? (Kya hai ye?)

Cancellation = ek in-flight (chal rahi) HTTP request ko rok dena, uski response ka intezaar chhodkar. RxJS mein unsubscribe() (ya operators jaise switchMap) HTTP Observable se unsubscribe karke underlying request ko abort kar deta hai.

2. Why do we need it? (Zaroorat kyun hai?)

Typeahead search: user har keystroke pe request bhejता hai — purani responses bekaar (ya out-of-order aa kar galat result dikha sakti). User navigate away kar gaya toh chal rahi request ki response ki zaroorat nahi. Cancellation waste + race conditions rokta.

3. What problem does it solve?

Bina cancellation — 2 problems
  • Race conditions: "reac" ki slow response "react" ki fast response ke baad aati → stale result dikhta.
  • Wasted work + leaks: navigate-away ke baad response aati, component gone → error/leak, bandwidth waste.

4. Internal Working (Engine ke andar kya hota hai?)

unsubscribe = abort

HTTP Observable ke teardown logic mein request abort hota hai. Jab tum subscription.unsubscribe() karte ho (ya switchMap naya value pe purani inner ko unsubscribe karta hai, ya takeUntilDestroyed component destroy pe unsubscribe karta hai), Angular underlying request (fetch backend ke saath AbortController se) cancel kar deta hai — sirf JS-side ignore nahi, asli network request bhi abort.

Diagram — switchMap auto-cancel (typeahead)
 type "r"  ─▶ switchMap ─▶ request(r)   ─── cancelled ✗ (superseded)
 type "re" ─▶ switchMap ─▶ request(re)  ─── cancelled ✗
 type "rea"─▶ switchMap ─▶ request(rea) ─── cancelled ✗
 type "reac"─▶ switchMap ─▶ request(reac) ─▶ ✓ only LATEST survives

5. Execution Flow

input change ─▶ debounceTime ─▶ distinctUntilChanged ─▶ switchMap(http.get)
        ─▶ new keystroke ─▶ previous inner Observable unsubscribed ─▶ request ABORTED
        ─▶ only latest response reaches subscriber

6. Architecture — cancellation tools

Manual unsubscribe()Store subscription, call .unsubscribe() — explicit but verbose.
switchMapNaya emission → purani inner request auto-cancel. Typeahead ka core.
takeUntilDestroyed()Component destroy pe auto-unsubscribe (in-flight cancel).
AsyncPipeTemplate destroy pe auto-unsubscribe.

Interceptors

Covers: HttpInterceptorFn · withInterceptors · req.clone (immutable) · chaining order · auth · error · loading · retry · HttpContext · DI-based (legacy)

1. What is it? (Kya hai ye?)

Interceptor ek middleware hai jo har HTTP request/response ke beech mein baithता hai. Har request outgoing pe (aur response incoming pe) intercept karke modify/observe/short-circuit kar sakta hai — auth header add, logging, error handling, loading spinner, retry — sab centralized.

2. Why do we need it? (Zaroorat kyun hai?)

Cross-cutting concerns (auth token har request pe, 401 pe logout, har request log) ko har service call mein repeat karna DRY nahi + error-prone. Interceptor inhe ek jagah, automatically, saari requests pe apply karta hai.

3. What problem does it solve?

Duplication + scattered logic. Auth, logging, error handling, loading UI, retry policy — ye sab request-agnostic hain. Interceptor inhe application-wide middleware bana deta hai, services ko clean rakhkar.

4. Internal Working (Engine ke andar kya hota hai?)

Chain + immutable clone

Interceptors ek chain banate hain — request declared order mein aage jaati hai (out), response ulte order mein wapas (in). Har interceptor (req, next) leta hai: next(req) call karke aage bhejता hai. Request immutable hai — modify karne ke liye req.clone({ ... }) se naya banana padta (original nahi chhedte). Response stream pe pipe karke observe/transform.

Diagram — interceptor chain
 REQUEST  ─▶ [logging] ─▶ [auth] ─▶ [retry] ─▶ backend ─▶ SERVER
                                                            │
 RESPONSE ◀─ [logging] ◀─ [auth] ◀─ [retry] ◀─ backend ◀───┘
            (reverse order on the way back — response handling)

5. Execution Flow

http.get() ─▶ interceptor[0] req.clone(+header) ─▶ next(req)
        ─▶ interceptor[1] ... ─▶ backend ─▶ response
        ─▶ interceptor[1] pipe(catchError) ─▶ interceptor[0] pipe(tap log) ─▶ subscriber

6. Architecture — functional (modern) vs DI (legacy)

Functional (HttpInterceptorFn)Plain function (req, next) => ..., inject() for DI. withInterceptors([...]). Recommended.
DI-based (class)HttpInterceptor class + HTTP_INTERCEPTORS + withInterceptorsFromDi(). Legacy, being phased out.
OrderArray declared order = execution order (out); reverse on response.
HttpContextPer-request metadata (e.g. skip-auth flag) via HttpContextToken.

export const SKIP_LOADING = new HttpContextToken(() => false);

export const loadingInterceptor: HttpInterceptorFn = (req, next) => {
  if (req.context.get(SKIP_LOADING)) return next(req);   // opt-out
  const loader = inject(LoadingService);
  loader.show();
  return next(req).pipe(finalize(() => loader.hide()));  // hide on complete/error
};

// per-request opt-out:
this.http.get('/api/poll', { context: new HttpContext().set(SKIP_LOADING, true) });

10. Code Explanation — Line by Line

  1. HttpInterceptorFn = (req, next) => ... — functional interceptor signature.
  2. inject(AuthService) — DI inside functional interceptor (injection context available).
  3. req.clone({ headers: ... }) — request immutable; clone karke modify.
  4. next(cloned) — chain mein aage forward.
  5. finalize(() => loader.hide()) — success/error dono pe spinner off.
  6. HttpContext — per-request flags (skip-auth, skip-loading).

11. Common Mistakes

Traps
  • Request ko directly mutate karna (req.headers.set without clone) → immutable, no effect. req.clone().
  • next(req) return/call na karna → chain toot jaati, request kabhi nahi jaati.
  • Interceptor order galat (error before auth, ya loading after error) → wrong behavior.
  • Loading spinner finalize ke bina → error case mein spinner atka.
  • Legacy HttpClientModule + HTTP_INTERCEPTORS — functional + withInterceptors use karo.
  • Auth interceptor ka token refresh recursive loop bina careful handling — 401 pe refresh phir retry, but infinite avoid.

12. Best Practices

Do this
  • Functional interceptors (withInterceptors) — tree-shakable, predictable, no circular-DI.
  • Ek interceptor = ek concern (auth / error / loading / retry alag).
  • Order intentional: loading → auth → (retry) → error (response handling reverse).
  • HttpContext se per-request opt-outs (skip-auth for public endpoints).
  • Spinner ko finalize se hide (success + error dono).

13. Performance

Interceptor har request pe chalta — heavy sync work (large logging, JSON stringify) har call slow karega. Keep them lean. Loading interceptor mein counter (concurrent requests) rakho warna ek request ke complete pe spinner off ho jaayega jab doosri chal rahi ho. Retry interceptor global backoff se server-friendly.

15. Interview Questions

EASY Interceptor kya hai, kis liye use hota hai?

HTTP requests/responses ke liye middleware. Har request/response ko intercept karke modify/observe/short-circuit karta hai. Common uses: auth header add, logging, error handling (401 logout), loading spinner, retry, caching. Cross-cutting concerns ko centralize karta hai.

MEDIUM Functional vs DI-based interceptor?

Functional (HttpInterceptorFn, plain function + inject(), withInterceptors([...])) — recommended: tree-shakable, predictable order, no class boilerplate, circular-DI safe. DI-based (class implementing HttpInterceptor + HTTP_INTERCEPTORS + withInterceptorsFromDi()) legacy hai aur phase-out ho raha.

MEDIUM Interceptor mein request kaise modify karte ho aur kyun clone?

HttpRequest immutable hai — directly modify nahi hota. req.clone({ headers: req.headers.set(...), ... }) se ek naya request banate hain aur next(cloned) forward karte hain. Immutability ensure karti hai ki ek request doosri ko affect na kare aur retries safe rahein.

HARD Multiple interceptors ka order kaise decide hota hai? Ek loading spinner interceptor kaise sahi banaoge?

withInterceptors([a, b, c]) — array order = request pe execution order (a→b→c), response pe reverse (c→b→a). Loading spinner: show() at start, finalize(() => hide()) (success+error dono). Multiple concurrent requests ke liye ek request-counter rakho (increment on show, decrement on finalize) taaki spinner tabhi hide ho jab saari requests complete, na ki pehli ke complete pe.

SCENARIO Auth interceptor har request pe token lagata hai, par ek public login/signup endpoint pe token nahi bhejना. Kaise skip karo?

HttpContextToken banao (SKIP_AUTH), interceptor mein req.context.get(SKIP_AUTH) check karke skip karo. Call site pe { context: new HttpContext().set(SKIP_AUTH, true) } pass karo. (Ya URL-based check, par HttpContext cleaner + type-safe hai.)
