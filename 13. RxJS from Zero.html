S-01

Observable — The Stream

Covers: stream concept · cold vs hot · lazy execution · producer · creation (of/from/interval/fromEvent/new Observable) · Observable vs Promise

1. What is it? (Kya hai ye?)

Observable ek stream of values over time hai — ek "pipe" jismein se 0, 1, ya infinite values aa sakti hain, aaj ya future mein. Ek array socho jiske elements ek saath nahi, samay ke saath aate hain. Mouse clicks, HTTP response, timer ticks — sab streams hain.

2. Why do we need it? (Zaroorat kyun hai?)

Async cheezein sirf "ek future value" nahi hoti — kabhi multiple values over time hoti hain (clicks, keystrokes, WebSocket messages). Promise sirf ek value deta hai aur cancel nahi hota. Observable multiple values + cancellation + composition (operators) deta hai — reactive programming ka core.

3. What problem does it solve?

Promise ki limitations
  • Sirf ek value (stream of clicks handle nahi hota).
  • Cancel nahi ho sakta (fire ho gaya toh ho gaya).
  • Eager — banate hi chal padta.
  • Composition weak — retry/debounce/switch manually.
Observable

Multiple values · cancellable · lazy · 100+ operators se composable. Ek unified model for all async.

4. Internal Working (Engine ke andar kya hota hai?)

Lazy + Cold ka matlab

Observable banane se kuch nahi hota — ye sirf ek "recipe" hai (ek function jo batata hai kaise values produce karni hain). Kaam tabhi shuru hota hai jab koi subscribe() kare. Ye "lazy" hai. Aur cold Observable har subscriber ke liye producer ko dobara chalata hai — 3 subscribers = 3 independent executions (jaise 3 alag HTTP requests). Iske ulta hot Observable ek shared producer se sab subscribers ko same values bhejता hai (S-04 Subjects).

Diagram — cold vs hot
 COLD (default): each subscriber gets its OWN execution
   sub A ─▶ [producer runs] ─▶ 1,2,3
   sub B ─▶ [producer runs again] ─▶ 1,2,3   (independent)

 HOT (Subject-based): one shared producer, multicast
   producer ─▶ 1,2,3 ─┬─▶ sub A
                      └─▶ sub B   (same values, shared)

5. Execution Flow

 new Observable(fn)  ─▶ NOTHING happens (lazy recipe)
        │
   .subscribe(observer) ─▶ fn(observer) runs
        ─▶ observer.next(v)  (0..many times, over time)
        ─▶ observer.complete()  OR  observer.error(e)   ← terminal (stream ends)
        ─▶ teardown/cleanup runs

6. Architecture — creation functions

of(1,2,3)Given values ek-ek karke emit, phir complete. Synchronous.
from([...]) / from(promise)Array/iterable/Promise ko Observable mein convert.
interval(1000) / timer(d, p)Har interval pe number emit (infinite).
fromEvent(el, 'click')DOM events ko stream banata (hot).
new Observable(fn)Custom producer — full control.
EMPTY / NEVER / throwErrorSpecial: turant complete / kabhi nahi / error.

9. Advanced Example — custom Observable + teardown

typescript
import { Observable } from 'rxjs';

const timer$ = new Observable<number>(subscriber => {
  let count = 0;
  const id = setInterval(() => subscriber.next(count++), 1000);
  // teardown — unsubscribe pe chalega (cleanup!)
  return () => { clearInterval(id); console.log('cleaned up'); };
});

const sub = timer$.subscribe(v => console.log(v));
setTimeout(() => sub.unsubscribe(), 3500);   // 0,1,2 then cleanup

10. Code Explanation — Line by Line

  1. new Observable(subscriber => {...}) — producer function; subscribe pe chalta hai.
  2. subscriber.next(v) — ek value emit; jitni baar chaho.
  3. return () => {...} — teardown; unsubscribe/complete pe cleanup (leak-prevention).
  4. sub.unsubscribe() — stream stop + teardown trigger.

11. Common Mistakes

Traps
  • Observable ko "already running" samajhna — ye lazy hai, subscribe bina kuch nahi.
  • Cold Observable ko multiple jagah subscribe karke wonder karna ki request 3 baar kyun gayi (har subscribe = nayi execution).
  • Infinite Observable (interval) ko unsubscribe na karna → leak.
  • Custom Observable mein teardown na dena → resources (timers/listeners) leak.
  • Observable ko Promise samajhna (ek value expect karna) — multiple emit ho sakte.

12. Best Practices

Do this
  • Streams ko $ suffix se naam do (users$) — convention.
  • Custom Observables mein hamesha teardown return karo.
  • Shared cold data → shareReplay (S-08) se multicast + cache.
  • Prefer built-in creation functions (of/from/fromEvent) over manual new Observable.

13. Performance

Cold Observable ka biggest gotcha: multiple subscribers = multiple executions (duplicate HTTP). shareReplay(1) se ek execution multicast karo. Infinite streams (interval, fromEvent) bina unsubscribe leak + CPU waste karte. Lazy nature actually perf-friendly — jab tak zaroorat na ho, kuch execute nahi hota.

14. Edge Cases

Synchronous emitof(1,2,3) subscribe hote hi sab emit — "over time" jaruri nahi async.
Never completesinterval/fromEvent khud complete nahi hote — take/takeUntil chahiye.
Observable vs PromisePromise eager + single + non-cancellable; Observable lazy + multi + cancellable.

15. Interview Questions

EASY Observable vs Promise?

Promise: ek future value, eager (banate hi chalta), non-cancellable, no operators. Observable: 0..many values over time, lazy (subscribe pe chalta), cancellable (unsubscribe), 100+ operators se composable. Multiple values ya cancellation chahiye → Observable.

EASY Cold vs Hot Observable?

Cold: har subscriber ke liye producer alag se chalta hai (independent execution) — jaise HTTP (har subscribe = nayi request). Hot: ek shared producer sab subscribers ko same values multicast karta (jaise Subject, fromEvent). Cold ko hot banana → share/shareReplay.

MEDIUM "Observable lazy hota hai" — iska practical impact?

Observable banane pe producer chalta hi nahi — sirf subscribe pe. Practical: http.get() call karke subscribe na karo toh request kabhi nahi jaati. Aur har subscribe cold Observable ko dobara chalata (duplicate requests). Isliye AsyncPipe/shareReplay important.

SCENARIO Ek service method 3 components use karte hain aur network tab mein 3 identical requests dikhte hain. Kyun aur fix?

HTTP Observable cold hai — har component ka subscribe (ya | async) alag execution trigger karta hai. Fix: source pe shareReplay(1) lagao — pehla subscriber request fire karega, response cache hoga, baaki cached value paayenge (ek request, multicast).

16. Revision Notes

⚡ 30-second recall
  • Observable = stream of 0..many values over time. LAZY (subscribe pe chalta) + cancellable + composable.
  • COLD = per-subscriber execution (HTTP); HOT = shared multicast (Subject/fromEvent).
  • Create: of/from/interval/timer/fromEvent/new Observable. Custom → return teardown.

S-02

Observer & Subscription

Covers: Observer (next/error/complete) · subscribe overloads · Subscription object · unsubscribe · teardown · terminal events · finite vs infinite

1. What is it? (Kya hai ye?)

Observer woh consumer hai jo stream ko "sunta" hai — ek object with 3 callbacks: next (har value), error (fail), complete (stream khatam). Subscription woh object hai jo subscribe() return karta hai — ise unsubscribe() karke stream band karte ho.

2. Why do we need it? (Zaroorat kyun hai?)

Observable "recipe" hai, par usse values nikaalne ke liye koi consume karna padta — that's the Observer. Aur running stream ko rokne (cancellation, cleanup) ke liye Subscription ka handle chahiye. In dono ke bina Observable useless hai.

3. What problem does it solve?

Observer 3 alag channels deta hai — success values, error, completion — clean separation. Subscription lifecycle management deta hai: jab tak chahiye suno, phir band karke resources free karo (leak prevention).

4. Internal Working (Engine ke andar kya hota hai?)

next/error/complete contract

Ek Observable zero ya more next emit kar sakta hai, phir ya to ek complete ya ek error — dono nahi, aur uske baad kuch nahi. error/complete terminal hain: stream khatam, teardown chalta, further emissions ignore. Isliye error ke baad next nahi aata. Ye "grammar" (next* (error | complete)?) reactive contract hai.

Diagram — the contract
 next ─▶ next ─▶ next ─▶ complete    ✓ (finite, success)
 next ─▶ next ─▶ error               ✓ (failed — terminal)
 next ─▶ next ─▶ ... (never ends)    ← infinite (interval) — YOU must unsubscribe

 grammar:  next* (error | complete)?     terminal = teardown runs

5. Execution Flow

 obs.subscribe({ next, error, complete })
        │
        ├─ producer emits ─▶ next(v)  (repeat)
        ├─ success ─▶ complete() ─▶ teardown ─▶ (auto-unsubscribed)
        ├─ failure ─▶ error(e)   ─▶ teardown ─▶ (auto-unsubscribed)
        └─ manual  ─▶ sub.unsubscribe() ─▶ teardown (stop early)

6. Architecture — subscribe forms

Full observersubscribe({ next, error, complete }) — teeno callbacks.
next onlysubscribe(v => ...) — shorthand, sirf values.
SubscriptionReturn object — unsubscribe(), closed, add(child).
Auto-cleanupcomplete/error → auto-unsubscribe; infinite → manual.

10. Code Explanation — Line by Line

  1. subscribe({ next, error, complete }) — Observer object; teen channels.
  2. next har value pe; error ek baar (terminal); complete ek baar (terminal).
  3. const sub = ...subscribe(...) — Subscription handle capture.
  4. sub.unsubscribe() — stream stop + teardown; infinite streams ke liye must.

11. Common Mistakes

Traps
  • Infinite stream subscribe karke unsubscribe na karna → #1 memory leak (S-10).
  • error callback na dena → unhandled error crash/console noise.
  • Error ke baad next expect karna — error terminal hai, stream khatam.
  • Subscription.add() chaining pe over-rely — flawed pattern; takeUntil/AsyncPipe better.
  • Nested subscribes (subscribe ke andar subscribe) → callback hell + leaks. Flattening operators (S-05) use karo.

12. Best Practices

Do this
  • Templates mein AsyncPipe — auto subscribe + unsubscribe (leak-free).
  • Manual subscribe zaroori ho toh takeUntilDestroyed() se cleanup (S-10).
  • Hamesha error handle karo (ya catchError).
  • Nested subscribe kabhi nahi — flattening operators (switchMap etc.).

13. Performance

Leaked subscriptions (khaas infinite streams) memory + CPU khaate hain — har destroy nahi hui subscription callback chalati rehti. AsyncPipe/takeUntilDestroyed se automatic cleanup. Nested subscribes na sirf leaky balki uncontrolled concurrency — flattening operators controlled + cancellable.

14. Edge Cases

Double unsubscribeSafe — no-op (closed already true).
Unsubscribe before emitProducer teardown chalta, koi value nahi aati.
Finite auto-completeHTTP/of complete karte hi auto-unsubscribe — manual zaroori nahi (par AsyncPipe still cleaner).

15. Interview Questions

EASY Observer ke 3 methods?

next(value) — har emitted value. error(err) — failure (terminal, stream khatam). complete() — success end (terminal). Contract: next* phir optionally ek error YA complete, dono nahi.

MEDIUM Kaunse Observables ko manually unsubscribe karna zaroori hai?

Infinite/long-lived streams jo khud complete nahi hote — interval, fromEvent, Subject-based, valueChanges. Finite streams (HTTP, of) complete pe auto-unsubscribe ho jaate. Par best practice: AsyncPipe/takeUntilDestroyed se saare cases handle karo.

SCENARIO Ek component interval(1000) subscribe karta hai, phir destroy ho jaata, par console mein numbers aate rehte hain. Fix?

Subscription destroy pe cleanup nahi hui — leak. Fix: interval(1000).pipe(takeUntilDestroyed()).subscribe(...) — component destroy pe auto-unsubscribe. Ya AsyncPipe use karo template mein. Manual: ngOnDestroy mein sub.unsubscribe().

16. Revision Notes

⚡ 30-second recall
  • Observer = { next, error, complete }. Contract: next* (error|complete)? — terminal ends stream.
  • subscribe() → Subscription; unsubscribe() stops + teardown. Infinite → MUST unsubscribe.
  • Prefer AsyncPipe / takeUntilDestroyed(). Never nest subscribes.

S-03

Operators

Covers: pipe() · pure functions · creation vs pipeable operators · map/filter/tap/take/scan · marble diagrams · chaining · immutability

1. What is it? (Kya hai ye?)

Operators wo functions hain jo ek Observable ko leke naya Observable return karte hain — stream ko transform, filter, combine karke. map, filter, tap, debounceTime — sab operators. Inhe .pipe() ke andar chain karte ho.

2. Why do we need it? (Zaroorat kyun hai?)

Raw stream se seedhe kaam nahi chalta — usse transform karna padta (map values), filter (sirf valid), debounce (rate limit), combine (dependent data). Operators ye declaratively karte hain, imperative loops/flags ke bina. Yehi RxJS ki asli power hai.

3. What problem does it solve?

Async logic ka spaghetti (nested callbacks, manual state, flags) ko clean, composable, declarative pipeline mein badalna. "Har keystroke pe, 300ms wait karke, agar value badli ho, API call karo, purani cancel karke" — ek pipe mein.

4. Internal Working (Engine ke andar kya hota hai?)

pipe = function composition

Har pipeable operator ek function hai jo Observable leta hai aur naya Observable deta hai. pipe(a, b, c) unhe left-to-right compose karta hai: source → a → b → c. Har operator source ko subscribe karta hai, values ko transform karke apne subscriber ko forward karta hai. Operators pure hain — original stream ko modify nahi karte, naya banate hain (immutable).

Diagram — pipe composition (marble)
 source:        --1--2--3--4--|
 .pipe(
   filter(x=>x%2==0)   --2-----4--|      keep even
   map(x=>x*10)   )     --20----40-|      transform
 result:               --20----40-|

5. Execution Flow

 source$ .pipe(map(f), filter(g), take(2))
        │
   subscribe ─▶ source emits ─▶ map(f) ─▶ filter(g) ─▶ take(2) ─▶ your observer
              (each value flows through the chain, left to right)

6. Architecture — operator categories

Transformationmap, scan, pluck, flattening (switchMap etc. — S-05)
Filteringfilter, take, first, distinctUntilChanged, debounceTime
Combinationmerge, concat, forkJoin, combineLatest (S-06)
Utilitytap (side-effects), delay, finalize, timeout
ErrorcatchError, retry (S-09)
Creation (not piped)of, from, interval — Observables banate hain (S-01)

typescript

10. Code Explanation — Line by Line

  1. .pipe(...) — operators ki chain; left-to-right execute.
  2. filter(pred) — sirf true wali values aage.
  3. map(fn) — har value ko transform.
  4. tap(fn) — side-effect (log/debug), value badalta nahi.
  5. scan(reducer, seed) — running accumulate (jaise Array.reduce, par per-emit).

11. Common Mistakes

Traps
  • Operators ke saath source modify hone ki umeed — ye pure hain, naya stream dete hain.
  • map mein side-effects (state mutate) — map pure hona chahiye; side-effects tap mein.
  • Creation operator (of) ko .pipe ke andar use karna — creation vs pipeable confuse.
  • Nested subscribe jab flattening operator (switchMap) chahiye — S-05.
  • RxJS 6 style import (rxjs/operators) — v7+ mein sab 'rxjs' se import ho jaata.

12. Best Practices

Do this
  • Side-effects sirf tap mein; map/filter pure rakho.
  • Chain ko readable rakho — ek pipe mein logical flow.
  • Marble diagram se sochо — "input marbles → output marbles".
  • Import 'rxjs' se (v7+ unified); tree-shaking automatic.

13. Performance

Har operator ek subscription layer add karta — typically negligible, par hot paths mein zaroorat se zyada operators avoid. tap ke bhaari side-effects har emit pe chalte. distinctUntilChanged/filter jaldi lagakar downstream work kam karo. Pure operators tree-shakable → chhota bundle.

14. Edge Cases

Operator order mattersfilter phir map vs map phir filter — alag semantics + perf.
Empty streammap/filter kuch nahi karte agar source complete without emit.
Custom operatorEk function jo Observable→Observable — reusable pipe logic.

15. Interview Questions

EASY pipe() kya karta hai?

Multiple operators ko left-to-right compose karta hai. Source Observable in operators se guzarta hai (transform/filter/etc.) aur naya Observable banta hai — original unchanged (pure). source.pipe(op1, op2, op3).

EASY map vs tap?

map value ko transform karke naya value emit karta hai (stream badalta). tap side-effect (log/debug/analytics) karta hai par value ko as-is aage bhejta (stream unchanged). Transform → map; side-effect → tap.

MEDIUM Creation vs pipeable operators?

Creation operators (of, from, interval, fromEvent) naya Observable banate hain — pipe ke bahar. Pipeable operators (map, filter, switchMap) existing Observable ko .pipe() ke andar transform karte hain. Ek stream shuru karta, doosra use manipulate.

SCENARIO Ek pipe mein map ke andar tumne component property update kar di aur ajeeb bugs aa rahe. Kyun?

map pure hona chahiye — sirf value transform, koi side-effect nahi. State mutation map mein daalne se multiple subscriptions/re-runs pe unpredictable behavior aata. Side-effects ko tap mein move karo, ya better — state ko stream se derive karo (AsyncPipe).

16. Revision Notes

⚡ 30-second recall
  • Operators: Observable → new Observable (pure). Chain via .pipe(a, b, c) left-to-right.
  • Transform=map/scan, filter=filter/take/distinctUntilChanged, side-effect=tap.
  • Creation (of/from) vs pipeable (map/filter). Import from 'rxjs' (v7+).

Subjects (4 types)

Covers: Subject · BehaviorSubject · ReplaySubject · AsyncSubject · Observable+Observer both · multicast/hot · state management · asObservable

1. What is it? (Kya hai ye?)

Subject ek special object hai jo Observable AUR Observer dono hai — yani tum ise subscribe bhi kar sakte ho, aur ispe next() karke values push bhi kar sakte ho. Ye hot aur multicast hai: ek Subject ke saare subscribers ko same values milti hain. Iske 4 variants alag "memory" behavior dete hain.

2. Why do we need it? (Zaroorat kyun hai?)

Regular Observable cold + unicast hai. Par kabhi tumhe chahiye: ek jagah se value push karna aur multiple listeners ko bhejna (event bus, shared state, cross-component communication). Subject exactly ye deta hai — imperative "emit" + multicast.

3. What problem does it solve?

Multicasting (ek source, multiple listeners, same values) aur imperative emission (code se next() push). Angular mein: simple state services, event buses, cross-component signals — sab Subjects se (ya ab signals se).

4. Internal Working (Engine ke andar kya hota hai?)

4 variants = alag "memory"

Sabhi Subjects multicast hain; farak sirf naye subscriber ko kya milta hai:
• Subject — subscribe ke baad wali values only (no memory).
• BehaviorSubject — ek initial/latest value hamesha yaad; subscribe pe turant current value.
• ReplaySubject(n) — last n values yaad; subscribe pe woh sab replay.
• AsyncSubject — sirf last value, woh bhi complete hone ke baad.

Diagram — what a late subscriber gets
 emitted:  1 -- 2 -- 3 --[late subscriber joins here]-- 4 -- complete

 Subject          late sub gets:  4              (nothing before join)
 BehaviorSubject  late sub gets:  3, 4           (latest, then new)
 ReplaySubject(2) late sub gets:  2, 3, 4        (last 2 buffered, then new)
 AsyncSubject     late sub gets:  4              (only last, AFTER complete)

5. Execution Flow

 const s = new BehaviorSubject(0);
 s.subscribe(A);        A ─▶ 0 (initial, immediate)
 s.next(1);             A ─▶ 1
 s.subscribe(B);        B ─▶ 1 (current value on join)
 s.next(2);             A ─▶ 2,  B ─▶ 2   (multicast to both)

6. Architecture — the 4 types

SubjectNo memory. Pure event bus. Late subscriber = future values only.
BehaviorSubject(init)Needs initial value; always has "current". .value sync read. State ke liye best.
ReplaySubject(n)Buffers last n (optionally time-window). Late subscriber replay.
AsyncSubjectOnly final value, only on complete. Rare (jaise ek computation ka result).

7. Real Project Example — simple state service

cart.service.ts
@Injectable({ providedIn: 'root' })
export class CartService {
  private itemsSubject = new BehaviorSubject<Item[]>([]);   // initial: empty
  items$ = this.itemsSubject.asObservable();               // expose read-only

  add(item: Item) {
    this.itemsSubject.next([...this.itemsSubject.value, item]);  // .value = current
  }
  get count() { return this.itemsSubject.value.length; }
}
// any component: cartService.items$ | async  → always current cart, multicast

Flattening Operators

Covers: higher-order Observables · switchMap · mergeMap · concatMap · exhaustMap · when to use which · nested-subscribe fix

1. What is it? (Kya hai ye?)

Flattening operators tab use hote hain jab ek Observable ki har value se ek naya Observable banta hai (jaise "har search term pe ek HTTP call"). Ye ek higher-order Observable (Observable of Observables) ko flatten karke ek single stream banate hain. Chaar variants alag concurrency strategy dete hain.

2. Why do we need it? (Zaroorat kyun hai?)

Common pattern: ek stream (clicks/keystrokes/route params) ki har value pe async operation (HTTP). Agar tum map use karo toh Observable-of-Observables milta hai (nested, useless). Aur nested subscribe callback hell + no cancellation. Flattening operators inner Observable ko subscribe + flatten karke, cancellation/ordering handle karte hain.

3. What problem does it solve?

Nested subscribe — anti-pattern
// ❌ callback hell + no cancel + leak
this.search$.subscribe(term => {
  this.http.get(`/api?q=${term}`).subscribe(res => { ... });
});
✅ flattening operator
this.search$.pipe(switchMap(term => this.http.get(`/api?q=${term}`))).subscribe(...)

4. Internal Working — the 4 strategies

Concurrency ka farak (RAT-DE mnemonic nahi — bas samjho)

Jab outer se nayi value aaye jabki purani inner abhi chal rahi ho, tab kya karein:
• switchMap — purani inner ko cancel karo, nayi pe switch. (Latest jeetta.)
• mergeMap — sabko parallel chalao, jaise-jaise complete ho emit. (No order.)
• concatMap — queue karo; ek complete hone ke baad agli. (Order preserved.)
• exhaustMap — jab tak current chal rahi, nayi ko ignore karo. (First jeetta.)

Diagram — same input, 4 behaviors
 outer:   --A------B--C-------|   (each maps to inner taking ~2 units)

 switchMap:  A cancelled by B, B cancelled by C ─▶ only C result
 mergeMap:   A, B, C all run in parallel ─▶ all results (any order)
 concatMap:  A fully, then B, then C ─▶ all results, IN ORDER
 exhaustMap: A runs; B,C ignored (arrived while A busy) ─▶ only A result

5. Execution Flow

 outer emits term ─▶ project(term) → inner Observable (e.g. http.get)
        │
   switchMap: unsubscribe previous inner ─▶ subscribe new ─▶ flatten value
   (concurrency strategy decides what happens to overlapping inners)

6. Architecture — decision table

OperatorOn overlapUse case
switchMapCancel previousSearch/typeahead, route-param fetch — only latest matters
mergeMapAll parallelIndependent parallel tasks (multiple uploads) — order irrelevant
concatMapQueue, in orderSequential mutations (ordered saves) — order matters
exhaustMapIgnore new while busyLogin/submit button — prevent double-submit

  • "Ignore while busy" (submit/login) → exhaustMap.
  • Nested subscribe kabhi nahi.

Combination Operators

Covers: forkJoin · combineLatest · zip · parallel HTTP · dependent streams · emit timing differences · merge/concat

1. What is it? (Kya hai ye?)

Combination operators multiple Observables ko ek mein jodte hain. Teen sabse important: forkJoin (sab complete hone pe last values), combineLatest (kisi ke change pe sabki latest), zip (index-wise pair). Farak: kab aur kya emit hota hai.

2. Why do we need it? (Zaroorat kyun hai?)

Aksar ek screen ko multiple sources chahiye — 3 parallel API calls jinke sab aane pe render karo; ya do filters jinme se koi bhi badle toh list refresh; ya do streams ko pair karo. Combination operators ye coordination declaratively karte hain.

3. What problem does it solve?

Multiple async sources ko coordinate karna — bina manual counters/flags ("teeno aa gaye kya?"). forkJoin parallel-then-combine, combineLatest reactive-latest, zip strict-pairing deta hai.

4. Internal Working — emit timing

Teeno ka core farak

• forkJoin — saare source complete hone ka wait; phir har ek ki last value ek array/object mein, ek baar. (Finite sources chahiye — jaise HTTP.)
• combineLatest — har source ne kam-se-kam ek value emit ki ho, phir kisi bhi source ke change pe sabki latest combine karke emit. (Ongoing.)
• zip — sources ko index-wise pair — 1st with 1st, 2nd with 2nd. Slowest source dictate karta.

Diagram — timing
 A: --1------2----|      B: ----a------b--|

 forkJoin:      -------------[2,b]|     (both complete → last values, once)
 combineLatest: ----[1,a]-[2,a]-[2,b]   (any change → latest of each)
 zip:           ----[1,a]----[2,b]      (pair by index: 1&a, 2&b)

5. Execution Flow

 forkJoin([a$, b$, c$]) ─▶ wait ALL complete ─▶ emit [lastA, lastB, lastC] ─▶ complete
 combineLatest([a$, b$]) ─▶ each emitted once ─▶ on any emit ─▶ [latestA, latestB]
 zip([a$, b$]) ─▶ pair index 0, index 1, ... ─▶ [a0,b0], [a1,b1]

6. Architecture — pick one

forkJoinParallel finite calls, sab aane pe combine. Multiple HTTP at once. "Promise.all" of RxJS.
combineLatestOngoing streams; any change → latest snapshot. Dependent filters/form.
zipIndex-wise pairing. Rare — jab do streams ko strictly pair karna ho.
merge / concatInterleave (any) / sequential (one after other) — combine values, not tuples.

Rate-Limiting & Timing

Covers: debounceTime · throttleTime · distinctUntilChanged · auditTime · sampleTime · debounce vs throttle · when to use

1. What is it? (Kya hai ye?)

Rate-limiting operators high-frequency streams (keystrokes, scroll, resize, clicks) ko "seedha kar dete hain" — kitni baar aur kab value aage jaaye control karke. Do main: debounceTime (rukne ka wait) aur throttleTime (fixed interval pe ek).

2. Why do we need it? (Zaroorat kyun hai?)

User har keystroke pe API call = server flood + wasted work. Scroll har pixel pe handler = jank. Rate-limiting emissions ko kam karke performance + UX improve karta — sirf meaningful moments pe react karo.

3. What problem does it solve?

Event storms (typing, scrolling, resizing) ko manageable rate mein convert karna. Redundant work (same value pe dobara), noise (har micro-event) ko filter karke sirf relevant emissions.

4. Internal Working — debounce vs throttle

Core farak

• debounceTime(ms) — value ko hold karo; agar ms tak koi nayi value na aaye tabhi emit. Yani "typing ruk gayi" pe fire. Har naya value timer reset karta.
• throttleTime(ms) — pehli value turant emit, phir ms tak baaki ignore, phir agli allow. Yani "har ms mein max ek". Continuous stream ko steady rate pe.

Diagram — debounce vs throttle
 input:        x-x-x-x--------x-x----|   (bursts of events)

 debounceTime: ---------x---------x--|   (emit after pause — last of burst)
 throttleTime: x----------x---x-----|   (first, then 1 per window)

 keystroke ─▶ debounceTime(300) ─▶ (wait 300ms silence) ─▶ distinctUntilChanged
        ─▶ (skip if same as last) ─▶ switchMap(http) ─▶ results

throttleTime(ms)Max one per ms window. Scroll/resize/rapid clicks. First of window.
auditTime(ms)Like throttle but emits last value of window (not first).
sampleTime(ms)Emit latest value every ms tick (periodic snapshot).
distinctUntilChanged()Skip if value == previous. Paired with debounce.

Multicasting & Sharing

Covers: shareReplay · share · unicast vs multicast · refCount footgun · caching HTTP · cold→hot · bufferSize

1. What is it? (Kya hai ye?)

Multicasting = ek Observable execution ko multiple subscribers ke saath share karna (cold → hot). shareReplay aur share operators ek source ko multicast karke duplicate executions (jaise duplicate HTTP) rokte hain, aur late subscribers ko cached value dete hain.

2. Why do we need it? (Zaroorat kyun hai?)

Cold Observable har subscriber ke liye dobara chalta — 3 components ek HTTP service subscribe karein = 3 identical requests. Multicasting ek execution ko share karta, aur shareReplay result cache karke naye subscribers ko turant deta.

3. What problem does it solve?

Duplicate work (duplicate HTTP, repeated computation) aur cold-source ki inefficiency. shareReplay = "ek baar fetch, sabko share + cache". Classic HTTP caching pattern.

4. Internal Working (Engine ke andar kya hota hai?)

S-09

Error Handling

Covers: catchError · throwError · retry({count,delay}) · EMPTY · of(fallback) · error is terminal · inner catch in flattening · finalize

1. What is it? (Kya hai ye?)

RxJS error handling = jab stream error kare (HTTP fail, exception) toh use gracefully manage karna — recover (fallback value), retry, ya transform-and-rethrow. Main tools: catchError, retry, throwError, EMPTY.

2. Why do we need it? (Zaroorat kyun hai?)

Error ek Observable mein terminal hai — bina handle kiye stream mar jaata hai aur (khaas UI streams jaise valueChanges-driven pipes mein) dobara kaam nahi karta. Proper handling stream ko alive rakhta ya gracefully recover karta.

3. What problem does it solve?

Silent stream death

Agar ek search pipe (valueChanges → switchMap(http)) mein HTTP error aaya aur handle nahi kiya, toh poora stream complete/dead ho jaata — aage typing pe kuch nahi hota. User ko lagता hai search toot gaya. Yeh sabse common RxJS bug hai.

4. Internal Working (Engine ke andar kya hota hai?)

catchError returns a NEW Observable

catchError error ko pakadta hai aur ek naya Observable return karta hai jo error wale ki jagah continue karta. of(fallback) → recover (value emit + complete). throwError(() => e) → re-propagate. EMPTY → chup-chaap complete. Critical: outer stream ko alive rakhna ho toh error ko inner (switchMap ke andar) catch karo — bahar catch karne se outer already dead ho chuka hota.

Diagram — where to catch (inner vs outer)
 search$ .pipe(
   switchMap(q => http.get(q).pipe(catchError(() => of([]))))  ✓ inner: outer survives
 )
 vs
 search$ .pipe(
   switchMap(q => http.get(q)),
   catchError(() => of([]))     ✗ outer: after 1 error, search$ is DEAD
 )

5. Execution Flow

 http error ─▶ catchError(err => ...)
   ├─ of(fallback)      ─▶ emit fallback, complete (recovered)
   ├─ throwError(()=>e) ─▶ error propagates to subscriber's error cb
   └─ EMPTY             ─▶ complete silently (no value)
 retry({count,delay}) ─▶ re-subscribe (new request) N times before failing

6. Architecture — error tools

catchError(fn)Catch + return replacement Observable. Recover or rethrow.
retry({count, delay})Re-subscribe on error (S-06 of Ch12). Transient failures.
throwError(() => e)Create an error-Observable (factory form, RxJS 7+).
EMPTY / of(x)Complete silently / emit fallback then complete.
finalize(fn)Runs on complete OR error (cleanup, hide spinner).

S-10

Memory Leaks & Unsubscribe

Covers: leak sources · AsyncPipe · takeUntilDestroyed · takeUntil+Subject · finite auto-complete · toSignal · what needs cleanup

1. What is it? (Kya hai ye?)

Memory leak tab hota hai jab ek subscription component destroy hone ke baad bhi alive rehti hai — callback chalti rehti, memory + CPU khaati, purani component instance ko GC hone se rokti. Unsubscribe patterns ise prevent karte hain.

2. Why do we need it? (Zaroorat kyun hai?)

SPA mein components baar-baar mount/unmount hote hain (routing). Har leaked subscription accumulate hoti — 100 navigations = 100 zombie subscriptions running. Result: slow app, growing memory, ghost callbacks (destroyed component pe update → errors).

3. What problem does it solve?

Zombie subscriptions ki accumulation. Proper cleanup se destroyed components ki subscriptions band ho jaati, memory free hoti, aur "cannot update destroyed component" errors nahi aate.

4. Internal Working — kya leak karta hai?

Finite vs infinite

Finite streams (HTTP, of, timer(x)) complete karte hain → complete pe auto-unsubscribe → usually no leak. Infinite/long-lived streams (interval, fromEvent, Subject-based, form valueChanges, router events) khud complete nahi hote → manual/declarative cleanup zaroori, warna leak. Rule: agar stream khud khatam nahi hota, tumhe khatam karna padega.

Diagram — what leaks
 HTTP / of / timer   ─▶ completes ─▶ auto-cleanup ─▶ ✓ usually safe
 interval / fromEvent ─▶ never completes ─▶ MUST clean up:
   ├─ AsyncPipe (template destroy)
   ├─ takeUntilDestroyed()  (component destroy)
   └─ takeUntil(destroy$)   (manual signal)

5. Execution Flow

 subscribe ─▶ stream alive ─▶ component destroyed
   without cleanup ─▶ subscription STILL alive ─▶ leak (zombie callback)
   with takeUntilDestroyed/AsyncPipe ─▶ auto-unsubscribe ─▶ teardown ─▶ ✓ freed

6. Architecture — cleanup patterns (best → worst)

AsyncPipe (| async)Best — template subscribe/unsubscribe automatic. No TS code.
takeUntilDestroyed()v16+ — auto-unsubscribe on destroy. Manual subscribe ka cleanest.
toSignal(obs$)Observable → signal; cleanup automatic. Modern.
takeUntil(destroy$)Classic — Subject in ngOnDestroy. Verbose but works.
Manual ngOnDestroyStore + unsubscribe(). Error-prone (bhoolna easy).

SCENARIO Live dashboard har 5s update karna hai, par ek fail hone pe poora polling ruk jaata hai. Fix?

Fetch error outer stream ko kill kar raha. Inner catchError lagao: switchMap(() => api.get().pipe(catchError(() => of(null)))) — ab ek fetch fail ho toh us tick pe fallback (null) aayega par polling (timer) chalti rahegi. Cleanup takeUntilDestroyed.

HARD Dependent dropdowns (country→state→city) 3 levels — kaise banaoge?

Chain switchMaps: country.valueChanges.pipe(switchMap(c => loadStates(c))) → states dropdown. State change pe city: state.valueChanges.pipe(switchMap(s => loadCities(s))). Har level pe parent change purani child-request cancel karta (switchMap). Reset child selections on parent change. Inner catchError for resilience.
