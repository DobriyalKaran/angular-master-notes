S-01

signal() — Writable State

Covers: writable signal · read () · set · update · fine-grained reactivity · equality function · asReadonly · signal vs BehaviorSubject · zoneless

1. What is it? (Kya hai ye?)

Signal ek reactive value container hai — ek box jismein value rehti hai, aur jab value badalti hai toh woh sab jagah jo us value ko "padh" rahe the automatically notify ho jaate. signal(0) ek writable signal banata; count() se padho, count.set(5) se likho.

2. Why do we need it? (Zaroorat kyun hai?)

Purane Angular mein change detection Zone.js pe depend karta tha — koi bhi async event (click, timer, HTTP) poore component tree ko re-check karwata, bhale kuch badla ho ya nahi. Ye inefficient. Signals fine-grained reactivity dete hain: Angular ko exactly pata hota hai kaunsa signal kahan use hua, toh sirf wahi update hota. Yehi zoneless Angular ka foundation hai.

3. What problem does it solve?

Purane approaches ka dard
  • Zone.js — har event pe poora tree check; over-rendering.
  • BehaviorSubject — kaam karta par verbose: .value, | async, subscribe/unsubscribe, leak-management.
  • Template mein | async ka boilerplate, aur derived state ke liye RxJS operators.
Signal

Synchronous read (count()), auto dependency-tracking, no subscribe/unsubscribe, no | async, fine-grained updates. Template mein seedhe {{ count() }}.

4. Internal Working (Engine ke andar kya hota hai?)

Producer / Consumer dependency graph

Har signal ek producer hai. Jab tum ek reactive context (template, computed, effect) ke andar count() call karte ho, Angular us context ko signal ka consumer register kar deta hai — ek dependency edge banti hai. Jab count.set() se value badalti hai, signal apne saare consumers ko "dirty" mark karta hai, aur woh re-evaluate hote hain. Ye graph automatic + dynamic hai — tumhe dependencies declare nahi karni. Signal by default referential equality (===) se check karta: agar nayi value purani ke barabar hai, consumers notify nahi hote (no wasted work).

Diagram — dependency graph
 count = signal(0)
     │  (read in these consumers)
     ├──▶ template {{ count() }}      ─ re-renders on change
     ├──▶ computed(() => count()*2)   ─ recomputes on change
     └──▶ effect(() => log(count()))  ─ re-runs on change

 count.set(1) ─▶ mark consumers dirty ─▶ only THESE update (not whole tree)

5. Execution Flow

 const count = signal(0);
 count();            ─▶ read → 0 (registers dependency if in reactive context)
 count.set(5);       ─▶ write → notify consumers (if 5 !== old)
 count.update(n => n+1); ─▶ read+write in one → 6
 count.asReadonly(); ─▶ read-only view (no set/update) for exposing

6. Architecture — the API

signal(initial)Create writable signal with initial value.
sig()Read (call it). In reactive context → registers dependency.
sig.set(value)Replace value.
sig.update(fn)Derive new from old: sig.update(c => c + 1).
sig.asReadonly()Read-only view — expose without letting outsiders write.
signal(v, { equal })Custom equality (e.g. deep) to control change detection.

7. Real Project Example (Angular context)

counter.component.ts
import { Component, signal } from '@angular/core';

@Component({
  selector: 'app-counter',
  template: `
    <p>Count: {{ count() }}</p>
    <button (click)="increment()">+</button>
  `,
})
export class Counter {
  count = signal(0);                          // writable signal
  increment() { this.count.update(c => c + 1); }  // no subscribe, no | async
}
Note

Template mein count() — call karna zaroori (signal ek function hai). {{ count }} (bina call) galat — woh function print karega.

9. Advanced Example — objects & immutability, custom equality

typescript
const user = signal({ name: 'Karan', age: 25 });

// ❌ mutate → same reference → NO update (referential equality)
user().age = 26;

// ✅ new reference → update fires
user.update(u => ({ ...u, age: 26 }));

// custom equality (deep) — control when consumers notify
const list = signal([1, 2, 3], { equal: (a, b) => a.length === b.length });

10. Code Explanation — Line by Line

  1. signal(0) — writable signal, initial 0.
  2. count() — read; reactive context mein dependency register.
  3. set(v) — full replace; update(fn) — old se new derive.
  4. Objects: immutable update ({...u, ...}) zaroori — mutation same ref = no notify.
  5. { equal } — custom equality se change-detection control.

11. Common Mistakes

Traps
  • Template mein {{ count }} (bina ()) — signal call karna bhoolna.
  • Object/array signal ko mutate karna (arr().push()) → same reference → update nahi hota. Immutable update karo.
  • set vs update confuse — new value = set; old se derive = update.
  • Signal ko RxJS ki tarah subscribe karne ki koshish — signal sync hai, seedhe () se padho.
  • Public writable signal expose karke bahar se uncontrolled writes — asReadonly().

12. Best Practices

Do this
  • Objects/arrays hamesha immutably update (update(x => ({...x}))).
  • State ko service mein rakho, asReadonly() expose, mutation methods se controlled.
  • Template mein seedhe signal() call — no | async needed.
  • Prefer signals over BehaviorSubject for synchronous local/UI state.

13. Performance

Signals ka poora point perf hai — fine-grained updates matlab sirf affected DOM/computeds update, poora tree nahi (Zone.js ke opposite). Referential equality wasted updates rokti (same value set = no-op). Zoneless Angular signals pe hi khada hai. Bade objects pe custom equal se unnecessary notifications control karo.

14. Edge Cases

Same value setset with equal value → consumers notify nahi (referential equality).
Reading outside reactive contextSeedhe sig() value deta, par dependency register nahi hoti (e.g. event handler mein).
Signal in constructorField initializer / constructor injection context mein banao.

15. Interview Questions

EASY Signal kya hai aur kaise read/write karte ho?

Reactive value container. signal(0) banata; count() (call karke) padho; count.set(5) replace; count.update(c => c+1) old se derive. Reactive context (template/computed/effect) mein read karne pe dependency auto-register hoti hai — value badle toh sirf woh consumers update.

EASY Signal vs BehaviorSubject?

Dono "current value + notify on change". Signal: synchronous read (()), auto dependency-tracking, no subscribe/unsubscribe (no leaks), template mein no | async, fine-grained CD. BehaviorSubject: async (subscribe), .value for sync, manual cleanup, | async in template. Synchronous UI state → signal; async streams → RxJS.

MEDIUM Object signal update kiya par UI refresh nahi hua. Kyun?

Signal referential equality (===) se check karta hai. Object ko mutate kiya (user().age = 26) toh reference same rehti — signal ko lagता value nahi badli, consumers notify nahi hote. Fix: immutable update — user.update(u => ({...u, age: 26})) — nayi reference se change fire hoga.

SCENARIO Ek state service bana rahe ho, chahte ho components padh sakein par directly write na kar sakein. Kaise?

Private writable signal + public asReadonly(): private _count = signal(0); readonly count = this._count.asReadonly();. Components store.count() padh sakte, par .set() nahi (readonly signal pe set nahi hota). Mutation service ke methods (jo _count.set() karte) se controlled. BehaviorSubject ke asObservable() jaisa pattern.

16. Revision Notes

⚡ 30-second recall
  • signal(init) → read sig(), write set()/update(). Function — call it (even in template).
  • Referential equality — objects/arrays IMMUTABLE update, warna no notify. Custom {equal} to tune.
  • Fine-grained reactivity (foundation of zoneless). Expose via asReadonly(). No subscribe/leak.

S-02

computed() — Derived State

Covers: derived signal · memoization · lazy evaluation · glitch-free · auto dependency tracking · pure computation · read-only · vs method

1. What is it? (Kya hai ye?)

computed() ek derived, read-only signal banata hai — ek value jo doosre signals se calculate hoti hai. const total = computed(() => price() * qty()). Jab price ya qty badle, total automatically recompute hota. Ye memoized aur lazy hai.

2. Why do we need it? (Zaroorat kyun hai?)

Derived values (subtotal, filtered list, full name) ko manually sync karna error-prone hai — kaunse changes pe kya recompute? computed ye automatic karta: dependencies track karke, sirf zaroorat pe recompute, aur result cache karke. Template mein calculation ki jagah clean derived signal.

3. What problem does it solve?

Manual state synchronization aur redundant computation. Bina computed ke: ya toh template mein bhaari expression (har CD pe re-run), ya manual "jab X badle toh Y update karo" (bug-prone). computed dono solve karta — declarative + memoized.

4. Internal Working (Engine ke andar kya hota hai?)

Lazy + memoized + glitch-free

• Lazy — computed ki body tab tak nahi chalti jab tak koi use padhe na (aur value stale ho). Agar koi nahi padh raha, computation skip.
• Memoized — ek baar compute hone ke baad value cache; agli read pe dobara compute nahi (jab tak dependency na badle).
• Auto-tracked — body ke andar jo signals padhे jaate hain, wahi dependencies ban jaate (dynamic).
• Glitch-free — ek synchronous batch mein multiple dependency changes ho toh computed sirf ek baar final value pe settle hota — intermediate inconsistent states expose nahi karta.

Diagram — lazy + memoized
 price=signal(10)  qty=signal(2)
 total = computed(() => price() * qty())

 read total() ─▶ compute (10*2=20), CACHE 20
 read total() ─▶ return cached 20 (no recompute)
 qty.set(3)   ─▶ mark total STALE (but don't compute yet — lazy)
 read total() ─▶ recompute (10*3=30), cache 30

5. Execution Flow

 dependency (price/qty) changes ─▶ computed marked STALE (not recomputed)
        ─▶ someone reads total() ─▶ recompute ─▶ cache ─▶ return
        (if nobody reads → never recomputes — lazy)

6. Architecture

computed(fn)Derived read-only signal. Body reads other signals.
Read-onlyNo set/update — value purely derived (writable version → linkedSignal, S-04).
Auto-tracked depsSignals read inside fn become dependencies (dynamic each run).
Chainablecomputed can read other computeds — graph builds.

7. Real Project Example (Angular context)

cart computeds
import { signal, computed } from '@angular/core';

export class CartComponent {
  items = signal<CartItem[]>([]);

  subtotal = computed(() => this.items().reduce((s, i) => s + i.price * i.qty, 0));
  tax      = computed(() => this.subtotal() * 0.18);         // chains on subtotal
  total    = computed(() => this.subtotal() + this.tax());   // chains on both

  // template: {{ total() | currency:'INR' }} — auto-updates when items change
}

9. Advanced Example — filtered/derived list

typescript
const todos = signal<Todo[]>([...]);
const filter = signal<'all' | 'active' | 'done'>('all');

const visibleTodos = computed(() => {
  const f = filter();
  const list = todos();
  return f === 'all' ? list
       : f === 'active' ? list.filter(t => !t.done)
       : list.filter(t => t.done);
});
const activeCount = computed(() => todos().filter(t => !t.done).length);

10. Code Explanation — Line by Line

  1. computed(() => ...) — body doosre signals padhta; wahi dependencies.
  2. subtotal pe tax/total chain — computed graph.
  3. Read-only — sirf derive, koi set nahi.
  4. Lazy: agar template total() na padhe, recompute nahi.
  5. Memoized: dependency unchanged → cached value return.

11. Common Mistakes

Traps
  • Computed body mein side-effects (state mutate, HTTP, console spam) — computed pure hona chahiye. Side-effects → effect (S-03).
  • Computed mein async (Promise/HTTP) — computed synchronous hai; async ke liye resource/RxJS.
  • Manually recompute karne ki koshish — computed khud manage karta.
  • Computed ko writable expect karna — read-only; writable+derived chahiye toh linkedSignal (S-04).
  • Non-signal values (plain props) pe depend karke expect karna ki auto-update ho — sirf signals track hote.

12. Best Practices

Do this
  • Derived state ke liye HAMESHA computed (na ki manual sync ya template calculation).
  • Computed pure rakho — no side-effects, no async.
  • Chain computeds for clean layered derivation (subtotal → tax → total).
  • Template mein bhaari expression ki jagah computed — memoized, ek hi baar calc.

13. Performance

Computed memoization ka bada perf win — ek expensive calc (filter/sort of large list) sirf tab re-run jab dependency badle, aur agar multiple jagah padha jaaye toh ek hi baar compute (cached). Template mein direct method call ({{ compute() }}) har CD pe re-run hota; computed nahi. Lazy nature: unused computeds kabhi compute nahi hote.

14. Edge Cases

Conditional dependenciesBody mein if se kabhi ek signal read, kabhi doosra — deps dynamically track hote (jo us run mein padha).
Diamond dependencyA→B, A→C, B&C→D: glitch-free ensures D ek baar consistent value pe settle.
Equal resultComputed ka naya result purane === ho toh downstream consumers notify nahi (custom equal bhi).

15. Interview Questions

EASY computed() kya hai?

Ek derived, read-only signal jo doosre signals se value calculate karta hai. Dependencies auto-track hoti hain — jo signal body mein padha jaaye. Dependency badle toh computed recompute (lazy + memoized). Jaise total = computed(() => price() * qty()).

MEDIUM Computed "lazy + memoized" ka kya matlab?

Lazy: body tab tak nahi chalti jab tak koi value padhe na (koi consumer nahi → compute nahi). Memoized: ek baar compute hone par value cache hoti; agli read pe dobara compute nahi jab tak dependency na badle. Isse redundant calculation avoid hoti — bada perf win for expensive derivations.

MEDIUM Template mein {{ getTotal() }} (method) vs {{ total() }} (computed) — farak?

Method har change-detection cycle pe re-run hota (Angular ko nahi pata kab result badlega). Computed memoized hai — sirf tab recompute jab uske signal dependencies badlein, warna cached value. Bade/expensive calc ke liye computed drastically kam runs — better perf.

SCENARIO Computed ke andar tumne ek API call daal di aur ajeeb behavior aa raha. Kyun galat?

Computed pure + synchronous hona chahiye — sirf derive, koi side-effect/async nahi. API call (side-effect + async) computed mein daalna: har recompute pe fire ho sakta, race conditions, aur computed sync value expect karta. Async data ke liye resource()/rxResource() ya RxJS + toSignal use karo; side-effects effect() mein.

16. Revision Notes

⚡ 30-second recall
  • computed(() => ...) = derived read-only signal. Auto-tracks signals read inside.
  • LAZY (no read → no compute) + MEMOIZED (cache until deps change) + glitch-free.
  • PURE — no side-effects/async. Chain freely. Beats template method calls (memoized).

S-03

effect() — Side Effects

Covers: side-effects · auto-tracking · timing (CD cycle) · cleanup (onCleanup) · untracked · allowSignalWrites removed · injection context · when NOT to use

1. What is it? (Kya hai ye?)

effect() ek function register karta hai jo tab-tab chalta hai jab uske andar padhe gaye koi bhi signal badalte hain. Ye side-effects ke liye hai — logging, DOM manipulation, localStorage sync, 3rd-party libs — yani reactive value change pe "kuch karna" (value derive karna nahi, woh computed ka kaam).

2. Why do we need it? (Zaroorat kyun hai?)

Kabhi signal change pe world ke saath interact karna padta — console log, localStorage.setItem, chart library ko update, analytics event. Ye "pure derivation" nahi (computed ka domain), balki external side-effect hai. effect ye reactively karta, dependencies auto-track karke.

3. What problem does it solve?

Reactive side-effects ko cleanly express karna — bina manual "jab X change ho toh ye karo" wiring ke. Par ye escape hatch hai: zyada tar reactivity computed/template se honi chahiye; effect sirf jab bahar duniya se baat karni ho.

4. Internal Working (Engine ke andar kya hota hai?)

Timing + tracking + cleanup

• Auto-track — effect ki body mein padhe signals uski dependencies; koi badle toh effect re-run.
• Timing (v19+) — effects ab change detection cycle ka part hain (pehle microtask queue the). Parent effects child se pehle chalte. Pehla run effect register hone ke baad first CD pe.
• Cleanup — effect fn ko onCleanup milta; next run se pehle (aur destroy pe) previous cleanup chalta — timers/listeners clear karne ko.
• Auto-destroy — injection context mein banao toh component destroy pe auto-cleanup (DestroyRef).
• allowSignalWrites removed (v19) — ab effect ke andar signal write default allowed (pehle flag chahiye tha), par soch-samajhkar (infinite loop se bacho).

Diagram — effect lifecycle
 effect(() => {
   const v = count();          ← tracked dependency
   localStorage.setItem('c', v);  ← side effect
   onCleanup(() => {...});      ← runs before next run & on destroy
 });
 count change ─▶ cleanup(prev) ─▶ re-run effect ─▶ new side effect
 component destroy ─▶ final cleanup ─▶ effect stopped (auto)

5. Execution Flow

 register effect ─▶ first CD ─▶ run (track deps) ─▶ side effect
   dep changes ─▶ (next CD) ─▶ onCleanup(prev) ─▶ re-run ─▶ side effect
   destroy ─▶ onCleanup(final) ─▶ stopped

6. Architecture

effect(fn)Register reactive side-effect. Injection context (field/constructor).
onCleanup(fn)Inside effect — cleanup before next run & on destroy.
untracked(fn)Read a signal WITHOUT making it a dependency.
effect(fn, { injector })Outside injection context — pass injector.
{ manualCleanup }Opt out of auto-destroy (rare; then destroy manually).

7. Real Project Example (Angular context)

persist to localStorage
import { Component, signal, effect } from '@angular/core';

export class SettingsComponent {
  theme = signal<'light' | 'dark'>('dark');

  constructor() {
    // re-runs whenever theme() changes; auto-cleaned on destroy
    effect(() => {
      localStorage.setItem('theme', this.theme());
      document.body.className = this.theme();
    });
  }
}

10. Code Explanation — Line by Line

  1. effect(() => ...) — body ke signals track; change pe re-run.
  2. onCleanup(fn) — next run se pehle + destroy pe cleanup (timers/listeners).
  3. untracked(() => sig()) — value padho par dependency mat banao (unnecessary re-runs avoid).
  4. Constructor/field mein banao → injection context → auto-destroy.

11. Common Mistakes

Traps
  • effect ko state-sync ke liye use karna (jab computed/linkedSignal behtar) — #1 anti-pattern. Effect last resort.
  • Effect mein signal write jo apni hi dependency badle → infinite loop. (allowSignalWrites hata, par responsibility tumhaari.)
  • Cleanup na dena jab timer/listener banaya → leak.
  • Async data fetch effect mein — race conditions; resource()/RxJS use karo.
  • Injection context ke bahar bina injector effect banana → error.
  • Har signal read ko track hone dena jab kuch ko untracked hona chahiye → over-firing.

12. Best Practices

Do this
  • Effect sirf external side-effects ke liye (log, DOM, storage, 3rd-party). Derivation → computed. State reset → linkedSignal.
  • Timers/listeners banao toh onCleanup hamesha.
  • Jo signals trigger nahi karne chahiye unhe untracked se padho.
  • Signal writes effect mein soch-samajhkar (loop se bacho); prefer derived state.

13. Performance

Effects CD cycle ka part hain — bhaari sync work har run pe CD slow karega. Over-tracking (unnecessary deps) se over-firing — untracked se scope karo. Effect abuse (state sync) unnecessary re-runs + hard-to-debug cascades deta; computed/linkedSignal zyada efficient + predictable. Cleanup se resources leak nahi karte.

14. Edge Cases

Signal write in effectAllowed (v19+) par apni dependency likhna → infinite loop. Careful / prefer derived.
View queries in effectTiming change (v19): effects CD ke dauraan; view-query-dependent effects ka behavior verify.
afterRenderEffectDOM-measurement side-effects ke liye alag primitive (post-render).
manualCleanupAuto-destroy opt-out — phir EffectRef.destroy() khud karo.

15. Interview Questions

EASY effect() kab use karte ho, computed ki jagah kyun nahi?

Effect side-effects ke liye — logging, localStorage, DOM, 3rd-party libs — jab signal change pe bahar duniya se interact karna ho. Computed derivation ke liye (naya value calculate). Rule: naya value chahiye → computed; kuch "karna" (side-effect) chahiye → effect. Effect last resort hai.

MEDIUM Effect mein cleanup kaise aur kyun?

Effect fn ko onCleanup callback milta hai — ye next re-run se pehle aur component destroy pe chalta. Timers (setInterval), event listeners, subscriptions jaise resources clear karne ko: effect((onCleanup) => { const id = setInterval(...); onCleanup(() => clearInterval(id)); }). Bina cleanup → leak.

MEDIUM untracked() kya karta hai?

Ek signal ko reactive context (effect/computed) ke andar padhne deta hai bina use dependency banaye — yani us signal ke change pe effect re-run nahi hoga. Use case: effect ko ek signal (jaise theme) pe react karana hai par doosre (jaise userId) ki latest value chahiye par uspe trigger nahi karna. untracked(() => userId()).

SCENARIO Selected item ko options change hone pe reset karna hai. Junior ne effect use kiya. Better?

Effect se state-sync anti-pattern hai — options aur selected ko independent treat karta, cascades + potential loops. Better: linkedSignal (S-04) — ek writable signal jo options (source) pe auto-reset ho par user manually bhi .set() kar sake. Ye exactly is "dependent selection" case ke liye banaya gaya.

16. Revision Notes

⚡ 30-second recall
  • effect(fn) = reactive SIDE-effects (log/DOM/storage/3rd-party). NOT for derivation (computed) or state-sync (linkedSignal).
  • Auto-tracks deps; runs in CD cycle (v19+); onCleanup for timers/listeners; auto-destroy in injection context.
  • allowSignalWrites removed — writes allowed but beware infinite loops. untracked() to read without depending.

S-04

linkedSignal() — Writable + Reset

Covers: writable derived signal · source + computation · auto-reset on source change · shorthand vs full form · previous value · dependent selection · vs computed vs effect

1. What is it? (Kya hai ye?)

linkedSignal() ek writable signal jo ek source se linked hai. Ye computed aur signal ke beech ka gap bharta: computed ki tarah source change pe auto-reset hota hai, par plain signal ki tarah tum .set() bhi kar sakte ho. Perfect for "dependent selection" — jaise options list badle toh selected item reset ho, par user manually bhi select kar sake.

2. Why do we need it? (Zaroorat kyun hai?)

Ek common problem: state jo usually ek source se derive hoti hai par user override bhi kar sakta. computed read-only hai (override nahi). Plain signal auto-reset nahi karta (source badle toh stale selection reh jaati). Pehle log effect+signal se ye hack karte the (anti-pattern). linkedSignal ise cleanly solve karta.

3. What problem does it solve?

The gap
  • computed → auto-reset ✓ but read-only ✗ (user override nahi).
  • signal → writable ✓ but no auto-reset ✗ (stale on source change).
  • Old hack: effect(() => selected.set(default)) — messy, loop-prone.
linkedSignal

Writable ✓ AND auto-reset on source change ✓. Dono ek primitive mein.

4. Internal Working (Engine ke andar kya hota hai?)

Computed value + writable override

linkedSignal ek source signal (ya signals) track karta hai aur ek computation se apna value derive karta. Jab source badalta hai, computation dobara chalti aur signal ka value reset ho jaata (override discard). Par jab tak source same hai, tum .set()/.update() se value override kar sakte ho — woh override hold hota. Full form mein computation ko previous value bhi milti hai (source aur pichhla state) taaki smart reset logic likh sako (jaise "agar purana selection abhi bhi list mein hai toh use rakho").

Diagram — linkedSignal behavior
 options = signal(['Std','Express','Overnight'])
 selected = linkedSignal(() => options()[0])   // reset to first

 selected()            ─▶ 'Std'   (derived from source)
 selected.set('Express') ─▶ 'Express'  (manual override holds)
 options.set(['Drone','Teleport']) ─▶ RESET ─▶ 'Drone' (source changed)

5. Execution Flow

 source unchanged ─▶ manual .set()/.update() ─▶ value holds (override)
 source CHANGES   ─▶ computation re-runs ─▶ value RESETS (override discarded)
 (full form: computation(source, previous) → smart reset)

6. Architecture — two forms

ShorthandlinkedSignal(() => source()[0]) — reset to computed value on source change.
Full formlinkedSignal({ source, computation: (src, prev) => ... }) — access previous value/source for smart reset.
WritableHas set/update (unlike computed).
StatusExperimental v19 → stabilized ~v20. Import from @angular/core.

7. Real Project Example (Angular context)

dependent selection — keep valid selection
import { signal, linkedSignal } from '@angular/core';

export class ProductPicker {
  variants = signal<Variant[]>([...]);      // source (changes when product changes)

  // full form: keep previous selection IF still valid, else reset to first
  selected = linkedSignal<Variant[], Variant>({
    source: this.variants,
    computation: (variants, previous) => {
      const stillThere = previous && variants.find(v => v.id === previous.value.id);
      return stillThere ?? variants[0];      // smart reset
    },
  });

  choose(v: Variant) { this.selected.set(v); }   // user override (writable!)
}

S-05

Signal Inputs

Covers: input() · input.required() · default value · alias · transform · model() two-way · output() · vs @Input decorator · reactive inputs

1. What is it? (Kya hai ye?)

Signal inputs parent-to-child data ko signals ke roop mein receive karte hain — @Input() decorator ka signal-based replacement. name = input<string>() ek read-only signal deta hai jo parent ke binding se update hota. Ab input reactive hai: use computed/effect mein directly use kar sakte ho.

2. Why do we need it? (Zaroorat kyun hai?)

Purana @Input() ek plain property tha — uspe react karne ke liye ngOnChanges (clunky) chahiye tha, aur computed/effect mein use nahi hota tha. Signal inputs input ko reactive banate: value change automatically downstream computeds/effects/template ko flow karti — ngOnChanges ki zaroorat khatam.

3. What problem does it solve?

Inputs ko baaki signal reactivity ke saath integrate karna. ngOnChanges boilerplate hatana. Required inputs ko type-safe banana (input.required — undefined nahi). Transform/alias declaratively.

4. Internal Working (Engine ke andar kya hota hai?)

Read-only signal fed by binding

input() ek read-only signal banata hai — child use likh nahi sakta (parent owns it). Jab parent binding ki value badalti hai, Angular signal ko update kar deta, aur uske consumers (computed/effect/template) react karte. input.required() koi default nahi leta aur type non-undefined hoti (compiler enforce karta ki parent bind kare). model() ek writable signal deta jo two-way binding ([(value)]) support karta — child bhi likh sakta aur parent ko propagate hota.

Diagram — input vs model
 PARENT [name]="user"  ─▶ child: name = input<string>()   (read-only signal)
                                    child reads name(), cannot set

 PARENT [(value)]="x"  ─▶ child: value = model<string>()  (writable, two-way)
                                    child value.set(...) ─▶ propagates to parent

5. Execution Flow

 parent binding changes ─▶ input signal updates ─▶ computed/effect/template react
 model: child value.set() ─▶ emits to parent ─▶ parent state updates (two-way)

6. Architecture — the input family

input<T>()Optional input, T | undefined. Read-only signal.
input(default)With default value → type T (never undefined).
input.required<T>()Required — parent MUST bind; type T.
input(d, { alias, transform })Alias (external name) + transform (e.g. booleanAttribute, numberAttribute).
model<T>()Writable, two-way ([(x)]). Replaces @Input+@Output pair.
output<T>()Signal-era event emitter (replaces @Output EventEmitter).

}

Covers: viewChild · viewChildren · contentChild · contentChildren · required · read option · vs @ViewChild decorator · timing · reactive queries

1. What is it? (Kya hai ye?)

Signal queries child elements/components/directives ko signals ke roop mein grab karte hain — @ViewChild/@ContentChild decorators ka signal replacement. chart = viewChild<ChartComponent>('chart') ek signal deta jo template ke us element pe point karta.

2. Why do we need it? (Zaroorat kyun hai?)

Signal Store (NgRx)

Covers: signalStore · withState · withComputed · withMethods · withHooks · patchState · signalStoreFeature · rxMethod · signalState · providedIn

1. What is it? (Kya hai ye?)

NgRx SignalStore (@ngrx/signals) signals ke upar ek lightweight, extensible state-management layer hai. signalStore(...) ek injectable service banata jismein state (signals), computed (derived signals), aur methods (actions) ek jagah — builder-style with* features se. Redux-style boilerplate ke bina structured state.

2. Why do we need it? (Zaroorat kyun hai?)

Plain signals chhote/local state ke liye great, par bade shared state (multiple components, business logic, async) ke liye structure chahiye — warna scattered signals + services ka mess. Classic NgRx (actions/reducers/effects/selectors) powerful par heavy boilerplate. SignalStore beech ka sweet spot: structured, extensible, kam boilerplate, signals-native.

3. What problem does it solve?

Shared/feature-level state ka organization — state + derived + actions co-located, reusable features, less boilerplate than Redux-style. Signal-native (sab kuch signals, template mein direct read, fine-grained CD).

4. Internal Working (Engine ke andar kya hota hai?)

Builder pattern — order matters

signalStore ek sequence of features (withState, withComputed, withMethods, withHooks, withProps) leta hai, RxJS pipe ki tarah — har feature store ko incrementally build karta aur pichle sabko access kar sakta (isliye order matters). withState state ko signals mein badalta; withComputed derived signals add karta; withMethods actions (jo patchState se state update karte). State updates patchState(store, partial | updaterFn) se — immutable. Same-key member override pe dev-mode warning.

Diagram — store composition

   { providedIn: 'root' },
   withState({ count: 0 }),          ─▶ count: Signal<number>
   withComputed(s => ({              ─▶ double: Signal (derived)
     double: computed(() => s.count()*2) })),
   withMethods(s => ({               ─▶ actions
     inc: () => patchState(s, { count: s.count()+1 }) })),
   withHooks({ onInit, onDestroy }),  ─▶ lifecycle
 )   // ↑ each feature sees everything defined ABOVE it

5. Execution Flow

 inject(Store) ─▶ read store.count() / store.double() (signals)
        ─▶ call store.inc() ─▶ patchState (immutable) ─▶ signals update
        ─▶ template + computeds react (fine-grained)

6. Architecture — the features

signalStore(...)Creates injectable store. { providedIn:'root' } = singleton.
withState(initial)State slices → signals.
withComputed(s => ({...}))Derived signals (private with _ prefix).
withMethods(s => ({...}))Actions; inject() services; patchState.
withHooks({onInit,onDestroy})Lifecycle (e.g. initial fetch).

7. Real Project Example (Angular context)

books.store.ts
import { computed, inject } from '@angular/core';
import { signalStore, withState, withComputed, withMethods, withHooks, patchState } from '@ngrx/signals';

type BooksState = { books: Book[]; loading: boolean; query: string };

export const BooksStore = signalStore(
  { providedIn: 'root' },
  withState<BooksState>({ books: [], loading: false, query: '' }),

  withComputed(({ books, query }) => ({
    filtered: computed(() => books().filter(b => b.title.includes(query()))),
    count: computed(() => books().length),
  })),

  withMethods((store, api = inject(BooksApi)) => ({
    setQuery(q: string) { patchState(store, { query: q }); },
    async load() {
      patchState(store, { loading: true });
      const books = await api.getAll();
      patchState(store, { books, loading: false });
    },
  })),

  withHooks({ onInit(store) { store.load(); } }),
);
// component: private store = inject(BooksStore);  template: {{ store.filtered() }}

  1. { providedIn: 'root' } — singleton injectable store.
  2. withState<T>(initial) — har key ek signal ban jaati.
  3. withComputed(({books,query}) => ...) — pichhle state signals ko destructure, derived add.
  4. withMethods((store, api=inject(...)) => ...) — actions; service inject; patchState.
  5. patchState(store, { ... }) — immutable partial update.
  6. withHooks.onInit — store init pe initial load.

11. Common Mistakes

Traps
  • State ko directly mutate karna — hamesha patchState (immutable).
  • Feature order galat — withComputed jo baad-wale state pe depend kare toh withState pehle hona chahiye.
  • Same key ko do features mein define karke unintended override (dev warning aati hai — dhyaan do).
  • Ek monolithic giant store — feature-based split karo (signalStoreFeature).
  • Async ke liye patchState pehle loading, phir await, phir result — loading flag bhoolna.
  • RxJS-heavy side-effects ke liye rxMethod na use karke manual subscribe.

12. Best Practices

Do this
  • Shared/feature state → SignalStore; chhota local state → plain signals.
  • Derived data → withComputed (memoized selectors).
  • Reusable logic → signalStoreFeature (withRequestStatus, withLoading).
  • patchState for all updates (immutable). Split stores by feature.
  • RxJS side-effects → rxMethod; entity CRUD → withEntities.

13. Performance

SignalStore signals pe khada — fine-grained CD (sirf affected consumers update). withComputed memoized selectors redundant calc rokte. Feature-split stores lazy-loadable + focused. Classic NgRx ke muqable kam boilerplate + no action/reducer indirection = simpler + often faster dev. patchState immutable so change detection precise.

15. Interview Questions

EASY NgRx SignalStore classic NgRx Store se kaise alag?

SignalStore signals-native + lightweight: state/computed/methods ek builder mein co-located, kam boilerplate, no actions/reducers/effects indirection. Classic Store Redux-style (actions → reducers → selectors → effects) — zyada boilerplate, RxJS-heavy. SignalStore modern default for most apps; classic for very large/strict-unidirectional needs.

MEDIUM withState, withComputed, withMethods ka order kyun matter karta?

Features pipe ki tarah sequentially compose hote — har feature apne se pehle define kiye members ko access kar sakta. withComputed jo state pe depend karta hai, uske pehle withState chahiye; withMethods jo computed use kare, uske pehle withComputed. Galat order → member available nahi.

MEDIUM State kaise update karte ho aur kyun patchState?

patchState(store, { key: value }) ya patchState(store, s => ({ ...s, ... })). Ye immutable update ensure karta — naya state object, purana untouched — jisse signals sahi se change detect karein aur time-travel/predictability bani rahe. Direct mutation signals ko confuse karta (referential equality).

SCENARIO Multiple stores mein same "loading/error status" logic repeat ho raha. DRY kaise?

signalStoreFeature se ek reusable custom feature banao — withRequestStatus() jo withState({status}) + withComputed({isPending, isFulfilled}) + status updaters bundle kare. Phir kisi bhi store mein signalStore(withRequestStatus(), ...) plug karo. Composition se DRY.

S-08

Signals vs RxJS

Covers: sync vs async · when to use which · toSignal · toObservable · interop patterns · pull vs push · not a replacement · hybrid apps

1. What is it? (Kya hai ye?)

Signals aur RxJS dono reactive hain, par alag problems ke liye. Signals = synchronous state (current value, derived state, UI). RxJS = asynchronous streams (events over time, HTTP, WebSocket, complex async coordination). Ye section decide karna sikhata hai kab kaunsa, aur dono ko toSignal/toObservable se bridge karna.

2. Why do we need it? (Zaroorat kyun hai?)

Signals aane ke baad confusion: "kya RxJS dead hai? sab signals se karun?" Nahi. Dono complementary hain. Galat tool = pain (signals se async coordination hard, RxJS se simple sync state overkill). Sahi choice = clean code. Interview mein ye ek favourite discussion hai.

3. What problem does it solve?

"Kaunsa reactive tool kahan" ka decision. Aur hybrid reality: aksar dono chahiye — HTTP (RxJS) ka result signal state mein (toSignal), ya signal change pe debounced stream (toObservable).

4. Internal Working — pull vs push

Core mental model

• Signals = "pull" + synchronous. Value hamesha current — jab chaho sig() se pull karo. Koi "time" dimension nahi; bas latest state. Glitch-free, memoized.
• RxJS = "push" + asynchronous. Stream over time values push karta — 0..many, kab aayenge pata nahi. Rich operators (debounce, retry, switchMap, combineLatest) time/async coordinate karte.
Signals lack: time-based operators (debounce, throttle), multi-value-over-time async, cancellation semantics. RxJS lacks: synchronous "current value" simplicity, template-friendliness (no | async). Isliye complementary.

Diagram — decision
 Synchronous "what is the value NOW?" ─────▶ SIGNAL
   (UI state, derived values, form state, toggles, counters)

 Asynchronous "values OVER TIME" ─────────▶ RxJS
   (HTTP, WebSocket, debounced search, retry, event streams)

 Bridge:  observable$ ──toSignal()──▶ signal     (consume stream as state)
          signal ──toObservable()──▶ observable$  (react with operators)

5. Execution Flow — interop

 HTTP (RxJS) ─▶ toSignal(http$) ─▶ signal in template {{ data() }}
 search signal ─▶ toObservable ─▶ debounceTime+switchMap(http) ─▶ toSignal ─▶ results()

6. Architecture — when to use which

Use SIGNALS forUI/local state, derived state (computed), form values, toggles, counters, template binding, shared feature state (SignalStore).
Use RxJS forHTTP, WebSocket/SSE, debounced/throttled search, retry/cancel, event streams, complex multi-stream async (combineLatest/forkJoin), typeahead.
toSignal(obs$)Observable → signal (auto subscribe/unsubscribe). Consume async as state.
toObservable(sig)Signal → observable. Apply RxJS operators to signal changes.

7. Real Project Example — interop both ways

RxJS search → signal state
import { toSignal, toObservable } from '@angular/core/rxjs-interop';

query = signal('');                         // signal (UI state)

results = toSignal(                          // ← back to signal for template
  toObservable(this.query).pipe(             // ← signal to stream
    debounceTime(300),                       // RxJS: time-based
    distinctUntilChanged(),
    switchMap(q => this.api.search(q)),       // RxJS: cancellation
  ),
  { initialValue: [] as Result[] }
);
// template: {{ results() }} — no | async, no manual subscribe

9. Advanced Example — decision cheat

right tool per case
// ✅ SIGNAL — synchronous UI state
sidebarOpen = signal(false);
fullName = computed(() => `${first()} ${last()}`);

// ✅ RxJS — async coordination (then bridge to signal)

// ❌ don't force async coordination into signals (no debounce/switchMap)
// ❌ don't use RxJS BehaviorSubject for simple local toggles (overkill)

10. Code Explanation — Line by Line

  1. toObservable(signal) — signal changes ko stream banata (RxJS operators lagane ko).
  2. debounceTime/switchMap — RxJS ki taakat (signals mein nahi).
  3. toSignal(obs$, { initialValue }) — stream ko wapas signal (template-friendly, auto-cleanup).
  4. Result: synchronous results() — no | async, no leak.

11. Common Mistakes

Traps
  • "Sab kuch signals" — async coordination (debounce/retry/switchMap) signals mein force karna painful.
  • Simple local toggle ke liye BehaviorSubject + | async — overkill; signal use karo.
  • toSignal bina initialValue jab stream sync value na de — undefined issues.
  • Signal ko RxJS ki tarah subscribe karne ki koshish (signal sync — seedhe call).
  • Computed mein async ghusaana — RxJS + toSignal ya resource() use karo.
  • Interop overuse — har jagah bridge; jahan ek hi tool kaafi wahan simple rakho.

12. Best Practices

Do this
  • Synchronous state/derived/UI → signals. Async streams/coordination → RxJS.
  • Async result ko UI mein → toSignal (template-friendly, auto-cleanup).
  • Signal pe time-based/async logic → toObservable + operators → back to toSignal.
  • Dono ko complementary samjho — "vs" nahi, "and". Right tool per job.

13. Performance

Signals sync state ke liye lighter (no subscription machinery, fine-grained CD, template mein no | async). RxJS async ke liye efficient (cancellation via switchMap, debounce se fewer calls). Galat tool perf/complexity cost deta. toSignal auto subscribe/unsubscribe leak-free. Hybrid apps mein sahi split = clean + fast.

14. Edge Cases

resource()/httpResourceNewer signal-native async — kuch RxJS-for-HTTP cases replace karta.
Injection contexttoSignal/toObservable injection context (ya passed injector) chahiye.

15. Interview Questions

EASY Signals ne RxJS ko replace kar diya?

Nahi — complementary hain. Signals synchronous state (current value, derived, UI) ke liye; RxJS asynchronous streams (HTTP, events over time, debounce/retry/switchMap coordination) ke liye. Signals mein time-based operators aur multi-value-over-time nahi. Dono saath use hote (interop se bridge).

MEDIUM Signals kab, RxJS kab?

Signals: UI/local state, computed derived state, form values, toggles, template binding, shared feature state (SignalStore). RxJS: HTTP, WebSocket, debounced search, retry/cancellation, event streams, multi-stream coordination (combineLatest/forkJoin), typeahead. Sync "value now" → signal; async "over time" → RxJS.

MEDIUM toSignal aur toObservable kya karte hain?

toSignal(obs$) — Observable ko signal mein badalta (auto subscribe + unsubscribe, template-friendly, no | async). toObservable(sig) — signal ko observable banata taaki RxJS operators (debounce/switchMap) laga sako. Interop bridge — aksar signal→toObservable→operators→toSignal pattern.

SCENARIO Search box (signal) chahiye jo debounced HTTP kare aur result template mein dikhe bina | async. Kaise?

Signal → RxJS → signal: query = signal(''); phir results = toSignal(toObservable(query).pipe(debounceTime(300), distinctUntilChanged(), switchMap(q => api.search(q))), { initialValue: [] }). RxJS debounce/cancellation deta (signals mein nahi), toSignal wapas template-friendly signal deta — no | async, no leak.

⚡ 30-second recall

  • Complementary, not "vs". Signals lack time-operators (debounce/switchMap); RxJS lacks sync simplicity.
  • Bridge: toSignal(obs$) (stream→state, no | async), toObservable(sig) (state→operators).
