S-01

NgRx — Overview & Redux Pattern

Covers: the NgRx family · Redux/Flux pattern · unidirectional data flow · single source of truth · when to use / when NOT · pros & cons

1. What is it? (Kya hai ye?)

NgRx Angular ke liye reactive state-management libraries ka ek family hai. Core: @ngrx/store (Redux-style global store), plus @ngrx/effects, @ngrx/entity, @ngrx/signals (SignalStore), @ngrx/component-store. Idea: app ki state ko ek single, predictable, immutable container mein rakho jahan changes ek fixed, unidirectional flow se hote hain.

2. Why do we need it? (Zaroorat kyun hai?)

Chhoti app mein state components/services mein bikhri chal jaati. Par jaise app badhti hai: same data multiple components mein chahiye, prop-drilling (parent→child→grandchild) shuru hoti, "kis component ne state badli" trace karna mushkil, async flows (load/success/error) tangle hote. NgRx state ko centralize karke, changes ko explicit (actions) aur traceable banata.

3. What problem does it solve?

Bina state-management ka dard
  • Prop-drilling — data ko layers ke through pass karna.
  • Scattered state — same data ki multiple copies, sync issues.
  • Untraceable changes — kis ne kab kya badla? Debug hard.
  • Tangled async — loading/error flags har jagah.
NgRx

Single source of truth · unidirectional predictable flow · explicit traceable changes (actions + DevTools time-travel) · centralized async (effects).

4. Internal Working — the unidirectional loop

Redux pattern: one-way data flow

Data ek circle mein flow karta hai, hamesha ek hi direction: Component dispatch karta ek Action → Reducer (pure function) us action + current state se naya state banata → Store update hota → Selectors se derived slices → Component (template) react karta. Side-effects (HTTP) Effects handle karte: action sunte, async karte, naya action dispatch karte. Ye strict one-way flow predictability deta — koi bhi state change ek action se hi hota, isliye traceable + testable + time-travel debuggable.

Diagram — the NgRx loop
   Component ──dispatch(Action)──▶ Reducer ──▶ Store (state)
       ▲                             ▲              │
       │                             │              │ select
       │                          (new action)      ▼
       └───────── Selector ◀──────────────────── derived state
                                     ▲
                Effects ──(HTTP/async)┘  listen to action → dispatch result action

5. Execution Flow

 user clicks ─▶ dispatch(loadUsers) ─▶ Effect: HTTP ─▶ dispatch(loadUsersSuccess({users}))
   ─▶ Reducer: state = {...state, users, loading:false} ─▶ Store updated
   ─▶ Selector(selectUsers) ─▶ template {{ users$ | async }} re-renders

@ngrx/effectsSide-effects (HTTP, async) as RxJS streams reacting to actions.
@ngrx/entityNormalized collections (CRUD adapters) — less boilerplate for lists.
@ngrx/signalsSignalStore — modern, signal-native, less boilerplate. (S-08)
@ngrx/component-storeLocal, component-scoped reactive state. (S-09)

7. Real Project Example — when NgRx fits

Use NgRx when
  • State shared across many components / features.
  • Complex async flows (load/cache/optimistic updates).
  • State must persist beyond a component's lifecycle.
  • Team needs structure, traceability, testability at scale.
DON'T use NgRx when
  • Small app / mostly local state — plain signals ya services kaafi.
  • Simple parent-child data — input()/output().
  • You'd add boilerplate without real shared-state complexity.

8. Beginner Example — the mental model

the 3 questions NgRx answers
// 1. WHAT happened?   → Action    (loadUsers, loadUsersSuccess)
// 2. HOW does state change? → Reducer  (pure: (state, action) => newState)
// 3. WHERE do side-effects go? → Effect  (HTTP, then dispatch result)
// + WHAT do components read? → Selector  (memoized derived state)

9. Advanced Example — classic vs signal-era

two eras of NgRx
// CLASSIC (Redux): @ngrx/store — Actions + Reducers + Effects + Selectors
//   verbose but max structure/traceability; RxJS-based; best for large enterprise.

// SIGNAL-ERA: @ngrx/signals SignalStore — withState/withComputed/withMethods
//   signal-native, far less boilerplate; the modern default for most apps.

// LOCAL: @ngrx/component-store — component-scoped reactive state.

10. Code Explanation — the loop pieces

  1. Action — ek event ("kya hua"): loadUsers.
  2. Reducer — pure fn ("state kaise badle"): (state, action) => newState.
  3. Effect — side-effect ("async kahan"): action sunо → HTTP → naya action.
  4. Selector — derived read ("component kya padhe"): memoized slice.
  5. Sab immutable + unidirectional — isliye predictable + debuggable.

11. Common Mistakes

Traps
  • NgRx ko har chhoti app mein daalna — over-engineering; boilerplate without benefit.
  • Local UI state (dropdown open/close) ko global store mein rakhna — local state local rahe.
  • Store ko directly mutate karna — immutability + unidirectional flow todta.
  • Business logic reducers mein daalna — reducers pure; logic effects mein.
  • Classic Store choose karna jab SignalStore kaafi + simpler hota (modern default).

12. Best Practices

Do this
  • NgRx tabhi jab genuine shared-state complexity ho — warna signals/services.
  • Modern apps: SignalStore default; classic Store for large/strict-unidirectional needs.
  • Unidirectional flow respect karo — state sirf actions/methods se badle.
  • DevTools use karo — time-travel se debugging superpower.

13. Performance

NgRx khud lightweight; perf mostly selectors ki memoization + OnPush change detection se aati (derived state stable references). Galat use (bahut granular actions, non-memoized selectors) overhead deta. SignalStore fine-grained CD (signals) se aur efficient. Immutability precise change detection enable karti.

Server state vs client stateServer cache (TanStack Query) alag concern ho sakta — sab store mein mat thoso.
Feature vs root stateRoot (app-wide) + lazy feature state (route-scoped) — modular.
MigrationClassic → SignalStore incrementally; Events plugin unidirectional flow rakhta.

S-02

Store

Covers: state container · provideStore · provideState · root vs feature state · dispatch · select · state shape · immutability · devtools

1. What is it? (Kya hai ye?)

Store NgRx ka central state container hai — ek single immutable object (state tree) jismein poori app ki (ya feature ki) state rehti. Components isse do tarike se interact karte: dispatch(action) (change trigger) aur select(selector) (state padho, as Observable ya signal).

2. Why do we need it? (Zaroorat kyun hai?)

Ek hi jagah state rakhne se single source of truth milta — koi duplicate/out-of-sync copies nahi. Store observable/signal-based hai, toh state change automatically consumers ko flow karta. Aur ye immutable + centralized hai, toh predictable + debuggable.

3. What problem does it solve?

State ki scattered copies aur manual sync. Store single tree deta jahan se sab derive hota, aur dispatch/select ka clean contract deta — components ko nahi pata state kaise change hoti (bas action dispatch), sirf padhte hain (select).

4. Internal Working (Engine ke andar kya hota hai?)

State tree + observable/signal

Store ek single immutable state object hold karta — top level pe feature-wise slices ({ users: {...}, cart: {...}, ... }). Jab tum dispatch(action) karte ho, store us action ko saare reducers ko bhejta; reducers naya state object banate (purana untouched); store naya state set karta aur consumers (selectors → components) ko notify karta. select ek slice ka Observable (ya selectSignal se signal) deta jo us slice ke badalne pe emit karta. Immutability critical hai — naya reference se change detection precise + time-travel possible.

Diagram — store as single tree
 AppState = {
   users:  { entities: {...}, loading: false },   ← feature slice
   cart:   { items: [...] },                       ← feature slice
   router: {...},                                   ← router-store
 }
 dispatch(action) ─▶ reducers ─▶ NEW AppState ─▶ notify selects
 select(selectUsers) ─▶ Observable<UsersState> (emits on change)

5. Execution Flow

 provideStore() + provideState(feature) ─▶ store built
 component: store.dispatch(loadUsers()) ─▶ reducer/effect ─▶ state updated
 component: users$ = store.select(selectUsers) ─▶ template | async
 (or) users = store.selectSignal(selectUsers)  ─▶ template {{ users() }}

6. Architecture — registration & access

provideStore()Root store (app.config). Replaces StoreModule.forRoot().
provideState(feature)Feature state (lazy/route-scoped). Replaces forFeature().
provideStoreDevtools()Redux DevTools (time-travel).
store.dispatch(action)Trigger a state change.
store.select(selector) / selectSignalRead slice as Observable / signal.

7. Real Project Example (Angular context)

app.config.ts — standalone registration

import { provideEffects } from '@ngrx/effects';
import { provideStoreDevtools } from '@ngrx/store-devtools';

export const appConfig: ApplicationConfig = {
  providers: [
    provideStore(),                       // root store
    provideState(usersFeature),           // feature state (createFeature)
    provideEffects([UsersEffects]),
    provideStoreDevtools({ maxAge: 25, logOnly: !isDevMode() }),

  private store = inject(Store);
  users = this.store.selectSignal(usersFeature.selectUsers);  // signal read

  ngOnInit() { this.store.dispatch(UsersActions.loadUsers()); }
  // template: @for (u of users(); track u.id) { ... }
}

11. Common Mistakes

Traps

  • select ki jagah poora state subscribe karke manually filter — selectors use karo (memoized).

  • Legacy StoreModule.forRoot/forFeature naye standalone apps mein — provideStore/provideState.
  • Dispatch ke turant baad state read karke sync result expect karna — flow async ho sakta (effects).

12. Best Practices

  • State ko feature slices mein organize karo; lazy features apni state khud laayein.
  • Read via selectors (select/selectSignal), na ki raw state.
  • DevTools enable karo (dev mode) — time-travel debugging.

13. Performance

Store perf mostly selector memoization + OnPush se — stable derived references unnecessary re-renders rokti. selectSignal fine-grained CD deta. Bade state trees mein deep nesting avoid; feature-split lazy state initial bundle chhota rakhta. Immutability precise change detection enable karti.

15. Interview Questions

EASY Store ke saath component kaise interact karta?

Do tarike: store.dispatch(action) — ek change trigger karta (kya hua batata). store.select(selector) (Observable) ya store.selectSignal(selector) (signal) — state slice padhta. Component ko nahi pata state kaise badalti (bas dispatch), sirf derived state padhta (select).

EASY Modern standalone app mein store kaise register karte ho?

MEDIUM Root state vs feature state?

Root state (provideStore) app-wide, bootstrap pe available. Feature state (provideState in lazy route providers) route/feature-scoped — us feature ke load hone pe hi state register hoti, tree-shakable + modular. Bade apps mein feature-based lazy state initial bundle chhota + organization behtar.

SCENARIO State store se select kiya, mutate kar diya seedhe, ab weird bugs. Kyun?

Store ka state immutable hai — select se mila reference ko mutate karna (users.push(...)) store ke internal state ko corrupt karta, change detection + DevTools time-travel todta, aur unpredictable behavior deta. State sirf actions dispatch karke, reducers ke through (naya object banakar) badle. Read-only treat karo.

S-03

Actions

Covers: createAction · props · type strings · createActionGroup · emptyProps · action hygiene · event-driven naming · dispatch

1. What is it? (Kya hai ye?)

Action ek plain object hai jo describe karta hai ki kya hua — ek event. Har action ka ek unique type string aur optionally payload (props) hoti. createAction('[Users Page] Load Users'). Components inhe dispatch karke state change trigger karte.

2. Why do we need it? (Zaroorat kyun hai?)

NgRx mein state directly nahi badalti — har change ek explicit "event" (action) se hota. Ye traceability deta: DevTools mein har action dikhta ("kya hua, kab"). Aur ye decouples: component sirf batata "ye event hua"; reducers/effects decide karte state kaise react kare.

3. What problem does it solve?

Untraceable, implicit state changes. Actions har change ko named, explicit event bana dete — audit trail (DevTools), decoupling (dispatcher ko nahi pata handler kaun), aur clear intent ("[Login Page] Login Clicked" self-documenting).

4. Internal Working (Engine ke andar kya hota hai?)

Action = { type, ...payload }

createAction ek action creator function return karta — call karne pe ek action object banta { type: '[Users Page] Load Users', ...props }. type unique identifier hai (convention: '[Source] Event'). Jab dispatch hota, ye object store ke through saare reducers + effects ko broadcast hota; jo iska type handle karte, react karte. Payload props<{...}>() se type-safe. createActionGroup related actions ka group ek jagah banata (boilerplate kam).

Diagram — action flow
 createAction('[Users] Load', props<{page}>()) ─▶ creator fn
     dispatch(loadUsers({page:1}))
         ─▶ { type:'[Users] Load', page:1 }  (broadcast)
             ├─▶ reducers matching this type react
             └─▶ effects with ofType(loadUsers) react

5. Execution Flow

 define action ─▶ component dispatch(action(payload)) ─▶ action object created
   ─▶ store broadcasts ─▶ reducers (on) + effects (ofType) receive ─▶ react

6. Architecture — the API

createAction(type)No-payload action. createAction('[App] Init').
createAction(type, props<T>())With typed payload.
createActionGroup({source, events})Group of related actions (v14+). Less boilerplate.
emptyProps()Explicit "no payload" in a group.
Type convention'[Source] Event' — [Users Page] Load Users. Source = where; Event = what.

good vs bad naming

10. Code Explanation — Line by Line

  1. createActionGroup({ source, events }) — ek source ke related actions.
  2. Event keys → camelCased creators ('Load Users' → loadUsers).
  3. props<{...}>() — type-safe payload; emptyProps() — no payload.
  4. Type auto: '[Users Page] Load Users'.
  5. Separate groups for page-events vs API-results — clear flow.

11. Common Mistakes

Traps
  • Ek generic action ko multiple jagah reuse karna (setLoading) — origin trace nahi hota. One event = one action.
  • Command-style naming (setUsers) instead of event-style (loadUsersSuccess) — actions "kya hua" describe karte, "kya karo" nahi.
  • Type strings mein source-prefix na dena — DevTools mein confusing.
  • Payload ko untyped (props<any>) — type-safety lose.
  • Bahut zyada actions (har micro-step) — noise; meaningful events pe rakho.

12. Best Practices

Do this
  • createActionGroup for related actions — kam boilerplate.
  • Event-style naming: '[Source] What Happened' (unique per source).
  • Ek real event = ek action; reuse avoid (traceability).
  • Payload hamesha typed (props<T>()).
  • Separate source groups: Page (intent), API (result), etc.

13. Performance

Actions khud negligible (plain objects). Perf indirectly: bahut granular actions zyada dispatch/reduce cycles = overhead. Meaningful events pe rakho. Action creators tree-shakable. DevTools mein bahut noise slow debugging.

S-04

Reducers

Covers: createReducer · on() · pure functions · immutability · initial state · createFeature · auto-generated selectors · no business logic

1. What is it? (Kya hai ye?)

Reducer ek pure function hai jo current state + ek action leke naya state return karta hai: (state, action) => newState. Ye batata hai state har action pe kaise badle. createReducer(initialState, on(action, handler), ...).

2. Why do we need it? (Zaroorat kyun hai?)

State change ka logic ek predictable, testable jagah chahiye. Reducers pure hain — same (state, action) hamesha same newState dete, koi side-effect nahi. Isse state transitions deterministic, unit-testable, aur time-travel-able bante.

3. What problem does it solve?

Unpredictable, scattered state mutations. Reducers state-change logic ko ek jagah, pure + immutable form mein rakhte — predictability, testability, aur DevTools time-travel (kyunki har state pichle + action se derivable hai).

4. Internal Working (Engine ke andar kya hota hai?)

Pure + immutable

Dispatch pe store har reducer ko (currentState, action) deta. Reducer on(action, (state, props) => newState) handlers check karta — jo action match kare, uska handler chalta aur naya state object return karta (spread {...state, ...changes} se — purana kabhi mutate nahi). Agar koi handler match na kare, state as-is return. Pure ka matlab: no HTTP, no Date.now()/random, no mutation — sirf input se output. Yehi purity time-travel + predictability enable karti.

Diagram — reducer purity

S-05

Effects

Covers: createEffect · functional effects · ofType · Actions stream · switchMap · error handling (catchError) · dispatch result action · non-dispatching · concatLatestFrom

1. What is it? (Kya hai ye?)

Effects side-effects (HTTP, WebSocket, localStorage, navigation) ko handle karte — reducers pure hote hain, toh async/impure kaam kahin aur chahiye. Ek effect ek RxJS stream hai jo actions ko sunta hai, side-effect karta, aur (usually) ek naya action dispatch karta.

2. Why do we need it? (Zaroorat kyun hai?)

Reducers pure — HTTP/async wahan nahi. Effects ye gap bharte: loadUsers action suno → API call karo → result se loadSuccess/loadFailure dispatch karo. Async logic ek isolated, testable, RxJS-powered jagah — components se decoupled.

3. What problem does it solve?

Side-effects ko unidirectional flow mein integrate karna, cleanly. Component sirf intent dispatch karta (loadUsers); effect async handle karke result action deta; reducer state update karta. Isse async logic centralized, RxJS operators (switchMap/retry) available, aur components thin rehte.

4. Internal Working (Engine ke andar kya hota hai?)

Actions stream → filter → side-effect → new action

NgRx ek Actions Observable expose karta — har dispatched action ka stream. Effect is stream ko ofType(someAction) se filter karta (sirf woh action), phir ek flattening operator (usually switchMap) se side-effect (HTTP) chalata, aur result ko map se ek naye action mein badalta jo automatically dispatch ho jaata. Errors ko inner catchError (switchMap ke andar) se handle karna zaroori — warna effect stream mar jaata (aur phir woh action kabhi handle nahi hota). Non-dispatching effects ({ dispatch: false }) side-effect karte par action return nahi (jaise logging/navigation).

Diagram — effect pipeline
 Actions$ ─▶ ofType(loadUsers) ─▶ switchMap(() =>
                api.getUsers().pipe(
                  map(users => loadSuccess({users})),   ← dispatched
                  catchError(e => of(loadFailure({error})))  ← inner! stream survives
                ))
           ─▶ result action auto-dispatched ─▶ reducer

5. Execution Flow

 dispatch(loadUsers) ─▶ effect: ofType matches ─▶ switchMap(HTTP)
   success ─▶ map ─▶ loadSuccess({users}) ─▶ (auto-dispatch) ─▶ reducer
   error   ─▶ inner catchError ─▶ loadFailure({error}) ─▶ (auto-dispatch) ─▶ reducer

Selectors

Covers: createSelector · createFeatureSelector · memoization · composition · derived state · view-model selectors · parameterized · OnPush

1. What is it? (Kya hai ye?)

Selectors pure functions hain jo store se specific state slices padhte aur derive karte hain. createSelector se banate; ye memoized hain — inputs unchanged toh cached result. Components selectors se state read karte (store.select(selector)).

2. Why do we need it? (Zaroorat kyun hai?)

Components ko poora state tree nahi, specific derived slices chahiye ("active users count", "filtered cart"). Selectors ye derivation ek reusable, memoized jagah rakhte — components decoupled from state shape, aur memoization se redundant recomputation + unnecessary re-renders rukte.

3. What problem does it solve?

State-shape coupling aur redundant computation. Selectors ek "read layer" dete: component sirf selector jaanta (state shape nahi — shape badle toh sirf selector update). Memoization derived values ko cache karke OnPush ke saath efficient rendering deti.

4. Internal Working (Engine ke andar kya hota hai?)

Memoization + composition

createSelector(...inputSelectors, projector) input selectors ke results ko project karta. Memoization: agar input selectors ke results referentially unchanged hain (===), projector dobara nahi chalta — cached result return hota. Isliye immutability important (naya reference = recompute, same = cached). Selectors composable hain — ek selector doosre ko input le sakta, ek dependency graph banake. createFeatureSelector ek feature ka top-level slice pick karta (starting point).

Diagram — memoized composition
 createFeatureSelector('users') ─▶ selectUsersState
        │

        │
 createSelector(selectUsers, u => u.filter(active)) ─▶ selectActiveUsers
        │  (memoized: inputs unchanged → cached, projector skipped)
        ▼
 store.select(selectActiveUsers) ─▶ Observable (emits only on real change)

5. Execution Flow

 state change ─▶ input selectors re-evaluate
   inputs === previous ─▶ cached result (projector NOT run)
   inputs changed      ─▶ projector runs ─▶ new result ─▶ emit to component

1. What is it? (Kya hai ye?)

@ngrx/signals ka SignalStore signals ke upar ek lightweight, signal-native state management hai. signalStore(...) ek injectable store banata jismein state (signals), computed (derived signals), aur methods (actions) ek builder-style with* features se — classic Redux boilerplate ke bina. Modern NgRx ka default recommendation.

2. Why do we need it? (Zaroorat kyun hai?)

Classic Store (Actions/Reducers/Effects/Selectors) powerful par verbose — ek simple feature ke liye 4 files. SignalStore signals-native hai: state directly signals, no | async, fine-grained CD, aur sab (state/computed/methods) ek jagah — drastically kam boilerplate, same predictability.

3. What problem does it solve?

Classic NgRx ka boilerplate + RxJS overhead simple cases mein. SignalStore structured state-management deta with far less ceremony, signal-native reactivity (template mein direct read), aur composable features — 80% cases ke liye modern sweet spot.

4. Internal Working (Engine ke andar kya hota hai?)

Builder — order matters (recap from Ch14)

signalStore features (withState, withComputed, withMethods, withHooks, withProps) ko RxJS pipe ki tarah sequentially compose karta — har feature pichhle sab members ko access kar sakta (isliye order matters). withState state ko signals mein badalta; withComputed derived signals; withMethods actions (jo patchState se immutable update karte). Sab signals — fine-grained CD, template mein direct call. signalStoreFeature se reusable custom features. RxJS side-effects rxMethod se.

import { computed, inject } from '@angular/core';
import { signalStore, withState, withComputed, withMethods, withHooks, patchState } from '@ngrx/signals';
import { rxMethod } from '@ngrx/signals/rxjs-interop';
import { tapResponse } from '@ngrx/operators';
import { pipe, switchMap, tap } from 'rxjs';

type State = { users: User[]; loading: boolean; query: string };

export const UsersStore = signalStore(
  { providedIn: 'root' },
  withState<State>({ users: [], loading: false, query: '' }),
  withComputed(({ users, query }) => ({
    filtered: computed(() => users().filter(u => u.name.includes(query()))),
    count: computed(() => users().length),
  })),
  withMethods((store, api = inject(UsersApi)) => ({
    setQuery(q: string) { patchState(store, { query: q }); },
    loadAll: rxMethod<void>(pipe(
      tap(() => patchState(store, { loading: true })),
      switchMap(() => api.getAll().pipe(tapResponse({
        next: (users) => patchState(store, { users, loading: false }),
        error: () => patchState(store, { loading: false }),
      )))),
    ),
  })),
  withHooks({ onInit(store) { store.loadAll(); } }),
);
// component: store = inject(UsersStore);  template: {{ store.filtered() }}

Component Store

Covers: @ngrx/component-store · local/component-scoped state · select · updater · effect · setState/patchState · tapResponse · lifecycle-bound · vs global Store

1. What is it? (Kya hai ye?)

@ngrx/component-store ek local, component-scoped reactive state solution hai. Global Store ke unlike, ye ek component (ya feature) ki lifetime se bandha hota hai — component destroy pe state + subscriptions clean ho jaate. Standalone (@ngrx/store pe depend nahi karta). API: select, updater, effect, patchState.

2. Why do we need it? (Zaroorat kyun hai?)

Kabhi complex local state chahiye (ek smart component ka form + async + derived state) jo global store mein daalna over-engineering ho aur plain component properties se manage karna messy. ComponentStore is beech ka solution deta — structured reactive local state, lifecycle-bound, RxJS-powered, bina global store ke.

3. What problem does it solve?

"Local state jo complex hai par global nahi honi chahiye." Global store mein daalne se pollution + boilerplate; plain properties se async/derived state messy. ComponentStore local reactive state ko clean, encapsulated, auto-cleaned form mein deta.

4. Internal Working (Engine ke andar kya hota hai?)

Lifecycle-bound reactive state

ComponentStore<State> ko component ke providers mein diya jaata — toh iski instance component ke saath banti aur destroy hoti. State reactive hai: select(fn) derived Observables deta, updater(fn) synchronous state updates ke liye, effect(fn) async side-effects (jo ek Observable ko subscribe karta, component destroy pe auto-unsubscribe). Important: ComponentStore effect ko "call" karne ka matlab ek internal Subject pe value push karna — effect stream Observable ko subscribe karke reactively handle karta. Errors ko tapResponse (@ngrx/operators) se safely handle karo (stream alive rakhta).

Diagram — ComponentStore lifecycle
 provided at component level ─▶ store created with component
   select(fn)   ─▶ derived Observable<T>
   updater(fn)  ─▶ sync state update
   effect(fn)   ─▶ async side-effect (subscribes Observable)
 component destroyed ─▶ store destroyed ─▶ all subscriptions cleaned (auto)

5. Execution Flow

 constructor: super(initialState)
 select(s => s.x) ─▶ vm$ (template | async)
 updater((s, v) => ({...s, x:v})) ─▶ call to update sync
 effect(trigger$ => trigger$.pipe(switchMap(HTTP), tapResponse(...))) ─▶ async
 destroy ─▶ auto-cleanup

import { ComponentStore } from '@ngrx/component-store';
import { tapResponse } from '@ngrx/operators';
import { switchMap, tap, Observable } from 'rxjs';

interface MoviesState { movies: Movie[]; loading: boolean; }

@Injectable()   // provided at component level (not root)
export class MoviesStore extends ComponentStore<MoviesState> {
  private api = inject(MoviesApi);
  constructor() { super({ movies: [], loading: false }); }

  readonly movies$ = this.select((s) => s.movies);
  readonly loading$ = this.select((s) => s.loading);

  readonly setLoading = this.updater((s, loading: boolean) => ({ ...s, loading }));

  readonly load = this.effect((trigger$: Observable<void>) =>
    trigger$.pipe(
      tap(() => this.setLoading(true)),
      switchMap(() => this.api.getAll().pipe(tapResponse({
        next: (movies) => this.patchState({ movies, loading: false }),
        error: () => this.setLoading(false),
      )))
    ));
}
// component: providers: [MoviesStore]; store.load(); movies$ = store.movies$;

11. Common Mistakes

Traps
  • Type-based folders (all actions in one folder, all reducers in another) — feature ko change karne ke liye 5 folders mein jaana. Feature-based better.
  • Sab kuch global root state mein — feature state lazy rakho.
  • Dumb components mein store inject karna — presentational ko store-agnostic rakho (inputs/outputs).
  • Facade ke bina bade app mein — components NgRx internals se tightly coupled.
  • Business logic components mein — effects/services mein.
  • Over-architecting chhoti app — YAGNI; complexity match karo.

12. Best Practices

Do this
  • Feature-based folders; colocate state with feature; lazy-load.
  • Dumb (presentational) + smart (container) component split.
  • Facade for decoupling in larger apps (optional but scales well).
  • Consistent naming (*.actions.ts, *.effects.ts, *.feature.ts).
  • Match tool to complexity — signals/SignalStore first, classic Store when justified.

13. Performance

Architecture perf: lazy feature state → smaller initial bundle. Dumb components + OnPush + memoized selectors → minimal re-renders. Facade with selectSignal → fine-grained updates. Feature-based structure → better tree-shaking + lazy-loading boundaries. Good structure indirectly enables all the perf patterns.

15. Interview Questions

MEDIUM NgRx project ko kaise structure karoge?

Feature-based folders — har feature apna folder with components/ (dumb), containers/ (smart), state/ (actions/reducer/effects/selectors), optional facade, aur lazy route (provideState/provideEffects). Shared cheezein shared/, singletons core/. Type-based (saare actions ek folder) avoid — feature-based colocation discoverable + lazy-loadable + scalable.

MEDIUM Facade pattern kya hai, pros/cons?

Ek service jo store interactions (dispatch/select) encapsulate karta — components sirf facade use karte, NgRx internals nahi jaante. Pros: decoupling (store refactor components ko affect nahi), cleaner components, easier testing/mocking. Cons: extra indirection layer, kuch log ise unnecessary abstraction maante. Bade apps/teams mein useful; chhoti mein optional.

MEDIUM Dumb vs smart components?

Dumb (presentational): sirf @Input/@Output (ya input()/output()), no store/service — reusable, testable, store-agnostic. Smart (container): store se connect (dispatch/select), data dumb components ko pass karta. Ye separation reusability + testability deta aur state logic ko UI se decouple karta.

SCENARIO Ek team ne saare actions ko store/actions/, reducers ko store/reducers/ mein rakha (type-based). Ek feature change karne mein pain ho raha. Better?

Feature-based structure mein migrate karo — har feature ka action/reducer/effect/selector ek feature/state/ folder mein colocated. Ek feature change karne pe sab ek jagah (5 alag folders nahi), feature deletable/lazy-loadable, aur naye devs ke liye navigable. Type-based folders scale pe fragment kar dete hain.
