S-01

Change Detection — Internal Working

Covers: what CD is · component/view tree · dirty checking · LView & TView · how bindings are compared · the CD tree · no "re-render" myth

1. What is it? (Kya hai ye?)

Change Detection (CD) woh mechanism hai jisse Angular tumhare component ki state (properties) ko template (DOM) ke saath sync rakhta hai. Jab this.count badla, CD hi decide karta hai ki template ka {{ count }} update ho. Simple shabdon mein: CD "model → view" sync engine hai.

2. Why do we need it? (Zaroorat kyun hai?)

Angular declarative hai — tum template mein {{ count }} likhte ho, manually DOM update nahi karte. Toh kisi ko toh check karna padta "count badla kya? Agar haan, DOM patch karo". Yahi CD karta. Bina iske tumhe har change pe manually element.textContent = ... likhna padta (imperative, error-prone).

3. What problem does it solve?

Manual DOM synchronization. CD automate karta: tum sirf component state badlo, CD binding-by-binding compare karke sirf badle DOM nodes update karta. Declarative UI ka foundation — "state describe karo, sync framework kare".

4. Internal Working (Engine ke andar kya hota hai?)

Component tree = view tree; dirty checking

Har Angular app ek component tree hai (root se neeche). Internally har component ka ek view (runtime data structure LView — Logical View — + static TView shared blueprint) hota. Jab CD chalti, Angular is tree ko top-to-bottom (root se leaves) traverse karta. Har view mein template ke har binding (jaise {{ count }}, [value]) ka pichla value stored hota. CD current value ko pichle se === (identity) se compare karta — dirty checking. Agar badla, DOM update + naya value store. Ye per-binding comparison compiler ne generate ki template instructions se hota (super fast, monomorphic). Important: Angular kabhi poora component "re-render" nahi karta — sirf changed bindings ke DOM nodes patch karta.

Diagram — the component/view tree
        AppRoot (view)
        /          \
   Header        UsersPage (view)
                 /        \
          UserList      Sidebar
          (view)        (view)
   CD traverses TOP → BOTTOM, checking each view's bindings
   each binding: current === previous ?  no → update that DOM node

5. Execution Flow

 something triggers CD (event/async/signal)
   ─▶ CD starts at root view
   ─▶ for each view (top-down): run its template's update instructions
       ─▶ each binding: compare current vs stored previous (===)
           changed  → patch that DOM node + store new value
           same     → skip
   ─▶ recurse into child views
   ─▶ done: DOM in sync with state

6. Architecture — the pieces

Component treeRoot → children; mirrors your template nesting.
LViewPer-instance runtime array: bindings' current + previous values, DOM refs, state flags.
TViewStatic template blueprint shared by all instances of a component.
Dirty checkingPer-binding === compare vs stored previous value.
Update instructionsCompiler-generated per-template functions (fast, no reflection).

7. Real Project Example (Angular context)

what the compiler roughly generates (conceptual)

// compiler → update function (runs each CD for this view):
function updateView(lView) {
  // ɵɵtextInterpolate compares & patches ONLY if changed
  if (bindingUpdated(lView, 0, ctx.title))      setText(node0, ctx.title);
  if (bindingUpdated(lView, 1, ctx.user.name))  setText(node1, ctx.user.name);
  // bindingUpdated = current !== previous ? (store new, return true) : false
}

9. Advanced Example — why immutability & identity matter

=== comparison consequences
// CD compares by reference (===) for objects/arrays:
this.user.name = 'New';        // SAME object reference → OnPush may MISS it
this.user = { ...this.user, name: 'New' };  // NEW reference → detected

// primitives compare by value:
this.count = this.count + 1;   // 1 !== 0 → detected fine
// This is WHY OnPush + immutability go together (S-05).

10. Code Explanation — Line by Line

  1. Component tree = view tree; har view ka ek LView (runtime state).
  2. CD root se start, top-to-bottom har view visit.
  3. Har binding ka current value stored previous se === compare.
  4. Changed → us specific DOM node ko patch + naya value store.
  5. Objects reference se compare — isliye immutability matters (OnPush).
  6. No full re-render — sirf changed bindings ke nodes touch hote.

11. Common Mistakes

Traps
  • Sochna Angular poora component "re-render" karta har change pe — nahi, sirf changed bindings patch hote.
  • Objects ko mutate karke expect karna CD detect karega (OnPush mein miss) — reference badlo.
  • Template mein heavy function calls ({{ compute() }}) — har CD pe chalti (S-04 perf).
  • CD ko "slow magic" samajhna — actually per-binding ===, bahut fast; problem tab jab bahut baar/bahut components.
  • Template expressions mein side-effects — CD unhe bar-bar chala sakti (pure rakho).

12. Best Practices

Do this
  • Template expressions PURE + cheap rakho (no heavy compute / side-effects).
  • Objects/arrays immutably update karo (naya reference) — CD-friendly.
  • Signals prefer karo (fine-grained, CD ko exact batate kya badla).
  • Samjho CD per-binding hai — design accordingly (OnPush, trackBy).

13. Performance

CD ki cost = (views checked) × (bindings per view) × (CD frequency). Per-binding === individually tez hai; problem scale pe — hazaaron bindings × frequent ticks. Optimizations: OnPush (views skip), signals (targeted), trackBy/track (list diffing), pure template expressions. No-full-re-render design already efficient; goal hai unnecessary checks minimize karna.

15. Interview Questions

EASY Change detection kya hai?

Woh mechanism jo component ki state (properties) ko template (DOM) ke saath sync rakhta hai. Jab data badalta, CD binding-by-binding compare (dirty checking) karke sirf badle DOM nodes update karta. Ye Angular ke declarative UI ka foundation — tum state describe karte ho, CD DOM sync karta.

MEDIUM Angular internally CD kaise karta hai (dirty checking)?

Har component ka ek view (LView runtime + TView blueprint) hota. CD component tree ko top-to-bottom traverse karti; har view mein compiler-generated update instructions har binding ka current value stored previous se === compare karti. Changed → us DOM node ko patch + naya value store. Ye per-binding comparison hi "dirty checking" hai.

MEDIUM "Angular re-renders the component" — sahi hai?

Nahi, ye common misconception hai. Angular poora component/template dobara render nahi karta. CD sirf bindings check karti aur agar koi badla toh sirf us affected DOM node ko update karti — baaki DOM untouched. Isliye CD efficient hai; "re-render" React-style virtual-DOM reconciliation nahi, targeted binding patching hai.

SCENARIO Tumne this.user.name = 'X' kiya par UI update nahi hua (OnPush component). Internally kyun?

CD objects ko reference (===) se compare karti. user.name mutate karne se user ka reference same rehta — OnPush strategy ko lagta "kuch nahi badla" (input reference unchanged), toh view skip. Fix: naya reference do — this.user = {...this.user, name: 'X'} — ab reference badla, CD detect karti. Yehi immutability + OnPush ka connection hai.

The Rendering Cycle

Covers: the tick · top-down traversal · binding evaluation · DOM update · mark→refresh phases · lifecycle hooks order · synchronous cycle · dev double-check

1. What is it? (Kya hai ye?)

Rendering cycle (ek "tick") woh ek complete pass hai jo CD component tree pe chalati — root se leaves tak, har view ke bindings evaluate karke DOM sync karti, aur beech mein lifecycle hooks fire karti. Ek tick = ek synchronous run.

2. Why do we need it? (Zaroorat kyun hai?)

State kabhi bhi badal sakti (click, HTTP, timer, signal). Rendering cycle ek organized, predictable pass deta jahan Angular saari pending changes ko ek baar mein DOM mein flush karta — na ki har micro-change pe alag-alag. Ye batching + ordering (hooks) deta.

3. What problem does it solve?

Chaotic, uncoordinated DOM updates. Ek structured tick ensure karta: sab bindings ek consistent snapshot se evaluate hon, hooks sahi order mein chalein, aur DOM ek coordinated pass mein update ho — predictable + debuggable rendering.

4. Internal Working — the two phases of a tick

Mark dirty → refresh (synchronize)

Ek tick conceptually do phases mein: (1) Mark phase — trigger (event/signal/markForCheck) affected views ko "dirty" mark karta (aur ancestors ko root tak, taaki traversal un tak pahunche). (2) Refresh/synchronize phase — Angular tree ko top-down traverse karta; har view jo checkable hai (dirty ya CheckAlways), uske bindings evaluate karta, lifecycle hooks fire karta (ngOnInit once, phir har tick ngDoCheck → ngOnChanges if inputs changed → content/view hooks), aur DOM patch karta. Poora tick synchronous hai. Dev mode mein ek extra "checkNoChanges" pass hota jo verify karta ki bindings tick ke dauraan stable rahe — na rahe toh ExpressionChangedAfterItHasBeenCheckedError.

Diagram — one tick
 TRIGGER (event / signal / markForCheck / async)
   │
   ▼  PHASE 1: mark dirty (view + ancestors to root)
 root ──dirty──▶ page ──dirty──▶ list  (path marked)
   │
   ▼  PHASE 2: refresh (top-down traversal)
 for each checkable view:
    ngDoCheck / ngOnChanges → evaluate bindings → update DOM → child hooks
   │
   ▼  (dev only) checkNoChanges pass → ExpressionChanged...Error if unstable
 DOM now in sync

5. Execution Flow — hooks order within a cycle

 tick ─▶ (per component, top-down)
   ngOnChanges (if @Input changed)
   ngOnInit    (first tick only)
   ngDoCheck
   ngAfterContentInit / ngAfterContentChecked
   ngAfterViewInit / ngAfterViewChecked
   → bindings evaluated, DOM updated
 (ngOnDestroy fires when a view is removed, not part of normal refresh)

6. Architecture — what triggers a tick

DOM events(click), (input) — handler runs, then a tick.
Async (Zone mode)setTimeout, Promise, XHR — Zone.js schedules a tick (S-03).
SignalsA read signal changes → schedules a targeted tick (S-08).
markForCheck()Marks dirty for the NEXT tick (S-06).
ApplicationRef.tick()Manually run a full CD pass.

  2. Refresh pass top-down: har checkable view visit.

11. Common Mistakes

Traps
  • Post-check hooks (ngAfterViewChecked) mein bound state badalna → ExpressionChangedAfterItHasBeenChecked.
  • Template mein function calls — har tick pe chalti (unpredictable frequency + perf).
  • Sochna hooks har baar sabhi fire karte — ngOnInit once, baaki har checkable tick.
  • ngDoCheck mein heavy work — har tick pe chalta.
  • Tick ko async samajhna — ek tick synchronous hai.

12. Best Practices

Do this
  • Bound values ko within-a-tick stable rakho (post-check hooks mein mutate mat karo).
  • Template expressions cheap + pure; function calls avoid (precompute / signals / pipes).
  • Lifecycle hooks light rakho (especially per-tick ngDoCheck/AfterViewChecked).
  • Signals se targeted ticks — poore tree ke bajaye.

13. Performance

Ek tick ki cost = checkable views × bindings + hooks work. Frequent ticks (Zone.js har async pe) sabse bada perf drain. Reduce: OnPush (fewer checkable views), signals (targeted ticks), zoneless (no blanket async ticks), light hooks, pure expressions. Har tick synchronous hai — heavy work main thread block karta (jank).

15. Interview Questions

EASY Ek CD "tick" kya hai?

Ek trigger (event/async/signal) ke baad usually ek tick chalta jo saari pending changes DOM mein flush karta.

MEDIUM Ek tick mein lifecycle hooks kis order mein?

HARD ExpressionChangedAfterItHasBeenCheckedError kyun aata hai?

Agar badla (jaise ngAfterViewChecked mein bound property mutate ki), Angular throw karta — kyunki iska matlab view aur model ek hi tick mein inconsistent ho gaye. Fix: post-check hooks mein bound state mat badlo; stable value / setTimeout / signal use karo. Production mein ye check off hota.

SCENARIO Ek component har second slow lagta, aur DevTools mein CD bar-bar chal rahi. Rendering cycle ke lens se debug kaise?

Probably bahut-frequent ticks + heavy per-tick work. Check: (1) Template mein function calls ({{ compute() }}) — har tick pe chalti. (2) ngDoCheck/ngAfterViewChecked mein heavy logic. (3) Zone.js koi frequent async (timer/polling) har baar tick trigger kar raha. Fixes: OnPush se checkable views kam, signals se targeted ticks, heavy async runOutsideAngular (S-03), template expressions precompute, ya zoneless (S-08).

S-03

Zone.js

Covers: what Zone.js is · monkey-patching async APIs · NgZone · onMicrotaskEmpty → tick · runOutsideAngular · run · eventCoalescing · downsides

1. What is it? (Kya hai ye?)

Zone.js ek library hai jo browser ke async APIs (setTimeout, Promise, addEventListener, XHR/fetch, etc.) ko monkey-patch karti hai — unhe wrap karke Angular ko notify karti jab koi async kaam khatam hota. Angular is signal ko use karke CD trigger karta. Simple: Zone.js Angular ki "aankh-kaan" thi jo batati "kuch async hua, shayad state badli, CD chalao".

2. Why do we need it? (Zaroorat kyun thi?)

Angular ko kaise pata chale ki state badli aur CD chalani chahiye? State to kisi bhi async callback (timer, HTTP response, click) mein badal sakti. Manually har jagah CD trigger karna painful. Zone.js ne ye automatic kar diya — har async operation ke baad Angular ko implicitly bata deti, toh developer ko CD ke baare mein sochna hi nahi padta tha.

3. What problem does it solve?

Zone.js ka fayda (historically)

"Automatic change detection" — developer ko kabhi manually CD trigger nahi karna padta. Koi bhi async (click/timer/HTTP) ke baad UI apne aap update. Zero-config reactivity — yahi Angular ka "it just works" tha.

Par iska cost

Har async op → poore app pe CD (over-checking). Bade bundle (~zone.js size). Confusing stack traces (patched async). Yahi cheezein zoneless (S-08) ki wajah bani.

4. Internal Working (Engine ke andar kya hota hai?)

Monkey-patch → NgZone → onMicrotaskEmpty → tick

App start pe (zone.js polyfill) browser ke async APIs ko replace kar diya jaata patched versions se. Angular ek special zone banata: NgZone ("Angular zone"). Jab bhi patched async operation Angular zone ke andar complete hoti, NgZone track karta ki koi pending microtask/macrotask baaki hai ya nahi. Jab saare pending async khatam ho jaate (zone "stable"), NgZone ka onMicrotaskEmpty event fire hota, jise Angular subscribe karke ApplicationRef.tick() (full CD pass) chalata. Yani: Zone.js state-change detect nahi karta — bas "async activity hui" detect karta aur uske baad CD trigger karta (isliye over-triggering). eventCoalescing: true multiple rapid events ko ek tick mein batch karta.

Diagram — how Zone.js triggers CD
 app bootstrap: zone.js patches setTimeout, Promise, addEventListener, XHR...
   │
 async op runs inside NgZone (e.g. button click handler / http response)
   │
 NgZone tracks pending tasks ─▶ all done? ─▶ onMicrotaskEmpty fires
   │
 Angular subscribes ─▶ ApplicationRef.tick() ─▶ FULL top-down CD pass
   (triggers even if NO real state changed — just async happened)

5. Execution Flow

 (click) ─▶ patched addEventListener ─▶ handler runs (in NgZone)
   ─▶ handler done, no pending tasks ─▶ onMicrotaskEmpty ─▶ tick() ─▶ CD
 setTimeout(fn) ─▶ patched ─▶ fn runs in NgZone ─▶ onMicrotaskEmpty ─▶ tick()

6. Architecture — NgZone API

NgZoneAngular's zone wrapper; emits onMicrotaskEmpty, onStable, etc.
ngZone.run(fn)Run code INSIDE Angular zone (will trigger CD).
ngZone.runOutsideAngular(fn)Run code OUTSIDE zone (NO CD triggered) — perf escape hatch.
provideZoneChangeDetection({eventCoalescing:true})Batch rapid events into fewer ticks.
polyfills (angular.json)Where zone.js is loaded; removed for zoneless.

7. Real Project Example — runOutsideAngular

escaping CD for a hot loop
export class CanvasComponent {
  private zone = inject(NgZone);

  startAnimation() {
    // requestAnimationFrame fires ~60x/sec — DON'T trigger CD each frame
    this.zone.runOutsideAngular(() => {
      const loop = () => { this.draw(); requestAnimationFrame(loop); };
      loop();
    });
  }

  onDone(result: string) {
    // when you DO need the UI to update, re-enter the zone:
    this.zone.run(() => { this.status = result; });   // triggers CD
  }
}

10. Code Explanation — Line by Line

  1. Bootstrap pe zone.js async APIs ko patch karta.
  2. Async ops NgZone ke andar chalti, pending tasks track hote.
  3. Saare tasks done → onMicrotaskEmpty → Angular tick().
  4. Tick full CD pass — chahe state badla ho ya na (over-trigger).
  5. runOutsideAngular — zone ke bahar, no CD (perf).
  6. run — wapas zone mein, CD trigger.

11. Common Mistakes

Traps
  • Sochna Zone.js state-changes detect karta — nahi, sirf "async activity" detect karta, phir blanket CD.
  • High-frequency async (mousemove, rAF, WebSocket flood) ko zone ke andar chalana — har baar full CD.
  • runOutsideAngular mein state badalke UI update expect karna — CD trigger nahi hoga; run se re-enter karo.
  • 3rd-party lib (jo apna async karti) zone ke bahar chale → UI update na ho (manual run/markForCheck).
  • Zone.js ko "the change detector" samajhna — ye sirf trigger hai; CD alag mechanism (S-01).

12. Best Practices

Do this (Zone.js apps)
  • High-frequency async (animations, rAF, heavy timers) → runOutsideAngular.
  • UI-affecting update ke waqt ngZone.run(...) se re-enter.
  • provideZoneChangeDetection({eventCoalescing:true}) — rapid events batch.
  • Better: OnPush + signals; best (modern): zoneless (S-08) — zone.js hi hata do.

13. Performance

Zone.js ka core perf issue: over-triggering — har async op (even irrelevant) full CD pass karwati. Plus bundle size (~zone.js payload) aur patched-async stack traces (debug harder). runOutsideAngular + coalescing help karte, par fundamental fix zoneless hai (signals precisely batate kab CD chahiye). Isliye Angular zone.js se door ja raha.

15. Interview Questions

EASY Zone.js kya karti hai Angular mein?

Zone.js browser ke async APIs (setTimeout, Promise, events, XHR) ko monkey-patch karti hai, taaki jab koi async operation complete ho, Angular ko implicitly pata chale aur woh change detection trigger kar sake. Isse "automatic CD" milta — developer ko kabhi manually CD chalani nahi padti. Ye Angular ka classic "it just works" reactivity tha.

MEDIUM Zone.js exactly CD kaise trigger karti hai?

NgZone pending microtasks/macrotasks track karta; jab saare khatam (zone stable), onMicrotaskEmpty event fire hota, jise Angular subscribe karke ApplicationRef.tick() (full top-down CD) chalata. Note: Zone.js state-change detect nahi karta — sirf "async activity hui" — isliye woh blanket/over-trigger karta (even jab state na badli ho).

Jab high-frequency async ho jo UI ko affect nahi karti — animations, requestAnimationFrame, frequent timers, heavy polling. Zone ke andar ye har baar full CD trigger karti (waste). ngZone.runOutsideAngular(fn) mein ye code zone ke bahar chalta — koi CD nahi. OnPush Strategy

Covers: ChangeDetectionStrategy.OnPush · reference-based checks · the 4 triggers · immutability · skipping subtrees · async pipe · signals · mixed trees

1. What is it? (Kya hai ye?)

ChangeDetectionStrategy.OnPush ek optimization hai — is strategy wale component ko Angular har tick pe check nahi karta. Woh view tab tak "skip" hoti hai jab tak koi specific trigger use "dirty" na kar de. Matlab: "sirf tab check karo jab kuch relevant push (badla) ho".

2. Why do we need it? (Zaroorat kyun hai?)

Default har view har tick check karta (S-04) — waste. OnPush Angular ko batata "is component ki output sirf iske inputs (aur kuch triggers) pe depend karti; agar woh nahi badle, mujhe skip kar do". Isse bade tree ke poore subtrees skip ho jaate — bada perf win.

3. What problem does it solve?

Default ka over-checking. OnPush CD ko selective banata: sirf woh views check hoti jinke inputs/state actually badle. Bade apps mein ye "har baar sab check" se "sirf zaroorat pe kuch check" ka shift deta — measurable performance.

4. Internal Working — the 4 triggers

OnPush view checkable hoti hai sirf jab...

Ek OnPush view normally skip hoti, par ye 4 cheezein use dirty (checkable) bana deti:

  1. Input reference change — ek @Input ko naya reference (=== se alag) mile. (Mutation, same ref → NO trigger.)
  2. Event from the view (or its children) — component ke template mein koi (click)/DOM event fire ho.
  3. Async pipe emit — template mein | async ne naya value emit kiya (internally ye markForCheck call karta).
  4. Explicit markForCheck() (ya signal read in template jo badla — signal-based components).

Jab in mein se koi hota, Angular us view (aur ancestors ko root tak) ko dirty mark karta; next tick pe woh dirty path check hoti. Warna OnPush subtree poora skip.

Diagram — OnPush skips unless triggered
 tick ─▶ root ✓
          ├─ A (OnPush)  ⏭ SKIPPED  (no new input / event / async / markForCheck)
          │   └─ A1      ⏭ skipped (parent skipped → subtree skipped)
          └─ B (OnPush)  ✓ CHECKED  (got a new @Input reference)
              └─ B1      ✓ checked
 skip an OnPush parent → its whole subtree is skipped that tick

5. Execution Flow

 trigger (new input ref / event / async emit / markForCheck / signal)
   ─▶ mark view + ancestors dirty (to root)
   ─▶ next tick: traverse — dirty path checked, non-dirty OnPush subtrees SKIPPED
   ─▶ after check, dirty flag cleared (until next trigger)

Covers: ChangeDetectorRef.markForCheck · mark path-to-root dirty · for the NEXT tick · does NOT run CD itself · async pipe uses it · OnPush + imperative updates

1. What is it? (Kya hai ye?)

markForCheck() ChangeDetectorRef ka method hai jo current view aur uske saare ancestors ko root tak "dirty" mark kar deta — taaki agli CD tick mein woh path check ho. Ye OnPush components ke liye hai: "mere paas kuch badla, next tick pe mujhe check karna".

2. Why do we need it? (Zaroorat kyun hai?)

OnPush view normally skip hoti. Par kabhi state imperatively badalti hai — jaise ek service ka callback, WebSocket message, ya manual subscription — jahan koi automatic trigger (new input ref / async pipe) nahi. Aise mein tum khud Angular ko batao "check karna hai" via markForCheck().

3. What problem does it solve?

OnPush ka blind-spot: imperative/non-reference state changes. markForCheck ek manual "mujhe next tick check karo" signal deta, taaki OnPush component reliably update ho jab automatic triggers na hon (manual subscriptions, timers, external callbacks).

4. Internal Working (Engine ke andar kya hota hai?)

Marks a path, does NOT run CD

Yaad rakhne wali sabse important baat: markForCheck() CD ko turant chalata nahi. Ye sirf ek flag set karta — current view ko dirty, aur upar chalte hue har ancestor ko bhi dirty, root tak. Iska matlab: agli baar jab koi CD tick chalti (jaise zone.js se, ya zoneless scheduler se), traversal is dirty path ko check karega (kyunki dirty views + unke ancestors checkable ho jaate). Agar koi tick hi na chale, markForCheck akela kuch visible nahi karta — woh sirf "next tick ke liye" mark karta. async pipe internally har emit pe yahi call karta — isliye async pipe OnPush mein bina manual kaam ke chalti.

Diagram — markForCheck marks the path

   root ★dirty
     └─ B ★dirty
         └─ C ★dirty   ← called here; marks C + ancestors to root
 (no CD runs NOW — just flags set)
 NEXT tick ─▶ traversal follows dirty path ─▶ C checked

5. Execution Flow

 imperative state change (service callback / manual subscribe / timer)
   ─▶ cdr.markForCheck()   (sets dirty flags: view + ancestors to root)
   ─▶ ... eventually a tick runs (zone / scheduler) ...
   ─▶ dirty path traversed & checked ─▶ DOM updates ─▶ flags cleared

markForCheck()Marks view + ancestors dirty; CD runs on the NEXT tick (async, deferred). Marks UP to root.
detectChanges()Runs CD NOW, synchronously, on THIS view + its descendants (S-07). Goes DOWN.
DirectionmarkForCheck: up (to root). detectChanges: down (to children).
TimingmarkForCheck: deferred. detectChanges: immediate.

1. What is it? (Kya hai ye?)

detectChanges() ChangeDetectorRef ka method hai jo CD ko abhi, synchronously chalata — is component par aur uske descendants (children) par. markForCheck ke ulta: woh mark karta (next tick), ye turant run karta (this view down).

2. Why do we need it? (Zaroorat kyun hai?)

Kabhi tumhe CD ko turant chalana hota — jaise ek detached view (jo tune CD tree se hataya perf ke liye) ko manually update karna, ya kisi precise moment pe DOM sync karna. detectChanges ye local, on-demand control deta.

3. What problem does it solve?

On-demand, localized synchronous CD. Especially "detached view" pattern ke liye — jahan tum ek high-frequency component ko CD tree se detach karte ho aur controlled intervals pe khud detectChanges karte ho (jaise har 10th frame), poore-app CD ke bina.

4. Internal Working (Engine ke andar kya hota hai?)

Runs CD now, this view + descendants (not ancestors)

detectChanges() turant ek local CD pass chalata: is view ke bindings check karta aur phir neeche (descendants) traverse karta — par upar (ancestors) nahi. Ye synchronous hai (call return hone tak DOM updated). Iska bhai ChangeDetectorRef ka poora family deta fine control: detach() ek view ko CD tree se hata deta (normal ticks use skip karengे), reattach() wapas jodta, markForCheck() (S-06) next-tick ke liye up-mark karta, aur checkNoChanges() (dev) verify karta bindings stable. Detached view pattern: detach() + apni marzi se detectChanges() = surgical control over kab ek subtree update hota.

Diagram — detectChanges goes DOWN, now
 detectChanges() on view B (SYNCHRONOUS, immediate):
   root      (ancestors NOT touched)
     └─ B ✓  ← starts here
         ├─ B1 ✓
         └─ B2 ✓   (descendants checked, top-down from B)
 vs markForCheck(): marks UP to root, runs on NEXT tick.

5. Execution Flow — detached view pattern

 constructor: cdr.detach()          ← view removed from normal CD ticks
 high-frequency update (e.g. per frame):
   this.data = ...; 
   if (frame % 10 === 0) cdr.detectChanges();   ← manual, throttled CD
 → app-wide ticks never check this view; you control exactly when it updates

6. Architecture — the ChangeDetectorRef family

detectChanges()Run CD NOW on this view + descendants (sync, downward).
markForCheck()Mark dirty up to root for the NEXT tick (S-06).
detach()Remove this view from the CD tree (skipped by ticks).
reattach()Re-add a detached view to the CD tree.
checkNoChanges()Dev assert: throws if bindings changed (stability check).
ApplicationRef.tick()Run a full app-wide CD pass (from root).

7. Real Project Example — detached high-frequency view

throttled manual CD
@Component({ selector: 'app-ticker', template: `{{ price }}` })
export class TickerComponent implements OnInit, OnDestroy {
  private cdr = inject(ChangeDetectorRef);
  private feed = inject(PriceFeed);
  price = 0;
  private sub?: Subscription;

  ngOnInit() {
    this.cdr.detach();                 // opt out of app-wide CD
    this.sub = this.feed.prices$.subscribe((p) => {
      this.price = p;                  // updates arrive ~100x/sec
      this.cdr.detectChanges();        // we decide when to render (locally)
    });
  }
  ngOnDestroy() { this.sub?.unsubscribe(); }
}

11. Common Mistakes

Traps
  • detectChanges ko markForCheck ki jagah routinely use karna — forced sync CD zyada costly + can cause re-entrancy.
  • detach() ke baad markForCheck se update expect karna — detached view ticks pe check nahi hoti; detectChanges/reattach chahiye.
  • detectChanges ko CD ke andar (ek hook mein) call karke nested-CD / ExpressionChanged errors trigger.
  • detach karke ngOnDestroy mein cleanup (unsubscribe) bhoolna — leaks.
  • Sochna detectChanges ancestors bhi check karta — sirf this view + descendants.

12. Best Practices

Do this
  • Default choice markForCheck (deferred, batched); detectChanges sirf jab turant/local sync chahiye.
  • detach + detectChanges ka use high-frequency isolated views ke liye (tickers, canvases).
  • Detached views ka lifecycle carefully manage (reattach/cleanup).
  • Modern apps: pehle signals/OnPush try karo — ye low-level CDR calls ki zaroorat kam karte.

13. Performance

detectChanges synchronous + immediate — agar bar-bar call ho toh costly (har call ek CD pass). markForCheck aksar behtar kyunki multiple marks ek tick mein batch. Par detached-view + throttled detectChanges pattern high-frequency data (real-time feeds, animations) ke liye poore-app CD se bacha kar bada win de sakta — surgical control. Signals modern alternative jo zyaadaatar cases mein ye manual work hata dete.

EASY detectChanges() kya karta hai?

CD ko turant, synchronously chalata — is view par aur uske descendants (children) par, ancestors par nahi. Call return hone tak DOM updated. Ye markForCheck se alag hai jo sirf next-tick ke liye mark karta (up to root). detectChanges = "abhi, yahan se neeche check karo".

HARD markForCheck vs detectChanges — poora farak?

Direction: markForCheck UP mark karta (view→ancestors→root); detectChanges DOWN check karta (view→descendants). Timing: markForCheck deferred (next tick); detectChanges immediate/synchronous (abhi). Scope: markForCheck path ko checkable banata taaki tick check kare; detectChanges khud ek CD pass chalata. Use: markForCheck OnPush + imperative updates ke liye (batched); detectChanges detached views / precise immediate sync ke liye.

MEDIUM detach()/reattach() kab use karoge?

Jab ek view high-frequency updates receive karti (real-time price ticker, canvas animation) jise poore-app CD mein include karna wasteful hai. detach() use CD tree se hata deta (ticks skip), phir tum detectChanges() se controlled intervals pe (jaise har 10th frame) manually render karte ho. reattach() wapas normal CD mein jodta — jaise off-screen widget ko hide pe detach, show pe reattach.

SCENARIO Ek real-time chart 100 updates/sec receive kar raha aur app janky ho gaya. ChangeDetectorRef se kaise optimize?

Chart view ko detach() karo (app-wide CD se nikaalo — 100 ticks/sec se bacho). S-08

Zoneless Angular

Covers: provideZonelessChangeDetection · signals as trigger · what schedules CD · removing zone.js · migration · v20.2 stable · v21 default · OnPush stepping-stone · testing

1. What is it? (Kya hai ye?)

Zoneless Angular matlab Zone.js ke bina change detection. Zone.js ki jagah (jo har async pe blanket CD trigger karti thi), Angular ab explicit signals pe CD schedule karta — signals, template events, async pipe, markForCheck, aur input changes. Enable: provideZonelessChangeDetection(). v20.2 se stable, v21+ mein naye apps ka default.

2. Why do we need it? (Zaroorat kyun hai?)

Zone.js ke teen bade dard (S-03): (1) over-triggering — har async pe full CD, chahe state na badli ho; (2) bundle size (~zone.js payload); (3) confusing patched-async stack traces. Zoneless in teeno ko fix karta — CD sirf tab jab actually kuch relevant badla, chhota bundle, saaf debugging.

3. What problem does it solve?

Zoneless ke faayde
  • Predictable CD — chalti sirf inputs/signals/events pe, "jab kahin kuch async hua" pe nahi.
  • Performance — no blanket over-checking; less wasted CD.
  • Smaller bundle — zone.js hata do (~KBs saved).
  • Cleaner stack traces — no monkey-patched async noise.

4. Internal Working (Engine ke andar kya hota hai?)

A scheduler notified by explicit triggers

Zoneless mode mein Zone.js hata diya jaata — koi async monkey-patching nahi. Iski jagah Angular ke paas ek internal ChangeDetectionScheduler hota jise explicit notifications milti hain jab CD zaroori: ek signal (jo template ya component mein use ho) badla, ek template event (click) fire hua, markForCheck() call hua (async pipe isko karti), ek @Input badla, ya host binding update hua. In notifications pe scheduler ek CD pass schedule karta (microtask timing pe coalesced). Key insight: CD ka mechanism same hai (Default/OnPush strategies, top-down traversal, dirty checking) — sirf trigger badla: Zone.js ke "kuch bhi async" se "explicitly notified changes" pe. Isliye signals zoneless ke saath ideal — woh scheduler ko precisely batate kya badla.

Diagram — zone vs zoneless trigger
 ZONE.JS:    any async (timer/http/event/mousemove) ─▶ tick() ─▶ full CD
                (over-triggers: fires even when nothing changed)

 ZONELESS:   signal change ─┐
             template event ─┼─▶ scheduler ─▶ CD pass (coalesced)
             async pipe emit ─┤     (fires only on EXPLICIT change signals)
             markForCheck    ─┘
 CD mechanism identical; only WHAT starts it differs.
