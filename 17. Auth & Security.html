S-01

JWT (JSON Web Tokens)

Covers: structure (header.payload.signature) · base64url · claims (iss/sub/aud/exp/iat) · signing · HS256 vs RS256 · stateless auth · NOT encrypted · verification

1. What is it? (Kya hai ye?)

JWT (JSON Web Token) ek compact, self-contained token hai jo do parties ke beech claims (statements about a user) securely transmit karta. Ye teen base64url-encoded parts ka dot-separated string hai: header.payload.signature. Login ke baad server ek JWT deta; client har request mein bhejta; server signature verify karke bharosa karta.

2. Why do we need it? (Zaroorat kyun hai?)

HTTP stateless hai — har request independent. Server ko har request pe "ye user kaun hai" pata hona chahiye. Purana tareeka: server-side session (server memory/DB mein session store, cookie mein session-id). JWT stateless alternative deta: user ki identity + claims token ke andar, cryptographically signed — server ko kuch store nahi karna, sirf signature verify karo.

3. What problem does it solve?

Stateful server sessions ki scaling problem. Sessions server memory/DB mein rehti — multiple servers pe sticky sessions ya shared store chahiye. JWT self-contained (identity token mein hi) + signed hai, toh koi bhi server bina shared session store ke verify kar sakta — horizontal scaling + microservices friendly.

4. Internal Working — the 3 parts

header . payload . signature

Header (base64url JSON): algorithm + type, jaise { "alg": "HS256", "typ": "JWT" }. Payload (base64url JSON): the claims — user data + metadata, jaise { "sub": "123", "email": "a@b.com", "roles": ["admin"], "exp": 1735689600 }. Signature: header+payload ko secret/private-key se sign karke banaya — HMACSHA256(base64url(header) + "." + base64url(payload), secret) (HS256 ke liye). Server jab token receive karta, woh header+payload se signature dobara compute karke received signature se match karta — match hua toh token untampered + trustworthy. Critical: payload sirf base64url-encoded hai, encrypted NAHI — koi bhi decode karke padh sakta. Signature tampering rokti (integrity), confidentiality nahi deti.

5. Execution Flow

 login(user, pass) ─▶ server validates ─▶ signs JWT ─▶ returns to client
 client stores token ─▶ each request: Authorization: Bearer <token>
 server: verify signature + exp ─▶ valid → extract claims → authorize
                                  invalid/expired → 401

6. Architecture — claims & algorithms

Registered claimsiss (issuer), sub (subject/user id), aud (audience), exp (expiry), iat (issued-at), nbf (not-before), jti (id).
Custom claimsApp-specific: roles, email, tenantId, etc.
HS256 (symmetric)One shared secret signs & verifies. Simple; secret must stay server-side.
RS256 (asymmetric)Private key signs, public key verifies. Better for distributed systems / microservices.

9. Advanced Example — HS256 vs RS256 choice

who can verify?
// HS256: same secret signs AND verifies.
//   → every service that verifies needs the SECRET (which can also SIGN).
//   → fine for a monolith; risky to share the signing secret widely.

// RS256: auth server holds PRIVATE key (signs). Others get PUBLIC key (verify only).
//   → microservices verify without the ability to mint tokens.
//   → public keys often published at a JWKS endpoint (/.well-known/jwks.json).
// Rule of thumb: distributed/multi-service → RS256.

10. Code Explanation — Line by Line

  1. JWT = 3 dot-separated base64url parts: header, payload, signature.
  2. Payload holds claims (user id, roles, exp) — readable, not secret.
  3. Signature = crypto over header+payload with secret/private-key.
  4. Server verifies by recomputing signature + checking exp.
  5. Client can decode for UX, but only the server's verification is authoritative.

11. Common Mistakes

Traps
  • JWT payload ko "secret" samajhna — ye readable hai (base64url, not encrypted). Sensitive data mat daalo.
  • Client-side decode pe auth-decisions lena — server MUST verify signature; client bypass ho sakta.
  • alg: none accept karna (server-side) — unsigned token accept = massive vuln.
  • Long-lived JWTs (no short exp) — stateless tokens easily revoke nahi hote; short expiry + refresh use karo.
  • HS256 secret ko multiple services mein share karna — koi bhi token mint kar sakta.

12. Best Practices

Do this
  • Short-lived access JWTs (minutes) + refresh tokens (S-03) — revocation window chhota.
  • Server pe hamesha verify: signature, exp, iss, aud.
  • Distributed systems → RS256 (public-key verify, private-key sign).
  • Minimal claims — sirf zaroori (small token, less exposure).
  • Client-side decode sirf UX ke liye (username dikhana), security ke liye nahi.

13. Performance

JWT ka perf win: stateless verification — server ko session lookup (DB/cache hit) nahi karna, bas signature verify (CPU-only, fast). Ye horizontal scaling + microservices mein bada fayda. Trade-off: token size (claims zyada → bade headers har request pe). RS256 verify HS256 se thoda mehenga (asymmetric), par public-key caching se manageable. Minimal claims rakho.

14. Edge Cases

RevocationStateless JWT easily revoke nahi hota — short expiry + refresh-token invalidation, ya blocklist.
Clock skewexp/nbf checks mein thodi tolerance (leeway) — server clocks slightly differ.
Key rotationkid (key id) header se multiple keys support; JWKS endpoint.
JWEAgar payload confidentiality chahiye → JWE (encrypted), na ki plain JWT.

15. Interview Questions

EASY JWT kya hai aur kis 3 parts se banta?

JSON Web Token — ek compact, self-contained, signed token jo user claims transmit karta. Teen base64url parts, dot-separated: header (alg + typ), payload (claims — user id, roles, exp), aur signature (header+payload ka crypto over secret/private-key). Server signature verify karke claims pe bharosa karta — stateless auth.

MEDIUM JWT encrypted hota hai? Payload mein password rakh sakte?

Nahi — standard JWT signed hota, encrypted nahi. Payload sirf base64url-encoded hai, koi bhi decode karke padh sakta (jwt.io pe paste karke). Signature tampering rokti (integrity), reading nahi (confidentiality). Isliye passwords/secrets/sensitive data payload mein kabhi mat daalo. Agar confidentiality chahiye toh JWE (encrypted variant) use karo.

MEDIUM HS256 vs RS256 — kab kaunsa?

HS256 symmetric — ek shared secret sign aur verify dono karta; jo verify kar sakta woh sign bhi kar sakta. Monolith ke liye theek. RS256 asymmetric — auth server private key se sign karta, baaki services public key se sirf verify karte (mint nahi kar sakte). Distributed systems/microservices mein RS256 better — signing capability centralized rehti, public keys (JWKS endpoint) se widely verify.

SCENARIO Frontend dev JWT decode karke roles check kar raha aur admin UI dikha raha. Backend pe bhi check hai. Kya ye sahi hai?

UI dikhane ke liye client-side decode theek hai (UX — admin button dikhana). Par ye security boundary nahi — client-side check bypass ho sakta (attacker JWT edit kar sakta ya API directly call kar sakta). Asli enforcement backend pe honi chahiye: har protected API signature verify kare aur roles/permissions check kare. Client-side check sirf UX; server-side check security. Dono chahiye, par kabhi client pe akele bharosa mat karo.

  • Claims: iss/sub/aud/exp/iat + custom (roles). Stateless: server verifies signature, stores nothing.
  • HS256 (shared secret) vs RS256 (private-sign/public-verify, for distributed). Short expiry + refresh. Server-side verify is authoritative.

S-02

OAuth 2.1 & OpenID Connect

Covers: OAuth framework · roles · Authorization Code + PKCE · access vs id token · OIDC · implicit/password grants removed · scopes · client credentials

1. What is it? (Kya hai ye?)

OAuth 2.0 (aur iska modern consolidation OAuth 2.1) ek authorization framework hai — ye define karta hai ki ek app kaise securely ek user ki taraf se kisi resource (API) tak access le. OpenID Connect (OIDC) OAuth ke upar ek thin authentication layer hai jo "user kaun hai" batata (via an id_token). Simple: OAuth = authorization (kya access), OIDC = authentication (kaun ho).

2. Why do we need it? (Zaroorat kyun hai?)

Users ko apna password har app ko dena assurance nahi (phishing, breach). OAuth delegated authorization deta: user identity provider (Google, Auth0, apna auth server) pe login karta, aur app ko sirf ek token milta — password kabhi app ke paas nahi jaata. "Login with Google" isi ka example. Standardized, secure, revocable.

3. What problem does it solve?

Password sharing + insecure custom auth. OAuth ek battle-tested standard deta jahan credentials ek trusted authorization server ke paas rehte, apps ko scoped, expiring tokens milte, aur user kabhi bhi access revoke kar sakta — bina password badle. Custom auth ke roll-your-own vulnerabilities se bachta.

4. Internal Working — the roles & the Authorization Code + PKCE flow

4 roles + the recommended flow

Roles: Resource Owner (user), Client (tumhari app), Authorization Server (issues tokens, e.g. Auth0/Keycloak/Google), Resource Server (the API). SPAs ke liye recommended flow = Authorization Code + PKCE: (1) app ek random code_verifier banata, uska SHA-256 hash code_challenge auth server ko bhejta; (2) user auth server pe login karta; (3) auth server ek short-lived, single-use authorization code app ko redirect karta; (4) app us code + original code_verifier ko token endpoint pe bhejta; (5) auth server verifier ka hash challenge se match karke tokens deta (access + optionally id + refresh). PKCE (RFC 7636) ensure karta ki code intercept karne wala bhi tokens na le sake (uske paas verifier nahi). OAuth 2.1 PKCE mandatory karta aur implicit flow (token in URL) hata deta.

Diagram — Authorization Code + PKCE
 app: verifier=random; challenge=SHA256(verifier)
   │  1. /authorize?...&code_challenge=challenge
   ▼
 [Authorization Server]  ← user logs in
   │  2. redirect back with ?code=ABC (short-lived, single-use)
   ▼
 app: 3. POST /token { code:ABC, code_verifier:verifier }
   ▼
 [Authorization Server] verifies SHA256(verifier)==challenge
   │  4. returns { access_token, id_token, refresh_token }
   ▼
 app uses access_token as: Authorization: Bearer <access_token>

5. Execution Flow — tokens you get back

 access_token  → sent to APIs (authorization: "what can I access")   [often a JWT]
 id_token      → OIDC: WHO the user is (name/email claims)           [always a JWT]
 refresh_token → get new access tokens without re-login (S-03)
 scope=openid  → triggers OIDC (you get an id_token)

6. Architecture — grant types (what to use)

Authorization Code + PKCEUSE for SPAs, mobile, web apps. The recommended flow for all user-facing clients.
Client CredentialsMachine-to-machine (no user) — server-to-server with a secret.
Device AuthorizationDevices without a good browser (TVs, CLIs).
Implicit❌ REMOVED in OAuth 2.1 — token in URL fragment, leaks. Never use for new apps.
Password (ROPC)❌ REMOVED in OAuth 2.1 — app handles user's raw password. Avoid.

9. Advanced Example — why PKCE, why not implicit

the security reasoning
// IMPLICIT (dead): /authorize returns #access_token=... IN THE URL
//   → token leaks via browser history, logs, Referer header. No PKCE possible.

// AUTH CODE + PKCE: returns a single-use CODE, exchanged server-side-style for tokens.
//   even if an attacker steals the code, without the code_verifier it's useless.
//   OAuth 2.1 mandates PKCE (S256) for ALL clients — public and confidential.

10. Code Explanation — flow recap

  1. App generates code_verifier + code_challenge (SHA-256).
  2. User logs in at the authorization server.
  3. Server returns a short-lived, single-use authorization code.
  4. App exchanges code + verifier → access/id/refresh tokens.
  5. PKCE proves same app initiated + completed — code theft useless.

11. Common Mistakes

Traps
  • Implicit flow use karna (purane tutorials) — OAuth 2.1 mein removed; token URL mein leak.
  • SPA mein client secret embed karna — JS bundle se extract ho sakta (public client, no secret).
  • PKCE skip karna — code interception/injection attacks.
  • OAuth (authorization) ko authentication samajhna — user identity ke liye OIDC (openid scope + id_token).
  • Redirect URI ko loose match rakhna — exact match honi chahiye (open redirect vuln).

12. Best Practices

Do this
  • SPAs → Authorization Code + PKCE (S256). Kabhi implicit nahi.
  • Vetted library ya BFF pattern (S-08) use karo — OAuth khud roll mat karo.
  • Exact redirect URI matching; har environment register.
  • Least-privilege scopes (orders.read, na ki full_access).
  • User identity chahiye → OIDC (openid scope, verify id_token).

13. Performance

OAuth ka overhead mostly ek-baar-ka login redirect + token exchange (network round-trips). Uske baad access token (usually JWT) se stateless API calls — fast. Refresh tokens (S-03) re-login avoid karte (better UX). Token caching (in-memory) redundant exchanges rokti. Perf se zyada yahan security primary concern hai; correct flow choose karna performance se important.

15. Interview Questions

EASY OAuth aur OIDC mein farak?

OAuth 2.x ek authorization framework hai — decide karta ki app kis resource (API) tak user ki taraf se access le sakta (access_token deta). OpenID Connect (OIDC) OAuth ke upar ek authentication layer — batata "user kaun hai" (id_token deta, user claims ke saath). openid scope OAuth request ko OIDC banata. Short: OAuth = kya access, OIDC = kaun ho.

MEDIUM SPA ke liye kaunsa OAuth flow aur kyun?

HARD Implicit flow kyun deprecated/removed hua?

Implicit flow access token ko directly URL fragment mein return karta tha (#access_token=...). Ye token browser history, server logs, aur Referer header mein leak ho sakta; sender-constrain nahi ho sakta; aur token endpoint bypass karne se PKCE se protect nahi ho sakta. Authorization Code + PKCE ye saari problems solve karta (code single-use, tokens URL mein nahi, PKCE protection) aur SPAs mein utna hi achha kaam karta — isliye OAuth 2.1 implicit ko poori tarah hata deta.

S-03

Refresh Tokens

Covers: access vs refresh token · short vs long lifetimes · rotation · reuse detection (token family) · inactivity expiry · secure storage · revocation

1. What is it? (Kya hai ye?)

Refresh token ek long-lived credential hai jiska ek hi kaam hai: naya access token lena jab purana expire ho jaaye — bina user ko dobara login karwaye. Access token short-lived (minutes) hota; refresh token long-lived (days/weeks). Jab access token expire, app refresh token ko auth server ko bhejta aur naya access token (aur usually naya refresh token) paata.

2. Why do we need it? (Zaroorat kyun hai?)

Do competing goals: security chahti short-lived access tokens (chori ho toh window chhota), par UX chahti user bar-bar login na kare. Refresh token ye balance deta — access token 5-15 min ka rakho (chota risk window), aur jab woh expire ho, refresh token se silently naya le lo (user ko pata bhi nahi chalta).

3. What problem does it solve?

Security-vs-UX tension. Sirf long-lived access tokens = chori pe bada damage. Sirf short-lived (no refresh) = user har 10 min login kare (terrible UX). Refresh token: access token short (security), session long (UX via silent renewal). Plus refresh tokens revoke karke ek session end kar sakte (better control).

4. Internal Working — rotation & reuse detection

Rotation: har refresh pe naya refresh token

Modern secure setups mein refresh token rotation hoti: jab tum refresh token exchange karte ho, auth server ek naya access token aur ek naya refresh token deta, aur purana refresh token invalidate kar deta (single-use). Isse agar refresh token chori ho, uski life bahut chhoti. Reuse detection: agar ek already-rotated (purana) refresh token dobara use hota — matlab kisi ne chori kiya aur original + attacker dono use kar rahe — auth server poore "token family" ko invalidate kar deta (dono logged out), theft detect karke. Plus inactivity expiry (jaise 7 din use na ho toh invalid) absolute expiry se alag. Ye layered defense refresh tokens ko zyada safe banata.

Diagram — rotation & reuse detection
 login ─▶ access(5m) + refresh_A
 access expires ─▶ POST refresh_A ─▶ access' + refresh_B  (refresh_A now INVALID)
 access' expires ─▶ POST refresh_B ─▶ access'' + refresh_C (refresh_B now INVALID)

 ⚠ attacker replays refresh_A (already rotated) ─▶ server sees REUSE
    ─▶ invalidate the WHOLE family (A,B,C...) ─▶ everyone re-authenticates

5. Execution Flow

 request with access token ─▶ 401 (expired)
   ─▶ app: POST /token { grant_type: refresh_token, refresh_token }
   ─▶ new access (+ new refresh via rotation)
   ─▶ retry original request with new access ─▶ success (user unaware)

6. Architecture — lifetimes & storage

Access tokenShort-lived (5–15 min). Sent to APIs. In-memory (JS variable) ideally.
Refresh tokenLong-lived (days). Only to the auth server's token endpoint. HttpOnly cookie / BFF ideally.
RotationEach use issues a new refresh token; old invalidated.
Reuse detectionOld (rotated) token reused → invalidate whole family.
RevocationServer can revoke a refresh token to end a session.

7. Real Project Example — the refresh call

auth.service.ts — refresh method
@Injectable({ providedIn: 'root' })
export class AuthService {
  private http = inject(HttpClient);
  private accessToken: string | null = null;   // in-memory (safer than localStorage)

  refresh(): Observable<string> {
    // refresh token ideally sent via HttpOnly cookie (withCredentials), not body
    return this.http.post<{ accessToken: string }>(
      '/auth/refresh', {}, { withCredentials: true }   // cookie carries refresh token
    ).pipe(
      tap(res => this.accessToken = res.accessToken),  // store new access in memory
      map(res => res.accessToken),
    );
  }
  getAccessToken() { return this.accessToken; }
}

11. Common Mistakes

Traps
  • Refresh token ko localStorage mein rakhna — XSS se chori. HttpOnly cookie / BFF prefer.
  • Long-lived access token (refresh ki zaroorat na pade) — chori pe bada damage window.
  • Rotation ke saath concurrent refreshes fire karna → user unexpectedly logged out (S-07).
  • Rotation/reuse-detection na karna — chori hua refresh token indefinitely valid.
  • Refresh token ko regular API calls mein bhejna — sirf auth server ke token endpoint pe.

12. Best Practices

Do this
  • Access short (5-15 min, in-memory); refresh long (HttpOnly/Secure/SameSite cookie ya BFF).
  • Refresh token rotation + reuse detection enable karo.
  • Inactivity expiry (unused N days → invalid) + absolute expiry.

  • Logout pe refresh token server-side revoke karo.

Refresh flow user ke liye seamless — background renewal, no re-login (great UX). Perf cost minimal (occasional token-endpoint call). Short access tokens API-side stateless verification rakhte (no per-request DB). Rotation slight server-side state (track current family), par security worth it. 15. Interview Questions

EASY Access token aur refresh token mein farak?

Access token short-lived (minutes) hota aur har API request mein bheja jaata (authorization). Refresh token long-lived (days) hota aur sirf ek kaam ke liye — naya access token lena jab purana expire ho, bina re-login. Ye security (short access = chota chori-window) aur UX (long session, silent renewal) balance karta.

HARD Refresh token rotation aur reuse detection kya hai?

Rotation: har refresh exchange pe auth server naya access aur naya refresh token deta, purana refresh invalidate (single-use). Chori hua token ki life chhoti. Reuse detection: agar koi already-rotated (purana) refresh token dobara use kare — indicating theft (original + attacker dono use kar rahe) — server poore token family ko invalidate kar deta, sabko re-auth pe bhejta. Ye stolen-token damage drastically limit karta.

MEDIUM Refresh token kahan store karna chahiye?

Ideally HttpOnly, Secure, SameSite cookie mein (JS access nahi kar sakta → XSS-safe), ya BFF pattern mein server-side session mein (browser ko refresh token milta hi nahi). localStorage/sessionStorage avoid — koi bhi same-origin JS (XSS) padh sakta. Access token in-memory (JS variable) theek hai kyunki short-lived, par refresh token (long-lived, powerful) ko sabse zyada protect karo.

SCENARIO Users randomly logged out ho rahe. Logs mein "refresh token reuse" errors. Kya ho raha?

Probably concurrent refresh: jab access token expire hota aur multiple requests ek saath 401 paate, har ek refresh token se refresh try karta. Rotation ke saath pehla success (token rotate ho gaya), baaki purane (ab-invalid) token se refresh karte → reuse-detection trigger → family invalidate → logout. Fix: single shared in-flight refresh (mutex) — ek hi refresh chale, baaki uska wait karein aur naye token se retry karein (S-07). Multi-tab ho toh BFF/centralized refresh best.

⚡ 30-second recall

  • Rotation: each refresh → new refresh token, old invalidated. Reuse of old → invalidate whole family (theft defense).
  • Concurrent refresh + rotation = accidental logout → use single shared refresh (S-07). Revoke on logout.

HTTP Interceptors

Covers: HttpInterceptorFn · attach Bearer token · withInterceptors · provideHttpClient · order matters · immutable clone · 401 handling · functional vs class

1. What is it? (Kya hai ye?)

HTTP interceptor ek function hai jo har outgoing HTTP request (aur incoming response) ke beech mein baithta hai — use inspect ya transform kar sakta before it goes out / comes back. Auth ke context mein iska primary kaam: har request mein automatically Authorization: Bearer <token> header attach karna, taaki har service mein manually na karna pade.

2. Why do we need it? (Zaroorat kyun hai?)

Har API call mein manually auth header lagana repetitive + error-prone (ek jagah bhool gaye = 401). Interceptor ek central chokepoint deta — ek jagah token attach karo, poori app ke har request pe apply. Same jagah errors (401/403) bhi centrally handle kar sakte. DRY + consistent.

3. What problem does it solve?

Cross-cutting HTTP concerns (auth headers, error handling, logging, retries) ko har call mein duplicate karna. Interceptor inhe ek reusable pipeline stage mein centralize karta — auth token har request pe, error handling har response pe, bina har service ko chhue.

4. Internal Working (Engine ke andar kya hota hai?)

A chain; requests are immutable

Modern Angular mein interceptors functional hain: HttpInterceptorFn — ek function jo (req, next) leta. Tum provideHttpClient(withInterceptors([a, b, c])) se register karte, aur woh ek chain banate — request a → b → c → backend se guzarti, response ulta wapas. HttpRequest immutable hai — header add karne ke liye req.clone({ setHeaders }) se naya request banao, phir next(cloned) call karo. next chain ka agla handler hai (ya backend). Order matters: auth interceptor error interceptor se pehle (token attach ho, phir error handle). inject() function interceptor ke andar services access karne deta.

Diagram — the interceptor chain
 request ─▶ authInterceptor ─▶ errorInterceptor ─▶ [Backend]
              (add Bearer)        (catch 401/403)        │
 response ◀── errorInterceptor ◀── (handle errors) ◀─────┘
 req.clone({ setHeaders }) → immutable; next(req) → pass along the chain

5. Execution Flow

 http.get('/api/x') ─▶ authInterceptor: clone + add Authorization header
   ─▶ next(cloned) ─▶ errorInterceptor: next().pipe(catchError(...))
   ─▶ backend ─▶ response flows back up the chain ─▶ component

6. Architecture — the API

provideHttpClient(withInterceptors([...]))Register functional interceptors (order = array order).
req.clone({ setHeaders })Immutable header add (never mutate req).
inject(Service)Access DI inside the interceptor function.
Class-based HttpInterceptorLegacy — functional is recommended for new code.

7. Real Project Example (Angular context)

auth.interceptor.ts — attach the Bearer token
import { HttpInterceptorFn } from '@angular/common/http';
import { inject } from '@angular/core';
import { AuthService } from './auth.service';

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const auth = inject(AuthService);
  const token = auth.getAccessToken();

  // don't attach to auth endpoints / external URLs
  if (!token || req.url.includes('/auth/')) return next(req);

  const authReq = req.clone({                         // immutable clone
    setHeaders: { Authorization: `Bearer ${token}` },
  });
  return next(authReq);
};

// app.config.ts
provideHttpClient(withInterceptors([authInterceptor, errorInterceptor]));
//                                   ↑ order: auth first, then error handling

9. Advanced Example — an error interceptor (401 → logout)

error.interceptor.ts
export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  const auth = inject(AuthService);
  const router = inject(Router);
  return next(req).pipe(
    catchError((err: HttpErrorResponse) => {
      if (err.status === 401) { auth.logout(); router.navigate(['/login']); }
      if (err.status === 403) { router.navigate(['/forbidden']); }
      return throwError(() => err);
    })
  );
};
// (a smarter 401 handler that REFRESHES instead of logging out → S-07)

  4. next(authReq) — modified request ko chain mein aage bhejo.
  5. Register: withInterceptors([auth, error]) — order = execution order.

11. Common Mistakes

Traps
  • req ko directly mutate karna (req.headers.set) — immutable; clone use karo.
  • Token ko auth/login endpoints pe bhi attach karna — infinite loops / wrong behavior. Skip logic rakho.
  • Interceptor order galat — auth error ke baad (token attach hi nahi hoga before send in some flows).
  • next(req) return na karna — request hang.
  • External/3rd-party URLs pe apna Bearer token bhejना — token leak. URL check karo.

12. Best Practices

Do this

  • Always req.clone(...) — never mutate.
  • Token sirf apne API origins pe attach (URL allowlist); auth endpoints skip.
  • Order: auth → error (→ refresh logic in the error/auth interceptor, S-07).
  • Ek interceptor = ek responsibility (auth, error, logging alag).

13. Performance

Interceptors lightweight — per-request function call. Token attach O(1). Perf ka dhyaan: heavy sync work interceptor mein har request pe chalega (avoid). Retry/refresh logic mein infinite loops se bacho (guard flags). Central error handling redundant per-call error code hatakar cleaner + consistent. Functional interceptors better tree-shaking dete class-based se.

Route Guards

Covers: CanActivateFn · CanMatchFn · inject() · functional guards · redirect to login · returnUrl · UrlTree · lazy-load protection · CanDeactivate

1. What is it? (Kya hai ye?)

Route guards functions hain jo decide karte hain ki koi navigation allow ho ya nahi. Auth ke context mein: "kya ye user is protected route pe ja sakta?" Agar logged-in nahi → login pe redirect. Modern Angular mein ye functional guards hain — CanActivateFn, CanMatchFn — jo inject() se services access karte.

2. Why do we need it? (Zaroorat kyun hai?)

Kuch routes sirf authenticated (ya specific-role) users ke liye hone chahiye — dashboard, admin panel, profile. Bina guard, koi bhi URL type karke protected page pe pahunch sakta (client-side). Guard navigation ko intercept karke unauthorized access rokta aur user ko login/forbidden pe bhejta.

3. What problem does it solve?

Unprotected client-side routes. Guard ek declarative "is route ke liye ye condition honi chahiye" deta — auth check ek jagah, route config mein clean. (Yaad rahe: client-side guard sirf UX/navigation control hai; asli data protection backend pe.)

4. Internal Working (Engine ke andar kya hota hai?)

Functional guards + return types

Ek functional guard ek function hai jo true (allow), false (block), ya ek UrlTree (redirect) return karta — ya inka Observable/Promise (async checks ke liye). Router navigation se pehle guard chalata: CanActivateFn route activate hone se pehle, CanMatchFn route match (aur lazy chunk load) hone se pehle. inject() function guard ke andar DI deta (AuthService, Router) — no constructor. Redirect ke liye router.createUrlTree(['/login'], { queryParams: { returnUrl } }) ek UrlTree return karo — router us pe navigate karega. Class-based guards ab bhi kaam karte par functional recommended (less boilerplate, better tree-shaking, cleaner DI).

                 ─▶ router redirects to /login
 CanMatchFn: runs BEFORE lazy chunk loads → blocks download too

import { inject } from '@angular/core';
import { CanActivateFn, Router } from '@angular/router';
import { AuthService } from './auth.service';

export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.isLoggedIn()) return true;

  // redirect to login, remember where the user wanted to go
  return router.createUrlTree(['/login'], {
    queryParams: { returnUrl: state.url },
  });
};

// routes
export const routes: Routes = [
  { path: 'dashboard', canActivate: [authGuard], loadComponent: () => import('./dashboard') },
  { path: 'admin', canMatch: [authGuard], loadChildren: () => import('./admin.routes') }, // blocks chunk
];

RBAC (Role-Based Access Control)

Covers: roles vs permissions · role guards (parameterized) · UI-level control (directives) · least privilege · roles in JWT claims · server-side enforcement · RBAC vs ABAC

1. What is it? (Kya hai ye?)

RBAC (Role-Based Access Control) authorization ka ek model hai jahan users ko roles assign hoti hain (admin, editor, viewer), aur roles ke saath permissions jude hote (kya kar sakte). "Ye user kya kar sakta" ka jawab role se aata. Authentication (kaun ho) ke baad ka step: authorization (kya karne ki ijaazat).

2. Why do we need it? (Zaroorat kyun hai?)

Sab authenticated users equal nahi — ek admin sab kar sakta, ek viewer sirf padh sakta. Har user ke liye individual permissions manage karna unmanageable. Roles ek abstraction dete: permissions ko roles mein group karo, users ko roles do — scalable, manageable authorization.

3. What problem does it solve?

Fine-grained, scalable authorization. Individual-user permissions chaos ban jaate. RBAC roles ke through structure deta — ek jagah role→permissions define, users ko roles assign, aur routes/UI/APIs ko role/permission se guard. Least-privilege enforce karna aasan.

4. Internal Working — roles in claims, checked everywhere

Roles flow: token → guard/UI → API

Login ke baad user ke roles usually JWT claims mein aate ({ "roles": ["admin"], "permissions": ["orders.write"] }). Angular-side, ek AuthService in roles ko expose karta. Phir teen jagah check hota: (1) Route guards — parameterized roleGuard('admin') jo route data ya argument se required role check kare; (2) UI-level — ek structural directive (*hasRole) ya signal-based check jo buttons/sections role ke hisaab se dikhaye/chhupaaye; (3) sabse important — server-side, jahan har protected API endpoint role/permission verify kare. Client-side checks sirf UX (irrelevant options chhupana) — bypass ho sakte, isliye enforcement backend pe. Roles vs permissions: role = collection of permissions; fine-grained checks permissions pe (orders.write), coarse checks roles pe (admin).

S-07

Token Refresh Flow

Covers: interceptor catches 401 · call refresh · retry original request · concurrent-request queuing · the single-flight mutex · shareReplay · avoiding refresh storms · logout on failure

1. What is it? (Kya hai ye?)

Token refresh flow woh production-grade mechanism hai jo access token expire hone pe automatically naya le aata aur failed request ko retry karta — user ko kuch pata nahi chalta. Angular mein ye usually ek interceptor mein implement hota: 401 catch karo → refresh karo → original request naye token ke saath retry karo. Asli challenge: jab multiple requests ek saath 401 paate.

2. Why do we need it? (Zaroorat kyun hai?)

Access token short-lived (S-03) — beech kaam mein expire ho sakta. Bina auto-refresh, user ko random 401s ya forced logout milega (bad UX). Refresh flow seamlessly session continue karta. Par naive implementation (har 401 pe alag refresh) rotation ke saath toot jaata (S-03) — isliye ek carefully-designed single-flight flow chahiye.

3. What problem does it solve?

Do problems: (1) expired-token requests ka silent recovery; (2) concurrent refresh storm — jab access expire hote hi 5 requests ek saath 401 paate, sabhi refresh trigger kar dete → 5 parallel refreshes → rotation invalidate + duplicate work + possible logout. Proper flow ek hi refresh chalata; baaki uska wait karte, phir naye token se retry.

4. Internal Working — the single-flight refresh

One refresh in flight; others queue

Core idea: ek shared state (jaise ek BehaviorSubject<string|null> for the current/new token, plus an isRefreshing flag). Jab ek request 401 paata: (1) agar koi refresh already chal raha, ye request us pending refresh ko wait karta (subject ke naye-token emit hone tak), phir naye token se retry; (2) agar koi refresh nahi chal raha, ye request refresh start karta (isRefreshing=true, subject ko null), refresh Observable ko shareReplay/shared rakhta taaki concurrent callers same result paayein, aur success pe naya token subject pe emit karke sab waiting requests ko unblock karta. Refresh fail hone pe → logout + login redirect. Ye "single-flight" (ek hi refresh at a time) pattern rotation-safe hai — parallel refreshes nahi hote.

Diagram — concurrent 401 handling
 t0: access token expires
 req A ─401─▶ no refresh running → START refresh (isRefreshing=true, token$=null)
 req B ─401─▶ refresh already running → WAIT on token$
 req C ─401─▶ refresh already running → WAIT on token$
        refresh completes → token$.next(newToken), isRefreshing=false
 req A,B,C ─▶ all retry with newToken ─▶ success
 (only ONE refresh call happened → rotation-safe, no accidental logout)

7. Real Project Example (Angular context)

refresh.interceptor.ts — single-flight (simplified)
let isRefreshing = false;
const token$ = new BehaviorSubject<string | null>(null);

export const refreshInterceptor: HttpInterceptorFn = (req, next) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  return next(req).pipe(
    catchError((err: HttpErrorResponse) => {
      // only handle 401s, and never for the refresh call itself
      if (err.status !== 401 || req.url.includes('/auth/refresh')) {
        return throwError(() => err);
      }
      if (isRefreshing) {
        // a refresh is already running → wait for the new token, then retry
        return token$.pipe(
          filter((t): t is string => t !== null),
          take(1),
          switchMap(t => next(addToken(req, t))),
        );
      }
      // start a single refresh
      isRefreshing = true;
      token$.next(null);
      return auth.refresh().pipe(
        switchMap(newToken => {
          isRefreshing = false;
          token$.next(newToken);                 // unblock waiting requests
          return next(addToken(req, newToken));   // retry this one
        }),
        catchError(e => {
          isRefreshing = false;
          auth.logout(); router.navigate(['/login']);
          return throwError(() => e);
        }),
      );
    }),
  );
};

function addToken(req: HttpRequest<unknown>, token: string) {
  return req.clone({ setHeaders: { Authorization: `Bearer ${token}` } });
}

11. Common Mistakes

Traps
  • Naive per-401 refresh → parallel refreshes → rotation breaks → logout (the classic bug).
  • Refresh endpoint ke 401 ko refresh karne ki koshish → infinite loop. Skip karo.
  • isRefreshing flag reset na karna on error → subsequent requests permanently stuck waiting.
  • Waiting requests ko naye token se retry karna bhoolna (purane token se retry → phir 401).
  • Refresh fail pe logout na karna → user broken state mein.

12. Best Practices

Do this
  • Single-flight refresh — ek refresh at a time; baaki wait + retry with new token.
  • Refresh endpoint ke requests ko skip karo (loop-proof).
  • Refresh fail → logout + redirect; flag hamesha reset (success aur error dono).
  • Proactive pre-expiry refresh + reactive 401-catch dono (smoothest UX).
  • Multi-tab → BFF/centralized refresh (cross-tab rotation issues avoid).

13. Performance

Single-flight ka bada win: N concurrent 401s pe sirf ek refresh call (N nahi) — network + server load kam, aur rotation-safe. Proactive refresh 401 round-trips hi avoid karta (requests pehli baar mein succeed). shareReplay/shared subject waiting requests ko efficiently unblock karta. Refresh storm (dozens of parallel refreshes) rokna server-side + client-side dono pe healthy.

Security Best Practices

Covers: XSS & Angular sanitization · CSRF/XSRF & HttpClient protection · token storage (in-memory / httpOnly / BFF) · HTTPS · CSP · least privilege · dependency hygiene

1. What is it? (Kya hai ye?)

Security best practices woh cross-cutting defenses hain jo poori app ko common web attacks se bachati hain — mukhya taur pe XSS (malicious script injection), CSRF (forged requests), aur token theft (insecure storage). Ye ek "feature" nahi, ek discipline hai jo har layer pe apply hoti.

2. Why do we need it? (Zaroorat kyun hai?)

Ek achhi auth flow bhi bekaar hai agar attacker XSS se token chura le, ya CSRF se user ki taraf se actions kar de, ya token localStorage se leak ho jaaye. Security best practices ye ensure karti ki tumhara auth infrastructure actually secure ho — sirf "kaam kare" nahi, "safely kaam kare".

3. What problem does it solve?

Real-world attack vectors. XSS (script injection → token theft/actions), CSRF (cross-site forged requests), token leakage (insecure storage), man-in-the-middle (no HTTPS). Best practices in vectors ko systematically close karti — defense in depth.

4. Internal Working — the big three & Angular's defenses

XSS, CSRF, token storage

XSS (Cross-Site Scripting): attacker malicious JS inject karta (jaise ek comment field mein <script>) jo victim ke browser mein chalta — tokens chura sakta, actions kar sakta. Angular ki defense: by default Angular saara interpolated content ko contextually auto-sanitize/escape karta — {{ userInput }} safe render hota (script execute nahi hota). Danger tab jab tum [innerHTML] ya bypassSecurityTrust* se ise bypass karte ho with untrusted data.

CSRF/XSRF (Cross-Site Request Forgery): attacker site victim ke browser se tumhare API pe forged request bhejwaata (cookies auto-attach hote). Angular's HttpClient defense: ek XSRF-TOKEN cookie padhta aur X-XSRF-TOKEN header mein bhejta — server dono match karke verify karta (double-submit). Cross-site attacker cookie padh nahi sakta (same-origin), toh header set nahi kar sakta.

Token storage: localStorage/sessionStorage XSS se readable — tokens wahan mat rakho. Access token in-memory; refresh token HttpOnly cookie / BFF (S-03).
