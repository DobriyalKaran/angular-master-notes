S-01

Performance Mindset & Measurement

Covers: Core Web Vitals (LCP/INP/CLS) · measure-first discipline · Angular DevTools profiler · Chrome DevTools · Lighthouse · performance budgets · finding the real bottleneck

1. What is it? (Kya hai ye?)

Performance optimization ki foundation ek mindset hai: guess mat karo, measure karo. Pehle metrics define karo (kya "fast" ka matlab), phir profile karke asli bottleneck dhoondho, phir usi ko fix karo, phir dobara measure karke verify. Baaki poora chapter techniques deta — par ye section decide karta ki kaunsi technique kab lagani hai.

2. Why do we need it? (Zaroorat kyun hai?)

Bina measurement ke optimization = guesswork. Log aksar galat cheez optimize karte (jo slow lagti, na ki jo actually slow hai) — effort waste, complexity add. "Premature optimization is the root of all evil." Measure-first ensure karta ki tum bottleneck pe kaam karo, aur prove kar sako ki fix ne actually help kiya.

3. What problem does it solve?

Blind optimization. Data ke bina tum galat area optimize karte, ya micro-optimizations pe time waste karte jinka user-facing impact zero. Measurement objective priorities deta: sabse bada user-facing slowdown pehle, aur har change ka verified impact.

4. Internal Working — Core Web Vitals

The three user-centric metrics

Google ke Core Web Vitals real user-experience measure karte: (1) LCP (Largest Contentful Paint) — sabse bada content element (hero image/heading) kitni jaldi visible; loading speed. Target < 2.5s. (2) INP (Interaction to Next Paint) — user interaction (click/type) se next visual update tak ka delay; responsiveness. Target < 200ms. (INP ne 2024 mein FID ko replace kiya.) (3) CLS (Cumulative Layout Shift) — page load ke dauraan content kitna "jump" karta; visual stability. Target < 0.1. Angular perf work in teeno ko target karta: LCP (fast load — lazy/SSR/images), INP (responsive — change detection/zoneless), CLS (stable — image dimensions/skeleton). Baaki metrics: TTI (time to interactive), FCP (first contentful paint), TBT (total blocking time).

10. Code Explanation — the discipline

  1. Define "fast" via metrics (Core Web Vitals + app-specific).
  2. Measure with the right tool (Lighthouse overall, DevTools for detail).
  3. Identify the single biggest bottleneck (data, not intuition).
  4. Apply the matching technique (rest of this chapter).
  5. Re-measure → verify the improvement → repeat.

11. Common Mistakes

Traps
  • Optimize without measuring — galat area fix, real bottleneck untouched.
  • Sirf lab data (Lighthouse) pe bharosa — field data (real users) alag ho sakta.
  • Micro-optimizations pe time (jinka user-facing impact ~0) jab bada issue elsewhere.
  • Fast machine pe test karna — low-end devices/slow networks pe throttle karke test karo.
  • Fix karke re-measure na karna — "improvement" assume karna without proof.

12. Best Practices

Do this
  • Measure-first, always. Bottleneck data se find karo, guess se nahi.
  • Core Web Vitals target karo (LCP/INP/CLS) — user-centric metrics.
  • Lab (Lighthouse) + field (web-vitals/CrUX) dono track karo.
  • CPU/network throttle karke low-end conditions simulate.
  • Budgets set karo (angular.json) — regressions CI mein catch.

S-02

Change Detection Optimization

Covers: OnPush · signals · zoneless · no function calls in templates · pure pipes · memoization · runOutsideAngular · detach · immutability

1. What is it? (Kya hai ye?)

Change detection (CD) optimization ka matlab: Angular kitni baar aur kitne components ke bindings check karta, use minimize karna. Kyunki CD har interaction/async pe chalti (Ch16), unnecessary/expensive CD sabse common runtime slowdown hai. Techniques: OnPush, signals, zoneless, pure expressions, aur zone-escaping.

2. Why do we need it? (Zaroorat kyun hai?)

Default CD har tick pe poore tree ko check karti (Ch16 S-04). Bade apps + frequent triggers (zone.js) mein ye INP (responsiveness) ka #1 killer ban jaata — main thread block, janky interactions. CD optimize karke tum fewer + cheaper checks karte, UI snappy rehta.

3. What problem does it solve?

Over-checking aur expensive re-evaluation. OnPush views skip karta; signals targeted updates dete; zoneless blanket-async ticks hatata; pure expressions redundant recompute rokte; runOutsideAngular high-frequency async ko CD se bahar rakhta. Sab milkar CD cost (frequency × components × work) minimize.

4. Internal Working — the CD cost levers

Reduce: frequency, scope, per-check work

CD cost = frequency (kitni baar tick) × scope (kitne views checked) × per-check work (bindings + expressions). Har lever attack karo: Scope — OnPush se non-dirty subtrees skip (Ch16 S-05); Frequency — zoneless (Ch16 S-08) se sirf real changes pe tick, blanket-async nahi; aur runOutsideAngular se high-frequency async (rAF, mousemove) ko zone ke bahar (no ticks); Per-check work — template mein {{ fn() }} function calls avoid (har tick re-run) — pure pipes ya computed signals se memoize. Signals teeno improve karte: fine-grained (sirf affected consumers update), zoneless-native, aur computed memoized. Modern approach: signals + OnPush + (eventually) zoneless.

5. Execution Flow — signals + OnPush

6. Architecture — the techniques

SignalsFine-grained reactivity; computed memoizes; OnPush/zoneless-friendly.
ZonelessCD only on explicit changes; no blanket async ticks (Ch16 S-08).
No template fn callsReplace {{ fn() }} with computed / pure pipe / precomputed value.

runOutsideAngular / detachEscape CD for high-frequency work (Ch16 S-03/S-07).

7. Real Project Example — replacing function calls

the #1 template perf fix
// ❌ runs on EVERY change detection tick:
// template: {{ getFullName() }}  {{ items.filter(i => i.active).length }}

// ✅ computed signal — memoized, recomputes only when deps change:
fullName = computed(() => `${this.first()} ${this.last()}`);
activeCount = computed(() => this.items().filter(i => i.active).length);
// template: {{ fullName() }}  {{ activeCount() }}

// ✅ or a pure pipe (memoized by input):
// template: {{ items | activeCount }}

8. Beginner Example — OnPush + signals

the modern default component
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,   // skip unless triggered
  template: `{{ count() }} — {{ double() }}`,
})
export class CounterComponent {
  count = signal(0);
  double = computed(() => this.count() * 2);   // memoized
  inc() { this.count.update(n => n + 1); }      // fine-grained update
}

9. Advanced Example — runOutsideAngular for hot paths

escaping CD for 60fps work
private zone = inject(NgZone);
startAnimation() {
  this.zone.runOutsideAngular(() => {          // no CD per frame
    const loop = () => { this.render(); requestAnimationFrame(loop); };
    loop();
  });
}
onComplete(result: string) {
  this.zone.run(() => this.status.set(result));  // re-enter → one CD
}
// 60 frames/sec without 60 CD passes/sec.

10. Code Explanation — Line by Line

  1. OnPush — view har tick check nahi hoti, sirf trigger pe.
  2. Signals — count.set sirf affected views mark karta (targeted).
  3. computed — derived value memoized (deps na badle toh cached).
  4. Template fn calls → computed/pure pipe (no per-tick re-run).
  5. runOutsideAngular — high-frequency async CD-free; run se re-enter.

11. Common Mistakes

Traps
  • Template mein function/getter calls — har tick re-run (sabse common CD perf bug).
  • OnPush ke saath mutable objects — reference same → stale UI (Ch16 S-05).
  • Impure pipe use karna jahan pure kaafi — pure memoizes, impure har tick.
  • High-frequency async zone ke andar — har event full CD.
  • OnPush blindly sab jagah bina measure — complexity without proven benefit (S-01).

12. Best Practices

Do this
  • Signals-first + OnPush naye components mein — fine-grained + skippable.
  • Template expressions cheap + pure; derived values → computed/pure pipes.
  • High-frequency async → runOutsideAngular; enterprise → zoneless (Ch16 S-08).
  • Immutable updates (naya reference) OnPush ke saath.
  • Measure (S-01) — CD optimization jahan profiler hotspot dikhaye.

13. Performance

CD optimization INP (responsiveness) ka biggest lever. OnPush bade trees mein checked-views drastically kam karta. Signals fine-grained — sirf badla widget update. Zoneless blanket over-triggering hatata. Template function-call elimination + pure pipes redundant compute rokte. runOutsideAngular 60fps work ko main-thread-friendly rakhta. Combined: janky → smooth interactions. (Deep dive: Chapter 16.)

EASY CD optimize karne ke main tareeke?

MEDIUM Template mein {{ getData() }} kyun perf problem hai aur fix?

Har CD tick pe woh function re-execute hota (Angular result cache nahi karta). Default CD + zone.js mein har mousemove/timer/click → tick → function phir chalti — heavy function pe massive waste. Fix: value ko computed signal banao (memoized — deps na badle toh cached), ya pure pipe use karo (input se memoized), ya change pe precompute karke ek property mein rakho. Signals fine-grained reactivity dete — jab ek signal badalta, sirf woh views/computeds update hote jo use padhte hain, na ki poora tree. computed automatically memoize karta. Signals OnPush views ko auto-mark karte aur zoneless ke saperfect fit hain (scheduler ko precisely batate kya badla). Yani signals CD ke teeno levers — frequency, scope, per-check work — improve karte.

SCENARIO Ek canvas-heavy dashboard 60fps animate kar raha par poora app janky ho gaya. CD-side fix?

Animation loop (rAF, 60x/sec) probably zone ke andar chal raha — har frame full CD trigger. Fix: loop ko ngZone.runOutsideAngular(...) mein rakho (zone bypass, no CD per frame). Jab actually UI status update karni ho, ngZone.run(() => signal.set(...)) se re-enter karke ek CD. Ya canvas component ko detach() karke throttled detectChanges() (Ch16 S-07). Isse 60fps rendering ke saath main thread CD-free rehta.

S-03

Efficient List Rendering (track)

Covers: @for track · trackBy · DOM node reuse · stable identity · why re-creation is slow · track by id vs index · list diffing

1. What is it? (Kya hai ye?)

@for (aur purana *ngFor) lists render karta. track Angular ko batata ki kaunsi property se ek item ko uniquely identify kare (usually item.id). Isse jab list badalti, Angular purane DOM nodes ko reuse kar sakta un items ke liye jo same hain — sab kuch destroy+recreate karne ke bajaye. Modern Angular mein @for ke saath track mandatory hai.

2. Why do we need it? (Zaroorat kyun hai?)

Bina track, Angular list badalne pe items ko identify nahi kar pata — toh kaafi cases mein poore list ke DOM nodes destroy karke naye banata (expensive: DOM ops + component re-init + lost state jaise input focus/scroll). track se Angular "ye wahi item hai" samajhkar nodes reuse karta — bahut fewer DOM operations.

3. What problem does it solve?

Unnecessary DOM re-creation. Bade/frequently-updating lists mein (tables, feeds), bina track har update poore DOM ko re-render karwa sakta — slow + janky + lost UI state. track stable identity deta, toh sirf actually-changed items ke nodes touch hote — minimal DOM work.

4. Internal Working — identity-based diffing

track = "kaunsa item wahi item hai"

Jab list array badalti, Angular purani aur nayi list ko diff karta. track expression (jaise track item.id) har item ki identity key deta. Angular keys ko compare karta: jo key dono lists mein hai → uska DOM node reuse (bas data update); jo nayi hai → naya node create; jo hat gayi → node remove. Bina track (ya track $index jab items reorder hote), Angular identity galat samajh sakta — jaise index-based tracking mein agar tum list ke start pe item insert karo, saare indices shift ho jaate, toh Angular sochta har item badal gaya → sab re-render. track item.id (stable unique id) sabse reliable hai. Reuse se DOM ops, component re-init, aur UI state (focus/scroll/animations) sab preserve hote.

Diagram — track reuses nodes
 old: [A, B, C]   new: [A, X, B, C]   (X inserted)

 track item.id:  A→reuse, B→reuse, C→reuse, X→create   (1 DOM insert)
 no track / bad: possibly destroy B,C + recreate as X,B,C  (many DOM ops)

 stable id ─▶ Angular knows A,B,C are the SAME items ─▶ reuse their nodes

5. Execution Flow

 list changes ─▶ Angular diffs old vs new by track key
   key exists in both ─▶ REUSE node, update bindings
   key is new         ─▶ CREATE node
   key removed        ─▶ DESTROY node
 result: minimal DOM mutations (only real changes)

6. Architecture — the syntax

@for (item of items; track item.id)Modern control flow; track is REQUIRED.
*ngFor="let i of items; trackBy: fn"Legacy; trackBy function returns the key.
track item.idBest — stable unique identity.
track $indexOK only for static/append-only lists (bad if items reorder/insert).
Reuse benefitsFewer DOM ops, preserved component state, smoother animations.

11. Common Mistakes

Traps
  • track $index reorder/insert wali lists mein — spurious re-renders + lost state.
  • Legacy *ngFor bina trackBy — object-reference-based diffing = poora re-render on new array.
  • track expression mein object return karna (na ki primitive key) — identity comparison toot sakta.
  • Sochna track cosmetic hai — bade lists mein ye major perf + state-preservation.
  • Har render pe naya array banakar (immutable) bina track — sab items "naye" lagte.

12. Best Practices

Do this
  • @for ke saath hamesha track item.id (stable unique id).
  • $index sirf static/append-only lists mein (no reorder/insert).
  • Legacy *ngFor → hamesha trackBy function (id return).
  • Bade lists → track + virtual scrolling (S-04) combine.
  • Track key primitive (id/string) rakho, object nahi.

13. Performance

track ka win DOM operations mein — bade lists (hundreds/thousands rows) jo frequently update hote (live tables, feeds) mein, proper track destroy+recreate ke storm ko reuse mein badal deta: fewer DOM mutations, no component re-init, preserved state/animations. Ye ek chhota syntax change hai par high-impact — profiler mein list-render time drastically girta. Virtual scrolling (S-04) ke saath combine = large-list gold standard.

track item.id stable unique identity deta — items reorder/insert hone pe bhi Angular sahi samajhta kaunsa item wahi hai, nodes reuse karta. track $index position (index) se track karta — agar list ke start/middle mein item insert ya reorder ho, indices shift ho jaate, Angular sochta har item badal gaya → unnecessary re-renders + lost state (focus/scroll). id almost hamesha better; $index sirf static/append-only lists mein safe.

MEDIUM Bina track (ya bina trackBy) kya hota hai?

Legacy *ngFor bina trackBy object reference se diff karta — agar tum immutably naya array banate (naye object references), Angular sochta saare items naye hain → poora list destroy+recreate. Ye DOM-heavy, component re-init karta, aur UI state (input focus, scroll, animations) lose karta. Modern @for track ko mandatory bana deta taaki ye trap avoid ho. Hamesha stable id se track karo.

SCENARIO Ek editable table mein type karte waqt input focus randomly khoti hai jab background data refresh hota. Kyun aur fix?

Probably list track $index (ya no proper track) use kar rahi, aur data refresh pe array reorder/replace ho raha. Angular items ki identity galat samajhta → rows ke DOM nodes destroy+recreate → focused input ka element hi replace ho jaata → focus lost. Fix: track item.id (stable unique id) use karo — Angular existing rows ko same identify karke unke DOM nodes (aur focus) reuse karta, sirf badla data update. Focus preserved.

⚡ 30-second recall

  • track item.id (stable) >> track $index (breaks on reorder/insert → lost state).

S-04

Virtual Scrolling & Large Data

Covers: CDK virtual scroll · windowing · cdk-virtual-scroll-viewport · *cdkVirtualFor · itemSize · only render visible · pagination · infinite scroll

1. What is it? (Kya hai ye?)

Virtual scrolling (windowing) ek technique hai jismein ek badi list ke sirf woh items DOM mein render hote jo abhi viewport mein visible hain (plus thoda buffer) — chahe list mein 10,000 items hon. Jaise user scroll karta, DOM nodes recycle hote. Angular CDK ka cdk-virtual-scroll-viewport + *cdkVirtualFor ye deta.

2. Why do we need it? (Zaroorat kyun hai?)

10,000 rows ko DOM mein render karna = 10,000 DOM nodes + components. Ye browser ko ghutne pe la deta: slow initial render, huge memory, janky scroll. Par screen pe sirf ~20 rows dikhte hain! Virtual scrolling sirf woh ~20 render karke baaki ka illusion deta (spacer se scroll height maintain) — 10,000 ka experience, 20 ka cost.

3. What problem does it solve?

Large-list DOM explosion. Thousands of rows = unusable perf. Virtual scrolling DOM node count ko constant (~visible + buffer) rakhta chahe data kitna bhi ho — O(visible) instead of O(total). Initial render fast, memory low, scroll smooth.

4. Internal Working — windowing

Render only the window; recycle on scroll

cdk-virtual-scroll-viewport ek fixed-height scrollable container hai. Ye total list ki height ko calculate karta (itemSize × count) aur ek bada spacer se woh height "fake" karta — toh scrollbar sahi lagta jaise poori list hai. Par actually sirf viewport mein fit hone wale items (+ buffer above/below) ke DOM nodes create hote (*cdkVirtualFor). Jaise user scroll karta, viewport calculate karta ki ab kaunse items visible hone chahiye, aur DOM nodes ko recycle karta — purane (scrolled-out) nodes ka data naye (scrolling-in) items se replace, position adjust. Yani DOM node count constant rehta (~visible), data window shift hota. itemSize (fixed row height) se calculations fast; variable heights ke liye autosize (thoda costly).

 │  (not in    │   height)    │ item 4992│  ← only these render
 │   DOM)      │              │ ...      │

 └─────────────┘              └──────────┘

   *cdkVirtualFor renders only visible+buffer items

   ─▶ recycle DOM nodes with new items' data ─▶ constant node count

AlternativesPagination, infinite scroll — for server-side large data.

Lazy Loading & Code Splitting

Covers: loadComponent · loadChildren · standalone lazy routes · code splitting · chunks · smaller initial bundle · faster startup · route-level splitting

1. What is it? (Kya hai ye?)

Lazy loading matlab code ko on-demand load karna — jab actually zaroorat ho, na ki upfront. Angular routes ko lazy banate ho (loadComponent/loadChildren): ek feature ka code ek alag chunk (bundle) mein split hota jo tabhi download hota jab user us route pe jaata. Initial bundle chhota → faster startup.

2. Why do we need it? (Zaroorat kyun hai?)

Agar poori app ek bade bundle mein ho, user ko homepage dekhne ke liye admin-panel + settings + reports sab ka code download karna padega — bhale woh kabhi na jaaye. Ye initial load slow karta (LCP/TTI). Lazy loading sirf zaroori code pehle bhejta, baaki jab chahiye tab — startup fast.

3. What problem does it solve?

Monolithic bundle bloat. Sab kuch upfront = bada initial download = slow first load. Code splitting app ko route-based chunks mein todta; user sirf woh download karta jo use chahiye. Initial bundle minimal, subsequent features on-demand.

4. Internal Working — build-time code splitting

Dynamic import() → separate chunk

Jab tum route mein loadComponent: () => import('./admin.component') likhte ho, build tool (esbuild/webpack) us dynamic import() ko dekhkar us component (aur uski dependencies) ko ek alag chunk file mein split kar deta — main bundle se bahar. Runtime pe, jab user us route pe navigate karta, Angular router us chunk ko dynamically fetch karta (network request), phir component render. Ye route-level code splitting hai. Standalone components ne ise aur clean banaya (loadComponent for a single standalone component; loadChildren for a lazy route sub-tree). Guards (canMatch, Ch17 S-05) chunk download ko bhi gate kar sakte — unauthorized users ke liye code hi na bheja jaaye. Standalone components (v14+, default v17+) better tree-shaking + finer splitting dete NgModules se.

Diagram — code splitting
 build: main.js (home + shell)   admin-chunk.js   reports-chunk.js
        ↑ loaded on startup       ↑ loaded when     ↑ loaded when
                                    /admin visited     /reports visited

 initial download = main.js only (small) → fast startup
 navigate /admin ─▶ fetch admin-chunk.js on demand ─▶ render

5. Execution Flow

 startup ─▶ only main bundle loads
 user clicks "Admin" ─▶ router fetches admin chunk ─▶ component renders
 (canMatch guard can block the fetch entirely for unauthorized users)

6. Architecture — the API

loadComponent: () => import(...)Lazy-load a single standalone component.
loadChildren: () => import(...)Lazy-load a sub-route tree (returns Routes / NgModule).
StandaloneDefault in v17+; cleaner splitting, better tree-shaking than NgModules.
canMatch guardGate the chunk download (Ch17 S-05) — security + perf.

Deferrable Views (@defer)

Covers: @defer blocks · triggers (on idle/viewport/interaction/hover/timer/immediate) · when condition · prefetch · @placeholder/@loading/@error · in-page code splitting

1. What is it? (Kya hai ye?)

@defer (Angular v17+) ek template-level primitive hai jo ek block ke components ko lazy-load karta — bina route ke. Tum template mein @defer (on viewport) { <heavy-chart /> } likhte ho, aur Angular us chart ke code ko alag chunk mein split karke tabhi load karta jab trigger (viewport mein aana, etc.) fire ho.

2. Why do we need it? (Zaroorat kyun hai?)

Route-level lazy loading (S-05) sirf routes ke liye. Par ek page pe bhi heavy components ho sakte (charts, maps, rich editors, comments) jo initial render ke liye zaroori nahi. Pehle inhe lazy karna manual + painful tha. @defer declarative, built-in way deta — in-page components ko trigger-based lazy-load karo, initial bundle + render se hata do.

3. What problem does it solve?

Heavy in-page components ka upfront cost. Ek dashboard mein ek bhaari chart initial load slow kar sakta bhale woh below-the-fold ho. @defer use viewport-aane-tak (ya interaction-tak) tal deta — initial bundle chhota, initial render fast, aur heavy code sirf jab actually chahiye.

4. Internal Working — triggers & sub-blocks

A trigger loads a separate chunk; sub-blocks fill the gap

@defer apne andar ke components ko ek alag chunk mein code-split karta (route lazy-load jaisa, par in-page). Chunk tab load hota jab trigger fire ho: on idle (browser idle — default), on viewport (block viewport mein aaya), on interaction (user clicked/typed on it), on hover, on immediate (turant, after non-deferred content), on timer(2s), ya when <condition> (boolean/signal). Jab tak defer content load nahi hota, @placeholder dikhta; loading ke dauraan @loading; fail pe @error. prefetch on ... se code ko trigger se pehle background mein fetch kar sakte (jaise prefetch on idle + on interaction — idle pe download, interaction pe render — instant feel). Ye SSR + incremental hydration (S-09) ka bhi foundation hai.

on idle (browser idle, default), on viewport (block visible), on interaction (click/type), on hover, on immediate (non-deferred content ke baad turant), on timer(t), aur when <expr> (custom boolean/signal). Plus prefetch on ... — code ko render-trigger se pehle background mein download (jaise prefetch on idle; on interaction = idle pe fetch, interaction pe instant render). Sub-blocks: @placeholder/@loading/@error.

MEDIUM Route lazy-loading vs @defer — kab kaunsa?

Route lazy-loading (loadComponent/loadChildren) poore routes ko lazy-load karta — feature-level splitting. @defer ek page ke andar individual heavy components ko lazy-load karta — in-page granularity, trigger-based (viewport/interaction). Route → alag page/feature; @defer → same page pe below-fold/on-demand components (chart, comments). Dono complementary — routes se coarse splitting, @defer se fine.

SCENARIO Ek dashboard fast load karna hai par usmein ek heavy analytics chart (below the fold) initial render slow kar raha. Fix?

@defer (on viewport; prefetch on idle) mein chart ko wrap karo. Ab chart ka code initial bundle mein nahi (alag chunk), aur initial render mein @placeholder (correct height ka skeleton — CLS avoid) dikhta. Browser idle hone pe chunk background mein prefetch ho jaata; jab user scroll karke chart tak pahunchta, woh instantly render hota (already fetched). Hero + above-fold content instant, chart on-demand — LCP/TTI dono better.

  • Sub-blocks: @placeholder (before, needs dimensions for CLS) / @loading / @error. Defer non-critical/below-fold. Basis for incremental hydration.

Preloading Strategies

Covers: PreloadAllModules · NoPreloading · custom PreloadingStrategy · network-aware · selective preload · anticipatory loading · withPreloading

1. What is it? (Kya hai ye?)

Preloading strategy decide karti hai ki lazy route chunks (S-05) ko background mein kab download kiya jaaye — user ke actually navigate karne se pehle. Lazy loading initial bundle chhota rakhta par navigation pe fetch-delay deta; preloading us delay ko hide karta by fetching chunks ahead of time (idle bandwidth mein).

2. Why do we need it? (Zaroorat kyun hai?)

Pure lazy loading ka downside: jab user pehli baar ek lazy route pe jaata, chunk download hone tak wait (spinner). Preloading initial load ke baad, background mein, likely-needed chunks pehle se fetch kar leta — toh navigation instant lagta. Best of both: chhota initial bundle + snappy navigation.

3. What problem does it solve?

Lazy-loading navigation delay. Chhota initial bundle achha, par har first-visit pe chunk-fetch lag annoying. Preloading initial critical load complete hone ke baad idle time mein chunks fetch karke ye delay eliminate karta — user ko lazy-loading ka size benefit milta bina navigation penalty ke.

4. Internal Working — strategies

When to fetch lazy chunks in the background

Router ek PreloadingStrategy leta jo har lazy route ke liye decide karta preload karna hai ya nahi. Built-in: NoPreloading (default — kuch preload nahi, pure lazy); PreloadAllModules (initial load ke baad saare lazy chunks background mein fetch — simple, aggressive). Custom strategy se selective control: ek PreloadingStrategy implement karke sirf specific routes preload karo (jaise route data: { preload: true } flag se), ya network-aware (Network Information API se — slow connection pe skip, fast pe preload), ya priority-based. Preloading initial critical rendering ke baad chalti (taaki startup slow na ho) aur idle bandwidth use karti. Configure: provideRouter(routes, withPreloading(PreloadAllModules)).

          NoPreloading      → none (fetch on navigate)
          PreloadAllModules → all lazy chunks
          Custom            → selected routes / network-aware
              │
 user navigates ─▶ chunk already cached ─▶ INSTANT (no fetch wait)

   ─▶ idle: preload lazy chunks in background
 navigate to /admin ─▶ chunk already fetched ─▶ render instantly

  }

9. Advanced Example — network-aware preloading

respect the user's connection
@Injectable({ providedIn: 'root' })
export class NetworkAwarePreload implements PreloadingStrategy {
  preload(route: Route, load: () => Observable<any>): Observable<any> {
    const conn = (navigator as any).connection;
    const slow = conn && (conn.saveData || /2g/.test(conn.effectiveType));
    return slow ? of(null) : load();   // skip preload on slow / save-data
  }
}
// don't burn a user's limited mobile data preloading chunks they may not need.

  3. PreloadAllModules — sab preload; custom — selective.
  4. Network-aware — connection quality check karke decide.
  5. Preloading startup ke baad, background idle bandwidth mein.

Traps

Image & Asset Optimization

Covers: NgOptimizedImage · ngSrc · priority (LCP) · fill · placeholder · automatic srcset/lazy · width/height (CLS) · image loaders/CDN · fonts · static assets

1. What is it? (Kya hai ye?)

Images aksar page ke sabse bade assets hote — LCP (largest content) usually ek image hi hoti. Angular ka NgOptimizedImage directive image loading best practices ko automate karta: LCP image prioritize, baaki lazy-load, correct sizing (CLS avoid), responsive srcset, aur CDN loaders. Plus fonts/static assets ki hygiene.

2. Why do we need it? (Zaroorat kyun hai?)

Un-optimized images = slow LCP + layout shift (CLS) + wasted bandwidth. Manually har image pe priority hints, lazy loading, srcset, dimensions set karna tedious + error-prone. NgOptimizedImage in best practices ko ek simple ngSrc API mein bundle karta, aur dev-time warnings deta jab kuch galat (jaise LCP image bina priority).

3. What problem does it solve?

Image performance mistakes — the most common Core Web Vitals killer. Oversized images, lazy-loaded LCP image, missing dimensions (layout shift), no responsive sizing. NgOptimizedImage in sabko address karta with sensible defaults + guardrails, directly LCP + CLS improve karke.

4. Internal Working — what NgOptimizedImage does

ngSrc + automatic best practices

ngSrc use karke (na ki src), directive image loading control karta. Key behaviors: (1) priority — LCP image pe lagao; ye fetchpriority="high" + loading="eager" set karta aur server-render pe ek <link rel=preload> generate karta — LCP image jaldi load. (2) Non-priority images ko automatically loading="lazy" (viewport aane par load). (3) width+height required — aspect ratio se browser space reserve karta → no layout shift (CLS). (4) Automatic srcset generation (width/height ya ngSrcset+sizes se) — device-appropriate image size download. (5) fill mode — background-image ka better alternative (parent position:relative). (6) placeholder — low-res blurred placeholder (CDN loader ke saath). (7) Image loaders (Imgix/Cloudinary/custom via IMAGE_LOADER) — CDN se optimized URLs. Plus dev warnings: LCP image bina priority, oversized files, missing dimensions.

   → loading=lazy (auto) → loads when scrolled to

SSR, Hydration & Prerendering

Covers: SSR (@angular/ssr) · full-app hydration · event replay · incremental hydration (@defer hydrate) · hybrid rendering / render modes · prerendering (SSG) · LCP/SEO/TTI

1. What is it? (Kya hai ye?)

SSR (Server-Side Rendering) app ko server pe render karke ready HTML browser ko bhejta — user turant content dekhta (fast first paint + SEO). Phir hydration us static HTML ko client-side "activate" karta (event listeners + interactivity). Incremental hydration (v19) isse selective banata — parts ko on-demand hydrate. Prerendering (SSG) build-time pe HTML generate karta.

2. Why do we need it? (Zaroorat kyun hai?)

Pure client-side rendering (CSR) mein browser ko pehle blank page + poora JS download + execute karna padta phir content dikhta — slow LCP/FCP, poor SEO (crawlers ko blank/JS-dependent page). SSR ready HTML deta (fast visible content + crawlable), aur hydration usse interactive banata. Best of server-rendered speed + client-side interactivity.

3. What problem does it solve?

Slow first paint + weak SEO of CSR. CSR "blank until JS runs"; SSR content pehle deta. Par naive SSR mein hydration poore app ko ek saath activate karta (JS-heavy, slow TTI). Incremental hydration ye solve karta — sirf zaroori parts hydrate, baaki on-trigger. Prerendering static pages ka SSR-server-cost bhi hata deta.

4. Internal Working — the SSR → hydration spectrum

Render on server, activate on client — progressively

SSR (@angular/ssr, provideClientHydration()): server app render karke HTML bhejta; browser use turant dikhata. Phir full-application hydration (v16) us HTML ko client pe reuse karta (DOM re-create nahi, existing markup se attach — non-destructive) aur poore app ko interactive banata. Problem: full hydration poore app ka JS ek saath bootstrap karta — bada app slow-to-interactive. Event Replay (withEventReplay(), v18, v19 SSR mein default): hydration complete hone se pehle user ke events (clicks) capture karta aur hydration ke baad replay karta — koi interaction lost nahi. Incremental Hydration (withIncrementalHydration(), v19): @defer (hydrate on ...) se app ko "hydration zones" mein todta — block SSR-render hota (placeholder nahi, actual content) par uski interactivity trigger (idle/viewport/interaction) tak defer hoti. Yani less JS upfront, on-demand hydrate. (Auto-includes event replay.) Hybrid Rendering (v19): per-route render mode set karo — Server (SSR), Client (CSR), ya Prerender (build-time SSG). Har page ki zaroorat ke hisab se.

Diagram — CSR vs SSR vs incremental hydration
 CSR:  [blank] ─▶ download JS ─▶ execute ─▶ content + interactive
        (slow first paint, weak SEO)

 SSR + full hydration:
   server HTML ─▶ content VISIBLE fast ─▶ hydrate WHOLE app ─▶ interactive
   (fast paint; but big hydration = delayed TTI)

 SSR + incremental hydration:
   server HTML ─▶ content visible ─▶ hydrate only what's needed, on trigger
   (fast paint + minimal JS upfront + on-demand interactivity)

5. Execution Flow

 event replay captures early clicks
 @defer (hydrate on interaction) block ─▶ hydrates when user interacts
   ─▶ replays the captured event ─▶ seamless

withIncrementalHydration()Selective hydration via @defer hydrate (v19). Auto-includes event replay.

Content sab SSR-rendered hai (visible + SEO), par JS sirf zaroori widgets ke liye on-demand load/hydrate hota — fast TTI + INP. Event replay early clicks preserve karta.

S-10

Network & Data Optimization

Covers: HTTP caching · shareReplay · debounceTime/distinctUntilChanged · request dedup · pagination · prefetch via resolvers · switchMap cancellation · httpResource

1. What is it? (Kya hai ye?)

Network & data optimization ka matlab: kam, chhote, aur smarter HTTP requests bhejna. Techniques: results cache karo (repeat requests avoid), duplicate requests dedupe/share karo, user input pe debounce karo (har keystroke pe request na bhejo), stale requests cancel karo, aur data pagination/prefetch se manage karo. RxJS operators + HttpClient in sabko enable karte.

2. Why do we need it? (Zaroorat kyun hai?)

Network aksar app ka slowest part hai. Redundant requests (same data baar-baar), chatty requests (har keystroke pe search), aur huge payloads app ko slow + server ko overloaded karte. In sabko optimize karke tum fewer/faster requests bhejte — snappier UI, less bandwidth, less server load.

3. What problem does it solve?

Redundant, chatty, aur heavy network traffic. Bina optimization: same data multiple components alag-alag fetch karte, search har keystroke pe fire karta, stale responses race karte, huge lists ek saath aate. Caching, dedup, debounce, cancellation, aur pagination ye sab address karke network ko efficient banate.

4. Internal Working — the techniques

Cache, share, debounce, cancel, paginate

Caching: ek fetched result ko store karke repeat requests avoid — HTTP interceptor (Ch17 S-04) se, ya shareReplay(1) se ek Observable ka result cache+share (multiple subscribers ek hi request share karte, na ki har koi naya). Debounce: debounceTime(300) — user type karna ruke tab request (har keystroke nahi); distinctUntilChanged() — same value repeat pe skip. Cancellation: switchMap — naya request aaye toh purana (stale) cancel (Ch11 RxJS) — search-as-you-type mein race conditions rokta. Dedup: in-flight identical requests ko share karo (ek request, multiple consumers). Pagination: data page-by-page (poora dataset nahi). Prefetch: route resolvers se navigation se pehle data fetch (ya idle pe likely-next data). httpResource (v20 experimental): signal-based reactive HTTP — URL signal badle toh auto re-fetch, response ek signal.

1. What is it? (Kya hai ye?)

Ye final layer: app ko efficiently build + ship karna aur runtime mein clean rakhna. 2. Why do we need it? (Zaroorat kyun hai?)

3. What problem does it solve?

Bloated shipping + runtime degradation. Bina in: bundle silently grow karta, dead code ship hota, assets bade jaate, repeat visits slow, main thread block hota, aur leaked subscriptions memory kha jaate. SCENARIO App time ke saath slow hota jaata — jitna zyada use karo utna laggy, kabhi tab crash. Diagnosis + fix?
