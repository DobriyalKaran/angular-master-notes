S-01

CLI Internals

Covers: @angular/cli · @angular-devkit · schematics · builders · angular.json · ng commands · code generation · the architect API

1. What is it? (Kya hai ye?)

Angular CLI (ng command) ek command-line tool hai jo Angular projects ko scaffold, generate, build, serve, aur test karta. Andar se ye do core pieces pe khada hai: Schematics (code generation/transformation engine) aur Architect / Builders (tasks jaise build/serve/test run karne ka engine). angular.json in sabko configure karta.

2. Why do we need it? (Zaroorat kyun hai?)

Bina CLI, tumhe manually TypeScript config, bundler config, test setup, folder structure, aur boilerplate likhna padega — har project mein, error-prone. CLI ye sab standardize + automate karta: ek command se consistent project, ek command se component generate, ek command se optimized build. Ye "convention over configuration" deta.

3. What problem does it solve?

Tooling complexity + inconsistency. Modern frontend build setup (TS + bundler + AOT + optimization + SSR) bahut moving parts. CLI in sabko ek zero-config, versioned, upgradable abstraction ke peeche chhupa deta — tum ng build likhte ho, woh poora toolchain orchestrate karta. Plus ng update se framework migrations automate.

4. Internal Working — schematics + architect

Two engines: code-gen (schematics) & task-runner (architect)

CLI internally @angular-devkit packages use karta. Do main engines: (1) Schematics (@angular-devkit/schematics) — ek code generation + transformation engine. Jab tum ng generate component chalate, ek "schematic" run hoti jo files ko ek virtual file system (Tree) mein describe karti — files add/modify karti (component TS/HTML/spec, plus module/routing updates) — phir Tree ko disk pe commit karti. Schematics idempotent + composable hoti; ng update ki migrations bhi schematics hain (code ko transform karti). (2) Architect (@angular-devkit/architect) — ek task orchestration API. Har command (build, serve, test) ek builder ko map karti (jaise @angular-devkit/build-angular:application). Builder ek function hai jo options leta aur task run karta (build → esbuild invoke karta). angular.json project structure, targets (build/serve/test), aur unke builders + options define karta. So: ng generate → schematics; ng build/serve/test → architect + builders; angular.json = the config that wires it all.

Build Process

Covers: esbuild + Vite · application builder · AOT compilation · tree-shaking · minification · code splitting · ESM output · dev server / HMR

1. What is it? (Kya hai ye?)

Build process woh pipeline hai jo tumhare TypeScript + templates + styles ko browser-ready optimized JavaScript/CSS bundles mein convert karta. Modern Angular (v17+) mein ye esbuild (actual bundler) + Vite (dev server) pe based hai, application builder ke through. Ye compile (AOT), bundle, tree-shake, minify, aur split karta.

2. Why do we need it? (Zaroorat kyun hai?)

Browser TypeScript, Angular templates, ya bahut saari alag files directly efficiently nahi chala sakta. Build process source ko: compile (TS→JS, templates→Ivy instructions), optimize (dead code hatao, minify), aur package (chunks jo lazy-load ho) karta. Bina achhe build system ke, bundles bade + builds slow hote.

3. What problem does it solve?

Slow builds + bloated output. Purana Webpack-based system enterprise apps pe slow tha (single-threaded JS bundler). esbuild (Go, multi-threaded) drastically fast hai. Application builder AOT + tree-shaking + minification + code splitting + SSR ko ek integrated, fast pipeline mein deta — chhote bundles, tez builds.

4. Internal Working — esbuild + Vite

esbuild bundles; Vite serves. AOT + tree-shake + split.

Angular v17+ ka default application builder (@angular-devkit/build-angular:application) do tools use karta: esbuild (Go mein likha, multi-core parallel bundler/minifier — actual "building" ye karta) aur Vite (dev server — native ESM + near-instant HMR deta; Vite bundle nahi karta, woh sirf dev-time "messenger" hai). Pipeline: (1) AOT compilation — Angular compiler (ngtsc, S-03) TS + templates ko Ivy instructions wale JS mein compile karta (build-time pe, browser mein nahi). (2) Bundling — esbuild modules ko bundles mein combine karta. (3) Tree-shaking — unused exports/code drop (ESM + Ivy locality se possible). (4) Code splitting — dynamic import() (lazy routes, @defer) par alag chunks. (5) Minification — whitespace/names shrink. (6) Output — modern ESM with content-hashed filenames. Production build (ng build) sab optimizations on; dev (ng serve) Vite se fast HMR, minimal optimization. Old Webpack browser builder ab deprecated hai. Application builder client bundle + optional SSR server + prerendering ek saath handle karta.

The Angular Compiler

Covers: ngtsc · AOT vs JIT · decorators → static definitions · ɵcmp/ɵfac/ɵprov/ɵinj · TypeScript transformer · compilation model · @angular/compiler

1. What is it? (Kya hai ye?)

Angular compiler woh cheez hai jo tumhare declarative Angular code (@Component, templates, decorators) ko plain, runnable JavaScript mein badalti hai. Modern compiler ka naam ngtsc (the Ivy compiler) hai — ye ek TypeScript transformer hai jo tsc ke saath chalti aur decorators ko static class definitions (ɵcmp, ɵfac, etc.) mein compile karti.

2. Why do we need it? (Zaroorat kyun hai?)

Angular ka syntax (decorators, HTML templates, bindings) browser directly nahi samajhta. Compiler in high-level constructs ko efficient JS instructions mein translate karta — templates ko rendering code mein, decorators ko runtime metadata mein. Bina compiler, Angular ka declarative model chal hi nahi sakta. Aur AOT compilation isse build-time pe karke browser ko compiler bhejne se bachata.

3. What problem does it solve?

High-level Angular code ko runnable code mein badalna, efficiently. Compiler declarative templates + metadata ko optimized Ivy instructions + static definitions mein translate karta. AOT mode isse build-time pe karke: (a) browser mein compiler ki zaroorat khatam (chhota bundle, fast startup), (b) template errors build-time pe pakde jaate, (c) rendering code pehle se ready.

4. Internal Working — ngtsc & the definitions

Decorators → static ɵ definitions, as a TS transformer

ngtsc Angular ka Ivy-era compiler hai, jo ek TypeScript compiler plugin/transformer ke roop mein implement hai — yani ye normal tsc compilation ke andar hook karke chalti. Har Angular decorator ke liye, compiler ek static field generate karti class pe (decorator ko erase karke): @Component → ɵcmp (component definition — template function, selectors, inputs/outputs, directiveDefs) + ɵfac (factory — instance kaise banega). @Injectable → ɵprov (provider def) + ɵfac. @Directive → ɵdir. @Pipe → ɵpipe. @NgModule → ɵmod + ɵinj (injector def). Ye ɵ (theta) prefix "framework-private" denote karta. Template ka compilation (S-05) ɵcmp ke andar template instruction function banata. AOT (Ahead-of-Time): ye sab build-time pe hota — production default. Browser ko ready JS milta, compiler nahi. JIT (Just-in-Time): compilation browser mein runtime pe (needs @angular/compiler in bundle) — historically dev/testing. AOT v9+ (Ivy) se default hai. Locality (S-04) ki wajah se ngtsc har file ko independently compile kar sakti — incremental + fast.

   emit static defs: ɵcmp / ɵfac / ɵprov / ɵmod / ɵinj
   erase decorators
   → plain JS, no compiler needed at runtime (AOT)

Ivy Architecture

Covers: Ivy rendering engine · locality · tree-shakability · instruction set · LView / TView · ɵɵ instructions · incremental compilation · why Ivy replaced View Engine

1. What is it? (Kya hai ye?)

Ivy Angular ka rendering + compilation engine hai (v9 se default) — woh architecture jo decide karta hai templates kaise compile hote aur views kaise runtime pe render/update hote. Iske do signature ideas: instructions (templates → ɵɵ function calls) aur locality (har component independently compile hota). Runtime data ye LView/TView structures mein rakhta.

2. Why do we need it? (Zaroorat kyun hai?)

Purana engine (View Engine) global compilation maangta (har component ko doosron ki metadata chahiye thi — .ngfactory files, slow builds, bade bundles, poor tree-shaking). Ivy ne isse locality se replace kiya: har component apne aap compile hota, sirf apni template + used deps ki public API jaanke. Isse fast incremental builds, chhote bundles, better tree-shaking, aur naye features (@defer, signals, hydration) possible hue.

3. What problem does it solve?

View Engine ki global-compilation bottleneck. Ivy: (a) locality — component-by-component compile (incremental, fast); (b) tree-shakability — instructions tree-shakable functions, unused framework code drops (chhote bundles); (c) modern-feature foundation — @defer, signals, incremental hydration sab Ivy ke instruction+LView model pe khade hain.

4. Internal Working — instructions, LView, TView

Templates become instructions; runtime lives in LView/TView

Instructions: Ivy mein template ek function ban jaata hai jo Ivy runtime instructions (ɵɵ prefixed — "theta theta") call karta: ɵɵelementStart, ɵɵtext, ɵɵproperty, ɵɵadvance, etc. (S-05). Ye instructions tree-shakable standalone functions hain — jo use na hon, drop. Locality: component ko compile karne ke liye compiler ko sirf uski apni template aur uski dependencies ki public API chahiye (unki full metadata nahi) — isliye har file independently compile hoti (incremental builds). Runtime data structures: jab component render hota, Ivy do parallel structures banata: TView (Template View) — component type ka shared, static data (template ka "blueprint": instructions ki list, directive defs, binding metadata). Ek TView per component type, saari instances share karti — jaise ek "class". LView (Logical/Live View) — ek component instance ka live data (ek array): component instance, DOM node references, current binding values, injected tokens, child view pointers. Ek LView per instance — jaise ek "object". LView apne TView ko point karta (LView[TVIEW]). Ye class/instance separation memory bachata (shared static data duplicate nahi hota) aur fast rendering deta.

Template Compiler

Covers: template → instruction function · create vs update passes · RenderFlags · binding instructions · ɵɵadvance · slot allocation · embedded views (@if/@for)

1. What is it? (Kya hai ye?)

Template compiler Angular compiler ka woh hissa hai jo specifically tumhare HTML template ko ek instruction function mein badalta. Ye template ko parse karke ek two-phase function generate karta: ek create pass (DOM structure ek baar banati) aur ek update pass (bindings har change detection pe update karti). Har binding ko LView mein ek "slot" milta.

2. Why do we need it? (Zaroorat kyun hai?)

HTML templates + Angular bindings ({{ }}, [prop], (event), @if, @for) browser ke liye directly executable nahi. Template compiler inko efficient imperative instructions mein translate karta — jo exactly batate DOM kaise banao aur kaunse bindings kab update karo. Ye Angular ke declarative templates ko fast runtime rendering se jodta hai.

3. What problem does it solve?

Declarative template → efficient runtime code. Bina compilation, template ko runtime pe interpret karna padta (slow, needs compiler in bundle). Template compiler isse pre-generated instruction function mein badalta: create pass DOM structure once banata, update pass sirf changed bindings touch karta (dirty checking, S-10). Efficient + no runtime template parsing.

4. Internal Working — create/update passes & slots

Two passes, indexed slots, ɵɵadvance

Template compiler HTML ko parse karke ek template AST banata, phir ek instruction function emit karta jiska signature hai (rf: RenderFlags, ctx). rf ek bitmask hai: RenderFlags.Create (=1) aur RenderFlags.Update (=2). Function ke andar do blocks: if (rf & 1) { ... } — create pass, DOM structure banata (ɵɵelementStart, ɵɵelement, ɵɵtext, ɵɵtemplate for structural blocks) — ye ek baar chalta (view creation pe). if (rf & 2) { ... } — update pass, bindings evaluate + update karta (ɵɵproperty, ɵɵtextInterpolate, ɵɵattribute) — ye har change detection pe chalta. Har element/binding ko ek numbered slot index milta LView array mein (compiler decls = declaration count aur vars = binding var count declare karta). ɵɵadvance(n) instruction "current node pointer" ko n slots aage badhati — taaki update instructions sahi node/binding slot pe apply hon (create order aur update order match karein). Bindings ke liye compiler ek slot mein last value store karta; update pass naya value compare karke (dirty check) sirf change pe DOM touch karta. Structural constructs (@if, @for, *ngIf) embedded views banate: ɵɵtemplate ek "container" declare karta, aur block ka content ek alag sub-template function + child LView(s) ban jaata (conditionally/repeatedly created).

Diagram — the two-pass template function

   if (rf & 1 /*Create*/) {        ← runs ONCE (view creation)
     ɵɵelementStart(0, 'h1');       slot 0 = <h1>

 }

Dependency Injection Engine

Covers: injector tree · environment vs element injectors · resolution algorithm · bloom filters · R3Injector · tokens & providers · hierarchical resolution · tree-shakable providers · resolution modifiers

1. What is it? (Kya hai ye?)

Dependency Injection (DI) engine woh system hai jo services (dependencies) ko create karke components/services ko provide karta — tum inject(X) ya constructor param likhte ho, engine sahi instance dhoondh ke deta. Andar se ye ek hierarchical injector tree hai: environment injectors (app/module-level) aur element injectors (component-tree-level), jo ek resolution algorithm se tokens ko instances mein resolve karte.

2. Why do we need it? (Zaroorat kyun hai?)

Bina DI, har class ko apni dependencies khud new karni padegi — tight coupling, hard testing, no shared singletons, manual wiring. DI dependencies ko invert karta: tum sirf "mujhe X chahiye" declare karte ho, engine lifecycle + sharing manage karta. Isse loose coupling, easy mocking (testing), aur controlled sharing (singletons/scoped) milta.

3. What problem does it solve?

Manual dependency wiring + coupling. DI dependencies ko centralize + automate karta: shared services (root singletons), scoped services (component-level), aur swappable implementations (tokens) sab declaratively. Hierarchical injectors scoping dete (kahan service available hai), aur tree-shakable providers unused services ko bundle se drop karte.

4. Internal Working — two trees, bloom-filter resolution

Element injector tree → environment injector tree, with bloom filters

Angular DI ke do parallel hierarchies hain: (1) EnvironmentInjector tree — application/module-level. Root environment injector bootstrap pe banta (S-06); lazy-loaded routes apne child environment injectors bana sakte. Ivy ka implementation R3Injector hai. Ye providedIn:'root' services + appConfig.providers host karta. (2) ElementInjector tree — component/directive tree ke saath, har DOM element jiske component/directive providers hon, uska apna element injector. Resolution algorithm: jab ek component inject(X) maangta, engine: pehle us component ke element injector se shuru karke upar element injector tree mein chadhta (parent elements) — jab tak X mile; agar element tree mein na mile, phir environment injector hierarchy mein jaata (current → parents → root → platform); agar kahin na mile → NullInjector jo NullInjectorError throw karta. Bloom filters: element injectors ko fast banane ke liye, Ivy har injector pe ek bloom filter (ek bitmask) rakhta jo probabilistically batata "is injector mein ye token ho sakta hai ya definitely nahi". Resolution ke dauraan, engine bloom filter se quickly un injectors ko skip karta jinmein token definitely nahi hai — poora tree slow-search karne ke bajaye. (Bloom filter false-positive de sakta but false-negative nahi — toh "definitely not" skip safe hai.) Tokens: class, InjectionToken, ya string. Providers: useClass/useValue/useFactory/useExisting. Tree-shakable: providedIn:'root' service pe ɵprov deta — agar service inject na ho, tree-shaken (module providers array ke ulta, jo tree-shakable nahi).

Rendering Pipeline

Covers: createLView · renderView (create pass) · refreshView (update pass) · lifecycle hooks order · component tree traversal · view creation → first render → updates

1. What is it? (Kya hai ye?)

Rendering pipeline woh end-to-end runtime flow hai jo ek component ko instance se lekar DOM tak le jaata, aur phir update karta. Ye do phases orchestrate karta: view creation (createLView + renderView — DOM structure banao ek baar) aur view refresh (refreshView — bindings update + child components refresh + lifecycle hooks, har CD pe). Ye S-04/S-05 ke pieces ko ek chalti hui process mein jodta.

2. Why do we need it? (Zaroorat kyun hai?)

Instructions + LView/TView (S-04) aur create/update passes (S-05) alag concepts hain — inhe ek coordinated runtime process mein orchestrate karna padta. Rendering pipeline ye karta: kis order mein views create hon, DOM kab attach ho, bindings kab refresh hon, child components kab process hon, aur lifecycle hooks kab fire hon. Bina ye orchestration, individual pieces useless hain.

3. What problem does it solve?

Coordinating creation, updates, children, aur lifecycle. Pipeline deterministic ordering deti: parent view create → children create → DOM attach → first CD; phir har CD pe parent refresh → child inputs update → child refresh → hooks in correct order. Ye ensure karta ki data flows top-down consistently aur hooks predictable order mein chalein.

4. Internal Working — createLView, renderView, refreshView

Create the view, render it once, then refresh it each CD

Jab ek component render hota, pipeline ye steps chalati: (1) createLView — us component instance ke liye ek naya LView array allocate hota (S-04), uske TView (blueprint, once per type) se linked. (2) renderView (create pass) — template function rf=Create ke saath chalti (S-05) — DOM structure build hota (ɵɵelementStart, etc.) LView slots mein, event listeners attach hote. Ye ek baar. Child components encounter hone pe unke LViews bhi create hote (recursive). (3) DOM host element se attach. (4) refreshView (update pass) — ye CD ka core hai (S-10): template function rf=Update chalti (bindings evaluate + DOM update if changed), phir child components ko refresh karti (unke inputs update karke, phir unka refreshView), aur lifecycle hooks correct order mein fire karti. Lifecycle order (roughly per view): ngOnChanges (input changes) → ngOnInit (first only) → ngDoCheck → child content/view hooks → ngAfterContentInit/Checked → ngAfterViewInit/Checked. Ye har change detection par (except once-only hooks). Data top-down flow karta: parent apne bindings evaluate karke child inputs set karta, phir child refresh. Ye "unidirectional data flow" pipeline ka core hai — isliye "expression changed after checked" error aata jab child parent ka state badalta (top-down violate).

Agar koi child (ya lifecycle hook) ek value badalta jise parent is pass mein already render kar chuka, toh doosre verification check mein woh binding ka value alag dikhta → ye error. Root cause: unidirectional top-down flow violate karna (child mid-pass parent-bound state mutate kar raha). Fix: CD pass ke dauraan bound state child se mat badlo; signals use karo, ya restructure karo taaki data sirf parent→child bahe.

1. What is it? (Kya hai ye?)

1. What is it? (Kya hai ye?)

Signals Angular ka fine-grained reactivity primitive hai (v16+). Andar se ye ek reactive dependency graph hai jo producers (values jo change notify karti — signals, computed) aur consumers (jo values pe depend karti — effects, templates, computed) ke beech banti. Ek clever push/pull algorithm se ye glitch-free aur efficient updates deta.

2. Why do we need it? (Zaroorat kyun hai?)

Traditional CD (zone.js) poore component tree ko check karta har async event pe — coarse + wasteful. Signals fine-grained reactivity dete: exactly woh cheezein re-run/update hoti jo actually changed value pe depend karti, aur exactly tab. Ye zoneless CD (Ch16), efficient derived state (computed), aur predictable reactivity enable karta. Internals samajhna "signals kaise glitch-free + lazy hote" ka answer deta.

3. What problem does it solve?

Coarse, redundant reactivity + glitches. Zone-based CD over-checks; naive push-based reactivity "diamond problem" (ek value ko multiple baar recompute) + glitches (intermediate stale values) deta. Signals ka push/pull dono solve karta: change eagerly propagate hoti (correctness) par recomputation lazy hoti (efficiency) — glitch-free, minimal computation.

4. Internal Working — the reactive graph, push/pull, versions

Producers & consumers, bidirectional edges, push dirtiness / pull values

Angular signals ka core @angular/core/primitives/signals mein hai. Har participant ek ReactiveNode interface implement karta — jo producer, consumer, ya dono ho sakta. Producer = value jo change notifications deti (writable signals, computed). Consumer = reactive context jo producers pe depend karti (effects, templates, computed). computed dono hai (doosre signals consume karta, naya value produce karta). Jab ek consumer ek producer ko read karta (reactive context mein), unke beech ek bidirectional edge banti — dependency graph. valueVersion: har producer ek monotonically-increasing valueVersion rakhta — jab producer semantically-naya value produce karta, version increment hota. Jab consumer producer se read karta, current version dependency edge mein save hota — taaki baad mein consumer check kar sake "kya mera dependency badla?". Push/pull algorithm (glitch-free): jab ek source signal set hoti, "dirtiness" eagerly PUSH hoti graph mein downstream (consumers ko "tum shayad stale ho" mark karti) — par actual recomputation nahi. Recomputation lazily PULL hoti — sirf jab koi consumer apna value read karta (jaise template CD mein), tab woh apne dependencies ki versions check karke, agar koi badla, recompute karta. Ye do-phase (eager dirty-push, lazy value-pull) diamond problem solve karta: ek value multiple paths se depend ho toh bhi ek hi baar recompute (na ki intermediate glitchy states). computed lazy + memoized: computed tabhi recompute karta jab (a) koi dependency badla ho AND (b) uska value actually read ho — warna cached value. effect: ek consumer; set ke saath synchronously nahi chalta — schedule hota (microtask), batches (multiple deps ek saath badlein toh ek hi run), aur destroy pe auto-cleanup. Live consumers & CD: component template ek live consumer hai — template mein signal read karna view ko us signal ka consumer bana deta; signal change → view Dirty mark + CD schedule (S-10). Equality: set/update pe naya value purane se compare hota (Object.is ya custom equal) — agar equal, consumers notify hi nahi hote (no wasted work).

9. Advanced Example — glitch-free (diamond problem)

why push/pull avoids double computation
const a = signal(1);
const b = computed(() => a() + 1);   // depends on a
const c = computed(() => a() + 2);   // depends on a
const d = computed(() => b() + c()); // depends on b AND c  → "diamond"

// naive eager push: set(a) → recompute b → recompute d, then recompute c
//   → recompute d AGAIN (d computed twice; glitchy intermediate value)
// signals push/pull: set(a) pushes dirtiness to b,c,d (no compute);
//   reading d() pulls → b,c recompute if needed → d computes ONCE, correct.
// → glitch-free: no double-compute, no stale intermediate shown.

11. Common Mistakes

Traps
  • Sochna computed eagerly recompute hota — lazy hai (read pe hi).
  • Effect ko synchronous samajhna — scheduled (microtask), batched.
  • Signal ko function ki tarah call na karna (x() vs x) — read = dependency register.
  • Computed ke andar side effects — computed pure hona chahiye (effect use karo).

Do this

  • Side effects → effect (not computed); auto-cleanup use karo.
  • Immutable updates (naya reference) ya custom equal — change detect ho.
  • Templates mein signals read karo — live-consumer CD (OnPush+signals, Ch18 S-02).
  • Push/pull + memoization pe bharosa — over-optimize mat karo.

13. Performance

Signals internals hi unki perf hain: (1) Lazy computed — sirf read pe + dep-change pe recompute (memoized) — wasted computation nahi. (2) Push/pull glitch-free — diamond problem mein ek hi recompute, no double work. (3) Equality check — same value pe consumers notify hi nahi — no wasted CD. (4) Fine-grained CD — signal change sirf consuming views ko dirty marks (targeted, S-10), poora tree nahi. (5) Batched effects — multiple sync changes → ek effect run. Isliye signals + OnPush/zoneless (Ch16/Ch18) = Angular ki most efficient reactivity.

15. Interview Questions

EASY Signals internally kaise organized hain?

Ek reactive dependency graph ke roop mein. Har node ek ReactiveNode hai jo producer (value jo change notify kare — signal, computed), consumer (jo depend kare — effect, template, computed), ya dono (computed) ho sakta. Jab consumer producer ko read karta (reactive context mein), unke beech bidirectional edge banti. Signal change hone pe consumers ko notify kiya jaata (dirty marked), aur reads pe values pulled/recomputed hote. Ye graph fine-grained reactivity deta — exactly woh cheezein update hoti jo changed value pe depend karti.

HARD Signals "glitch-free" kaise hote? Push/pull algorithm explain karo.

Signals ek push/pull algorithm use karte, do phases mein. Jab ek source signal set hoti: Phase 1 (eager PUSH) — "dirtiness" turant downstream graph mein propagate hoti, dependent nodes (computed, template, effect) ko "tum shayad stale ho" mark karti — par koi recomputation nahi. Phase 2 (lazy PULL) — jab koi consumer apna value actually read karta (template CD, effect run), tab woh apne dependencies ki valueVersion check karta, aur sirf jo actually badla usko recompute karta. Ye "diamond problem" solve karta: agar ek node D, B aur C dono pe depend karta jo dono A pe depend karte, naive eager approach D ko do baar compute karega (glitchy intermediate). Push/pull mein A set → dirtiness B,C,D ko push (no compute); D read → B,C recompute if needed → D ek hi baar compute, correct final value. No double-compute, no stale intermediate shown = glitch-free.

MEDIUM computed lazy aur memoized kaise hai?

Lazy: computed creation pe evaluate nahi hota — uska computation function tabhi chalta jab koi consumer uska value read karta (total()). Agar koi read na kare, kabhi compute nahi hota. Memoized: ek baar compute hone pe result cache hota; dobara read pe, agar uske dependencies mein se koi nahi badla (valueVersion check), toh cached value return hota — recompute nahi. Recompute sirf tab jab (a) koi dependency badla AND (b) value read ho. Isse expensive derivations har CD pe re-run hone se bach jaate — bahut efficient derived state.

SCENARIO Ek computed mein tum ek object signal se derive karte ho, par mutation ke baad computed update nahi hota. Internally kyun?

Signals default equality ke liye Object.is use karte hain. Jab tum object ko mutate karte ho (obj.field = x) aur signal.set(obj) karte ho (ya set hi nahi karte), reference wahi rehta — Object.is(old, new) true return karta, toh signal sochta value "same" hai aur consumers ko notify hi nahi karta (no valueVersion bump). Isliye computed (jo us signal ka consumer hai) ko dirtiness push nahi hoti, woh cached value hi deta. Fix: immutable update — naya reference banao (signal.set({ ...obj, field: x })) taaki Object.is different dekhe aur change propagate ho; ya signal creation pe custom equal function do jo structural comparison kare. Immutability signals ke saath key hai.

Hydration Internals

Covers: ngh markers · serialized view info · DOM claiming · non-destructive hydration · incremental hydration · DEHYDRATED_BLOCK_REGISTRY · event replay (jsaction) · @defer hydrate · mismatch handling

1. What is it? (Kya hai ye?)

Hydration woh process hai jismein client-side Angular server-rendered (SSR) static HTML ko "activate" karta — usse re-create karne ke bajaye existing DOM ko reuse (claim) karke, aur usme interactivity (event listeners, bindings) add karke. Internally ye serialized view info (DOM pe ngh markers) use karta taaki client server ke DOM structure ko match karke usse adopt kar sake.

2. Why do we need it? (Zaroorat kyun hai?)

SSR fast content + SEO deta (server HTML). Par bina hydration, client us HTML ko throw karke poora phir se render karega (destructive) — flicker, wasted work, lost server benefit. Hydration non-destructive reuse deta: server ka DOM rehta (no flicker), client bas usse interactive banata. Aur incremental hydration isse selective karke JS aur kam karta.

3. What problem does it solve?

Destructive re-rendering after SSR. Naive client bootstrap server DOM ko ignore karke re-create karta — flicker + double work. Hydration server DOM ko claim karke ye eliminate karta. Incremental hydration ek aur problem solve karta: full hydration poore app ka JS ek saath chahiye (slow TTI) — incremental sirf zaroori parts hydrate karta.

4. Internal Working — markers, claiming, incremental, event replay

Serialize on server, claim on client, hydrate incrementally, replay events

Serialization (server): SSR ke dauraan, Angular HTML render karke uske saath serialized hydration info bhi bhejta — DOM nodes pe ngh (Angular hydration) attributes annotate karta jo ek serialized data structure (ek script mein, TransferState-style) ko reference karte. Ye info batati view structure: kitne nodes, containers kahan, template boundaries — taaki client match kar sake. DOM claiming (client): jab client hydrate karta (provideClientHydration()), woh apna create pass (S-05) chalata par document.createElement ke bajaye existing server DOM nodes ko "claim" (adopt) karta — ngh info se guide hoke. Ye non-destructive hai: DOM nodes wahi rehte (no flicker/re-create), Angular bas unpe LView bind karke listeners/bindings attach karta. Incremental hydration (withIncrementalHydration(), v19): @defer (hydrate on ...) se app "hydration zones" mein bat-ta. Deferred block ka content SSR-render hota (placeholder nahi, actual content) par uski hydration (interactivity) trigger (idle/viewport/interaction) tak deferred — woh block "dehydrated" rehta. Ek DEHYDRATED_BLOCK_REGISTRY in dehydrated blocks ko track karta. Event replay (withEventReplay(), v18, v19 SSR default; incremental hydration mein auto-included): hydration se pehle, ek global event listener (document root pe) user events capture karta — SSR-rendered elements pe jsaction attributes mark karte kaunse events capture karne — aur hydration complete hone pe woh captured events replay hote. Isse pehle-click-before-hydration lost nahi hota. Mismatch: agar client render server DOM se differ kare (different markup), hydration mismatch error → Angular us part ko re-render (fallback). Guard: browser-only APIs (window/document) ko isPlatformBrowser/afterNextRender se guard karo, warna server/client divergence.
