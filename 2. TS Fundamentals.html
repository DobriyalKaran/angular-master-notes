S-01

Type System Fundamentals

Covers: Structural typing · Type inference · any vs unknown vs never vs void · Narrowing · strict mode · Type erasure

1. What is it?

TypeScript = JavaScript + ek static type layer jo sirf compile time pe exist karti hai. Types code ke shapes ka contract hain — "is variable mein aisa data hoga" ka promise jo compiler enforce karta hai. Run hone se pehle saare types erase ho jaate hain — browser ko pure JS milta hai. TS runtime nahi badalta, sirf likhne ka experience aur safety badalta hai.

2. Why do we need it?

Java background se aa rahe ho toh ye argument already jeete hue ho: compiler ko galti pakadne do, user ko nahi. JS mein user.nmae (typo) runtime pe undefined deta hai — production mein, kisi customer ke browser mein. TS mein wahi typo editor mein red underline hai, commit hone se pehle. Angular ne TS ko day-1 se isi liye adopt kiya: framework-scale codebase bina types ke maintain karna impossible tha.

3. What problem does it solve?

  • Runtime type errors → compile-time errors — bugs ki sabse badi category shift-left.
  • Documentation that can't lie — function signature hi contract hai; comments purane ho jaate hain, types nahi (compile hi nahi hoga).
  • Refactoring confidence — field rename karo, compiler har broken usage dikha dega. 500-file Angular app mein ye superpower hai.
  • Tooling — autocomplete, go-to-definition, safe renames — sab type information se aate hain.

4. Internal Working

Structural typing (duck typing with a compiler): Java nominal hai — type ka naam matter karta hai (implements likhna padta hai). TS structural hai — shape matter karti hai. Agar object mein required properties hain, wo type se compatible hai, chahe usne kabhi us interface ka naam suna bhi na ho.

interface Point { x: number; y: number; }
const p = { x: 1, y: 2, label: 'home' };
const pt: Point = p;   // ✅ shape match — extra props variable assignment mein OK
// Java hota toh: "p Point implement nahi karta" — TS ko naam se matlab nahi

Inference + widening: compiler expressions se types khud nikaalta hai. let s = 'hi' → type string (widened — badal sakta hai). const s = 'hi' → type 'hi' (literal — badal hi nahi sakta, toh narrow rakha).

Control flow narrowing: compiler tumhare if/typeof/instanceof checks ko padhta hai aur har branch mein type ko narrow kar deta hai — runtime checks se compile-time knowledge.

Diagram — The Big Four special types

9. Advanced Example — never se exhaustiveness checking

type PaymentStatus = 'created' | 'authorized' | 'captured' | 'failed';

function statusLabel(s: PaymentStatus): string {
  switch (s) {
    case 'created':    return 'Order placed';       // L1
    case 'authorized': return 'Payment authorized';
    case 'captured':   return 'Payment complete';
    case 'failed':     return 'Payment failed';
    default: {
      const exhaustive: never = s;                 // L2
      throw new Error(`Unhandled: ${exhaustive}`); // L3
    }
  }
}

10. Code Explanation — Line by Line

  • L1: Har case ke andar s ka type narrow hai — sirf wo ek literal. Control flow analysis in action.
  • L2: Magic line. Agar saare cases handle ho gaye, toh default mein s ka type never hai (kuch bacha hi nahi) — assignment ✅. Kal koi 'refunded' status add kare union mein, toh default mein s ka type 'refunded' hoga — never ko assign nahi hota — compile error exactly us switch pe jo update karna bhoola.
  • L3: Runtime safety net bhi — defense in depth. Ye pattern Razorpay-clone ke payment states mein literally life-saver hai; production mein naya status aaya aur UI handle karna bhool gaye wala bug compile-time pe mar jaata hai.

11. Common Mistakes

❌ Galtiyan
  • any ko "temporary fix" bolke daalna — any infectious hai; usse nikli har value bhi unchecked. Legacy boundary pe bhi unknown use karo.
  • as casting se compiler ko chup karana: (data as User).name — compiler ko jhooth bola, runtime pe sach samne aayega. as = "mujhe tumse zyada pata hai" ka claim; 90% baar galat hota hai.
  • ! (non-null assertion) ki barsaat: user!.profile!.email! — teen jagah "trust me bro". Optional chaining + narrowing use karo.
  • Har cheez pe explicit annotation likhna: const x: number = 5 — noise hai; inference ko kaam karne do. Annotate karo boundaries pe: function params, return types (public APIs), class fields.

12. Best Practices

✅ Production rules
  • strict: true non-negotiable — naya Angular project default deta hai; legacy migrate karna ho toh flag-by-flag.
  • External data (API responses, localStorage, URL params) ko unknown maano aur boundary pe validate karo — types compile-time promise hain, network runtime reality hai.
  • Return types public functions pe explicit likho — implementation change hone pe contract accidentally na badle.
  • satisfies operator (TS 4.9+): validation without widening — const config = {...} satisfies AppConfig — check bhi hua, literal types bhi bache.

13. Performance

Runtime cost: zero — types erase hote hain, bundle mein ek byte nahi jaata. Asli cost build time pe hai: complex types (Section 8 wale) type-checker ko slow kar sakte hain. Angular CLI isliye type-checking ko alag process mein aur incremental rakhta hai. Team-scale tip: tsc --noEmit --extendedDiagnostics se check time profile hota hai; giant unions (1000+ members) aur deeply recursive conditional types hi usually culprits hote hain.

14. Edge Cases

  • Excess property check: object literal ko directly assign karo toh extra properties error hain (const p: Point = {x:1, y:2, z:3} ❌), par variable ke through allowed (upar wala example ✅). Fresh literals pe TS strict, purani values pe structural.
  • const obj = { status: 'active' } — status ka type string hai (widened), 'active' nahi! Union expect karne wali jagah fail hoga. Fix: as const ya satisfies.
  • typeof null === 'object' — JS ka historic bug TS narrowing mein bhi bite karta hai; typeof x === 'object' null exclude nahi karta.
  • Empty interface {} ka matlab "empty object" nahi — "kuch bhi non-null" hai. const x: {} = 42 ✅ (surprise!). Object chahiye toh Record<string, unknown>.

15. Interview Questions

EASY any aur unknown mein difference — kab kaunsa?

Dono "kuch bhi ho sakta hai" bolte hain, par any checking band kar deta hai (kuch bhi karo, koi error nahi) jabki unknown use karne se pehle narrowing force karta hai. Rule: incoming untyped data = unknown; any = last resort with eslint-disable comment jaisa shame.

EASY TypeScript runtime pe kya karta hai?

Kuch nahi — trick question hai! Types compile pe erase; browser pure JS chalata hai. Isi liye interfaces pe instanceof nahi lagta aur API response apne aap validate nahi hota. Runtime validation chahiye toh code likhna padta hai (type guards, zod jaisi libs).

MEDIUM Structural vs nominal typing — aur iska ek practical consequence?

Nominal (Java): compatibility naam/declaration se. Structural (TS): shape se — same properties = compatible, bina kisi implements ke. Consequence: (1) test mocks banana easy — poori class implement kiye bina shape-compatible object kaafi; (2) gotcha: do logically-alag types same shape ke hon (UserId = string, OrderId = string) toh interchangeable ho jaate hain — branded types se fix hota hai.

MEDIUM never type ke 3 real uses batao.

(1) Exhaustiveness check — switch ke default mein never assignment, union badhne pe compile error. (2) Functions jo return hi nahi karte (throw-only, infinite loop) — callers ko pata hai aage code unreachable. (3) Conditional types mein "is branch ko union se hatao" ka tool — Exclude internally never return karke members filter karta hai (Section 8).

HARD Type widening kya hai, kab hota hai, aur as const exactly kya karta hai?

Widening = literal type ko general banane ka inference behavior: let x = 'a' mein 'a' → string, kyunki let reassignable hai. const primitives pe literal preserve hota hai, par object properties phir bhi widen hoti hain (object mutable hai). as const teen kaam karta hai: har property literal type, sab readonly, arrays → readonly tuples. Isi se const ROLES = ['admin','user'] as const + typeof ROLES[number] = 'admin' | 'user' — single source of truth pattern (Section 3 mein enum alternative yahi hai).

SCENARIO API response type mismatch se production bug aaya — TS ne kyun nahi pakda, aur process fix kya hoga?

Root cause samjhao: this.http.get<User>(url) mein User sirf assertion hai — HttpClient runtime pe kuch validate nahi karta; backend ne field rename kiya, TS khush, UI toota. Fixes ladder: (1) boundary validation — zod/valibot schema se parse, ya haath ka type guard; (2) contract source of truth — OpenAPI spec se types generate (backend-frontend drift structurally impossible); (3) CI mein contract tests. Punch line: "types compile-time contract hain; network boundary pe runtime enforcement lagana architecture ka kaam hai" — ye framing SDE-3 level hai.

S-02

Interfaces & Type Aliases

Covers: interface vs type · extends vs intersection · Declaration merging · Optional/readonly · Index signatures · Unions

1. What is it?

Interface = object shape ka named contract. Type alias = kisi bhi type ka naam — object shape, union, primitive, tuple, function, conditional... sab kuch. Interface sirf object-ish shapes describe karta hai; type alias poore type system ka naming tool hai. Dono runtime pe erase (Section 1 yaad hai).

2. Why do we need it?

Bina named types ke har function signature mein {id: number; name: string; email: string} repeat hota — DRY violation type level pe. Named types se: (1) domain vocabulary banti hai (Order, PaymentIntent — code business language bolta hai), (2) ek jagah change, har jagah propagate, (3) IDE mein hover karo aur poora contract dikhe.

3. What problem does it solve?

  • Shape duplication → single source of truth.
  • API contracts — frontend-backend ke beech ka agreement code mein likha hua.
  • Union modeling (type alias exclusive) — "ye value in 4 mein se ek hai" ko first-class banana; discriminated unions impossible states ko unrepresentable bana dete hain.

4. Internal Working — interface vs type ka asli difference

Declaration merging internals: compiler same scope mein same naam ke saare interface declarations ko ek combined type maanta hai. Ye "feature" nahi, mechanism hai jisse poora .d.ts ecosystem chalta hai — Window pe property add karna, library types augment karna (Section 9 mein full use).

11. Common Mistakes

❌ Galtiyan
  • Optional ? ka overuse — "pata nahi hoga ya nahi" ko model karne ke bajaye har field optional kar dena. Result: har usage pe ?. ki barish aur asli contracts kho gaye. API truly optional bheje tabhi ?.
  • readonly ko runtime freeze samajhna — sirf compile-time hai (Ch1 ke const jaisa). Runtime immutability = Object.freeze ya discipline.
  • Discriminant field bhoolna aur union ke members ko manually narrow karne mein as lagana — discriminated union banao, narrowing free milegi.
  • Boolean flags se state model karna (isLoading + isError + data?) — impossible states representable. Interviewer ke saamne discriminated union bolo, seniority dikhegi.
  • Ek hi file mein galti se same naam ke do interfaces — merging chup-chaap ho jaayegi, "kahan se aayi ye property" ka confusion.

12. Best Practices

✅ Production rules
  • Team convention > dogma. Common sane default: object shapes/public contracts = interface (better errors, merging for augmentation), unions/tuples/derived types = type (option bhi nahi hai).
  • API models ek models/ folder mein, backend ke saath review hote hue — ya OpenAPI se generated.
  • Server-owned fields readonly; state machines discriminated unions.
  • Interface ko chhota rakho — 15+ fields ka interface aksar 2-3 concepts ko mila raha hota hai; compose karo (extends / intersection).

13. Performance

Compile-time angle: extends (interface) compiler ke liye cached, named relationship hai; bade intersection chains (A & B & C & ...) har use pe re-computed flatten hote hain — massive codebases mein interfaces marginally faster check hote hain. TS team ki khud ki guidance: prefer interfaces for object shapes jab dono kaam karein. Runtime: dono zero cost, obviously.

14. Edge Cases

  • Intersection mein same prop alag types: {x: string} & {x: number} → x: never — silently! Value banana impossible, error tab jab koi assign kare. extends yahi cheez declaration pe hi pakad leta.
  • Interface class ko bhi extends kar sakta hai — class ki instance shape le leta hai (rare but legal).
  • Optional (tags?: string[]) vs explicit undefined (tags: string[] | undefined) alag hain: pehle mein key absent ho sakti hai, dusre mein key deni hi padegi (value undefined sahi). exactOptionalPropertyTypes flag isko aur strict karta hai.
  • Function type ko interface se likha ja sakta hai (callable signature) — arrow-style alias zyada readable, par overloads model karne ke liye interface style better (multiple call signatures).

15. Interview Questions

EASY Interface vs type — 3 concrete differences.

(1) type unions/tuples/primitives naam de sakta hai, interface sirf object-ish shapes; (2) interface declaration merging karta hai, type duplicate = error; (3) extend style: extends vs &, aur conflicts pe extends clear error deta hai jabki intersection never bana deta hai.

EASY readonly interface property runtime pe mutation rokti hai?

Nahi — compile-time only. Types erase hote hain. Runtime immutability ke liye Object.freeze (shallow) ya immutable update discipline (Ch1 Section 4). Ye cross-questioning trap hai — confidently "nahi" bolo with reason.

MEDIUM Discriminated union kya hai aur "make impossible states unrepresentable" ka matlab?

Union jisme har variant ek common literal field (discriminant) carry karta hai; us field pe check karo toh compiler poora variant narrow kar deta hai. Boolean-flags model mein loading+data+error independent hain — 2³ combinations, mostly invalid. Union model mein sirf valid states ka type exist karta hai; invalid state likhi hi nahi ja sakti. Bugs ko impossible banana > bugs ko pakadna.

MEDIUM Declaration merging kab kaam aati hai — 2 real cases?

(1) Global augmentation: declare global { interface Window { Razorpay: RazorpayConstructor } } — checkout SDK window pe aata hai, types se officially batao (Section 9 mein poora). (2) Library augmentation: kisi lib ke interface mein apni custom property add karna (e.g., Express Request pe user field — Node side). Merging hi wo mechanism hai jo ye bina source badle possible banata hai.

HARD {x: string} & {x: number} kya banta hai aur kyun error nahi aata declaration pe?

x ka type string & number = never (koi value dono nahi ho sakti). Intersection lazy structural operation hai — compiler declaration pe sirf type banata hai, invalid hona tab pata chalta hai jab koi use/assign kare, aur error message door hoke confusing hota hai. Isi liye object hierarchies mein extends prefer: conflict declaration site pe hi loud error. Ye nuance batao toh interviewer ko pata chal jaata hai tumne production TS likha hai.

SCENARIO Backend team API response mein naye optional fields add karti rehti hai aur kabhi-kabhi fields rename kar deti hai. Frontend types kaise architect karoge ki drift jaldi pakda jaye?

(1) Types ko haath se sync karna band — OpenAPI/Swagger spec se codegen (single source of truth); rename = generated type change = compile errors sab affected jagah. (2) Boundary validation runtime pe (Section 1 wala zod/guard) — kyunki "optional add karna" backward-compatible hai par rename breaking hai, guard failed parse ko monitoring mein bhejta hai. (3) Types ko readonly + narrow banao, app-internal models ko API models se alag rakho (mapping layer) — API churn app types tak na phaile. (4) Contract tests CI mein. Layered answer = system thinking.

S-03

Enums (aur unke Modern Alternatives)

Covers: Numeric/String enums · Reverse mapping · const enum · Generated JS · Literal union + as const alternative · Template usage

1. What is it?

Enum = named constants ka fixed set. TS ki gini-chuni features mein se ek jo runtime code generate karti hai — baaki sab types erase hote hain, enum ek asli JS object banata hai. Isi "special" hone ki wajah se ye TS community ka sabse debated feature bhi hai.

2. Why do we need it?

Magic values ka ilaj: if (order.status === 2) — 2 kya hai? Kisko pata. Named constants readable + typo-proof + refactor-safe hote hain. Java devs ke liye enum ghar jaisa lagta hai — lekin TS mein literal unions aksar wahi kaam better karte hain, aur ye trade-off samajhna hi is section ka asli maal hai.

3. What problem does it solve?

  • Magic numbers/strings → self-documenting names.
  • Valid values ka closed set — OrderStatus type ke bahar ki value assign nahi hoti.
  • Centralized change — value badli toh ek jagah.

4. Internal Working — Generated JS dekho (yahi interview ka gold hai)

// TS:
enum Direction { Up, Down }        // numeric: Up=0, Down=1

// Compiled JS — IIFE se object banta hai:
var Direction;
(function (Direction) {
  Direction[Direction["Up"] = 0] = "Up";      // DOUBLE assignment!

// Result: { Up: 0, Down: 1, 0: "Up", 1: "Down" }  ← REVERSE MAPPING

String enums (enum Status { Active = 'ACTIVE' }) reverse mapping nahi banate — sirf one-way object. const enum object banata hi nahi — compiler har usage ko literal value se inline kar deta hai (Direction.Up → 0), zero runtime footprint. S-04

Functions in TypeScript

Covers: Function types · Optional/default/rest params · Overloads · this parameter · void ka secret · Contextual typing

1. What is it?

Function type = parameters + return ka contract: (q: string) => Promise<Result[]>. TS functions ko first-class types deta hai — Chapter 1 mein seekha functions values hain; ab un values ke bhi types hain. Overloads ek function ke multiple call-shapes declare karte hain; this parameter batata hai function kis context pe chalega.

2. Why do we need it?

Angular/RxJS code functions-as-arguments se bhara hai: operators, callbacks, effects, guards. In sab jagah callback ka galat shape = subtle runtime bug. Typed function signatures se: callback ke params auto-typed milte hain (contextual typing — map(order => ...) mein order ka type likha nahi, mila), aur galat callback compile pe hi reject.

3. What problem does it solve?

  • Callback contracts — "is function ko kya milega, kya lautana hai" enforce hota hai.
  • API ergonomics — overloads se ek function alag input-shapes pe alag precise return types de sakta hai.
  • this ki safety — Chapter 1 ka this-loss ab compile-time pe pakda ja sakta hai.

4. Internal Working

Contextual typing: jab function expected type wali jagah likha jaata hai, compiler params ke types expected signature se ulta infer karta hai. arr.map(x => x.price) — x ka type map ke declared signature se aaya. Isi liye inline callbacks pe annotations nahi likhne padte.

Parameter bivariance vs contravariance: method-style declarations (onClick(e: MouseEvent): void) params ko bivariantly check karti hain (loose — DOM APIs practical rahein isliye); arrow-style properties (onClick: (e: MouseEvent) => void) strictFunctionTypes ke saath contravariantly (sound). Ek line mein yaad: function jo KAM specific input accept kare wo substitute ho sakta hai, zyada specific nahi.

void ka secret: return type void ka contextual matlab hai "return value ignore hogi", "return mat karo" nahi. Isliye (x: T) => void expect karne wali jagah value-returning function bhi chalta hai — aur yahi design hai jisse arr.forEach(x => results.push(x)) legal hai (push number return karta hai!).

❌ Galtiyan

S-06

Generics

Covers: Generic functions/interfaces/classes · Constraints (extends) · keyof constraint · Defaults · Inference · Signal<T>, FormControl<T>, HttpClient

1. What is it?

Generic = type ka parameter. Jaise function values ke parameters leta hai, generic type-level pe parameters leta hai: function first<T>(arr: T[]): T — "koi bhi type T ho, T[] doge toh T milega". Ek baar likho, har type ke liye precisely typed — any wala compromise nahi, per-type duplication bhi nahi.

2. Why do we need it?

Generics ke bina do hi raste the: (a) any — safety gayi (first(users) se any milta, aage ka sab unchecked), ya (b) har type ka alag function — firstUser, firstOrder... DRY ki maut. Generic relationship preserve karta hai: "jo andar gaya, wahi type bahar aayega". Angular tumhe roz generics consume karwata hai: signal<number>(0), http.get<Order[]>(url), FormControl<string> — ye section tumhe producer bana dega.

3. What problem does it solve?

  • Type-safe reuse — ek container/utility, saare types.
  • Input↔output linkage — return type input pe depend karta hai, compiler ye rishta track karta hai.
  • API precision — library functions callers ke types ke saath "morph" ho jaate hain.

4. Internal Working

Inference: call pe compiler arguments se type params solve karta hai — first([1,2,3]) mein T[] ↔ number[] match karke T = number. Explicit first<number>(...) tabhi jab inference ambiguous ho ya tum override karna chaho. Multiple candidates milein (e.g., T do params mein) toh best common type banta hai ya error.

Constraints (extends): bina constraint ke T ke andar compiler kuch assume nahi kar sakta — t.length ❌ kyunki T number bhi ho sakta hai. T extends { length: number } bolke tum compiler ko capability ki guarantee dete ho — ab andar .length ✅ aur bahar sirf compatible types accepted. Constraint = "T kuch bhi ho, par kam se kam ye ho".

Erasure (Java se important difference): TS generics puri tarah erase — runtime pe T naam ki koi cheez nahi. Java ki tarah bhi reflection se nahi milta, C# ki tarah reified toh bilkul nahi. Consequence: if (x instanceof T) ❌; runtime pe type chahiye toh value pass karo (constructor, validator fn).

11. Common Mistakes

❌ Galtiyan

  • <T>(x: T) likhna jab (x: unknown) kaafi tha — agar return input pe depend nahi karta toh generic ceremony hai.
  • Constraint ke bina T pe operations maangna, phir as any se "fix" — constraint likho, wahi toh contract hai.
  • Runtime pe T use karne ki koshish (new T(), instanceof T) — erased! Constructor chahiye toh param mein lo: create<T>(Ctor: new () => T): T.
  • Java instinct: har class ko generic banana. TS mein functions + inference itne powerful hain ki generic classes kam lagti hain — zaroorat pe hi.

12. Best Practices

✅ Production rules
  • Naming: T/K/V chhote scopes mein theek; complex generics mein descriptive — TEntity, TResponse, EventMap.
  • Inference ko kaam karne do — explicit type args tabhi jab zaroori (empty array, override).
  • Defaults se ergonomic APIs: Repository<T, ID = string> — common case free, special case possible.
  • Constraint minimum rakho — jitna kam maangoge, utne zyada types ke saath reusable.

13. Performance

Runtime: zero — na code duplicate hota hai (C++ templates jaisa nahi), na dispatch cost. Compile: har alag instantiation cache hoti hai; problem sirf deeply recursive generics (Section 8 ke DeepX types) ya massive union × generic combinations pe — "Type instantiation is excessively deep" error wahi hai. App-level code mein kabhi issue nahi hoga; library-level type gymnastics mein depth budget yaad rakhna.

14. Edge Cases

  • T extends string mein T literal ho sakta hai ('hello') — string se narrow. Constraints upper bound hain, exact type nahi.
  • Generic ke andar conditional narrowing tricky: function f<T extends string | number>(x: T) mein typeof x === 'string' se x narrow hota hai par T nahi — return T ho toh casts lag sakte hain (known design friction).
  • Default type param inference ko replace nahi karta — inference fail hone pe hi default lagta hai.
  • const type parameter (TS 5.0): function f<const T>(x: T) — arguments as-const jaise infer hote hain, literals preserve. Modern API design tool.
  • Variance annotations in/out (TS 4.7) — library-level; interfaces pe variance explicitly declare karne ke liye. Naam jaano, use rare.

1. What is it?

Utility types = TS ke built-in type transformers — types jo types lete hain aur naye types return karte hain (type-level functions!). Aur type operators (keyof, typeof, T[K]) wo primitives hain jinse ye transformers bane hain. Sab lib.d.ts mein plain TS code hain — koi compiler magic nahi, Section 8 ke mapped/conditional types se likhi gayi ordinary definitions.

2. Why do we need it?

Real apps mein ek entity ke kai roop hote hain: full User, create-form ka user (id nahi), patch-request ka user (sab optional), list-view ka user (3 fields). Har roop ka alag interface haath se likhna = duplication + drift (ek mein field add hua, baaki bhool gaye). Utility types se sab roop ek source se derive hote hain — source badla, sab roop khud update, mismatch compile error.

3. What problem does it solve?

  • Type drift — derived types kabhi out-of-sync nahi hote.
  • DTO boilerplate — Create/Update/Patch shapes free mein.
  • Values↔types bridge — typeof se runtime constants ke types nikalte hain (config, as-const objects).

4. Internal Working — The operators (building blocks)

11. Common Mistakes

❌ Galtiyan
  • Partial<T> ko "sab kuch chalega" mode banana — service internals tak Partial phailaya toh har jagah undefined checks. Partial sirf boundary shapes (patch DTO) pe; andar full types.
  • Omit ki keys unchecked hain (K extends keyof any): Omit<Order, 'idd'> — typo, koi error nahi, kuch omit hi nahi hua! Strict version khud banao: type StrictOmit<T, K extends keyof T> = Omit<T, K>.
  • Record<string, V> har jagah — keys ka union pata hai toh use likho, exhaustiveness + autocomplete dono milenge. Record<string, V> lookup pe har value V deta hai, missing key ka jhooth (noUncheckedIndexedAccess flag isi ke liye hai).
  • Derived types ko copy-paste expand karke commit karna — "IDE ne expand karke dikhaya toh maine likh diya" — drift wapas aa gaya.
  • Deep expectations: Partial/Readonly shallow hain — nested objects untouched. Deep versions Section 8 mein khud banayenge.

✅ Production rules

13. Performance

Compile-time only; evaluation lazy + cached. 14. Edge Cases

  • keyof union pe = common keys only (keyof (A | B) = keys jo dono mein hain); intersection pe sab keys. Ulta lagta hai pehli baar — value-assignability se socho toh sahi hai.
  • Partial<T> + exactOptionalPropertyTypes: {name: undefined} explicitly dena vs key absent hona alag treat hota hai.
  • Pick/Omit discriminated unions ko flatten kar dete hain (union pe directly lagao toh distribution nahi hoti default mein) — union-safe versions ke liye distributive wrapper chahiye (S8).
  • ReturnType<typeof overloadedFn> sirf aakhri overload ka return deta hai — overloads + utility types ka known friction.
  • Readonly<T> methods ko nahi chhoota; aur readonly T[] pe push/pop types se hi gayab hote hain — mutation compile pe hi impossible (Ch1 S4 immutability ka type-level enforcement).

15. Interview Questions

EASY Pick aur Omit mein kya farak, aur dono ka ek-ek use case?

Pick = whitelist (ye keys rakho): list-item views. Omit = blacklist (ye keys hatao): create DTO jisme server-generated id/createdAt nahi. Chhota surface chahiye toh Pick; "sab kuch except" toh Omit. Bonus: Omit = Pick + Exclude composition.

EASY TS ka typeof JS ke typeof se kaise alag hai?

JS typeof runtime operator hai — string deta hai ('object', 'string'...). TS typeof type position mein compile-time operator hai — value ka poora static type deta hai. type C = typeof appConfig compile pe evaluate, runtime pe exist hi nahi.

MEDIUM Partial<T> ki definition likho aur -? modifier kya karta hai?

type Partial<T> = { [K in keyof T]?: T[K] } — mapped type jo har key pe ? add karta hai. -? ulta: optionality remove (Required ki definition). Similarly -readonly readonly hatata hai. Modifiers add (+, implicit) ya remove (-) dono ho sakte hain — mapped types ka control panel.

MEDIUM Record<Status, Meta> vs { [k: string]: Meta } — exhaustiveness angle se compare karo.

Union-keyed Record: saari keys required — missing status = compile error (S3 wala table pattern), extra key bhi error, lookup ka result guaranteed Meta. String index signature: koi bhi key, kuch bhi missing, lookup pe Meta ka jhootha vaada (undefined possible). Bounded domain = Record with union; truly dynamic = index signature + noUncheckedIndexedAccess.

HARD Omit<T, K> typos kyun nahi pakadta jabki Pick pakadta hai — aur fix?

Definitions dekho: Pick ka K extends keyof T — invalid key constraint fail = error. Omit ka K extends keyof any (i.e., string|number|symbol) — deliberately loose rakha gaya (union se omit, related types pe reuse ke liye). Cost: Omit<Order,'idd'> silently no-op. Fix: type StrictOmit<T, K extends keyof T> = Omit<T, K> — apne codebase mein yahi export karo. Ye definition-level answer "maine lib.d.ts padha hai" signal deta hai.

SCENARIO Form draft feature: user aadha form bhar ke save kare, baad mein resume. Types kaise design karoge ki draft aur final submission alag contracts hon?

(1) Canonical CheckoutForm interface (full, required). (2) Draft = Partial<CheckoutForm> — storage/API dono mein; nested sections hain toh DeepPartial (S8) ya section-wise Partials: { address?: Partial<Address>; payment?: Partial<Payment> } — deliberate structure blind DeepPartial se better. (3) Submit boundary pe narrow: ek validator/type-guard isComplete(draft): draft is CheckoutForm — guard pass hone ke baad hi submit fn (jo full type maangta hai) call ho sakta hai; "incomplete submit" compile-time pe unrepresentable. (4) Server pe bhi wahi validation (types wire cross nahi karte — S1 ka mantra). Ye Partial + type guard + boundary validation ka combo hai — teen sections ka gyaan ek answer mein.

S-08

Advanced Types — Mapped, Conditional, infer, Template Literals

Covers: Mapped types + key remapping (as) · +/- modifiers · Conditional types · Distributivity · infer · Template literal types · DeepPartial

1. What is it?

Ye TS ka type-level programming toolkit hai: mapped types = types pe loop (for key in keys), conditional types = types pe if-else (T extends U ? X : Y), infer = types ke andar se pattern-match karke hissa nikaalna, template literal types = string types pe concatenation/pattern. Milake: ek chhoti functional language jo types compute karti hai — Section 7 ke saare utilities isi se bane the.

2. Why do we need it?

Kyunki real APIs mein types ke beech mechanical relationships hote hain: "event object ki har key ka handler", "har field ka getter", "Observable ke andar kya hai", "route path se params". Ye relationships haath se maintain karo toh drift; type-level programming se compute karo toh hamesha sync. Interview reality: 4-5 YOE ke baad yahi section junior aur senior TS ko alag karta hai.

3. What problem does it solve?

  • Derived shapes at scale — Getters, EventHandlers, FormModel — sab source se computed.
  • Wrapper unwrapping — Promise/Observable/Array ke andar ka type nikaalna (infer).
  • String contracts — 'order:created' jaise patterns, route paths, CSS vars — typed strings.

4. Internal Working

A) Mapped types — loop + modifiers + remap

type Mapped<T> = {
  // [K in keyof T]  ← loop variable K, har key pe
  // -readonly / +?  ← modifiers add/remove
  // as NewKey       ← KEY REMAPPING (TS 4.1+)
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K]
};

interface Order { id: string; amount: number; }
type OrderGetters = Mapped<Order>;
// { getId: () => string; getAmount: () => number }  ← computed!

as never remap = key ko drop karna — filtered mapped types isi se bante hain (sirf function keys rakho, etc.).

B) Conditional types + distributivity (THE exam topic)

type IsString<T> = T extends string ? true : false;
  DISTRIBUTIVITY: naked type param + union = condition HAR member pe
  alag-alag chalti hai, results ka union banta hai:

  Exclude<'a'|'b'|'c', 'b'>
    = ('a' extends 'b' ? never : 'a')   → 'a'
    | ('b' extends 'b' ? never : 'b')   → never   (union se GAYAB)
    | ('c' extends 'b' ? never : 'c')   → 'c'
    = 'a' | 'c'          ← never union mein evaporate ho jaata hai!

  Band karna ho: [T] extends [U] ? ... : ...   (tuple-wrap = non-distributive)

C) infer — type pattern matching

type UnwrapPromise<T> = T extends Promise<infer U> ? U : T;
// "Agar T Promise<kuch> ke pattern se match kare, us 'kuch' ko U naam do"
type A = UnwrapPromise<Promise<Order>>;   // Order
type B = UnwrapPromise<number>;            // number (no match → fallback)

D) Template literal types

type EventName = `${'order'|'user'}:${'created'|'updated'}`;
// 'order:created'|'order:updated'|'user:created'|'user:updated' — cross product!
// + intrinsic helpers: Uppercase, Lowercase, Capitalize, Uncapitalize

5. Execution Flow — compiler ek advanced type kaise evaluate karta hai

  type R = ApiResult<typeof fetchOrders>
     │ 1. Alias expand — definition substitute
     │ 2. Conditional? → T resolve karo → union hai + naked param? → DISTRIBUTE
     │ 3. infer positions pe structural match — candidates collect
     │ 4. Mapped? → keyof compute → har key pe remap/modifier apply
     │ 5. Result cache (same inputs → same output, dobara compute nahi)
     ▼ Lazy: types tabhi evaluate jab USE hote hain — infinite-ish recursion
       tabhi pakdi jaati hai jab expand karni pade (depth limit ~ recursion guard)

6. Architecture — apne utility types ki library

// 1. DeepPartial — S7 ka shallow limitation fix (recursion!)
type DeepPartial<T> = {
  [K in keyof T]?: T[K] extends object ? DeepPartial<T[K]> : T[K];

10. Code Explanation — Line by Line

  • L1: Path extends string constraint — template matching sirf string types pe.
  • L2: Template pattern mein do infer — compiler string ko structurally todta hai: : ke baad param, / ke baad rest. Greedy nahi, pattern-guided.
  • L3: Recursion! Rest pe khud ko call + mapped type se dono params merge. Types mein loop nahi hota — recursion hi iteration hai (FP mindset).
  • L4: Base-ish case — aakhri param (aage koi / nahi).

11. Common Mistakes

❌ Galtiyan
  • Distributivity ko bhool jaana — IsString<'a' | 1> ko false expect karna; asal mein true | false = boolean milta hai (per-member!). Whole-union check chahiye toh [T] extends [string].
  • never input pe conditional se result expect karna — naked param pe never = empty union = distribute over nothing = result never. Famous head-scratcher.
  • DeepPartial ko sab pe maar dena — Date, Map, Observable bhi "object" hain, unke andar bhi ghus jaayega. Production DeepPartial mein in types ke liye early-return branches likho.
  • Type-level showoff — jo cheez do explicit interfaces se saaf ho jaati, uspe 5-level conditional. Error messages team ke liye punishment ban jaate hain. Power tool = zaroorat pe.
  • infer positions ka overconfidence — multiple infer candidates (covariant positions) union bante hain, contravariant intersection — is nuance ke bina complex infers debugging nightmare.

12. Best Practices

✅ Production rules
  • Har custom utility ke saath test types likho: type _t1 = Expect<Equal<ExtractParams<'/a/:id'>, {id: string}>> — type-level unit tests (type-challenges style).
  • Intermediate aliases + comments — 3+ operations ho toh naam do har stage ko; error messages aur compile perf dono sudhrenge.
  • Utilities ko types/ mein centralize; ad-hoc inline conditionals PR review mein flag karo.
  • Recursion pe depth-consciousness — tail-recursive patterns (accumulator param) TS 4.5+ mein zyada depth allow karte hain.

13. Performance

Ye wahi section hai jahan compile perf sach mein bigad sakti hai: template literal cross-products explode karte hain (`${A}-${B}-${C}` = |A|×|B|×|C| members — 3 unions of 100 = 1,000,000 types!), deep recursion "excessively deep" errors, aur giant distributive conditionals editor lag. Budget rules: unions chhote rakho jab cross ho rahe hon, recursion sirf bounded structures pe, aur tsc --extendedDiagnostics + TS server logs se hot types identify karo. Runtime: zero — chahe type kitna bhi zabardast ho, JS mein sirf plain code jaata hai.

14. Edge Cases

  • Homomorphic mapped types ([K in keyof T] direct) modifiers preserve karte hain — optional/readonly source se copy hote hain; non-homomorphic ([K in SomeUnion]) nahi. Isi se Partial nested optionality sahi rakh paata hai.
  • Mapped type arrays/tuples pe lagao toh (homomorphic case) array/tuple hi rehta hai — Readonly<string[]> = readonly string[], object nahi banta. Spec ka thoughtful corner.
  • infer X extends string (TS 4.7+) — inferred type pe constraint inline; extra conditional bacha deta hai.
  • Conditional types deferred rehte hain jab T unresolved generic ho — function body ke andar isliye kabhi-kabhi "assignability unknown" errors; resolution call site pe hoti hai.
  • Recursive conditional + union = dono combine: har member apni recursion chalata hai. Power aur perf-risk ka multiplication yahi hai.

EASY Mapped type kya hai — Partial ki definition se samjhao.

EASY infer keyword kya karta hai?

Conditional type ke extends pattern mein ek "capture variable" declare karta hai — match hua toh us position ka type us naam se milta hai. T extends Array<infer U> ? U : never = element type nikaalo. Type-level destructuring bolo toh bhi sahi hai.

MEDIUM Distributive conditional types — kya, kab, aur band kaise karte hain?

Band karna: dono sides tuple-wrap [T] extends [U] — ab whole-union comparison. Bonus: never input distributive mein never hi return karta hai (empty union).

MEDIUM ReturnType<T> khud likho aur samjhao.

type MyReturnType<T extends (...args: any[]) => any> = T extends (...args: any[]) => infer R ? R : never. Constraint ensure karta hai function hi aaye; pattern mein return position pe infer R capture; match guaranteed hai (constraint ki wajah se) par syntax conditional maangta hai. Follow-up ready: overloaded function pe last signature hi milti hai.

HARD DeepPartial likho aur uske 3 production pitfalls batao.

type DeepPartial<T> = { [K in keyof T]?: T[K] extends object ? DeepPartial<T[K]> : T[K] }. Pitfalls: (1) built-ins bhi "object" hain — Date/Map/Set/Observable ke andar ghus jaata hai; special-case branches chahiye (T extends Date ? T : ...); (2) functions object extend karte hain — methods corrupt ho sakte hain; (3) arrays: element optional karna hai ya nahi — design decision, naive version (T[K])[number] semantics galat kar sakta hai; (4th bonus) recursion depth on huge models. Isliye libraries (type-fest) ka battle-tested DeepPartial use karna bhi valid engineering answer hai.

(1) Source of truth: interface EventPayloads { order_created: {...}; user_login: {...} }.   • Untyped JS boundary — declaration files se koi bhi JS library typed ho jaati hai.
  • Global scripts — declare global se window-injected SDKs officially typed.
  • Metadata attach karna — decorators class ke saath configuration jodte hain jo framework padh sake.
  • Type-only dependencies — import type se circular/runtime issues ke bina types share.

9. Advanced Example — apna decorator likhna (demystify!)

// TC39-style method decorator — performance logging
function measured<This, Args extends unknown[], R>(          // L1
  original: (this: This, ...args: Args) => R,
  context: ClassMethodDecoratorContext<This>                  // L2
) {
  return function (this: This, ...args: Args): R {           // L3
    const t0 = performance.now();
    try { return original.call(this, ...args); }             // L4
    finally {
      console.log(`${String(context.name)}: ${(performance.now() - t0).toFixed(1)}ms`);
    }
  };
}

class ReportService {
  @measured                                                   // L5
  buildHeatmapData(rows: PriceRow[]) { /* heavy transform */ }
}

10. Code Explanation — Line by Line

  • L1: Fully-generic signature — This/Args/R preserve hote hain, decorated method ka type nahi badalta callers ke liye. Generics (S6) yahan kaam pe.
  • L2: context — TC39 style ki pehchaan: kind ('method'), name, static/private flags, addInitializer. Legacy style hota toh (target, key, descriptor) milta — dono forms interview mein bolo.
  • L3–L4: Wrapper function return = original replace. this forward + try/finally — errors pe bhi timing log. Ch1 ka this-binding gyaan yahan directly lagta hai.
  • L5: Sirf @measured — arguments nahi. Configurable chahiye (@measured('label')) toh decorator factory: function jo decorator return kare — Angular ke saare decorators factories hain, isliye @Component({...}) mein parentheses hain!

11. Common Mistakes

❌ Galtiyan
  • (window as any).Razorpay se kaam chalana — 30 second bachaye, poore checkout flow ki type safety bech di. d.ts likho (7 wala) — 10 minute ka one-time kaam.
  • .d.ts mein implementation likhne ki koshish — declaration files mein sirf declare-shapes; logic wahan compile error hai.
  • declare module 'some-lib' mein : any ka shortcut stub — poora module any ho gaya, imports pe koi checking nahi. Minimum bhi likho toh used-surface ke real types likho.
  • Global augmentation ko kahin bhi random file mein daalna — module file mein declare global tabhi lagta hai jab file module ho (koi import/export ho); warna ambient/global confusion. Convention: types/ folder, project references saaf.
  • Decorators mein heavy logic — wo class definition pe chalte hain (app startup), na ki lazily; startup cost + testing pain. Decorator thin rakho; kaam service/function mein.

12. Best Practices

✅ Production rules
  • Boundary inventory rakho: har script-tag SDK, env constant, asset import ka apna .d.ts types/ mein — "any imports" lint rule se guard.
  • import type + verbatimModuleSyntax — type/value imports explicitly alag; bundler-friendly, cycle-safe.
  • Library publish karo toh: declaration: true, declaration maps bhi (go-to-definition consumers ke liye source tak jaaye).
  • Custom decorators sirf cross-cutting concerns pe (logging, caching, telemetry) — business logic kabhi nahi; aur Angular components mein framework ke primitives (signals APIs) prefer karo.

13. Performance

.d.ts / import type: pure compile-time — balki runtime bundle ghatate hain (type-only imports drop). Decorators: runtime cost hai — legacy emit har decorated class pe helper calls + (agar emitDecoratorMetadata on ho) design-type metadata arrays; sainkdon classes wale app mein measurable startup bytes. Angular AOT isi liye decorators ko build time pe consume karke unke runtime footprint ko practically zero kar deta hai — "decorators are a compile-time DSL for Angular" bolna interview mein sharp line hai.

14. Edge Cases

  • declare global sirf module files mein valid (kuch import/export ho); pure ambient .d.ts mein cheezein waise hi global hoti hain — dono modes mix karna common confusion.
  • Module augmentation (declare module '@angular/router') sirf existing module mein add karta hai — naya module declare karna alag cheez hai; augment mein original import bhi file mein hona chahiye.
  • @types packages ka version drift — lib v5, @types v4 = phantom errors; renovate/dependabot pairing rakho. Bundled-types libraries (Angular, RxJS) ye problem hi nahi rakhtin.
  • Legacy vs TC39 ek project mein mix nahi hote — experimentalDecorators flag ON = sab legacy semantics (Angular ka current mode), OFF (TS 5+) = standard. Library authors ke liye dono support karna painful — isi transition mein ecosystem abhi hai.
  • Decorator evaluation order: factories top-to-bottom evaluate, decorators bottom-to-top apply — multiple decorators stack karo toh wrapping order ulta hota hai.

15. Interview Questions

EASY .d.ts file kya hai aur @types packages kahan se aate hain?

Declaration file — sirf type declarations, zero implementation; JS code ko TS-consumable banati hai. @types/* = DefinitelyTyped community repo se published types un packages ke liye jo apne types bundle nahi karte. Resolution order: package ke own types → @types → nothing (any + error).

EASY import type kyun use karein jab normal import bhi types la sakta hai?

Guarantee ke liye ki import runtime output se poora erase ho — (1) circular dependency runtime pe nahi banti, (2) single-file transpilers (esbuild) ko ambiguity nahi, (3) accidental bundle inclusion nahi. verbatimModuleSyntax ise enforce karta hai. Ek line: "types ki dependency ≠ runtime ki dependency — import type ye distinction explicit karta hai."

MEDIUM window pe aane wale third-party SDK (script tag) ko type-safe kaise banaoge?

Types folder mein .d.ts: SDK ke options/instance ke interfaces + declare global { interface Window { Sdk: new (opts) => Instance } } — Window interface merging (S2) se augment hota hai. Ab window.Sdk fully typed, as-any casting gayab. Bonus: SDK load hone ki runtime guarantee alag problem hai — loader service + "script loaded" promise/signal se handle hoti hai; types sirf shape ka contract hain.

MEDIUM Decorator kab execute hota hai — class definition pe ya instantiation pe? Iska implication?

Definition pe — file/module evaluate hote waqt, ek baar per class. Instances banne se pehle sab decorators chal chuke hote hain. Implications: (1) decorator ke andar instance state available nahi (wo prototype/constructor level pe kaam karta hai); (2) heavy decorator = startup cost; (3) isi liye Angular AOT decorators ko build pe consume kar sakta hai — wo static, definition-time information hain.

HARD @Component decorator se render hone tak Angular ka pipeline explain karo — reflect-metadata ki zaroorat kyun nahi padti?

@Component ek decorator factory hai jo config ko class ke saath associate karta hai. JIT world mein ye metadata runtime pe padha jaata (reflect-metadata / compiler in browser). AOT mein Angular compiler build time pe TS AST se decorator config statically extract karta hai, template ko parse karke render instructions (ɵcmp static property — component definition) generate karta hai, aur DI ke liye ɵfac (factory). Runtime pe framework seedha in generated definitions ko use karta hai — decorator ka runtime role effectively khatam, reflect-metadata unnecessary. Isi architecture se: templates ki compile-time checking (strictTemplates), chhota runtime, tree-shakable output. Signals-era APIs (input()/viewChild()) isi static-analysis philosophy ko aur aage le jaati hain.

SCENARIO Team ko legacy jQuery plugin (script tag, window.$.myPlugin) Angular app mein use karna hai. Poora type-safe integration plan batao.

(1) Surface audit: plugin ka jitna API use hoga sirf utna type karo (full types likhne mein hafta mat lagao). (2) .d.ts: options/instance interfaces + declare global se Window (ya jQuery interface) augmentation — S2 merging. (3) Wrapper Angular service: plugin access ek jagah encapsulate — DOM-touching code Renderer2/afterNextRender ke saath (SSR safety — tumhare PR reviews wala point!), lifecycle pe destroy (Ch1 leaks). (4) Loader: script dynamically load + typed readiness promise; components sirf service se baat karein. (5) Boundary tests: wrapper ka contract test, plugin mocked. Ye answer types + architecture + SSR + memory sab dikhata hai — full-stack senior signal.
