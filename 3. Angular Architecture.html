Angular Overview — Framework ka Falsafa

Covers: Library vs framework · Angular ke 6 pillars · Ecosystem · AngularJS se Angular tak · Signals-first era · Trade-offs

1. What is it?

Angular ek opinionated, batteries-included application framework hai — TypeScript-first, Google dwara maintained. "Framework" ka matlab: tum uske andar likhte ho, wo tumhe call karta hai (Inversion of Control). React ek library hai — tum use call karte ho, baaki decisions (routing, HTTP, forms, DI, testing) tumhe khud jodne hote hain. Angular ye saare decisions pehle se de deta hai, ek coherent whole ke roop mein.

2. Why do we need it?

Ek 3-page site ke liye framework overkill hai. Angular ka case tab banta hai jab: 50+ developers, 5 saal ka lifecycle, 500 screens, aur team members rotate hote rehte hain. Aise environment mein sabse mehenga cheez decisions ka bikhraav hai — har team apna routing solution, apna state library, apna form pattern. Angular ka core value proposition: consistency at scale. Enterprise (banking, healthcare, internal tools) mein isi liye dominant hai.

3. What problem does it solve?

  • Decision fatigue & fragmentation → router, HTTP, forms, DI, testing — sab official aur integrated.
  • Onboarding cost → ek Angular dev doosre Angular codebase mein 2 din mein productive; structure predictable hai.
  • Long-term maintenance → ng update + automated migrations (schematics) — breaking changes ka rasta bana hua hai.
  • Type safety end-to-end → templates bhi type-checked (strictTemplates, Ch2 S1).

  );

10. Code Explanation — Line by Line

  • L1: imports array mein sirf class ka reference hai. Compile time pe compiler ChildComponent ka static δcmp field dekhta hai (uska selector, inputs, outputs) — uska template nahi.
  • L2: Template mein <app-child /> ko compiler resolve karta hai imports list se — matching selector. Ye AngularJS/ViewEngine se bada shift hai jahan poora NgModule graph compile-time pe chahiye tha (global compilation scope).
  • Kyun matter karta hai: Locality ki wajah se Ivy ne teen cheezein possible ki — (1) incremental compilation (badla hua component hi rebuild), (2) libraries ko compiled ship karna (npm packages Angular version se decouple), (3) tree shaking (unused component definitions bundle se nikal jaate hain). Ye ek architectural constraint hai jisne teen alag-alag problems solve ki — yahi acchi architecture ka nishaan hai.

11. Common Mistakes

Galtiyan
  • Angular ko "sirf ek library" samajhna — IoC framework hai; uske lifecycle aur DI ke against ladoge toh code ganda hoga.
  • AngularJS (1.x) aur Angular (2+) ko ek samajhna — ye alag frameworks hain. Interview mein "$scope" bol diya toh red flag.
  • Purane tutorials follow karna — NgModules, constructor injection, *ngIf, RxJS-for-everything. 2026 ka Angular standalone + signals + @if hai. Content ki freshness check karo.
  • Framework choice ko technical religion banana — actual questions ye hain: team size? lifespan? hiring pool? bundle budget? existing ecosystem?
  • "Angular slow hai" maan lena — Ivy + signals + zoneless ke baad ye claim data ke bina nahi tikta.

12. Best Practices

Production rules
  • Naya code hamesha: standalone components, inject(), signals, new control flow (@if/@for), input()/output() functions.
  • ng update regularly — ek-ek version karke; do saal ruk gaye toh migration project ban jaata hai.
  • Angular ki official DI/lifecycle ke saath chalo, uske around workaround mat banao.
  • Framework version ke saath apne mental model ko update karo — Angular tezi se badla hai; 2020 ka gyaan aaj half-wrong hai.

13. Performance

Framework-level performance ke teen levers: (1) Bundle size — Angular ka runtime baseline React se bada hai, par tree shaking + lazy routes ke baad real apps ka farak choti si hoti hai; measure karo, maano mat. (2) Change detection cost — default Zone.js CD poore tree ko check karta hai; OnPush se subtree skip, signals se sirf affected views. Ye Chapter 4 ka poora vishay hai. (3) Startup (TTI) — AOT compilation (S5) ne runtime compiler hata diya, SSR + hydration (S8) ne first paint tez kiya. Baseline expectation: modern Angular app ka initial chunk 150–250 KB gzipped range mein hona chahiye; usse zyada = audit karo.

14. Edge Cases

  • Angular Elements — Angular component ko custom element (Web Component) bana ke non-Angular page mein use karna; micro-frontend scenarios mein.
  • Multiple Angular apps ek page pe — possible hai par platform-level state share nahi hota; zone conflicts classic problem hai.
  • AngularJS + Angular hybrid (ngUpgrade) — legacy migration ka rasta; ab practically historical, par purani enterprise codebase mein mil sakta hai.
  • Angular bina Zone.js — provideZonelessChangeDetection(); tab CD sirf signals/explicit markForCheck se trigger hoti hai. Third-party libs jo setTimeout se state badalti hain, wo tootengi.

15. Interview Questions

EASY Angular framework hai ya library — farak kya hai?

Framework — Inversion of Control: tum uske structure ke andar likhte ho, wo tumhare code ko call karta hai (lifecycle hooks, CD cycle). Library (React) mein tum uske functions call karte ho. Practical implication: framework opinionated hota hai (routing, DI, HTTP sab andar), library ke saath ecosystem tum assemble karte ho.

EASY AngularJS aur Angular mein kya difference hai?

Alag frameworks hain, sirf naam mila-julta hai. AngularJS (1.x): JS, $scope, digest cycle, 2-way binding by default, controllers. Angular (2+): TypeScript, components, unidirectional data flow, Zone.js/Signals se CD, AOT compilation, hierarchical DI. Rewrite tha, upgrade nahi.

MEDIUM Angular ke architectural pillars kaunse hain aur wo ek saath kaise kaam karte hain?

Components (UI atoms), Templates (declarative view, compile hoke JS instructions), DI (services ki hierarchical supply), Change Detection (model → DOM sync), Router (URL ↔ component tree), Compiler+Runtime (Ivy). Flow: compiler template ko instructions mein badalta hai → runtime component instantiate karta hai, DI se dependencies inject karta hai → instructions DOM banate hain → state badalne pe CD update instructions chalata hai. Har pillar ka ek clear job hai, aur inka contact point Ivy ka component definition (δcmp) hai.

MEDIUM Angular ka "locality" principle kya hai aur usne kya unlock kiya?

Ek component compile karne ke liye compiler ko sirf usi component ki information chahiye — dependencies ka source nahi, sirf unke public types/selectors. Isse teen cheezein mili: (1) incremental builds — ek file badli toh sirf wahi recompile; (2) partial compilation — libraries pre-compiled ship kar sakti hain (Angular version se decoupled); (3) tree shaking — component definitions plain static properties hain, unused drop ho jaati hain. Ye Ivy ka core design constraint tha, ViewEngine mein global compilation scope tha (NgModule graph chahiye tha).

HARD Angular ka evolution (NgModules → standalone, Zone.js → signals) ek hi direction mein kyun ja raha hai?

Common thread: implicit global machinery ko explicit local declarations se replace karna. NgModule ek global compilation+DI scope tha — component ko nahi pata tha uske paas kya available hai (module decide karta tha); standalone mein component apni dependencies khud declare karta hai (locality). Zone.js ek global monkey-patch tha jo har async operation intercept karta tha aur "kuch badla hoga shayad" bol ke poore tree ki CD chalata tha; signals mein har reactive value apne consumers ko khud jaanti hai, toh sirf affected views invalidate hote hain. Dono cases mein: global+implicit → local+explicit. Faayda: better tree shaking, better incremental compilation, better performance predictability, aur (sabse important) code padhne wale ko dikh jaata hai ki kya ho raha hai. React ka safar bhi (class components → hooks → compiler) isi philosophy pe hai.

SCENARIO Tumhari company ek naya internal admin platform bana rahi hai — 30 devs, 5 saal ka roadmap, 200+ screens, mostly CRUD + reports. Tech lead poochta hai: Angular ya React? Kaise decide karoge?

Framework choice ko decision framework ki tarah treat karo, opinion ki tarah nahi. (1) Team: 30 devs, rotation hogi — consistency ka premium high hai; Angular ki opinionated structure onboarding cost ghatati hai. React choose kiya toh pehle hi din ek "golden path" (routing, state, forms, data-fetching ka fixed stack) define karna padega, warna 5 saal mein 5 patterns milenge. (2) Lifespan: 5 saal mein framework migrations hongi — Angular ka ng update + automated schematics maintenance cost predictable rakhta hai; React ecosystem mein har library apni breaking-change timeline pe chalti hai. (3) Workload shape: CRUD + reports = forms-heavy. Angular ke typed reactive forms (Ch2 S6/S8) yahan strong hain. (4) Hiring pool + existing expertise: team already Angular jaanti hai toh switch ka cost sirf syntax nahi, poora ecosystem knowledge hai. (5) Constraints jo choice ko flip karein: agar bundle-size critical public-facing app hota, ya team chhoti aur experimental hoti, ya React Native saath chahiye hota — jhukav React ki taraf hota. Punchline: is context mein Angular strong default hai, par decision "kaunsa better" se nahi, "kis dimension pe optimize kar rahe hain" se aata hai.

Angular CLI & Workspace

Covers: ng commands · Schematics (code generation) · Builders (task execution) · angular.json · ng update migrations · Monorepo workspaces

1. What is it?

Angular CLI = ek command-line tool jo do bilkul alag cheezein karta hai, aur inhe alag samajhna hi is section ka core hai:

  • Schematics — code generate/modify karte hain (ng generate, ng new, ng update ki migrations). Ye files likhte hain.
  • Builders — tasks execute karte hain (ng build, ng serve, ng test). Ye code chalate hain.

Aur angular.json = workspace ka config file jo batata hai kaunsa project kaunsa builder kis configuration ke saath use karega.

2. Why do we need it?

Bina CLI ke ek Angular project set up karne ka matlab: TypeScript config, Angular compiler plugin, bundler (esbuild/webpack), dev server, HMR, test runner, SSR setup, production optimizations — sab haath se wire karna. Aur phir har Angular version upgrade pe ye poora setup dobara tune karna. CLI ye poora build infrastructure abstract karta hai aur — critically — ng update ke through maintain bhi karta hai.

3. What problem does it solve?

  • Setup complexity → ng new se production-grade config, day 1.
  • Boilerplate → ng g c se component + spec + registration, consistent naming.
  • Upgrade pain → ng update package versions bump karta hai aur code migrations chalata hai (deprecated APIs auto-rewrite).
  • Team consistency → sab log same generator use karte hain, structure same rehta hai.

9. Advanced Example — apna custom schematic (blueprint enforcement)

// schematics/feature/index.ts — team ka standard feature structure generate karo
import { Rule, Tree, apply, url, template, move, mergeWith, chain } from '@angular-devkit/schematics';

export function feature(options: { name: string }): Rule {        // L1
  return (tree: Tree) => {                                        // L2
    const templateSource = apply(url('./files'), [                // L3
      template({ ...options, ...strings }),                       // L4
      move(`src/app/features/${options.name}`),                   // L5
    ]);
    return chain([
      mergeWith(templateSource),                                  // L6
      addRouteToAppRoutes(options.name),                         // L7: AST edit
    ]);
  };
}
// Usage: ng g ./schematics:feature --name=payments
// Banega: payments.routes.ts, payments.service.ts, pages/, ui/, models/
// aur app.routes.ts mein lazy route AUTOMATICALLY add ho jaayega.

10. Code Explanation — Line by Line

  • L1: Schematic = factory function jo Rule return karta hai. Rule ek pure-ish transformation hai: Tree → Tree.
  • L2: Tree = virtual filesystem. Yahan kuch bhi disk pe nahi likha ja raha — sirf staged.
  • L3: url('./files') — template files ka folder (jaise __name@dasherize__.service.ts.template).
  • L4: template() — placeholders fill karta hai; strings helpers (dasherize, classify, camelize) naming conventions handle karte hain.
  • L5: Generated files ko target path pe move.
  • L6: mergeWith — staged files ko main tree mein merge.
  • L7: Yahi asli power hai: existing file ko AST parse karke surgical edit (route array mein entry add) — regex se nahi. ng update ki migrations bilkul isi mechanism pe chalti hain, isiliye wo tumhare formatting ko todti nahi.

11. Common Mistakes

Galtiyan
  • ng update ko skip karke npm install @angular/core@latest — packages upgrade ho gaye, migrations nahi chale. Deprecated API code mein pade rehte hain aur agla upgrade aur mushkil.
  • Do-teen versions ek saath jump karna — migrations version-by-version likhi jaati hain; 17 se 20 seedha jaoge toh beech ki migrations skip ho jaayengi. Ek-ek karke chalo (update.angular.dev guide follow karo).
  • angular.json ko haath se bina samjhe edit karna — builder options schema-validated hain; galat key silently ignore ho sakti hai.
  • Generated files ko blindly accept karna — --dry-run pehli baar hamesha; naya CLI version output badal sakta hai.
  • Har jagah ng g c ke defaults use karna par angular.json mein team defaults na set karna — 20 devs, 20 style choices.

12. Best Practices

Production rules
  • angular.json mein schematic defaults commit karo (OnPush, scss, standalone) — convention automation se enforce ho, code review se nahi.
  • Upgrade ritual: ek minor version at a time → ng update → tests → commit. Har version ka apna commit — bisect karna aasan.
  • Repetitive structure (feature folder) ho toh custom schematic likho — ek din ka kaam, saal bhar ka consistency.
  • Builder configurations se environments handle karo (--configuration=staging), environment.ts file-replacement se nahi (ab legacy pattern hai).
  • CI mein ng build --configuration=production hi chalao — dev build ka behavior alag hota hai (S7).

13. Performance

CLI ki performance do jagah dikhti hai. Dev loop: ng serve ab esbuild + Vite pe hai — cold start seconds mein, rebuild sub-second (webpack era mein 10–30s common tha). Ye locality principle (S1) + esbuild ki speed ka combined faayda hai. CI build: --configuration=production full optimization chalata hai — sabse mehenga step hai; caching (node_modules, Angular ka build cache .angular/cache) se CI time aadha ho jaata hai. Ek practical measurement: ng build --stats-json + bundle analyzer se dekho time kahan ja raha hai; blind optimization mat karo.

14. Edge Cases

  • Multi-project workspace: ng g app / ng g library — ek angular.json mein kai projects. Libraries ng-packagr se build hoti hain (partial compilation, S5).
  • Nx / custom monorepo tools Angular CLI ke builders ko wrap karte hain — Nx apne executors deta hai par underlying Angular builders hi chalte hain.
  • ng update third-party packages ke liye bhi migrations chala sakta hai agar unhone ng-update collection publish ki ho (Angular Material karta hai).

10. Code Explanation — Line by Line

  • L1: export * re-export — barrel ka module graph mein har member ka edge ban jaata hai.
  • L2: Ergonomically sundar. Cost: import graph mein @shared/ui node ke saare children aa gaye. Chhote barrels (5-6 files, sab pure) practically theek hain; app-wide mega-barrel (src/app/index.ts) disaster hai.
  • L3: Deep import — precise edge. Path aliases ise readable rakhte hain. Bundle analysis mein bhi saaf dikhta hai kaun kis pe depend karta hai.
  • Nuance: "Barrel files evil hain" ek meme ban gaya hai — sach ye hai ki bade, deep, cyclic-prone barrels evil hain. Library ka public API surface (public-api.ts) ek barrel hi hota hai aur bilkul sahi hai — kyunki wahan wo intentional contract hai, convenience shortcut nahi.

11. Common Mistakes

Galtiyan
  • Type-first structure (components/, services/) — 50 files ke baad navigate karna dard ban jaata hai, lazy loading impossible.
  • shared/ ko dumping ground banana — jo cheez samajh na aaye kahan rakhein, wo shared mein. Chhe mahine mein shared/ hi poora app hai.
  • Cross-feature direct imports (features/orders se features/payments/...) — do features ab ek doosre ke bina deploy/delete nahi ho sakte. Zaroorat ho toh shared contract core/ ya shared/ mein uthao.
  • Mega-barrel + circular imports — index.ts A ko export karta hai, A index.ts se import karta hai. Runtime pe undefined, debugging mein ghanton (Ch1 S8 ka TDZ/cycle wala point).
  • Structure decide karke rules enforce na karna — lint boundaries ke bina structure sirf ek suggestion hai.

12. Best Practices

Production rules
  • Feature-first + core/shared. Folder ka naam business domain bole ("payments"), framework concept nahi ("services").
  • Path aliases (@core/*, @features/*) — ../../../ kabhi nahi. Refactor pe imports nahi tootenge.
  • Rule of three: shared mein promote tabhi jab 3 consumers hon.
  • Dependency rules ko ESLint boundaries plugin (ya Nx tags) se enforce karo — CI mein fail ho, review mein nahi.
  • Har feature ka apna *.routes.ts — lazy boundary aur folder boundary ek hi jagah.
  • Deletability test: "kya main is feature ka folder delete karke build chala sakta hoon?" Agar nahi, coupling hai.

13. Performance

Structure ka direct performance impact bundle graph ke through aata hai. Feature-first + per-feature routes = clean lazy chunks; type-first = sab kuch initial bundle mein kyunki koi natural split point hi nahi. Barrels lazy chunk boundaries ko "leak" kara sakte hain: reports chunk ne @shared/ui barrel import kiya, barrel ne 15 components pull kiye, chunk 40KB bada ho gaya — source-map-explorer ya ng build --stats-json se ye saaf dikhta hai. Build-time impact bhi hai: barrels + cycles se incremental rebuild ka invalidation scope badta hai, dev loop slow.

14. Edge Cases

  • Do features ko sach mein ek doosre ka data chahiye — solution: shared contract (interface + injection token) core/ mein, implementations apni-apni features mein. Direct import phir bhi nahi (Ch2 S5 ka abstract-token pattern yahan fit hota hai).
  • Monorepo (Nx): features libraries ban jaati hain, dependency rules tags se enforce hoti hain, aur affected-graph se CI sirf badle hue projects test karta hai.
  • Micro-frontends: har feature ka apna deploy — structure boundaries ab network boundaries ban jaati hain (Module Federation). Overhead bada hai; 30-dev team ke liye aksar overkill.
  • shared/ mein components ke saath signals-based stateful services aa jaayen toh wo "shared" nahi raha — hidden global state ban gaya. Shared = stateless.

15. Interview Questions

EASY core/, shared/, features/ mein kya rakhoge?

core/: app-wide singletons jo ek hi baar chahiye — interceptors, guards, config service, error handler. shared/: stateless reusable cheezein — dumb UI components, pipes, utils; koi business logic nahi. features/: business domains, har ek self-contained (routes, services, pages, ui, models). Direction: features → shared/core, kabhi ulta nahi.

EASY Smart aur dumb components mein farak?

Smart (container/page): services inject karta hai, data laata hai, state rakhta hai, routed hota hai. Dumb (presentational): sirf input() leta hai, output() emit karta hai, koi dependency nahi. Faayda: dumb components reusable, trivially testable (koi mocking nahi), aur OnPush ke saath naturally efficient.

MEDIUM Feature-first structure type-first se better kyun hai?

Teen thos reasons: (1) Cohesion — ek feature pe kaam karte waqt ek folder mein reh sakte ho, teen mein jump nahi. (2) Lazy loading — feature folder = route boundary = bundle chunk; type-first mein natural split point hai hi nahi. (3) Deletability — feature retire hua toh ek folder delete; type-first mein orphan services har jagah reh jaate hain. Bonus: folder structure business domain "scream" karti hai, framework nahi — naye dev ko app ka mental model turant milta hai.

MEDIUM Barrel files ke fayde aur nuksaan?

Fayda: clean import statements, ek jagah se public surface control. Nuksaan: (1) barrel import poore module ko evaluate karta hai — accidental code pull, lazy chunk mein extra weight; (2) circular import ka risk (barrel ↔ member) jo runtime pe undefined deta hai; (3) incremental rebuild ka invalidation scope badta hai. Nuanced position: library ka public-api.ts barrel sahi hai (intentional contract); app ke andar deep imports + path aliases behtar hain.

HARD Dependency rules ko sirf convention na rakh kar enforce kaise karoge, aur kyun?

Kyun: koi bhi rule jo human review pe depend karti hai, deadline wale hafte mein tootegi — aur ek baar toot gayi toh "already broken hai" ka justification aage bhi chalta rahega. Kaise: (1) eslint-plugin-boundaries ya @nx/enforce-module-boundaries — har folder ko ek tag do (type:feature, type:shared, type:core) aur allowed dependencies declare karo; violation = CI fail. (2) Path aliases ko is tarah design karo ki galat import likhna asuvidhajanak ho. (3) madge/dependency-cruiser se circular dependency detection CI mein. (4) Codeowners — core/ ke changes ko architecture review chahiye. Deep point: architecture ek constraint system hai, ek diagram nahi. Constraint agar automated nahi hai, toh wo exist hi nahi karta.

SCENARIO Ek 4-saal purani Angular app mili hai: 1,200 files, type-first structure, sab kuch eagerly loaded, initial bundle 2.8 MB. Tumhe incremental restructuring karna hai bina feature freeze ke. Plan?

Big-bang rewrite mana hai — plan phased hona chahiye. (1) Measure first: ng build --stats-json + source-map-explorer se dekho bundle mein kya hai; madge se circular deps aur dependency graph nikalo. Data ke bina restructuring religion hai. (2) Boundaries dhundo: routes se natural feature boundaries pehle se maujood hain — code kahin bhi ho, routes batati hain ki kaun kiske saath jaata hai. (3) Strangler pattern: naya code hamesha naye features/ structure mein; purana wahin rehne do. Ek lint rule: naye files ko naya structure follow karna hoga. (4) Lazy load pehle, refactor baad mein: sabse bada route ko lazy karo (loadChildren) — bundle win turant milti hai aur wo route ka code implicitly ek folder mein khinch aata hai. Highest-impact route se shuru karo. (5) Circular deps todo — warna lazy chunks split nahi honge (bundler sab ek chunk mein daal dega). Ye aksar asli blocker hota hai. (6) Enforce as you go: jaise-jaise features migrate hon, boundary lint rules add karo — ratchet, kabhi peeche nahi. (7) Track a metric: initial bundle size CI mein budget ke saath (S7) — har PR pe visible. Message: restructuring ek metric-driven, incremental exercise hai; iska output size + rebuild time mein dikhna chahiye, warna sirf files move hui hain.

1. What is it?

Bootstrap = wo process jisme ek plain HTML page ek zinda Angular application ban jaata hai. Concretely: platform banta hai → root injector (DI container) banta hai providers se → root component instantiate hota hai → uska template DOM mein render hota hai → change detection loop chalu ho jaata hai. main.ts is process ka entry point hai, app.config.ts uska configuration.

2. Why do we need it?

Kyunki Angular app ko chalne ke liye ek environment chahiye: DI container jisme services rahein, ek renderer jo DOM ko chhue, error handler, zone (ya zoneless scheduler), router ki initial navigation. Ye sab kisi ek jagah wire hona chahiye, aur wo jagah bootstrap hai. Interview angle: bootstrap samajhna matlab poore framework ka "power on" sequence samajhna — DI, CD, rendering sab yahin se shuru hote hain.

3. What problem does it solve?

  • Explicit composition root — app ki saari global wiring ek file mein, chhupi hui nahi.
  • Tree-shakable configuration — provideRouter() jaisi functions import na karo toh router bundle mein aata hi nahi (NgModules mein ye mushkil tha).
  • Startup ordering — config load, auth restore, feature flags — sab render se pehle (initializers).
  • Multiple platforms — same app browser mein, server pe (SSR), ya test mein bootstrap ho sakti hai; sirf platform badalta hai.

  • L1 (legacy): declarations — ek component exactly ek NgModule mein declare ho sakta tha. Ye "kaun kis module mein hai" ka mental overhead poora hataya gaya standalone mein.
  • L2: HttpClientModule import kiya = poora module bundle mein, chahe tum sirf ek GET karo. provideHttpClient(withFetch()) mein sirf wahi features aate hain jo tum import karte ho — yahi tree-shakable providers ka point hai.
  • L3: bootstrap: [AppComponent] — module ko batana padta tha kaunsa root component hai. Ab bootstrapApplication(AppComponent) mein direct hai.
  • L4: platformBrowserDynamic — "dynamic" ka matlab tha: runtime compiler shipped (JIT), templates browser mein compile hote the. 11. Common Mistakes

Galtiyan
  • provideAppInitializer mein Promise return na karna — app initializer ke complete hone se pehle render ho jaayegi; config undefined, aur ye bug staging mein (fast network) kabhi reproduce nahi hoga.
  • Initializer mein bhaari blocking work (bade config fetches, analytics init) — har user ka first paint utna slow. Initializer mein sirf wahi jo render se pehle sach mein chahiye.
  • Root providers mein sab kuch daalna — @Injectable({providedIn: 'root'}) tree-shakable hai, app.config.ts ka array nahi. Service ko root mein tabhi likho jab token override kar rahe ho.
  • platformBrowserDynamic naye code mein use karna — JIT compiler bundle mein ghusa deta hai.
  • Bootstrap ka rejected promise ignore karna — .catch() lagao, warna startup errors silent (Ch1 S7 unhandled rejection).
  • SSR ke saath browser-only APIs (window, localStorage) bootstrap-time pe touch karna — server pe crash. afterNextRender() ya platform check use karo.

12. Best Practices

Production rules
  • app.config.ts ko composition root ki tarah treat karo: sirf global wiring, koi business logic nahi. Bada ho jaye toh core/providers/*.ts mein tod ke ...routerProviders spread karo.
  • Services providedIn: 'root' se register karo (tree-shakable); config array sirf tokens/overrides ke liye.
  • Initializers: ek, chhota, timeout ke saath. Fail ho toh graceful degrade (default config) — startup gate crash gate na ban jaye.
  • Functional interceptors + functional guards (Ch2 S4 CanActivateFn) — testable, tree-shakable.
  • Naye projects: provideZonelessChangeDetection() se shuru karo — baad mein migrate karna mushkil hai.

13. Performance

Bootstrap seedha TTI (Time to Interactive) pe baithta hai, isliye yahan har millisecond user ko dikhta hai. Teen levers: (1) Initializers ka critical path — har blocking initializer first paint ko delay karta hai; parallel karo (Promise.all, Ch1 S7) aur non-critical ko bootstrap ke baad chalao. (2) Root providers ka eager instantiation — useClass providers lazily banti hain (pehle inject pe), par useFactory jo turant kuch karta hai wo startup pe cost hai. (3) Bundle mein kya aaya — provideAnimationsAsync() vs provideAnimations() ka farak ~30-50KB ka ho sakta hai; JIT compiler (platformBrowserDynamic) ~100KB. SSR + hydration (S8) bootstrap ki dikhne wali cost ko chupata hai: user ko server-rendered HTML turant milta hai, hydration background mein.

14. Edge Cases

  • Multiple apps ek page pe: bootstrapApplication do baar — platform injector shared, application injectors alag. State share nahi hota.
  • importProvidersFrom(SomeNgModule) — legacy NgModule-based libraries ko standalone config mein bridge karta hai. Escape hatch hai; jahan library ne provideXxx de diya ho, wahi use karo.
  • Testing mein bootstrap alag hota hai — TestBed.configureTestingModule({providers}) apna injector banata hai; main.ts chalta hi nahi.
  • SSR mein bootstrapApplication server pe chalti hai (@angular/platform-server) — same app config, alag platform providers; DOM ki jagah server-side renderer.
  • APP_INITIALIZER token (purana) vs provideAppInitializer() (naya) — dono kaam karte hain; naya inject()-friendly hai.
  • Bootstrap ke pehle koi error (module-level code mein throw) — Angular ka error handler abhi bana bhi nahi; browser console mein raw error milta hai.

EASY main.ts ka kya kaam hai?

EASY app.config.ts mein kya jaata hai?

MEDIUM Standalone bootstrap NgModule bootstrap se better kyun hai?

(1) Tree shaking: provideHttpClient(withFetch()) sirf use kiye gaye features laata hai; HttpClientModule poora module laata tha. (2) Locality: component apni dependencies khud declare karta hai — "ye component kis module mein declare hai" wala indirection khatam. (3) Kam concepts: declarations/imports/exports/entryComponents ka mental model gaya. (4) AOT by default: platformBrowserDynamic JIT compiler bundle mein laata tha. Trade-off imaandaari se: NgModule ek jagah se bahut saare components ko dependencies dene ka shortcut deta tha; standalone mein har component ki apni imports list — thoda verbose, par explicit.

MEDIUM Platform injector aur root (environment) injector mein farak?

Platform injector per-platform hota hai (ek page pe kai apps ho toh shared) — DOM adapter jaisi platform-level cheezein. Root/environment injector per-application hai — tumhare app.config.ts ke providers yahan register hote hain. Chain: NullInjector ← Platform ← Root Environment ← Route-level environment injectors ← Element injectors. "No provider for X" error ka matlab: is poori chain mein token nahi mila.

HARD bootstrapApplication call se pehle paint tak ka sequence detail mein batao — aur APP_INITIALIZER exactly kahan fit hota hai?

(1) main.js ESM evaluate hota hai; module-level side effects chalte hain. (2) bootstrapApplication platform inject karta/banata hai. (3) Root EnvironmentInjector banta hai: Angular ke internal core providers + appConfig.providers. Providers register hote hain, instantiate nahi (lazy). (4) Zone setup (ya zoneless scheduler). (5) Initializers chalte hain — provideAppInitializer ke functions inject-context mein execute hote hain; jo Promise/Observable return karte hain unka wait hota hai. Yahi wo point hai jahan bootstrap rukta hai; isliye config yahan load karte hain. (6) Root component instantiate: <app-root> host element dhundha jaata hai, ElementInjector chain banti hai, constructor + field initializers chalte hain (inject() yahan valid hai). (7) Template ka creation pass: Ivy instructions DOM nodes banate hain, bindings initialize hote hain, child components recursively create hote hain (S6/S8). (8) Pehla CD cycle poora hone pe browser paint karta hai. (9) Router initial navigation resolve karta hai (guards/resolvers — ye paint ke baad ho sakta hai). (10) ApplicationRef.isStable true — SSR/e2e isi signal ka wait karte hain. Key insight: initializers step 5 pe hain, yaani injector ready hai par koi component nahi bana — isliye wahan component-level DI nahi milti, sirf environment-level.

SCENARIO App ko chalne se pehle server se feature flags aur tenant config chahiye. Par ye call kabhi-kabhi 3 second leti hai aur kabhi fail bhi hoti hai. Bootstrap kaise design karoge?

Naive answer: "APP_INITIALIZER mein daal do." Senior answer resilience aur perceived performance dono sochta hai. (1) Kya sach mein render se pehle chahiye? Tenant theming/routing decisions — haan. Feature flags jo sirf ek deep page pe use hote hain — nahi; wo route resolver ya lazy service mein ja sakte hain. Critical path chhota karo. (2) Timeout + fallback: initializer mein Promise.race([configLoad(), timeout(2000)]) (Ch1 S7 ka pattern) — timeout pe cached/default config se aage badho, background mein refresh karo. App kabhi white-screen na dikhe. (3) Failure policy: fail hone pe catchError se default config; error ko monitoring pe bhejo. Bootstrap crash = poora app dead, ye acceptable nahi. (4) Caching: last-known-good config localStorage mein; agli baar wahi se instant boot, phir revalidate (stale-while-revalidate). (5) Perceived performance: index.html mein ek lightweight skeleton/spinner — ye Angular se pehle dikhta hai, bootstrap ke baad replace ho jaata hai. SSR ho toh ye problem largely gayab (server pe config resolve karke HTML bhejo). (6) Parallel: config + auth restore ek saath (Promise.all), sequential nahi. Ye answer startup performance + failure modes + caching + SSR — sab touch karta hai, jo exactly SDE-3 bar hai.

S-05

Compilation — JIT, AOT & Ivy Compiler

Covers: JIT vs AOT · ngtsc pipeline · Template → instructions · δcmp / δfac · Template type-checking · Partial compilation · ngcc (historical)

1. What is it?

Angular compilation = tumhare @Component decorated classes aur HTML templates ko plain JavaScript mein badalna. Template koi cheez nahi hai jo browser samajhta ho — usse instructions (function calls ki sequence) mein compile karna padta hai. Ye kaam karne wala compiler hai ngtsc (Angular's TypeScript Compiler) — jo TypeScript compiler ke upar ek plugin ki tarah baithta hai.

Do modes: AOT (Ahead-of-Time — build ke waqt, default) aur JIT (Just-in-Time — browser mein runtime pe, ab practically dead).

2. Why do we need it?

Template ek DSL hai: {{ user.name }}, @if, [value], (click) — ye HTML nahi hai. Koi na koi isse executable code mein badlega. Sawaal sirf ye hai: kab — user ke browser mein (JIT, har user har baar) ya tumhare CI mein (AOT, ek baar sab ke liye)? Jawab obvious hai, par uske consequences deep hain: AOT ki wajah se templates type-check ho sakte hain, compiler bundle se nikal jaata hai, aur errors production se pehle mil jaate hain.

3. What problem does it solve?

  • Template errors runtime pe milna → AOT + strictTemplates se compile-time pe.
  • Compiler bundle mein aana → AOT ke baad browser ko compiler ki zaroorat hi nahi (~100KB bacha).
  • Startup pe compilation ka time → browser ko ready-made instructions milti hain.
  • Library-app version coupling → partial compilation se libraries kai Angular versions ke saath chalti hain.

4. Internal Working — ngtsc pipeline

  order-card.component.ts  (TS + template)
        |
        v  [1] TypeScript parse --> AST
        |
        v  [2] ANALYSIS: @Component decorators dhundho, metadata padho
        |      (selector, template, imports, inputs, outputs)
        |
        v  [3] TEMPLATE PARSE: HTML --> template AST
        |      (elements, bindings, @if/@for blocks, pipes)
        |
        v  [4] TYPE CHECK: template AST se ek "Type Check Block" (TCB)
        |      naam ka fake TS function generate hota hai --> tsc use check karta hai
        |      ► isi liye {{ user.nmae }} template mein bhi ERROR deta hai
        |

        |      static class properties: δcmp (component def), δfac (factory)
        |
        v  [6] EMIT: plain .js (decorators gayab, instructions maujood)

Type Check Block (TCB) ngtsc ka sabse elegant trick hai: template ko ek synthetic TypeScript function mein translate kiya jaata hai jahan har binding ek expression ban jaati hai. Phir normal tsc use type-check karta hai. Matlab template type-checking ke liye Angular ne apna type checker nahi likha — TS ka reuse kiya. Isliye tumhare Ch2 ke saare type rules (narrowing, generics, strict null) templates mein bhi lagte hain.

5. Execution Flow — template se instructions tak (asli output)

  static δfac = function() { return new BadgeComponent(); };   // factory (DI)

  static δcmp = δδdefineComponent({                              // component definition
    type: BadgeComponent,
    selectors: [['app-badge']],
    decls: 2,        // kitne DOM nodes banane hain (span + text)
    vars: 1,         // kitni bindings track karni hain
    template: function(rf, ctx) {                             // rf = RenderFlags
      if (rf & 1 /* Create */) {                             // L1
        δδelementStart(0, 'span', 0);                       // L2
        δδtext(1);                                          // L3
        δδelementEnd();
      }
      if (rf & 2 /* Update */) {                             // L4
        δδadvance(1);                                        // L5
        δδtextInterpolate(ctx.label);                          // L6
      }
    },
  });
}

10. Code Explanation — Line by Line

  • L1 → L2: inject() field initializer mein hai, toh factory simple hai — dependency class construct hote waqt injection context se milti hai. Constructor injection hota toh δfac mein δδdirectiveInject(CartService) dikhta.
  • L3: dependencies — component ke imports ka compiled roop. Static array hai, isliye bundler dekh sakta hai ki CurrencyPipe use ho raha hai. Ye locality (S1) ka concrete implementation hai.
  • L4: Pipes ko creation mein ek "slot" milta hai — kyunki impure/stateful pipes ko instance chahiye aur pure pipes ko memoization ke liye pichli value yaad rakhni hai. Slot index (1) LView array mein position hai (S6).
  • L5: [class.big] ek dedicated instruction ban gayi — generic "set attribute" nahi. Ivy ka instruction set specialized hai: har binding type ka apna optimized instruction. Isiliye runtime interpretation nahi karni padti.
  • L6: δδpipeBind2(slot, bindingIndex, value, arg) — pipe ko call karta hai par pehle check karta hai ki inputs badle hain ya nahi (pure pipe memoization). δδtextInterpolate bhi pehle purani value se compare karta hai — same hai toh DOM ko chhoota hi nahi. Ye dirty-checking hai, aur ye compiled code mein baked hai — runtime pe "kya badla" dhundhne ka generic algorithm nahi chalta. Yahi Ivy ki speed ka source hai.

11. Common Mistakes

Galtiyan
  • strictTemplates: false rakhna "kyunki errors aa rahe the" — AOT ka aadha faayda phenk diya. Errors migrate karke on karo.
  • Template mein complex expressions likhna ({{ items.filter(x => x.active).length }}) — ye har CD cycle mein re-evaluate hota hai (S8), aur type-check block bhi bloat karta hai. computed() mein nikaalo.
  • Ye samajhna ki AOT "template ko HTML string ki tarah render" karta hai — nahi, wo template ko code banata hai. Isliye dynamic template strings (runtime pe banaya HTML) AOT mein compile nahi ho sakte.
  • Purane blog posts se ngcc ke workarounds copy karna — v16+ mein wo exist hi nahi karta.
  • Library banate waqt full compilation force karna (compilationMode: "full") — app ke Angular version se coupling ban jaayegi. Libraries ke liye partial default hai, use rehne do.

12. Best Practices

Production rules
  • strictTemplates: true + typeCheckHostBindings: true — templates ko TS code ki tarah treat karo.
  • Template mein sirf simple property access aur signal calls; logic component class ya computed() mein.
  • Libraries: ng-packagr + partial compilation (default) — kabhi manually override mat karo.
  • Build errors ko compile-time gift samjho — // @ts-ignore ya $any() se dabao mat ($any() template ka as any hai — Ch2 S1 ka wahi zeher).
  • CI mein production build hi source of truth ho; dev build ke optimizations alag hote hain.

13. Performance

Build-time cost: ngtsc ka sabse mehenga step aksar template type-checking (TCB generation + tsc) hota hai — bade apps mein ye 40-60% time le sakta hai. Incremental builds (locality!) isse manage karte hain: sirf badle hue components aur unke dependents recheck hote hain. tsc --extendedDiagnostics aur .angular/cache se profile karo.

Runtime benefit: AOT ke baad browser ko koi compilation nahi karni — instructions ready hain. JIT vs AOT ka startup farak bade apps mein 300ms–1s+ ka hota tha, plus ~100KB compiler. Aur AOT compiled code khud optimized hai: dirty-checking baked in (upar L6), specialized instructions, no runtime metadata lookup.

14. Edge Cases

  • TestBed mein components JIT compile ho sakte hain (agar pre-compiled na hon) — isliye tests mein wo compiler present hota hai. Production bundle se alag rasta.
  • $any(expr) template escape hatch — type checking bypass. Third-party template edge cases mein rarely justified; comment ke saath.
  • ngDevMode — dev builds mein extra checks (ExpressionChangedAfterItHasBeenCheckedError!) inject hoti hain; production build mein ye poora code shake ho jaata hai. Isliye kuch errors sirf dev mein dikhte hain — feature hai, bug nahi.
  • Component ke template vs templateUrl — build ke waqt dono inline ho jaate hain; runtime pe koi HTTP request nahi hoti (AOT mein).
  • Angular linker (partial → full) app ke build mein babel step hai — custom builders/monorepo tools mein isko skip kar dena "library toot rahi hai" ka common root cause hai.

15. Interview Questions

EASY AOT aur JIT mein farak?

AOT: templates build time pe compile hote hain; compiler bundle mein nahi jaata; template errors compile pe milte hain; startup fast. JIT: browser mein runtime pe compile; compiler ship karna padta hai (~100KB); errors runtime pe. Aaj AOT default hai har build mein (dev + prod); JIT practically sirf kuch testing scenarios mein.

EASY Compiled Angular component mein δcmp kya hai?

Compiler dwara generate ki gayi ek static property jisme component ki poori definition hoti hai: selector, decls/vars count, aur template function (creation + update instructions). Decorator compile ke baad gayab ho jaata hai; runtime Angular sirf δcmp padhta hai. Isliye reflect-metadata ki zaroorat nahi aur definition tree-shakable hai.

MEDIUM Angular template ko type-check kaise karta hai — apna type checker likha hai kya?

Nahi. ngtsc template AST se ek synthetic TypeScript function generate karta hai — Type Check Block (TCB) — jisme har binding ek TS expression ban jaati hai (context = component instance). Phir normal tsc use check karta hai. Faayda: TS ke saare rules (strict null, narrowing, generics) free mein templates pe lag jaate hain, aur errors TS ki hi vocabulary mein aate hain. strictTemplates is checking ki strictness control karta hai.

MEDIUM Partial compilation kya hai aur kyun zaroori hai?

Isse pehle ye problem ngcc se handle hoti thi (node_modules rewrite) — slow aur flaky; v16 mein hataya gaya.

HARD Ivy ka compiled output "dirty checking baked in" kaise karta hai — ek interpolation ka example se explain karo.

{{ label }} compile hoke update block mein δδtextInterpolate(ctx.label) ban-ta hai. Ye instruction andar se: (1) current binding index (LView array ka slot) se pichli value uthata hai; (2) nayi value se Object.is comparison karta hai; (3) same hai toh return — DOM touch nahi; (4) alag hai toh LView mein nayi value store karke nativeNode.textContent = value set karta hai. Har binding ka apna slot hai (isliye vars: N count hota hai), aur δδadvance(n) instruction pointer ko aage badhata hai. Deep point: change detection ek generic tree-walking algorithm nahi hai — wo compiled per-component update function ka execution hai, jisme comparisons pehle se placed hain. Isliye Ivy predictable aur fast hai, aur isiliye decls/vars compile time pe count ho paate hain (memory pre-allocate ho jaati hai, S6).

Ivy Runtime Architecture

Covers: LView / TView · Instruction set · Slot allocation (decls/vars) · Incremental DOM · Locality · Tree shaking · ViewEngine se comparison

1. What is it?

Ivy = Angular ka compiler + runtime (v9 se default). Runtime side pe uska dil do data structures hain:

  • TView (Template View) — static, per-component-type, ek hi baar banta hai. "Is component ka blueprint": kitne nodes, kaunse bindings, kaunse directives.
  • LView (Logical View) — dynamic, per-component-instance. Ek flat array jisme actual DOM nodes, binding values, child views, injector — sab slots mein pade hain.

Aur instructions — wo functions (δδelementStart, δδtextInterpolate) jo compiled template call karta hai. Ivy ka runtime basically "instructions ka interpreter nahi, executor hai" — kaam compiled code karta hai, runtime sirf state (LView) manage karta hai.

2. Why do we need it?

Ivy se pehle ViewEngine tha: template ek generic data structure mein compile hota tha, jise runtime interpret karta tha. Interpreter ka matlab: poora interpretation engine bundle mein aana zaroori hai, chahe tumhari app uska 10% use kare. Aur wo engine har node type, har binding type handle karta — generic = slow + bada.

Ivy ne ise ulta kar diya: template ko code mein compile karo, runtime ko chhota rakho. Har feature ek alag instruction function hai; use nahi kiya toh import hi nahi hua toh bundle mein nahi aaya. Ye "tree shakable runtime" hai.

3. What problem does it solve?

  • Bundle size → sirf use kiye gaye instructions ship hote hain (Hello World ~30KB tak gira).
  • Debugging → stack trace mein tumhare template ka code dikhta hai, framework ke interpreter ka nahi. ng.getComponent($0) jaise debug APIs.
  • Incremental builds → locality (S1) — per-component compilation.
  • Memory predictability → decls/vars compile-time pe pata hain, LView pre-allocated array hai.

10. Code Explanation — Line by Line

  • L1: track $index ka matlab: "position hi identity hai". Reorder pe har position ka content badla, toh Angular ke liye har item "changed" hai. Component instances position se bandhe hain — unki internal state (scroll, input focus, half-filled form) galat row pe chali jaati hai. Ye correctness bug hai, sirf performance nahi.
  • L2: track order.id = stable identity. Ivy ka δδrepeaterTrackByIdentity purane LViews ko naye array ke saath match karta hai, aur teen operations decide karta hai: keep (kuch mat karo), move (DOM node ko reposition, LView reuse), create/destroy (sirf naye/hataye items). Ye Ivy ke naye @for ka algorithm hai — purane *ngFor se tez, kyunki wo IterableDiffer abstraction se guzarta tha.
  • Deep implication: LView reuse ka matlab hai component instance reuse. Constructor dobara nahi chalega, ngOnInit dobara nahi chalega, @Input sirf badli hui values pe set hongi. Isliye track ka sahi hona sirf speed nahi, identity semantics hai. Interview mein yehi angle bolo.

13. Performance

Ivy ke design se teen practical performance rules nikalte hain. (1) Bindings ki count matter karti hai — har binding ek vars slot + har CD cycle mein ek comparison. Ek template mein 200 bindings = 200 comparisons per cycle, per instance. Bade lists mein multiply hota hai. (2) Component count matter karta hai — har component instance ek LView (array allocation) + ek CD tree node + potentially ek ElementInjector. (3) track matter karta hai — galat track = LView destroy/create = DOM churn + GC pressure (Ch1 S8). Measurement ka rasta: Angular DevTools ka profiler — wo per-component CD time dikhata hai, aur "change detection cycles" ka flame graph deta hai. Guess mat karo, profile karo.

14. Edge Cases

  • __ngContext__ — har host DOM node pe ek hidden property jo LView tak pahunchati hai. Debug APIs ka mechanism yahi hai. Memory profiler mein detached DOM ke saath ye LView bhi retain ho sakta hai (Ch1 S8 leak pattern).
  • Embedded views (<ng-template>, @if, @for) ke apne LViews hote hain, parent LView ke slot mein reference. Isliye @if ke andar ka content hide nahi hota — destroy hota hai (LView + DOM dono).
  • ngDevMode flag runtime pe extra assertions on karta hai; production build mein wo poora code path shake ho jaata hai — isliye kuch errors sirf dev mein.
  • Dynamic components (createComponent()) manually LView banate hain — ViewContainerRef ke through parent LView se attach; destroy karna tumhari zimmedari.
  • Standalone components ke dependencies static array mein hote hain; dynamic imports ke saath compiler use resolve nahi kar sakta — isliye truly dynamic component loading import() + createComponent ka pattern maangta hai.

15. Interview Questions

EASY Ivy kya hai?

Angular ka compiler + runtime (v9 se default), jo ViewEngine ko replace karta hai. Templates ko generic data structures ki jagah JavaScript instructions mein compile karta hai. Faayde: chhota tree-shakable bundle, better debugging (stack traces + ng.* APIs), incremental compilation (locality), aur predictable performance.

EASY Angular Virtual DOM use karta hai?

Nahi. Angular incremental DOM use karta hai: compiled template ki update-instructions har binding ko pichli value se compare karke seedha DOM mutate karti hain. Koi intermediate tree nahi banta, koi diff nahi hota. Faayda: update pe ~zero allocation, GC pressure kam. Trade-off: templates ko compile karna zaroori (DSL), JSX jaisi runtime flexibility nahi.

MEDIUM LView aur TView mein farak — aur ye separation kyun?

TView = static blueprint, per component type, ek baar banta hai: node structure (TNodes), directive defs, kitne decls/vars. LView = dynamic state, per component instance: actual DOM node references, binding ki pichli values, child views, injector. Separation ka faayda: 1000 rows = 1000 LViews par sirf 1 TView — static memory ek baar. Aur LView ek flat array hai (index access, monomorphic — V8 friendly), object nahi.

MEDIUM decls aur vars kya hain, aur compile time pe kyun count hote hain?

decls = template mein kitne DOM nodes/directives create hone hain; vars = kitni bindings ki purani values track karni hain. Dono compile time pe count ho jaate hain kyunki template static hai. Isse LView array ka exact size pata chal jaata hai — ek baar allocate, kabhi grow nahi. Aur δδadvance(n) se binding pointer O(1) mein aage badhta hai, koi map lookup nahi. Predictable memory + fast access.

HARD @for mein track sirf performance optimization hai ya correctness ka issue bhi? Detail mein.

Dono — aur correctness wala pehlu zyada important hai. track Angular ko batata hai "do renders ke beech ye item wahi item hai". Uske basis pe Ivy purane LView ko reuse (keep/move) karta hai ya destroy+create. Agar track $index ho aur list reorder ho, toh Angular ke liye position 0 ka item "badal gaya" — wo existing LView ko naye data se update karta hai. Consequence: us row ke component instance ki internal state (focus, scroll, half-typed input, chal rahi animation, subscription) galat data ke saath jud jaati hai. Constructor/ngOnInit nahi chalte kyunki instance destroy hi nahi hua. Ye silent data-corruption class ka bug hai — QA aksar miss karta hai. Stable ID se track karne pe Ivy identity match karke DOM nodes ko move karta hai (LView reuse) — state item ke saath travel karti hai. Performance benefit (kam DOM operations, kam allocations) uska side effect hai. Isliye Angular ne @for mein track ko mandatory bana diya.

SCENARIO Tumhare heatmap grid (1000 rows × 20 columns) mein scroll janky hai aur har filter change pe 800ms freeze hota hai. Ivy ke knowledge se debug + fix plan?

(1) Measure: Angular DevTools profiler se CD cycle record karo — dekho kaunse components ka detectChanges time kha raha hai, aur kitne instances hain. Chrome Performance tab se DOM node count + GC activity. Guess mat karo. (2) Instance count ghatao: agar har cell ek component hai toh 20,000 LViews + 20,000 CD nodes hain. Cell ko plain <td> banao, sirf row-level component rakho (ya wo bhi nahi). Ye aksar single biggest win hai. (3) Bindings ginno: per cell 4 bindings × 20,000 = 80,000 comparisons per CD cycle. [style.background] jaise bindings ko CSS classes se replace karo (ek binding, ya static class). (4) track verify karo: track cell.id — filter pe reorder ho raha hai toh $index se poora grid recreate ho raha hoga. Ye 800ms ka bada hissa ho sakta hai. (5) CD scope chhota karo: grid component ko OnPush + inputs ko signals se drive karo, taaki filter change pe sirf grid ka subtree check ho, poora app nahi. Zoneless ho toh aur precise. (6) Virtualization: 1000 rows sach mein DOM mein chahiye? CDK cdk-virtual-scroll-viewport se sirf visible ~30 rows render karo — 20,000 nodes se 600 nodes. Ye structural fix hai, baaki micro-optimizations. (7) Verify with numbers: har step ke baad profiler dobara — kaunse change ne kitna diya, wo document karo. Punchline: pehle instance aur node count attack karo (structural), phir CD scope (OnPush/signals), phir micro (bindings). Ye ordering hi senior thinking dikhati hai.

11. Common Mistakes

Galtiyan
  • Dynamic import() mein computed path (import(`./f/${name}`)) — bundler static analysis nahi kar sakta; ya toh chunk nahi banta ya poora folder bundle ho jaata hai.
  • Budgets set na karna (ya maximumError ko itna bada rakhna ki kabhi fire hi na ho) — bundle size chupke se badhta rehta hai, 6 mahine baad koi nahi jaanta kab kya bada hua.
  • Lazy route banaya par uske andar ka module kisi eager file se bhi import ho gaya — chunk ban ke bhi initial bundle mein aa jaata hai. (Barrel files ka classic gunaah, S3.)
  • Bundle analysis kabhi na karna — "app dheere hai" ka 70% jawab source-map-explorer ke ek screenshot mein hota hai.
  • Production build ko dev build maan ke debug karna — ngDevMode assertions aur error messages prod mein alag (ya gayab) hote hain.
  • PreloadAllModules ko blindly on karna — sab chunks download ho gaye, code splitting ka faayda hi khatam (mobile data pe bura). Custom strategy likho (sirf likely routes).

12. Best Practices

Production rules
  • Budgets CI mein error level pe — initial bundle ka regression PR pe hi pakda jaye.
  • Har feature route lazy; bhaari widgets @defer mein. Default: kuch bhi eager tabhi jab first paint ke liye chahiye.
  • Har release se pehle (ya monthly) bundle analysis — top 10 contributors ki list maintain karo.
  • Content hashing on + long cache headers (1 year, immutable) on hashed files; index.html pe no-cache.
  • @defer ke saath hamesha @placeholder aur @error — layout shift aur stale-chunk failures dono handle.
  • Third-party libs ko import karne se pehle bundlephobia check karo; moment-class packages (large + no tree shaking) reject karo.

13. Performance

Build performance: esbuild ne webpack-era ke minute-lambe builds ko seconds mein badla; par bade apps mein bottleneck ab bundling nahi, type checking (S5 ka TCB) hota hai. .angular/cache ko CI mein persist karo — 2x tak speedup. Nx jaise tools affected graph se sirf badle hue projects build/test karte hain.

Runtime performance: teen numbers track karo — (1) initial bundle size (gzipped/brotli, transfer size — raw nahi), (2) chunk count aur waterfall depth (bahut zyada chhote chunks = HTTP overhead), (3) LCP/TTI real devices pe (Lighthouse CI). Ek practical target: initial JS < 250KB compressed. Uske upar har 100KB mobile pe ~150-300ms parse+execute cost hai.

14. Edge Cases

  • Stale chunk 404: deploy ke baad purana tab naya chunk maangta hai jo hash badalne se ab nahi hai. Solutions: @error block, service worker ka SwUpdate prompt, ya server-side fallback jo purane hashes ko kuch der rakhe.
  • Side-effectful imports (import './polyfill-something') tree shaking rok dete hain — package.json ka "sideEffects" field bundler ko batata hai kis file pe bharosa karein.
  • CSS ka tree shaking JS jaisa nahi hota — component styles scoped hote hain (emulated encapsulation) par global styles poore aate hain.
  • SSR build do outputs deta hai (browser/ + server/); server bundle mein browser-only code na jaye iske liye platform checks zaroori.
  • Prerendering (SSG) build time pe routes render karta hai — dynamic routes ke liye getPrerenderParams chahiye.
  • Source maps production mein: upload karo error tracking pe, serve mat karo (ya hidden-source-map).

Rendering Pipeline

Covers: Creation vs update pass · tick() · Zone.js ka role · OnPush & dirty marking · Signals & zoneless · SSR + Hydration · afterRender hooks

1. What is it?

Rendering pipeline = wo poora loop jo "component ki state" ko "screen pe pixels" tak le jaata hai, aur phir har change pe unhe sync rakhta hai. Do halves hain:

  • Creation — ek baar: DOM nodes banao, listeners lagao, child components create karo (S6 ka creation pass).
  • Update (Change Detection) — baar-baar: bindings ki nayi values ko purani se compare karo, badli hui ko DOM mein likho.

Asli sawaal ye hai jo poore Angular ecosystem ko define karta hai: update kab chalaayein? Iska jawab teen daur mein badla: Zone.js → OnPush → Signals.

2. Why do we need it?

Kisi bhi UI framework ka core problem yahi hai: state badli, DOM ko pata kaise chale? Do raste hain — (a) framework ko batao ("setState") ya (b) framework khud pata kar le. Angular ne originally (b) chuna: Zone.js se har async operation intercept karo aur uske baad maan lo "shayad kuch badla" → poore component tree ko check karo. Ye ergonomically brilliant tha (tum plain this.count++ likho, sab kaam karta hai) par computationally mehenga.

3. What problem does it solve?

  • Manual DOM sync → declarative templates + automatic update.
  • Kab check karein → Zone.js (async ke baad), OnPush (input/event/async pipe), Signals (exact consumer).
  • Slow first paint → SSR se HTML pehle, hydration se interactivity baad mein.
  • Wasteful checks → OnPush se subtree skip, signals se surgical updates.

  • L2: mousemove 60+ times/second fire hota hai. Zone ke andar hota toh 60 tick() per second = 60 baar poora tree check. Ye classic jank ka source hai (Ch1 S6 ka task-chunking wala theme).
  • L3–L4: Jab sach mein UI update chahiye, tabhi zone.run() se wapas Angular ke andar aao — ab CD chalega, ek baar. Ye "opt-out phir opt-in" pattern hai.

Ye ek acchi misaal hai ki architecture ka change ek poori class ke bugs ko kaise mita deta hai.

11. Common Mistakes

Galtiyan
  • Template mein function call ({{ getTotal() }}) — har CD cycle mein chalta hai, poore tree mein har baar. computed() use karo (memoized + tracked).
  • ChangeDetectorRef.detectChanges() ko har jagah "fix" ki tarah lagana — root cause chhupa diya. Pehle poochho: state kis mechanism se badli, aur Angular ko kyun nahi pata chala?
  • OnPush ke saath object mutate karna (this.data.items.push(x)) — reference wahi hai, Angular ko change dikha hi nahi (Ch1 S4 ka immutability point). Naya reference banao ya signals use karo.
  • ExpressionChangedAfterItHasBeenChecked ko setTimeout(() => ...) se dabana — bug wahin hai, ek tick baad. Root cause: CD ke beech state mutate ho rahi hai.
  • Zoneless enable karke third-party lib use karna jo setTimeout se state badalti hai — CD trigger nahi hogi. Wrapper mein signal write ya ApplicationRef.tick() chahiye.
  • SSR mein window/document/localStorage component constructor mein — server pe crash. afterNextRender() ya isPlatformBrowser check.

12. Best Practices

Production rules
  • Naye apps: zoneless + signals se shuru karo. Purane: har component OnPush, phir signals mein migrate.
  • Derived state hamesha computed() mein, template mein kabhi logic nahi.
  • Browser-only side effects afterNextRender()/afterRenderEffect() mein — SSR-safe by construction (tumhare PR reviews ka ye recurring theme hai).
  • Immutable updates (Ch1 S4) — OnPush aur signals dono reference equality pe chalte hain.
  • High-frequency events (scroll, mousemove, drag) — zoneless mein natural, zone apps mein runOutsideAngular.
  • Performance debugging: Angular DevTools profiler — kaun kitni baar check hua, kitna time.

13. Performance

CD ka cost = (kitni baar chali) × (kitne bindings check kiye). Zone.js pehle factor ko bada karta hai (har async operation), Default strategy doosre ko (poora tree). OnPush doosre ko kaatta hai, zoneless+signals dono ko. Practical numbers: 1000 bindings ka app, 60fps pe mousemove → 60,000 comparisons/sec sirf ek pointer hilane se. Wahi app signals ke saath: 1 comparison.

Doosra bada lever hydration hai. SSR bina hydration ke = server ne HTML banaya, client ne use phenk ke dobara banaya (destructive) — double work, flicker. Hydration ke saath DOM reuse hota hai. Incremental hydration (@defer (hydrate on viewport)) isse aur aage le jaata hai: below-the-fold ka JS load hi nahi hota jab tak zaroorat na ho — LCP aur TTI dono improve.

14. Edge Cases

  • markForCheck() upar (ancestors) dirty karta hai, detectChanges() abhi is component se neeche chalata hai, ApplicationRef.tick() root se poora. Teeno alag tools hain — confuse mat karo.
  • Zoneless mein CD trigger hone ke sources: signal writes (jo template/effect ne consume kiye), markForCheck(), host listeners, AsyncPipe emissions, ChangeDetectorRef APIs. Baaki kuch nahi.
  • effect() CD ka hissa nahi hai — wo signals ke change pe chalta hai; usme DOM likhna ya state badalna anti-pattern hai (glitchy loops). Rendering side effects afterRenderEffect() mein.
  • Hydration mismatch: server aur client ka DOM alag ho (random values, Date.now(), browser-only branching) → Angular error deta hai aur us subtree ko destructively re-render karta hai. ngSkipHydration escape hatch hai (third-party DOM-mutating libs ke liye).
  • ngDevMode ki double-check (ExpressionChanged...) sirf dev mein — production mein wo error kabhi nahi dikhega, par bug maujood rahega.
  • @defer + hydration: hydrate on viewport ka matlab hai server ne HTML bheja, JS baad mein aayega — SEO safe, interactivity deferred.

EASY Change detection kya hai aur kab trigger hoti hai (Zone.js wale Angular mein)?

Component ki state aur DOM ko sync karne ka process. Zone.js ne async APIs (setTimeout, events, Promise, XHR) patch ki hui hain; jab koi async task khatam hota hai aur microtask queue khaali hoti hai, NgZone onMicrotaskEmpty emit karta hai aur ApplicationRef.tick() chalti hai — jo root se poore tree ke update-instructions chalati hai. Angular ko ye nahi pata kya badla; wo sab check karta hai.

EASY OnPush strategy kya karti hai?

Component ko tabhi check karti hai jab wo "dirty" ho: (1) koi input ka reference badla, (2) component ya uske children mein host event chala, (3) template ke AsyncPipe ne emit kiya, (4) markForCheck() call hua. Warna poora subtree skip. Isliye OnPush ke saath immutable updates zaroori hain — object mutate karoge toh reference wahi rahega aur change detect nahi hoga.

MEDIUM markForCheck() aur detectChanges() mein farak?

markForCheck(): is component se lekar root tak ke path ko "dirty" mark karta hai — kuch turant nahi hota; agli CD cycle mein ye component check hoga. detectChanges(): turant, isi component se neeche ka subtree synchronously check karta hai. Ek scheduling hai, doosra execution. Aur ApplicationRef.tick() root se poora tree. OnPush + async update ke liye usually markForCheck() chahiye.

MEDIUM Signals ne change detection ko fundamentally kaise badla?

Zone.js model "push-based invalidation without information" tha: kuch async hua → sab check karo. Signals ek dependency graph banate hain: jab template ne count() read kiya, Angular ne record kiya ki "ye view is signal ka consumer hai". count.set() pe signal apne consumers ko exactly notify karta hai, aur sirf wo views dirty hote hain. Matlab CD ka scope O(tree) se O(affected views) ho gaya. Isi wajah se Zone.js optional ho paaya (zoneless) — kyunki ab "kuch badla hoga" guess karne ki zaroorat nahi, framework ko theek-theek pata hai.

HARD ExpressionChangedAfterItHasBeenCheckedError kyun aati hai, aur ye dev-only kyun hai?

Dev mode mein Angular har CD cycle ke baad ek verification pass chalata hai: sab bindings dobara evaluate karke check karta hai ki values wahi hain. Agar alag hain, matlab CD pass ke dauran ya baad kisi code ne state badal di (aksar ngAfterViewInit, child ka output parent ki state badal raha ho, ya ek shared service jo CD ke beech mutate ho). Ye unidirectional data flow ka violation hai. Production mein ye check nahi chalta (performance) — par bug rehta hai: UI ek frame purana dikhega, ya worst case do-tarfa update loop ban jaayega jo har cycle mein re-trigger hoga. Fixes: state ko CD se pehle set karo (ngOnInit/constructor), ya afterNextRender() mein (jo cycle ke bahar hai), ya signals use karo (unka notification mechanism CD scheduler ke saath sahi tarike se integrate hai). setTimeout se "fix" karna sirf error ko chhupata hai — ek tick baad wahi flow, ab bina warning ke. Ye error framework ka diagnostic hai, dushman nahi.

SCENARIO Ek Angular app ko zoneless mein migrate karna hai. Wo 4 saal purani hai, 300 components, kuch third-party libs bhi hain. Step-by-step migration strategy?

Zoneless "ek flag" nahi hai — ye contract change hai: pehle Angular guess karta tha, ab tumhe batana padega. (1) Pre-requisite phase: sab components OnPush pe lao (ye zoneless ka rehearsal hai — OnPush mein bhi tumhe explicitly dirty mark karna padta hai). Har jagah jahan markForCheck()/detectChanges() lagana pada, wahi jagah zoneless mein bhi attention maangegi. (2) State ko signals mein migrate karo — component-by-component: fields → signal(), getters → computed(), Observables → toSignal(). Ye asli kaam hai aur isi se CD automatically sahi hoti hai. (3) Zone-dependent patterns dhundho: setTimeout/setInterval ke baad state update (ab CD trigger nahi hogi), manual addEventListener, third-party callbacks (chart lib ka onComplete, WebSocket handlers). Har ek ko signal write mein convert karo — signal write CD schedule karta hai. (4) Third-party libs: jo apne callbacks se Angular state badalti hain, unhe wrapper service mein encapsulate karo jo signal set kare. Agar lib DOM khud mutate karti hai, wo zoneless mein bhi theek hai (Angular ko farak nahi padta). (5) Enable + test: provideZonelessChangeDetection() ek feature branch mein; e2e suite chalao — jo screens "freeze" dikhein, wahan koi update path missing hai. Ye debugging aasan hai: symptom hamesha "UI update nahi hua", aur cause hamesha "state Angular ko bataye bina badli". (6) Incremental rollout agar possible ho: ek route/feature ko pehle. (7) Bundle win verify karo: Zone.js ~15KB + har async call ka overhead gaya. Sabse important framing jo bolni hai: "zoneless migration asal mein signals migration hai; zone hatana uska aakhri step hai, pehla nahi."

S-09

Dependency Graph & DI Architecture

Covers: Injector hierarchy · EnvironmentInjector vs ElementInjector · Resolution algorithm · Provider types · providedIn · Resolution modifiers · Module dependency graph · Circular deps

1. What is it?

Do "dependency graphs" hain jo naam mein similar hain par bilkul alag hain — aur seniors ye distinction saaf rakhte hain:

  • Runtime DI graph — kaunsi service kis service pe depend karti hai, aur wo kis injector se resolve hoti hai. Ye Angular ka DI system hai.
  • Static module graph — kaunsi file kis file ko import karti hai. Ye bundler ka domain hai (S3, S7) — chunks aur circular deps yahin se aate hain.

Ye section dono ko cover karta hai, kyunki architecture ke sawaal aksar dono ko touch karte hain.

2. Why do we need it?

DI ke bina: har component apni dependencies khud new karta — testing impossible (mock kahan lagaoge?), configuration hard-coded, aur singletons ke liye global variables. DI inversion deta hai: component bolta hai "mujhe PaymentGateway chahiye", aur kaun sa concrete implementation milega ye bahar se decide hota hai (Ch2 S5 ka abstract-token pattern). Hierarchical DI isse aur aage le jaata hai: same token, alag scope mein alag instance — jo Angular ka sabse under-used superpower hai.

3. What problem does it solve?

  • Coupling → consumers abstraction pe depend karte hain, implementation pe nahi.
  • Testability → token override karke mock inject karo.
  • Scoping → service ka lifetime = injector ka lifetime (per-app, per-route, per-component).
  • Configuration → InjectionToken se values/config inject.
  • Tree shaking → providedIn: 'root' = unused service bundle se drop.

9. Advanced Example — skipSelf aur recursive tree components

@Component({
  selector: 'app-tree-node',
  providers: [NodeContextService],                          // L1: har node apna
  template: `
    <div>{{ node().name }} (depth {{ ctx.depth }})</div>
    @for (child of node().children; track child.id) {
      <app-tree-node [node]="child" />      <!-- recursion! -->
    }
  `,
})
export class TreeNodeComponent {
  node = input.required<TreeNode>();

  protected ctx = inject(NodeContextService);                  // L2: apna
  private parentCtx = inject(NodeContextService, {              // L3
    skipSelf: true,
    optional: true,                                            // L4: root pe parent nahi
  });

  constructor() {
    this.ctx.depth = (this.parentCtx?.depth ?? -1) + 1;       // L5
  }
}

10. Code Explanation — Line by Line

  • L1: Component-level provider — har <app-tree-node> instance ka apna NodeContextService (ElementInjector mein). Ye DOM hierarchy ke saath naturally nest hota hai.
  • L2: Normal inject() — apne ElementInjector se milta hai (step 1 mein hi hit).
  • L3: skipSelf: true — "apna wala mat do, upar se dhundho". Yahan wo parent tree-node ka instance dega, kyunki ElementInjector chain DOM structure follow karti hai.
  • L4: Root node ka koi parent tree-node nahi — optional ke bina NullInjectorError aayega. Ye recursive components ka classic pattern hai.
  • L5: Depth calculate ho gayi bina koi input pass kiye, bina koi global state ke. Har node ko apna context mila jo parent se derive hota hai. Ye hierarchical DI ka asli demo hai — Angular ki wo capability jo React (context ke bina) aur baaki frameworks mein itni natural nahi. Interview mein skipSelf ka ek real use case bolna instant depth signal hai.

11. Common Mistakes

Galtiyan
  • Har service ko providedIn: 'root' — phir flow-scoped state ko manually reset karna padta hai (aur bhool jaate ho). Scope ko lifetime se match karo.
  • Component ke providers array mein service daal ke ye samajhna ki singleton hai — har instance ka apna banega. Do jagah use kiya toh do instances, silent bug.
  • Interface ko token banana — runtime pe erase (Ch2 S9). Abstract class ya InjectionToken.
  • useClass vs useExisting confuse karna: useClass naya instance banata hai, useExisting alias hai (wahi instance). Do tokens ek instance chahiye toh useExisting.
  • inject() ko injection context ke bahar call karna (setTimeout ke andar, ya method mein) — runtime error. Field initializer, constructor, ya runInInjectionContext() ke andar hi.
  • Circular imports ko circular DI samajhna — do alag problems. Circular imports bundler/TDZ ka issue hai (Ch1 S8); circular DI (A → B → A) runtime error deta hai.

12. Best Practices

Production rules
  • Default: providedIn: 'root' stateless services ke liye (tree-shakable, singleton).
  • Scoped state (wizard, editor, feature session) = route ya component providers — lifetime automatic.
  • Swappable implementations = abstract class token + useClass/useFactory (Ch2 S5).
  • Config = InjectionToken<T> with factory — typed, tree-shakable, testable.
  • inject() fields mein (constructor injection se better — inheritance friendly, Ch2 S5).
  • Circular deps: madge --circular CI mein; DI cycle aaye toh design smell hai — ek third service nikaalo.

13. Performance

DI ka runtime cost low hai par zero nahi: har inject() ek chain walk hai (aksar 1-3 levels), aur har ElementInjector ek LView slot leta hai. Component-level providers ka matlab hai har instance pe ek injector — 10,000 rows mein ye add up hota hai. Rule: bade lists ke row components mein providers mat daalo.

Tree shaking angle: providedIn: 'root' service ka reference tabhi banta hai jab koi inject() kare — nahi kiya toh bundler drop kar deta hai. app.config.ts ke providers array mein daali gayi service hamesha bundle mein hai (array mein reference hai). Isliye services ko config array mein sirf tab daalo jab token override kar rahe ho.

Module graph: circular imports lazy chunking ko tod dete hain (bundler dono ko ek chunk mein daal deta hai) aur incremental rebuild ka scope badha dete hain (S3, S7). Ye build-time performance ka silent killer hai.

14. Edge Cases

  • runInInjectionContext(injector, () => inject(X)) — inject() ko bahar use karne ka legal rasta (e.g. ek callback mein).
  • InjectionToken with {providedIn: 'root', factory: () => ...} — tree-shakable config token, bina providers array ke.
  • multi: true providers merge hote hain hierarchy mein nahi — child injector ka multi-provider parent wale ko replace karta hai (concat nahi). Common surprise.
  • Lazy route ke providers uska apna EnvironmentInjector banate hain — route destroy pe wo injector aur uske instances destroy (ngOnDestroy chalega). Eager routes mein ye nahi hota.
  • @Self() + directive: element pe do directives, ek doosre ka service chahiye — self se sirf usi element se.
  • DI cycle (A inject B, B inject A) — NG0200: Circular dependency in DI. Fix: teesri service, ya forwardRef (band-aid), ya design revisit.

15. Interview Questions

EASY providedIn: 'root' ka matlab kya hai?

Service root EnvironmentInjector mein register hai — app-wide singleton. Aur ye tree-shakable hai: registration service ki apni file mein hai, toh agar koi use nahi karta, bundler use drop kar deta hai. app.config.ts ke providers array mein daalne se ye faayda nahi milta (array reference hamesha bundle mein hai).

EASY Component ke providers array mein service daalne se kya hota hai?

Har component instance ke liye alag service instance banta hai (component ke ElementInjector mein). Component destroy hone pe service bhi destroy (ngOnDestroy chalega). Per-component state ke liye useful; galti se app-wide singleton samajh liya toh do instances ka silent bug.

MEDIUM EnvironmentInjector aur ElementInjector mein farak?

ElementInjector DOM/component hierarchy follow karta hai — component ke providers/viewProviders aur us element pe lagi directives yahan rehti hain. EnvironmentInjector logical/module scope hai — root (app.config), lazy route providers, platform. Resolution pehle ElementInjector chain upar chadhta hai (component → parent component → ... → root component), phir EnvironmentInjector chain mein jaata hai (route → root → platform → NullInjector). Isliye component-level provider hamesha root wale ko "jeet" jaata hai.

MEDIUM skipSelf aur self modifiers ka ek-ek real use case?

skipSelf: recursive tree component jahan har node apna context provide karta hai par parent ka context bhi chahiye (depth calculate karne ke liye) — inject(Ctx, {skipSelf: true, optional: true}). self: ek directive jo sirf apne host element pe provide ki gayi service accept kare, parent se accidentally na uthaye (strict local contract) — form directives mein common. Dono resolution ki starting position badalte hain, algorithm wahi rehta hai.

HARD inject(SomeService) call pe Angular kya karta hai, step by step? Aur NullInjectorError exactly kab aati hai?

(1) Angular current injection context se active injector uthata hai (isliye inject() sirf field initializer/constructor/runInInjectionContext mein valid hai — bahar context hi nahi hota). (2) Agar wo ElementInjector hai: us element ke providers/viewProviders aur directives mein token dhundho. Nahi mila toh parent element ka injector — DOM/view hierarchy mein upar chadho. (3) Root component ke ElementInjector ke baad EnvironmentInjector chain: pehle route-level (agar lazy route ke andar ho), phir root, phir platform. (4) Har injector mein: token → provider record; agar instance cache mein hai toh wahi return (per-injector singleton), warna factory chalao (jo recursively apni dependencies inject karega) aur cache mein daalo. (5) Chain ka aakhri node NullInjector hai jo har request pe throw karta hai — yahi NG0201: No provider for X ka source hai. Modifiers is walk ko modify karte hain: skipSelf step 2 ko apne injector se shuru nahi karta, self use aage badhne nahi deta, optional NullInjector ke throw ko null mein badal deta hai, host element chain ko host component pe rok deta hai. Deep point: instances lazy hain — provider register hone ka matlab instantiate hona nahi; pehle inject() pe hi constructor chalta hai.

SCENARIO Ek multi-tenant SaaS app hai. Har tenant ka apna theme, feature flags, aur API base URL hai. Aur ek "impersonation" mode hai jahan admin doosre tenant ke context mein browse karta hai. DI se ye architecture kaise design karoge?

Ye DI scoping ka perfect problem hai. (1) Config as token: export const TENANT_CONFIG = new InjectionToken<TenantConfig>('tenant') — interface nahi, token (Ch2 S9). Services inject(TENANT_CONFIG) karti hain, kabhi global se nahi padhtin. (2) Root pe default: bootstrap ke initializer (S4) se current tenant ka config load, useValue/useFactory se root injector mein provide. (3) Impersonation = child injector: admin jab kisi tenant ko impersonate kare, us route subtree ke liye providers: [{provide: TENANT_CONFIG, useValue: otherTenantConfig}] — route-level EnvironmentInjector. Us subtree ke saare components/services automatically doosra config dekhte hain; koi if (isImpersonating) condition kahin nahi. Impersonation route chhoda → injector destroy → wapas normal. Ye hi hierarchical DI ka killer use case hai. (4) API base URL: HTTP interceptor inject(TENANT_CONFIG) se URL prefix kare — par dhyan: interceptors root injector mein chalte hain, toh route-scoped config unhe nahi dikhega. Solution: request context (HttpContext) se tenant pass karo, ya per-tenant HttpClient instance route injector mein. Ye wo edge case hai jo interviewer sunna chahta hai. (5) Theme: config-driven CSS custom properties, component-level nahi. (6) Testing: TestBed mein token override — poora tenant simulate, zero mocking of internals. (7) Security note: impersonation ka authorization server-side enforce ho; DI sirf UI context hai. Ye answer DI hierarchy + tokens + interceptor edge case + testing + security touch karta hai — SDE-3 bar.
