S-01

Component & Metadata

Covers: Component kya hai · @Component ka har option · Metadata → δcmp · Lifecycle hooks ka map · Standalone default

1. What is it?

Component = ek TypeScript class + uski metadata. Class mein state aur behavior hota hai; metadata (@Component({...})) Angular ko batati hai is class ko UI mein kaise badalna hai: kaunse selector pe render karo, kaunsa template use karo, kaunse styles lagao, kaunsi dependencies chahiye.

Ch3 S5 se yaad karo: @Component decorator compile time pe gayab ho jaata hai. Uski jagah class pe ek static property aati hai — δcmp — jisme wo saari information encoded hoti hai, plus generated template function. Yaani metadata ek compile-time DSL hai, runtime object nahi.

2. Why do we need it?

UI ko todna padta hai. 200-screen app ek file mein likhna asambhav hai. Component boundary teen cheezein deta hai: (1) encapsulation — internal state bahar nahi leak karti, styles bahar nahi bikharte; (2) reuse — ek button 400 jagah; (3) change detection boundary — OnPush/signals ke saath component hi wo unit hai jise skip ya update kiya ja sakta hai (Ch3 S8).

3. What problem does it solve?

  • Monolithic templates → composable units.
  • Global CSS collisions → scoped styles (S4).
  • Manual DOM wiring → declarative template + bindings.
  • Testing → ek component = ek testable unit with clear inputs/outputs.

6. Architecture — ek modern component ka anatomy

@Component({
  selector: 'app-order-card',
  changeDetection: ChangeDetectionStrategy.OnPush,   // hamesha
  imports: [CurrencyPipe, StatusBadgeComponent],     // locality
  template: `...`,
  styleUrl: './order-card.component.scss',
  host: { '[class.selected]': 'selected()' },        // S8
})
export class OrderCardComponent {
  // 1. Dependencies (private — template ko nahi chahiye)
  private analytics = inject(AnalyticsService);

  // 2. Inputs (public API — parent set karta hai)
  order = input.required<Order>();
  selected = input(false, { transform: booleanAttribute });

  // 3. Outputs (public API — parent sunta hai)
  cardClicked = output<string>();

  // 4. Derived state (protected — template use karega, Ch2 S5)
  protected label = computed(() => STATUS_META[this.order().status].label);

  // 5. Methods
  protected onClick() {
    this.analytics.track('card_click');
    this.cardClicked.emit(this.order().id);
  }
}

Ye ordering (deps → inputs → outputs → derived → methods) ek convention hai jo har component ko skim-karne layak banati hai. Public API (inputs/outputs) hamesha upar, taaki koi bhi 5 second mein samajh le ki component ka contract kya hai.

9. Advanced Example — lifecycle hooks vs modern equivalents

// PURANA WAY
export class OldComponent implements OnInit, OnChanges, OnDestroy {
  @Input() userId!: string;
  user: User | null = null;
  private sub?: Subscription;

  ngOnChanges(changes: SimpleChanges) {                     // L1
    if (changes['userId']) this.load();
  }
  ngOnInit() { this.load(); }                              // L2: duplicate call!
  private load() {
    this.sub?.unsubscribe();                                 // L3: manual
    this.sub = this.api.getUser(this.userId).subscribe(u => this.user = u);
  }
  ngOnDestroy() { this.sub?.unsubscribe(); }              // L4
}

// MODERN WAY — 4 lines, zero lifecycle hooks
export class NewComponent {
  userId = input.required<string>();                        // L5
  protected user = resource({                              // L6
    params: () => ({ id: this.userId() }),
    loader: ({ params, abortSignal }) => fetchUser(params.id, abortSignal),
  });
}

10. Code Explanation — Line by Line

  • L1: ngOnChanges ek SimpleChanges object deta hai — stringly-typed (changes['userId']), koi type safety nahi, aur har input ke liye if likhna padta hai. Ye API ka design 2016 ka hai.
  • L2: Classic bug: ngOnChanges pehle chalta hai (initial inputs ke saath), phir ngOnInit. Dono load() call kar rahe hain — pehla page load pe do HTTP requests. Ye production mein bahut common hai.
  • L3–L4: Manual subscription management. Bhoole toh memory leak (Ch1 S8 ka Big Five #1). load() race condition bhi de sakta hai — purana response naye ke baad aa gaya toh stale data (Ch1 S7 ka scenario).
  • L5: input.required() ek signal hai. Parent value badle toh signal update hota hai — koi hook nahi.
  • L6: resource() userId signal ko track karta hai; badalne pe purani request abort karke nayi chalata hai (abortSignal!). Component destroy pe apne aap cleanup. Loading/error states built-in (user.isLoading(), user.error()). Teen bugs (duplicate call, leak, race) API design se hi impossible ho gaye. Ye lesson yaad rakho: lifecycle hooks aksar missing reactivity ka symptom hote hain.

11. Common Mistakes

Galtiyan
  • changeDetection set na karna — Default strategy har CD cycle mein poora subtree check karti hai (Ch3 S8). OnPush ko default banao (angular.json schematics config se, Ch3 S2).
  • ngOnInit mein wo kaam karna jo constructor/field initializer mein ho sakta tha. Rule: ngOnInit tabhi jab inputs chahiye (aur ab wo bhi signals se better).
  • ngDoCheck implement karna — har CD cycle mein chalta hai, har component ke liye. 99% cases mein galat tool.
  • Constructor mein HTTP call ya DOM access — constructor DI ke liye hai, side effects ke liye nahi. DOM ke liye afterNextRender() (SSR-safe).
  • Component ko sirf "template ka container" samajhna — wo ek CD boundary, ek DI scope, aur ek lifecycle unit bhi hai.
  • ngOnChanges + ngOnInit dono mein same load logic — double fetch.

12. Best Practices

Production rules
  • OnPush hamesha (ya zoneless + signals, jahan strategy irrelevant ho jaati hai).
  • Member ordering: deps (private) → inputs → outputs → derived/computed (protected) → methods.
  • Template-facing members protected, internals private, public sirf asli API (Ch2 S5).
  • Lifecycle hooks ko last resort samjho — pehle signals/computed/resource/afterNextRender socho.
  • Component granularity: reuse, CD boundary, ya genuine complexity — teen mein se ek reason ho.
  • Har component ka public contract (inputs/outputs) 10 se kam — usse zyada matlab component do cheezein kar raha hai.

13. Performance

Har component instance ki fixed cost hai (Ch3 S6): ek LView array, ek TNode set, ek CD tree node, aur agar providers hain toh ek ElementInjector. 100 components ke page pe ye kuch nahi; 20,000 cells wale grid mein ye sab kuch hai. Doosri cost bindings ki hai — har binding ek vars slot + har CD cycle mein ek Object.is comparison.

Practical rules: (1) repeated leaf content ko component mat banao, plain elements rakho; (2) OnPush se CD subtree skip karo; (3) template mein function calls ({{ getTotal() }}) — har cycle mein chalti hain, computed() mein daalo; (4) heavy children ko @defer mein (Ch3 S7). Measure Angular DevTools profiler se — wo per-component CD time dikhata hai.

14. Edge Cases

  • exportAs: 'orderCard' — template mein #c="orderCard" se component instance milta hai (S7). Library components (jaise ngModel) isi se apna API expose karte hain.
  • Dynamic components: viewContainerRef.createComponent(Comp) — manually LView banta hai; setInput() se inputs, destroy tumhari zimmedari.
  • Component inheritance (extends BaseComponent) — metadata inherit nahi hoti (selector, template alag chahiye), par inputs/outputs/lifecycle inherit hote hain. Aksar hostDirectives (S8) ya composition better hai.
  • ngOnChanges signal inputs ke saath bhi chalta hai (agar implement karo) — par SimpleChanges mein wahi purani API. Mixing avoid karo.
  • Constructor mein inject() valid hai, par setTimeout ke andar nahi — injection context sirf synchronous construction ke dauran (Ch3 S9).

15. Interview Questions

EASY Component aur Directive mein farak?

Component ek directive hai jiska apna template hota hai. Directive (attribute directive) existing element ka behavior badalta hai — koi template nahi, koi view nahi. Internally: component ka δcmp hota hai (template function ke saath), directive ka δdir. Ek element pe ek hi component ho sakta hai, par kai directives.

EASY @Component metadata runtime pe kahan jaati hai?

Kahin nahi — decorator compile pe erase ho jaata hai (Ch3 S5). Uski information class ki static δcmp property mein compile ho jaati hai, generated template function ke saath. Isliye Angular ko reflect-metadata nahi chahiye aur component definitions tree-shakable hain.

MEDIUM Lifecycle hooks ka order batao aur bolo kaunse aaj bhi zaroori hain.

Order: constructor → ngOnChanges (inputs set hone pe, ngOnInit se pehle) → ngOnInit → ngDoCheck → ngAfterContentInit → ngAfterContentChecked → ngAfterViewInit → ngAfterViewChecked → (repeat DoCheck/Checked har cycle) → ngOnDestroy. Aaj kaunse zaroori: ngOnDestroy (par DestroyRef/takeUntilDestroyed better), aur kabhi-kabhi ngOnChanges legacy inputs ke saath. Baaki ke modern replacements: input signals + computed (ngOnChanges), field initializers (ngOnInit), afterNextRender (ngAfterViewInit, SSR-safe), effect. ngDoCheck almost hamesha galat tool hai.

MEDIUM Kab naya component banana chahiye aur kab nahi?

Banao jab: (1) reuse hoga (3+ jagah — "rule of three"), (2) CD boundary chahiye (bhaari subtree jo OnPush se skip ho sake), (3) genuine complexity/state ka alag tukda hai. Mat banao sirf "template chhota karne" ke liye — har component ki cost hai: ek LView, ek CD node, ek injector (agar providers hon). 20,000-cell grid mein har cell ko component banana measurable jank deta hai.

HARD ngOnChanges + ngOnInit ka classic double-fetch bug explain karo, aur signals usse kaise structurally rokte hain.

Sequence: Angular pehle inputs set karta hai, phir ngOnChanges call karta hai (initial values ke saath bhi!), phir ngOnInit. Log dono mein load() likh dete hain — ngOnChanges "input badla toh reload" ke liye, ngOnInit "initial load" ke liye. Pehle mount pe dono chalte hain: do identical HTTP requests. Aur agar dono ke responses out of order aayein toh stale data bhi. Signals mein ye bug likhna hi mushkil hai: input() ek signal hai, aur computed()/resource()/effect() uske value ko track karte hain — "initial" aur "changed" ka koi distinction hi nahi hai, ek hi reactive dependency hai. resource() to purani request abortSignal se cancel bhi kar deta hai. Deep lesson: lifecycle hooks imperative timing ke liye the; reactivity ne timing ko declarative bana diya, toh hooks ki zaroorat hi khatam ho gayi. Yahi Angular ke evolution ka pattern hai (Ch3 S1): implicit machinery → explicit data flow.

SCENARIO Code review mein ek component aaya: 800 lines, 12 inputs, 6 outputs, 4 lifecycle hooks, aur 3 services inject. Kya feedback doge?

Symptoms se root cause pe jao, list mat ginwao. (1) 12 inputs = multiple responsibilities: component shayad 2-3 alag cheezein kar raha hai. Inputs ko group karo — kaunse saath badalte hain? Har group ek sub-component ka candidate hai. (2) Smart + dumb mila hua: 3 services inject hain, matlab data fetching bhi kar raha hai aur presentation bhi. Container/presentational split karo (S9): ek smart wrapper jo services se baat kare, aur chhote dumb children. (3) 4 lifecycle hooks = reactivity gap: aksar ngOnChanges + ngOnInit + ngAfterViewInit ka matlab hai imperative sync ho rahi hai. Signals/computed se replace karke dekho kitne hooks bachte hain — usually zero. (4) Testing ka smell: poochho "iska ek unit test kaisa dikhega?" — 12 inputs ka setup + 3 mocks = test hi 100 lines. Ye sabse convincing argument hota hai. (5) Refactor plan: ek saath sab mat todo — pehle presentational children nikaalo (behavior nahi badlega, tests pehle likh lo), phir state ko signals mein, phir hooks hataao. Har step alag PR. (6) Kya nahi bolna: "800 lines bahut hain" — line count symptom hai, cause nahi. Ek 800-line component jo ek hi cheez karta ho theek ho sakta hai; ek 200-line component jo teen kaam kare, nahi.

S-02

Selectors

Covers: Element / attribute / class selectors · Compound & :not() · Matching algorithm · Prefixes · Selectorless (future)

1. What is it?

Selector = wo CSS-jaisa pattern jo Angular ko batata hai: "template mein ye pattern mile toh ye component/directive lagao". selector: 'app-order-card' ka matlab hai <app-order-card> element pe ye component instantiate hoga.

Important: selector CSS selector nahi hai — ek chhota subset hai. Compile time pe (Ch3 S5) ngtsc ise parse karke δcmp.selectors mein ek array-of-arrays mein encode kar deta hai, aur template ke har element ke against match karta hai.

2. Why do we need it?

Kyunki template HTML mein tumhe framework ko batana padta hai ki kaunsa custom tag ya attribute "zinda" hai. Do design choices possible thin: (a) import karo aur naam se use karo (React ka JSX model), ya (b) selector se match karo (Angular). Angular ne (b) chuna kyunki templates HTML hain, JS nahi — HTML mein "imported symbol" ka concept nahi hai. Isliye selector wo bridge hai jo declarative HTML ko TypeScript classes se jodta hai.

3. What problem does it solve?

  • HTML ↔ class mapping — declarative markup ko code se jodna.
  • Attribute directives — existing elements ko behavior dena bina wrapper element ke.
  • Progressive enhancement — <button appRipple> — native button rehta hai, behavior add hota hai.
  • Scoped matching — sirf wahi components match hote hain jo imports mein hain (locality, Ch3 S1).

6. Architecture — prefix convention aur uski wajah

// angular.json mein enforce karo:
"prefix": "app"

// Component: element selector, kebab-case, prefixed
selector: 'app-order-card'       // ✅
selector: 'order-card'           // ❌ future HTML element se clash ho sakta hai
selector: 'appOrderCard'         // ❌ HTML case-insensitive hai

// Directive: attribute selector, camelCase, prefixed
selector: '[appHighlight]'       // ✅
selector: '[highlight]'          // ❌ kisi aur lib se clash

// Library ka apna prefix (Angular Material = 'mat')
selector: 'mat-button', '[matTooltip]'

Prefix sirf convention nahi — W3C custom elements spec kehti hai ki custom element names mein hyphen hona chahiye, aur bina prefix ke tum future native HTML elements se clash kar sakte ho (kal ko <card> standard ban gaya toh?). Attribute directives pe prefix isliye ki do libraries ka same attribute naam collide na kare — aur collision ka error message bahut confusing hota hai.

9. Advanced Example — :not() aur compound selectors

// Scenario: sab <a> tags ko track karna hai, par external links ko nahi
@Directive({
  selector: 'a[routerLink]:not([appSkipTracking])',          // L1
})
export class LinkTrackingDirective {
  private analytics = inject(AnalyticsService);
  private link = inject(RouterLink);                          // L2: same element se!

  constructor() {
    inject(ElementRef).nativeElement.addEventListener('click', () =>
      this.analytics.track('nav', { to: this.link.href })
    );
  }
}

// Matches:
<a routerLink="/orders">Orders</a>                          // ✅ tracked
<a routerLink="/x" appSkipTracking>Internal</a>            // ❌ :not() ne skip kiya
<a href="https://x.com">External</a>                        // ❌ routerLink nahi hai
<button routerLink="/y">Go</button>                         // ❌ 'a' element nahi hai

10. Code Explanation — Line by Line

  • L1: Teen conditions ek saath: element a hona chahiye, routerLink attribute hona chahiye, aur appSkipTracking nahi hona chahiye. Ye compound + negation matching hai. :not() ke andar sirf simple selectors allowed hain — nested :not() ya comma nahi.
  • L2: Yahi asli power hai: inject(RouterLink) — usi element pe lagi doosri directive ka instance mil gaya. Ye ElementInjector se aata hai (Ch3 S9): ek element pe lagi saari directives ek hi injector share karti hain. Isliye ek directive doosri directive ke saath compose kar sakti hai bina parent se kuch pass kiye.
  • Deep point: Attribute selectors + ElementInjector = Angular ka "mixin" system. Tum behavior ko element pe attach kar sakte ho, aur wo behavior usi element ki doosri behaviors ko dekh sakti hai. Ye React mein natural nahi hai (wahan HOC/hooks se hota hai). Interview mein inject(SomeDirective) ka ye use case batao — instant depth.

11. Common Mistakes

Galtiyan
  • Prefix na lagana — future native elements aur third-party libs se collision. angular.json mein prefix set karo, lint rule enforce karega.
  • Component ka selector camelCase mein (appOrderCard) — HTML tag names case-insensitive hain, kaam nahi karega. Elements kebab-case, attributes camelCase.
  • Native elements ko wrapper component mein lapetna (<app-button>) — accessibility, form participation, keyboard behavior sab manually re-implement karna padta hai. Attribute selector use karo.
  • imports mein component add karna bhool jaana — Angular chup-chaap <app-card> ko unknown element samajh leta hai (ya strictTemplates ke saath error). "Component render hi nahi ho raha" ka #1 cause.
  • Class selectors (.card) use karna — styling aur behavior couple ho gaye; koi CSS refactor tumhari directive tod dega.
  • Do components ek hi element pe match karana — compile error ("More than one component matched").

12. Best Practices

Production rules
  • Components: app-kebab-case element selectors. Directives: [appCamelCase] attribute selectors.
  • Native semantics preserve karo — button/anchor/input pe attribute selector, wrapper nahi.
  • Compound selectors se contract enforce karo: selector: 'button[appButton]' — koi <div appButton> nahi likh sakta.
  • Ek element pe ek component, kai directives — behavior composition ka natural rasta.
  • Library banao toh unique prefix, aur exportAs se instance expose karo.

13. Performance

Selector matching compile time pe hoti hai — runtime cost zero. Par ek indirect cost hai: har component ki imports list mein jitne zyada items, ngtsc ko utne zyada selectors test karne padte hain per element. Bade barrel imports (Ch3 S3) se yahan bhi build slow hota hai.

Runtime pe har matched directive ek instance + LView slot leti hai (Ch3 S6). Ek element pe 5 directives = 5 instances. Attribute directives ko chhota rakho; heavy logic services mein.

14. Edge Cases

  • ng-content ka select attribute bhi selector syntax use karta hai (S7): <ng-content select="[slot=header]" />.
  • Selector mein attribute value: input[type=checkbox] — exact match, partial nahi. Regex nahi.
  • Dynamically banaye gaye HTML ([innerHTML]) pe koi selector match nahi hoga — Angular usse sirf sanitized string ki tarah dekhta hai.
  • hostDirectives (S8) selector-independent hain — wo bina attribute likhe apply hoti hain.
  • Selectorless (experimental, v20+): Angular ek aisa mode explore kar raha hai jahan template mein directly class name use ho — <OrderCard [order]="o" /> — koi selector nahi, koi imports array nahi (import statement hi kaafi). Ye React ke model ke close hai. Abhi experimental hai; interview mein "direction pata hai" batane layak.

15. Interview Questions

EASY Component aur directive ke selectors mein convention ka farak?

Component: element selector, kebab-case, prefixed — app-order-card. Directive: attribute selector, camelCase, prefixed — [appHighlight]. Wajah: component ek nayi cheez add karta hai (element), directive existing element ko modify karta hai (attribute). Aur HTML tags case-insensitive hain, isliye elements kebab-case mein.

EASY Component render nahi ho raha — pehli cheez kya check karoge?

Parent ke imports array mein wo component hai ya nahi (standalone mein). Ye sabse common cause hai. Doosra: selector spelling/case. Teesra: strictTemplates on hai toh compiler batayega ("is not a known element") — off ho toh Angular chup-chaap use unknown HTML element maan leta hai.

MEDIUM Native button ke liye wrapper component vs attribute selector — kaunsa better aur kyun?

Attribute selector (selector: 'button[appButton]'). Wrapper (<app-button>) mein tumhe native button ki har cheez manually re-implement karni padti hai: keyboard activation (Enter/Space), type="submit" ka form behavior, disabled ka focus behavior, screen-reader semantics, focus ring. Attribute selector mein native element wahi rehta hai — tum sirf styling aur extra behavior add karte ho. Angular Material yahi karta hai. Bonus: compound selector (button[appButton], a[appButton]) se tum allowed hosts ko compile-time pe restrict bhi kar sakte ho.

MEDIUM Ek element pe kitni directives aur kitne components lag sakte hain?

Components: exactly ek (do match hue toh compile error) — kyunki component apna view aur template laata hai, do views ek element pe nahi ho sakti. Directives: jitni chaho, sab apply hongi. Aur wo ek doosre ko inject() kar sakti hain, kyunki ek element ki saari directives ek hi ElementInjector share karti hain (Ch3 S9). Yahi Angular ka behavior-composition mechanism hai.

HARD Selector matching runtime pe hoti hai ya compile time pe? Iska ek practical consequence batao.

Compile time pe. ngtsc template AST ke har element ko component ki imports list ke selectors ke against match karta hai, aur codegen mein directly directive indices embed kar deta hai (δcmp.dependencies). Runtime pe koi selector engine nahi chalta. Consequences: (1) [innerHTML] se inject kiya gaya <app-card> kabhi component nahi banega — wo sirf unknown HTML element rahega; dynamic components ke liye createComponent() chahiye. (2) Locality (Ch3 S1): sirf imports mein declared selectors test hote hain, poore app ke nahi — isliye do features same selector use kar sakti hain bina collision ke (though confusing hoga). (3) Performance: matching ka runtime cost zero hai; par bade imports arrays build time badhate hain. (4) Ye locality hi selectorless design ko possible bana rahi hai — agar matching compile-time pe hai, toh selector ki jagah class reference bhi kaam kar sakta hai.

SCENARIO Tumhe ek design system banana hai jisme Button, Input, Card hon. Har ek ke liye selector strategy decide karo, aur batao ek naya dev galat use na kar paaye — ye kaise ensure karoge?

Har component ka selector uske semantics se aata hai. (1) Button: attribute selector on native hosts — selector: 'button[dsButton], a[dsButton]'. Native semantics (form submit, keyboard, a11y) free; aur compound selector se <div dsButton> compile-time pe hi impossible. (2) Input: selector: 'input[dsInput], textarea[dsInput]' — same reasoning; wrapper mein ngModel/reactive forms integration, native validation, aur autofill sab tootenge. (3) Card: ye genuinely nayi cheez hai (koi native equivalent nahi) → element selector ds-card, content projection ke saath (<ng-content select="[dsCardHeader]">, S7). Misuse rokne ke tools: (a) compound selectors — sabse strong, compile error deta hai; (b) required inputs (input.required, S5) — bina zaroori data ke use hi na ho; (c) :not() se anti-patterns block karo ('button[dsButton]:not([disabled])' jaisa agar policy ho); (d) template type-checking (strictTemplates) galat binding pakad legi; (e) lint rules for banned patterns; (f) Storybook + docs — par ye aakhri defense hai. Principle jo bolna hai: "API ko aisa design karo ki galat use compile na ho — documentation aakhri sahara hai, pehla nahi." Ye Ch2 ka "make impossible states unrepresentable" ka component-level version hai.

S-03

Templates

Covers: Interpolation & binding syntax · Template expressions ki limits · @if/@for/@switch · Template reference variables · ng-template / ng-container / @defer

1. What is it?

Template = HTML + Angular ka binding syntax. Ye ek declarative DSL hai jo compile hoke JavaScript instructions ban jaata hai (Ch3 S5/S6). Isme char tarah ki bindings hain, teen control-flow blocks hain, aur do "invisible" elements (ng-template, ng-container).

Sabse important baat jo log bhool jaate hain: template ek expression language hai, poora JavaScript nahi. Jaan-boojh ke restricted hai.

2. Why do we need it?

Do design philosophies hain: JSX (template = JS, poori power) ya restricted DSL (template = declarative markup). Angular ne DSL chuna, aur uske teen thos faayde hain: (1) static analysis — compiler template ko samajh sakta hai, isliye AOT + type checking + tree shaking (Ch3 S5); (2) optimization — compiler jaanta hai kaunsi binding kis DOM node pe jaati hai, toh targeted update instructions bana sakta hai; (3) discipline — template mein logic likhna mushkil hai, isliye logic component mein rehti hai. Restriction ek feature hai.

3. What problem does it solve?

  • Manual DOM manipulation → declarative bindings.
  • String templating ki XSS → automatic contextual sanitization.
  • Imperative show/hide/loop → @if/@for blocks.
  • Runtime template errors → compile-time type checking (strictTemplates).

6. Architecture — control flow blocks (v17+)

// @if — *ngIf ka replacement, ab built-in (koi import nahi)
@if (user(); as u) {          // as = narrowed alias (Ch2 S1!)
  <p>{{ u.name }}</p>              // u yahan non-null hai, TS bhi jaanta hai
} @else if (loading()) {
  <app-spinner />
} @else {
  <p>No user</p>
}

// @for — track MANDATORY hai (Ch3 S6: correctness issue!)
@for (order of orders(); track order.id) {
  <app-order-row [order]="order" />
} @empty {
  <p>Koi order nahi</p>            // *ngFor mein ye tha hi nahi
}
// Contextual variables: $index, $first, $last, $even, $odd, $count

// @switch
@switch (state().status) {
  @case ('success') { <app-data [d]="state().data" /> }   // narrowed! (Ch2 S2)
  @case ('error')   { <app-error [m]="state().message" /> }
  @default          { <app-spinner /> }
}

Naye blocks sirf syntax sugar nahi hain: (1) built-in hain — CommonModule import nahi chahiye; (2) type narrowing better hai (@if (x; as y) aur @switch discriminated unions ke saath, Ch2 S2); (3) @for ka reconciliation algorithm naya aur tez hai (*ngFor IterableDiffer abstraction se guzarta tha); (4) track ab compulsory hai — Angular ne ek poori bug class ko compile error bana diya.

11. Common Mistakes

Galtiyan
  • Template mein function calls: {{ getTotal() }}, [data]="filterItems()" — har CD cycle mein chalte hain. computed() use karo.
  • [attr.disabled]="isDisabled" jab [disabled] chahiye tha — attribute ki presence hi disable karti hai, toh [attr.disabled]="false" bhi disable kar deta hai! (false string ban jaata hai.) Property use karo, ya [attr.disabled]="isDisabled() || null".
  • Template mein complex logic — nested ternaries, chained filters. Component ya computed() mein nikaalo; template padhne layak rehna chahiye.
  • *ngIf/*ngFor naye code mein — @if/@for use karo (built-in, faster, better narrowing, @empty).
  • @for mein track $index jab items reorder hote hain — Ch3 S6 wala correctness bug.
  • $any($event.target).value ka overuse — ye template ka as any hai. Better: (input)="onInput($event)" aur component mein typed handler.
  • innerHTML se user content — Angular sanitize karta hai par bypassSecurityTrustHtml se bypass karna XSS ka direct rasta.

12. Best Practices

Production rules
  • Template mein zero logic: sirf property access, signal calls, aur pipes. Baaki sab computed().
  • strictTemplates: true — template ko TS code ki tarah type-check karo (Ch3 S5).
  • Naya control flow (@if/@for/@switch); @for mein stable track.
  • ng-container se grouping — extra DOM nodes mat banao (DOM node count = CD aur memory cost).
  • Customization ke liye ng-template + ngTemplateOutlet (content projection se zyada flexible jab context pass karna ho).
  • Property binding default; attr. sirf jab native property na ho (aria-*, colspan, SVG).

13. Performance

Teen bade levers. (1) Binding count — har binding ek vars slot + har cycle ek comparison (Ch3 S6). 50 bindings × 1000 rows = 50,000 comparisons per cycle. (2) Function calls in template — memoization nahi hoti; computed() mein daalne se wo dependency-tracked aur cached ho jaata hai. (3) DOM node count — har <div> wrapper ek node hai; ng-container se bachao.

Aur @defer (Ch3 S7) template-level lazy loading deta hai — bhaari children ka JS tabhi aata hai jab trigger ho. Ye ab template ka hissa hai, build config ka nahi.

14. Edge Cases

  • @if (obs$ | async; as data) — async pipe + alias, ek hi subscription (bina as ke multiple | async = multiple subscriptions!). Signals mein ye problem hi nahi.
  • @for ka $index ek local variable hai; nested loops mein shadowing hoti hai — let idx = $index se alias banao.
  • ng-container pe structural directive lag sakti hai, par uska koi host element nahi — #ref lagane pe kya milega? (Nothing useful — ye deliberately non-element hai.)
  • Event binding mein $event ka type: native events pe Event (narrow karna padta hai), component outputs pe emitted type (typed!).
  • Key modifiers: (keydown.control.enter), (keyup.escape) — Angular pseudo-events. Custom modifiers nahi bana sakte.
  • {{ }} ke andar | bitwise OR nahi hai — wo pipe operator hai. Bitwise chahiye toh component mein.

15. Interview Questions

EASY Property binding aur attribute binding mein farak?

[prop] DOM property set karta hai (live JS state), [attr.x] HTML attribute (initial/serialized value). Zyada tar cases mein property chahiye. Attribute tab jab native property exist na kare: [attr.aria-label], [attr.colspan], SVG attributes. Classic trap: [attr.disabled]="false" element ko disable kar deta hai (attribute ki presence hi kaafi hai) — || null chahiye.

EASY Template mein function call kyun bura hai?

Kyunki wo har change detection cycle mein execute hoti hai — Angular ko nahi pata result cache ho sakta hai ya nahi. Default CD + frequent events (mousemove, scroll) ke saath ye hazaaron calls per second ban jaata hai. computed() memoized hai (dependencies na badlein toh re-evaluate nahi) aur signal-based CD mein view ko dirty bhi nahi karta.

MEDIUM ng-template aur ng-container mein farak?

ng-container: grouping ke liye "invisible element" — DOM mein kuch nahi banta, par uska content turant render hota hai. Use: <tr> ke andar wrapper chahiye bina invalid HTML ke. ng-template: ek blueprint — render nahi hota jab tak koi explicitly na kare (ngTemplateOutlet, structural directive, ya createEmbeddedView). Use: conditional content, customizable slots, render-prop pattern. Ek jagah hota hai par dikhta nahi; doosra dikhta hai par jagah nahi leta.

MEDIUM @if/@for/@switch ne *ngIf/*ngFor/*ngSwitch se kya behtar diya?

(1) Built-in — koi CommonModule import nahi (locality, kam bundle). (2) Type narrowing — @if (x; as y) aur @switch discriminated unions ke saath saaf narrow karte hain. (3) Performance — @for ka naya reconciliation algorithm IterableDiffer abstraction ke bina; benchmarks mein kaafi tez. (4) API — @empty block, @else if, mandatory track (jo ek poori bug class ko compile error bana deta hai, Ch3 S6). (5) Better error messages aur readable syntax (structural directive ka <ng-template> desugaring chhupa hua tha).

HARD Template expression language deliberately restricted kyun hai? Teen concrete faayde batao.

(1) Static analyzability — compiler har expression ka AST bana sakta hai, isliye AOT compilation, type checking (Type Check Blocks, Ch3 S5), aur "kaunsi binding kaunse DOM node ko chhuti hai" ka precise mapping possible hai. Agar template mein arbitrary JS hota (assignments, closures, global access) toh ye analysis toot jaati. (2) Optimization — restricted expressions ka matlab hai compiler jaanta hai ki koi side effect nahi hoga (mostly), isliye wo dirty-checking ko instructions mein bake kar sakta hai (textInterpolate khud compare karta hai). Arbitrary JS mein ye guarantee nahi. (3) Discipline/architecture — template mein logic likhna mushkil banane se logic component class mein rehti hai, jahan wo testable aur reusable hai. Ye "pit of success" design hai. (4 bonus) Security — global scope (window, eval) accessible nahi, aur interpolation contextually sanitized hai. Trade-off imaandaari se: JSX zyada flexible hai aur "sirf JS hai" ka mental model simple hai; Angular ne flexibility becha kar compile-time guarantees kharidi hain. React ab compiler la raha hai — yaani wo bhi kuch analysis wapas chahta hai.

SCENARIO Tumhe ek reusable DataTable component banana hai jisme consumer har column ka cell rendering customize kar sake (badge, currency, link, custom). Template APIs se kaise design karoge?

Do valid approaches hain, dono jaano. Approach A — ng-template + ngTemplateOutlet (render props): consumer templates deta hai, table unhe context ke saath render karta hai. <app-table [rows]="rows"><ng-template appCell="status" let-row><app-badge [status]="row.status" /></ng-template></app-table>. Table ke andar contentChildren(CellTemplateDirective) (S7) se saare templates collect karo, column name se map banao, aur [ngTemplateOutlet]="tpl" [ngTemplateOutletContext]="{$implicit: row, index: i}" se render. Pros: maximum flexibility, context pass ho sakta hai, lazy (jo column dikh raha hai wahi render). Cons: API thoda ceremony wali. Approach B — content projection with slots (S7): simpler, par per-row context pass nahi kar sakte — isliye tables ke liye A hi chahiye. Type safety: Ch2 S6 ka ColumnDef<T, K extends keyof T> pattern + directive pe generic input; TS 4.7+ ka ngTemplateContextGuard static method se let-row ko typed banao — ye wo detail hai jo interviewer ko impress karti hai (static ngTemplateContextGuard(dir, ctx): ctx is CellContext<T>). Performance: har cell template ek embedded view = ek LView (Ch3 S6); 20,000 cells pe ye bhaari hai — isliye default rendering (plain interpolation) rakho aur template sirf custom columns ke liye. Fallback: agar consumer ne template nahi diya toh default cell. Ye answer template APIs + generics + performance + type safety sab jodta hai.

S-04

Styles & View Encapsulation

Covers: Component styles · Emulated / None / ShadowDom · _ngcontent attributes · :host, :host-context · ::ng-deep · CSS custom properties

1. What is it?

Component styles = CSS jo sirf us component ke template pe lagti hai. "View encapsulation" wo mechanism hai jisse Angular ye scoping achieve karta hai. Teen modes hain, aur teeno browser mein bilkul alag kaam karte hain:

  • Emulated (default) — Angular CSS aur DOM dono ko rewrite karke scoping simulate karta hai. Koi browser feature use nahi hota.
  • None — koi scoping nahi; styles global ho jaati hain.
  • ShadowDom — asli browser Shadow DOM; native isolation.

2. Why do we need it?

CSS ka original design global hai. Ek .card { padding: 8px } rule poore app ke har .card ko chhoo leti hai. 200-component app mein iska matlab: har naya CSS rule ek potential regression hai, koi bhi class rename dar ke saath hota hai, aur "ye style kahan se aa rahi hai?" debugging ka daily ritual ban jaata hai. Encapsulation ye kehti hai: component ki CSS component ke bahar nahi jaayegi.

3. What problem does it solve?

  • Global namespace collisions → scoped rules, BEM jaisi naming discipline ki zaroorat khatam.
  • Refactoring fear → component delete karo, uski CSS bhi gayi.
  • Specificity wars → scoping ke baad ladai kam.
  • Design system boundaries → theming custom properties se controlled, accidental override se nahi.

10. Code Explanation — Line by Line

  • L1: Bare ::ng-deep ka compiled roop hai sirf .mat-mdc-dialog-surface { ... } — koi scoping attribute nahi. Ye rule <head> mein inject ho jaati hai aur poore app pe lagti hai. Aur component destroy hone pe bhi rule wahin rehti hai (styles remove nahi hoti). Yaani ye chhupa hua global CSS hai — global se bhi bura, kyunki dhundhne mein nahi milti.
  • L2–L3: :host prefix se compiled selector [_nghost-abc] .mat-... banta hai — ab scope host ke andar contained hai. Ye ::ng-deep use karne ka ekmatra acceptable tareeka hai.
  • L4: Asli trap: overlays (dialogs, tooltips, menus) CDK ke through document.body mein render hote hain — DOM tree mein tumhare host ke andar nahi. Toh :host ::ng-deep unhe kabhi match nahi karega. Log ghanton yahan atakte hain.

"Global CSS" bura nahi hai — chhupa hua global CSS bura hai. ::ng-deep officially deprecated hai; Angular team ka long-term answer CSS custom properties aur @scope jaisi native CSS features hain.

11. Common Mistakes

Galtiyan
  • :host { display: block } bhool jaana — custom elements default display: inline hote hain; width/height/margin ignore honge. Ye #1 layout bug hai.
  • Bare ::ng-deep — chhupa hua global CSS. Kam se kam :host ::ng-deep.
  • encapsulation: None "styles kaam nahi kar rahi" ka fix samajhna — ab us component ki saari CSS global hai, aur problem 10 guna badi.
  • ShadowDom mode casually on karna — global styles (resets, fonts, theming) andar nahi aayengi; CDK overlays aur third-party libs tootengi.
  • Child ke internals ko parent se style karna (app-card .title) — ye emulated mein kaam nahi karega (child ke elements pe parent ka _ngcontent nahi hai). CSS variables se contract banao.
  • Har component mein poora design system dobara define karna — variables global (:root) mein, component sirf consume kare.

12. Best Practices

Production rules
  • Emulated default rakho. None sirf genuinely global cheezon ke liye (aur unhe styles.scss mein rakho, component mein nahi).
  • Har component mein :host { display: ... } explicitly likho.
  • Theming = CSS custom properties. Component apne variables :host pe defaults ke saath declare kare — ye uska styling API hai.
  • ::ng-deep ko lint rule se ban karo; exception ke liye :host ::ng-deep + comment + ticket.
  • Design tokens :root mein; dark mode :host-context ya root class se.
  • Component styles chhote rakho (anyComponentStyle budget, Ch3 S7) — 4KB se bada style file aksar design system ki jagah component mein tokens duplicate kar raha hota hai.

13. Performance

Bundle: component styles uske JS chunk ka hissa hain (compiled string) — lazy route ki CSS bhi lazy aati hai. Ye webpack-era ke "poori CSS ek file mein" se bada sudhar hai.

Runtime: pehli baar component render hone pe ek <style> tag inject hota hai — per component type, per instance nahi. 1000 cards = 1 style tag. Emulated mode ka attribute overhead: har element pe ek extra attribute (DOM size thoda badhta hai, ignorable).

Selector cost: rewritten selectors (.title[_ngcontent-abc]) browser ke liye normal attribute selectors hain — fast. ::ng-deep wale unscoped rules poore document pe match hote hain — wahi mehenge hain, aur sabse zyada tab jab wo descendant selectors hon.

14. Edge Cases

  • Projected content (<ng-content>, S7) pe parent ka _ngcontent hota hai, child ka nahi — isliye child ki CSS projected content pe nahi lagti. Ye by design hai (content parent ka hai). ::ng-deep ya :host se target karna padta hai.
  • :host-context() ancestors mein dhundhta hai — document tak. Powerful, par thoda mehenga (browser ko upar chadhna padta hai).
  • ShadowDom mode: ::slotted() se projected content style hota hai; global CSS custom properties + inherited properties hi andar aati hain.
  • Dynamically added classes (Renderer2 se, S8) pe encapsulation attribute nahi hoti — CSS match nahi karegi. Angular ke bindings use karo.
  • styles array mein CSS string vs styleUrl — build pe dono same treatment; inline styles chhote components mein readable.
  • Angular v20+ mein @scope native CSS ka support aa raha hai — encapsulation ka future browser-level hoga.

15. Interview Questions

EASY Component ki CSS doosre components pe kyun nahi lagti?

View encapsulation ki wajah se. Default (Emulated) mode mein Angular har component ke DOM elements pe ek unique attribute (_ngcontent-abc) lagata hai aur CSS rules ko rewrite karke us attribute selector se scope kar deta hai (.title[_ngcontent-abc]). Koi Shadow DOM nahi — sirf attribute selectors se simulate kiya gaya scoping.

EASY :host kya hai aur uske bina kya problem hoti hai?

:host component ke apne host element ko target karta hai (<app-card> tag khud). Bina uske: custom elements ka default display: inline hota hai, toh width, height, margin, padding sab weirdly behave karte hain. Har component mein :host { display: block } (ya flex/grid) explicitly likhna chahiye.

MEDIUM Encapsulation ke teen modes ka farak, aur kab kaunsa?

Emulated (default): attribute rewriting se scoping; component ki CSS bahar nahi jaati, par global CSS andar aa sakti hai (ek-tarfa). 95% cases. None: koi scoping nahi, styles global. Sirf genuinely global styles ke liye — par unhe styles.scss mein rakhna behtar hai. ShadowDom: native browser Shadow DOM, dono taraf se isolation — global CSS andar bhi nahi aati (sirf inherited props aur CSS variables). Use tab jab Web Components ship kar rahe ho ya strict isolation chahiye; par CDK overlays aur global theming toot sakti hai.

MEDIUM ::ng-deep kya karta hai aur wo deprecated kyun hai?

Wo encapsulation todta hai: uske aage ka selector bina scoping attribute ke emit hota hai, yaani rule global ban jaati hai. Aur wo <head> mein inject hoke rehti hai, component destroy hone pe bhi. Isliye ye "chhupa hua global CSS" hai — global se bhi khatarnaak, kyunki koi styles.scss mein dhundhne nahi jaayega. Deprecated isliye ki iska koi standard equivalent nahi tha aur ab CSS custom properties + native @scope better answers hain. Zaroori ho toh hamesha :host ::ng-deep (contained) + comment.

HARD Parent apne child component ke andar ki class ko style kyun nahi kar sakta, aur sahi solution kya hai?

Kyunki emulated encapsulation mein child ke template ke elements pe child ka _ngcontent-child attribute hota hai, parent ka nahi. Parent ki rule app-card .title compile hoke app-card[_ngcontent-parent] .title[_ngcontent-parent] banti hai — jo kabhi match nahi karegi. Ye bug nahi, encapsulation ka poora point hai: child ke internals uska private implementation hain, aur parent unpe depend karega toh child ka koi bhi refactor parent ko tod dega (yahi CSS ki purani duniya ka dard tha). Sahi solutions, priority order mein: (1) CSS custom properties — child :host pe apne variables defaults ke saath declare kare (--card-title-color); ye uska intentional styling contract hai, bilkul inputs ki tarah. Custom properties inherited hoti hain, isliye encapsulation cross karti hain — by design. (2) Input-driven classes — variant = input<'primary'|'danger'>() aur child apne andar class lagaye (S8 ka host binding). (3) Content projection — jo parent style karna chahta hai, wo parent hi provide kare (<ng-content>, S7); projected content pe parent ka _ngcontent hota hai. (4) Last resort: :host ::ng-deep with a comment explaining why the API is insufficient — aur ideally library pe issue file karo. Framing jo bolni hai: "CSS bhi ek API hai; child ki internals pe style likhna wahi galti hai jo uski private property ko chhoona."

SCENARIO Design system banana hai jisme light/dark theme, brand colors per-tenant configurable, aur 40 components hon. Styling architecture kya hogi?

Layered token architecture. (1) Primitive tokens (:root, global styles.scss): raw values — --ds-color-pink-500: #ff2d6f, spacing scale, font sizes. Ye kabhi directly component mein use nahi hote. (2) Semantic tokens: --ds-surface-bg, --ds-text-primary, --ds-accent — primitives ko point karte hain. Theme switch inhi ko remap karta hai: .theme-dark { --ds-surface-bg: #0c0c11 }. (3) Component tokens: har component :host pe apne variables semantic tokens ke defaults ke saath declare kare — --card-bg: var(--ds-surface-bg). Ye component ka public styling API hai. (4) Theming switch: root pe ek class (document.documentElement.classList) — :host-context(.theme-dark) se bhi ho sakta hai par root-class + variables simpler aur tez hai. Runtime pe zero JS: sirf ek class toggle, CSS baaki karta hai. (5) Per-tenant branding: tenant config se :root pe custom properties inject karo (bootstrap ke initializer se, Ch3 S4 — ek <style> tag ya style.setProperty). Naya tenant = naye values, zero rebuild. (6) Encapsulation: Emulated everywhere; ::ng-deep lint-banned; components sirf apne variables consume karte hain. (7) SSR: theme class server pe hi set karo (cookie se), warna FOUC. (8) Governance: budget per component style (4KB), aur ek lint rule jo raw hex values ko component styles mein block kare — sab tokens se aaye. Key insight: CSS custom properties encapsulation ko intentionally cross karti hain — wahi wo API surface hai jo tumhe design karni hai, warna log ::ng-deep se tod ke ghusenge.

Inputs

Covers: input() / input.required() · transform · alias · model() two-way · @Input legacy · setter pattern · Route param binding

1. What is it?

Input = component ka data contract — parent se child mein data aane ka official rasta. Modern Angular mein input ek signal hai: order = input.required<Order>(). Parent binding se value set karta hai ([order]="o"), child use signal ki tarah padhta hai (this.order()).

Ye purane @Input() decorator se sirf syntax ka farak nahi hai — architectural farak hai, aur wahi is section ka core hai.

2. Why do we need it?

Bina inputs ke parent-child communication ke liye shared mutable state ya direct instance access chahiye — dono coupling aur bugs laate hain. Input ek explicit, typed, one-directional channel deta hai: parent bolta hai, child sunta hai. Aur ab jab wo signal hai, toh child ko "value badli" ka notification bhi automatically milta hai — ngOnChanges ki zaroorat khatam.

3. What problem does it solve?

  • Untyped props → compile-time checked bindings (strictTemplates).
  • Missing required data → input.required() — bina bind kiye compile error.
  • Manual change detection → input signal badla toh dependent computed/view apne aap update.
  • Coercion boilerplate → transform function inline.

// PURANA PATTERN: input pe react karne ke liye setter
export class OldChart {

  @Input()
  set data(value: Point[]) {              // L1

  data = input([], {                                         // L4

  constructor() {
    afterRenderEffect(() => this.redraw(this.processed()));       // L6: DOM side effect
  }
}

10. Code Explanation — Line by Line

  • L1: Setter input — pehle ye ekmatra tareeka tha "input badla toh kuch karo" ke liye. Problem: setter ek hi jagah mein teen kaam karta hai.
  • L2: Derivation setter mein — agar normalize mehenga hai aur input do baar same value ke saath set ho, toh dobara chalega (Angular Object.is check karta hai, toh same reference pe nahi — par same content, naya reference pe chalega).
  • L3: DOM work setter mein — ye CD cycle ke beech chal raha hai. Yahi ExpressionChangedAfterItHasBeenChecked (Ch3 S8) ka classic source hai. Aur SSR pe crash, kyunki server pe DOM nahi.
  • L4: transform — sirf shape conversion, pure function. Input aate hi ek baar chalta hai. (Note: transform mein bhaari kaam mat karo; wo har binding update pe chalega.)
  • L5: computed — pure derivation, memoized. data() na badle toh normalize dobara nahi chalega. Aur ye lazy hai: koi na padhe toh evaluate hi nahi hota.
  • L6: afterRenderEffect — DOM side effect, render ke baad, browser-only (SSR-safe). Cleanup automatic (component destroy pe).
  • Lesson: Setter ek tool tha jisse teen alag cheezein karwayi jaati thi. Signals ne har concern ko apna tool diya: transform (conversion), computed (derivation), effect/afterRenderEffect (side effect). Interview mein ye "ek API ne teen responsibilities collapse kar di thi" wali framing bolna maturity dikhata hai.

  • @Input() data!: X — non-null assertion ka matlab hai "compiler chup raho"; parent bhool gaya toh runtime pe undefined. input.required() use karo.
  • Setter input mein side effects (HTTP, DOM) — CD ke beech chalta hai, SSR pe todta hai. computed/effect mein todo.
  • Template mein {{ order }} likhna jab order ek signal hai — {{ order() }}. (Compiler strictTemplates ke saath pakad lega — signal object print ho jaayega warna.)
  • transform mein bhaari computation — wo har input update pe chalta hai, aur memoized nahi hai. Derivation computed mein.
  • Bahut saare inputs (10+) — component do kaam kar raha hai. Object input (config) bhi anti-pattern ho sakta hai (poora object badla toh sab kuch dirty).

12. Best Practices

Production rules
  • Naya code: hamesha input()/input.required(). @Input sirf legacy files mein.
  • Zaroori data input.required() — API ko aisa banao ki galat use compile na ho (Ch2 ka theme).
  • Derived state computed(), kabhi getter ya ngOnChanges se manual sync nahi.
  • Boolean attributes pe transform: booleanAttribute — <app-x disabled> kaam karega ('' → true).
  • Inputs immutable treat karo — child kabhi mutate na kare (signal inputs structurally rokte hain).
  • Router se: withComponentInputBinding() (Ch3 S4) → route params seedhe inputs mein bind hote hain.

13. Performance

Har input ek binding hai — ek vars slot + har parent CD cycle mein ek Object.is comparison (Ch3 S6). Comparison sasta hai; asli cost tab hai jab comparison fail hota hai (naya reference) aur child ka subtree dirty ho jaata hai.

Isliye: (1) parent template mein [config]="{ a: 1 }" mat likho — har CD cycle mein naya object literal = har baar child dirty. computed() ya field mein rakho. (2) Array/object inputs ko stable rakho — filter/map template mein mat karo. (3) Signal inputs ke saath granularity better hai: computed(() => this.order().amount) sirf tab re-run hoga jab order badle, aur uska consumer view hi dirty hoga.

14. Edge Cases

  • input.required() ko constructor/field-init mein padhna → runtime error ("Input is required but no value is available yet"). Inputs ngOnInit ke time set hote hain; computed/template mein padho.
  • transform ka type: input ka accept type transform ke parameter se aata hai, aur read type return se. InputSignalWithTransform<ReadT, WriteT>.
  • Static attribute vs binding: disabled (attribute) se '' milta hai — booleanAttribute use karo. [disabled]="false" (binding) se actual false.
  • Dynamic components: componentRef.setInput('order', o) — signal inputs ke saath bhi kaam karta hai.
  • ngOnChanges signal inputs ke saath bhi fire hota hai (agar implement karo) — par mixing confusing hai; ek hi paradigm mein raho.
  • Host directives (S8) apne inputs expose kar sakti hain: hostDirectives: [{directive: Tooltip, inputs: ['text: tooltipText']}].

Ye Ch2 ka structural typing + API design ka sundar use hai: same object, do alag views — Angular ke liye writable, tumhare liye readable. Aur model() ke case mein ModelSignal<T> set/update expose karta hai plus ek implicit output emit karta hai — yaani two-way binding bhi isi mechanism ka extension hai (S6).

(3) Plain @Input → input(): mechanical (ng update ki migration ye kaafi had tak karti hai). Template mein () lagao. Required wale input.required(). (7) Ek PR per concern, tests har step pe green. (8) Aakhri mein: OnPush → (agar app zoneless ja rahi hai) verify karo ki har update path signal se guzarta hai (Ch3 S8 ka migration). S-06

Outputs

Covers: output() · EventEmitter internals · $event · outputFromObservable / outputToObservable · Two-way binding & model() · Event bubbling ka mith

1. What is it?

Output = child se parent ko event bhejne ka channel. cardClicked = output<string>() declare karo, this.cardClicked.emit(id) se fire karo, parent (cardClicked)="handle($event)" se sunta hai.

Ek galatfehmi pehle hi door kar lo: Angular outputs DOM events nahi hain. Wo bubble nahi karte, stopPropagation nahi hota, Event object nahi hai. Wo sirf ek subscription mechanism hai — internally ek Subject (ya uska simplified version).

2. Why do we need it?

Unidirectional data flow ka doosra aadha hissa. Input parent→child data laata hai; output child→parent intent bhejta hai. Child bolta hai "user ne delete click kiya" — delete karna parent ka kaam hai. Ye separation hi dumb components ko reusable banati hai: same <app-order-row> ek jagah delete kare, doosri jagah archive kare — component ko farak nahi padta.

3. What problem does it solve?

  • Child ka parent pe direct dependency → child ko parent ka pata bhi nahi.
  • Callback props ki untyped mess → typed events (output<string>() → $event: string).
  • Two-way state sync → model() se ek line mein.
  • Subscription leaks → output() ka teardown Angular manage karta hai.

4. Internal Working — EventEmitter ka sach

// Angular ka EventEmitter (legacy):
class EventEmitter<T> extends Subject<T> {     // ← RxJS Subject hai!
  emit(value: T) { super.next(value); }
}
// Iska matlab: EventEmitter pe .subscribe(), .pipe(), .next() sab chalta hai.
// AUR YAHI PROBLEM HAI — log use stream ki tarah use karne lagte the:
//   this.myOutput.pipe(debounceTime(300)).subscribe(...)   // ye output ka kaam nahi!
//   this.someService.stream$ = this.myOutput;              // leak ka rasta
  output() ne kya badla:

    output<T>()  -->  OutputEmitterRef<T>
                       .emit(value)
                       .subscribe(fn)   [internal use — DestroyRef se auto-cleanup]

    ► NAHI hai: .pipe(), .next(), .complete(), Observable ka kuch bhi
    ► RxJS pe dependency KHATAM (output() import karne se RxJS nahi aata)
    ► Subscription component destroy pe apne aap cleanup (Ch1 S8 leak class gayi)

  Design philosophy: "Output ek EVENT CHANNEL hai, ek STREAM nahi.
                     Stream chahiye toh outputFromObservable() se explicitly banao."

$event ka type output ke generic se aata hai: output<Order>() → template mein $event: Order, aur strictTemplates handler ka signature check karta hai (Ch3 S5).

  child: this.deleted.emit(order.id)

  ► Aur isliye: emit ke andar se emit (loop) infinite recursion de sakta hai.

10. Code Explanation — Line by Line

  • L1: Har intermediate component ko event forward karna padta hai. Do levels tak theek hai; teen se zyada = design smell. Har forwarding line ek jagah hai jahan koi rename/refactor tootega.
  • L2: Shared service + signal — C directly store update karta hai, A directly padhta hai. Critical detail: service ko A ke providers mein rakho (providedIn: 'root' nahi) — tab wo us subtree ka scoped state hai, global nahi (Ch3 S9). Do parallel A instances ka apna-apna store hoga. Ye Angular ka React-Context equivalent hai, par zyada powerful.
  • L3: Native CustomEvent with bubbles: true — ye sach mein DOM tree mein upar jaata hai. Web Components ka standard pattern.
  • L4: Parent kisi bhi ancestor level pe sun sakta hai ($event.detail se payload). Trade-off: type safety kamzor (CustomEvent ka detail any hai jab tak tum declare global se HTMLElementEventMap augment na karo — Ch2 S9!), aur ye Angular ke declarative model se bahar hai. Micro-frontends ya Angular Elements mein justified.
  • Rule of thumb: 1-2 levels = outputs. 3+ levels ya siblings = scoped service + signals. Cross-app = router/global store.

11. Common Mistakes

Galtiyan
  • EventEmitter ko stream ki tarah use karna — output.pipe(...), ya use service mein pass karna. Wo ek event channel hai. RxJS chahiye toh outputFromObservable().
  • Output ko subscribe karke bhool jaana (dynamic components mein) — output() ke saath Angular cleanup karta hai, par manual EventEmitter.subscribe mein tum zimmedaar ho (Ch1 S8).
  • Output ka naam onXxx rakhna (onDelete) — tab template mein (onDelete) padhta hai, jo on-on jaisa lagta hai. Naam past-tense event ho: deleted, saved, closed.
  • Child ke andar hi action complete karna aur output "notification" ke roop mein bhejna — child ne delete kar diya, phir parent ko bataya. Ab child ko service inject karni padi, wo dumb nahi raha. Intent bhejo, action parent kare.
  • Output emit ke andar se dobara emit — sync hai, infinite loop possible.
  • Two-way binding ([(x)]) ka overuse — do-tarfa data flow debug karna mushkil hai. Forms/simple values ke liye theek, business state ke liye nahi.

12. Best Practices

Production rules
  • Naya code: output(). EventEmitter sirf legacy.
  • Naming: past tense ya noun-event (saved, selectionChanged), on prefix nahi.
  • Child intent bheje, parent action kare — yahi dumb component ka contract (S9).
  • Payload minimal rakho: output<string>() (id) aksar output<Order>() se behtar; parent ke paas already data hai.
  • Debouncing/throttling child ke andar (outputFromObservable) — parent ko clean events milen.
  • 3+ levels ki forwarding dikhe toh rukho — scoped service + signals pe jao.

13. Performance

Output emit khud sasta hai (ek loop over listeners, sync). Asli cost uske baad aati hai: handler state badalta hai → CD trigger. High-frequency outputs (mousemove forwarding, scroll position) parent ki CD ko har frame trigger kar sakte hain.

Fixes: (1) child ke andar hi throttle/debounce karo (outputFromObservable + auditTime); (2) zone apps mein runOutsideAngular se listener lagao aur sirf zaroori emits zone.run() mein (Ch3 S8); (3) zoneless mein signal write hi CD schedule karta hai, toh sirf emit ke handler mein signal write ko throttle karo.

output() ka ek bonus: RxJS import nahi aata, toh bundle chhota (EventEmitter Subject extend karta tha).

14. Edge Cases

  • model() ka output naam automatically xChange hota hai — [(value)] ke liye value + valueChange chahiye. Custom two-way bhi bana sakte ho: value = input() + valueChange = output().
  • model.required() bhi hai — two-way binding compulsory.
  • Output ka handler $event ke bina bhi ho sakta hai: (closed)="close()".
  • Native DOM events aur outputs ka naam clash: agar component pe click naam ka output ho, wo native click ko shadow kar dega. Isliye prefix/unique names.
  • Dynamic components: componentRef.instance.saved.subscribe(...) — output() ke saath outputToObservable() ya direct .subscribe() (jo OutputRef deta hai — manually unsubscribe karo).
  • Host directives (S8) outputs bhi expose kar sakti hain: hostDirectives: [{directive: D, outputs: ['x: aliasX']}].

HARD Output emit synchronous hai — iske do practical consequences batao.

Consequences: (1) Re-entrancy / infinite loops: parent ka handler child ke input ko badal de, jisse child dobara emit kare — sync recursion, stack overflow ya ExpressionChangedAfterItHasBeenChecked (Ch3 S8). Two-way bindings mein ye classic hai; isliye model() mein Angular value comparison karke redundant emits rokta hai. (2) CD timing: emit CD cycle ke dauran ho sakta hai (jaise @for ke andar ka event handler), aur handler ne state badli toh wo state usi cycle mein already-checked components ke liye stale ho sakti hai — phir wahi dev-mode error. (3) Error propagation: handler mein exception child ke emit() call stack mein propagate karti hai — child ka code beech mein ruk sakta hai (emit ke baad ki lines nahi chalengi). Isliye emit ko method ki aakhri line rakhna ek acchi aadat hai. Design insight: sync emit deliberate hai — predictable ordering deta hai (async hota toh "kab tak state update hui?" ka jawab hi nahi hota), par re-entrancy ki zimmedari developer pe daalta hai.

Zero prop drilling, zero output forwarding, aur signals ki wajah se sirf do nodes ka view update hoga (purana selected + naya) — poora tree nahi. (2) Recursive component + skipSelf agar parent context bhi chahiye (Ch3 S9 ka pattern). (3) Native CustomEvent ek valid alternative hai (bubbles: true), par type safety weak aur Angular ke declarative model se bahar — micro-frontend boundaries pe justified. (4) Kya nahi karna: global store (providedIn: 'root') — do trees ek doosre ka selection dekhne lagenge; ya viewChild se parent ka reference lena (child ko parent pe depend karwana = reusability khatam). Framing: "outputs 1-2 levels ke liye hain; usse gehra ho toh problem communication ka nahi, state ownership ka hai — state ko us scope mein rakho jahan sab consumers use dekh sakein."

ViewChild & ContentChild

Covers: viewChild() / contentChild() signal queries · static vs dynamic · View vs Content ka farak · ng-content projection · Timing · read option

1. What is it?

Queries = template ke andar se element ya component instances ka reference nikalne ka mechanism. Do parivaar hain, aur unka farak samajhna hi is section ka 60% hai:

  • View queries (viewChild, viewChildren) — mere apne template mein jo hai.
  • Content queries (contentChild, contentChildren) — jo parent ne mere andar project kiya (<ng-content> ke through).
  <app-card>                          <-- card ka HOST
    <h1>Title</h1>                     <-- PARENT ne likha = CONTENT
  </app-card>                              (card ke liye contentChild)

  card.component.html:
    <div class="wrapper">              <-- card ne khud likha = VIEW
      <ng-content />                        (card ke liye viewChild)
    </div>

  ► "Kisne likha?" — yahi sawaal hai. Maine likha = view. Parent ne = content.

2. Why do we need it?

Declarative bindings 95% cases cover karte hain. Baaki 5% mein tumhe imperative access chahiye: input pe focus karo, canvas pe draw karo, scroll position set karo, ek third-party lib ko DOM node do, ya projected children ko coordinate karo (tabs, accordion). Queries wahi escape hatch hain — controlled escape hatch, kyunki Angular tumhe reference lifecycle ke saath deta hai.

3. What problem does it solve?

  • document.querySelector → scoped, typed, lifecycle-aware references (aur SSR pe crash nahi).
  • Parent-child coordination → container apne projected children ko dhundh sakta hai (Tabs → Tab).
  • Timing ki uncertainty → signal queries se "abhi available hai ya nahi" reactive ho gaya.

4. Internal Working — signal queries (v17.2+)

// Modern (signal-based)
searchInput = viewChild<ElementRef<HTMLInputElement>>('searchBox');
// type: Signal<ElementRef<HTMLInputElement> | undefined>   ← undefined possible!

canvas = viewChild.required<ElementRef<HTMLCanvasElement>>('cv');
// type: Signal<ElementRef<HTMLCanvasElement>>   ← required, par abhi bhi timing hai

rows = viewChildren(RowComponent);
// type: Signal<readonly RowComponent[]>

tabs = contentChildren(TabComponent);
// projected tabs ka signal — @for se add/remove hone pe apne aap update!

// LEGACY (decorator)
@ViewChild('searchBox') input!: ElementRef;              // ngAfterViewInit tak undefined
@ViewChild('searchBox', { static: true }) input!: ElementRef;  // ngOnInit mein mil jaayega
@ViewChildren(RowComponent) rows!: QueryList<RowComponent>;   // .changes Observable
  KYUN SIGNAL QUERIES BEHTAR HAIN:

  Decorator queries:                    Signal queries:
  ───────────────────────               ─────────────────────────
  Timing: ngAfterViewInit tak wait      Signal — kabhi bhi padho, reactive
  static flag ka confusion              Koi static flag nahi
  QueryList (custom class, .changes)    Plain readonly array signal
  ! assertion (Ch2 S1 ka zeher)         .required() ya `| undefined`
  Manual subscribe to .changes          computed()/effect() se auto-track

  ► static: true ka matlab tha: "ye element @if/@for ke andar NAHI hai,
    isliye creation ke waqt hi mil jaayega". Signal queries mein ye
    distinction gayab — signal khud batata hai kab value aayi.

5. Execution Flow — query kab resolve hoti hai

  Component creation (Ch3 S6 ka creation pass)
      |
      v  Template ke DOM nodes bane, child components instantiate hue
      |
      v  Angular ne TView ki QUERY definitions ke against nodes match kiye
      |     (query bhi compile time pe pata hai! decls/vars ki tarah)
      v
  View queries resolve --> viewChild signal ki value set
      |
      v  ngAfterViewInit (legacy hook yahan chalta tha)
      |
  ► @if(false) ke andar ka element: query signal = undefined
  ► @if turn ho gaya true --> element bana --> query signal UPDATE ho gaya
     --> uske consumers (computed/effect) chal gaye. Reactive!

  CONTENT queries pehle resolve hoti hain:
      content projection parent ke creation ka hissa hai
      --> ngAfterContentInit --> phir view --> ngAfterViewInit

6. Architecture — content projection (ng-content)

// card.component.html — multi-slot projection
<div class="card">
  <header><ng-content select="[cardHeader]" /></header>     // L1: named slot
  <div class="body"><ng-content /></div>                       // L2: default slot
  <footer><ng-content select="app-card-actions" /></footer>   // L3: by selector
</div>

// Usage
<app-card>
  <h2 cardHeader>Order #123</h2>          // header slot mein
  <p>Amount: 499</p>                      // default slot mein
  <app-card-actions>...</app-card-actions>
</app-card>

Do critical facts projection ke baare mein: (1) Projected content parent ke context mein evaluate hota hai — uski bindings parent ki properties padhti hain, aur uspe parent ka _ngcontent attribute hota hai (S4: isliye child ki CSS uspe nahi lagti). (2) Projected content hamesha create hota hai, chahe child use render kare ya na kare — <ng-content> ko @if ke andar rakhna kaam nahi karta jaise tum socho. Lazy chahiye toh <ng-template> + ngTemplateOutlet (S3).

// tabs.component.ts — container, projected tabs ko coordinate karta hai
@Component({
  selector: 'app-tabs',
  imports: [NgTemplateOutlet],
  template: `
    <div class="tab-bar">
      @for (tab of tabs(); track tab.label()) {
        <button [class.active]="tab === active()" (click)="select(tab)">
          {{ tab.label() }}
        </button>
      }
    </div>
    <div class="panel">
      <ng-container [ngTemplateOutlet]="active()?.content() ?? null" />
    </div>
    <div hidden><ng-content /></div>   <!-- tabs ko project hone do -->
  `,
})
export class TabsComponent {
  protected tabs = contentChildren(TabComponent);          // signal array!
  private selected = signal<TabComponent | null>(null);
  protected active = computed(() => this.selected() ?? this.tabs()[0]);
  protected select(t: TabComponent) { this.selected.set(t); }
}

10. Code Explanation — Line by Line

  • L1: Query ka locator ek component class hai → default read wahi component instance hai. this.chartCmp().redraw() — child ki public method call. (Sochlo: kya ye input/output se better solve hota? Aksar haan.)
  • L2: Same element, alag "read". read batata hai us node se kya nikalna hai: ElementRef, ViewContainerRef, TemplateRef, ya koi bhi directive/token jo us element ke injector mein ho (Ch3 S9!). Yahi wo jagah hai jahan ElementInjector ka gyaan kaam aata hai.
  • L3: ViewContainerRef — dynamic components ka anchor. container().createComponent(SomeComp).
  • L4: @if ke andar ka element — signal undefined ho sakta hai. Legacy mein yahi static: false tha aur ngAfterViewInit mein bhi undefined milta agar condition false ho, phir ngAfterViewChecked mein poll karna padta.
  • L5: Yahi signal queries ka superpower hai: query ek signal hai, toh effect/computed uspe react kar sakte hain. Element aaya/gaya — automatic notification, koi QueryList.changes.subscribe() nahi, koi manual unsubscribe nahi. Legacy queries ke saath ye pattern ngAfterViewChecked + manual diffing tha.

11. Common Mistakes

Galtiyan
  • Query ko constructor mein padhna — abhi template create hi nahi hua. Signal undefined dega (ya .required() throw karega). afterNextRender/effect use karo.
  • viewChild se child ki state mutate karna — this.childCmp().someProperty = 5. Ye input/output contract ko bypass karta hai; child ka koi bhi refactor tootega. Queries sirf imperative APIs (focus, scroll, play) ke liye.
  • View vs content ka confusion: parent ne jo project kiya wo viewChild se nahi milega — contentChild chahiye. Ye #1 "query undefined aa rahi hai" ka cause hai.
  • <ng-content> ko @if ke andar rakh ke ye maan lena ki content lazy hai — content hamesha banta hai (parent ke context mein). Lazy chahiye toh ng-template.
  • Ek se zyada <ng-content> bina select ke — sirf pehla content paayega, baaki khaali.
  • document.querySelector use karna — SSR pe crash, aur component destroy hone ke baad stale reference (Ch1 S8 leak).
  • Legacy @ViewChild(..., {static: true}) ko "faster" samajhna — ye sirf timing flag hai, aur wo bhi tab kaam karta hai jab element @if/@for ke andar na ho.

12. Best Practices

Production rules
  • Naya code: viewChild()/contentChildren() signal queries. Decorator queries legacy.
  • Queries last resort hain — pehle poochho: input/output/binding se ho sakta hai?
  • DOM work hamesha afterNextRender()/afterRenderEffect() mein (SSR-safe, Ch3 S8).
  • Container components (Tabs, Accordion, Stepper) mein contentChildren + signals = clean declarative API.
  • read: ElementRef | ViewContainerRef | SomeDirective — jo chahiye wahi maango.
  • Child ki method call karni ho toh use uska public API banao (documented, tested), na ki internal property poke karo.

13. Performance

Queries compile-time pe TView mein register hoti hain (Ch3 S6) — matching creation ke dauran hota hai, aur updates tabhi jab matching nodes add/remove hon. Cost chhoti hai, par contentChildren/viewChildren ke saath har structural change (@for mein item add) query result rebuild karta hai — bade lists pe viewChildren avoid karo.

Projection ka apna cost: projected content parent ke LView mein banta hai aur child ke template mein place hota hai — do views ka coordination. Ek <ng-content> free hai; 5 named slots aur nested projection layers add up karte hain. Aur yaad rakho: projected content hamesha create hota hai — bhaari content ko ng-template/@defer ke peeche rakho.

14. Edge Cases

  • viewChild ke locator: template ref string ('box'), component/directive class, ya TemplateRef. Strings ke saath typos runtime pe silently undefined.
  • @ViewChildren ka QueryList Angular ka custom class hai (Array nahi) — .toArray(), .changes, .first. Signal viewChildren() plain readonly array deta hai.
  • Content queries descendants: true (default v17+) — nested projected elements bhi match hote hain.
  • <ng-content> ka select selector syntax use karta hai (S2) — attribute, class, element sab.
  • Projected content ka DI: wo parent ke ElementInjector se resolve hota hai, child ke nahi — isliye viewProviders projected content ko nahi dikhte (Ch3 S9).
  • Fallback content: <ng-content>Default text</ng-content> — v18+ mein supported (pehle nahi tha).
  • Dynamic components (createComponent) queries se nahi milte — wo template ka hissa nahi hain.

15. Interview Questions

EASY ViewChild aur ContentChild mein farak?

Sawaal ek hi hai: "wo element kisne likha?" viewChild — maine apne template mein likha. contentChild — parent ne mere tags ke beech likha aur wo <ng-content> se project hua. Timing bhi alag: content queries pehle resolve hoti hain (ngAfterContentInit), phir view (ngAfterViewInit) — kyunki projected content parent ke creation ka hissa hai.

EASY viewChild se mila reference constructor mein kyun undefined hota hai?

Constructor ke waqt template abhi create nahi hua — DOM nodes aur child components exist hi nahi karte. Query creation pass ke dauran resolve hoti hai. Legacy mein ngAfterViewInit ka wait karna padta tha; signal queries mein signal undefined se shuru hota hai aur value aane pe update ho jaata hai, toh effect()/afterNextRender() se react karo.

MEDIUM Signal queries ne decorator queries se kya behtar kiya?

(1) Timing declarative — static flag ka confusion gaya; signal khud batata hai kab value hai. (2) Reactive — computed()/effect() se track ho jaati hain; QueryList.changes.subscribe() + manual unsubscribe ki zaroorat nahi. (3) Type safety — viewChild() deta hai Signal<T | undefined>, viewChild.required() deta hai Signal<T>; ! assertion (Ch2 S1) ki zaroorat nahi. (4) Plain arrays — viewChildren() readonly array deta hai, custom QueryList class nahi.

MEDIUM ng-content ke andar ka content kis context mein evaluate hota hai?

Parent ke context mein. Jo bindings tum <app-card>{{ someValue }}</app-card> mein likhte ho, wo parent ki someValue padhti hain — card ki nahi. Aur us content pe parent ka _ngcontent attribute hota hai, isliye card ki CSS uspe nahi lagti (S4). Doosri important baat: projected content hamesha create hota hai, chahe child use render kare ya na kare — <ng-content> ko @if ke andar rakhne se content lazy nahi hota. Lazy chahiye toh <ng-template> + ngTemplateOutlet.

HARD read option kya karta hai aur wo Angular ke DI se kaise juda hai?

read batata hai ki matched node se kaunsa token nikalna hai. Default: agar locator component/directive class hai toh uska instance; agar template ref string hai toh ElementRef (ya us element ka component, agar ho). Par tum kuch bhi maang sakte ho — ElementRef, ViewContainerRef, TemplateRef, ya koi bhi directive/token. Mechanism: har matched element ka apna ElementInjector hai (Ch3 S9), jisme uske component + saari directives + providers registered hote hain. read basically us ElementInjector se token resolve karta hai. Isliye viewChild(MyDir, {read: SomeService}) bhi kaam karta hai agar SomeService us element ke injector mein provide ho. Ye samajhna do cheezein unlock karta hai: (1) read: ViewContainerRef se dynamic component insertion ka anchor; (2) ek element pe lagi doosri directive ka instance nikalna — bilkul waise hi jaise ek directive doosri ko inject() karti hai (S2 ka pattern). Yaani queries aur DI ek hi underlying structure (ElementInjector) ke do interfaces hain.

SCENARIO Ek Accordion component banana hai: multiple panels, ek waqt mein ek open (ya multiple, configurable), keyboard navigation, aur consumer panels ko @for se dynamically render kar sake. Design karo.

(1) Structure: <app-accordion> container + <app-panel> children, projected. Panel apna content <ng-template> mein rakhe (lazy — band panel ka content DOM mein na ho; ye perf aur correctness dono ke liye, warna 20 panels ke chart sab render ho jaayenge). (2) Discovery: panels = contentChildren(PanelComponent) — signal, toh @for se panels add/remove hone pe apne aap update. Legacy QueryList.changes mein ye manual subscribe tha. (3) State ownership: open state container ke paas — openIds = signal<Set<string>>(new Set()). Panel computed(() => container.isOpen(this.id())) se apna state padhe. Panel ko container ka reference DI se milega (inject(AccordionComponent)) — kyunki wo uska ancestor hai, ElementInjector chain se resolve hoga (Ch3 S9). Ye output forwarding se better hai. (4) Mode: multi = input(false, {transform: booleanAttribute}) — single mode mein open pe baaki band. (5) Keyboard: container pe host: {'(keydown.arrowdown)': 'focusNext()'} (S8); panels ke headers ko viewChildren se nahi — contentChildren(PanelComponent) se instances mile hain, unke headerEl (unka apna viewChild) se focus karo. Roving tabindex pattern. (6) A11y: header <button> ho (native, S2!), aria-expanded, aria-controls, panel pe role="region". (7) Testing: panels ko mock kiye bina test karo — container ka contract "kitne panels open hain" hai. (8) Gotcha jo bolna chahiye: content projection se aaye panels ka DI parent ke injector se resolve hota hai, isliye container ke viewProviders unhe nahi milenge — providers use karo. Ye detail 90% candidates nahi jaante.

HostBinding & HostListener

Covers: host object (modern) · @HostBinding / @HostListener (legacy) · Renderer2 · hostDirectives composition · Event options · a11y patterns

1. What is it?

Host element = wo DOM element jispe component/directive baitha hai. <app-card> component ka host hai; <button appRipple> directive ka host hai.

  • Host binding — host element ki property/attribute/class/style ko component ki state se bind karna.
  • Host listener — host element pe (ya window/document pe) event sunna.

Modern syntax metadata ke host object mein hai; purana syntax @HostBinding/@HostListener decorators mein.

2. Why do we need it?

Component apne andar ko template se control karta hai — par apne host ko nahi. Host toh parent ke template mein likha hai. Do galat raste hain: (a) parent har jagah class lagaye (<app-card class="card selected"> — component ka internal state parent ki zimmedari ban gaya), ya (b) component nativeElement.classList.add() kare (imperative, SSR pe crash, CD ke bahar). Host bindings teesra, sahi rasta hain: declarative, component ke andar, CD ke saath integrated.

3. What problem does it solve?

  • Self-styling → component apna host state khud reflect kare ([class.selected]).
  • Accessibility → role, aria-*, tabindex component ke saath travel karein, parent bhoole nahi.
  • Attribute directives → behavior attach karne ka poora mechanism (ripple, tooltip, autofocus).
  • Direct DOM manipulation → declarative bindings, SSR-safe.

4. Internal Working

5. Execution Flow — event target ka resolution

  '(click)'          --> host element pe listener
  '(window:resize)'  --> window pe listener       [component destroy pe auto-removed!]
  '(document:keydown)' --> document pe listener   [same]
  '(body:scroll)'    --> body pe

  ► Global listeners ka cleanup Angular karta hai — Ch1 S8 ka Big Five #3
    (global listener leak) is API ke saath structurally impossible.
    Manual addEventListener(window, ...) mein tumhe khud hataana padta.

  ► ZONE.JS (Ch3 S8): host listeners patched hain, toh har event CD trigger karta hai.
    High-frequency events (mousemove, scroll) pe ye jank deta hai.
    Zoneless mein: sirf signal write CD schedule karega.

9. Advanced Example — Renderer2 aur event options

@Directive({ selector: '[appInfiniteScroll]' })
export class InfiniteScrollDirective {
  private el = inject<ElementRef<HTMLElement>>(ElementRef);
  private renderer = inject(Renderer2);                     // L1
  private zone = inject(NgZone);
  reachedEnd = output<void>();

  constructor() {
    // ❌ host: {'(scroll)': 'onScroll()'} — har scroll event pe CD (Ch3 S8)

    // ✅ zone ke bahar listen karo, sirf zaroori pe andar aao
    this.zone.runOutsideAngular(() => {                     // L2
      const unlisten = this.renderer.listen(                 // L3
        this.el.nativeElement, 'scroll',
        throttle(() => {
          if (this.isNearBottom()) {
            this.zone.run(() => this.reachedEnd.emit());     // L4
          }
        }, 200)
      );
      inject(DestroyRef).onDestroy(unlisten);                // L5
    });
  }

  private isNearBottom() {
    const e = this.el.nativeElement;
    return e.scrollTop + e.clientHeight >= e.scrollHeight - 200;
  }
}

  • L1: Renderer2 ek abstraction hai DOM ke upar. Kyun? Kyunki Angular sirf browser mein nahi chalta — SSR (server pe DOM nahi), aur Web Workers. Renderer2 platform-specific implementation ko delegate karta hai. nativeElement.style.x = y likhne pe wo server pe crash karega; renderer.setStyle() nahi.
  • L2: Zone ke bahar (Ch3 S8) — scroll 60+ times/sec fire hota hai; har event pe tick() = poore tree ki CD = jank.
  • L3: renderer.listen() ek unlisten function return karta hai — cleanup ka handle. (Ye Ch1 S8 ka teardown pattern hai, framework-level pe.)
  • L4: Jab sach mein kuch hua, tab zone.run() se wapas Angular ke andar — output emit hoga aur CD chalegi. Ek hi baar, throttled.
  • L5: DestroyRef.onDestroy() — declarative cleanup, ngOnDestroy hook likhe bina.
  • Zoneless mein (Ch3 S8): L2 aur L4 dono ki zaroorat nahi — koi zone hai hi nahi, toh scroll listener CD trigger karta hi nahi. Sirf signal write ya emit se downstream state change hi CD schedule karega. Yaani architecture ke change ne ek poora defensive pattern obsolete kar diya — yahi progress ka nishaan hai.

11. Common Mistakes

Galtiyan
  • nativeElement.style/classList/innerHTML se direct DOM manipulation — SSR pe crash, CD ke bahar, aur XSS ka rasta (innerHTML). Host bindings ya Renderer2.
  • High-frequency host listeners ((scroll), (mousemove)) zone apps mein — har event ek tick(). Throttle + runOutsideAngular, ya zoneless.
  • @HostBinding ke saath getter — wo har CD cycle mein call hota hai. Signal/computed use karo.
  • Host mein complex expressions — '[class.x]': 'a() && b() || c()'. Ek computed() banao; host mein sirf uska naam.
  • role/tabindex parent ke bharose chhodna — component apna a11y contract khud laaye.
  • Component pe class input lena aur [class] host binding dono — dono lad-te hain ([class] string overwrite kar sakta hai). [class.x] granular bindings safe hain.
  • Inheritance se behavior share karna jab hostDirectives se ho sakta tha.

12. Best Practices

Production rules
  • Naya code: host: {} object (metadata ke saath ek jagah). Decorators legacy.
  • Host bindings ke expressions ko computed() mein rakho — memoized aur testable.
  • A11y attributes (role, aria-*, tabindex) component ke host mein — travel karte hain component ke saath.
  • Behavior composition = hostDirectives, inheritance nahi.
  • DOM chhoona ho toh Renderer2 + afterNextRender (SSR-safe); global listeners host syntax se ((window:resize)) — cleanup automatic.
  • High-frequency events: throttle + zoneless (ya runOutsideAngular).
  • typeCheckHostBindings: true (Ch3 S3) — host expressions bhi type-checked.

13. Performance

Host bindings component ke hostBindings function mein compile hote hain aur uski CD cycle mein evaluate hote hain — bilkul template bindings jaise. Har host binding = ek slot + ek comparison. 10 host bindings × 1000 instances = 10,000 comparisons per cycle. Chhota rakho.

Host listeners ka cost do jagah: (1) listener registration (creation, ek baar), (2) event fire hone pe CD trigger — yahi asli cost hai zone apps mein. (mousemove) host listener 60fps pe poore app ki CD chala sakta hai. Solutions: throttling, runOutsideAngular, ya (best) zoneless + signals.

hostDirectives ka cost: har directive ek instance + apne host bindings. 3 host directives = 3 instances per host element. Design system components mein ye consciously budget karo.

14. Edge Cases

  • '[attr.x]': 'value() || null' — null attribute ko remove karta hai; false string "false" ban jaata hai (S3 ka trap).
  • Global targets: window:, document:, body:. Cleanup automatic, par listener har instance ka apna hai — 1000 components = 1000 window listeners. Shared service better.
  • hostDirectives ke inputs/outputs explicitly expose karne padte hain (inputs: ['x']), warna consumer unhe bind nahi kar sakta.
  • Host directive ka selector ignore hota hai — wo bina attribute likhe apply hoti hai.
  • Ek element pe do directives same class bind karein toh dono apply hoti hain (merge), par same property pe conflict ka behavior undefined-ish hai — avoid.
  • @HostListener('document:click') + component-level provider = classic leak-free pattern hai, par har instance apna listener lagata hai — dropdown/menu mein ye multiply ho sakta hai.
  • Static host attributes ('role': 'article') template mein overridden ho sakte hain (<app-card role="region">) — static values merge hote hain, binding jeetti hai.

15. Interview Questions

EASY HostBinding kya karta hai?

Component/directive ke host element (jis element pe wo baitha hai) ki property, attribute, class, ya style ko component ki state se bind karta hai. Modern syntax: metadata mein host: { '[class.selected]': 'selected()' }. Faayda: component apna host state khud manage karta hai, parent ko yaad rakhne ki zaroorat nahi, aur ye CD ka hissa hai (declarative, SSR-safe).

EASY Renderer2 kyun use karte hain, seedha nativeElement kyun nahi?

Kyunki Angular sirf browser mein nahi chalta. SSR (server pe koi DOM nahi) aur Web Workers mein nativeElement.style.x = y crash karega ya kaam nahi karega. Renderer2 ek abstraction hai jo platform-specific implementation ko delegate karti hai. Aur wo listen() se unlisten function deti hai (cleanup ka handle). Zyada tar cases mein host bindings Renderer2 se bhi behtar hain — declarative.

MEDIUM Host listener se window pe event sunne ka cleanup kaun karta hai?

Angular. host: { '(window:resize)': 'onResize()' } mein component destroy hone pe listener automatically remove ho jaata hai. Ye Ch1 S8 ke "Big Five" memory leaks mein se ek (global listener leak) ko API design se hi khatam kar deta hai. Manual window.addEventListener mein tumhe khud remove karna padta hai. Caveat: har component instance ka apna listener hota hai — 1000 instances = 1000 window listeners; shared service + ek listener better hai.

MEDIUM hostDirectives kya hai aur inheritance se better kyun?

hostDirectives: [RippleDirective, TooltipDirective] — component/directive apne host pe doosri directives "attach" kar sakta hai, bina consumer ke unhe likhe. Inheritance (extends BaseInteractive) se better isliye: (1) multiple behaviors compose ho sakte hain, single-parent limitation nahi; (2) har behavior alag se testable aur reusable; (3) koi protected-member coupling nahi (Ch2 S5 ka "composition over inheritance"); (4) inputs/outputs explicitly expose karne padte hain — API surface deliberate rehti hai. Angular Material apne components mein isi se ripple/a11y behaviors compose karta hai.

HARD host: {'(mousemove)': 'onMove($event)'} likhne se kya performance problem hoti hai, aur teen alag solutions?

Zone.js host listeners ko patch karta hai (Ch3 S8): har mousemove (60+/sec) ke baad onMicrotaskEmpty fire hota hai aur ApplicationRef.tick() chalti hai — poore component tree ki change detection, 60 baar per second. 1000 bindings wale app mein ye 60,000 comparisons/sec hai, aur browser jank karega. Solutions: (1) runOutsideAngular + Renderer2.listen: listener zone ke bahar register karo, throttle karo, aur sirf jab UI update chahiye tab zone.run() se andar aao. Purana par reliable. (2) OnPush + immutable state: CD chalegi phir bhi, par sirf dirty subtrees check honge — partial fix, root problem (tick frequency) wahi hai. (3) Zoneless + signals (best): Zone.js hai hi nahi, toh listener khud CD trigger nahi karta. Handler signal write karega tabhi affected views schedule honge — aur wo bhi coalesced (ek microtask/frame mein ek baar). Yahan runOutsideAngular ki zaroorat hi khatam ho jaati hai. Deep observation jo bolna chahiye: zone-era mein hum framework ke guessing ko manually correct karte the (runOutsideAngular = "yahan mat guess karo"); signals-era mein framework guess karta hi nahi, toh wo correction API obsolete ho gayi. Aur bonus: @if/@for se listener conditionally register/deregister ho — host mein ye control nahi hai, wahan Renderer2 chahiye.

SCENARIO Design system mein ek appTooltip directive chahiye: hover/focus pe dikhe, escape pe band ho, viewport ke bahar na jaaye, screen readers ko dikhe, aur 500 elements pe use ho. Design + performance considerations?

(1) Trigger events: host listeners se (mouseenter), (mouseleave), (focus), (blur) — focus/blur zaroori hain (keyboard users; mouse-only tooltip a11y fail hai). Escape ke liye (document:keydown.escape) — par sirf tab register ho jab tooltip open ho (warna 500 document listeners!). Isliye escape wala listener Renderer2.listen se conditionally, open hone pe. Ye edge case hai jo interviewer dhundhta hai. (2) Rendering: tooltip host ke andar nahi ho sakta (overflow/z-index/stacking context). CDK Overlay ya ek portal service se body mein render karo — lekin phir S4 ka lesson: uski CSS :host ::ng-deep se target nahi hogi; panelClass + global styles. (3) Positioning: viewport collision detection — CDK ConnectedPositionStrategy (flexible positions with fallbacks). Khud likhoge toh getBoundingClientRect + afterNextRender (SSR-safe, Ch3 S8). (4) A11y: aria-describedby host pe (tooltip ka id), tooltip pe role="tooltip". Host binding se: '[attr.aria-describedby]': 'open() ? id : null'. (5) Performance at 500 instances: har directive instance ka apna host binding set (chhota rakho — 2-3 max); tooltip ka DOM lazy banao (sirf open pe ViewContainerRef.createComponent, band pe destroy) — 500 hidden tooltip DOMs mat banao; ek shared overlay service jo ek hi tooltip instance reuse kare (kyunki ek waqt mein ek hi dikhta hai) — ye sabse bada win hai. (6) Composition: hostDirectives se buttons/icons pe attach karo, taaki consumer ko har jagah appTooltip likhna na pade. (7) Cleanup: overlay destroy, document listener remove, timers clear — DestroyRef.onDestroy() mein (Ch1 S8). Ye answer host APIs + overlays + a11y + memory + perf sab jodta hai.

Component Architecture & Communication

Covers: Smart vs Dumb (container/presentational) · 7 communication patterns · State ownership · Kaunsa pattern kab · Testing implications

1. What is it?

Do concepts jo ek doosre ke bina adhoore hain:

  • Smart (container) components — services inject karte hain, data fetch karte hain, state own karte hain, routed hote hain. "Kya" jaante hain.
  • Dumb (presentational) components — sirf input() lete hain, output() emit karte hain. Koi service, koi HTTP, koi router. "Kaise dikhna hai" jaante hain.

Aur communication = in components ke beech data/events kaise behte hain. Angular mein saat patterns hain, aur seniority ka test hai: kaunsa kab.

2. Why do we need it?

Bina is separation ke har component apni data fetching karta hai — 50 components, 50 jagah HTTP calls, 50 jagah loading states, aur ek bhi component test karne ke liye poora HTTP stack mock karna padta hai. Aur reuse asambhav: <app-order-row> jo khud OrderService se data laata hai, use tum kisi doosre context mein use nahi kar sakte.

Container/presentational split ye kehta hai: data ki zimmedari kam jagah rakho, presentation ki zyada jagah reuse karo.

3. What problem does it solve?

  • Untestable components → dumb components ka test = inputs do, DOM check karo. Zero mocking.
  • Unreusable UI → presentation data source se decoupled.
  • Scattered state → state ownership clear (kaun badalta hai, kaun padhta hai).
  • Prop drilling → sahi pattern chunne se (service/DI) drilling ki zaroorat hi nahi.

11. Common Mistakes

Galtiyan
  • Dumb components mein services inject karna — ek baar Router ya HttpClient ghusa, wo component reusable nahi raha aur uska test 3 guna bada ho gaya.
  • viewChild ko data ke liye use karna — child ki internals pe dependency. Imperative APIs ke liye hi.
  • Sab kuch providedIn: 'root' — flow-scoped state global ban gayi, manual reset ki zaroorat, aur do parallel instances ka conflict.
  • Prop drilling ko sehna — 4 levels tak input pass kar rahe ho? Wo signal hai ki state galat jagah baithi hai, ya scoped service chahiye.
  • Navigable state ko component mein rakhna — filter/page refresh pe gaya, link share nahi hota, back button toota.
  • Store ke signals ko writable expose karna — readonly cart = this._cart.asReadonly() use karo; mutations methods se.
  • Har cheez ke liye NgRx uthana — 90% apps ko signals + scoped services kaafi hain. Store tab jab: time-travel debugging, complex derived state graphs, ya team-wide strict conventions chahiye.

12. Best Practices

Production rules
  • Routed page = smart. Baaki sab dumb, jab tak proven zaroorat na ho.
  • Dumb component ka contract: sirf input() + output(), koi inject() (a11y/format services chhod ke).
  • State ownership: sabse neeche wala common ancestor. Scope = lifetime.
  • Navigable state (filter, page, sort, id, tab) → URL. Ephemeral UI state → component signal.
  • 3+ levels ya siblings → scoped service (providers array mein, root mein nahi).
  • Store pattern: private writable signals, public asReadonly() + computed(), mutations methods se.
  • viewChild sirf imperative DOM/component APIs ke liye — kabhi data ke liye nahi.

13. Performance

Architecture ka performance pe seedha asar hai. (1) Dumb + OnPush = CD boundary: parent ki state badli par child ka input reference wahi hai → child ka poora subtree skip (Ch3 S8). Smart components jo har jagah services se subscribe karte hain, wo boundaries todte hain. (2) Scoped services + signals = fine-grained updates: sirf wo components jo us signal ko consume karte hain, dirty hote hain — prop drilling mein toh raste ke har component ki CD chalti hai. (3) Immutable inputs (Ch1 S4) — reference equality se hi OnPush kaam karta hai; mutation se child update hi nahi hoga (galat), ya poora tree Default CD mein chalega (dheema).

Ek concrete measurement: 5-level nesting mein prop drilling se ek value badalne pe 5 components ki CD chalti hai; scoped signal se sirf consumer ki.

14. Edge Cases

  • Kuch "dumb" components ko services chahiye hoti hain (i18n, date formatting, a11y announcer) — ye acceptable hai; rule data/business services ke liye hai.
  • Scoped service + @if: agar provider component @if ke andar destroy ho, uska service instance bhi gaya — state loss. Provider ko stable ancestor pe rakho.
  • Content projection ke saath DI: projected children parent ke injector se resolve karte hain (Ch3 S9, S7 ka edge case) — container ke viewProviders unhe nahi dikhte, providers dikhte hain.
  • Two-way (model()) + OnPush: model set hone pe output emit hota hai, parent state badalti hai, phir input wapas aata hai — ek extra cycle. Simple values ke liye theek, bade objects ke liye nahi.
  • Router state + component input binding: query params string hote hain — transform: numberAttribute (S5) se coerce karo.
  • Sibling communication bina common ancestor ke (do routed outlets) — root service ya router state hi rasta hai.

MEDIUM Do sibling components ke beech communication kaise karoge?

MEDIUM viewChild se child ki method call karna kab theek hai?

Sirf tab jab wo cheez declaratively possible hi nahi: input.focus(), element.scrollIntoView(), video.play(), canvas context, third-party lib initialization. Data padhna (child().value) ya state badalna (child().reset()) — ye input/output/model se hona chahiye, warna child ki internals pe dependency ban jaati hai aur wo reusable nahi rehta. Rule: queries imperative DOM/browser APIs ke liye hain, component communication ke liye nahi.

HARD "State kahan rehni chahiye" ka decision framework batao — aur URL kab involve hota hai?

Teen sawaal, is order mein. (1) Kaun padhta hai? Saare consumers ka sabse neeche wala common ancestor — state wahin rakho, usse upar nahi (warna unnecessary CD scope aur coupling). (2) Kaun badalta hai? Agar mutations ancestor se door hain (deep child), toh input/output drilling ke bajaye us ancestor pe scoped service provide karo — signals se consumers directly padhenge/likhenge. Scope ka lifetime state ke lifetime se match karna chahiye: wizard state = route-level provider (flow chhoda, state gayi); auth = root. (3) Kya user is state ko share/refresh/back kar sakta hai? Ye wo sawaal hai jo log poochte hi nahi. Filter, pagination, sort, selected tab, open item ka id, search query — haan, ye URL mein hone chahiye: link shareable, refresh survive kare, back button expected kaam kare, aur SSR pe server sahi content de sake. Tooltip open, hover state, unsaved draft, animation state — nahi, ye component signals hain. Modern ergonomics: withComponentInputBinding() se route params seedhe input() mein aate hain, toh URL-state use karna ab component-state jitna hi aasan hai — purana excuse ("router se padhna verbose hai") ab valid nahi. Bonus: URL state ko single source of truth banao — component usse derive kare (computed), uski copy na rakhe, warna do sources of truth aur sync bugs.

SCENARIO Ek dashboard hai: left mein filter panel, right mein 3 widgets (chart, table, summary). Filter badle toh teeno update hon. Table ka row select ho toh chart highlight kare. Aur user ko dashboard ka link share karna hai. Architecture design karo.

Do alag state categories hain — inhe alag treat karna hi answer ka core hai. (1) Filter = navigable state → URL. Query params (?from=...&to=...&category=...). Filter panel router.navigate se update kare; dashboard page withComponentInputBinding() se inputs le (from = input.required<string>()). Link shareable, refresh-safe, back button kaam karta hai, aur SSR pe server sahi data render kar sakta hai. Widgets ko filter input se milta hai — teeno dumb rehte hain. (2) Row selection = ephemeral UI state → scoped signal. URL mein daaloge toh har row click pe history entry banegi (back button 50 baar dabana padega). Dashboard page ka selectedRowId = signal<string|null>(null), ya (agar teeno widgets ko chahiye aur nesting deep hai) ek DashboardStore jo page ke providers mein ho — root mein nahi, taaki do dashboard instances ka apna state ho. (3) Data fetching = smart page. Page resource() se data laata hai (filter signals params mein → filter badla toh purani request abort, nayi chale). Loading/error states page handle kare, widgets ko sirf data mile. (4) Widgets = dumb. chart: input.required<readonly Point[]>() + highlightedId = input<string|null>(). table: data input + rowSelected = output<string>(). Zero services. Zero HTTP. OnPush → filter badalne pe sirf wahi widgets re-render hon jinke inputs badle. (5) Table → chart communication: table output → page signal set → chart ko input se mila. Siblings kabhi directly baat nahi karte — ancestor mediate karta hai. Agar nesting 3+ ho jaye toh scoped store. (6) Testing: har widget ka test = inputs do, DOM assert karo. Page ka test = router + service mock. (7) Performance: readonly array types (Ch2 S7) + immutable updates → OnPush sahi kaam karega; chart ko @defer (on viewport) (Ch3 S7) agar bhaari lib ho. Jo cheez ye answer prove karti hai: "kaunsa pattern" ka jawab "kaunsi state" se aata hai, aur state ka pehla classification navigable vs ephemeral hai.
