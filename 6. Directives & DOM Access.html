S-01

Directive ka Falsafa & Anatomy

Covers: Directive kya hai · δdir vs δcmp · Teen prakaar · Lifecycle · Ek element pe kai directives · Kab directive banayein

1. What is it?

Directive = ek class jo DOM element ko behavior deti hai. Component bhi ek directive hai — bas uske paas ek template bhi hai.

  DIRECTIVE (base concept)
      |
      |-- Attribute Directive    behavior add karti hai, koi template nahi
      |                          <button appRipple>
      |
      |-- Structural Directive   DOM ka SHAPE badalti hai (add/remove views)
      |                          <div *appIfRole>
      |
      |-- COMPONENT              directive + apna template + apna view
                                 <app-card>

  ► Compile hoke:  Directive --> static δdir  |  Component --> static δcmp
  ► δcmp mein δdir ki saari properties hain + template fn, decls, vars, styles.
    Yani component literally directive ka superset hai.

2. Why do we need it?

Kyunki har behavior ke liye component banana galat hai. Ek tooltip, ek ripple, ek autofocus, ek permission check — ye sab ek existing element ke saath judte hain, uski jagah nahi lete. Agar sirf components hote, toh <app-tooltip><button>Pay</button></app-tooltip> likhna padta — ek extra DOM node, native button ki a11y semantics tooti hui (Ch4 S2), aur do behaviors ek saath lagane ke liye nesting.

Directives Angular ka behavior composition system hain: ek element pe ek component, par jitni chaho directives.

3. What problem does it solve?

  • Wrapper hell → behavior element pe attach hota hai, wrap nahi karta.
  • Native semantics ka nuksan → <button> button hi rehta hai.
  • Cross-cutting concerns → analytics, permissions, focus management — sab reusable directives.
  • Inheritance ki rigidity → multiple behaviors compose ho sakte hain (S9).

9. Advanced Example — ek element pe kai directives, aur unka aapsi rishta

<button appButton appTooltip="Pay now" appAnalytics="pay_click" [disabled]="loading()">
  Pay
</button>

// Is button pe:
//   - ButtonComponent    (selector: 'button[appButton]') — component, ek hi ho sakta
//   - TooltipDirective
//   - AnalyticsDirective
//   Teeno ka instance banega, teeno ek hi ElementInjector share karenge (Ch3 S9)

@Directive({ selector: '[appAnalytics]' })
export class AnalyticsDirective {
  event = input.required<string>({ alias: 'appAnalytics' });

  // SAME ELEMENT ki doosri directives inject ho sakti hain!
  private tooltip = inject(TooltipDirective, { optional: true, self: true });  // L1
  private button = inject(ButtonComponent, { optional: true, self: true });   // L2
  private analytics = inject(AnalyticsService);                              // L3

  constructor() {
    inject(ElementRef).nativeElement.addEventListener('click', () => {
      this.analytics.track(this.event(), {
        tooltip: this.tooltip?.text(),                                       // L4
        variant: this.button?.variant(),
      });
    });
  }
}

10. Code Explanation — Line by Line

  • L1–L2: Yahi directives ka superpower hai. Ek element ki saari directives (+ uska component) ek hi ElementInjector mein register hoti hain (Ch3 S9). Toh koi bhi directive doosri ko inject() kar sakti hai. self: true ka matlab: sirf isi element se dhundho, parent chain mein mat jao. optional: true kyunki wo directive lagi bhi ho sakti hai, nahi bhi.
  • L3: Normal service injection — environment injector chain se (root).
  • L4: Ab analytics directive ko pata hai ki is button ka tooltip kya hai, uska variant kya hai — bina parent se kuch pass kiye. React mein ye natural nahi hai (wahan HOC/context/hooks ka jugaad karna padta). Ye Angular ka structural advantage hai.
  • Caveat: ye coupling bhi hai. AnalyticsDirective ab TooltipDirective ko jaanti hai. optional: true se soft rakha hai, par ideal design mein ye information ek shared token/service se aani chahiye, na ki concrete directive class se. Interview mein dono side bolo.

11. Common Mistakes

Galtiyan
  • Har chhote behavior ke liye directive — har instance ek object + LView slot + ElementInjector entry. 1000 rows pe measurable.
  • Directive ko component ki tarah use karna (wrapper banana) — native semantics tootengi (Ch4 S2).
  • Parent ke imports mein directive add karna bhool jaana — Angular chup-chaap ignore kar deta hai (attribute unknown attribute ban jaata hai; strictTemplates bhi ise nahi pakadta!). Ye #1 "meri directive chal hi nahi rahi" ka cause hai.
  • Directive selector bina prefix ([highlight]) — library collision.
  • Structural directive banana jahan @if kaafi tha (Ch5 S10) — extra abstraction, extra instances.
  • Directive mein bhaari logic — wo har instance mein duplicate hoti hai. Logic service mein, directive patli.

12. Best Practices

Production rules
  • Directive = behavior on existing element. Component = naya UI unit. CSS = pure styling. Pipe = pure transformation.
  • Selector: [appCamelCase], prefix mandatory, compound se host restrict karo (button[appRipple]).
  • Directive patli rakho: state signals mein, logic services mein, DOM kaam Renderer2/host bindings se.
  • exportAs do agar template ko instance chahiye (#t="appTooltip").
  • Directive-to-directive DI use karo (self: true, optional: true) — par concrete classes ke bajaye tokens prefer karo.
  • Naye code mein signals: input(), output(), host object.

13. Performance

Har matched directive per element: (1) ek instance (δfac()), (2) ek LView slot (Ch3 S6), (3) ek ElementInjector entry agar wo DI use karti hai (Ch3 S9), (4) uske hostVars ke barabar binding slots + har CD cycle mein utne comparisons (S8).

Numbers mein: 5,000 rows × 2 directives = 10,000 instances + 10,000 injector entries. Ye heap mein dikhta hai aur creation time mein bhi. Isliye bade lists mein: directives ko row-level pe rakho (cell-level pe nahi), ya CSS/delegation se kaam chalao (Ch5 S3 ka data-* + CSS pattern, Ch5 S5 ka event delegation).

Aur yaad rakho: @if/@for (blocks) ne *ngIf/*ngFor (directives) ko replace kiya hi isliye tha — directive overhead hatane ke liye (Ch5 S8).

14. Edge Cases

  • Directive inheritance: base class pe @Directive() (bina selector ke) lagana zaroori hai, warna uske inputs/lifecycle inherit nahi honge ("abstract directive" pattern).
  • Component ek directive selector se bhi match kar sakta hai (selector: 'button[appButton]') — tab wo attribute directive jaisa lagta hai par uska apna template/view hota hai.
  • Ek element pe do components match → compile error. Do structural directives → compile error. Attribute directives → unlimited.
  • Directive ke ngOnChanges host element ke inputs pe nahi, uske apne inputs pe chalta hai.
  • Directives @if/@for ke andar ke elements pe lagi hon toh view destroy pe unka ngOnDestroy chalta hai.
  • hostDirectives se lagi directive ka selector ignore hota hai (S9).

15. Interview Questions

EASY Component aur directive mein farak?

Component ek directive hai jiska apna template (aur isliye apna view/LView) hota hai. Directive sirf existing element ka behavior/appearance badalti hai. Compile hoke component δcmp banta hai (jisme δdir ki saari cheezein hain + template function, decls, vars, styles), directive δdir. Ek element pe ek component, par kai directives.

EASY Directives ke teen prakaar?

(1) Components — template ke saath. (2) Attribute directives — element ka behavior/appearance badalti hain ([appHighlight]). (3) Structural directives — DOM ka shape badalti hain, views add/remove karke (*appIfRole); ye TemplateRef + ViewContainerRef use karti hain.

MEDIUM Ek element pe kai directives kaise coordinate karti hain?

Us element ki saari directives (aur uska component) ek hi ElementInjector mein register hoti hain (Ch3 S9). Isliye koi bhi directive doosri ko inject(OtherDirective, {self: true, optional: true}) se le sakti hai — self se sirf usi element se, optional se agar wo lagi na ho toh null. Yahi Angular ka behavior-composition mechanism hai; React mein ye natural nahi (HOC/context chahiye). Caveat: ye concrete-class coupling banata hai — bade design systems mein shared token behtar hai.

MEDIUM Meri custom directive lag hi nahi rahi — kya check karoge?

Sabse pehla: parent component ke imports array mein directive add ki hai? Standalone mein ye zaroori hai, aur bhoolne pe Angular chup-chaap ignore karta hai — attribute ko plain HTML attribute maan leta hai, koi error nahi (component ke ulta, jahan "not a known element" milta hai). Doosra: selector spelling/case ([appHighlight] camelCase attribute). Teesra: element pe compound selector match ho raha hai (button[appX] par tumne div pe lagaya). Chautha: hostDirectives se toh nahi laga rahe (wahan selector ignore hota hai).

HARD Component "directive ka superset" hai — iske teen concrete implications batao.

(1) Compiled structure: δcmp mein δdir ki saari fields hain (selectors, inputs, outputs, hostBindings, hostVars, exportAs, features) plus template function, decls/vars, styles, encapsulation, dependencies. Yani runtime ke liye component ek directive hai jo apna view bhi create karta hai. (2) API parity: jo kuch directive kar sakti hai, component bhi kar sakta hai — host bindings, exportAs, DI, lifecycle, hostDirectives. Aur component ka selector attribute-based bhi ho sakta hai ('button[appButton]'), toh "component = element selector" ek convention hai, rule nahi. (3) Injector semantics: component aur uske element ki directives ek hi ElementInjector share karti hain — isliye directive component ko inject kar sakti hai aur ulta bhi. Aur viewProviders sirf component ke view ko dikhte hain, uske host pe lagi directives ko nahi (kyunki wo host ke injector mein hain) — ye ek subtle rule hai jo tabs/form-control implementations mein bite karta hai. (4 bonus) hostDirectives dono pe kaam karti hain, jo composition ko uniform banata hai (S9).

SCENARIO Team ne 40 custom directives bana di hain: appMarginTop, appTextCenter, appBold, appIfAdmin, appTrackClick, appTooltip... Code review mein kya bologe?

Har directive ko ek framework se test karo: kya ye behavior hai, ya styling hai, ya control flow hai, ya derivation hai? (1) appMarginTop, appTextCenter, appBold = pure styling. Ye CSS classes/utility classes honi chahiye — zero JS, zero instances, zero CD cost. Har directive per element ek object + LView slot + injector entry hai (Ch3 S9). 1000 elements pe ye 1000 objects hain ek margin ke liye. Delete karo, CSS classes banao. (2) appIfAdmin = control flow. @if (auth.isAdmin()) se ho jaata hai — koi directive instance nahi, better narrowing, aur security ka reminder: DOM se hataana authorization nahi hai (server pe enforce karo). Structural directive tab likho jab reuse + microsyntax ka asli value ho. (3) appTrackClick, appTooltip = asli directives. Ye behavior hain, existing element pe attach hote hain, reusable hain. Inhe rakho — par patla rakho (logic service mein), aur hostDirectives se compose karo (S9). (4) Systemic fix: ek decision doc — "directive banane se pehle ye 4 sawaal". Aur ek lint rule ya PR checklist. (5) Migration: styling directives sabse pehle hataao (mechanical, zero risk, turant perf win). Phir control-flow wali. (6) Measurement: pehle aur baad mein ek bade list page ka heap snapshot + CD time — numbers hi team ko convince karenge. Punchline: directive ek behavior primitive hai, ek CSS shorthand nahi.

S-02

Built-in Attribute Directives

Covers: NgClass · NgStyle · NgModel · KeyValueDiffer internals · ControlValueAccessor · Kyun native bindings ne inhe replace kiya

1. What is it?

Angular ke saath aane wali attribute directives — jo CommonModule/FormsModule se aati hain:

  • [ngClass] — classes ka object/array/string se set
  • [ngStyle] — styles ka object se set
  • [(ngModel)] — template-driven forms ka two-way binding

Pehli do ab legacy hain — native [class]/[style] bindings (Ch5 S4) unse har metric pe better hain. Teesri reactive forms ke saamne legacy hai. Par inhe samajhna zaroori hai: (a) purani codebases mein milengi, (b) inka internals ek acchi lesson hai ki directive-based abstraction ka cost kya hota hai.

2. Why do we need it?

Historically: Ivy se pehle native class/style bindings ka koi robust merging system nahi tha. ngClass ne dynamic classes ka problem solve kiya. ngModel ne form controls ko DOM se jodne ka standard diya.

Aaj: [class]/[style] compiler instructions hain apne precedence algorithm ke saath (Ch5 S4). ngClass ki zaroorat khatam. ngModel ki zaroorat reactive forms ne khatam ki (jahan FormControl value ka owner hai, koi two-way binding nahi).

3. What problem does it solve?

  • (Historical) Dynamic classes/styles → object/map se declarative set.
  • DOM ↔ model sync → ngModel + ControlValueAccessor.
  • Aaj ka asli faayda: legacy code padhne aur migrate karne ki samajh.

10. Code Explanation — Line by Line

  • L1–L2: Template mein object literal = har CD cycle mein naya reference (Ch5 S2 ka trap). Ye ngClass ke saath khaaskar bura hai kyunki uska input setter reference change pe hi trigger hota hai.
  • L3–L4: Naya reference milte hi NgClass ko pata nahi ki content same hai — uske paas differ hi ekmatra rasta hai. KeyValueDiffer.diff() purane records (linked list) ko traverse karke naye object se compare karta hai. Ye ek generic abstraction hai, aur uska cost object size ke proportional hai.
  • L5: Aur ye poora kaam "kuch nahi badla" ka jawab dene ke liye hua. Ye classic abstraction tax hai.
  • L6: Native classProp mein compiler ne pehle hi jaan liya ki kaunsi class kis expression pe depend karti hai. Ek boolean, ek comparison, ek classList.toggle (sirf change pe). Koi intermediate object, koi differ, koi ngDoCheck.
  • Bada sabak: yahi wo pattern hai jo poore Angular mein repeat hota hai (Ch5 S10 ka closing point): generic runtime abstraction (differ, IterableDiffer, Zone.js ka guessing) ko specialized compiled code se replace karo. ngClass → classProp, *ngFor → δδrepeater, *ngIf → δδconditional, Zone → signal graph. Ek hi philosophy, chaar jagah.

11. Common Mistakes

Galtiyan
  • Naye code mein ngClass/ngStyle — legacy hain; native bindings har metric pe better.
  • [ngClass] ko template ke [class.x] ke saath mix karna — precedence mein ngClass (directive host) neeche hai, template jeetta hai (Ch5 S4). "Kaam nahi kar raha" ka #1 cause.
  • Object literal inline ([ngClass]="{a: x}") — naya reference har cycle, differ har baar chalta hai.
  • ngModel + reactive forms ek saath ([formControl] ke saath [(ngModel)]) — deprecated, warnings.
  • ngModel ke async value update pe bharosa — wo microtask mein set hota hai; (ngModelChange) ke turant baad model stale ho sakta hai.
  • ngStyle ka unit syntax ({'width.px': w}) — string keys, koi type safety.

12. Best Practices

Production rules
  • Naye code mein ngClass/ngStyle ban karo (ESLint rule). Native [class.x]/[style.x.px].
  • Dynamic key sets zaroori hon toh [class]="computed()" — stable reference.
  • Forms: reactive + typed (nonNullable, getRawValue()). ngModel sirf legacy maintenance.
  • Migration automated: ng generate @angular/core:control-flow structural ke liye; ngClass manual (par mechanical).
  • Legacy code padhte waqt: ngDoCheck dikhe toh performance suspect karo.

13. Performance

Concrete numbers (typical, 1000 elements, 3 dynamic classes):

  • ngClass: 1000 directive instances, 1000 KeyValueDiffer objects (+ internal record linked lists), 1000 ngDoCheck calls per CD cycle, aur inline object literals ke saath 1000 allocations per cycle.
  • Native: 3000 binding slots, 3000 Object.is comparisons per cycle, zero allocations, zero instances.

Aur bundle: NgClass/NgStyle ka code + KeyValueDiffers infrastructure import hota hai. Native bindings compiler instructions hain — jo already runtime mein hain.

ngModel ka apna cost: har control pe NgModel directive + ControlValueAccessor + form registration + async update. Reactive forms mein FormControl ek plain object hai (koi directive per control nahi, sirf formControlName directive jo bind karti hai).

14. Edge Cases

  • ngClass aur static class merge hote hain — ngClass static classes ko nahi hataata (differ sirf apne set ko track karta hai).
  • ngStyle mein null value style hata deti hai (native jaisa hi).
  • NgModel name attribute maangta hai <form> ke andar (control registration ke liye) — standalone mein [ngModelOptions]="{standalone: true}".
  • NgForm aur NgModel ka exportAs (#f="ngForm") — Ch5 S7 ka pattern.
  • CommonModule import karna standalone mein poore module ko laata hai — sirf zaroori directives import karo (NgTemplateOutlet etc.), ya (better) unki zaroorat hi khatam karo.
  • Custom ControlValueAccessor likhna abhi bhi valid hai — custom form controls ke liye (S9 mein hostDirectives ke saath compose ho sakta hai).

15. Interview Questions

EASY ngClass aur [class.x] mein farak?

ngClass ek directive hai: har element pe instance + KeyValueDiffer + ngDoCheck har CD cycle mein. [class.x] ek compiler instruction hai (δδclassProp): ek binding slot, ek Object.is comparison. Native tez, chhota, aur precedence mein upar (template > directive host, Ch5 S4). Naye code mein ngClass use nahi karna chahiye.

EASY ngModel kaise kaam karta hai?

[(ngModel)] desugars hota hai [ngModel] + (ngModelChange) mein. NgModel directive element pe lagi ControlValueAccessor (jaise DefaultValueAccessor for input) se baat karti hai: writeValue() se model → DOM, aur registerOnChange() se DOM → model. Value updates ek microtask mein defer hote hain (historically ExpressionChangedAfterItHasBeenChecked se bachne ko).

MEDIUM KeyValueDiffer kya karta hai aur uska cost kya hai?

NgClass/NgStyle ko ek object milta hai; unhe pata karna hota hai kaunsi keys add/remove/change hui. KeyValueDiffer ek generic diffing abstraction hai jo purane state ke records (linked lists) rakhta hai aur har diff() call pe naye object se compare karta hai. Cost: ek differ object per element, aur ngDoCheck har CD cycle mein (chahe kuch badla ho ya nahi). Aur agar template mein inline object literal hai, toh har cycle naya reference = differ hamesha chalta hai. Native classProp mein compiler pehle hi jaanta hai kaunsi class kis expression pe depend karti hai — ek comparison, zero allocation.

MEDIUM "ngClass kaam nahi kar raha" — kya check karoge?

Sabse pehle precedence (Ch5 S4): ngClass ek directive host binding hai, aur template ka [class.x] uspe jeetta hai (template > directive host). Agar kisi ne same class ko template mein prop-bind kiya hai, ngClass silently ignore ho jaayega. Doosra: object literal ka reference (naya har cycle, differ chalega par sahi result dega). Teesra: CommonModule/NgClass imports mein hai? (Directive silent fail karti hai, Ch6 S1.) Chautha: naya code hai toh ngClass use hi mat karo.

HARD NgClass ka internals aur native class binding ka internals compare karo — aur bataao ye comparison Angular ke evolution ke baare mein kya kehta hai.

NgClass: ek directive class, jo Renderer2, ElementRef, KeyValueDiffers inject karti hai; input setter object ko normalize karta hai; ngDoCheck har CD cycle mein differ.diff() chalata hai jo change records (added/removed/changed) generate karta hai; phir renderer.addClass/removeClass. Yani: ek instance, ek injector entry, ek differ object, N records, aur ek per-cycle hook — sab runtime pe. Native [class.a]="x": compiler ne compile time pe hi jaan liya ki class a expression x pe depend karti hai, aur δδclassProp('a', x) emit kiya — jo LView slot se pichli value uthata hai, Object.is karta hai, aur change pe classList.toggle. Zero allocation, zero instance, zero hook. Ye comparison Angular ke poore evolution ka microcosm hai: *ngIf → δδconditional, *ngFor+IterableDiffer → δδrepeater ka keyed algorithm, Zone.js ka "kuch badla hoga" guessing → signal graph ka exact dependency tracking. Har case mein: ek generic runtime abstraction ko specialized compiled code se replace kiya gaya. Ye Ivy ki locality (Ch3 S1) se possible hua — kyunki compiler ab har template ko poori tarah samajh sakta hai. Aur yahi wajah hai ki naye Angular mein "directive" ka role sikud ke genuine custom behavior tak reh gaya hai; framework ke apne concerns ab primitives hain.

SCENARIO Ek 4-saal purani app hai: 300 components, har jagah ngClass/ngStyle, template-driven forms, *ngIf/*ngFor. Modernization plan banao — priority, risk, aur measurable outcome ke saath.

Sequence risk aur payoff se decide karo, na ki "kaunsa sabse purana lagta hai". (1) Measure first: ek representative heavy page pe baseline lo — bundle size, CD time (Angular DevTools profiler), directive instance count (ng.getDirectives ya heap snapshot), aur ek key interaction ka latency. Bina numbers ke ye "cleanup" hai, "modernization" nahi. (2) Control flow migration (sabse pehle) — automated hai (ng generate @angular/core:control-flow), mechanical, aur review karna aasan. Payoff: CommonModule imports hatte hain (bundle), directive instances khatam (CD), aur track mandatory hone se chhupe hue correctness bugs saamne aate hain (Ch5 S10) — inhe fix karna migration ka sabse valuable side effect hai. Risk: track chunne mein galti; isliye reorder-able lists pe ek test likho. (3) ngClass/ngStyle → native — manual par mechanical. Yahan ek chhupa hua bug milega (Ch5 S4): jahan ngClass aur template [class.x] dono ek class control kar rahe the, wahan behavior badal jaayega (kyunki precedence layer hi hat gayi). Isliye ek-ek component, visual regression tests ke saath. Payoff: instances + differs + ngDoCheck per element gaya. (4) Signals + OnPush — ab jab template layer saaf hai, CD scope attack karo. Ye sabse bada perf win hai par sabse zyada soch maangta hai (Ch4 S5 ka input migration). (5) Forms (sabse aakhir, sabse mehenga) — template-driven se reactive ek architecture change hai: validation, typing, dynamic controls, testing sab badalta hai. Feature-by-feature, naye forms hamesha reactive, purane tabhi jab us feature pe kaam ho raha ho. Governance: har step ke baad lint rule add karo (ratchet — kabhi peeche nahi). Naya ngClass/*ngIf CI mein fail ho. Reporting: har phase ka before/after number. Aur ek baat imaandaari se bolo: 4-saal purani app ka har hissa migrate karne ki zaroorat nahi — jo code stable hai aur badal nahi raha, use chhod do; effort wahan lagao jahan development active hai ya performance dikh rahi hai.

3. What problem does it solve?

  • Template as value → ngTemplateOutlet (render props, Ch5 S8).
  • Component as value → ngComponentOutlet (dynamic components, plugin systems).

4. Internal Working — NgIf poora

// Simplified NgIf — 40 lines mein poora concept
@Directive({ selector: '[ngIf]' })
export class NgIf<T> {
  private vcr = inject(ViewContainerRef);              // L1: anchor
  private thenTpl = inject<TemplateRef<NgIfContext<T>>>(TemplateRef);  // L2
  private elseTpl: TemplateRef<NgIfContext<T>> | null = null;
  private thenView: EmbeddedViewRef<any> | null = null;
  private context = new NgIfContext<T>();               // L3: {$implicit, ngIf}

  @Input() set ngIf(condition: T) {
    this.context.$implicit = condition;
    this.context.ngIf = condition;
    this.updateView();
  }
  @Input() set ngIfElse(tpl: TemplateRef<any>) { this.elseTpl = tpl; this.updateView(); }

  private updateView() {
    if (this.context.$implicit) {
      if (!this.thenView) {                             // L4: idempotent!
        this.vcr.clear();
        this.thenView = this.vcr.createEmbeddedView(this.thenTpl, this.context);
      }
    } else { /* else branch ya clear */ }
  }

  // TYPE NARROWING KA JAADU:
  static ngTemplateGuard_ngIf: 'binding';                       // L5
  static ngTemplateContextGuard<T>(dir: NgIf<T>, ctx: any):
    ctx is NgIfContext<Exclude<T, false | 0 | '' | null | undefined>> {  // L6
    return true;
  }
}

6. Architecture — NgFor ka IterableDiffer

  *ngFor="let item of items; trackBy: trackFn"

  ngDoCheck() {
    const changes = this.differ.diff(this.items);      // IterableDiffer
    if (changes) this.applyChanges(changes);
  }

  IterableDiffer.diff() ke andar:
    - har item ke liye ek IterableChangeRecord object
    - linked lists: _additionsHead, _movesHead, _removalsHead, _identityChangesHead
    - purani collection se compare, records ko in lists mein daalo

  applyChanges():
    changes.forEachOperation((record, prevIdx, currIdx) => {
       record.previousIndex == null  --> vcr.createEmbeddedView(...)
       currentIndex == null          --> vcr.remove(prevIdx)
       else                          --> vcr.move(view, currIdx)
    })

  ► Do abstraction layers: differ (records) + directive (VCR operations)
  ► @for ka δδrepeater LView-aware hai — koi record objects, direct operations
  ► trackBy OPTIONAL tha (default: reference identity) --> derived arrays pe
    chupchaap sab recreate. @for ne track MANDATORY kar diya (Ch5 S10).

10. Code Explanation — Line by Line

  • L1: Surprise: [ngSwitch] ek attribute directive hai (koi * nahi). Wo sirf current value hold karta hai aur cases ko coordinate karta hai.
  • L2–L3: Har *ngSwitchCase ek alag structural directive instance hai. Wo apne aap decide karta hai render karna hai ya nahi. Isliye do identical cases dono render ho jaate hain — kisi ne "pehla match, phir stop" enforce nahi kiya. C ke switch ka fallthrough nahi, par uniqueness bhi nahi.
  • L5: NgSwitchCase inject(NgSwitch, {host: true}) se parent ko dhundhta hai — DI ke through. Isliye cases nested elements mein bhi ho sakte hain (jab tak host chain mein NgSwitch mile). Ye flexibility hai, par ye implicit coupling bhi hai jo debug karna mushkil banata hai.

Aur type narrowing free milti hai (discriminated unions, Ch2 S2). Production rules

13. Performance

*ngFor per use: instance + IterableDiffer + har CD cycle ngDoCheck + per-item IterableChangeRecord objects. @for: δδrepeater ka keyed algorithm, LView-aware, koi intermediate records.

Aur bundle: CommonModule ke saath NgIf, NgForOf, NgSwitch, differs infrastructure — sab aata hai. Blocks compiler instructions hain, already runtime mein.

Render-props pattern (customizable list/table rows with per-item context, Ch5 S8) sirf ngTemplateOutlet se possible hai. Same logic ngComponentOutlet ke liye — dynamic components.

HARD NgSwitch ka architecture explain karo — aur @switch se uska farak.

[ngSwitch] ek attribute directive hai (structural nahi!) jo current value hold karta hai aur ek internal match-counter rakhta hai. *ngSwitchCase aur *ngSwitchDefault structural directives hain jo inject(NgSwitch, {host: true}) se parent ko DI ke through dhundhti hain. Har case apne aap apni value ko NgSwitch ki value se === compare karta hai aur khud decide karta hai view create karni hai ya nahi; NgSwitch match-count badhata hai taaki NgSwitchDefault ko pata chale koi match hua ya nahi. Consequences: (1) do identical cases → dono render honge (koi uniqueness check nahi, koi "pehla match" semantics nahi); (2) cases ko direct child hona zaroori nahi (DI host chain se jud-te hain) — flexible par implicit; (3) har case = ek directive instance + injector entry. @switch mein compiler matching branch ka index compute karke ek δδconditional(idx) emit karta hai: ek hi view, pehla match, koi directive instance, koi DI coordination. Yani ek runtime coordination protocol (DI + counting + per-case decisions) ko ek compile-time index computation se replace kar diya gaya — bilkul wahi pattern jo ngClass→classProp aur IterableDiffer→δδrepeater mein hai.

(3) Contract: har widget ek interface implement kare (WidgetConfig input, configChange output) — ya better, ek abstract class DI token (Ch2 S5) taaki type safety mile. ngComponentOutletInjector se per-widget scoped services de sakte ho (Ch3 S9). S-04

Custom Attribute Directives

Covers: Selector design · Inputs & alias · exportAs · Signals-era patterns · DI ke saath · Testing · Kab banayein

1. What is it?

Custom attribute directive = tumhari apni class jo kisi existing element ka behavior ya appearance badalti hai. Koi template nahi, koi wrapper nahi — sirf ek attribute lagao aur element "smart" ho jaata hai.

<button appRipple>Click</button>
<input appAutofocus />
<a appExternalLink href="https://x.com">Link</a>
<div [appTooltip]="'Help text'">?</div>

2. Why do we need it?

Cross-cutting behaviors ke liye. Ek app mein 200 buttons hain, un sab pe analytics tracking chahiye. Component banaoge toh har button ko wrap karna padega (native semantics gayi, Ch4 S2). Base class se inherit karoge toh single-inheritance ki jail (S9). Service call har jagah likhoge toh 200 jagah duplicate.

Directive: <button appTrack="pay_click">. Ek attribute, behavior attached, element wahi ka wahi.

3. What problem does it solve?

  • Behavior duplication → ek jagah likho, har jagah lagao.
  • Wrapper components → native element intact.
  • Inheritance rigidity → kai behaviors ek saath (S9).
  • Template clutter → declarative attribute, imperative code chhupa hua.

@Directive({

10. Code Explanation — Line by Line

  • L1: exportAs ke bina #dd se element milta (Ch5 S7 rule #3). exportAs se directive instance milta hai. Yahi ngModel, ngForm, matMenuTrigger ka pattern hai.
  • L2: asReadonly() — bahar wale sirf padh sakte hain, mutations sirf public methods se. Ye Ch2 S5 ka encapsulation, signals mein. Bina iske consumer dd.isOpen.set(true) karke internal invariants tod sakta hai.
  • L3: Public API deliberate hai: toggle, open, close. Ye directive ka contract hai — documented, tested, semver ke saath.
  • L4–L6: Template ab directive ki state padh sakta hai aur uske methods call kar sakta hai — bina component class mein kuch likhe. Ye ek chhota, self-contained widget banata hai.
  • Design nuance: @if (dd.isOpen()) template mein hai, toh menu ka markup consumer ke paas hai — flexible. Agar directive khud menu render karti toh wo component ban jaati. Yahi directive vs component ki line hai: directive behavior deti hai, markup nahi. (Aur agar markup + behavior dono chahiye, toh component + hostDirectives, S9.)

11. Common Mistakes

Galtiyan
  • imports mein add nahi kiya — silent failure (attribute ignore ho jaata hai, koi error nahi). Directives ka #1 bug.
  • Prefix na lagana ([tooltip]) — library collision, aur error message confusing.
  • Directive mein DOM ko seedha chhoona (nativeElement.style) — SSR crash (S6). Host bindings ya Renderer2.
  • Bhaari logic directive mein — har instance mein duplicate. Service mein rakho, directive patli.
  • Mouse-only interactions (mouseenter bina focus) — keyboard users ke liye feature dead.
  • Writable signal public expose karna — asReadonly() use karo.
  • Generic selector ([appRipple]) jab compound se contract enforce ho sakta tha (button[appRipple]).

12. Best Practices

Production rules
  • Selector: prefixed camelCase attribute. Compound (button[appX]) se allowed hosts restrict karo.
  • Input alias = directive naam (idiomatic), ya appXSomething for extras.
  • State signals mein; public read-only (asReadonly()), mutations methods se.
  • DOM kaam host bindings se; bache toh Renderer2 + afterNextRender (S6/S7).
  • exportAs tab jab template ko instance chahiye.
  • A11y: keyboard parity (focus/blur ke saath mouseenter/mouseleave), ARIA attributes host mein.
  • Logic services mein, directive ek patla adapter.

13. Performance

Har directive instance ka cost (S1): ek object, ek LView slot, ek ElementInjector entry (agar DI use kare), aur hostVars ke barabar binding slots + comparisons per CD cycle.

Practical rules: (1) directive ko element ke level pe rakho jahan wo zaroori hai — 20,000 cells pe nahi, 1000 rows pe (ya 1 grid pe, delegation se). (2) Host bindings kam rakho (har ek ek comparison per cycle). (3) High-frequency host listeners (mousemove) zone apps mein CD trigger karte hain — throttle ya zoneless (Ch5 S5). (4) Directive ke andar bhaari computed() theek hai (memoized), par har instance ka apna hoga — shared derivation service mein rakho.

14. Edge Cases

  • Static attribute vs binding: appTooltip="Help" (string) vs [appTooltip]="expr". input() dono handle karta hai; boolean ke liye transform: booleanAttribute (Ch4 S5).
  • Directive ka input naam native property se clash (value) — binding directive ko jaayegi, native property ko nahi (Ch5 S2 ka shadowing). Prefix karo.
  • Directive component ke host pe bhi lag sakti hai (<app-card appTooltip="x">) — dono ek ElementInjector share karenge.
  • @Directive() bina selector ke = abstract base directive (inheritance ke liye; par S9 dekho).
  • Directive hostDirectives bhi laga sakti hai (S9) — nested composition.
  • Testing: directive ko test karne ke liye ek host test component banao (@Component({template: '<div appX>'})), ya TestBed.createComponent + viewChild.

HARD Directive silent fail kyun karti hai jab component "not a known element" error deta hai?

Practical impact: "meri directive chal hi nahi rahi" debug karte waqt pehla check imports array hai, aur DevTools mein ng.getDirectives($0) se confirm karo ki instance bana ya nahi. Bachne ke tareeke: (1) directive ko ek shared "UI kit" barrel se export karo aur ek lint rule likho; (2) compound selectors (button[appRipple]) se galat host pe lagne ka risk kam karo; (3) directive ka koi observable effect ho (class, aria attribute) taaki missing directive turant dikhe; (4) integration test jo assert kare ki directive instance maujood hai. Ye ek design trade-off hai — Angular ne HTML ki extensibility preserve ki, par debuggability ki keemat pe.

S-05

Custom Structural Directives

Covers: Microsyntax ka poora grammar · TemplateRef + ViewContainerRef · Idempotent updates · Template guards & narrowing · Kab likhna chahiye (aur kab nahi)

1. What is it?

Structural directive = ek directive jo DOM ka shape badalti hai — views create/destroy/move karke. Uske paas do superpowers hain jo attribute directives ke paas nahi:

  • TemplateRef — us content ka blueprint jispe wo lagi hai (Ch5 S8)
  • ViewContainerRef — ek anchor jahan views insert ho sakti hain

Aur ek syntax gift: * prefix, jo element ko automatically <ng-template> mein wrap kar deta hai.

2. Why do we need it?

Imaandaar jawab: aaj kal aksar zaroorat nahi hai. @if/@for/@switch (Ch5 S10) ne 90% use cases cover kar liye, aur wo directives se tez hain (koi instance, koi injector). Custom structural directive tab likho jab:

  • Ek reusable, named abstraction chahiye (*appIfRole="'admin'") jo poore app mein consistent ho
  • Ya ek genuinely naya rendering pattern (repeat with delay, virtual windowing, permission + fallback)

Par unka internals samajhna zaroori hai — kyunki Angular ke apne NgIf/NgFor yahi the (S3), aur ViewContainerRef ka gyaan dynamic UIs mein har jagah kaam aata hai.

3. What problem does it solve?

  • Repeated conditional logic → ek named directive.
  • Custom rendering patterns → views ko manually control karo.
  • Type narrowing in templates → template guards.
  • Dynamic view management → ViewContainerRef ki poori API.

4. Internal Working — microsyntax ka grammar

  *appRepeat="count; let i = index; when: cond; as ctx"
   |         |      |               |          |
   |         |      |               |          +--> let-ctx="appRepeat"  (as = alias for main input)
   |         |      |               +-------------> [appRepeatWhen]="cond"
   |         |      +-----------------------------> let-i="index"
   |         +------------------------------------> [appRepeat]="count"   (main input = directive naam)
   +----------------------------------------------> element ko ng-template mein wrap karo

  DESUGARED:
    <ng-template appRepeat [appRepeat]="count" [appRepeatWhen]="cond"
                 let-i="index" let-ctx="appRepeat">
      <div>...</div>
    </ng-template>

  RULES:
    - Pehla expression --> input named after the directive
    - `keyword: expr`  --> input named `directiveName + Capitalize(keyword)`
    - `let x = key`    --> context binding let-x="key"
    - `let x`          --> let-x (i.e. $implicit)
    - `as x`           --> alias for the main input's value
    - Separators: `;` ya `,`

// Isliye *ngFor="let item of items; trackBy: fn" desugars to:
<ng-template ngFor let-item [ngForOf]="items" [ngForTrackBy]="fn">
//              ^^^^^ directive selector    ^^^^^^^ "of" --> ngForOf

// Views banao

6. Architecture — ek sahi structural directive

@Directive({ selector: '[appIfRole]' })
export class IfRoleDirective {
  private tpl = inject<TemplateRef<IfRoleContext>>(TemplateRef);
  private vcr = inject(ViewContainerRef);
  private auth = inject(AuthService);

  role = input.required<Role>({ alias: 'appIfRole' });
  elseTpl = input<TemplateRef<void> | null>(null, { alias: 'appIfRoleElse' });

  private shown = signal<'then' | 'else' | 'none'>('none');      // L1: current state

  constructor() {
    effect(() => {
      const allowed = this.auth.hasRole(this.role());
      const want = allowed ? 'then' : (this.elseTpl() ? 'else' : 'none');

      if (want === this.shown()) return;                        // L2: IDEMPOTENT!

      this.vcr.clear();
      if (want === 'then') this.vcr.createEmbeddedView(this.tpl, { $implicit: this.role() });
      if (want === 'else') this.vcr.createEmbeddedView(this.elseTpl()!);
      this.shown.set(want);
    });
  }

  // Type narrowing ke liye (S3 ka mechanism)
  static ngTemplateContextGuard(dir: IfRoleDirective, ctx: unknown):
    ctx is { $implicit: Role } { return true; }               // L3
}

// Use:
<button *appIfRole="'admin'; else noAccess">Delete</button>
<ng-template #noAccess><span class="muted">No access</span></ng-template>

constructor() {
  effect(() => {
    const allowed = this.auth.hasRole(this.role());   // L1: dependency

//   - hasRole() ka koi bhi dependency badla (user object ka naya reference,

//   - Andar ka component: constructor dobara, ngOnInit dobara, HTTP dobara
//   - Form input ki value gayi, focus gaya, animation restart

10. Code Explanation — Line by Line

  • L1: effect apni saari signal dependencies track karta hai — role() aur auth.hasRole() ke andar ke saare signals. Koi bhi badla, effect dobara chalega.
  • L2–L3: Blind clear() + createEmbeddedView(). Result same dikhta hai (view maujood hai), par wo naya view hai. Andar ke component ka poora lifecycle dobara chala.
  • L4: Guard: current rendered state ko track karo aur sirf transition pe kaam karo. Yahi NgIf ka updateView() karta hai (S3 ka L4).
  • Deep point: Ye ek general principle hai — DOM/view operations idempotent hone chahiye. Wahi principle jo React ke reconciler mein hai, jo @for ke keep/move/create/destroy mein hai (Ch5 S10), aur jo δδconditional mein hai (pichla branch index yaad rakhta hai). "Desired state" compute karo, "current state" se compare karo, sirf diff apply karo. Jo bhi manually views manage karta hai, use ye khud implement karna padta hai — aur bhoolne pe bug silent hota hai.
  • Testing hook: is bug ko pakadne ka tareeka — andar ke component ke constructor mein console.count, phir koi unrelated signal badlo. Count badha? Bug hai.

11. Common Mistakes

Galtiyan
  • Non-idempotent updates — har change pe clear() + create. Silent state loss.
  • Structural directive likhna jahan @if kaafi tha — extra instance, injector, aur worse narrowing.
  • ngTemplateContextGuard bhoolna — let-i ka type any (Ch5 S7).
  • Ek element pe do structural directives — compile error (* do baar wrap nahi kar sakta).
  • Microsyntax ke input naming rules galat samajhna — when: x → [appRepeatWhen], na ki [when].
  • Permission directives ko security samajhna — DOM se hataana authorization nahi hai.
  • ViewContainerRef se banayi views ka cleanup bhoolna (imperative usage mein).

12. Best Practices

Production rules
  • Pehle poochho: @if/@for se ho sakta hai? Aksar haan. Directive tabhi jab reuse + naming ka asli value ho.
  • Updates idempotent — desired vs current state compare karo, sirf diff apply.
  • ngTemplateContextGuard hamesha (typed let- variables).
  • Narrowing chahiye toh ngTemplateGuard_<input>: 'binding' (S3).
  • else template support do ([appXElse] input) — binary hide/show aksar kaafi nahi.
  • Logic service mein; directive sirf view management kare.
  • Security: authorization server pe. Directive sirf UI convenience.

13. Performance

Structural directive per use: instance + ElementInjector entry (TemplateRef + ViewContainerRef inject karti hai) + input setter chain + (agar ngDoCheck ho) per-cycle hook. @if mein ye sab nahi — ek δδconditional instruction (Ch5 S8).

Asli cost view creation ka hai: har createEmbeddedView = ek LView array + DOM nodes + CD tree node (Ch3 S6). Isliye:

  • Idempotency critical hai — unnecessary destroy/create sabse bada waste.
  • Reorder ke liye move()/detach()+insert(), na ki remove+create.
  • Context ko mutate karo (view.context.index = 5) + markForCheck(), view recreate mat karo.
  • 1000 uses = 1000 instances; bade lists mein blocks use karo.

14. Edge Cases

  • * syntax ke bina bhi structural directive use ho sakti hai: <ng-template [appIfRole]="'admin'"> — explicit form.
  • Microsyntax mein as: *appIfRole="role as r" → let-r="appIfRole" (main input ka alias, context se).
  • TemplateRef inject nahi hoga agar directive * ke bina (plain attribute ki tarah) lagi ho — runtime error.
  • Structural directive ng-container pe bhi lag sakti hai (kyunki wo bhi ng-template mein wrap ho jaata hai).
  • EmbeddedViewRef.rootNodes se DOM nodes mil jaate hain (rare, DOM measurement ke liye).
  • Views ka detectChanges() manually call karna pad sakta hai agar tumne detach() kiya ho (CD tree se hat gayi).
  • Signals-era mein effect() se view manage karna common hai; par effect mein DOM writes karna glitch de sakta hai — afterRenderEffect() socho agar timing matter kare.

15. Interview Questions

EASY Structural directive kaunse do tokens inject karti hai?

TemplateRef — us content ka blueprint jispe wo lagi hai (* ne use ng-template mein wrap kar diya). Aur ViewContainerRef — ek anchor jahan views insert/remove/move hoti hain. Poora kaam: condition ke hisaab se vcr.createEmbeddedView(tpl) ya vcr.clear().

EASY *appX="a; b: c; let d = e" desugar karke dikhao.

<ng-template appX [appX]="a" [appXB]="c" let-d="e">...</ng-template>. Rules: pehla expression → directive naam wala input; keyword: expr → directiveName + Capitalize(keyword) input; let x = key → context binding. Isliye *ngFor="let i of items" mein of → [ngForOf].

MEDIUM Structural directive mein idempotency kyun zaroori hai?

Kyunki update logic har input change / effect run pe chalega — aur agar tum blindly clear() + createEmbeddedView() karoge, toh view destroy + recreate hogi chahe condition badli ho ya nahi. Andar ke component ka constructor, ngOnInit, HTTP calls dobara chalenge; form input ki value, focus, aur animations reset ho jaayenge. Aur ye silent hota hai. Fix: current rendered state track karo, desired state se compare karo, sirf transition pe kaam karo — bilkul jaise NgIf aur δδconditional karte hain.

MEDIUM Aaj custom structural directive kab likhni chahiye?

Kam. @if/@for/@switch 90% cases cover karte hain, aur wo tez hain (koi directive instance/injector) aur better narrowing dete hain. Likho jab: (1) ek domain concept ko naam dena hai jo 50+ jagah repeat hota hai (*appIfFeature) aur consistency ka value hai; (2) ek genuinely naya rendering pattern hai jo blocks se possible nahi (custom windowing, delayed repeat, portal-based). Aur likho toh idempotent updates + ngTemplateContextGuard ke saath.

HARD ViewContainerRef ki API se @for ka keep/move/create/destroy kaise implement hota hai?

ViewContainerRef deta hai: createEmbeddedView(tpl, ctx, {index}), get(i), move(viewRef, newIndex), detach(i), insert(viewRef, i), remove(i), length. Reconciliation isse aise banti hai: (1) purani views ka ek map banao trackKey → ViewRef; (2) nayi collection traverse karo — key mila map mein? → keep (view reuse; uska context mutate karo naye item/index se, aur markForCheck() — view recreate mat karo!) aur agar position galat hai toh move(); key nahi mila? → create at index; (3) jo purani keys nayi collection mein nahi → destroy (remove()). Critical insight: context object ko mutate karna hi wo cheez hai jisse $index/$last update hote hain bina view recreate kiye — aur isliye component instance zinda rehta hai aur uski state item ke saath travel karti hai (Ch5 S10 ka correctness argument). Angular ka δδrepeater yahi karta hai, par LView-level pe (VCR abstraction ke bina, isliye tez); *ngFor yahi karta tha VCR + IterableDiffer ke through (S3). Aur detach+insert ka combo ek view ko CD tree se temporarily hata ke wapas daal sakta hai — virtual scrolling implementations isi se view pooling karti hain.

SCENARIO Ek *appVirtualFor banani hai jo 100,000 items ki list mein sirf visible ~30 rows render kare, scroll pe views recycle kare. Design karo — aur batao kya tum ye khud likhoge.

Pehle imaandaari: ye mat likho — @angular/cdk/scrolling (cdk-virtual-scroll-viewport) battle-tested hai, aur virtual scrolling ke edge cases (variable heights, sticky headers, keyboard nav, a11y, resize, browser scroll anchoring) mein saal lagte hain. Interview mein ye judgement khud ek acchha jawab hai. Par design poochha hai, toh mechanism batao: (1) Measurement: viewport height + item height (fixed pehle version mein) → visibleCount = ceil(vh / ih) + buffer. Scroll offset se startIndex = floor(scrollTop / ih). (2) Spacer strategy: total height ko fake karne ke liye ek spacer element (height = total * ih) ya transform-based offset. Warna scrollbar galat hoga. (3) View recycling — core: ViewContainerRef mein sirf visibleCount views rakho. Scroll pe: jo views window se bahar gayi unhe detach() karke ek pool mein rakho (destroy mat karo!); nayi rows ke liye pool se view lo, uska context mutate karo ($implicit = newItem; index = newIndex), markForCheck(), aur insert() sahi position pe. Ye createEmbeddedView se 10x sasta hai (koi LView allocation, koi DOM creation). Yahi CDK karta hai. (4) Scroll listener: runOutsideAngular + rAF throttle (ya zoneless, Ch5 S5) — warna har scroll event pe poori CD. Position update ko ek signal mein likho. (5) Context guard: ngTemplateContextGuard se let-item typed. (6) A11y: aria-rowcount = total (100,000), aria-rowindex = absolute index — warna screen reader "30 rows" bolega (Ch5 S3). Keyboard nav ko scrollIntoView ke saath sync karna padega. (7) Edge cases jo tumhe maarenge: variable item heights (measurement cache + estimation), fast scroll (blank rows — buffer + rAF), browser scroll anchoring, resize observer, aur nested scroll containers. (8) Testing: 100k items ke saath scroll performance (frame timing), aur ek test ki view count constant rehta hai. Punchline: mechanism samajhna zaroori hai (view pooling + context mutation), par production mein CDK use karo — aur agar custom likhna hi pade toh CDK ke VirtualScrollStrategy interface se extend karo, poora scroll engine dobara mat likho.

S-06

ElementRef & DOM Access

Covers: ElementRef ka contract · nativeElement ke teen khatre · SSR · XSS · Timing · Kab justified hai · DOCUMENT token

1. What is it?

ElementRef = ek patla wrapper jiske andar nativeElement property hai — us DOM element ka direct reference jispe tumhari directive/component baithi hai.

class ElementRef<T = any> {
  nativeElement: T;      // bas itna hi. Ek property.
}

const el = inject<ElementRef<HTMLInputElement>>(ElementRef);
el.nativeElement.focus();

Angular ke source mein iske upar ek comment hai jo effectively kehta hai: "Iska use security risk hai. Renderer2 ya template APIs use karo." Ye chapter samjhaega ki wo warning literal kyun hai — aur wo 5% cases kaunse hain jahan ye justified hai.

2. Why do we need it?

Kyunki kuch cheezein declarative APIs se possible hi nahi: input.focus(), element.scrollIntoView(), canvas.getContext('2d'), video.play(), getBoundingClientRect(), ya ek third-party library ko ek DOM node dena (chart.js, map libs). In sab ke liye asli DOM node chahiye.

Angular ka stance: ye escape hatch hai, default nahi. Aur uske teen thos reasons hain.

3. What problem does it solve?

  • Imperative browser APIs → focus, scroll, measure, media control.
  • Third-party libs → jo DOM node maangti hain.
  • Measurement → layout reads jo bindings se possible nahi.

4. Internal Working — teen khatre

  KHATRA 1 — PLATFORM (SSR)
  ────────────────────────────
    Server pe DOM nahi hai. Angular server pe ek DOM-like structure banata hai
    (@angular/platform-server), par wo browser ka HTMLElement nahi hai.
      el.nativeElement.focus()      --> server pe crash ya undefined
      el.nativeElement.style.x = y  --> kaam nahi karega
    Aur Web Workers mein toh DOM hai hi nahi.

  KHATRA 2 — SECURITY (XSS)
  ────────────────────────────
    el.nativeElement.innerHTML = userContent;
       --> Angular ka sanitizer BYPASS ho gaya (Ch5 S2)
       --> <img src=x onerror="steal()"> chal jaayega
    Template bindings sanitize karte hain; nativeElement nahi.

  KHATRA 3 — CHANGE DETECTION / TIMING
  ────────────────────────────
    constructor mein: element bana hai par uske children nahi (creation pass chal raha)
    ngOnInit mein:    view abhi render nahi hui
    CD ke beech DOM padhna: layout thrashing + ExpressionChanged errors (Ch3 S8)
    Aur Angular ko nahi pata ki tumne DOM badla — uske bindings overwrite kar denge.

          const h = node.scrollHeight;                            // L4: ek READ

      });

10. Code Explanation — Line by Line

  • L1: input event har keystroke pe — zone ke andar hota toh har keystroke pe poori CD (Ch5 S5). Zoneless apps mein ye line nahi chahiye.
  • L2: requestAnimationFrame — DOM reads/writes ko frame boundary pe le jaao. Isse browser ko batch karne ka mauka milta hai.
  • L3–L5: Layout thrashing ka core: browser scrollHeight jaisi geometry padhne ke liye pending style changes ko flush karke layout compute karta hai. Agar tum read → write → read → write karte ho, toh har read ek forced synchronous layout trigger karta hai (DevTools mein "Forced reflow" warning). Solution: saare reads pehle, saare writes baad mein — ya jaise yahan, ek reset-write, ek read, ek final write, sab ek rAF callback mein.
  • Aur Renderer2 kyun (L3/L5): node.style.height = ... bhi kaam karta, par renderer.setStyle platform-safe hai aur Angular ke rendering pipeline se guzarta hai (S7). Yahan hum already afterNextRender-jaisi browser-only jagah mein hain, toh dono theek hain — par habit consistent rakhna behtar hai.
  • Behtar alternative: modern CSS mein field-sizing: content (ya textarea ko grid trick se auto-grow) — zero JavaScript. Ye poora directive delete ho sakta hai. Hamesha poochho: kya ye CSS se ho sakta hai? (Ch5 S3 ka theme.)

11. Common Mistakes

Galtiyan
  • nativeElement.innerHTML = userContent — sanitizer bypass, direct XSS (Ch5 S2).
  • Constructor/ngOnInit mein DOM access — element ready nahi, SSR pe crash.
  • document.querySelector — global, SSR pe crash, aur component scope ke bahar (stale refs, Ch1 S8 leak).
  • nativeElement.classList.add() jab Angular bhi wahi class bind kar raha hai — Angular ka binding overwrite kar dega (uska styling context tumhari manual class ko nahi jaanta, Ch5 S4).
  • Read → write → read loops (layout thrashing) — forced reflows, jank.
  • isPlatformBrowser se DOM code wrap karna jab afterNextRender cleaner tha.
  • Third-party lib instance ka cleanup bhoolna — DestroyRef.onDestroy(() => chart.destroy()).

12. Best Practices

Production rules
  • DOM access ki ladder follow karo: template binding > host binding > Renderer2 > ElementRef.
  • ElementRef ka use hamesha afterNextRender/afterRenderEffect ke andar (SSR-safe by construction).
  • ElementRef<HTMLInputElement> — generic do, any mat chhodo.
  • Global document/window ke bajaye DOCUMENT token (aur inject(WINDOW) custom token).
  • innerHTML kabhi nahi — DomSanitizer ke saath bhi soch ke (Ch5 S1/S2).
  • Measurement: reads batch karo, writes batch karo, rAF mein. Aur pehle poochho: CSS se ho sakta hai?
  • Third-party libs ka lifecycle: create in afterNextRender, destroy in DestroyRef.

13. Performance

Do bade issues. (1) Layout thrashing: geometry properties (offsetHeight, scrollTop, getBoundingClientRect(), getComputedStyle()) padhna browser ko pending styles flush karne pe majboor karta hai. Read-write-read pattern mein har read ek forced synchronous layout hai — ek loop mein ye O(n) reflows deta hai. Fix: reads pehle, writes baad mein, ya ResizeObserver/IntersectionObserver (jo browser ke apne timing pe chalte hain).

(2) CD ke saath conflict: manually lagayi class/style Angular ke styling context mein nahi hai (Ch5 S4). Agla CD cycle jab classMap ya styleMap apni value apply karega, wo tumhari manual change ko silently mita sakta hai. Aur ye intermittent bug hai — sirf tab dikhta hai jab wo binding update ho.

Best move aksar ye hai ki DOM chhuo hi mat: state ko signal mein rakho, aur DOM ko binding se update karwao.

14. Edge Cases

  • ElementRef ko viewChild(..., {read: ElementRef}) se bhi le sakte ho (Ch4 S7) — jab component/directive instance ke bajaye element chahiye.
  • ElementRef component ke host element ko point karta hai, uske template ke andar ke elements ko nahi.
  • SSR mein afterNextRender ka callback register hi nahi hota — koi if (isBrowser) nahi chahiye.
  • Hydration (Ch3 S8) ke saath: server-rendered DOM reuse hota hai, toh afterNextRender mein element wahi hai jo server ne bheja tha.

15. Interview Questions

EASY ElementRef kya hai aur uska use kyun discourage kiya jaata hai?

Ek wrapper jiske andar nativeElement — asli DOM node. Discourage isliye: (1) SSR/Web Worker pe DOM nahi hai, code crash karega; (2) innerHTML se sanitizer bypass = XSS; (3) manual DOM changes Angular ke bindings se conflict karti hain (styling context, Ch5 S4) aur CD ke saath timing issues deti hain. Alternatives: template/host bindings, Renderer2, aur imperative APIs ke liye ElementRef + afterNextRender.

EASY Input pe focus kaise karoge SSR-safe tareeke se?

inject(ElementRef<HTMLInputElement>) aur afterNextRender(() => el.nativeElement.focus()). afterNextRender server pe register hi nahi hota, aur browser mein paint ke baad chalta hai — element definitely DOM mein hai. Constructor/ngOnInit mein focus karne se element ready nahi hoga.

MEDIUM nativeElement.classList.add('x') se kya problem ho sakti hai?

Angular ka styling system (Ch5 S4) ek internal context maintain karta hai jisme har source (template, directive host, component host) ki class values hoti hain. Tumhari manual class us context mein nahi hai. Jab koi [class] map binding update hogi, wo apne diff ke hisaab se classes set karegi aur tumhari manual class ko silently mita sakti hai. Aur ye intermittent hai — sirf us binding ke update pe. Sahi tareeka: class ko state se derive karo ([class.x]="signal()") ya Renderer2.addClass use karo aur uss class ko kisi binding se mat chhuo.

MEDIUM Layout thrashing kya hai aur Angular mein usse kaise bachenge?

Geometry properties (offsetHeight, getBoundingClientRect, scrollHeight) padhne pe browser pending style changes flush karke layout compute karta hai. Agar tum read → write → read → write karte ho, har read ek forced synchronous layout trigger karta hai — loop mein ye jank deta hai. Bachne ke liye: saare reads pehle, saare writes baad mein; requestAnimationFrame mein batch karo; ya ResizeObserver/IntersectionObserver use karo. Angular ka afterRenderEffect phases (earlyRead, write, read) deta hai jo exactly isi ordering ko enforce karte hain.

HARD afterNextRender aur ngAfterViewInit mein kya farak hai — aur SSR ke context mein kaunsa sahi hai?

ngAfterViewInit ek lifecycle hook hai jo component ke view creation ke baad chalta hai — server pe bhi chalta hai, aur browser mein paint se pehle, change detection cycle ke andar. Toh: (1) SSR pe wahan DOM APIs (focus, getBoundingClientRect) crash karengi — isPlatformBrowser guard chahiye; (2) CD cycle ke andar hone ki wajah se state badalne pe ExpressionChangedAfterItHasBeenChecked (Ch3 S8); (3) paint se pehle hone ki wajah se layout measurements stale/incomplete ho sakti hain. afterNextRender browser-only hai — uska callback server pe register hi nahi hota, toh koi platform check nahi chahiye. Wo CD cycle ke bahar, paint ke baad chalta hai, aur uske phases (earlyRead → write → mixedReadWrite → read) DOM reads/writes ko batch karke layout thrashing rokte hain. afterRenderEffect uska reactive version hai (signals track karta hai). Practical rule: koi bhi browser-only DOM interaction → afterNextRender/afterRenderEffect. ngAfterViewInit ab sirf legacy code aur non-DOM view-related setup ke liye. Ye Angular ke SSR-first design ka hissa hai: API aisi banao ki galat cheez likhna mushkil ho, guard lagana yaad na rakhna pade.

SCENARIO Ek appTooltip directive banani hai jo hover pe tooltip dikhaye, viewport ke bahar na jaaye, SSR-safe ho, aur 500 elements pe use ho. DOM access ka design karo.

(1) Rendering ka decision — sabse pehla: tooltip ko host ke andar mat banao (overflow/z-index/stacking context problems). Use ek overlay container (document.body) mein render karo — CDK Overlay, ya ek chhota portal service jo ViewContainerRef.createEmbeddedView + appendChild kare. Iska matlab: DOM access unavoidable hai, par wo ek service mein centralized hai, 500 directives mein nahi. (2) SSR: tooltip sirf interaction pe banta hai, aur interaction browser mein hi hoti hai — toh creation code naturally browser-only hai. Par listeners host bindings se lagao (host: {'(mouseenter)': ..., '(focus)': ...}) — wo SSR pe register hi nahi hote. Positioning code afterNextRender mein. Global document ki jagah DOCUMENT token. (3) Positioning + layout thrashing: getBoundingClientRect() host ka, phir tooltip ka — dono reads pehle, phir ek write (transform). Loop mein mat karo. Better: CDK ka ConnectedPositionStrategy (flexible positions + fallbacks, viewport collision handled). Khud likhoge toh afterRenderEffect ke read/write phases use karo. (4) Position ke liye transform, na ki top/left — transform composite layer pe chalta hai, layout trigger nahi karta. (5) 500 instances ka design: (a) tooltip ka DOM lazy — sirf open pe banao, band pe destroy; 500 hidden tooltips mat banao. (b) Ek shared overlay instance reuse karo — ek waqt mein ek hi tooltip dikhta hai, toh ek hi DOM node kaafi hai. Ye sabse bada win hai. (c) Har directive mein sirf 2-3 host bindings. (6) A11y: [attr.aria-describedby] host pe (open hone pe tooltip ka id), tooltip pe role="tooltip". Aur focus/blur listeners — mouse-only tooltip a11y fail hai. Punchline: DOM access ko ek service mein centralize karo aur directives ko patla rakho; 500 directives = 500 chhote adapters, ek DOM engine.

1. What is it?

Renderer2 = DOM operations ke liye ek platform abstraction. element.style.color = 'red' ki jagah renderer.setStyle(element, 'color', 'red').

Sawaal: itni ceremony kyun? Jawab: kyunki Angular ka rendering layer pluggable hai. Tumhara code browser ke DOM ko directly nahi jaanta — wo ek Renderer2 interface ko jaanta hai, aur uska concrete implementation platform decide karta hai.

2. Why do we need it?

  Tumhara code
       |
       v  Renderer2 (interface)
       |
       +--> DomRenderer            (browser: document.createElement, etc.)
       +--> ServerRenderer         (SSR: ek DOM-like tree banata hai string ke liye)
       +--> AnimationRenderer      (@angular/animations ka wrapper)
       +--> EmulatedEncapsulationRenderer  (_ngcontent attributes lagata hai! Ch4 S4)
       +--> (Web Worker / NativeScript / custom renderers)

  ► Isliye tumhara ek hi directive browser aur server dono pe chalti hai.
  ► Isliye view encapsulation kaam karti hai — renderer hi attributes lagata hai.
  ► Isliye animations DOM operations ko intercept kar paati hain.

Aur ek chhupa hua faayda: hydration (Ch3 S8). Server aur client dono Renderer2 ke through DOM banate hain, toh Angular ko exactly pata hota hai kaunsa node kis instruction se aaya — isliye wo server-rendered DOM ko reuse kar sakta hai.

3. What problem does it solve?

  • Platform coupling → same code, alag renderers.
  • Encapsulation → renderer _ngcontent/_nghost attributes lagata hai.
  • Animations → renderer chain mein intercept.
  • Listener cleanup → listen() unlisten function deta hai.

7. Real Project Example — overlay service

@Injectable({ providedIn: 'root' })
export class OverlayService {
  private doc = inject(DOCUMENT);
  private renderer = inject(RendererFactory2).createRenderer(null, null);  // L1
  private appRef = inject(ApplicationRef);
  private injector = inject(EnvironmentInjector);

  open<C>(cmp: Type<C>, inputs?: Partial<C>): { close: () => void } {
    // 1. Host element banao — renderer se (SSR-safe, encapsulation-aware)
    const host = this.renderer.createElement('div');              // L2
    this.renderer.addClass(host, 'overlay-host');
    this.renderer.appendChild(this.doc.body, host);                // L3

    // 2. Component banao aur is host pe attach karo
    const ref = createComponent(cmp, {                              // L4
      environmentInjector: this.injector,
      hostElement: host,
    });
    Object.entries(inputs ?? {}).forEach(([k, v]) => ref.setInput(k, v));
    this.appRef.attachView(ref.hostView);                          // L5: CD tree mein daalo

    // 3. Escape listener — conditional, isliye renderer.listen
    const off = this.renderer.listen('document', 'keydown.escape', () => close());  // L6

    const close = () => {                                           // L7
      off();
      this.appRef.detachView(ref.hostView);
      ref.destroy();
      this.renderer.removeChild(this.doc.body, host);
    };
    return { close };
  }
}

9. Advanced Example — custom renderer (poore app ke DOM operations intercept karo)

// Use case: har DOM element pe automatically data-testid lagao (dev builds mein),
// ya har element creation ko log karo (debugging), ya SSR mein kuch skip karo.

export class TestIdRendererFactory implements RendererFactory2 {
  constructor(private delegate: RendererFactory2) {}          // L1: wrap karo

  createRenderer(host: any, type: RendererType2 | null): Renderer2 {
    const base = this.delegate.createRenderer(host, type);      // L2
    return new Proxy(base, {                                    // L3
      get(target, prop) {
        if (prop === 'createElement') {
          return (name: string, ns?: string) => {
            const el = target.createElement(name, ns);
            if (ngDevMode) target.setAttribute(el, 'data-created-by', 'angular');
            return el;
          };
        }
        return (target as any)[prop];
      },
    });
  }
  begin?() { this.delegate.begin?.(); }
  end?() { this.delegate.end?.(); }
}

10. Code Explanation — Line by Line

  • L1–L2: Decorator pattern — asli factory ko wrap karo, uska output modify karo. Angular ka apna AnimationRendererFactory exactly yahi karta hai: DOM renderer ko wrap karke animation-aware banata hai.
  • L3: Proxy se sirf ek method intercept kiya, baaki delegate ho gaye. Alternative: poora Renderer2 class implement karo (25+ methods) — verbose par explicit.
  • Ye kyun important hai (interview angle): ye samajhna ki Renderer2 ek indirection layer hai, aur us layer ka wajood teen features enable karta hai: (a) view encapsulation — emulated mode mein renderer har naye element pe _ngcontent-xyz lagata hai (Ch4 S4), isliye tumhare component ki CSS uspe lagti hai; (b) animations — AnimationRenderer DOM operations ko intercept karke transitions apply karta hai; (c) SSR + hydration — server pe alag renderer, aur client pe Angular jaanta hai kaunsa node kis instruction se bana, isliye DOM reuse kar sakta hai.
  • Practical warning: custom renderer poore app ke har DOM operation pe baithta hai. Ek chhoti performance galti ya edge case (SVG namespaces, hydration) poore app ko tod sakta hai. 99% cases mein hostDirectives (S9) ya ek shared service better hai.

11. Common Mistakes

Galtiyan
  • Renderer2 ko "sirf SSR ke liye" samajhna — wo encapsulation aur animations ka bhi mechanism hai.
  • Service mein inject(Renderer2) — kaam nahi karega (koi host element nahi). RendererFactory2.createRenderer(null, null).
  • renderer.listen() ka unlisten function ignore karna — leak (Ch1 S8).
  • Renderer2 se focus()/getBoundingClientRect() dhundhna — wo mutations API hai, commands/queries ke liye ElementRef.
  • Host binding se ho sakta tha wahan Renderer2 use karna — declarative better (CD-integrated, styling context).
  • document.createElement() se element banana aur expect karna ki component ki CSS lagegi — _ngcontent attribute nahi hai (Ch4 S4).
  • Custom renderer casually likhna — poore app ki DOM operations affect hoti hain.

12. Best Practices

Production rules
  • Pehle host/template bindings (declarative, CD-aware). Renderer2 tab jab imperative DOM mutation chahiye.
  • renderer.listen() ka unlisten hamesha DestroyRef.onDestroy() mein.
  • Services mein RendererFactory2.createRenderer(null, null).
  • Element creation renderer.createElement() se — encapsulation attributes free.
  • Commands/queries (focus, measure) → ElementRef + afterNextRender (S6).
  • Custom renderers: last resort, aur unhe hydration/animations ke saath test karo.
  • CSS variables: renderer.setStyle(el, '--x', v, RendererStyleFlags2.DashCase).

13. Performance

Renderer2 ka overhead ek extra function call hai — DOM operation ke saamne noise. Emulated encapsulation mein createElement ek extra setAttribute bhi karta hai (_ngcontent), jo ignorable hai.

Asli performance sawaal ye hai ki tum DOM ko kitni baar chhoo rahe ho, kaise nahi. Har manual DOM mutation Angular ke incremental DOM model (Ch3 S6) ke bahar hai — usko koi memoization nahi milti. Ek loop mein 1000 renderer.setStyle calls utne hi mehenge hain jitne 1000 direct assignments.

Behtar: state ko signal mein rakho, DOM ko bindings se update karwao — tab dirty-checking, batching, aur styling context sab free milte hain. Renderer2 use karo jahan bindings possible hi na hon (dynamic elements, body-level DOM, conditional listeners).

14. Edge Cases

  • renderer.listen('window'|'document'|'body', event, fn) — global targets, aur wahi unlisten pattern.
  • RendererStyleFlags2.DashCase CSS custom properties (--x) ke liye zaroori hai, warna camelCase conversion tod dega.
  • RendererStyleFlags2.Important se !important.
  • renderer.setProperty vs setAttribute — wahi property/attribute distinction (Ch5 S2/S3).
  • SVG elements ke liye createElement(name, 'svg') namespace ke saath.
  • Renderer2 ke begin()/end() hooks (optional) — animation renderers batching ke liye use karte hain.
  • Zoneless apps mein renderer.listen se lagaya listener CD trigger nahi karega — signal write karo.
  • Hydration ke saath: server-rendered nodes ko renderer.removeChild se hataoge toh hydration mismatch ho sakta hai.

HostBinding & HostListener (deep)

Covers: hostBindings function ka compiled roop · hostVars · Precedence · Global targets · Static vs dynamic · host object vs decorators · CD integration

1. What is it?

Host = wo element jispe directive/component baithi hai. Host bindings us element ki properties/attributes/classes/styles ko directive ki state se bind karti hain; host listeners us element (ya window/document) ke events sunti hain.

Ch4 S8 mein usage dekha tha. Yahan hum compiled output aur CD integration dekhenge — kyunki "host binding kab evaluate hoti hai" ka jawab hi zyada tar confusing bugs ka jawab hai.

2. Why do we need it?

Directive ke paas apna template nahi hai — toh wo apne host ko kaise style/observe kare? Do galat raste: (a) consumer har jagah class lagaye (directive ka internal state parent ki zimmedari), ya (b) directive nativeElement.classList.add() kare (imperative, SSR-unsafe, styling context ke bahar — S6). Host bindings teesra, sahi rasta hain: declarative, compiled, CD-integrated.

3. What problem does it solve?

  • Self-styling → directive apna host state khud reflect kare.
  • A11y contracts → role/aria-* component ke saath travel karein.
  • Listener cleanup → automatic (LView cleanup list).
  • Imperative DOM se bachna → SSR-safe by construction.

5. Execution Flow — host bindings kab chalti hain

  Host bindings us component/directive ki CHANGE DETECTION ka hissa hain,
  par ek subtle jagah pe:

  Parent component ka update block chal raha hai
       |
       v  child component ke inputs set hue
       v  child ke HOST BINDINGS evaluate hote hain
       |     (parent ke CD ke DAURAN — kyunki host element parent ke template mein hai!)
       v  child ka apna template update block chalta hai (agar dirty ho)

  ► Consequence 1: host bindings ka context CHILD hai, par unka timing PARENT ki CD hai.
  ► Consequence 2: OnPush child skip ho jaaye, toh uske host bindings bhi skip.
  ► Consequence 3: host binding expression har parent-CD-cycle mein evaluate hoti hai
                   --> function calls yahan bhi mehenge hain.

  hostVars = kitni dynamic host bindings --> utne LView slots + comparisons

6. Architecture — precedence (Ch5 S4 ka recap, host ke angle se)

  Ek class ko kai sources bind kar sakte hain. Priority:

    1. Template binding on the element      <app-card [class.x]="a">

    3. Component host binding               @Component({host: {'[class.x]': 'c'}})

  ► Component apne host pe DEFAULT deta hai; consumer template use override kar sakta hai.
  ► Do directives same class bind karein --> declaration order (unspecified-ish, avoid).
  ► Static host attrs (role: 'button') merge hote hain; template ka static attribute
    unhe override kar sakta hai.

10. Code Explanation — Line by Line

  • L1: Signal read — sasta, aur signal-based CD mein wo consumer ko track bhi karta hai.
  • L2–L4: Expressions jo har host-binding evaluation pe chalti hain. Aur host bindings parent ki CD cycle mein evaluate hoti hain (S5 ka execution flow) — toh parent ke kisi bhi unrelated change pe ye 1000 rows × 4 expressions chalengi.
  • L3/L5: Sabse bura: ek function call jo har baar array lookup + bucket computation karta hai. Interpolation ki tarah, host binding ka memoization bhi result pe hai (Object.is), call pe nahi — function chalega, phir result compare hoga (Ch5 S1 ka wahi do-layer wala point).
  • L6: computed() se: dependencies (row()) na badlein toh function chalta hi nahi, cached value milti hai. Aur zoneless mein wo view ko dirty bhi nahi karta.
  • Aur ek layer: [style.background] ki jagah [attr.data-intensity] + CSS (Ch5 S3) — tab ek chhota string binding hai aur color CSS mein. 20,000 cells pe ye teesra optimization hai. Ladder: function call → computed → CSS.

11. Common Mistakes

Galtiyan
  • Host binding mein function call/getter — har CD cycle. computed().
  • @HostBinding getter (legacy) — same problem, aur type checking kamzor.
  • Bahut saari host bindings (10+) — har ek ek slot + comparison, per instance.
  • High-frequency host listeners (mousemove, scroll) zone apps mein — har event ek tick() (Ch5 S5).
  • (document:click) per instance — 500 dropdowns = 500 document listeners. Shared service.
  • Precedence na samajhna: template ka [class.x]="false" directive ki host binding ko silently maar deta hai (Ch5 S4).
  • [attr.disabled] jab [disabled] chahiye tha (Ch5 S3 ka boolean attribute trap).

12. Best Practices

Production rules
  • host object (metadata) use karo, decorators nahi.
  • Har host expression ek signal read ya computed() — koi function call, koi getter.
  • Host bindings kam rakho; visual state data-* + CSS se (Ch5 S3).
  • A11y attributes host mein (component ke saath travel karein).
  • Global listeners: shared service se, per-instance nahi.
  • typeCheckHostBindings: true (Ch3 S3) — host expressions bhi type-check hon.
  • Conditional listeners Renderer2.listen se (host bindings static hain, S7).

13. Performance

hostVars = dynamic host bindings ki count = LView slots + comparisons per CD cycle, per instance. 5 bindings × 2000 rows = 10,000 comparisons per cycle — aur ye parent ki CD mein hota hai, toh parent ke kisi bhi change pe.

Host listeners ka cost registration mein nahi, event fire hone pe hai: zone apps mein har event ek tick(). eventCoalescing: true (Ch5 S5) mitigation, zoneless fix.

Optimization ladder: (1) function calls → computed(); (2) style bindings → data-* attribute + CSS rules; (3) OnPush/signals se CD scope chhota; (4) bade lists mein directive per row nahi, per grid + delegation.

14. Edge Cases

  • Static host attrs ('role': 'button') creation mein ek baar; template ka static attribute unhe override kar sakta hai.
  • '[attr.x]': 'expr || null' — null attribute remove karta hai (Ch5 S3). Boolean attributes ke liye zaroori.
  • Global targets: window:, document:, body:. Cleanup automatic, par per-instance listener.
  • Key modifiers host mein bhi: '(keydown.enter)', '(keydown.control.s)'.
  • $event host listener mein available hai: '(click)': 'onClick($event)'.
  • hostDirectives se aayi directives ki host bindings bhi apply hoti hain, aur unki precedence "directive host" level pe hai (S9).
  • Component aur uske host pe lagi directive dono host bindings kar sakti hain — precedence: directive > component.
  • Host bindings SSR pe bhi evaluate hoti hain (server-rendered HTML mein classes/attrs aate hain) — isliye wo afterNextRender se pehle bhi safe hain.

15. Interview Questions

EASY host object aur @HostBinding decorator mein farak?

Functionally same — host object modern preferred syntax hai (sab metadata ek jagah, kam decorators, TC39 decorators ke saath forward-compatible). Aur @HostBinding aksar getter ke saath use hota hai jo har CD cycle mein chalta hai. Dono compile hoke ek hi hostBindings function banate hain.

EASY Host listener ka cleanup kaun karta hai?

Angular. δδlistener handler ko LView ke cleanup list mein register karta hai; view destroy hone pe listener automatically remove ho jaata hai — (window:resize) jaise global targets pe bhi. Ye Ch1 S8 ke "Big Five" leaks mein se ek (global listener leak) ko API design se khatam kar deta hai. Manual addEventListener mein tumhe khud hataana padta hai.

MEDIUM Host bindings kis component ki CD cycle mein evaluate hoti hain?

Parent ki — kyunki host element parent ke template mein hai. Context child ka hota hai (ctx = directive instance), par timing parent ki CD hai. Do consequences: (1) parent ke kisi bhi change pe child ki host bindings re-evaluate hongi (isliye wahan function calls mehengi hain); (2) agar child OnPush hai aur skip ho jaata hai, uska template skip hoga par host bindings parent ke saath chal chuki hongi. Ye subtle hai aur "OnPush component ki class update ho rahi hai" wale confusion ka jawab hai.

MEDIUM hostVars kya hai?

Compiled directive definition (δdir) ka ek field: kitni dynamic host bindings hain. Har ek ke liye LView mein ek slot allocate hota hai (pichli value ke liye) aur har CD cycle mein ek Object.is comparison hota hai. Static host attributes ('role': 'button') hostAttrs mein jaate hain — wo creation mein ek baar lagti hain, hostVars mein nahi ginti.

HARD Ek directive ki host binding kaam nahi kar rahi — teen possible causes aur unka mechanism batao.

(1) Precedence loss (Ch5 S4): agar consumer ne template mein wahi class prop-bind ki hai (<button appLoading [class.btn--disabled]="false">), toh template binding (priority 1) directive host binding (priority 2) ko override kar deti hai — silently. Debug: DevTools mein class dikhti hi nahi, aur koi error nahi. Fix: alag class names (ek class = ek owner), ya state ko input se lo. (2) Directive lagi hi nahi (Ch6 S1/S4): parent ke imports mein add nahi kiya — attribute plain HTML attribute ban gaya, koi error nahi (unknown attribute legal hai). Debug: ng.getDirectives($0) se instance check karo. (3) Expression stale ya CD nahi chali: host binding parent ki CD mein evaluate hoti hai; agar parent OnPush hai aur dirty nahi hua, ya zoneless mein koi signal write nahi hua, toh binding re-evaluate hi nahi hogi. Debug: expression signal padh raha hai? Ya wo ek plain field hai jo bahar se mutate hua? (4 bonus) Manual DOM interference (S6): kisi ne nativeElement.classList.add() kiya, aur Angular ka classMap binding uske update pe use mita raha hai — kyunki manual class styling context mein nahi hai. Ye teeno/chaaron mechanisms alag layers pe hain (precedence, matching, CD, styling context) — aur ek senior engineer ka kaam hai ki wo pehle "kya instance bana?" poochhe, phir "kya expression evaluate hui?", phir "kya koi upar wali priority se override kar raha hai?".

SCENARIO Ek grid mein 5,000 rows hain, har row pe ek directive hai jiske 6 host bindings hain (selected, hovered, focused, background color, aria-selected, data-id). Scroll janky hai. Optimize karo.

(1) Measure pehle: Angular DevTools profiler se CD time; hostVars se calculate karo: 6 × 5,000 = 30,000 comparisons per CD cycle, plus har expression ka evaluation. Aur ye parent ki CD mein hota hai — matlab parent ke kisi bhi unrelated change pe. (2) Function calls hatao: har host expression ek signal read ya computed() honi chahiye. [style.background]: 'computeColor()' jaisa kuch bhi = har cycle mein 5,000 function calls. computed() se memoized (dependencies stable toh chalta hi nahi). Ye pehla, sasta win. (3) Visual state CSS mein bhejo (Ch5 S3 ka core lesson): [style.background] → [attr.data-intensity] (chhota string) + CSS rules (td[data-intensity="3"] { background: var(--heat-3) }). Ek style binding gaya, aur inline styles ka DOM bloat + style recalc bhi. Similarly hovered → CSS :hover (JS ki zaroorat hi nahi!) aur focused → :focus-visible. Ab 6 bindings se 3 bache. (4) Per-row directive hi chahiye? Selection/aria state row-level hai, par hover/focus CSS se ho gaye. Baaki ko grid-level directive + event delegation (Ch5 S5) se handle karo: ek listener grid pe, $event.target.closest('[data-row-id]'). Ab 5,000 directive instances se 1. Ye sabse bada structural win hai. (5) CD scope: grid ko OnPush + signals se drive karo, taaki parent ke unrelated changes pe rows ki host bindings evaluate hi na hon. Zoneless mein aur precise. (6) Virtualization: 5,000 rows DOM mein kyun hain? cdk-virtual-scroll se ~30 rows — tab 6 bindings bhi theek hain. Ye asli jawab hai; baaki micro-optimizations. Par a11y dhyan: aria-rowcount total dena hoga (Ch5 S3). (7) Ordering: virtualization → delegation/instance count → CSS-ification → computed() → CD scope. Structural pehle, micro baad mein. (8) Verify: har step ke baad profiler; kaunse change ne kitna diya, document karo. Punchline: host bindings sasti lagti hain kyunki declarative hain — par har ek ek comparison × instance count × parent CD frequency hai.

1. What is it?

hostDirectives (v15+) = ek component/directive apne host pe doosri directives attach kar sakta hai, bina consumer ke unhe likhe.

@Component({
  selector: 'button[appButton]',
  hostDirectives: [RippleDirective, FocusRingDirective, AnalyticsDirective],
})

// Consumer sirf ye likhta hai:
<button appButton>Pay</button>
// Aur teeno behaviors automatically attach ho jaate hain.

Ye Angular ka jawab hai ek purane sawaal ka: behavior reuse kaise karein bina inheritance ke?

2. Why do we need it?

Inheritance ki teen jails: (1) single parent — ripple aur tooltip aur analytics ek saath chahiye toh? (2) tight coupling — base class ke protected members subclasses ka contract ban jaate hain; base badla toh sab toote. (3) metadata inherit nahi hoti — Angular mein base class ka host/selector automatically nahi milta (abstract directive pattern chahiye).

hostDirectives se behaviors compose hote hain: har ek independently testable, reusable, aur unka apna encapsulated state.

3. What problem does it solve?

  • Multiple behaviors → ek host pe kai directives, koi inheritance nahi.
  • Consumer boilerplate → <button appButton appRipple appFocusRing> ki jagah sirf appButton.
  • Encapsulation → component apne internal behaviors chhupa sakta hai (inputs expose kiye bina).
  • Library design → Angular Material/CDK ka poora architecture isi pe ja raha hai.

10. Code Explanation — Line by Line

  • L1: hostDirectives compile-time metadata hai — δcmp.features mein baked. Runtime conditionals possible nahi. Ye deliberate hai (static analysis, tree shaking). Conditional behavior directive ke andar input se handle karo.
  • L2: Structural directives (TemplateRef maangti hain) host pe apply nahi ho sakti — host element ek template nahi hai. Ye ek fundamental restriction hai, arbitrary nahi.
  • L3: Sabse common gotcha. Default: hostDirective ke inputs/outputs private hain — consumer unhe bind nahi kar sakta. Ye actually ek feature hai (encapsulation: button ka ripple configure karna consumer ka kaam nahi), par pehli baar samajhne mein confusing hai. Explicit inputs: ['x'] se expose karo, aur 'x: alias' se rename bhi.
  • L4: Do host directives same class bind karein toh declaration order jeetta hai — unspecified-ish behavior, uspe rely mat karo (Ch5 S4). Aur poori composition ki precedence: template > hostDirectives > component host. Yani component apne host pe jo class lagata hai, uski hostDirective use override kar sakti hai — ye counter-intuitive lag sakta hai par consistent hai (directive host > component host).
  • Design lesson: composition ka poora point hai encapsulated behaviors. Agar tumhe har hostDirective ka har input expose karna pad raha hai, toh shayad consumer ko wo directives directly lagani chahiye (<button appButton appTooltip="x">) — hostDirectives tab use karo jab behavior component ka internal implementation detail ho.

11. Common Mistakes

Galtiyan
  • hostDirective ke inputs expose karna bhoolna — consumer bind nahi kar paayega, aur error message confusing hai.
  • Inheritance use karna jahan composition better tha — single parent ki jail.
  • Base class pe @Directive() lagana bhoolna (inheritance mein) — inputs/lifecycle inherit nahi honge.
  • Order-dependent host bindings — do hostDirectives same class bind kar rahi hain.
  • Har hostDirective ka har input expose kar dena — tab wo composition nahi, bas re-export hai.
  • Structural directive ko hostDirectives mein daalne ki koshish.
  • hostDirective mein inject(HostComponent) — circular (host directives pehle banti hain).

12. Best Practices

Production rules
  • Behavior reuse = hostDirectives. Inheritance sirf tab jab shared implementation ho (aur wo bhi rarely).
  • Har behavior ek chhoti, single-responsibility directive — alag se testable.
  • Inputs/outputs selectively expose karo — bakiya private (encapsulation ka faayda uthao).
  • Alias se naming saaf rakho ('appTooltip: tooltip').
  • Directives ke beech DI (self: true, optional: true) — concrete class ke bajaye token prefer karo bade systems mein.
  • Order-dependence se bacho: ek class = ek owner (Ch5 S4).
  • Selector-less directives (@Directive({})) sirf-hostDirectives behaviors ke liye — accidental direct use rok deti hain.

13. Performance

Har hostDirective = ek aur directive instance + LView slot + ElementInjector entry + uske hostVars ke comparisons per CD cycle (S8). Teen hostDirectives wale button ke 500 instances = 2,000 objects (button + 3 × 500).

Ye composition ka cost hai, aur wo real hai. Trade-off: code clarity aur reuse vs runtime objects. Practical rules: (1) design system ke core components mein 2-4 hostDirectives theek hain; (2) bade lists ke row components mein composition minimize karo; (3) behavior agar sirf CSS se ho sakta hai (:focus-visible, :hover) toh directive mat banao (Ch6 S1 ka lesson).

Tree shaking: hostDirectives static metadata hain, toh unused directives shake ho jaati hain — par jo component use kar raha hai wo hamesha aayengi.

14. Edge Cases

  • hostDirective ka selector ignore hota hai — wo bina attribute ke apply hoti hai. Isliye selector-less @Directive({}) ek clean pattern hai.
  • hostDirectives nested ho sakti hain: ek hostDirective ke apne hostDirectives ho sakte hain.
  • Instantiation order: hostDirectives pehle (declaration order mein), phir host khud. Isliye host apni hostDirectives inject kar sakta hai, ulta nahi.
  • hostDirectives ke lifecycle hooks chalte hain (ngOnInit etc.) — host ke pehle.
  • Directives bhi hostDirectives rakh sakti hain (sirf components nahi).
  • exportAs hostDirective ka kaam nahi karta — template se instance nahi milega; host ko inject karke re-expose karo.
  • Inheritance ke saath: base @Directive() (bina selector) hona chahiye, warna metadata inherit nahi hoti.

15. Interview Questions

EASY hostDirectives kya hai?

Ek component/directive apne host element pe doosri directives attach kar sakta hai, bina consumer ke unhe likhe: hostDirectives: [RippleDirective, TooltipDirective]. Consumer sirf <button appButton> likhta hai, aur teeno behaviors mil jaate hain. HostDirective ka selector ignore hota hai.

EASY hostDirective ke inputs consumer ko kaise dete hain?

Explicitly expose karke: hostDirectives: [{directive: TooltipDirective, inputs: ['appTooltip: tooltip']}]. Bina expose kiye wo private rehte hain — consumer bind nahi kar sakta. Ye ek feature hai (encapsulation), par pehli baar mein confusing lagta hai. Outputs bhi same tarah: outputs: ['shown: tooltipShown'].

MEDIUM hostDirectives inheritance se better kyun hai?

(1) Multiple behaviors — inheritance mein ek parent chain; composition mein jitne chaho. (2) Encapsulation — inherited members sab public/protected ho jaate hain aur subclass ka contract ban jaate hain; hostDirectives ke inputs default private hain. (3) Testing — har behavior alag se testable; inheritance mein base+subclass saath. (4) Metadata — Angular mein base class ka host/selector automatically inherit nahi hota (abstract directive pattern chahiye); hostDirectives ki bindings apply hoti hain. (5) Behaviors independently versioned/reused ho sakti hain.

MEDIUM Do directives ek element pe ek doosre se kaise baat karti hain?

DI se. Element ki saari directives (aur uska component) ek hi ElementInjector mein register hoti hain (Ch3 S9). Toh inject(OtherDirective, {self: true, optional: true}) — self se sirf usi element se dhundho, optional se wo na ho toh null. hostDirectives ke saath: wo host se pehle banti hain, isliye host unhe inject kar sakta hai; ulta nahi (circular). Design note: concrete directive class pe depend karna coupling hai — bade systems mein ek shared token (abstract class/InjectionToken) behtar hai.

HARD hostDirectives ki teen limitations aur unke design reasons batao.

(1) Static hain — hostDirectives compile-time metadata hai (δcmp.features mein), runtime conditionals possible nahi. Reason: static analysis, tree shaking, aur compile-time input/output validation (locality, Ch3 S1). Workaround: directive khud input se conditionally behave kare. (2) Structural directives nahi — wo TemplateRef inject karti hain, jo host element pe available nahi (host ek template nahi hai). Ye fundamental hai, arbitrary nahi. (3) Inputs/outputs default private — explicit exposure chahiye. Reason: encapsulation. Button ka internal ripple configure karna consumer ka kaam nahi. Trade-off: verbose, aur missing exposure ka error message confusing. (4 bonus) Precedence aur order — hostDirectives host se pehle instantiate hoti hain (isliye host unhe inject kar sakta hai, ulta nahi), aur unki host bindings component ki host bindings se jeet-ti hain (directive host > component host, Ch5 S4). Do hostDirectives same class bind karein toh declaration order — jispe rely nahi karna chahiye. (5) exportAs kaam nahi karta hostDirectives se — instance chahiye toh host inject karke re-expose kare. Bada design point: hostDirectives ek composition mechanism hai, ek re-export mechanism nahi. Agar har input expose karna pad raha hai, toh consumer ko directives directly lagani chahiye.

SCENARIO Design system mein 12 components hain (button, chip, tab, menu-item, list-item...) aur sab ko chahiye: ripple, focus-ring, disabled state, aur analytics. Aur teen components ko tooltip bhi. Architecture design karo.

(1) Behaviors ko atoms banao: RippleDirective, FocusRingDirective, DisabledDirective, AnalyticsDirective, TooltipDirective — har ek selector-less (@Directive({})) agar sirf composition ke through use hoti hai, ya selector ke saath agar standalone bhi lagti hai. Har ek: single responsibility, apna state, apne host bindings, alag unit test. (2) Pehla sawaal: kya ye sach mein directive hai? — focus-ring aksar CSS :focus-visible se ho jaata hai (zero JS, zero instances). disabled native elements pe already hai. Ye poochhna 12 × 500 instances bacha sakta hai (Ch6 S1). Jo bache, wahi directive banao. (3) Composition: har component apni hostDirectives declare kare. Inputs selectively expose: disabled aur analytics public (consumer ko chahiye), ripple/focusRing private (internal detail). Tooltip sirf teen components mein — unmein hi expose karo. (4) Behaviors ke beech coordination: ripple ko pata hona chahiye ki element disabled hai. inject(DisabledDirective, {self: true, optional: true}) — par concrete class pe depend karna 12 components ko couple karta hai. Behtar: ek DISABLED_STATE InjectionToken<Signal<boolean>> (Ch2 S5/Ch3 S9) jise DisabledDirective provide kare aur ripple consume kare. Ab ripple ko DisabledDirective ka pata bhi nahi — sirf contract ka. Ye scale karta hai. (5) Shared base? Nahi. Inheritance yahan trap hai: 12 components ka common base banaoge toh har naya behavior base mein add hoga aur sab pe lagega, chahe chahiye ya nahi. Composition mein har component apni list declare karta hai. (6) Performance budget: 4 hostDirectives × ek list mein 500 list-items = 2,000+ instances. Isliye: list-item jaise high-count components mein composition minimize karo (ripple ko CSS animation se, analytics ko parent-level delegation se, Ch5 S5). Design system ke "expensive" components (button, chip) mein composition theek hai. (7) Testing: har behavior ka apna spec (host test component ke saath); components ke specs sirf integration verify karein ("button pe ripple lagi hai"). (8) Documentation: har component ka "composed behaviors" section — kyunki consumer ko dikhta nahi ki kya laga hua hai. (9) Migration agar inheritance se aa rahe ho: ek-ek behavior nikaalo base se, directive banao, hostDirectives mein daalo, base khali hone pe delete. Har step pe tests green. Punchline: composition ka faayda encapsulation hai, aur uski keemat instance count hai — dono ko consciously budget karo.
