DI ka Falsafa — zero se

Covers: Bina DI ke duniya · Inversion of Control · DI ek design pattern hai · Angular ka flavor · Kya DI solve karta hai (aur kya nahi)

1. What is it?

Dependency Injection = ek object ko uski dependencies khud banane ke bajaye, wo bahar se di jaati hain.

// BINA DI — class apni dependencies khud banati hai
class OrderService {
  private http = new HttpClient(new XhrBackend(), new RequestOptions());
  private logger = new ConsoleLogger();
  private config = loadConfigFromWindow();
}

// DI KE SAATH — dependencies bahar se aati hain
class OrderService {
  constructor(
    private http: HttpClient,
    private logger: Logger,
    private config: AppConfig,
  ) {}
}
// Kaun banayega? Kaun decide karega ki Logger = ConsoleLogger ya SentryLogger?
// Wo kisi aur ka kaam hai. Yahi "inversion" hai.

2. Why do we need it?

Pehle wale OrderService ke saath teen problems hain, aur teeno structural hain:

  • Testing asambhav: test mein asli HTTP call jaayegi. Mock kahan lagaoge? new HttpClient() constructor ke andar hard-coded hai.
  • Configuration hard-coded: production mein Sentry logger chahiye, dev mein console. Ab if (isProd) service ke andar likhna padega.
  • Coupling: OrderService ab XhrBackend ko jaanta hai. Backend badla toh service badalni padegi — bhale hi uska HTTP ka use bilkul same ho.

DI teeno ko ek hi mechanism se solve karta hai: "kya chahiye" (type/token) aur "kaun sa concrete" (provider) ko alag kar do.

3. What problem does it solve?

  • Testability → token override karke mock inject karo.
  • Configurability → ek hi consumer, alag environments mein alag implementations.
  • Coupling → consumer abstraction pe depend karta hai (Ch2 S5 ka abstract-class token).
  • Lifecycle/scope → instance kab bane, kitni der jiye, kitne share karein — sab framework manage karta hai.
  • Wiring boilerplate → 50-level deep dependency graph automatically resolve hota hai.

4. Internal Working — DI teen alag cheezon ka naam hai

  Log "DI" ek cheez samajhte hain. Actually teen layers hain:

  1. DEPENDENCY INVERSION (principle, SOLID ka "D")
       High-level module low-level pe depend na kare; dono abstraction pe karein.
       OrderService --> Logger (interface)   NOT   OrderService --> ConsoleLogger

  2. INVERSION OF CONTROL (pattern)
       Object apni dependencies ka lifecycle control na kare; koi aur kare.
       "Don't call us, we'll call you."

  3. DEPENDENCY INJECTION (implementation of IoC)
       Dependencies constructor/field/setter se "inject" ki jaati hain.
       + DI CONTAINER (Angular ka injector) = wo cheez jo ye injection karti hai

  ► DI ke liye framework zaroori NAHI. Ye chalta hai:
        const service = new OrderService(realHttp, sentryLogger, config);
     Ye bhi DI hai! ("Poor man's DI" / manual DI)
  ► Framework (container) do cheezein add karta hai:
        (a) graph automatically resolve karna (transitive dependencies)
        (b) scope aur lifetime manage karna (singleton, per-route, per-component)

9. Advanced Example — DI kya nahi solve karta

// ❌ Galatfehmi 1: "DI se coupling khatam ho gayi"
class OrderService {
  private logger = inject(ConsoleLogger);   // L1: concrete class!
}
// Ye DI hai par dependency inversion NAHI hai. ConsoleLogger pe coupled ho.
// Inversion ke liye abstraction chahiye: inject(Logger) jahan Logger abstract hai.

// ❌ Galatfehmi 2: "sab kuch inject karo"
class DateUtil {
  private formatter = inject(DateFormatter);   // L2: pure function tha
}
// Pure functions ko inject karne ki zaroorat nahi — unka koi state nahi,
// koi lifetime nahi, unhe swap karne ki zaroorat nahi. `import { formatDate }` kaafi.

// ❌ Galatfehmi 3: "DI = singleton"
@Injectable({ providedIn: 'root' })
class WizardState { step = signal(1); }   // L3
// providedIn: 'root' = app-wide singleton = state ka lifetime app ka lifetime.
// Wizard chhodke wapas aao --> step abhi bhi 3. Scope ko LIFETIME se match karo (S6).

// ❌ Galatfehmi 4: "DI performance-free hai"
// Har ElementInjector ek LView slot hai; har inject() ek chain walk hai.
// 10,000 rows mein component-level providers = 10,000 injectors. (S9)

10. Code Explanation — Line by Line

  • L1: DI aur dependency inversion alag cheezein hain. inject(ConsoleLogger) ka matlab hai "container mujhe ConsoleLogger de" — tum abhi bhi concrete class pe depend ho. Inversion tab hai jab token ek abstraction ho (abstract class ya InjectionToken, S5). Interview mein ye distinction bolna instant seniority signal hai.
  • L2: DI ka cost hai (injector entry, resolution, indirection). Wo cost tab justified hai jab tumhe swap, scope, ya lifecycle chahiye. formatDate(d) ek pure function hai — use import karo. Har cheez ko service banana "enterprise cargo cult" hai.
  • L3: Sabse common architecture bug. providedIn: 'root' ka matlab sirf "singleton" nahi, uska matlab hai "iska lifetime app ka lifetime hai". Wizard/checkout/editor ka state route-level provider mein hona chahiye (S6) — flow chhoda, injector destroy, state gayi. Manual reset() likhna is bug ka symptom hai.
  • Deep point: DI ek scope aur lifetime ka architecture hai, ek "service milne ka tareeka" nahi. Har baar jab tum providedIn: 'root' likhte ho, tum ek lifetime decision le rahe ho. Zyada tar log wo decision unknowingly lete hain.

11. Common Mistakes

Galtiyan
  • DI aur dependency inversion ko ek samajhna — concrete class inject karna DI hai, inversion nahi.
  • Har cheez ko injectable service banana — pure functions ko import karo.
  • providedIn: 'root' default reflex — pehle poochho: iska lifetime kya hai?
  • DI ko "sirf testing ke liye" samajhna — scope, lifetime, aur configuration usse bade use cases hain.
  • Interfaces ko token banane ki koshish (runtime pe erase, S5).
  • Manual DI ko "purana" samajhna — chhoti utilities mein ek factory function bilkul theek hai.

12. Best Practices

Production rules
  • Inject karo jab: swap chahiye (test/env), scope chahiye, ya lifecycle chahiye. Warna import karo.
  • Abstraction pe depend karo (abstract class ya InjectionToken), concrete class pe nahi — jab wo cheez swap ho sakti hai.
  • Scope ko lifetime se match karo: root (app), route (feature session), component (instance).
  • Composition root (app.config.ts) mein wiring, business logic nahi (Ch3 S4).
  • inject() field initializer mein (constructor injection se better: inheritance-friendly, Ch2 S5).
  • Stateless services: providedIn: 'root' (tree-shakable). Stateful: scope soch ke.

13. Performance

DI ka runtime cost teen jagah hai: (1) resolution — har inject() ek injector chain walk hai (aksar 1–3 levels, S9 mein bloom filter isse tez karta hai); (2) instantiation — lazy hai (pehle inject() pe), par transitive graph poora banta hai; (3) memory — har ElementInjector ek LView slot leta hai (Ch3 S6).

Bundle side pe DI ka faayda hai: providedIn: 'root' tree-shakable hai — koi inject() na kare toh service bundle se drop ho jaati hai. providers: [Service] array mein daali gayi service hamesha bundle mein hai (array mein reference hai). Ye ek non-obvious but important distinction hai (S3).

14. Edge Cases

  • inject() sirf injection context mein valid hai (constructor, field initializer, factory, runInInjectionContext) — setTimeout ke andar nahi (S2).
  • Manual DI aur Angular DI mix ho sakte hain: useFactory ke andar new bilkul legal.
  • Testing mein TestBed apna injector banata hai; main.ts chalta hi nahi (Ch3 S4).
  • DI graph mein cycle possible hai (A→B→A) — runtime error, aur ye module import cycle se alag cheez hai (S8).
  • Ek class ke do instances chahiye? Do tokens banao (useExisting/useClass, S4) — ek token = ek instance per injector.

15. Interview Questions

EASY Dependency Injection kya hai?

Ek pattern jisme object apni dependencies khud nahi banata, wo bahar se di jaati hain (constructor/field/setter se). Faayde: testability (mock inject karo), configurability (env ke hisaab se alag implementation), aur decoupling (consumer abstraction pe depend karta hai). Angular ka DI container isme do cheezein add karta hai: transitive graph automatically resolve karna, aur scope/lifetime manage karna.

EASY DI ke bina Angular mein kya problem hoti?

Har component/service apni dependencies new karta — testing mein mock lagane ki jagah hi nahi hoti, environment-specific implementations ke liye if likhne padte, aur ek shared instance (singleton) ke liye global variables. Aur ek 50-level deep dependency graph manually wire karna padta.

MEDIUM DI aur Dependency Inversion mein farak?

Dependency Inversion ek principle hai (SOLID ka "D"): high-level module low-level pe depend na kare, dono abstraction pe karein. DI ek technique hai jisse dependencies bahar se aati hain. Tum DI use karke bhi inversion tod sakte ho: inject(ConsoleLogger) — ye DI hai, par tum concrete class pe coupled ho. Inversion tab hai jab token ek abstraction ho: inject(Logger) jahan Logger abstract class ya InjectionToken hai. DI inversion ko enable karta hai, guarantee nahi.

MEDIUM Har cheez ko service banake inject karna chahiye?

Nahi. DI ka cost hai (injector entry, resolution, indirection) aur wo tabhi justified hai jab: (1) implementation swap karni ho (test/environment), (2) scope/lifetime chahiye (per-route state), ya (3) shared mutable state hai. Pure functions (formatDate, slugify) ko simply import karo — unka koi state, lifetime, ya swap requirement nahi. "Sab kuch injectable" ek cargo-cult pattern hai.

HARD DI ke bina Angular kaam kar sakta hai? Aur DI container manual DI se kya extra deta hai?

Technically DI ek pattern hai, framework feature nahi — new OrderService(http, logger) bhi DI hai ("poor man's DI"). Angular ka container do cheezein add karta hai jo manually mushkil hain: (1) Transitive graph resolution: tum inject(OrderService) maangte ho, wo HttpClient maangta hai, wo HttpBackend maangta hai... container recursion se poora graph resolve karta hai aur har node ko cache karta hai. 200-service app mein ye manually 2000 lines ka wiring hota. (2) Scope aur lifetime: hierarchical injectors (S6) — same token alag subtree mein alag instance, aur us instance ka lifetime us injector ke lifetime se bandha (route destroy → service destroy → ngOnDestroy). Manual DI mein ye "kaun sa instance kahan pass karein" wala prop-drilling ban jaata hai. Bonus (3): tree shaking — providedIn: 'root' se unused service bundle se nikal jaati hai, kyunki registration service ki apni file mein hai. Practical takeaway: chhoti library ya utility ke liye factory function bilkul theek hai; app-level architecture ke liye container ka scope-management hi asli value hai, "service milna" nahi.

SCENARIO Ek team review mein bolti hai: "DI ka koi faayda nahi, hum toh sirf providedIn: 'root' use karte hain aur kabhi implementation swap nahi karte." Kya jawab doge?

Unki baat aadhi sahi hai — agar tum sirf providedIn: 'root' use kar rahe ho aur kabhi token override nahi karte, toh tum DI ka 20% use kar rahe ho, aur "singleton locator" pattern chala rahe ho. Par baaki 80% mein asli value hai: (1) Scope aur lifetime — ye DI ka sabse under-used hissa hai. Wizard/checkout/editor ka state route-level provider mein rakho: flow chhoda, injector destroy, state apne aap gayi, ngOnDestroy chala. Manual reset ka code delete ho jaata hai, aur "purana state dikh raha hai" wali bug class khatam. (2) Testing — token override se mock; koi jest.mock() ya module-mocking hack nahi. Ye tab dikhega jab wo ek service ko test karenge jiske andar teen dependencies hain. (3) Multi-tenant / feature flags — ek subtree ko alag config dena (S6/S7 ka scenario) — bina ek bhi if ke. (4) Configuration tokens — InjectionToken<ApiConfig> se typed config, aur environment-specific values ek jagah. (5) Interceptors, guards, validators — ye sab DI ke bina exist hi nahi kar sakte. Aur unka valid point maano: agar service stateless hai, kabhi swap nahi hogi, aur poore app mein ek hi hai — toh providedIn: 'root' bilkul sahi hai, aur uske liye abstraction banana over-engineering hai. Message: DI ek scope-management tool hai; "swap-ability" uska ek use case hai, uska poora point nahi.

Injector

Covers: Injector = Map + parent · NullInjector · EnvironmentInjector vs ElementInjector · inject() · Injection context · runInInjectionContext

1. What is it?

Injector = ek object jiske paas do cheezein hain:

  1. Ek map: token → provider record (kaise banana hai) + instance cache (bana hua kya hai)
  2. Ek parent pointer: agar mere paas nahi hai, toh upar poochho
// Conceptually, injector 20 lines ka hai:
class SimpleInjector {
  private records = new Map<any, ProviderRecord>();
  private instances = new Map<any, any>();
  constructor(private parent: SimpleInjector | null) {}

  get(token: any): any {
    if (this.instances.has(token)) return this.instances.get(token);   // cache hit
    const rec = this.records.get(token);
    if (rec) {
      const inst = rec.factory();      // factory ke andar recursive inject()
      this.instances.set(token, inst);   // per-injector singleton
      return inst;
    }
    if (this.parent) return this.parent.get(token);       // chain walk
    throw new Error(`NullInjectorError: No provider for ${token}`);
  }
}

Bas. Angular ka injector isse zyada optimized hai (S9 mein bloom filter), par mental model yahi hai.

2. Why do we need it?

Kyunki "dependencies bahar se aayein" (S1) ka matlab hai koi bahar hona chahiye jo unhe de. Wo bahar = injector. Aur parent chain isliye chahiye taaki scope ban sake: mere paas nahi hai toh parent se maango — jisse ek subtree apna override de sakta hai bina baaki app ko chhuye.

3. What problem does it solve?

  • Central wiring → token se instance milta hai, new kahin nahi.
  • Caching → ek token = ek instance per injector.
  • Scoping → parent chain se override possible.
  • Lazy instantiation → instance pehle inject() pe banta hai, register pe nahi.

7. Real Project Example — dynamic component with scoped config

@Injectable({ providedIn: 'root' })
export class WidgetHost {

  private appRef = inject(ApplicationRef);

  mount<C>(cmp: Type<C>, host: HTMLElement, config: WidgetConfig) {
    // Har widget ko apna config chahiye — child injector banao
    const scoped = createEnvironmentInjector(                     // L1
      [{ provide: WIDGET_CONFIG, useValue: config }],
      this.envInjector,

    const ref = createComponent(cmp, {                            // L2
      environmentInjector: scoped,
      hostElement: host,

      ref.destroy();
      scoped.destroy();                                          // L5: injector bhi!
    };
  }
}

      // const x = inject(Foo);   // NG0203
      runInInjectionContext(this.injector, () => {              // L6
        const x = inject(Foo);                                // ✅
      });
    });
  }

  onClick() {
    // ❌ Method mein context nahi — construction khatam ho chuka
    // const svc = inject(Foo);   // NG0203
    const svc = this.injector.get(Foo);                       // L7: ✅ direct get()
  }
}

10. Code Explanation — Line by Line

  • L1–L3: Field initializers class construction ke dauran chalte hain, aur Angular ne currentInjector set kar rakha hai. Isliye inject() valid hai. Ye constructor-parameter injection se behtar hai: inheritance mein super(...) ke saath parameters forward nahi karne padte (Ch2 S5).
  • L4: Constructor body bhi context mein hai. Par field initializers pehle chalte hain (JS semantics, Ch1 S5), toh L1 pehle, L4 baad mein.
  • L5: effect(), afterNextRender(), toSignal() — ye sab internally inject(DestroyRef) karte hain (cleanup ke liye), isliye unhe bhi injection context chahiye. Isliye wo constructor/field mein call hote hain, method mein nahi. (Ya {injector} option pass karo.)
  • L6: runInInjectionContext manually currentInjector set karta hai, callback chalata hai, phir restore karta hai. Ye ek stack push/pop hai. Isliye async callbacks mein bhi kaam karta hai.
  • L7: Method mein simplest fix: injector.get(Token) — ye inject() ki tarah context pe depend nahi karta, kyunki tum explicitly injector de rahe ho. Lekin lazy-service-lookup ek design smell ho sakta hai; aksar field injection hi sahi hota hai.

11. Common Mistakes

Galtiyan
  • inject() ko method/callback/setTimeout mein call karna → NG0203. Field initializer, ya injector.get(), ya runInInjectionContext.
  • effect()/afterNextRender() ko method mein call karna — wo bhi context maangte hain (ya {injector} option).
  • NullInjectorError ko galat padhna — error chain right-to-left hai; jo missing hai wo aakhri token hai.
  • createEnvironmentInjector() se bana injector destroy na karna → leak (Ch1 S8).
  • Injector ko service locator ki tarah use karna (injector.get() har jagah) — dependencies chhup jaati hain, testing mushkil. Explicit inject() fields better.
  • Ye samajhna ki ek token ke kai instances ho sakte hain ek injector mein — nahi, ek token = ek instance per injector.

12. Best Practices

Production rules
  • inject() field initializers mein (constructor params se better).
  • Deferred injection chahiye toh inject(Injector) store karo, runInInjectionContext use karo.
  • Manually banaye injectors (createEnvironmentInjector) ko destroy() karo — DestroyRef mein register.
  • Service locator pattern (injector.get() har jagah) se bacho — dependencies explicit rakho.
  • NullInjectorError ka chain padhna seekho — wo poora dependency path batata hai.
  • Testing mein TestBed.inject(Token) — wahi injector API.

13. Performance

Injector ke teen costs: (1) resolution — chain walk. EnvironmentInjector mein ye ek Map.get() per level hai. ElementInjector mein Angular ek bloom filter use karta hai (S9) taaki "is injector ke paas ye token hai hi nahi" ka jawab O(1) mein mil jaaye, poora Map dekhe bina. (2) Instantiation — lazy, par transitive graph poora banta hai pehle inject() pe. Heavy constructor = slow first access. (3) Memory — har ElementInjector LView mein slots leta hai; 10,000 rows × component-level providers = 10,000 injectors.

Practical: component-level providers bade lists ke row components mein mat daalo. providedIn: 'root' ka koi per-instance cost nahi.

14. Edge Cases

  • inject(Injector) tumhe current injector deta hai (Element ya Environment, jahan se call hua).
  • inject(EnvironmentInjector) hamesha environment-level deta hai (element skip).
  • Injector.NULL — ek public NullInjector instance (testing/edge cases).
  • injector.get(Token, defaultValue) — na mile toh default (throw nahi).
  • DestroyRef khud injectable hai; destroyRef.onDestroy(fn) = ngOnDestroy ka functional version.
  • Lazy route ka EnvironmentInjector route destroy pe destroy hota hai → uske services ka ngOnDestroy chalta hai.
  • TestBed ka injector har test ke baad reset hota hai.

15. Interview Questions

EASY Injector kya hai?

Ek container jiske paas do cheezein hain: ek map (token → provider record + instance cache) aur ek parent pointer. get(token) pe: cache dekho, phir apne records, phir parent se poochho; chain ke end pe NullInjector hai jo NullInjectorError throw karta hai. Instances lazily banti hain aur per-injector cache hoti hain.

EASY "inject() must be called from an injection context" (NG0203) kab aata hai?

Jab inject() constructor, field initializer, factory function, ya runInInjectionContext() ke bahar call ho — jaise setTimeout callback, koi method, ya subscribe callback. Angular ek internal "currentInjector" set karta hai sirf construction ke dauran. Fix: field mein inject karo, ya inject(Injector) store karke injector.get()/runInInjectionContext() use karo.

MEDIUM EnvironmentInjector aur ElementInjector mein farak?

EnvironmentInjector logical/module scope hai: root (app.config.ts + providedIn:'root'), lazy route providers, platform. ElementInjector DOM/view hierarchy follow karta hai: component ke providers/viewProviders aur us element pe lagi directives. Do parallel trees hain. Resolution pehle ElementInjector chain upar chadhta hai, phir EnvironmentInjector chain mein jump karta hai (S6). Isliye component-level provider hamesha root wale ko override karta hai.

MEDIUM runInInjectionContext kya karta hai?

Ek diye gaye injector ko temporarily "current injector" bana deta hai, callback chalata hai, phir purana restore kar deta hai (stack push/pop). Isse inject() (aur effect(), afterNextRender(), toSignal() jaise APIs jo internally inject(DestroyRef) karte hain) construction ke bahar bhi kaam karte hain — jaise setTimeout ya kisi method mein. Injector ko pehle field mein capture karna padta hai (private injector = inject(Injector)).

HARD inject() ko pata kaise chalta hai kaunsa injector use karna hai?

Angular ek module-scoped mutable variable rakhta hai — effectively ek "current injector" pointer (aur uske saath current "inject flags"). Jab Angular kisi class ko instantiate karta hai (component, directive, service factory), wo pehle setCurrentInjector(thatInjector) karta hai, phir δfac() chalata hai (jisme constructor + field initializers chalte hain), phir purana injector restore kar deta hai — ek try/finally stack ki tarah. inject(Token) bas us current injector se get() call karta hai. Consequences: (1) inject() sirf synchronous construction ke dauran valid hai — koi async callback context ke bahar hai (NG0203); (2) runInInjectionContext(injector, fn) manually yahi push/pop karta hai; (3) yahi mechanism effect(), toSignal(), afterNextRender() ko bhi context-bound banata hai, kyunki wo internally inject(DestroyRef) karte hain cleanup ke liye; (4) field injection (private x = inject(X)) constructor-parameter injection se equivalent hai runtime pe — dono δfac ke andar chalte hain. Design note: ye "ambient context" pattern React ke hooks jaisa hai (jahan useState ek current-fiber pointer pe depend karta hai) — dono mein wahi restriction hai: call site ka timing matter karta hai, position nahi.

SCENARIO Ek plugin system banana hai: har plugin ek Angular component hai, runtime pe load hota hai, aur usko apna scoped config + apna logger instance chahiye (jo plugin ka naam prefix kare). Injector-level design karo.

(1) Child EnvironmentInjector per plugin: createEnvironmentInjector([{provide: PLUGIN_CONFIG, useValue: cfg}, {provide: Logger, useFactory: () => new PrefixedLogger(cfg.name)}], parentEnvInjector). Ab plugin ke andar inject(Logger) uska apna prefixed logger dega, aur baaki app ka Logger untouched. Ye hierarchical DI ka killer use case hai — koi if (isPlugin) kahin nahi. (2) Component creation: createComponent(PluginCmp, {environmentInjector: childInjector, hostElement}) + appRef.attachView(ref.hostView) (warna CD nahi chalegi, Ch6 S7). (3) Cleanup — sabse zyada bhoola jaata hai: unmount pe appRef.detachView() → ref.destroy() → childInjector.destroy(). Injector destroy karne se uske andar banaye gaye services ka ngOnDestroy/DestroyRef callbacks chalte hain. Ye na karo toh plugin ka logger, uske subscriptions, uske timers sab zinda rehte hain (Ch1 S8). (4) Isolation ki limit: child injector parent ke saare providers dekh sakta hai (chain walk). Agar tum plugin ko app ke internal services se rokna chahte ho, toh DI kaafi nahi — wo trust boundary nahi hai. Sach mein isolation chahiye toh iframe/Web Worker/module federation, ya plugin ko ek restricted API object pass karo (capability-based). Ye distinction interviewer ko dikhta hai. (5) Lazy loading: plugin ka code import() se (static string paths, Ch3 S7); registry ek allowlist ho (server se aaya "component name" kabhi directly resolve mat karo — security). (6) Testing: har plugin ko apne child injector ke saath test karo; TestBed se parent banao, phir createEnvironmentInjector. (7) Debugging: ng.getInjector($0) se DevTools mein injector chain dikh sakti hai. Punchline: child injector = scoped configuration + scoped lifetime; par wo security boundary nahi hai.

Providers

Covers: Provider = token + recipe · providedIn ka tree shaking · Lazy instantiation · multi: true · Provider arrays · Kahan kya register karein

1. What is it?

Provider = ek nuskha (recipe) jo injector ko batata hai: "is token ke liye instance aise banao".

// Provider ke do hisse: TOKEN (kya maanga) + RECIPE (kaise banao)
{ provide: Logger,          useClass: ConsoleLogger }     // naya instance
{ provide: API_URL,         useValue: '/api/v2' }        // ready value
{ provide: Config,          useFactory: () => load() }  // runtime logic
{ provide: AbstractLogger,  useExisting: ConsoleLogger }  // alias (wahi instance)

// Shorthand: class token + useClass same class
providers: [OrderService]
// === { provide: OrderService, useClass: OrderService }

Injector ke andar ye ek Map entry ban jaata hai: token → ProviderRecord { factory, cachedValue } (S2).

2. Why do we need it?

Token bolta hai "mujhe Logger chahiye". Par Logger ek abstract class hai — usse instance nahi ban sakta. Kisi ko batana padega: "Logger maange toh SentryLogger do, aur usko DSN chahiye jo config se aata hai". Wo "batana" hi provider hai.

Aur providers hi wo jagah hain jahan configuration code se milti hai: same consumer, alag environment, alag provider.

3. What problem does it solve?

  • Token ↔ implementation mapping → swap possible.
  • Construction logic → useFactory se conditional/async-derived instances.
  • Configuration → useValue se constants, config objects.
  • Extension points → multi: true se plugin arrays (interceptors, validators).

4. Internal Working — do jagah register hote hain

10. Code Explanation — Line by Line

  • L1: providedIn: 'root' ka registration class ke saath compile hota hai — ek static δprov property (S9). Class ka koi reference nahi = class dead code = bundler use hata deta hai, uske transitive imports ke saath.
  • L2–L3: Yahan app.config.ts mein class ka literal reference hai. ESM static analysis (Ch1 S8) ke liye ye "used" hai. Bundler use rakhega, aur uske saath SomeHugeLibrary bhi.   • L4: InjectionToken ka providedIn + factory combo bhi tree-shakable hai. Ye config tokens ke liye best pattern hai (S5): typed, tree-shakable, aur default value bhi. useValue array mein daalne se ye faayda nahi milta.
  • Rule jo yaad rakho: services → providedIn. Token overrides aur config values → providers array. Providers array mein service daalne ka ekmatra reason hota hai: tumhe usko override karna hai (test, ya alag implementation).

11. Common Mistakes

Galtiyan
  • Services ko providers array mein daalna jab providedIn: 'root' kaafi tha — tree shaking gaya.
  • multi: true ko hierarchy mein merge samajhna — child parent ko replace karta hai.
  • Kuch providers multi: true ke saath aur kuch bina — runtime error ("Cannot mix multi providers and regular providers").
  • useFactory mein bhaari kaam bootstrap pe — wo pehle inject() pe chalega; agar wo initializer se call ho raha hai toh startup slow (Ch3 S4).
  • Component providers mein service daal ke singleton samajhna — har instance ka apna.
  • @Injectable() lagana bhool jaana un classes pe jo dusri dependencies inject karti hain aur providers array mein hain.
  • Provider order pe rely karna — baad wala jeetta hai, par ye fragile hai.

12. Best Practices

Production rules
  • Stateless services: @Injectable({providedIn: 'root'}). Bas.
  • Providers array sirf: token overrides, config values, provideXxx() functions.
  • Config tokens: new InjectionToken<T>('desc', {providedIn: 'root', factory}) — tree-shakable.
  • app.config.ts bada ho jaye toh core/providers/*.ts mein todo aur spread karo.
  • Scoped state: route/component providers (S6) — lifetime automatic.
  • multi: true extension points ke liye; hierarchy mein merge chahiye toh skipSelf se manually.

13. Performance

Bundle: providedIn = tree-shakable; providers array = always bundled. Ek 100KB monitoring SDK jo sirf ek route mein chahiye, agar root providers mein hai toh initial bundle mein hai (Ch3 S7 ka bundle analysis isse pakadta hai).

Startup: providers register karna sasta hai (Map entry). Instantiation lazy hai. Par useValue ki value evaluate ho chuki hoti hai (wo ek expression hai), aur useFactory pehle inject() pe chalta hai. Agar wo initializer se trigger ho raha hai toh startup path pe hai.

Memory: har injector mein har token ka ek instance. Component-level providers 10,000 rows pe = 10,000 instances.

14. Edge Cases

  • providedIn: 'any' — har lazy-loaded injector ka apna instance (root ka bhi ek). Rarely useful; confusion ka source.
  • providedIn: 'platform' — ek page pe kai Angular apps ke beech share.
  • Nested provider arrays flatten ho jaate hain: providers: [a, [b, [c]]] legal hai.
  • Provider type union hai: Type<any> | ValueProvider | ClassProvider | FactoryProvider | ExistingProvider | ....
  • EnvironmentProviders (provideXxx() ka return type) sirf environment injectors mein ja sakte hain — component providers mein nahi (compile error).
  • Same token do baar same array mein → baad wala jeetta.
  • multi providers ka order = declaration order (interceptor chain ka order isi se aata hai!).

15. Interview Questions

EASY Provider kya hai?

Ek recipe jo injector ko batati hai ki kisi token ke liye instance kaise banana hai: {provide: TOKEN, useClass/useValue/useFactory/useExisting: ...}. Injector ke andar ye token → factory ka record ban jaata hai; instance pehle inject() pe lazily banta hai aur cache hota hai.

EASY providedIn: 'root' aur providers array mein farak?

Dono se service root injector mein register hoti hai. Farak tree shaking ka hai: providedIn registration service ki apni file mein hai, toh koi inject() na kare toh class unreachable hai aur bundler use drop kar deta hai. providers: [Service] mein app.config.ts ka reference hai — class hamesha bundle mein hai. Rule: services → providedIn; token overrides/config → providers array.

MEDIUM multi: true kya karta hai?

Same token ke kai providers ko collect karke ek array banata hai, replace karne ke bajaye. inject(TOKEN) se array milta hai. Angular khud ye HTTP_INTERCEPTORS, NG_VALIDATORS, ROUTES ke liye use karta hai — extension points. Order = declaration order (interceptor chain ka order isi se aata hai). Trap: hierarchy mein child ka multi-provider parent wale ko replace karta hai, concat nahi.

MEDIUM Instances kab banti hain?

Lazily — pehle inject()/get() pe. Bootstrap pe sirf records (Map entries) banti hain. Pehle access pe factory chalti hai (jiske andar transitive inject() recursion hoti hai), phir instance us injector mein cache ho jaata hai. Isliye "singleton" ka matlab hai per injector singleton. Apwaad: useValue — value pehle se maujood hai.

HARD providedIn tree-shakable kyun hai aur providers array kyun nahi? Mechanism batao.

Ye ESM ke static module graph ka natural consequence hai (Ch1 S8). @Injectable({providedIn: 'root'}) compile hoke class pe ek static property banati hai: OrderService.δprov = δδdefineInjectable({token: OrderService, providedIn: 'root', factory: () => new OrderService()}). Yani registration class ke saath travel karta hai — koi doosri file us class ko reference nahi karti. Agar koi inject(OrderService) nahi karta, toh OrderService ka module graph mein koi incoming edge nahi hai, wo unreachable hai, aur bundler use (aur uske transitive imports ko) drop kar deta hai. Ulta, providers: [OrderService] mein app.config.ts literally OrderService ko import karke ek array mein daalta hai — ek live reference. Bundler ke liye wo "used" hai, chahe koi inject kare ya na kare. Practical impact: ek 100KB analytics SDK jo sirf ek lazy route mein chahiye, root providers array mein aa gaya toh initial bundle mein hai. Aur InjectionToken ka {providedIn: 'root', factory} pattern bhi isi wajah se tree-shakable hai. Design insight: Angular ne registration ko declaration site pe move karke tree shaking possible banayi — wahi locality principle (Ch3 S1) jo NgModule→standalone aur Zone→signals mein hai: global registry ko local declaration se replace karo.

SCENARIO App mein ek plugin system hai: teen alag features apne validators register karti hain, aur ek core service unhe run karta hai. Aur ek feature ko apne extra validators chahiye sirf apne route mein. Design karo.

(1) Extension point = multi token: export const VALIDATORS = new InjectionToken<readonly Validator[]>('validators'). Har feature apne validators {provide: VALIDATORS, useClass: EmailValidator, multi: true} se register kare. Core service inject(VALIDATORS) se array le. Ye Angular ke apne HTTP_INTERCEPTORS/NG_VALIDATORS ka exact pattern hai. (2) Order matters: multi providers declaration order mein aate hain. Agar order semantic hai (validation sequence), toh use explicit karo — ya har validator ko ek priority field do aur service sort kare, taaki registration order pe silently depend na karo. (3) Route-scoped extra validators — yahan trap hai: agar feature route ke providers mein {provide: VALIDATORS, useClass: X, multi: true} daaloge, toh us route mein inject(VALIDATORS) sirf [X] dega — root ke validators gayab. Multi providers hierarchy mein merge nahi hote, replace hote hain. Ye 90% devs ko surprise karta hai. (4) Merge ka pattern: route mein {provide: VALIDATORS, useFactory: () => [...inject(VALIDATORS, {skipSelf: true}), new X()]} — skipSelf se parent ka array lo, apna add karo (S7). Ya better: do tokens rakho — CORE_VALIDATORS (root, multi) aur FEATURE_VALIDATORS (route, multi), aur service dono ko inject karke combine kare. Ye explicit hai aur surprise-free. (5) Tree shaking: har feature ke validators uske apne chunk mein hone chahiye. Root providers array mein daaloge toh sab initial bundle mein aa jaayenge — feature ke route providers mein daalo (lazy chunk ke saath). (6) Type safety: token ko generic banao (InjectionToken<readonly Validator[]>) taaki inject() typed array de. (7) Testing: TestBed mein multi providers override karke specific validators test karo. Punchline jo bolni hai: multi providers extension points ke liye hain, aur hierarchy mein wo merge nahi hote — ye jaanna hi is design ka core hai.

1. What is it?

Chaar recipes — injector ko "instance kaise banao" batane ke chaar tareeke. Inka farak samajhna DI ka aadha gyaan hai.

RecipeKya karta haiNaya instance?

2. Why do we need it?

Kyunki "instance banana" ke alag-alag matlab hote hain. Kabhi ek class ko construct karna hai (useClass), kabhi ek constant chahiye (useValue), kabhi runtime pe decide karna hai kaunsa (useFactory), aur kabhi ek hi instance ko do naamon se accessible banana hai (useExisting).

Sabse zyada galat samjha jaane wala useClass vs useExisting hai — kyunki dono "same class" point karte lagte hain, par ek naya instance banata hai aur doosra nahi.

3. What problem does it solve?

  • Implementation swap → useClass.
  • Config/constants → useValue.
  • Conditional/derived construction → useFactory.
  • Ek instance, kai tokens → useExisting.

4. Internal Working — useClass vs useExisting

10. Code Explanation — Line by Line

  • L1–L2: useValue: window ek expression hai jo module evaluate hone pe chalti hai. Server pe window exist hi nahi karta → ReferenceError, bootstrap se bhi pehle. Aur ye error confusing hota hai kyunki stack trace mein providers array dikhta hai, koi component nahi.
  • L3: Subtle bug: Date.now() bootstrap ke waqt evaluate hota hai, na ki jab koi inject(START_TIME) kare. Agar tumhe "pehla use kab hua" chahiye tha, toh galat value milegi. useValue = eager evaluation.
  • L4: useFactory lazy hai — pehle inject() pe chalta hai. Aur inject(DOCUMENT) se platform-appropriate document milta hai (server pe bhi ek DOM-like object hota hai, Ch6 S6). defaultView browser mein window, server pe null.
  • L5: Factories compose hoti hain: STORAGE ka factory inject(WINDOW) karta hai, aur DI graph automatically resolve hota hai. Aur type mein | null hai — consumer ko SSR handle karne pe majboor karta hai (Ch2 S1 ka strict null). Ye "impossible states unrepresentable" ka DI version hai.
  • Rule: useValue sirf pure constants ke liye (strings, numbers, plain config objects). Kuch bhi jo environment/platform/timing pe depend karta hai → useFactory.

11. Common Mistakes

Galtiyan
  • useClass jab useExisting chahiye tha — do instances ban gaye, state split ho gayi, aur ye bug silent hai (dono "kaam kar rahe" lagte hain).
  • useValue: window/localStorage/Date.now() — eager evaluation, SSR crash ya galat value.
  • useFactory ke saath purana deps array — type-unsafe aur order-dependent. Factory ke andar inject() use karo.
  • useValue mein mutable object share karna — koi bhi consumer use mutate kar sakta hai. Object.freeze ya readonly types.
  • useFactory mein bhaari synchronous kaam — pehle inject() pe blocking (aur agar wo initializer se hai toh startup pe).
  • useExisting ke saath forwardRef bhoolna jab class abhi define nahi hui (circular file order).

12. Best Practices

Production rules
  • useClass: abstraction → implementation mapping. Ek token = ek instance.
  • useValue: sirf pure constants. Object.freeze() ya readonly types se seal karo.
  • useFactory: platform/environment-dependent ya derived instances. Andar inject(), deps array nahi.
  • useExisting: ek instance ko narrower interface se expose karna (read-only facade) — ek bahut acchha pattern.
  • SSR-sensitive globals (window, localStorage) hamesha useFactory + DOCUMENT token se.
  • Tokens ka {providedIn: 'root', factory} form prefer karo (tree-shakable, S3/S5).

13. Performance

useValue: zero instantiation cost, par eager evaluation — expression bootstrap pe chalti hai. Bhaari value banana yahan startup cost hai.

useClass/useFactory: lazy — pehle inject() pe. Uske transitive graph poora banta hai. Bhaari constructor = slow first access. Agar wo access ek initializer se hai (Ch3 S4), toh wo startup path pe hai.

useExisting: sabse sasta — ek redirect, koi naya object nahi. Aur memory bachata hai (do tokens, ek instance).

Bundle: providers array mein diya gaya useClass: Impl ka matlab hai Impl hamesha bundle mein (S3). Alternative: useFactory ke andar dynamic import()? Nahi — DI synchronous hai. Lazy chahiye toh route-level providers (S6).

14. Edge Cases

  • useFactory ka return null/undefined ho sakta hai — token ka type | null hona chahiye.
  • useExisting + multi: true — NG_VALUE_ACCESSOR ka classic pattern (custom form controls).
  • forwardRef(() => Class) — jab class abhi hoisted nahi hui (S8).
  • useValue ke saath class instance dena legal hai (useValue: new Foo()), par tab wo DI se resolve nahi hota — uski dependencies tumhe deni padengi.
  • useClass ke saath deps ignore hota hai (class apni DI khud karti hai); deps sirf useFactory ke liye hai.
  • Ek hi class ko do tokens pe useClass dene se do instances (aur agar wo stateful hai toh do state).
  • useValue objects ko Angular deep-clone nahi karta — wahi reference sab consumers ko.

15. Interview Questions

EASY useClass aur useValue mein farak?

useClass: Impl — injector Impl ka naya instance banata hai (uski dependencies inject karke), lazily, pehle inject() pe. useValue: obj — wo exact value return hoti hai; koi instantiation nahi, aur expression bootstrap ke waqt eagerly evaluate hoti hai. Isliye useValue: window SSR pe crash karta hai.

EASY useFactory kab use karte hain?

Jab instance banane ke liye runtime logic chahiye: environment ke hisaab se alag class, ya kisi doosri service se derived value, ya platform-safe globals (inject(DOCUMENT).defaultView). Factory injection context mein chalti hai, toh uske andar inject() use kar sakte ho (purana deps array ab zaroori nahi).

MEDIUM useClass aur useExisting mein kya farak hai? Ek bug ka example do.

useClass: CartService ek naya CartService instance banata hai us token ke liye. useExisting: CartService ek alias hai — token resolve hone pe injector CartService token ko resolve karta hai, yani wahi instance milta hai. Bug: providers: [CartService, {provide: ReadonlyCart, useClass: CartService}] — ab do alag cart instances hain. Ek component CartService.add() karta hai, doosra ReadonlyCart.items() padhta hai aur khaali array dekhta hai. Silent bug, kyunki dono "kaam kar rahe" lagte hain. Fix: useExisting.

MEDIUM useValue: window kyun galat hai?

Do wajah: (1) useValue ki expression eagerly evaluate hoti hai (module/bootstrap time pe), aur SSR mein window exist nahi karta → ReferenceError, component render se bhi pehle. (2) Ye timing bug bhi deta hai: useValue: Date.now() bootstrap ka time deta hai, use ka nahi. Sahi tareeka: new InjectionToken<Window|null>('window', {providedIn: 'root', factory: () => inject(DOCUMENT).defaultView}) — lazy, platform-safe, aur type mein null hone se consumer ko SSR handle karna padta hai.

HARD useExisting se "read-only facade" pattern kaise banate hain, aur wo achha design kyun hai?

Ek abstract class banao jisme sirf read API ho (abstract class CartReader { abstract readonly items: Signal<readonly Item[]>; abstract readonly total: Signal<number>; }), aur concrete service use extend kare aur mutations bhi de (add, remove). Phir {provide: CartReader, useExisting: CartService}. Ab display components inject(CartReader) karte hain — unhe add() dikhta hi nahi (compile-time!), par instance wahi hai, toh state consistent hai aur signals live hain. Faayde: (1) Capability segregation — kaun mutate kar sakta hai ye type system se enforce; ye Interface Segregation Principle ka DI-level implementation hai. (2) Ek instance — useClass hota toh do instances aur silent state split. (3) Testing — display components ko sirf CartReader ka stub chahiye. (4) Refactoring safety — agar kal mutations ek alag service mein chali jaayein, consumers nahi toote. (5) Zero runtime cost — useExisting ek redirect hai. Ye pattern Angular ke apne code mein hai (NG_VALUE_ACCESSOR, ControlContainer) aur design systems mein bahut kaam aata hai. Combine with Ch2 S5 ka abstract-class-as-token: abstract class ek hi cheez mein token, contract, aur type teeno hai.

SCENARIO Ek FeatureFlags service chahiye jo: server se flags load kare (startup pe), SSR pe cookies se initial value le, tests mein easily override ho, aur agar server fail ho toh defaults use kare. Providers kaise design karoge?

(1) Token design: export abstract class FeatureFlags { abstract has(flag: string): boolean; abstract readonly all: Signal<ReadonlySet<string>>; } — abstract class = token + contract + type (Ch2 S5). Consumers concrete class ko jaante hi nahi. (2) Initial value — useFactory, na ki useValue: useValue: JSON.parse(document.cookie...) SSR pe eagerly chalega aur platform-unsafe hai. Factory mein inject(DOCUMENT)/inject(REQUEST) (SSR ka request token) se cookies padho — lazy aur platform-safe. (3) Startup load: provideAppInitializer(() => inject(FeatureFlags).load()) (Ch3 S4). Par yahan discipline: initializer timeout ke saath ho (Promise.race), fail hone pe defaults use kare, aur kabhi app ko block na kare. Startup gate crash gate nahi banna chahiye. (4) Defaults: ek DEFAULT_FLAGS token ({providedIn: 'root', factory: () => DEFAULTS}) — tree-shakable, testable, aur factory mein consume ho. (5) Read-only facade: agar kahin flags set karne hote hain (dev tools, QA panel), toh FeatureFlagsAdmin alag token ho aur useExisting se wahi instance — production build mein wo token provide hi na ho. (6) SSR → client handoff: server ne jo flags resolve kiye, unhe TransferState se client tak bhejo — warna client dobara fetch karega aur flicker hoga (hydration mismatch, Ch3 S8). (7) Testing: {provide: FeatureFlags, useValue: {has: (f) => f === 'newCheckout', all: signal(new Set(['newCheckout']))}} — ek line, koi mocking library nahi. Ya useClass: InMemoryFeatureFlags. (8) Scoped overrides: ek route ke liye alag flags (A/B test) chahiye toh route providers mein token override (S6). Consumers mein ek bhi if nahi. Ye answer chaaron recipes ko unke sahi jagah pe use karta hai: useFactory (platform/derived), useValue (test stubs), useClass (implementation), useExisting (facade).

1. What is it?

InjectionToken<T> = ek unique runtime object jo DI map ki key ban sakta hai, aur jiske saath ek type parameter juda hota hai.

Sawaal: class hi token ban sakti hai (inject(OrderService)), toh alag token kyun chahiye? Jawab: kyunki non-class cheezein bhi inject karni hoti hain — strings, numbers, config objects, functions, arrays — aur unke paas koi runtime identity nahi hoti.

2. Why do we need it?

Ye chapter ka sabse important insight hai: DI ki keys runtime pe chahiye. TypeScript types runtime pe exist nahi karte (Ch2 S9).

// ✅ Do raste:

3. What problem does it solve?

  • Non-class dependencies → strings, config, functions inject karo.
  • Type safety → inject(API_URL) ka type string, any nahi.
  • Naam collisions → token object unique hai (string token collide karte the — purana Angular).
  • Tree-shakable config → {providedIn, factory} se default value bina providers array ke.

10. Code Explanation — Line by Line

  • L1–L2: Generic factory function se token ka type defaults se infer ho jaata hai — koi manual type annotation nahi. Ye Ch2 S6 ka inference-first design hai.
  • L3–L5: provideXxx() pattern (Angular ka apna idiom, Ch3 S4). Consumer ko provider object ka shape nahi likhna padta; ek typed function call. Aur Partial<T> (Ch2 S7) se sirf jo badalna hai wahi do.
  • L4: useFactory (na ki useValue) — taaki spread lazily ho aur agar defaults kisi doosre token se aayein toh inject() use kar saken.
  • L6: Sabse important trap. InjectionToken ki identity uska object reference hai, uska description string nahi. Agar do files mein new InjectionToken<number>('retry') likha hai, toh wo do alag tokens hain — ek se provide karo, doosre se inject karo, NullInjectorError milega. Aur error message dono ko "retry" bolega, jo debugging ko dukhdayi banata hai. Rule: token ek hi jagah define ho, aur wo export const ho.
  • Corollary: barrel files se token re-export karne mein savdhaan — agar circular imports ki wajah se module do baar evaluate ho jaaye (Ch1 S8), toh do token instances ban sakte hain. Ye ek real, bahut confusing bug hai.

11. Common Mistakes

Galtiyan
  • Interface ko token banane ki koshish — compile pe erase (Ch2 S9). InjectionToken ya abstract class.
  • Token do jagah define karna — do alag keys, NullInjectorError, aur error message identical.
  • InjectionToken<any> — type safety phenk di. Generic bharo.
  • String tokens ('apiUrl') — collision-prone, deprecated.
  • factory ke bina token banake providers array mein useValue — tree shaking ka faayda gaya (S3).
  • Token ka description khaali/generic (new InjectionToken('token')) — error messages bekaar.
  • Mutable config object useValue se — koi bhi consumer mutate kar sakta hai. Object.freeze/Readonly<T>.

12. Best Practices

Production rules
  • Config/constants/functions → InjectionToken<T>. Swappable services → abstract class.
  • Hamesha {providedIn: 'root', factory} — default value + tree-shakable.
  • Token ek file mein, export const, descriptive description string.
  • Generic bharo: InjectionToken<AppConfig>, kabhi <any> nahi.
  • Config objects Readonly<T> ya frozen — DI se aaya object shared hai.
  • provideXxx(overrides) helper functions likho — consumer ko provider shape na likhna pade.
  • Platform globals (window, localStorage) ke liye tokens + factory (SSR-safe, S4).

13. Performance

Token ka runtime cost: ek object allocation (ek baar, module load pe). Resolution mein wo bas ek Map key hai.

Bundle: {providedIn: 'root', factory} wala token tree-shakable hai — koi inject() na kare toh token, uska factory, aur factory ke transitive imports sab drop. Providers array mein useValue se ye faayda nahi milta (S3).

Startup: factory lazy hai. Par agar factory bhaari hai (config parse, crypto) aur wo bootstrap initializer se trigger hoti hai, toh wo startup path pe hai (Ch3 S4).

14. Edge Cases

  • InjectionToken ka toString() description deta hai — error messages isi se readable hote hain.
  • providedIn: 'any' tokens ke liye bhi kaam karta hai (har lazy injector ka apna instance).
  • Function inject karna legal hai: new InjectionToken<(x: number) => string>('formatter').
  • multi: true tokens ka type array hona chahiye: InjectionToken<readonly Validator[]>.
  • Token ka δprov (factory wala) providers array ke override se hamesha haar jaata hai (specific > default).
  • SSR mein REQUEST/RESPONSE tokens (@angular/ssr) — sirf server injector mein.
  • Circular token definitions (token A ka factory token B maange, B ka A) → DI cycle (S8).

15. Interview Questions

EASY InjectionToken kyun chahiye jab class hi token ban sakti hai?

Kyunki non-class dependencies bhi inject karni hoti hain: strings, numbers, config objects, functions, arrays. Aur DI ki keys runtime pe chahiye — TypeScript interfaces/types compile pe erase ho jaate hain (Ch2 S9), toh unhe key nahi bana sakte. InjectionToken<T> ek unique runtime object hai (Map key by reference) jiske saath ek compile-time type juda hota hai.

EASY Interface ko provide token bana sakte hain?

Nahi. Interface compile ke baad gayab ho jaata hai — runtime pe koi object hi nahi jo Map ki key bane. {provide: MyInterface, ...} compile error deta hai ("only refers to a type"). Do raste: InjectionToken<MyInterface> (value ke liye) ya abstract class (service ke liye, jo token + contract dono hai).

MEDIUM InjectionToken ka providedIn + factory kya deta hai?

Token apna default value khud laata hai — kisi providers array ki zaroorat nahi. Aur wo tree-shakable hai: registration token ke saath hai, toh koi inject() na kare toh token + factory + factory ke imports sab bundle se drop. Override abhi bhi possible hai (providers array ka specific provider default ko haraata hai). Factory injection context mein chalti hai, toh andar inject(OtherToken) use kar sakte ho.

MEDIUM Token vs abstract class — kab kaunsa?

InjectionToken<T>: config, constants, functions, arrays — jahan tumhe ek value chahiye, ek behavior contract nahi. Abstract class: swappable services — wo ek hi cheez mein token, contract (method signatures), aur type deti hai (Ch2 S5), aur consumer ko API dikhti hai. Rule of thumb: agar consumer uspe methods call karega toh abstract class; agar wo ek data value hai toh token.

HARD InjectionToken ki identity kya hai, aur usse kya bug ho sakta hai?

Identity = object reference. DI ka Map key by reference hoti hai (Ch1 S4), aur token ka description string sirf error messages ke liye hai — wo identity ka hissa nahi. Consequence: agar do files mein new InjectionToken<number>('retry') likha ho, toh wo do alag tokens hain. Ek se provide karo aur doosre se inject karo → NullInjectorError: No provider for InjectionToken retry — aur error message bilkul wahi naam bolega jo tumne provide kiya tha. Ye debugging ka nightmare hai. Rules: (1) token ek hi file mein, export const; (2) barrel files aur circular imports se savdhaan — agar module graph mein cycle ki wajah se module do baar evaluate ho jaaye (Ch1 S8 ka TDZ/cycle territory), toh do token instances ban sakte hain; (3) monorepo mein agar do packages same library ki do copies bundle karein (node_modules duplication), toh unke tokens alag honge — yahi peerDependencies aur npm dedupe ka DI-level reason hai. Ye aakhri wala real production issue hai: Angular Material ki do copies = do MAT_DIALOG_DATA tokens = "no provider" errors jo bilkul logical nahi lagte.

SCENARIO Ek reusable library bana rahe ho (@myorg/data-grid). Consumers ko configure karna hai: page size, date format, aur ek custom "export handler" function. Aur kuch consumers ko per-route alag config chahiye. Token design karo.

(1) Config shape: ek GridConfig interface (type only) + InjectionToken<Readonly<GridConfig>> (runtime key). Readonly isliye ki DI se aaya object shared hai — koi consumer use mutate na kar de. (2) Default + tree-shakable: export const GRID_CONFIG = new InjectionToken<Readonly<GridConfig>>('grid config', {providedIn: 'root', factory: () => DEFAULT_GRID_CONFIG}). Consumer kuch na kare toh defaults chalte hain, aur agar wo grid use hi na kare toh token bundle se drop. (3) provideGridConfig() helper: export function provideGridConfig(overrides: Partial<GridConfig>): Provider { return {provide: GRID_CONFIG, useFactory: () => Object.freeze({...DEFAULT_GRID_CONFIG, ...overrides})}; }. Consumer ko provider ka shape nahi likhna padta, aur partial override natural hai. useFactory (na ki useValue) — taaki future mein defaults kisi doosre token se aa saken. (4) Export handler = function token: export const GRID_EXPORT_HANDLER = new InjectionToken<(rows: readonly unknown[]) => void>('grid export', {providedIn: 'root', factory: () => defaultCsvExport}). Functions inject karna bilkul legal hai. Alternative: abstract class agar handler ke kai methods hon. (5) Per-route override: route ke providers: [provideGridConfig({pageSize: 100})] — route-level EnvironmentInjector (S6). Us subtree mein har grid 100 rows dikhayega; baaki app 25. Consumers mein zero conditionals, aur lifetime automatic. (6) Token identity discipline: token ko library ke public API (public-api.ts) se export karo, ek hi jagah define ho. peerDependencies mein Angular rakho taaki consumer ke node_modules mein library ki do copies na ho — warna do tokens, "no provider" errors. (7) Type-safe partial: Partial<GridConfig> mein nested objects ke liye DeepPartial chahiye ho sakta hai (Ch2 S8) — ya flat config rakho (better). (8) Docs: har token ka description string meaningful ho, kyunki wahi consumer ke error messages mein dikhega. Ye answer token identity, tree shaking, provideXxx idiom, hierarchical override, aur library packaging — sab jodta hai.

Hierarchical Injectors

Covers: Do parallel trees · Resolution algorithm · Route-level injectors · providers vs viewProviders · Scope = lifetime · Content projection ka trap

1. What is it?

Angular mein ek injector nahi, do parallel injector trees hain, aur resolution unke beech ek precise order mein chalta hai. Yahi wo cheez hai jo Angular DI ko baaki frameworks se alag banati hai — aur jo 90% devs use nahi karte.

  ELEMENT INJECTOR TREE              ENVIRONMENT INJECTOR TREE
  (DOM/view hierarchy)               (logical/module hierarchy)
  ───────────────────────            ──────────────────────────────
                                      NullInjector          (throws)
                                            ▲
                                            │
                                      PlatformInjector
                                            ▲
                                            │
                                      Root EnvironmentInjector
  <app-root>                                ▲   (app.config.ts + providedIn:'root')
     │                                      │
  <order-list>   ──────────────────────►  Route EnvironmentInjector
     │            (element chain khatam    (lazy route ke providers)
  <order-row>      hone pe yahan jump)
     │
  <price-cell>

  ► inject(Token) price-cell se:
      1. price-cell ka ElementInjector
      2. order-row ka ElementInjector
      3. order-list ka ElementInjector
      4. app-root ka ElementInjector
      5. Route EnvironmentInjector          <-- ab environment chain
      6. Root EnvironmentInjector
      7. PlatformInjector
      8. NullInjector --> throw NG0201

2. Why do we need it?

Kyunki "ek service ka ek instance" har zaroorat pura nahi karti. Teen alag zaroortein hain:

  • App-wide: auth, cart, feature flags — ek instance.
  • Feature-session: checkout wizard ka state — flow ke saath jiye aur mare.
  • Per-instance: har <app-editor> ka apna undo stack.

Hierarchical DI teeno ko ek hi mechanism se deta hai, aur — ye critical hai — lifetime automatic ho jaata hai: injector destroy, service destroy.

3. What problem does it solve?

  • Manual state reset → scope destroy pe state apne aap gayi.
  • Prop drilling of services → subtree mein inject() se mil jaata hai.
  • Conditional configuration → ek subtree ko alag config, bina if ke.
  • Multi-instance components → do parallel editors ka apna-apna state.

4. Internal Working — resolution algorithm

  inject(Token, flags) ka poora algorithm:

  START: current ElementInjector (agar component/directive context hai)
    │
    ├─ flags.skipSelf?  --> parent se shuru karo
    │
  LOOP over ElementInjector chain (DOM/view mein upar):
    │   token yahan hai? (bloom filter check, S9)
    │      HAAN --> instance banao/cache se do --> RETURN
    │      NAHI --> parent ElementInjector
    │
    │   flags.self?  --> loop mat karo, ek hi level dekho
    │   flags.host?  --> host component ke ElementInjector pe RUKO
    │
  ElementInjector chain khatam (root component ke baad)
    │
    ▼
  LOOP over EnvironmentInjector chain:
       Route injector --> Root injector --> Platform injector
    │
    ▼
  NullInjector
    │
    ├─ flags.optional? --> return null
    └─ warna --> throw NullInjectorError (NG0201)

Ek critical baat: jab ek component ke providers mein ek service hai, aur wo service khud kuch inject karti hai — wo resolution us component ke ElementInjector se shuru hoti hai, na ki root se. Isliye ek component-provided service uske parent components ke providers dekh sakti hai. Ye "injector jahan service provide hui, wahan se resolve hoti hai" rule hai.

6. Architecture — providers vs viewProviders

  <app-card>                          <-- card ka HOST element
    <app-child />                     <-- PROJECTED content (parent ne likha)
  </app-card>

  card.component.html:
    <div>
      <app-inner />                   <-- card ka VIEW (card ne likha)
      <ng-content />                  <-- yahan projected content aata hai
    </div>

  @Component({ selector: 'app-card', providers: [ServiceA], viewProviders: [ServiceB] })

    app-inner  (view)     --> ServiceA ✅   ServiceB ✅
    app-child  (content)  --> ServiceA ✅   ServiceB ❌ NAHI!

  ► providers:      view + projected content dono ko dikhte hain
  ► viewProviders:  SIRF component ke apne template ko

  ► Kyun? Projected content PARENT ke context mein banta hai (Ch5 S9).
    Uska ElementInjector chain parent se jaati hai, card ke view se nahi.
    Par card ka HOST element uske chain mein hai --> `providers` (host pe) dikhte hain.

Ye ek subtle mechanism hai: providers component ke host element ke injector mein jaate hain (jo projected content ka ancestor hai), viewProviders uske view ke injector mein (jo projected content ka ancestor nahi hai). Library authors ise internals chhupane ke liye use karte hain: "mera tab component apne internal TabRegistry ko consumer ke content se nahi dikhana chahta."

9. Advanced Example — interceptor ka scope trap

// Upar wala impersonation setup. Ab HTTP interceptor mein tenant chahiye:

export const tenantInterceptor: HttpInterceptorFn = (req, next) => {
  const tenant = inject(TENANT);                                   // L1
  return next(req.clone({ setHeaders: { 'X-Tenant': tenant.id } }));
};

// ❌ PROBLEM: interceptors ROOT injector mein register hote hain
//    (provideHttpClient(withInterceptors([...])) app.config.ts mein hai)
//    --> inject(TENANT) root injector se resolve hoga
//    --> route-scoped override DIKHEGA HI NAHI
//    --> impersonated route mein bhi ORIGINAL tenant ka header jaayega. Silent bug. // L2

// ✅ FIX 1: request ke saath context pass karo
export const TENANT_CTX = new HttpContextToken<string | null>(() => null);   // L3

// Service (jo route-scoped injector se resolve hui hai):
load() {
  const tenant = inject(TENANT);          // yahan sahi tenant milta hai
  return this.http.get(url, { context: new HttpContext().set(TENANT_CTX, tenant.id) });
}
// Interceptor: const id = req.context.get(TENANT_CTX);            // L4

// ✅ FIX 2: route-scoped HttpClient (bhaari, par poora isolation)
providers: [
  provideHttpClient(withInterceptors([tenantInterceptor])),        // L5: route mein!
]

10. Code Explanation — Line by Line

  • L1–L2: Yahi wo trap hai jo hierarchical DI ke saath aata hai. Ek service us injector se resolve hoti hai jahan wo provide hui thi, na ki jahan se wo call hui. Interceptor root mein register hai, toh uska inject(TENANT) root chain se jaata hai — route ka override wahan hai hi nahi. Aur ye bug silent hai: koi error nahi, bas galat header.
  • L3–L4: HttpContext Angular ka jawab hai: request ke saath data travel karta hai, injector se nahi. Caller (jo sahi injector mein hai) context set karta hai, interceptor use padhta hai. Ye explicit hai aur scope-safe.
  • L5: Alternative: route ke providers mein apna HttpClient banao, uske interceptors us injector se resolve honge. Cost: ek alag HttpClient instance (interceptor state, caching share nahi hoti).
  • General rule jo yaad rakho: agar koi cheez root mein register hai (interceptors, error handlers, guards resolve hone ke baad), toh usse route/component-scoped tokens nahi dikhte. Scoped configuration ko scoped consumers tak seemit rakho, ya data ko explicitly pass karo. Interview mein ye batana bahut strong signal hai — ye wo detail hai jo sirf production experience se aati hai.

11. Common Mistakes

Galtiyan
  • Sab kuch providedIn: 'root' — flow-scoped state global ban gayi; manual reset likhna padta hai (aur bhool jaate ho).
  • Component providers mein service daal ke singleton samajhna — har instance ka apna.
  • Root-registered services (interceptors) mein scoped tokens expect karna — silent galat value.
  • viewProviders aur providers ka farak na jaanna — projected content ko service nahi milti (ya galti se mil jaati hai).
  • Provider ko @if ke andar wale component pe rakhna — component destroy pe service bhi gayi (state loss).
  • Scoped service ko providedIn: 'root' se test karna — test pass, production mein alag behavior.
  • Bade lists ke row components mein providers — 10,000 injectors (S9).

12. Best Practices

Production rules
  • Scope = lifetime. Poochho: ye state kab tak jeeni chahiye? Root (app) / route (feature session) / component (instance).
  • Stateless services: providedIn: 'root'. Stateful: scope soch ke.
  • Wizard/editor/flow state → route-level providers. Manual reset code delete ho jaayega.
  • viewProviders se library internals ko projected content se chhupao.
  • Scoped tokens ke consumers bhi scoped hone chahiye — root services unhe nahi dekhengi.
  • Request-scoped data HttpContext se, injector se nahi.
  • Provider ko stable ancestor pe rakho (@if ke andar nahi).

13. Performance

Har ElementInjector apne LView mein slots leta hai (Ch3 S6, S9). Component-level providers = per-instance injector + per-instance service. 10,000 rows × ek provider = 10,000 injectors + 10,000 service instances. Bade lists ke row components mein providers mat daalo.

Resolution ka cost chain depth ke proportional hai, par Angular ka bloom filter (S9) "is injector mein ye token nahi hai" ka jawab O(1) mein deta hai — toh 20-level deep tree mein bhi walk sasta hai.

Route-level injectors ka cost ignorable hai (ek injector per route). Aur unka faayda bada hai: route destroy pe uske services destroy → subscriptions, timers, listeners sab cleanup (Ch1 S8).

14. Edge Cases

  • Lazy route ka injector route destroy pe destroy hota hai; eager routes ka nahi (wo root ka hissa hain).
  • Ek route ke providers uske children ko dikhte hain, siblings ko nahi.
  • Content projection ka DI: projected content parent ke ElementInjector se resolve hota hai (Ch5 S9) → viewProviders nahi dikhte.
  • createEnvironmentInjector() se manual child injector (S2) — destroy tumhara kaam.
  • ngOnDestroy services pe chalta hai agar unka injector destroy ho (root wale kabhi nahi).
  • Directive ke providers us element ke injector mein jaate hain — component ke saath share.
  • Same token component aur uske parent dono pe → component wala jeetta hai (nearest wins).

Wajah: projected content parent ke context mein banta hai (Ch5 S9), aur uski injector chain component ke host se hoke jaati hai, uske view se nahi. Library authors viewProviders se internals chhupate hain.

MEDIUM Wizard ka state kahan rakhoge aur kyun?

Route-level providers mein (@Injectable() bina providedIn). Kyunki state ka lifetime = flow ka lifetime. Sab steps ko same instance milega (parent route ka injector), aur flow chhodne pe injector destroy → service destroy (ngOnDestroy chalta hai) → wapas aane pe fresh state. providedIn: 'root' hota toh manual reset() likhna padta aur bhoolne pe "purana data dikh raha hai" wali bug. Scope ko lifetime se match karo — ye DI ka sabse under-used principle hai.

HARD Ek root-registered HTTP interceptor ko route-scoped token kyun nahi dikhta?

Kyunki ek service us injector ke context mein resolve hoti hai jahan wo provide hui thi, na ki jahan se wo call hui. provideHttpClient(withInterceptors([tenantInterceptor])) app.config.ts mein hai, toh interceptor ka inject(TENANT) root EnvironmentInjector se resolve hota hai. Route ka {provide: TENANT, useValue: otherTenant} route-level injector mein hai, jo root ka child hai — aur resolution hamesha upar jaati hai, neeche nahi. Toh interceptor ko hamesha root ka default milega. Aur ye silent bug hai: koi error nahi, bas galat X-Tenant header. Fixes: (1) HttpContext — caller (jo sahi injector mein hai) request ke saath data attach kare, interceptor req.context.get(TOKEN) se padhe. Explicit, scope-safe, aur testable. (2) Route-scoped HttpClient — route ke providers mein provideHttpClient(withInterceptors([...])); ab interceptor us injector se resolve hoga. Cost: alag HttpClient instance (caching/state share nahi). (3) Interceptor ko token ke bajaye ek root service se poochhne do jo current context track karti ho (par phir tum global mutable state pe wapas aa gaye). General principle: scoped configuration ke consumers bhi scoped hone chahiye. Yahi baat guards, resolvers, aur ErrorHandler pe bhi lagti hai.

SCENARIO Multi-tenant SaaS: har tenant ka apna theme, API base URL, aur feature flags. Admin "impersonate" mode mein doosre tenant ke context mein browse karta hai. Aur ek shared ReportService hai jo teeno cheezein use karti hai. Poora DI design karo.

(1) Tokens, na ki globals: TENANT, API_BASE_URL, FEATURE_FLAGS — sab InjectionToken with {providedIn: 'root', factory} (S5). Koi service kabhi window.__tenant nahi padhti. Ye poore design ki buniyaad hai. (2) Default (normal mode): root factories session/config service se current tenant ke values lein. (3) Impersonation = child injector: admin route ke providers mein teeno tokens override karo (useFactory se resolver ke data se). Us subtree ke saare components aur route-scoped services ko doosre tenant ke values milenge — ReportService mein ek bhi if (isImpersonating) nahi. Route chhoda → injector destroy → normal. Yahi hierarchical DI ka killer use case hai. (4) Yahan sabse bada trap: ReportService agar providedIn: 'root' hai, toh wo root injector se resolve hoti hai — usse route-scoped TENANT nahi dikhega! Fix: usse bhi route ke providers mein daalo (tab wo route injector se resolve hogi), ya use tenant ko parameter se lene do. Yahi baat HTTP interceptor pe lagti hai: root mein register hai, toh HttpContext se tenant pass karo, ya route-scoped HttpClient banao. Ye detail 90% candidates miss karte hain. (5) Theme: CSS custom properties (Ch4 S4) — tenant config se :root pe variables set karo. Zero JS per component. (6) Security: impersonation ka authorization server pe enforce ho. DI sirf UI context hai, trust boundary nahi. Aur har API call pe server tenant verify kare, header pe bharosa na kare. (7) SSR: server pe tenant request se (subdomain/cookie) aaye; TransferState se client ko bhejo warna hydration mismatch. (8) Testing: TestBed mein tokens override — poora tenant simulate, zero mocking of internals. Aur ek test jo verify kare ki impersonated route mein interceptor sahi header bhejta hai (wahi silent bug jo (4) mein tha). S-07

Resolution Modifiers

Covers: optional · self · skipSelf · host · Combinations · Recursive components · Form control patterns · Decorator vs options syntax

1. What is it?

Modifiers inject() ke resolution walk ko badalte hain — kahan se shuru karo, kahan ruko, aur na mile toh kya karo.

inject(Token, { optional: true });    // na mile toh null (throw nahi)
inject(Token, { self: true });        // sirf APNA injector dekho
inject(Token, { skipSelf: true });    // apna skip, PARENT se shuru
inject(Token, { host: true });        // host component tak, aage nahi

// Combine bhi kar sakte ho
inject(Token, { skipSelf: true, optional: true });

// Legacy decorator syntax (constructor injection era)
constructor(@Optional() @SkipSelf() private parent: TreeNode) {}

2. Why do we need it?

Default resolution "nearest wins, warna throw" hai (S6). Par kuch patterns ko precise control chahiye:

  • Recursive tree component ko parent ka context chahiye, apna nahi → skipSelf
  • Ek directive sirf usi element ki service accept kare, parent se galti se na uthaye → self
  • Optional dependency (agar module load ho toh use karo) → optional
  • Content-projected child host component se aage na dekhe → host

3. What problem does it solve?

  • Accidental parent resolution → self se strict local contract.
  • Recursive structures → skipSelf se parent ka instance.
  • Optional integrations → optional se graceful degradation.
  • Encapsulation boundary → host se lookup ko rok do.

4. Internal Working — walk kaise badalta hai

  Chain:  [child] --> [parent] --> [grandparent] --> ... --> [root env] --> NullInjector
             ▲
          inject() yahan se

  { self }:         child --> STOP. Na mila toh throw (ya null with optional)

    Angular runtime error deta hai.
  ► optional kisi bhi doosre ke saath combine ho sakta hai.

  `host` ka exact matlab (sabse confusing modifier):

// Recursive tree: har node apna context provide karta hai,
// par depth calculate karne ke liye PARENT ka context chahiye

@Injectable()

@Component({
  selector: 'app-tree-node',
  providers: [NodeContext],                                       // L1: har node apna
  template: `
    <div [style.padding-left.px]="ctx.depth * 16">{{ node().name }}</div>
    @for (child of node().children; track child.id) {
      <app-tree-node [node]="child" />
    }
  `,
})
export class TreeNodeComponent {
  node = input.required<TreeNode>();

  protected ctx = inject(NodeContext);                          // L2: apna
  private parentCtx = inject(NodeContext, {                       // L3
    skipSelf: true,
    optional: true,                                             // L4: root ka parent nahi
  });

  constructor() {
    this.ctx.depth = (this.parentCtx?.depth ?? -1) + 1;         // L5
    this.ctx.path = [...(this.parentCtx?.path ?? []), this.node().id];
  }
}

  b = inject(FORM_CTX, { self: true, optional: true });   // L4: null (input pe nahi hai)

10. Code Explanation — Line by Line

  • L3: Default: chain mein sabse nazdeek. input pe kuch nahi, app-form-card ke host pe hai → 'card'.
  • L4: self = sirf input element ka apna injector. Wahan FORM_CTX nahi hai (sirf FieldDirective hai) → null. self ka matlab hai "mere host element pe lagi directives/providers", na ki "meri component chain".
  • L5: host = chain mein upar chalo par host component ke element tak. appField ka host component FormCardComponent hai (kyunki wo uske content mein projected hai) → wahan tak dekho → 'card' mila. Agar FORM_CTX sirf app-page pe hota, toh host use nahi dhundhta (boundary crossing nahi).
  • L6: skipSelf = apna element skip, parent se shuru → wahi 'card'.
  • L7: viewProviders ke saath (S6) tasveer badal jaati hai: projected content ko wo providers dikhte hi nahi, kyunki uski chain host element se jaati hai, view se nahi. Ye host modifier ke saath milke Angular ke form directives ka poora architecture banata hai — ngModelGroup/formGroupName ka ControlContainer resolution exactly is mechanism pe khada hai.
  • Yaad rakhne ka tareeka: self = "mera element". host = "mera element se lekar us component tak jiske template mein main likha gaya hoon". skipSelf = "mere element ke upar se".

11. Common Mistakes

Galtiyan
  • self ko "mera component" samajhna — wo "mera element" hai (uspe lagi directives + providers).
  • skipSelf ke saath optional bhoolna — recursive tree ke root pe NullInjectorError.
  • self: true, skipSelf: true — contradictory, runtime error.
  • optional ke baad null check bhoolna — TS | null deta hai, use ignore mat karo.
  • host ko "root tak" samajhna — wo ek boundary hai (host component pe rukta hai).
  • Legacy decorators (@Optional()) aur options object mix karna — ek style chuno.
  • Modifiers ko overuse karna — agar bahut saare skipSelf chahiye, toh scope design galat hai.

12. Best Practices

Production rules
  • inject(T, {options}) syntax use karo, decorators nahi (functional, inheritance-friendly).
  • optional: true ke saath type mein | null aata hai — handle karo (Ch2 S1).
  • Recursive components: {skipSelf: true, optional: true} — dono chahiye.
  • Directive-to-directive DI (Ch6 S1): {self: true, optional: true}.
  • host tab jab content projection boundary matter karti ho (form directives).
  • Agar modifiers ki zaroorat baar-baar pad rahi hai, toh scope design (S6) revisit karo — modifiers fine-tuning hain, workaround nahi.

13. Performance

Modifiers walk ko chhota karte hain, toh technically faster hain: self ek level, host ek boundary tak. Par ye micro-optimization hai — DI resolution already sasti hai (bloom filter, S9).

Asli performance consideration: skipSelf wale recursive components mein har node ka apna injector + apna service instance hai. 1000-node tree = 1000 injectors + 1000 NodeContext objects. Deep trees mein ye add up hota hai. Alternative: ek shared service jo path signal se derive kare, ya @for ke $index/context se depth pass karo (input se).

14. Edge Cases

  • optional ke saath default value nahi milta — null milta hai. Default chahiye toh ?? fallback.
  • injector.get(Token, notFoundValue) — get() API mein default value ka option hai.
  • host aur skipSelf combine ho sakte hain ({skipSelf: true, host: true}) — parent se shuru, host pe ruko.
  • Component ke apne providers uske apne inject() ko dikhte hain (constructor mein) — kyunki wo uske ElementInjector mein hain.
  • viewProviders ke saath host ka behavior badalta hai (upar L7).
  • Legacy decorators (@Self(), @Host()) constructor params pe abhi bhi kaam karte hain.
  • inject(Token, {optional: true}) ka return type T | null hai — TS overloads se.

15. Interview Questions

EASY optional: true kya karta hai?

Agar poori chain mein token na mile toh NullInjectorError throw karne ke bajaye null return karta hai. Return type T | null ho jaata hai, toh TypeScript tumhe null-check karne pe majboor karta hai. Use: optional integrations (analytics ho toh track karo), aur recursive components ke root node pe (jahan parent nahi hai).

EASY skipSelf ka ek use case?

Recursive components: har tree node apna NodeContext provide karta hai, par depth calculate karne ke liye use parent ka context chahiye. inject(NodeContext, {skipSelf: true, optional: true}) — apna skip karo, upar se lo; root pe parent nahi hai isliye optional. Doosra use: singleton guard (agar parent chain mein already provided hai toh error throw karo).

MEDIUM self aur host mein farak?

self = sirf apna element ka injector dekho (uspe lagi directives + uske providers), aage bilkul mat jao. host = chain mein upar chalo, par us component ke element pe ruk jao jiske template mein tum likhe gaye ho (content projection boundary). Yani self ek level hai, host ek boundary. Angular ke form directives host use karte hain (ControlContainer) taaki wo apne enclosing form component ka container lein, kisi door ke ancestor form ka nahi.

MEDIUM self: true aur skipSelf: true ek saath?

Contradictory hai — "apna skip karo" aur "sirf apna dekho" ek saath possible nahi. Angular runtime error deta hai. Baaki combinations valid hain: {skipSelf, optional} (recursive components), {self, optional} (directive-to-directive DI), {host, optional} (form directives), aur {skipSelf, host} bhi.

HARD host modifier ka exact semantics kya hai, aur viewProviders uske saath kaise interact karta hai?

host: true lookup ko host component ke element injector tak seemit karta hai — yani us component ke element tak jiske template mein current element likha gaya hai. Content projection ke saath ye ek natural boundary banta hai: agar <input appField> <app-form-card> ke andar projected hai, toh appField ka host FormCardComponent ka element hai; host lookup wahan tak jaayega aur ruk jaayega — app-page (uska parent) tak nahi. Ab viewProviders ke saath twist: providers component ke host element ke injector mein jaate hain, aur viewProviders uske view ke injector mein (S6). Projected content ki chain host element se hoke jaati hai, view se nahi. Toh: providers → host lookup unhe dhundh lega; viewProviders → host lookup unhe nahi dekh paayega (aur na hi default lookup). Yahi mechanism Angular ke forms mein use hota hai: ngModelGroup apna ControlContainer viewProviders mein provide karta hai taaki uske apne template ke controls use lein, par projected content unhe accidentally na uthaye; aur child directives inject(ControlContainer, {host: true, optional: true}) se apna enclosing container dhundhti hain. Do features (host modifier + viewProviders) milke ek precise encapsulation boundary banate hain — ye Angular DI ka sabse elegant, aur sabse kam samjha gaya hissa hai.

SCENARIO Ek nested accordion banana hai: accordion ke andar accordion. Har accordion apne direct panels ko coordinate kare, nested wale ko nahi. Aur ek <app-panel> ko apne immediate accordion ka reference chahiye. DI se design karo.

(1) Panel → immediate accordion: panel inject(AccordionComponent) kare — default resolution "nearest wins" hai (S6), toh use apna immediate accordion mil jaayega, nested case mein bhi. Yahan koi modifier nahi chahiye; ye ElementInjector chain ka natural behavior hai. Par ek trap: agar panel <ng-content> se project hua hai (jo accordion mein hamesha hota hai), toh uski chain parent ke context se jaati hai — aur wo chain accordion ke host element se hoke jaati hai, isliye accordion ka component instance mil jaayega (host injector mein hai). (2) Accordion → direct panels: contentChildren(PanelComponent) (Ch4 S7) default mein descendants: true hai — yani nested accordion ke panels bhi mil jaayenge! Bug. Do fixes: (a) contentChildren(PanelComponent, {descendants: false}) — sirf direct children; par ye DOM structure pe depend karta hai (agar panel ek wrapper div mein hai toh miss ho jaayega). (b) Behtar: har panel apne accordion ko inject(AccordionComponent) se dhundhe aur usme khud ko register kare (accordion.register(this), DestroyRef.onDestroy(() => accordion.unregister(this))). Ab nesting automatically sahi hai — kyunki DI chain "nearest wins" hai, nested panel apne nested accordion mein register hoga. Ye Angular Material ka pattern hai. (3) Nested accordion ko parent ka reference chahiye? (e.g. "parent band ho toh child bhi band") — inject(AccordionComponent, {skipSelf: true, optional: true}). skipSelf se apna nahi, parent ka; optional kyunki top-level accordion ka koi parent nahi. Bilkul recursive-tree pattern (upar wala L3–L5). (4) self ka use: agar ek directive ([appPanelHeader]) chahti hai ki wo usi element pe lagi PanelComponent ko le (na ki upar wale panel ko), toh inject(PanelComponent, {self: true}). Bina self ke wo galti se ancestor panel utha legi — ek silent bug jo nested cases mein hi dikhega. (5) viewProviders se internals chhupao: accordion apna internal PanelRegistry agar viewProviders mein de, toh projected panels use nahi dekh paayenge (S6) — galat! Panels toh projected content hain. Isliye yahan providers chahiye, viewProviders nahi. Ye exactly wo detail hai jo tabs/accordion banate waqt logon ko ghanton ulajhati hai. (6) Testing: ek test jo nested accordion banaye aur assert kare ki outer accordion mein sirf 2 direct panels registered hain, 4 nahi. (7) Signals ke saath: registry ko signal<PanelComponent[]> rakho taaki open/close state reactive ho. Punchline: DI ki "nearest wins" chain nesting ko free mein handle karti hai — queries (contentChildren) nahi karti. Isliye registration-via-DI pattern nested components ke liye superior hai.

11. Common Mistakes

Galtiyan
  • DI cycle aur module import cycle ko ek samajhna — alag errors, alag tools, alag fixes.
  • forwardRef se DI cycle "fix" karna — error chhupa, design wahi.
  • Module cycle ka error kisi aur file mein dikhta hai — usi file ko debug karte rehna.
  • Barrel files se cycles banana (Ch3 S3) — index.ts ↔ member.
  • import type ki jagah value import — sirf type ke liye runtime dependency ban gayi (Ch2 S9).
  • Service locator (injector.get()) ko cycle ka standard fix banana — dependencies chhup jaati hain.
  • Cycle detection CI mein na hona — wo silently badhte rehte hain aur lazy chunking bhi todte hain (Ch3 S7).

12. Best Practices

Production rules
  • DI cycle = design smell. Shared state ko ek passive owner service mein nikaalo.
  • Reactive dependencies (signals) > imperative call chains — DAG naturally banta hai.
  • forwardRef sirf declaration-order problems ke liye (NG_VALUE_ACCESSOR, self-reference).
  • import type use karo jahan sirf type chahiye.
  • CI mein madge --circular (ya dependency-cruiser) — module cycles ratchet karo.
  • Barrel files app ke andar avoid karo; deep imports + path aliases (Ch3 S3).
  • Service locator (injector.get()) sirf last resort, comment ke saath.

13. Performance

DI cycles ka direct performance impact nahi — wo error hain, slow code nahi.

Module import cycles ka impact bada hai: (1) bundler lazy chunks split nahi kar paata (cycle ke dono sirey ek chunk mein aa jaate hain) — ye "lazy route ka code initial bundle mein aa gaya" ka classic root cause hai (Ch3 S7); (2) incremental rebuild ka invalidation scope badh jaata hai (dev loop slow); (3) module evaluation order fragile ho jaata hai — ek import reorder karne se production mein undefined aa sakta hai.

Isliye cycle detection ko CI mein rakhna ek performance practice bhi hai, sirf hygiene nahi.

14. Edge Cases

  • forwardRef ke saath inject(): inject(forwardRef(() => SomeClass)) — legal, par sochlo kyun zaroori hai.
  • Parent component ko child se inject karna (same file, parent pehle define) — forwardRef chahiye.
  • Constructor-injection era mein @Inject(forwardRef(() => X)) decorator syntax.
  • DI cycle sirf construction ke waqt banta hai; agar ek side lazy hai (getter, method) toh cycle detect nahi hoga — par design abhi bhi circular hai.
  • Do services jo ek doosre ke types use karti hain (par instances nahi) — import type se cycle gayab.
  • Circular InjectionToken factories (token A ka factory B maange, B ka A) → wahi NG0200.

15. Interview Questions

EASY NG0200 kab aata hai?

Jab DI graph mein cycle ho: A ka factory B maange aur B ka factory A. Angular instance slot mein pehle ek "circular" sentinel daalta hai, phir factory chalata hai — agar resolution ke dauran wo sentinel mil jaaye toh NG0200: Circular dependency in DI throw hota hai. Ye ek clear error hai (module import cycle se ulta, jo silent hota hai).

EASY forwardRef kya karta hai?

Ek reference ko lazy banata hai: forwardRef(() => MyClass) ek function return karta hai (ek marker property ke saath). Angular jab token resolve karta hai toh us function ko call karke asli class nikaalta hai — tab tak class define ho chuki hoti hai. Zaroorat isliye padti hai kyunki class declarations hoisted nahi hoti (TDZ, Ch1 S1), aur decorator class se pehle evaluate hota hai. Classic use: NG_VALUE_ACCESSOR mein class apne hi decorator ko reference karti hai.

MEDIUM DI cycle aur module import cycle mein farak?

DI cycle: A ko B chahiye, B ko A — runtime, injector resolution pe. Error saaf: NG0200. Module import cycle: a.ts aur b.ts ek doosre ko import karte hain — module evaluation pe (Ch1 S8), aur ek side pe class undefined mil sakti hai. Error confusing: "Cannot read properties of undefined (reading 'ɵprov')" ya "No provider for undefined", aur wo kisi teesri file mein dikhta hai. Tools alag: DI cycle Angular batata hai; module cycle ke liye madge --circular. Fixes alag: DI cycle = design (shared state ka owner nikaalo); module cycle = import type, barrel hatao, restructure.

MEDIUM forwardRef se DI cycle fix karna sahi hai?

Nahi — wo error chhupata hai, cycle nahi hataata. forwardRef ka legitimate use declaration order (TDZ) hai: class apne decorator ko reference kare, ya do classes ek hi file mein. Asli DI cycle (A ↔ B) mein fix design-level hona chahiye: (1) shared state ko ek passive owner service mein nikaalo (dono usko inject karein), (2) ek dependency ko callback/event se invert karo, ya (3) last resort: lazy resolution (injector.get() getter mein) — par wo service locator pattern hai aur dependencies chhupa deta hai.

HARD DI cycles kyun bante hain? Root cause aur uska universal fix batao.

Root cause almost hamesha ek hi hota hai: do services ek shared piece of state ke do hisse hain, aur dono use own karna chahti hain. Classic examples: AuthService ↔ ApiService (dono ko token chahiye), CartService ↔ PricingService (dono ko cart items), RouterService ↔ AnalyticsService (dono ko navigation events). Universal fix: (1) shared state ko ek passive owner service mein nikaalo — wo koi doosri service inject na kare (dependency graph mein leaf ho), sirf state aur uske primitive operations rakhe; (2) dono services us owner pe depend karein; (3) cycle gaya, aur bonus mein state ka single source of truth mil gaya (Ch4 S9). Signals ke saath ye aur natural hai: TokenStore ek signal rakhta hai, AuthService likhta hai, ApiService padhta hai (ya computed se derive karta hai) — koi kisi ko call nahi karta. Deeper insight: cycles imperative call-chains se bante hain ("A ko B se ye karwana hai"); reactive dependencies naturally ek DAG banati hain ("A is value pe depend karta hai"). Isliye signals-first architecture mein DI cycles kaafi kam hote hain. Aur agar cycle sach mein zaroori lage (jaise auth-refresh loop), toh use ek event ya callback se invert karo — ApiService ek onUnauthorized handler expose kare, AuthService use register kare. Ab ApiService ko AuthService ka pata bhi nahi.

SCENARIO Production build fail ho rahi hai: "Cannot read properties of undefined (reading 'ɵprov')" — ek file mein jo bilkul theek dikhti hai. Dev build chalti hai. Debug karo.

Ye classic module import cycle ka symptom hai, aur uske "dev mein chalta hai" hone ki wajah bhi hai. (1) Kyun ye error: δprov Angular ki compiled provider property hai (S9), jo class pe static hoti hai. "undefined ki ɵprov padh nahi sakte" ka matlab hai ki class hi undefined hai jab wo padhi ja rahi thi — yani module evaluation ke waqt uska binding abhi initialize nahi hua (Ch1 S8 ka TDZ/cycle territory). (2) Dev vs prod kyun alag: bundler ka module ordering, scope hoisting, aur tree shaking prod mein alag hote hain — cycle ka "kaunsa side pehle evaluate hua" badal jaata hai. Isliye cycles time-bombs hain: aaj chalte hain, kal ek unrelated import add karne se toot jaate hain. (3) Detect: npx madge --circular --extensions ts src/ chalao. Ya dependency-cruiser. Ye poori cycle chain deta hai. Error message se kabhi pata nahi chalega. (4) Common culprits: (a) barrel files — index.ts ne A ko export kiya, A ne index.ts se B import kiya (Ch3 S3); (b) type-only import jo value import hai — import { OrderService } sirf type annotation ke liye, par runtime import ban gaya. Fix: import type (Ch2 S9) — wo compile pe erase ho jaata hai aur cycle gayab; (c) shared constants/tokens jo do services ke beech ghoom rahe hain. (5) Fixes: import type lagao jahan possible; shared types/tokens ek alag leaf file mein nikaalo (*.model.ts, *.tokens.ts) jise dono import karein; barrel hatao aur deep imports + path aliases use karo. (6) Prevent: CI mein madge --circular — zero-tolerance ratchet. Aur ek bonus: cycles hataane se lazy chunking bhi theek hoti hai (Ch3 S7 — bundler cycle ke dono sirey ek chunk mein daal deta hai). (7) Aur agar ye DI cycle nikla (NG0200): tab error saaf hota aur fix design-level hota (shared state ka owner). Yahan error undefined ka hai, toh ye module cycle hai. Ye distinction hi debugging ka pehla step hai.

5. Execution Flow — Bloom filter (chapter ka crown jewel)

  PROBLEM: ElementInjector chain 20 levels deep ho sakti hai.
  Har level pe "kya is element pe ye directive/token hai?" check karna mehenga hai.
  Aur 99% baar jawab NAHI hota.

  SOLUTION: Har NodeInjector ke saath ek 256-bit BLOOM FILTER rakho.
  (LView mein 8 consecutive 32-bit numbers = 256 bits)

  ── REGISTRATION (creation time) ──
    Har directive/provider token ko ek unique ID milti hai: token.__NG_ELEMENT_ID__
    bucketIndex = (id & 0xE0) >> 5      // kaunsa 32-bit number (0-7)
    bitIndex    = id & 0x1F             // us number mein kaunsa bit (0-31)
    bloom[bucketIndex] |= (1 << bitIndex)

  ── LOOKUP (inject time) ──
    if ((bloom[bucketIndex] & mask) === 0)
         --> "is injector ke paas ye token DEFINITELY nahi hai"  → parent pe jao
    else --> "shayad hai" → ab actual directive list scan karo

  ► FALSE POSITIVES possible (do tokens ka same bit) --> phir scan karo, mil nahi
    toh parent. Correctness safe.
  ► FALSE NEGATIVES IMPOSSIBLE. Yahi bloom filter ki guarantee hai.
  ► Result: 20-level walk mein 19 levels ka jawab ek bitwise AND se.

// Simplified (real code: packages/core/src/render3/di.ts)
function bloomHashBitOrFactory(token: any): number {
  const id = token[NG_ELEMENT_ID];         // unique per token
  return typeof id === 'number' ? id & BLOOM_MASK : id;
}

function bloomHasToken(bloomHash: number, injectorIndex: number, lView: LView) {
  const mask = 1 << bloomHash;                        // bit position
  const b = bloomHash >> BLOOM_BUCKET_BITS;              // kaunsa bucket (0-7)
  const value = lView[injectorIndex + b];                // LView se 32-bit int
  return !!(value & mask);                              // ek bitwise AND
}

  • L2: False positive ka cost: ek directive-list scan jo fail hoga, phir parent pe jaana. Yani worst case wahi hai jo bina bloom filter ke har case hota. Best case (99%): ek bitwise AND.
  • L3: Yahi bloom filter ka mathematical contract hai, aur yahi use correctness-safe banata hai: false negatives impossible. Agar bit set nahi hai toh token yahan definitely nahi hai — parent pe jao, bina scan kiye. Ye "definitely not" wala jawab hi 99% cases mein chahiye hota hai.
  • L4: Ye ek classic systems-engineering trade-off hai: probabilistic data structure jo galat jawab de sakta hai, par sirf ek hi direction mein galat — aur wo direction safe hai (extra kaam, galat result nahi). Yahi trick databases (LSM trees), CDNs, aur browsers (Safe Browsing) mein use hoti hai.
  • Interview mein kaise bolna: "ElementInjector chain walk ko sasta banane ke liye Angular har node injector ke saath ek 256-bit bloom filter rakhta hai, LView ke 8 slots mein. Har token ki ek ID hoti hai; registration pe uska bit set hota hai. Lookup pe ek bitwise AND se pata chal jaata hai ki token definitely nahi hai — tab bina scan kiye parent pe jao. False positive pe ek scan waste hota hai, par false negative impossible hai." Ye 30 seconds mein poora mechanism hai.

11. Common Mistakes

Galtiyan

13. Performance

Ab poora picture: (1) Negative lookup (token yahan nahi hai) — ek bitwise AND, O(1). Ye 99% cases hai, isliye deep chains sasti hain. (2) Positive lookup — bloom hit ke baad TNode ki directive list ka linear scan (chhoti list). (3) Instantiation — ek baar per injector, phir cache. (4) Memory — ElementInjector = 9 LView slots; EnvironmentInjector = ek object + Map.

Iska practical matlab: DI resolution ka cost tumhari problem nahi hai. Injector aur instance count tumhari problem hai. 10,000 rows × component-level providers = 10,000 injectors (90,000 slots) + 10,000 service instances. Wahan attack karo, resolution pe nahi.

14. Edge Cases

  • δprov ka value field cached instance rakhta hai (root-provided services ke liye).
  • δfac inheritance ke saath: base class ka δfac derived class use karta hai agar derived ka apna na ho (isliye base pe @Injectable() chahiye, Ch6 S1).
  • __NG_ELEMENT_ID__ kuch special tokens pe negative hota hai (ElementRef, TemplateRef, ViewContainerRef, ChangeDetectorRef) — wo bloom filter se nahi, special-case code se resolve hote hain (kyunki wo har node pe "available" hain).
  • InjectFlags bitmask hai: Default=0, Host=1, Self=2, SkipSelf=4, Optional=8 (S7 ke modifiers).
  • Tree-shaken build mein δprov ke saath class hi drop ho jaati hai (S3).
  • Dev mode mein extra checks (ngDevMode) — production build mein wo code hat jaata hai.

15. Interview Questions

EASY ɵprov kya hai?

Ek compiled static property jo @Injectable se banti hai: Service.δprov = δδdefineInjectable({token, providedIn, factory}). Isme registration information hoti hai. Kyunki ye class ke saath travel karti hai, providedIn tree-shakable hai (S3). Aur δfac factory function hai jo instance banata hai (isliye reflect-metadata ki zaroorat nahi).

EASY "Cannot read properties of undefined (reading 'ɵprov')" ka matlab?

Jis class ka δprov padha ja raha tha, wo class hi undefined hai us waqt. Ye almost hamesha module import cycle hai (S8): cycle ki wajah se module evaluation ke waqt binding initialize nahi hua. Fix: madge --circular se cycle dhundho, import type lagao, barrel hatao. Ye DI ka problem nahi hai.

MEDIUM ElementInjector memory mein kahan rehta hai?

Wo ek alag object nahi hai — component ke LView array ke andar 9 consecutive slots hain: 8 numbers (256-bit bloom filter) + 1 number (parent injector ka encoded index + flags). Isliye uska creation cost bahut kam hai (array slots), aur chain walk cache-friendly hai (array indices, koi pointer chasing nahi). Ulta, EnvironmentInjector ek asli object hai (R3Injector) with a real Map — kyunki uske instances kam hain (root, per-route).

MEDIUM inject() ko bina injector pass kiye kaise pata chalta hai kahan se resolve karna hai?

Angular ek module-level "current injector" variable rakhta hai. Class instantiate karne se pehle wo setCurrentInjector(thatInjector) karta hai, phir δfac() chalata hai (jisme constructor + field initializers chalte hain), phir purana restore kar deta hai — ek stack push/pop. inject(Token) bas us current injector se resolve karta hai. Isliye inject() sirf construction ke dauran valid hai (NG0203 warna), aur runInInjectionContext() manually yahi push/pop karta hai (S2).

HARD Angular ElementInjector lookup ko kaise optimize karta hai? Bloom filter ka poora mechanism batao.

ElementInjector chain 20+ levels deep ho sakti hai, aur har level pe "kya is node pe ye token hai?" check karna mehenga hoga — jabki 99% baar jawab "nahi" hota hai. Isliye har NodeInjector ke saath ek 256-bit bloom filter rakha jaata hai: LView ke 8 consecutive 32-bit slots. Registration (creation time): har directive/provider token ko ek unique numeric ID milti hai (__NG_ELEMENT_ID__). Us ID se bucket (id >> 5, 0-7) aur bit (id & 31) nikaal ke bloom[bucket] |= 1 << bit. Lookup (inject time): same hash se bit compute karo aur ek bitwise AND: agar bit set nahi hai toh token is injector mein definitely nahi hai → seedha parent index pe jao, koi scan nahi. Agar set hai toh "shayad hai" → TNode ki actual directive list scan karo; mil gaya toh factory chalao aur cache karo, warna parent. Trade-off: 256 bits mein 256+ tokens collide kar sakte hain → false positives possible (ek waste scan), par false negatives impossible (registered token ka bit hamesha set hai). Yani probabilistic structure jo sirf "safe" direction mein galat hota hai. Result: deep chains mein 19 out of 20 levels ka jawab ek bitwise AND se. Special cases: ElementRef, TemplateRef, ViewContainerRef, ChangeDetectorRef ke IDs negative hote hain — wo bloom se nahi, special-case code se resolve hote hain kyunki wo har node pe available hain. Yahi trick databases (LSM tree ke SSTables), CDNs, aur browsers (Safe Browsing) use karte hain. Practical takeaway: DI resolution ka cost tumhari problem nahi hai; injector aur instance count hai.

SCENARIO Ek 20,000-row grid hai. Profiler dikhata hai ki component creation slow hai. Tumhe shak hai DI pe. Kaise investigate karoge aur kya fix karoge?

(1) Pehle measure, phir blame: Angular DevTools profiler se creation time; Chrome Performance tab se flame chart (kya δfac calls dikh rahi hain?); heap snapshot se object counts. "DI slow hai" ek hypothesis hai, conclusion nahi. (2) DI resolution ka cost check karo: negative lookups bloom filter se O(1) hain (bitwise AND), toh chain depth se ghabrao mat. Agar profiler mein δδdirectiveInject hot hai, toh problem resolution nahi, instantiation count hai. (3) Asli culprits, priority order mein: (a) Row/cell components ke providers — har instance ka apna ElementInjector (9 LView slots) + apna service instance. 20,000 × 1 provider = 20,000 injectors + 20,000 objects. Fix: service ko upar le jao (grid-level provider), aur row ko data input se do (Ch4 S9). (b) Har row pe directives (Ch6 S1) — har ek instance + injector entry + hostVars comparisons. Fix: CSS/delegation. (c) Har cell ko component banana (Ch4 S1) — LView + CD node + injector. Fix: plain elements + interpolation. (4) Sabse bada fix: virtualization. 20,000 rows DOM mein kyun hain? cdk-virtual-scroll se ~30 rendered — tab DI ki koi baat hi nahi. Aur aria-rowcount total dena (Ch5 S3). (5) Agar virtualization possible nahi (print view, export): (i) row-level component hataao, plain <tr> + @for; (ii) providers grid-level pe; (iii) track sahi (Ch5 S10); (iv) OnPush/signals. (6) Verify: har change ke baad heap snapshot mein instance count aur profiler mein creation time. Numbers ke bina "optimization" guessing hai. (7) Jo nahi karna: inject() ko injector.get() se replace karna, ya modifiers (self) se "resolution tez" karne ki koshish — ye micro-noise hai. Punchline jo bolni hai: Angular ka DI resolution bloom filter ki wajah se already tez hai; jo mehenga hai wo injector aur instance count hai — toh count kam karo, lookup optimize mat karo.
