S-01

Service ka Falsafa

Covers: Service = plain class · Angular ka service kuch special nahi hai · Kya service banti hai, kya nahi · Single Responsibility · Naming

1. What is it?

Angular mein "service" ka koi special meaning nahi hai. Wo ek plain TypeScript class hai jise tum DI ke through share karte ho. Bas.

// Ye ek service hai
@Injectable({ providedIn: 'root' })
export class OrderService {
  private http = inject(HttpClient);
  getAll() { return this.http.get<Order[]>('/api/orders'); }
}

// Ye bhi ek "service" hai — Angular ke liye koi farak nahi
export class Calculator {
  add(a: number, b: number) { return a + b; }
}
providers: [Calculator];   // @Injectable ki zaroorat bhi nahi (S2)

Angular ne kabhi "Service" naam ka koi base class ya interface nahi banaya. Koi extends AngularService nahi hai. Ye deliberate hai: service ek architectural role hai, ek framework construct nahi.

2. Why do we need it?

Component ka kaam hai UI — template render karna, user input lena, events emit karna (Ch4 S9). Jaise hi tum component mein HTTP call, business rules, ya shared state daalte ho, teen cheezein toot-ti hain:

  • Reuse: wo logic doosre component mein use nahi ho sakti.
  • Testing: logic test karne ke liye poora component + DOM chahiye.
  • Lifetime: component destroy hua toh logic aur state bhi gaye — par shayad wo state rehni chahiye thi.

Service ye teeno de deti hai: reusable, independently testable, aur uska lifetime tum decide karte ho (Ch7 S6 ka scope = lifetime).

3. What problem does it solve?

  • Fat components → logic bahar, component patla.
  • Duplicated logic → ek jagah likho.
  • Untestable code → class ka test = new + assert.
  • State ownership → state ka ek clear ghar.

10. Code Explanation — Line by Line

  • L1: Service version mein tumne DI ka cost liya: ek injectable class, ek injector entry, ek instance, aur consumer ko inject() karna padta hai. Kya mila? Kuch nahi — wo function stateless hai aur swap nahi hoga.
  • L2: Pure function: import karo, call karo, test karo. Zero framework coupling. Ye function ek Node script mein bhi chal sakta hai. Aur tree-shakable hai per function (service mein poori class aati hai).
  • L3: Ab picture badal gayi. Agar implementation runtime pe chuni jaani hai (country, feature flag, tenant), toh abstraction + DI ka poora point hai (Ch7 S1 ka "inject tab jab swap/scope/lifecycle chahiye").
  • L4: Yahi wo judgement hai jo "sab kuch service banao" wale cargo cult se bachata hai. DI ek tool hai, ek default nahi. Har injectable class ek runtime object, ek injector entry, aur ek indirection layer hai. Uska koi na koi faayda hona chahiye.
  • Practical heuristic: agar tumhari service mein sirf pure methods hain aur koi inject() nahi hai, toh wo shayad functions ka collection hai. Agar usme inject() hai (HTTP, config, doosri services) ya state hai, toh wo sahi mein service hai.

11. Common Mistakes

Galtiyan
  • Sab kuch service banana — pure functions ko @Injectable mein lapetna. DI ka cost, zero faayda.
  • UtilService, HelperService, CommonService — naam hi bata raha hai ki responsibility nahi hai. God class ka pehla kadam (S9).
  • Component mein HTTP + business rules + state — teeno alag reasons to change.
  • Service mein UI concerns (toast, dialog, router navigation) — ab wo service headless nahi rahi.
  • Ek OrderService jo API + store + facade + CSV export sab karti hai — 800 lines, 12 dependencies.
  • Service ko "logic ka dumping ground" samajhna — wo bhi component jitni hi disciplined honi chahiye.

12. Best Practices

Production rules
  • Service tab banao jab: shared state, swap-ability, ya lifecycle chahiye. Warna pure function.
  • Naam role batata ho: OrderApi, CartStore, OrderFacade, OrderMapper. *Service suffix vague hai.
  • Ek class = ek reason to change. Do reasons = do classes.
  • Services headless rahein — koi DOM, koi toast, koi navigation (wo UI layer ka kaam).
  • Business rules pure functions mein — framework-free, trivially testable.
  • Component: UI + ephemeral state. Service: shared state + I/O. Function: rules.

13. Performance

Har injectable service ka cost (Ch7 S9): ek injector record, ek instance, aur uska transitive graph. Root-provided services ka per-component cost zero hai — ek hi instance.

Bundle: pure functions per function tree-shakable hain; ek service class poori aati hai (ek method use karo, saare aa jaate hain). 40 methods wali UtilService mein se agar tum ek method use karte ho, toh baaki 39 aur unke imports bhi bundle mein hain. Ye UtilService anti-pattern ka concrete, measurable nuksaan hai.

Startup: services lazy hain (pehle inject() pe, Ch7 S3). Par agar ek service bootstrap initializer se touch hoti hai, uska poora dependency graph startup path pe aa jaata hai.

14. Edge Cases

  • Service mein ngOnDestroy chalta hai agar uska injector destroy ho (route/component-scoped). Root services pe kabhi nahi.
  • DestroyRef.onDestroy() service mein bhi kaam karta hai (functional cleanup).
  • Service inject() constructor/field mein hi kar sakti hai (injection context, Ch7 S2).
  • Ek service doosri service ko inject kar sakti hai — graph automatically resolve hota hai. Cycles se bacho (Ch7 S8).
  • Services mein signals bilkul valid hain (S5) — par effect() ke liye injection context chahiye.
  • Service ko component ki tarah "instantiate" karke test karna theek hai agar wo inject() use na karti ho.

15. Interview Questions

EASY Angular mein service kya hai?

Ek plain TypeScript class jise DI ke through share kiya jaata hai. Angular ne koi base class ya interface nahi banaya — "service" ek architectural role hai, framework construct nahi. Uska kaam: shared state rakhna, I/O karna (HTTP), ya swappable behavior dena — taaki components patle aur testable rahein.

EASY Kab service banani chahiye aur kab nahi?

Banao jab: shared mutable state ho, implementation swap karni ho (test/env), ya koi lifecycle ho (subscriptions, timers). Mat banao jab wo ek pure function hai — calculateTax(order) ko @Injectable mein lapetne se sirf indirection aati hai, koi faayda nahi. Practical test: agar class mein koi inject() nahi hai aur koi state nahi hai, toh wo shayad functions ka collection hai.

MEDIUM UtilService/HelperService kyun bura pattern hai?

(1) Naam mein hi responsibility nahi hai — "helper" kuch nahi batata, toh har cheez usme daal di jaati hai. God class ban jaata hai. (2) Multiple reasons to change — date formatting, CSV export, aur validation ek hi file mein; teen alag teams use touch karti hain. (3) Bundle — service class atomic hai; ek method use karo toh poori class + uske saare imports bundle mein aate hain. Pure functions per function tree-shakable hain. (4) Testing — ek helper test karne ke liye poori class ki dependencies chahiye. Fix: pure functions ko simple exported functions banao, domain ke hisaab se files mein.

MEDIUM Business logic component mein kyun nahi rakhni chahiye?

Teen wajah: (1) Reuse — doosra component wahi rule use nahi kar sakta; copy-paste hota hai aur do jagah diverge ho jaate hain. (2) Testing — rule test karne ke liye TestBed + DOM chahiye; ek pure function ka test ek line hai. (3) Lifetime — component destroy hua toh logic bhi gaya, par shayad wo state/behavior rehna chahiye tha. Sahi split: component = UI + ephemeral state; service = shared state + I/O; pure function = rules.

HARD "Single Responsibility" ko practically kaise apply karoge ek service pe?

SRP ka matlab "ek method" ya "chhoti class" nahi hai — uska matlab hai ek reason to change (Robert Martin ka original framing: "ek actor ke prati responsible"). Practical test: class ko dekh ke poochho, "ise badalne ke kaunse-kaunse karan ho sakte hain, aur wo karan kis team/stakeholder se aayenge?" Ek OrderService jisme getAll() (backend contract badla), calculateTax() (tax law badla), exportToCsv() (export format badla), aur showToast() (design system badla) hain — wo chaar alag stakeholders ko answer karti hai. Chaar reasons, chaar units: OrderApi (backend), calculateTax() pure function (domain), ordersToCsv() pure function (serialization), Notifier (UI). Ye split testing bhi aasan karta hai (har unit ki apni dependencies) aur bundle bhi (pure functions tree-shakable). Par savdhaan: SRP ko literally lene se 200 ek-method classes ban jaati hain, jo apni jagah ek anti-pattern hai (indirection tax). Rule of thumb: class 200-300 lines cross kare, ya usme do alag "domains" ki vocabulary dikhe (HTTP + tax), ya do alag teams uspe PR bhejein — tab split karo. Pehle se nahi.

SCENARIO Code review mein aaya: OrderService, 850 lines, 11 injected dependencies, methods mein HTTP calls, tax calculation, CSV export, toast notifications, aur router navigation. Kya bologe aur kaise refactor karoge?

(1) Symptom se cause pe jao: 850 lines symptom hai, cause hai "is class ko badalne ke chaar-paanch alag karan hain". Har method ko ek stakeholder assign karo: backend team (HTTP), finance/legal (tax), data team (CSV), design system (toast), product (navigation flow). Paanch stakeholders = paanch units. (2) 11 dependencies ka matlab: har naya test 11 mocks maangta hai. Ye sabse convincing argument hai — team ko test file dikhao. (3) Refactor plan, risk ke order mein: (a) Pure functions pehle nikaalo — calculateTax, ordersToCsv. Zero risk (koi DI nahi), turant testable, aur tree-shakable. Ye pehla PR hai aur wo aasani se merge hoga. (b) UI concerns nikaalo — toast aur navigation service se hatao; wo caller (component/facade) ka kaam hai. Service ko headless banao: wo result ya error return kare, decide na kare ki UI kya dikhaye. Ye service ko SSR-safe aur reusable banata hai. (c) API layer alag — OrderApi jo sirf HTTP + DTO mapping kare (S7). (d) State alag — agar service mein state hai toh OrderStore (S5). (e) Jo bacha, wo shayad ek OrderFacade hai (S7). (4) Har step pe tests pehle — current behavior lock karo, phir move karo. Behavior-preserving refactor ka yahi tareeka hai. (5) Ek PR per concern, aur pehla PR sabse aasan wala (pure functions) — momentum banta hai. (6) Kya nahi bolna: "850 lines bahut hain". Ek 850-line class jo ek hi cheez karti ho theek ho sakti hai; ek 200-line class jo teen kaam kare, nahi. Line count symptom hai, argument nahi.

@Injectable

Covers: Decorator kya karta hai · providedIn ke chaar roop · Kab zaroori hai, kab nahi · Inheritance · @Injectable vs providers array

1. What is it?

@Injectable() ek decorator hai jo do kaam karta hai:

  1. Class pe ek factory generate karta hai (δfac) — jo uski dependencies inject karke instance banata hai
  2. Agar providedIn diya hai, toh ek provider record bhi (δprov) — taaki koi providers array likhna na pade (Ch7 S3/S9)
@Injectable({ providedIn: 'root' })
export class OrderApi {
  private http = inject(HttpClient);
}

// Compiled (Ch7 S9):
OrderApi.δfac = () => new OrderApi();
OrderApi.δprov = δδdefineInjectable({ token: OrderApi, providedIn: 'root', factory: OrderApi.δfac });

2. Why do we need it?

Angular ko do cheezein pata honi chahiye: (a) is class ka instance kaise banega, aur (b) uske constructor mein kya-kya chahiye. Purane frameworks ye reflect-metadata polyfill se runtime pe padhte the. Angular (Ivy) ye compile time pe karta hai — decorator ek marker hai jo compiler ko batata hai "is class ki DI information generate karo".

Isliye @Injectable ka poora point hai: "compiler, is class ke liye δfac banao".

3. What problem does it solve?

  • reflect-metadata dependency → compile-time factory generation.
  • Providers array boilerplate → providedIn se self-registration.
  • Tree shaking → registration class ke saath travel karta hai (Ch7 S3).
  • Scope declaration → 'root' / 'platform' / 'any' / none.

4. Internal Working — kab @Injectable zaroori hai

  ZAROORI hai jab:
    - Class KHUD kuch inject karti hai (inject() ya constructor params)
      --> compiler ko δfac banana padega jo dependencies resolve kare

  ZAROORI NAHI jab:
    - Class ka constructor khaali hai AUR wo providers array mein hai
      class Calculator { add(a, b) { return a + b; } }
      providers: [Calculator]              // kaam karega!

  ZAROORI hai (bina dependencies ke bhi) jab:
    - providedIn use karna hai              @Injectable({providedIn: 'root'})
    - class ko koi doosri class INHERIT kar rahi hai jo inject karti hai

  ► Practical rule: HAMESHA lagao. Confusion ka koi faayda nahi,
    aur agar kal koi dependency add hui toh error nahi aayega.

// ❌ Classic bug
export class LoggerService {          // koi @Injectable nahi
  private http = inject(HttpClient);   // dependency!
}
providers: [LoggerService];
// Error: "Cannot resolve all parameters" ya NG0204
// Kyunki compiler ne δfac generate hi nahi kiya.

// ✅

11. Common Mistakes

Galtiyan
  • @Injectable lagana bhoolna jab class kuch inject karti ho → "Cannot resolve all parameters" / NG0204.
  • Abstract base class pe @Injectable() na lagana jab wo constructor injection use kare.
  • providedIn: 'any' ko "any injector se milega" samajhna — wo "har injector ka apna instance" hai.
  • Stateful service ko providedIn: 'root' — lifetime galat, manual reset chahiye (Ch7 S6).
  • @Injectable ke saath providers array bhi — tree shaking ka faayda gaya (Ch7 S3).
  • providedIn: 'platform' ko "aur bhi global" samajhna — wo micro-frontend ke liye hai.

12. Best Practices

Production rules
  • @Injectable hamesha lagao — confusion ka koi faayda nahi.
  • Stateless services: providedIn: 'root' (tree-shakable, ek instance).
  • Stateful services: scope soch ke — @Injectable() + route/component providers.
  • 'any' avoid karo jab tak explicit requirement na ho (aur comment likho).
  • Field injection (inject()) > constructor params — inheritance-friendly (Ch7 S2).
  • Abstract base classes pe bhi @Injectable() (ya sirf inject() use karo).

13. Performance

@Injectable ka runtime cost zero hai — wo compile pe erase ho jaata hai (Ch3 S5), sirf δfac/δprov static properties bachti hain.

Bundle: providedIn: 'root' tree-shakable hai. providers: [Service] array mein daalne se class hamesha bundle mein hai (Ch7 S3). Ek 100KB SDK jo sirf ek route mein chahiye, agar root providers array mein hai toh initial bundle mein hai.

Startup: instances lazy hain. Par providedIn: 'root' wali service jiske constructor mein bhaari kaam hai, pehle inject() pe wo cost deti hai — aur agar wo bootstrap initializer se touch hoti hai, toh startup path pe.

14. Edge Cases

  • @Injectable() bina options ke = "factory banao, par register mat karo" → providers array chahiye.
  • Service ka ngOnDestroy tabhi chalta hai jab uska injector destroy ho (root wale kabhi nahi).
  • useFactory providers ke saath @Injectable ki zaroorat nahi (factory tum de rahe ho).
  • @Injectable ke saath @Component/@Directive ek class pe nahi lag sakte.
  • Inherited δfac: derived class ka apna δfac na ho toh base ka use hota hai (isliye base pe decorator).
  • providedIn: SomeInjectionToken bhi possible tha (v14 tak) — ab deprecated.
  • Testing: TestBed mein providedIn: 'root' service automatically available hoti hai.

15. Interview Questions

EASY @Injectable kya karta hai?

Compiler ko batata hai ki is class ki DI information generate karo: ek δfac factory (jo dependencies inject karke instance banata hai) aur agar providedIn diya hai toh ek δprov provider record bhi. Isliye Angular ko reflect-metadata polyfill nahi chahiye — sab compile time pe hota hai.

EASY Kya @Injectable hamesha zaroori hai?

Technically nahi: agar class kuch inject nahi karti aur wo providers array mein hai, toh bina decorator ke bhi kaam karegi. Par zaroori hai jab: (a) class khud kuch inject karti ho, (b) providedIn use karna ho, ya (c) koi subclass use inherit kare jo inject karti ho. Practical rule: hamesha lagao — kal koi dependency add hui toh error nahi aayega.

MEDIUM providedIn ke chaar roop batao.

'root': root EnvironmentInjector, ek instance per app, tree-shakable — default choice. 'platform': platform injector, ek page pe kai Angular apps share karte hain (micro-frontends, Angular Elements). 'any': root ka apna instance, aur har lazy-loaded injector ka apna — rare, aur surprise ka source. Koi providedIn nahi (@Injectable()): kahin register nahi; tumhe providers array (route/component) mein daalna padega — scoped state ke liye yahi sahi hai.

MEDIUM providedIn: 'any' se kya bug ho sakta hai?

Har lazy-loaded route injector ko apna instance milta hai. Agar service mein state hai (cache, store), toh eager routes root ka instance share karenge par har lazy route ka apna hoga — teen alag caches. Symptom: "cache kaam nahi kar raha", aur wo dev mein aksar nahi dikhta (sab eager) balki production mein lazy chunks ke saath. 'any' NgModule-era ka artifact hai; standalone + 'root' ke baad uski zaroorat almost khatam.

HARD Inheritance ke saath @Injectable ka kya rule hai, aur field injection kyun better hai?

Angular ka δfac class pe static property hoti hai. Agar derived class ka apna δfac nahi hai, wo base ka use karta hai. Constructor parameter injection ke saath: base class ke constructor params ki metadata compile time pe generate hoti hai — agar base pe @Injectable() nahi hai toh wo metadata hai hi nahi, aur derived class δfac generate karte waqt "Class extends value has no constructor metadata" error deta hai. Fix: base pe bhi @Injectable() (abstract ho toh bhi). Field injection (private http = inject(HttpClient)) ke saath ye problem hai hi nahi: field initializers derived class ke δfac ke andar chalte hain (constructor ka hissa hain), aur wahan injection context maujood hai (Ch7 S2). Toh base class pe koi metadata chahiye hi nahi. Aur bonus faayde: (1) super(...) mein params forward nahi karne padte — base mein ek naya dependency add karo, derived classes ko chhoona nahi padta; (2) constructor khali rehta hai, toh usme setup logic saaf dikhta hai; (3) readonly fields natural hain. Yahi wajah hai ki Angular team ne inject() ko recommend kiya. Ek nuance: inject() field initializers base class ke pehle chalte hain ya baad mein? JS semantics: base ke field initializers super() ke dauran, derived ke super() ke baad (Ch1 S5) — toh base ke fields derived ke constructor body mein available hain.

SCENARIO Ek team ne providedIn: 'root' ko blanket rule bana diya hai. Ek naya requirement aaya: har open tab (workspace) ka apna document editor state ho, aur tab band hone pe wo state jaani chahiye. Kaise samjhaoge aur implement karoge?

(1) Problem ko state ownership ke terms mein rakho: providedIn: 'root' ek lifetime declaration hai — "is state ka lifetime app ka lifetime hai" (Ch7 S1/S6). Editor state ka lifetime tab ka lifetime hai. Mismatch hai. (2) Root ke saath kya karna padega: ek Map<tabId, EditorState> root service mein, manual create(tabId)/destroy(tabId), har component ko apna tabId pass karna, aur cleanup bhoolne pe memory leak. Yani tum manually ek injector re-implement kar rahe ho. Ye argument team ko convince karta hai. (3) Sahi design: @Injectable() (bina providedIn) + @Component({selector: 'app-editor', providers: [EditorState]}). Har <app-editor> instance ka apna EditorState; component destroy pe uska ElementInjector destroy, service ka ngOnDestroy/DestroyRef chalta hai, subscriptions aur timers cleanup ho jaate hain (Ch1 S8). Zero manual bookkeeping. (4) Agar tabs ek route se drive hote hain: route-level providers (per-route injector) — tab lifetime route ka hoga. (5) Shared cheezein root mein rehne do: DocumentApi (stateless HTTP) providedIn: 'root'; EditorState component-scoped; dono ek doosre ko inject kar sakte hain (scoped service root wali ko dekh sakti hai, ulta nahi — Ch7 S6). (6) Trap jo batana chahiye: agar koi root-level service (interceptor, error handler) ko current editor ka state chahiye, toh use nahi milega (Ch7 S6 ka scope trap) — wo root se resolve hoti hai. Data explicitly pass karo. (7) Team ke liye rule: "providedIn: 'root' stateless services ke liye default hai. Jaise hi service mein mutable state aaye, uska scope ek deliberate decision ban jaata hai: root (app-wide), route (feature session), ya component (instance)." Aur ek test: "kya is state ko kabhi reset karna padega?" Agar haan, toh scope galat hai.

S-03

Singleton

Covers: Per-injector singleton · "Singleton" ka asli matlab · Global state ke khatre · Stateless vs stateful · Singleton guard · SSR ka jaal

1. What is it?

Angular mein "singleton" ka matlab ek instance per injector hai — poore app ke liye nahi, poore process ke liye toh bilkul nahi.

  Root injector          --> CartService instance #1
  Route injector (admin) --> CartService instance #2   (agar wahan provide ki)
  Component injector     --> CartService instance #3   (agar wahan provide ki)

  ► "Singleton" ek SCOPE ka property hai, class ka nahi.
  ► providedIn: 'root'  = app-wide singleton (ek root injector hai)
  ► providers: [X]      = us injector ka singleton
  ► providedIn: 'any'   = har lazy injector ka apna "singleton" (S2!)

Aur SSR mein ek aur layer: har request ka apna root injector hota hai. Toh "app-wide singleton" server pe "request-wide singleton" ban jaata hai — ye distinction ek poori class of bugs ka source hai.

2. Why do we need it?

Kyunki kuch cheezein sach mein shared honi chahiye: auth token, HTTP client, cart, feature flags. Agar har component apna AuthService banata, toh login ek jagah hota aur baaki jagah pata hi na chalta.

Par shared state ka doosra chehra bhi hai: global mutable state — jo software engineering ki sabse purani samasya hai. DI ise "organized global state" bana deta hai (typed, injectable, testable), par global-ness khatam nahi karta.

3. What problem does it solve?

  • Shared state → ek source of truth.
  • Resource reuse → ek HTTP client, ek WebSocket connection.
  • Cross-component coordination → bina prop drilling ke (Ch4 S9).
  • Expensive initialization → ek baar (config parse, worker spawn).

4. Internal Working — instance kab banta hai

5. Execution Flow — SSR mein singleton

  BROWSER:
    Ek app instance --> ek root injector --> ek CartService
    Page reload --> sab kuch naya

  SERVER (SSR):
    Request 1 --> ek naya root injector --> CartService #1
    Request 2 --> ek naya root injector --> CartService #2
    (Angular Universal har request ke liye app bootstrap karta hai)

  ► Toh "app-wide singleton" server pe "request-wide" hai. Ye SAHI hai.
  ► KHATRA: module-level state (class ke bahar), jo requests ke beech share ho jaata hai:

      @Injectable({providedIn: 'root'})
      export class UserService {
        get user() { return cachedUser; }               // request 2 ko request 1 ka user!
      }

  ► Ye ek SECURITY bug hai (data leak between users), sirf correctness bug nahi.

6. Architecture — singleton guard

// Problem: koi galti se service ko do jagah provide kar de
// (root mein aur ek route mein) --> do instances --> state split

@Injectable({ providedIn: 'root' })
export class AuthService {
  constructor() {
    const existing = inject(AuthService, {                    // Ch7 S7
      skipSelf: true,     // mera nahi, parent chain se
      optional: true,     // na mile toh null
    });
    if (existing) {
      throw new Error(
        'AuthService pehle se provide hai. Ise dobara providers mein mat daalo.'
      );
    }
  }
}

// Ab agar koi route ke providers mein AuthService daalta hai,
// uska constructor parent chain mein root wala dhundh lega --> loud error.
// Silent state split se behtar hai loud crash.

10. Code Explanation — Line by Line

  • L1–L2: Service instance per-request hai, par requestCache module scope mein hai — wo module ke saath ek baar banta hai aur poore Node process mein share hota hai.
  • L3: Result: ek user ka data doosre user ko dikh sakta hai. Ye ek security incident hai, "cache bug" nahi. Aur browser mein ye kabhi reproduce nahi hoga (ek user, ek process).
  • L4: Fix simple hai: state class ke instance field mein rakho. Angular Universal har request pe naya root injector banata hai, toh naya instance, naya Map.
  • L5–L7: Module-level state ke saare roop: let variables, mutable exported objects, aur static class fields. Teeno SSR mein share hote hain. Static readonly constants theek hain (immutable).
  • Yaad rakhne ka tareeka: "Har request ka apna injector hota hai, apna module nahi." Agar tumhari app SSR karti hai (ya kabhi karegi), toh har let jo module scope mein hai, wo ek potential cross-user leak hai. ESLint rules (no-mutable-exports) aur code review mein ye pehla check hona chahiye.

11. Common Mistakes

Galtiyan
  • Module-level mutable state — SSR mein cross-request leak. Security bug.
  • Service ko providedIn: 'root' aur component providers mein — do instances, silent state split.
  • "Singleton" ko "global" samajhna — wo per-injector hai, aur SSR mein per-request.
  • Stateful service ka lifetime app-wide rakhna aur phir manual reset() likhna (Ch7 S6).
  • Constructor mein side effects (HTTP, timers) — lazy instantiation ki wajah se non-deterministic timing.
  • Static class fields se shared state banana — wahi module-scope problem.
  • Two browser tabs ko "do instances" samajhna — wo do processes hain, service ka koi role nahi.

12. Best Practices

Production rules
  • State hamesha instance fields mein. Module-level let/mutable exports ban karo (lint rule).
  • Stateless services root mein — safe. Stateful services ka scope ek deliberate decision.
  • Singleton guard (skipSelf + optional) un services pe jinka do instance hona bug hai.
  • Constructor patla: sirf DI. Side effects methods mein ya APP_INITIALIZER mein.
  • "Kya is state ko reset karna padega?" — agar haan, toh scope galat hai.
  • SSR: har module-level variable ko cross-user leak maano jab tak wo immutable na ho.

13. Performance

Singleton ka faayda hi performance hai: ek instance, ek initialization, ek shared cache. providedIn: 'root' ka per-component cost zero hai.

Lazy instantiation ka nuance: service pehle inject() pe banti hai. Agar ek bhaari service (config parse, crypto init, worker spawn) ka pehla consumer koi random component hai, toh wo cost ek random user interaction pe padegi — aur profiler mein wo interaction slow dikhegi, service nahi. Deterministic timing chahiye toh provideAppInitializer se startup pe warm karo (Ch3 S4), par tab wo startup path pe hai.

Memory: app-wide singleton kabhi destroy nahi hota. Agar usme unbounded cache hai, wo poore session mein badhta rehta hai (Ch1 S8). LRU/TTL lagao.

14. Edge Cases

  • SSR: har request ka apna root injector → app-wide singleton = request-wide.
  • TestBed har test ke liye naya root injector banata hai → test isolation free.
  • Root service ka ngOnDestroy browser mein kabhi nahi chalta (page unload pe process khatam).
  • SSR mein root injector destroy hota hai request ke baad → ngOnDestroy chalta hai (cleanup zaroori).
  • Angular Elements / micro-frontends: providedIn: 'platform' se kai apps ek instance share karte hain.
  • HMR (hot module replacement) dev mein module reload karta hai → module-level state reset, service instance nahi. Confusing behavior.
  • Do browser tabs = do processes = koi sharing nahi (jab tak BroadcastChannel/localStorage na use karo).

15. Interview Questions

EASY Angular mein singleton kya hota hai?

Ek instance per injector. providedIn: 'root' se root injector ka ek instance — jo app-wide singleton hai (browser mein). Par agar wahi service component ke providers mein bhi hai, toh us component ka apna instance banega. Yani "singleton" scope ka property hai, class ka nahi.

EASY Service ka constructor kab chalta hai?

Pehle inject() pe (lazy). Registration ke waqt sirf ek record banta hai; instance tab banta hai jab koi pehli baar maange, aur phir cache ho jaata hai. Isliye constructor mein side effects (HTTP calls, timers) daalna galat hai — unki timing non-deterministic ho jaati hai (kaunsa component pehle inject karega?).

MEDIUM Ek hi service ke do instances kaise ban sakte hain?

Agar wo providedIn: 'root' hai aur kisi component/route ke providers array mein bhi hai — toh us subtree mein alag instance banega ("nearest wins", Ch7 S6). Symptom: state split, aur koi error nahi. Doosra tareeka: providedIn: 'any' (har lazy injector ka apna). Bachne ke liye: singleton guard (inject(Self, {skipSelf: true, optional: true}) constructor mein, mile toh throw karo).

MEDIUM Stateless aur stateful singleton mein farak kyun matter karta hai?

Stateless service (OrderApi) ka instance sirf ek namespace hai — do instances hote toh bhi koi farak nahi padta, isliye providedIn: 'root' hamesha safe hai. Stateful service (CartStore) mein shared mutable state hai — ab uska scope ek correctness decision hai: kaun ise dekhta hai, kab tak jeeti hai, kab reset hoti hai. Aur agar tumhe kabhi manual reset() likhna pada, toh scope galat tha (Ch7 S6: scope = lifetime).

HARD SSR mein singleton ke saath kya khatra hai?

Angular Universal har request ke liye app bootstrap karta hai — naya root injector, naye service instances. Toh "app-wide singleton" server pe "request-wide" hai, jo bilkul sahi hai. Khatra module-level state se hai: let cache = new Map() module scope mein, ya mutable exported object, ya static class field. Module ek baar load hota hai aur poore Node process mein share hota hai — har request ka apna injector hota hai, apna module nahi. Result: User A ka data User B ki response mein aa sakta hai. Ye ek security incident hai, correctness bug nahi — aur browser mein kabhi reproduce nahi hoga (ek user, ek process). Isliye: (1) saara state service ke instance fields mein rakho; (2) module-level let aur mutable exports ko lint rule se ban karo (no-mutable-exports); (3) static fields sirf readonly constants ke liye; (4) SSR mein root injector request ke baad destroy hota hai, toh ngOnDestroy/DestroyRef se cleanup zaroori hai (browser mein wo kabhi nahi chalta, isliye log bhool jaate hain); (5) third-party libraries bhi module-level state rakh sakti hain — unhe SSR mein audit karo. Mantra: "Har request ka apna injector hota hai, apna module nahi."

Teeno bugs ek hi root cause ke alag chehre hain: state ka scope uske lifetime se match nahi karta (Ch7 S6). (a) Purana filter: providedIn: 'root' ka matlab hai "iska lifetime app ka lifetime hai". Filter ka lifetime page visit ka hai. Manual reset() ka option hai, par wo bhoola jaayega. (b) Do parallel views: ek root instance, do consumers — dono ko same filter dikhega. Ye batata hai ki state actually per-view hai. (c) SSR leak: agar filter module-level variable mein hai toh cross-request leak; agar service instance field mein hai toh SSR safe hai (har request ka apna injector) — is bug ka hona batata hai ki kahin module-level state hai. Pehle wahi dhundho. Fixes, sahi order mein: (1) Sabse pehle poochho: ye state navigable hai kya? Filter, page, sort — user in-ka link share karega, refresh karega, back button dabayega. Jawab: haan. Toh iska ghar URL hai (Ch4 S9), koi service nahi. Query params + withComponentInputBinding() se seedha component input mein. Ye teeno bugs ek saath fix karta hai: naya page = naye params; do views = do URLs (ya do outlets, apne params); SSR = server ko URL se hi sab pata hai (aur koi shared state nahi). (2) Agar URL possible nahi (bahut bada filter object): route-scoped store — @Injectable() + route providers. Route chhoda, injector destroy, state gayi. Par split-screen abhi bhi share karega agar dono ek hi route mein hain → tab component-scoped. (3) SSR audit: let/mutable exports/static fields dhundho. State instance field mein daalo. (4) Guard: FiltersService agar accidentally do jagah provide ho rahi hai toh singleton guard se loud error. (5) Test: ek e2e test jo filter set kare, navigate kare, wapas aaye, aur assert kare ki filter fresh hai. Aur SSR test jo do requests bhej ke isolation verify kare. Lesson jo bolna hai: "providedIn: 'root' ek default nahi, ek lifetime declaration hai. Aur jo state URL mein honi chahiye, wo kisi service mein nahi honi chahiye."

Business Logic

Covers: Business logic kya hai · Component vs service vs pure function · Domain layer · DTO vs domain model · Testability · Framework-free core

1. What is it?

Business logic = wo rules jo tumhare domain se aate hain, framework se nahi.

// Business logic (domain rules)
"Order 30 din tak refundable hai, agar wo captured state mein ho"
"Delhi mein GST 18%, baaki states mein 12%"
"Cart mein 10 se zyada items nahi"
"Discount tabhi lagega jab total > 500 ho aur user premium ho"

// NOT business logic
"HTTP call /api/orders pe jaati hai"          // infrastructure
"Dropdown escape pe band hota hai"              // UI behavior
"Loading spinner 200ms ke baad dikhta hai"     // UX
"Signal set karne pe CD chalti hai"            // framework

Ye distinction critical hai kyunki business logic tumhare app ki sabse lambi jeene wali cheez hai. HTTP client badlega, Angular version badlega, UI redesign hoga — par "Delhi mein 18% GST" 10 saal chalega. Isliye usko sabse kam framework se coupled hona chahiye.

2. Why do we need it (alag rakhna)?

Ek rule jo OrderListComponent.ngOnInit() ke andar likha hai, wo:

  • Test karne ke liye TestBed + DOM chahiye (100ms per test vs 0.1ms)
  • Doosre component mein reuse nahi ho sakta → copy-paste → do jagah diverge
  • Backend team ke saath share nahi ho sakta (jo shayad same rule TypeScript mein likh rahi hai)
  • Angular chhodne pe dobara likhna padega

3. What problem does it solve?

  • Test speed → pure functions milliseconds mein.
  • Reuse → ek rule, kai jagah.
  • Portability → domain core framework-free.
  • Reasoning → rules ek jagah, scattered nahi.

10. Code Explanation — Line by Line

  • L1: Sabse simple aur sabse powerful. Rule pure hai, test ek line hai, aur wo "kaunsa clock" ke sawaal se azaad hai. Caller (jo already impure hai) time deta hai.
  • L2–L3: Jab time ki zaroorat kai jagah hai (components, services), toh Clock ko ek abstract class token banao. Test mein useValue se frozen time — koi jest.useFakeTimers() nahi, koi global patching nahi.
  • L4: Ye rule ko impure bana deta hai. Uska test ab system time pe depend karta hai → flaky (midnight ke aas-paas fail hoga, timezone change pe fail hoga, CI mein alag). Ye ek real, bahut common flaky-test source hai.
  • L5: Same principle randomness ke liye. crypto.randomUUID() directly call karne se snapshot tests impossible ho jaate hain. Inject karo, test mein counter se deterministic ids do.
  • General principle: impurity ko boundary pe dhakelo. Domain core pure ho; time, randomness, HTTP, storage — sab bahar. Ye "functional core, imperative shell" pattern hai, aur wo Angular apps mein bilkul practical hai.

11. Common Mistakes

Galtiyan
  • Business rules component mein — untestable, unreusable, aur template se call hone pe har CD cycle mein chalti hain.
  • Rules ko service mein daalna jab wo pure functions ho sakti thi — DI ka cost, test mein TestBed.
  • Date.now()/Math.random() rules ke andar — flaky tests, midnight bugs.
  • DTO ko poore app mein circulate karna — backend ka rename 50 files todta hai; status: string se impossible states.
  • Domain folder mein Angular imports — core portable nahi raha.
  • Validation sirf UI mein — server pe bhi chahiye; aur domain rules dono jagah share ho sakti hain.

12. Best Practices

Production rules
  • Business rules = pure functions. Framework-free domain/ folder (lint rule se Angular imports ban).
  • Impurity (time, random, I/O) ko boundary pe dhakelo — parameter ya injectable abstraction.
  • DTO ≠ domain model. Ek mapper layer, ek jagah. Union types se impossible states khatam (Ch2 S2).
  • Rules ko computed() se consume karo — memoized, template mein call nahi.
  • Clock, IdGen jaise abstractions test mein deterministic banate hain.
  • Domain tests fast honi chahiye (100+ tests, <100ms total).

13. Performance

Pure functions ka do faayde: (1) Memoization — computed() mein daal do, dependencies stable toh dobara nahi chalte. Template se direct call (Ch5 S1) har CD cycle mein chalti hai. (2) Tree shaking — per-function; ek service class atomic hai.

Aur ek non-obvious: pure functions ko Web Worker mein bhej sakte ho (koi DI, koi framework). Bhaari calculations (pricing engine, 10,000 rows ka aggregation) ko worker mein chalane ke liye unka framework-free hona pehli shart hai.

Test performance bhi asli metric hai: 500 domain tests 200ms mein chalti hain; 500 TestBed tests 60 seconds. Ye developer velocity ka farak hai.

14. Edge Cases

  • Kuch rules ko async data chahiye (feature flags, config) — unhe parameter banao, service inject mat karo rule ke andar.
  • Validation rules frontend aur backend dono mein — agar dono TypeScript hain toh shared package (monorepo).
  • Money ko number mat rakho — floating point (0.1 + 0.2). Paise/cents integer, ya ek Money value object.
  • Domain model mein Date vs timestamp — timezone ka poora dukh; Temporal API aa rahi hai.
  • parseStatus() jaisa runtime validation zaroori hai — TypeScript types runtime pe nahi hote (Ch2 S9). Zod/valibot ya manual guards.
  • Pure function jo bade objects clone karti hai, wo "pure" hai par slow — purity aur performance alag axes hain.

S-05

Stateful Services & Stores

Covers: Signals store pattern · private writable + asReadonly · computed derivations · Mutations as methods · Async state · NgRx kab · Immutability

1. What is it?

Stateful service (ya "store") = ek service jiske paas shared mutable state hai. Signals ke zamaane mein uska ek standard shape ban gaya hai:

@Injectable({ providedIn: 'root' })
export class CartStore {
  // 1. PRIVATE writable signals — sirf ye class inhe badal sakti hai
  private _items = signal<readonly CartItem[]>([]);
  private _coupon = signal<Coupon | null>(null);

  // 2. PUBLIC read-only views
  readonly items = this._items.asReadonly();
  readonly coupon = this._coupon.asReadonly();

  // 3. DERIVED state — computed, memoized
  readonly count = computed(() => this.items().length);
  readonly subtotal = computed(() => sumPrices(this.items()));           // pure fn (S4)
  readonly total = computed(() => applyDiscount(this.subtotal(), this.coupon()));

  // 4. MUTATIONS — explicit, named, immutable updates
  add(item: CartItem) { this._items.update(l => [...l, item]); }
  remove(id: string) { this._items.update(l => l.filter(i => i.id !== id)); }
  clear() { this._items.set([]); this._coupon.set(null); }
}

Chaar hisse: private writable → public readonly → computed derivations → named mutations. Ye poore chapter ka sabse practical pattern hai.

2. Why do we need it?

Kyunki bina discipline ke shared state chaos ban jaati hai. Agar items ek public writable signal hai, toh koi bhi component store.items.set([]) kar sakta hai — aur ab "cart kisne khaali kiya" ka jawab dhundhne mein tumhe poora codebase grep karna padega.

Private writable + public readonly ek compile-time guarantee deta hai: state ke saare change points is class ke andar hain. Ye encapsulation hai (Ch2 S5), signals mein.

3. What problem does it solve?

  • Uncontrolled mutations → sirf named methods se.
  • Derived state ka sync → computed(), kabhi stale nahi.
  • Manual change detection → signal write → sirf consumers update.
  • Boilerplate → NgRx ke actions/reducers/effects/selectors ke bina.

4. Internal Working — chaar hisse detail mein

  1. PRIVATE WRITABLE
       private _items = signal<readonly CartItem[]>([]);
       ► `readonly CartItem[]` --> array ko mutate nahi kar sakte (Ch2 S7)
       ► private --> bahar se set/update nahi

  2. PUBLIC READ-ONLY
       readonly items = this._items.asReadonly();
       ► type: Signal<readonly CartItem[]>  (koi set/update nahi)
       ► RUNTIME pe wahi signal hai — koi copy nahi, koi cost nahi
       ► Compile-time enforcement (Ch2 S5 ka pattern)

  3. COMPUTED (derived)
       readonly total = computed(() => applyDiscount(this.subtotal(), this.coupon()));
       ► Memoized: dependencies na badlein toh function chalta hi nahi
       ► Lazy: koi na padhe toh evaluate hi nahi hota
       ► Glitch-free: subtotal aur coupon dono badlein toh total EK baar

  4. MUTATIONS (commands)

11. Common Mistakes

Galtiyan
  • Public writable signals — koi bhi store.items.set([]) kar sakta hai. asReadonly().
  • Mutation (items().push()) — signal notify nahi karta. readonly T[] type se compile-time block karo.
  • Teen alag signals (loading/error/data) — impossible states. Discriminated union ya resource().
  • Store mein UI concerns (toast, navigate, dialog) — store headless rahe (S1).
  • effect() mein state write karke doosri state derive karna — computed() use karo. Effects side effects ke liye hain.
  • Har feature ke liye NgRx uthana — boilerplate ka cost bina faayde ke.
  • Store ko providedIn: 'root' jab wo flow-scoped hai (Ch7 S6).

12. Best Practices

Production rules
  • Chaar hisse: private writable → asReadonly() public → computed() derivations → named mutations.
  • readonly T[] / Readonly<T> types — mutation compile-time pe block.
  • Derivations pure functions se (S4). Store sirf orchestrate kare.
  • Invariants mutations mein enforce karo (MAX_ITEMS check).
  • Async state: discriminated union ya resource(). Kabhi teen alag signals nahi.
  • Store headless: errors return karo (Result<T, E>), toast component dikhaye.
  • Scope = lifetime. Flow-scoped stores route providers mein.

13. Performance

computed() memoized aur lazy hai: dependencies na badlein toh function chalta hi nahi; koi na padhe toh evaluate hi nahi hota. Aur glitch-free: agar subtotal aur coupon dono ek hi mutation mein badlein, total ek hi baar recompute hoga (do baar nahi).

asReadonly() ka runtime cost zero hai — wo wahi signal object return karta hai, sirf type alag hai (Ch2 S5 ka structural typing). Koi wrapper, koi proxy.

Immutable updates ka cost: [...l, item] O(n) hai. 10,000-item cart mein har add pe 10,000 element copy. Bade collections ke liye: (a) Map/Set with structural sharing, (b) Immer-style patches, ya (c) items ko normalize karo (id → item map + id array). Par pehle measure karo — 1000 items tak spread bilkul theek hai.

14. Edge Cases

  • signal.set() reference equality (Object.is) check karta hai — same reference = no notification. Custom equal option se badal sakte ho.
  • computed() ke andar signal write karna → error (dev mode). Computed pure hona chahiye.
  • effect() mein apni hi dependency likhna → infinite loop (Ch5 S6).
  • Store mein effect() ke liye injection context chahiye (constructor/field) — ya {injector} option (Ch7 S2).
  • resource() params signal badalne pe purani request abort karta hai (AbortSignal) — race conditions structurally gayab.
  • Scoped store ka ngOnDestroy chalta hai; root store ka nahi (S3).
  • linkedSignal() (v19+) — ek writable signal jo kisi source se reset hota hai (e.g. selection jo list badalne pe clear ho).

15. Interview Questions

EASY Signals store ka standard shape kya hai?

Chaar hisse: (1) private writable signals (private _items = signal([])), (2) public read-only views (readonly items = this._items.asReadonly()), (3) derived state computed() se, (4) named mutation methods jo immutable updates karti hain. Isse state ke saare change points class ke andar rehte hain — compile-time enforced encapsulation.

EASY asReadonly() ka kya faayda?

Public API mein Signal<T> milta hai jisme set/update hai hi nahi — toh consumer state directly badal nahi sakta (compile error). Runtime cost zero hai: wahi signal object return hota hai, sirf type narrow hai. Isse "cart kisne clear kiya" ka jawab hamesha store ke andar milta hai.

MEDIUM Signal ke andar array ko push karne se UI update kyun nahi hota?

signal.set() reference equality (Object.is) se check karta hai ki value badli ya nahi. items().push(x) se array mutate hota hai par reference wahi rehta hai — aur agar tum set(sameArray) bhi karo toh signal use no-op maan lega. Fix: immutable update (update(l => [...l, x])). Aur behtar: type ko readonly T[] banao — tab push() compile hi nahi hoga (Ch2 S7).

MEDIUM Loading/error/data ke liye teen signals kyun galat hain?

Impossible states representable ban jaati hain: loading=true + error='x' + data=[...] ek saath. Template mein har combination handle karni padti hai, aur ek bhoolne pe bug. Fix: discriminated union ({status: 'loading'} | {status: 'success', data} | {status: 'error', message}) — @switch se exhaustive handling, aur har branch mein type narrow (Ch2 S2). Ya built-in resource() jo loading/error/value + auto-abort deta hai.

HARD Signals store aur NgRx mein kab kya chunoge?

Signals store 30 lines mein wahi core guarantees deta hai jo NgRx 120 mein: unidirectional flow (mutations sirf methods se), derived state (computed = selectors), immutability. NgRx ka extra value: (1) time-travel debugging (har change ek serializable action hai) — agar tumhe "user ne kya kiya" replay karna hai toh ye killer feature hai; (2) convention enforcement bade teams mein (40 devs, ek hi pattern); (3) Effects ka RxJS ecosystem complex async orchestration ke liye; (4) global state inspector. Signals store ka extra value: (1) scoping — store ko route/component injector mein daalo, lifetime automatic (Ch7 S6); NgRx mein feature state manually register/unregister; (2) zero boilerplate; (3) fine-grained CD; (4) type inference. Decision: complexity se decide karo, size se nahi. 200-component app jisme state simple hai — signals store. 20-component app jisme optimistic updates, undo/redo, offline sync, aur audit trail hai — NgRx. Middle ground: @ngrx/signals (SignalStore) — NgRx ki conventions + signals ki ergonomics, aur wo aaj default recommendation ban raha hai. Jo galat reason hai: "NgRx best practice hai" ya "bade apps mein sab use karte hain". Boilerplate ka cost real hai aur wo har feature pe lagta hai.

SCENARIO Ek collaborative document editor: optimistic updates (type karo, turant dikhe), server confirm/reject kar sakta hai, undo/redo chahiye, aur do users ek saath edit karein toh conflict resolve karna hai. State architecture design karo.

(1) Do state layers: confirmed state (server se confirmed) aur pending operations (local, unconfirmed). Displayed state = computed(() => applyOps(confirmed(), pending())). Ye separation optimistic updates ka core hai — reject hone pe sirf pending se op hatao, confirmed ko chhuo mat. Ek single doc signal rakhoge toh rollback impossible ho jaayega. (2) Operations as data: har edit ek serializable operation ({type: 'insert', pos, text, id, clientSeq}), na ki direct mutation. Isse undo/redo (inverse op), replay, aur conflict resolution sab possible hote hain. Ye NgRx-style "actions" ka value hai — par tumhe poora NgRx nahi chahiye, sirf ye discipline. (3) Undo/redo: do stacks of operations (undo, redo). undo() = last op ka inverse apply karo aur redo stack pe daalo. Critical: undo confirmed state pe nahi, operation log pe kaam karta hai. Aur collaborative mein "mera undo" chahiye, "kisi ka bhi undo" nahi — toh ops mein clientId. (4) Conflict resolution: ye sabse gehri baat hai. Do options: OT (Operational Transformation) — server ops ko transform karta hai; ya CRDT (Yjs, Automerge) — ops commutative hain, koi central server logic nahi. Naya code likh rahe ho toh CRDT (Yjs) — battle-tested, offline-first, aur Angular ke saath ek adapter se jud jaata hai. Khud OT likhna ek saal ka project hai; ye judgment interview mein bolna zaroori hai. (5) Angular layer: DocumentStore (component-scoped, per open document — Ch7 S6) jo Yjs doc ko wrap kare, uske observe events ko signals mein map kare (signal + Y.Doc.on('update')), aur commands expose kare. Store headless: koi toast, koi navigation. (6) Rejection handling: server reject kare toh pending op hatao, ek Result return karo, component decide kare ki toast dikhana hai ya silently retry (S1: store headless). (7) Performance: har keystroke pe poora doc signal set karoge toh 10,000-word doc pe jank. Solution: granular signals (per-paragraph/per-block), ya Yjs ke incremental updates + computed per block. (8) Persistence: pending ops ko IndexedDB mein (offline), reconnect pe flush. (9) Kya NgRx? Yahan actions/reducers ka model natural hai (ops = actions), aur time-travel debugging bahut valuable hai. Par CRDT library already state manage kar rahi hai — do state managers ka clash. Recommendation: Yjs source of truth, ek patla signals store adapter, NgRx nahi. Punchline: optimistic updates ka poora architecture "confirmed state + pending ops" hai; jo bhi library ho, ye separation hi asli design decision hai.

  • L3–L4: Route ke providers ek naya EnvironmentInjector banate hain us route (aur uske children) ke liye. Teeno steps usi injector se resolve karte hain → same instance. Ye "nearest wins" ka natural nateeja hai (Ch7 S6).
  • L5: Yahi poore chapter ka thesis hai: state ka lifetime injector ke lifetime se aata hai, tumhare reset() code se nahi. Har manual reset ek jagah hai jahan tum bhool sakte ho. Aur "purana data dikh raha hai" wali bug class isi se aati hai.
  • Ek nuance: agar wizard ka state URL mein hona chahiye (step number shareable ho, back button kaam kare), toh wo router ka kaam hai, store ka nahi (Ch4 S9). Store sirf wo state rakhe jo ephemeral hai (form drafts, validation state).

11. Common Mistakes

Galtiyan
  • Generic event bus — type safety gayi, dependency graph invisible, "kaun sunta hai" ka koi jawab nahi.
  • Sibling components ko viewChild se ek doosre ka reference dena — encapsulation toot gayi.
  • Har shared cheez providedIn: 'root' — flow-scoped state global ban gayi.
  • Prop drilling ko 5 levels tak sehna — scoped store ka signal.
  • Navigable state (filter/page/sort) service mein — refresh pe gayi, link share nahi hota (Ch4 S9).
  • toSignal() bina initialValue ke — type T | undefined, aur sync value na ho toh undefined.
  • Cross-tab sync ke liye service — do tabs = do processes. BroadcastChannel chahiye.

12. Best Practices

Production rules
  • 1-2 levels: input/output. 3+ ya siblings: scoped store (common ancestor ke providers mein).
  • Navigable state (filter, page, sort, id, tab) → URL, kisi service mein nahi.
  • Event bus ban karo. Typed stores + explicit inject() dependencies.
  • Scope = lifetime: wizard → route, editor → component, auth → root.
  • RxJS interop: toSignal() (initialValue ke saath) aur toObservable() jab operators chahiye.
  • Cross-tab: BroadcastChannel/storage events, SSR-safe wrapper ke saath.
  • Store headless: consumers decide karein UI kya dikhaye.

13. Performance

Scoped store + signals ka bada faayda: jab ek deep leaf state badalti hai, sirf consumers ki CD chalti hai. Prop drilling mein raste ka har intermediate component dirty hota hai (input badla), toh 5-level tree mein 5 components ki CD.

Concrete: 5-level nesting, ek value badalne pe — prop drilling: 5 components; scoped signal store: 1-2 components (jo signal padhte hain).

Event bus ka chhupa cost: har subscriber ek subscription hai; bhoolne pe leak (Ch1 S8). Aur RxJS Subject ke saath har emit sab subscribers ko chalata hai, chahe unhe zaroorat ho ya nahi. Signals mein computed lazy hai — koi na padhe toh evaluate hi nahi hota.

14. Edge Cases

  • Scoped store @if ke andar wale component pe provide kiya → toggle pe destroy+recreate, state loss. Stable ancestor pe rakho.
  • Projected content (ng-content) ka DI parent ke injector se resolve hota hai → container ke viewProviders use nahi dikhte (Ch7 S6).
  • toSignal() subscription DestroyRef se cleanup hota hai — injection context chahiye.
  • toObservable() signal ko ek effect se observe karta hai — har change pe emit, par sync nahi (microtask).
  • Do routed outlets (siblings) ka koi common component ancestor nahi → root service ya router state.
  • BroadcastChannel SSR pe nahi hai; storage event same-origin tabs ke liye (aur wo emit karne wale tab mein fire nahi hota).
  • Scoped store ke andar effect() — store destroy pe effect bhi cleanup.

15. Interview Questions

EASY Do sibling components kaise baat karte hain?

Agar nesting 3+ levels hai toh us ancestor ke providers mein ek scoped store rakho aur dono inject() karein — zero prop drilling, aur signals ki wajah se sirf consumers ki CD chalti hai.

EASY Scoped store kya hai?

@Injectable() (bina providedIn) service jise tum kisi component ya route ke providers array mein daalte ho. Us subtree ka apna instance banta hai, aur us injector ke destroy hone pe service bhi destroy (ngOnDestroy chalta hai). Isse state ka lifetime automatically scope se bandh jaata hai — manual reset ki zaroorat nahi.

MEDIUM Event bus pattern kyun bura hai?

(1) Type safety — payload aksar any; rename karo, koi compile error nahi. (2) Invisible dependency graph — "kaun emit karta hai" aur "kaun sunta hai" ka jawab dhundhne ke liye string grep. Compiler kuch nahi bata sakta. (3) Ordering aur error handling tumhara kaam. (4) Leaks — har subscribe ka cleanup. (5) Refactoring nightmare. Fix: typed store + explicit inject() — dependency compiler ko dikhti hai, IDE "find usages" kaam karta hai, aur computed() lazy hai.

MEDIUM toSignal aur toObservable kab use karte hain?

toSignal(obs$, {initialValue}) — jab source naturally stream hai (WebSocket, router events, HttpClient) aur tum use signal ki tarah consume karna chahte ho; subscription DestroyRef se auto-cleanup hoti hai (injection context chahiye). toObservable(sig) — jab tumhe RxJS operators chahiye jo signals mein nahi hain: debounceTime, switchMap (cancellation), retry. Modern alternative: resource() — params signal se, auto-abort, loading/error built-in — jo search-as-you-type ke 80% cases cover kar leta hai.

1. What is it?

Service architecture = ye faisla ki tumhari services kaunsi layers mein baithengi, aur kaun kis pe depend kar sakta hai. Chaar layers, aur dependency ka ek hi direction:

2. Why do we need it?

Bina layers ke har cheez har cheez pe depend karti hai. 18 mahine baad: ek HTTP endpoint badalne se 40 components toot-te hain, ek business rule teen jagah likhi hai (aur ab diverge ho chuki hai), aur naya dev nahi bata sakta ki "order total kahan calculate hota hai".

Layers ek hi cheez enforce karti hain: dependency direction. Wo direction hi architecture hai; baaki sab naming hai.

3. What problem does it solve?

  • Ripple effects → backend badla, ek mapper badla.
  • Duplicated rules → domain layer single source.
  • Untestable code → har layer independently testable.
  • Onboarding → "business rules yahan hain" ek folder.

10. Code Explanation — Line by Line

  • L1: Data layer ka kaam hai infrastructure errors (HTTP status codes) ko domain errors mein badalna. Upar wali layers ko kabhi HttpErrorResponse nahi dikhna chahiye — warna component error.status === 404 likhega, aur ab tumhara UI HTTP se coupled hai.

  • L3: Presentation decide karta hai ki error kaise dikhana hai — toast, inline, ya silently retry. Store ne ye decide kiya hota toh wo headless na rehti (S1).
  • L4: ErrorHandler sirf unexpected errors ke liye (bugs). Expected errors wahan pahunchne hi nahi chahiye — warna Sentry mein noise.
  • L6: Interceptors data layer ka concept hain, par unhe root injector mein register karna padta hai. Iska matlab: unhe route-scoped tokens nahi dikhte (Ch7 S6 ka scope trap). Request-scoped data HttpContext se bhejo.

12. Best Practices

Production rules
  • Dependency direction hamesha andar ki taraf: presentation → application → data → domain.
  • Domain framework-free (lint rule se enforce, S4).
  • Data layer domain models return kare, DTOs nahi.
  • Components sirf application layer (store/facade) se baat karein.
  • Facade tabhi jab wo orchestrate kare; pass-through facade delete karo.
  • Errors: HTTP → domain error (data), Result (application), UI decision (presentation).
  • Architecture ko lint rules se enforce karo (no-restricted-imports, boundaries plugin).
  • Chhoti app? Do layers kaafi. Layers zaroorat se aayein, ideology se nahi.

13. Performance

Bundle: layers ke saath ek feature ka poora graph ek chunk mein aa sakta hai — agar features ek doosre ko import na karein. Circular imports (Ch7 S8) lazy chunking ko todte hain: bundler cycle ke dono sirey ek chunk mein daal deta hai, aur tumhara "lazy" route initial bundle mein aa jaata hai.

Domain layer ka bonus: framework-free rules Web Worker mein chal sakte hain (S4). 10,000-row aggregation main thread se hataana tabhi possible hai jab wo code Angular pe depend na kare.

Indirection ka cost: ek facade jo sirf forward karta hai = ek extra injectable, ek injector entry, ek getter chain. Micro, par uska asli cost cognitive hai: har debug session mein ek extra file.

15. Interview Questions

EASY Service architecture ki layers kaunsi hain?

Presentation (components), Application (stores/facades — orchestration aur state), Data (API, mappers, storage), aur Domain (pure rules aur models). Dependency direction hamesha andar ki taraf: presentation → application → data → domain. Domain kisi pe depend nahi karta — wo framework-free hai.

EASY Component mein HttpClient inject karna kyun galat hai?

Presentation layer data layer se directly baat kar raha hai — dependency direction toot gayi. Consequences: component ko DTO shapes, HTTP errors, aur endpoint URLs ka pata chal jaata hai; backend badalne pe component badalta hai; aur component test karne ke liye HTTP mocking chahiye. Sahi: component → store/facade → API service.

MEDIUM Facade pattern kab use karna chahiye?

Jab wo sach mein orchestrate kare: ek use-case kai services (store + api + notifier) ko coordinate karta ho, ya component ko 4+ services inject karni pad rahi hon, ya ek feature ka public API define karna ho. Nahi jab wo sirf ek store ko forward kar raha ho — wo pass-through facade hai: ek file, ek indirection, zero value. Store ko directly inject karo. "Facade best practice hai" ek reason nahi hai.

MEDIUM Error handling har layer mein kaise hota hai?

Data: HTTP errors ko domain errors mein badlo (404 → OrderNotFoundError) — upar wali layers ko HttpErrorResponse kabhi na dikhe. Application: expected failures ko Result<T, E> se return karo, throw nahi — "refund fail hua" ek outcome hai, exception nahi. Presentation: Result ko UI mein map karo (toast/inline/retry) — ye decision component ka hai, store ka nahi (store headless). Global ErrorHandler: sirf unexpected errors (bugs) → Sentry.

HARD "Dependency direction" ka matlab kya hai aur use kaise enforce karoge?

Matlab: outer layers inner layers pe depend karti hain, ulta kabhi nahi. Domain (rules, models) sabse andar hai aur kisi pe depend nahi karta — na Angular pe, na HTTP pe, na store pe. Data layer domain models return karta hai (DTOs nahi). Application layer domain rules use karta hai aur data layer ko call karta hai. Presentation sirf application se baat karta hai. Kyun: jo cheez sabse kam badalti hai (business rules) wo sabse kam depend karti hai; jo sabse zyada badalti hai (UI, HTTP client) wo bahar hai. Isse change ka blast radius chhota rehta hai. Enforce kaise: (1) Lint rules — no-restricted-imports se domain/ mein @angular/* aur rxjs ban karo; components mein **/data/** ban karo. (2) Boundaries plugin (eslint-plugin-boundaries) ya Nx ka enforce-module-boundaries — layers ko tags do aur allowed edges declare karo. (3) CI mein fail — architecture ek document nahi, ek test hai. Testing Services

Covers: Plain new vs TestBed · Fakes vs mocks · Signals testing · HTTP testing · Store testing · Test speed

1. What is it?

Service testing = class ko isolate karke uska behavior verify karna. Angular mein teen levels hain, aur sabse tez wala aksar sabse sahi hai:

  LEVEL 1: PLAIN `new` (koi Angular nahi)
     const store = new CartStore(fakeApi);
     ► Sirf tab jab class inject() na kare (ya dependencies constructor se aayein)
     ► ~0.1ms per test

  LEVEL 2: TestBed.inject()
     TestBed.configureTestingModule({ providers: [...] });
     const store = TestBed.inject(CartStore);
     ► inject() use karne wali services ke liye
     ► ~5ms per test

  LEVEL 3: TestBed.createComponent() (integration)
     ► Component + template + CD
     ► ~50-100ms per test

  ► Pyramid: 100 pure function tests, 30 service tests, 10 component tests.
  ► Ulta pyramid (sab component tests) = 5-minute test suite = koi nahi chalata.

2. Why do we need it?

Services mein business-critical logic hoti hai (payments, permissions, cart). Component tests unhe cover karti hain bahut dheere aur bahut indirectly — ek edge case test karne ke liye poora DOM setup. Aur jab test fail hoti hai, wo nahi batati ki galti kahan hai.

3. What problem does it solve?

  • Fast feedback → milliseconds, na ki minutes.
  • Precise failures → "calculateTax galat hai", na ki "checkout page toota".
  • Edge case coverage → 20 cases likhna trivial hai.
  • Refactoring safety → behavior lock, implementation free.

11. Common Mistakes

Galtiyan
  • Sab kuch component test se cover karna — slow, brittle, aur failure ka locate karna mushkil.
  • Mocks jo type-safe nahi (jest.fn() objects) — API contract badla, test chup raha.
  • Implementation details assert karna (toHaveBeenCalled) — refactoring mehengi ho jaati hai.
  • httpMock.verify() bhoolna — pending requests silently pass ho jaati hain.
  • effect() test karte waqt TestBed.tick()/flushEffects() bhoolna.
  • Real HTTP ya real timers use karna — flaky tests.
  • Date.now() ko mock karne ke liye global patching — Clock inject karo (S4).
  • Private methods ko as any se test karna — public API ke through test karo.

12. Best Practices

Production rules
  • Pyramid: bahut saare pure-function tests, kuch service tests, kam component tests.
  • Typed fakes > untyped mocks. Ek testing/ folder mein reusable fakes.
  • Behavior assert karo (store.orders()), calls nahi (api.getAll).
  • Signals sync hain — computed turant padho. effect ke liye TestBed.tick().
  • HTTP: provideHttpClientTesting() + httpMock.verify() in afterEach.
  • Time/random ko inject karo (Clock, IdGen) — deterministic tests.
  • Invariants test karo — wahi domain ke rules hain.
  • Test likhna mushkil hai? Design theek karo, test nahi.

EASY Service ko test kaise karte hain?

MEDIUM Fakes aur mocks mein farak?

10. Code Explanation — Line by Line

  • L1: injector.get() yahan ek known workaround hai, aur uske saath ek TODO hai. Ye theek hai. Jo theek nahi wo hai injector.get() ko default pattern banana. Rule: har use ke saath ek comment "kyun".
  • L2: Ye event bus nahi hai, chahe naam similar lage. Farak: (a) typed — EventMap se payload compile-time pe check hota hai (Ch2 S6); (b) ek direction — koi subscribe nahi karta, ek concrete consumer hai (Segment/GA adapter); (c) koi subscription, isliye koi leak nahi. Event bus ka zeher any aur invisible graph hai, "events" nahi.
  • L3: Line count ek symptom hai. Ek 600-line parser jiska ek reason to change hai (markdown spec) bilkul theek hai. Ek 200-line service jo API + tax + CSV karti hai, nahi. Code review mein "ye bada hai" bolna weak argument hai; "iske chaar stakeholders hain" strong hai (S1).
  • L4: Yahan "reset" ek domain event hai (logout), na ki ek architectural band-aid. Farak ye hai: kya reset ek business action se trigger hota hai, ya isliye chahiye kyunki state galat scope mein hai? Agar tum ngOnInit mein store.reset() likh rahe ho, toh scope galat hai (S3).
  • Meta-lesson: anti-patterns ko dogma mat banao. Har rule ka ek "kab theek hai" hota hai, aur seniority uska pata hona hai. Interview mein "event bus hamesha bura hai" bolna junior signal hai; "event bus ka problem any payload aur invisible graph hai; agar typed aur single-consumer ho toh wo ek observer pattern hai jo theek hai" — ye senior signal hai.

11. Common Mistakes

Galtiyan (refactoring karte waqt)
  • Big-bang rewrite — 6 mahine, kabhi khatam nahi hota, business value zero.
  • Tests ke bina refactor — behavior badal gaya aur pata nahi chala.
  • Sabse mushkil hissa pehle — momentum nahi banta, team ka bharosa nahi banta.
  • Lint rules na lagana — tum saaf karte ho, team gandagi banati hai.
  • Har purani cheez migrate karna — 2 saal se na chhui gayi screens ko chhodo.
  • "Clean code" ka argument — business value mein baat karo (bug, velocity, bundle).
  • Anti-patterns ko dogma banana — har rule ka ek valid exception hota hai.

12. Best Practices

Production rules
  • Smell detect karne ka signal: "kitne reasons to change?" aur "test mein kitne mocks?"
  • Refactor ka order: tests → pure functions → UI concerns → data layer → state → facade.
  • Ek PR = ek concern. Sabse aasan pehle (momentum).
  • Ratchet: har step ke baad lint rule, CI mein fail.
  • Strangler pattern: naya code aaj se sahi, purana jab touch ho tab.
  • Metrics dikhao: duplicate rules, rename ka file count, suite runtime, bundle size.
  • Anti-patterns ke valid exceptions jaano — aur unhe comment mein justify karo.

13. Performance

Anti-patterns ka measurable cost:

  • God service → bundle (class atomic hai, S1), aur test suite (5+ mocks per test).
  • Event bus → har subscriber ek subscription; leaks (Ch1 S8); har emit sab subscribers ko chalata hai (signals mein computed lazy hai).
  • Wrong lifetime → root store jo kabhi destroy nahi hota = unbounded memory (S3).
  • Over-injection → har injectable ek injector record + instance; pure functions tree-shakable.
  • Circular imports (S7/Ch7 S8) → lazy chunking toot-ti hai; "lazy" route initial bundle mein.

Refactoring ka business case in numbers mein banao, "clean code" mein nahi.

14. Edge Cases

  • injector.get() valid: DI cycle workaround (comment ke saath), dynamic component creation.
  • Typed single-consumer "event" abstraction (analytics) event bus nahi hai.
  • Bada service theek hai agar uska ek hi reason to change ho (parser, formatter).
  • Root store with state theek hai agar lifetime sach mein app-wide ho (auth).
  • Facade theek hai agar wo orchestrate kare, forward nahi.
  • Legacy code mein anti-pattern chhodna theek hai agar wo code dead hai.
  • any theek hai third-party ke boundary pe — par turant narrow karo (Ch2 S1).

15. Interview Questions

EASY God service kya hai aur use kaise pehchante hain?

Ek service jo bahut saare unrelated kaam karti hai (API + business rules + export + UI notifications). Pehchan: "is class ko badalne ke kitne karan hain?" — 2 se zyada matlab split karo. Aur ek measurable proxy: test file mein 5+ mocks. Line count sirf symptom hai; ek 600-line parser theek ho sakta hai, ek 200-line multi-concern service nahi.

EASY Service locator pattern kyun bura hai?

injector.get(X) har jagah use karne se dependencies class ke signature mein nahi dikhtin — wo runtime pe chhup jaati hain. Consequences: test mein "kya provide karna hai" trial-and-error; "find usages" kaam nahi karta; compiler DI cycle detect nahi kar sakta; aur failures construction pe nahi, runtime pe. Explicit inject() fields use karo. Valid exception: DI cycle todna (comment ke saath), ya dynamic component creation.

MEDIUM Leaky abstraction ka ek example do.

OrderApi.getAll() jo Observable<HttpResponse<OrderDto[]>> return karta hai, aur component HttpErrorResponse ka status === 404 check karta hai. Ab presentation layer ko HTTP, DTOs, aur status codes ka pata hai. Backend REST se GraphQL pe gaya → 40 components badle. Aur 401 handling har component mein duplicate (jo interceptor ka kaam tha). Fix: data layer domain models return kare aur HTTP errors ko domain errors mein badle (S7).

MEDIUM "Kabhi reset() likhna pada" ek smell kyun hai?

Kyunki iska matlab hai ki state ka scope uske lifetime se match nahi karta. Agar wizard ka state route-scoped hota, toh route chhodne pe injector destroy hota aur state apne aap chali jaati (Ch7 S6). Manual reset() ek jagah hai jahan tum bhool sakte ho — aur "purana data dikh raha hai" wali bug class isi se aati hai. Nuance: logout pe auth store clear karna "manual reset" nahi — wo ek domain event hai.

HARD Event bus hamesha bura hai? Nuance batao.

Nahi — jo bura hai wo generic, untyped, multi-consumer event bus hai: Subject<{type: string; payload: any}>. Uske teen zeher: (1) any payload — rename karo, compiler chup; (2) invisible dependency graph — "kaun emit karta hai, kaun sunta hai" ka jawab sirf string grep se; (3) subscriptions ka cleanup aur ordering tumhara kaam. Par ek typed, single-consumer, one-direction abstraction bilkul theek hai: abstract class Analytics { track<K extends keyof EventMap>(e: K, p: EventMap[K]): void } — 50 jagah se call hota hai, ek concrete adapter consume karta hai, payload compile-time pe typed, koi subscription, koi leak. Ye observer pattern hai, event bus nahi. Similarly, RxJS Subject ek service ke andar (private) bilkul valid hai. Test: kya consumer aur producer ka rishta compiler ko dikhta hai? Agar haan, theek hai. Interview mein "event bus hamesha bura" bolna junior signal hai; ye distinction karna senior signal hai. Aur practical alternative aksar ek typed store hai (S6) — kyunki zyada tar "events" actually state changes hote hain, aur computed() unhe declaratively express kar leta hai.

SCENARIO Tum ek team mein naye SDE-3 ho. Codebase mein: 4 god services, ek event bus, sab kuch providedIn: 'root', aur test suite 8 minutes. Tumhare paas 3 mahine hain. Kya karoge (aur kis order mein)?

(1) Pehle 2 hafte: measure aur listen. Refactor mat karo. Data collect karo: suite ka breakdown, bug backlog (kitne bugs state/lifetime se aaye?), ek endpoint rename ka file count, bundle analysis. Aur team se poochho: "sabse zyada dard kahan hota hai?" Unka jawab tumhare plan se alag hoga, aur unka jawab zyada important hai — kyunki refactor unhe karna hai. (2) Business case banao, na ki "clean code" ka. "Payment bug production mein gaya kyunki wo rule component ke andar thi aur uska test likhna 80ms leta tha." "Endpoint rename 30 files badalta hai, har baar 2 din." Ye numbers leadership ko convince karte hain; "SOLID principles" nahi. (3) Pehla PR: sabse aasan, sabse visible. Ek duplicate business rule dhundho jo do jagah diverge ho chuki hai (ye almost hamesha milti hai) — use ek pure function mein nikaalo, 15 table-driven tests likho, aur ek real bug fix karo. Zero DI, zero risk. Ab team ne dekha ki ye kaam karta hai. (4) Mahina 1: pure functions + domain folder + lint rule. God services se rules nikaalte jao. Test suite tez hone lagegi (yahi team ko sabse zyada khushi degi). Ratchet: domain/ mein framework imports ban, CI mein fail. (5) Mahina 2: lifetime audit. Har stateful root service pe poochho: "kya iska reset karna padta hai?" Jo haan bolein, unhe route/component scope mein le jao (Ch7 S6). Ye bug-fixing hai, refactoring nahi — aur "purana data dikh raha hai" wali bugs gayab ho jaayengi. Saath mein: navigable state URL mein (Ch4 S9). (6) Mahina 3: data layer + event bus. Components se HttpClient hatao (mechanical, codemod). Event bus ke consumers ko typed stores se replace karo — ek-ek event, sabse zyada use hone wala pehle. Purane bus ko deprecate karo (lint warning), delete tab jab last consumer jaaye. (7) Jo nahi karoge: big-bang rewrite; har screen migrate karna (dead screens chhodo); NgRx introduce karna (naya boilerplate, purani problem); "architecture doc" likhna jo koi na padhe. (8) Governance (sabse important): har phase ke baad ek lint rule. Ratchet only forward. Aur ek PR checklist: "kitne reasons to change? kitne mocks? state ka lifetime?" (9) Teen mahine ke baad ka metric: suite time (8min → ~1min), duplicate rules (0), endpoint rename cost (30 files → 1), aur state-lifetime bugs (0 new). (10) Aur imaandaari: 3 mahine mein sab theek nahi hoga. Goal ye hai ki trajectory badle — naya code sahi ho, aur purana jab touch ho tab theek ho. Jo team 3 mahine baad khud ye rules enforce kare, wahi asli success hai.
