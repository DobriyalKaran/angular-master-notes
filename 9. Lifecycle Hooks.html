S-01

Lifecycle ka Falsafa & Poora Order

Covers: Hooks kyun exist karte hain · Creation vs update pass · Master execution diagram · Parent-child interleaving · Destroy ka reverse order

1. What is it?

Lifecycle hooks = wo methods jinhe Angular tumhare component/directive pe specific moments pe call karta hai. Aath classic hooks, ek fixed order:

  1. ngOnChanges              <-- inputs badle (decorator inputs only)
  2. ngOnInit                 <-- ek baar, inputs ready
  3. ngDoCheck                <-- har CD cycle
  4. ngAfterContentInit       <-- ek baar, projected content ready
  5. ngAfterContentChecked    <-- har CD cycle
  6. ngAfterViewInit          <-- ek baar, apna view + children ready
  7. ngAfterViewChecked       <-- har CD cycle
  8. ngOnDestroy              <-- destroy se pehle

  ► Directives ke paas 1,2,3,8 hain (koi view/content nahi — unka apna template nahi, Ch6 S1)
  ► Components ke paas saare aath.
  ► Aur teen modern render hooks (S9) jo lifecycle ke BAHAR chalte hain.

2. Why do we need it?

Kyunki component ka construction aur uska "ready hona" alag moments hain. Constructor ke waqt: inputs set nahi hue, template render nahi hua, children exist nahi karte, DOM mein element nahi hai. Har hook ek guarantee deta hai:

  • ngOnInit: "inputs set ho chuke hain"
  • ngAfterContentInit: "projected content initialize ho chuka hai, contentChild resolve ho chuki hai"
  • ngAfterViewInit: "mera template render ho chuka hai, viewChild resolve ho chuki hai"
  • ngOnDestroy: "ab cleanup ka aakhri mauka hai"

3. What problem does it solve?

  • Initialization timing → kab inputs bharose layak hain.
  • DOM access timing → kab viewChild undefined nahi hai.
  • Cleanup → subscriptions, timers, listeners (Ch1 S8).
  • Custom change detection → ngDoCheck (rare).

4. Internal Working — creation pass vs update pass

  Ivy do "passes" chalata hai (Ch3 S6):

  ┌─────────────────────── CREATION PASS (ek baar) ───────────────────────┐
  │  δδelement(), δδtext(), δδlistener(), directive instances,          │
  │  ElementInjector setup, LView allocate                                 │
  └───────────────────────────────────────────────────────────────────────┘
                                    │
  ┌─────────────────────── UPDATE PASS (har CD cycle) ────────────────────┐
  │  δδproperty(), δδtextInterpolate(), δδclassProp(),                 │
  │  input updates, LIFECYCLE HOOKS                                        │
  └───────────────────────────────────────────────────────────────────────┘

  ► Hooks UPDATE pass ka hissa hain, creation ka nahi!
  ► Isliye pehli CD cycle mein: creation pass chalta hai, PHIR update pass,
    aur usi update pass mein ngOnChanges + ngOnInit + ... chalte hain.
  ► constructor CREATION pass mein hai. Isliye wo hooks se PEHLE hai.

  ► TView (template ka blueprint) mein hooks ka registry hota hai:
      TView.preOrderHooks     [ngOnChanges, ngOnInit, ngDoCheck]
      TView.contentHooks      [ngAfterContentInit, ngAfterContentChecked]
      TView.viewHooks         [ngAfterViewInit, ngAfterViewChecked]
      TView.destroyHooks      [ngOnDestroy]
    Yahi "pre-order" aur "post-order" naming ka source hai.

9. Advanced Example — destroy order kyun ulta hai

// Init: child pehle "ready" hota hai (parent ke AfterViewInit se pehle)
// Destroy: PARENT pehle destroy hota hai, phir children

@Component({ selector: 'app-parent', template: `<app-child />` })
class Parent {
  private ws = inject(WebSocketService);
  ngOnDestroy() {
    this.ws.disconnect();                       // L1: parent pehle
  }
}

@Component({ selector: 'app-child' })
class Child {
  private ws = inject(WebSocketService);
  ngOnDestroy() {
    this.ws.send({ type: 'child-closing' });      // L2: ❌ connection band ho chuka!
  }
}

// Kyun ulta? Kyunki DESTROY ek TOP-DOWN operation hai:
//   parent ka LView destroy hota hai --> uske andar ke child LViews destroy hote hain
//   Har LView apne destroyHooks chalata hai, phir children ke.

// ✅ Sahi design: child parent ke resources pe destroy ke waqt depend na kare.
//    Cleanup order-independent hona chahiye.

10. Code Explanation — Line by Line

  • L1–L2: Ye ek asli bug pattern hai. Init order dekhkar log maan lete hain ki destroy bhi bottom-up hoga (jaise C++ destructors, ya React ke cleanup functions). Angular mein nahi.
  • Mechanism: View destroy ek recursive top-down operation hai. Parent ka LView destroy hota hai; wo pehle apne destroyHooks chalata hai (parent ka ngOnDestroy), phir apne child LViews pe recurse karta hai. Isliye parent pehle.
  • Yaad rakhne ka tareeka: "Init mein child pehle taiyaar hota hai (parent ko uska view chahiye). Destroy mein parent pehle jaata hai (wo poore subtree ka maalik hai)."
  • Practical rule jo isse nikalta hai: ngOnDestroy mein doosre components ya shared services ke state pe depend mat karo. Cleanup order-independent hona chahiye: apne subscriptions band karo, apne timers clear karo, apne listeners hatao. Bas. Koi coordination nahi.
  • Aur ek nuance: agar child ko "main band ho raha hoon" batana hai, toh wo construction ke waqt register kare aur parent uska cleanup handle kare — ya better, ek shared registry service jiska apna lifetime ho (Ch7 S6).

11. Common Mistakes

Galtiyan
  • Constructor mein inputs use karna — abhi set nahi hue.
  • ngOnInit mein viewChild use karna — view abhi nahi bani.
  • Destroy order ko bottom-up samajhna — wo top-down hai (parent pehle).
  • Ye maanna ki ngOnChanges signal inputs pe chalta hai — nahi chalta (S3).
  • Saare 8 hooks implement karna "just in case" — har hook ek CD-cycle cost hai.
  • Directive mein ngAfterViewInit dhundhna — directives ke paas view/content hooks nahi hain.
  • ngDoCheck ko "onChanges ka better version" samajhna — wo har CD cycle mein chalta hai.

12. Best Practices

Production rules
  • Sirf wahi hooks implement karo jo chahiye. Har hook TView ke registry mein ek entry + har cycle ek call.
  • implements OnInit likho — typo (ngOninit) silently kaam nahi karega.
  • Cleanup order-independent rakho; doosre components pe depend mat karo.
  • Signals-era mein: constructor + computed + effect + DestroyRef se 5 hooks gayab ho jaate hain (S10).
  • DOM access ke liye afterNextRender (S9), ngAfterViewInit nahi.
  • Hook methods patli rakho — logic services/functions mein (Ch8 S1).

13. Performance

Har implemented hook ka cost: (1) TView ke hook registry mein ek entry (creation-time, ek baar), (2) har CD cycle mein ek function call — sirf "Check" wale hooks ke liye (ngDoCheck, ngAfterContentChecked, ngAfterViewChecked).

Numbers mein: 5,000 rows × ngDoCheck + ngAfterViewChecked = 10,000 function calls per CD cycle. Agar app zone-based hai aur mousemove chal raha hai (60/sec), toh 600,000 calls per second — sirf khaali hooks ke liye. Ye ek asli production profile pattern hai.

"Init" hooks (ngOnInit, ngAfterContentInit, ngAfterViewInit) ek baar chalte hain — unka ongoing cost zero hai.

14. Edge Cases

  • Hooks ke naam string se match hote hain (ngOnInit) — implements OnInit sirf type-level hai. Typo = silent failure.
  • Directives ke paas sirf ngOnChanges, ngOnInit, ngDoCheck, ngOnDestroy hain.
  • Inherited hooks: base class ka ngOnInit chalta hai agar derived class use override na kare (aur override kare toh super.ngOnInit() tumhara kaam).
  • ngDoCheck implement karne se ngOnChanges band nahi hota — dono chalte hain.
  • Embedded views (@if/@for) ke andar ke components ka poora lifecycle hota hai; view destroy pe ngOnDestroy (S10).
  • Services ke paas sirf ngOnDestroy hai (aur wo tabhi jab unka injector destroy ho, S8).
  • Pipes ke paas ngOnDestroy hai (impure pipes ke liye useful).

15. Interview Questions

EASY Lifecycle hooks ka order batao.

constructor → ngOnChanges → ngOnInit → ngDoCheck → ngAfterContentInit → ngAfterContentChecked → ngAfterViewInit → ngAfterViewChecked → (repeat: ngOnChanges, ngDoCheck, ...Checked) → ngOnDestroy. Directives ke paas sirf ngOnChanges, ngOnInit, ngDoCheck, ngOnDestroy hain — kyunki unka apna view/content nahi hota.

EASY Parent aur child ke ngAfterViewInit mein kaunsa pehle chalta hai?

Child ka. Kyunki parent ka view tabhi "complete" hai jab uske saare children ke views ban chuke hon. Toh order: child ka poora lifecycle (ngOnInit → ngAfterViewInit), phir parent ka ngAfterViewInit. Ulta ngOnInit mein: parent ka pehle, phir child ka (kyunki child ke inputs parent ke template se aate hain).

MEDIUM ngOnDestroy ka order kya hai?

Parent pehle, phir children — init ke bilkul ulta. Kyunki view destroy ek top-down recursive operation hai: parent ka LView destroy hota hai, wo apne destroyHooks chalata hai, phir child LViews pe recurse karta hai. Practical implication: ngOnDestroy mein doosre components ya unke resources pe depend mat karo — cleanup order-independent hona chahiye.

MEDIUM Kaunse hooks parent ki change detection cycle mein chalte hain?

ngOnChanges, ngOnInit, aur ngDoCheck — kyunki child ke inputs parent ke template se set hote hain, toh ye "pre-order hooks" tab chalte hain jab parent apna update pass chala raha hota hai. ngAfterContent* aur ngAfterView* component ki apni CD ka hissa hain. Consequence: agar OnPush parent skip ho jaaye, toh child ka ngDoCheck bhi nahi chalega — chahe child ke apne signals badle hon.

HARD Hooks creation pass mein chalte hain ya update pass mein? Iska kya matlab hai?

Update pass mein (constructor creation pass mein hai). Ivy do passes chalata hai (Ch3 S6): creation pass mein δδelement(), directive instances, LView allocation; update pass mein δδproperty(), interpolations, aur lifecycle hooks. Pehli CD cycle mein dono chalte hain: pehle creation, phir update — aur usi update pass mein ngOnChanges/ngOnInit/... chalte hain. Compiled level pe, TView mein hook registries hain: preOrderHooks (ngOnChanges, ngOnInit, ngDoCheck), contentHooks, viewHooks, destroyHooks. "Pre-order" ka matlab hai ki ye hooks tab chalte hain jab CD tree ko top-down traverse kar rahi hoti hai (parent ke template ke andar, child ke element pe pahunchte hi), jabki content/view hooks "post-order" hain (child ka kaam khatam hone ke baad). Yahi naming Angular ke source mein hai. Practical consequences: (1) constructor mein inputs available nahi (wo update pass mein set hote hain); (2) hooks CD cycle ke andar chalte hain, isliye unme state badalne pe ExpressionChangedAfterItHasBeenChecked (S7); (3) hooks SSR pe bhi chalte hain (kyunki CD server pe bhi chalti hai) — isliye ngAfterViewInit mein DOM access galat hai, afterNextRender sahi (S9).

SCENARIO Ek 5,000-row grid hai. Profiler dikhata hai ki CD time ka 40% "lifecycle hooks" mein ja raha hai. Row component mein ngDoCheck, ngAfterViewChecked, aur ngAfterContentChecked teeno hain (khali methods, purane code se). Kya karoge?

(1) Pehle math karo: 5,000 rows × 3 "Checked" hooks = 15,000 function calls per CD cycle. Zone-based app mein mousemove pe 60 cycles/sec = 900,000 calls/sec — khaali methods ke liye. Ye profiler mein exactly wahi dikhega jo tumne dekha. (2) Khaali hooks delete karo — sabse aasan, zero-risk fix. Aur ye aksar milta hai: kisi ne implements AfterViewChecked likha, method khali chhoda, aur wo 3 saal se har cycle mein call ho raha hai. Angular ise optimize nahi kar sakta — method exist karta hai toh registry mein hai. (3) Jo hooks zaroori hain, unhe replace karo: ngDoCheck aksar "input object mutate hua kya" check karne ke liye hota hai — uska sahi jawab immutable updates + signals hai (Ch1 S4), ngDoCheck nahi. ngAfterViewChecked aksar DOM measurement ke liye — wo afterRenderEffect ke read phase mein jaana chahiye (S9), jahan wo CD ke bahar aur batched chalega. (4) Structural fix — sabse bada: 5,000 rows DOM mein kyun hain? cdk-virtual-scroll se ~30 rows → 90 hook calls, 15,000 nahi. Aur aria-rowcount total dena (Ch5 S3). Ye baaki sab optimizations ko irrelevant bana deta hai. (5) CD scope: rows OnPush + signal inputs (Ch4 S5) — par yaad rakho ki pre-order hooks (ngDoCheck) parent ki CD mein chalte hain, toh OnPush row bhi apne ngDoCheck se nahi bachega agar parent check ho raha hai. Zoneless (Ch3 S8) se CD cycles hi kam ho jaate hain. (6) Ordering: khaali hooks delete (5 min) → virtualization (2 din) → hooks ko modern APIs se replace (1 din) → zoneless (alag project). Sabse sasta pehle. (7) Verify: har step ke baad Angular DevTools profiler; hook calls ko console.count se gino. (8) Governance: ek lint rule ya PR checklist — "khaali lifecycle hook = reject". Punchline: har implemented "Checked" hook ek per-instance, per-cycle tax hai; unhe implement karne ka bar bahut ooncha hona chahiye.

16. Revision Notes

30-second recall

  • Init: child pehle ready (parent ke AfterViewInit se pehle). Destroy: parent pehle (top-down).

S-02

constructor vs ngOnInit

Covers: Constructor kab chalta hai · Inputs ready nahi · Field initializers · DI ka waqt · Testability · ngOnInit kyun exist karta hai

1. What is it?

constructor ek JavaScript cheez hai — class instantiate hone pe chalta hai. ngOnInit ek Angular hook hai — jab component "initialized" maana jaata hai.

  CREATION PASS                       UPDATE PASS
  ─────────────                       ───────────
  δfac() call hota hai
     field initializers chalte hain   ← inject() yahan valid (Ch7 S2)
     constructor body chalti hai
     |
     v
  LView slot mein instance store
  ElementInjector setup
     |
     +-------------------------------> inputs set hote hain
                                       ngOnChanges
                                       ngOnInit          ← ab inputs ready

  ► Constructor ke waqt: koi input set nahi, koi child, koi DOM.
  ► ngOnInit ke waqt: inputs set, par view abhi nahi bani (S7).

2. Why do we need it?

Sawaal: agar constructor already hai, toh ngOnInit kyun banaya?

Kyunki inputs constructor ke baad set hote hain. Angular pehle instance banata hai (δfac()), phir uspe inputs likhta hai. Toh koi bhi initialization jo inputs pe depend karti hai, wo constructor mein nahi ho sakti. ngOnInit wo guarantee deta hai: "ab inputs set ho chuke hain, kaam shuru karo."

Aur ek doosra reason: testability. Constructor mein side effects (HTTP calls) daalne se class ko test mein new karna hi khatarnaak ho jaata hai.

3. What problem does it solve?

  • Input-dependent init → ngOnInit.
  • Testability → constructor sirf DI, koi side effect.
  • Separation → "wiring" (constructor) vs "starting" (ngOnInit).

4. Internal Working — δfac() ke andar kya chalta hai

export class OrderComponent {
  // 1. Field initializers — ye constructor ka HISSA hain (JS semantics, Ch1 S5)
  private api = inject(OrderApi);          // L1
  protected count = signal(0);              // L2
  orderId = input.required<string>();          // L3: InputSignal object banta hai

  constructor() {
    // 2. Constructor body
    console.log(this.orderId());              // L4: ❌ NG0950 — required input abhi set nahi!
  }
}

// Compiled (Ch7 S9):
OrderComponent.δfac = function() {
  return new OrderComponent();     // fields + constructor body
};

//   setCurrentInjector(elementInjector)   ← isliye inject() field mein chalta hai

  CONSTRUCTOR MEIN:                     ngOnInit MEIN:
  ─────────────────                     ──────────────
  ✅ inject(...)                        ✅ input-dependent setup
  ✅ signal()/computed() declare        ✅ initial data fetch (input se params)

  ► ngOnInit    = STARTING (kaam shuru)

  orderId = input.required<string>();

12. Best Practices

Production rules
  • Constructor = wiring: inject(), signal declarations, effect()/afterNextRender()/DestroyRef registration.
  • ngOnInit = starting: input-dependent kaam.
  • Dependencies pehle declare karo, derived fields baad mein.
  • Signals-era mein ngOnInit aksar zaroori nahi: computed() + resource() input-driven hain (S10).
  • Constructor patla — koi side effect, koi HTTP, koi DOM.
  • effect() constructor mein register hota hai par chalta baad mein — toh usme inputs safe hain.

13. Performance

Constructor creation pass mein chalta hai, ek baar per instance. Uska cost tumhare code ka cost hai — par 10,000 rows pe wo 10,000 baar chalega, toh usme kuch bhi bhaari (regex compile, date parsing, object cloning) directly creation time mein add hoga.

ngOnInit bhi ek baar chalta hai, par update pass mein — yani pehli CD cycle mein. Agar usme synchronous heavy work hai, toh wo first paint ko block karega.

Signals ka faayda: computed() lazy hai — declare karne ka cost ek object allocation hai, evaluate tab hota hai jab koi padhe. Toh 10,000 rows mein 10,000 computed() declarations sasti hain agar unme se sirf visible wale padhe jaayein.

14. Edge Cases

  • effect() constructor mein register hota hai par pehli baar CD cycle mein chalta hai → usme inputs available hain.
  • computed() lazy hai → usme input.required() padhna safe hai (jab tak koi pehli CD se pehle na padhe).
  • afterNextRender() constructor mein register, browser mein paint ke baad chalta hai (S9).
  • Dynamic components (createComponent) mein setInput() constructor ke baad hota hai — wahi rule.
  • ngOnInit embedded views (@if) mein har baar chalta hai jab view recreate ho (S10).
  • Constructor-injected ChangeDetectorRef aur ElementRef available hain, par unka use timing-sensitive hai.
  • SSR pe constructor aur ngOnInit dono chalte hain (CD server pe bhi chalti hai).

15. Interview Questions

EASY constructor aur ngOnInit mein farak?

constructor JS ka hissa hai — class instantiate hone pe chalta hai (creation pass), aur us waqt inputs set nahi hue hote. ngOnInit Angular ka hook hai jo update pass mein chalta hai, inputs set hone ke baad. Toh: constructor mein dependencies inject karo aur declarations likho; ngOnInit mein wo kaam karo jo inputs pe depend karta hai.

EASY Constructor mein HTTP call kyun nahi karni chahiye?

Do wajah: (1) inputs abhi set nahi hue, toh request ke params galat honge; (2) testability — har new/createComponent pe HTTP chal padegi, aur test mein use rokna mushkil. Constructor sirf "wiring" ke liye ho: inject(), signal declarations, aur effect()/afterNextRender() registration.

MEDIUM effect() ko constructor mein kyun likhte hain, ngOnInit mein kyun nahi?

Kyunki effect() internally inject(DestroyRef) karta hai (cleanup ke liye), aur inject() sirf injection context mein valid hai — constructor, field initializer, factory, ya runInInjectionContext() (Ch7 S2). ngOnInit mein context khatam ho chuka hai → NG0203. Aur ye theek bhi hai: effect() register constructor mein hota hai par pehli baar chalta CD cycle mein hai — toh uske andar inputs available hote hain. Agar ngOnInit mein hi chahiye toh {injector: this.injector} option pass karo.

MEDIUM input.required() ko constructor mein padhne se kya hota hai?

NG0950: Input is required but no value is available yet. Kyunki inputs update pass mein set hote hain, constructor creation pass mein hai. Par computed(() => this.orderId()) aur effect(() => this.orderId()) constructor mein likhna safe hai — kyunki dono lazy hain: computed tab evaluate hota hai jab koi padhe, aur effect pehli CD cycle mein chalta hai. Tab tak input set ho chuka hota hai.

HARD Field initializers aur constructor body ka execution order kya hai, aur inheritance mein kya badalta hai?

JS semantics (Ch1 S5): class field initializers declaration order mein, top-to-bottom chalte hain, aur wo constructor body se pehle. Toh a = this.makeA(); private svc = inject(Svc); mein a ka initializer this.svc ko undefined dekhega — ek TypeError jiska message bilkul nahi batata ki wajah declaration order hai. Rule: dependencies pehle, derived fields baad mein. Inheritance mein: base class ke field initializers super() call ke dauran chalte hain; derived class ke field initializers super() ke turant baad, aur derived constructor body unke baad. Isliye derived ke field initializer mein base ka field (this.http) available hota hai. Angular ke liye implication: inject() field initializer mein chalta hai, aur wo poora δfac() ke andar hai jahan currentInjector set hai (Ch7 S2/S9). Isliye field injection inheritance mein kaam karta hai bina base pe koi metadata ke — jabki constructor-parameter injection mein base pe @Injectable() chahiye aur derived ko params super(...) mein forward karne padte hain (Ch8 S2). Base mein ek nayi dependency add karo: field injection mein zero derived classes badalti hain; constructor injection mein saari. Ye inject() ka structural faayda hai, "kam typing" nahi.

SCENARIO Ek component hai jo ngOnInit mein userId input se HTTP call karta hai. Ab requirement aayi: userId runtime pe badal sakta hai (parent se), aur badalne pe naya data chahiye, purani request cancel honi chahiye. Aur race condition nahi honi chahiye. Design karo.

(1) Kyun ngOnInit kaafi nahi: wo ek baar chalta hai. Input badla toh koi refetch nahi. Ye "component sirf ek baar load karta hai" wali classic bug hai. (2) Naive fix — ngOnChanges: ngOnChanges mein refetch. Par: (a) signal inputs pe ngOnChanges chalta hi nahi... actually chalta hai, par wo purana model hai (S3); (b) race condition — user tez-tez id badle toh purani slow request baad mein aa sakti hai aur naye data ko overwrite kar degi; (c) manual unsubscribe. (3) RxJS fix: toObservable(this.userId).pipe(switchMap(id => api.get(id))) → toSignal(). switchMap purani request cancel karta hai (Ch1 S7 ka cancellation). Ye kaam karta hai aur idiomatic tha. (4) Modern fix — resource(): resource({params: () => ({id: this.userId()}), loader: ({params, abortSignal}) => api.get(params.id, abortSignal)}). Ye chaar cheezein deta hai: params signal badle toh auto-refetch; purani request AbortSignal se abort; loading/error/value states built-in (Ch8 S5); aur koi ngOnInit, koi ngOnChanges, koi subscription. Race condition structurally impossible hai. (5) Deep point jo bolna hai: ngOnInit ek imperative hook hai ("ek baar, ab") jabki requirement declarative thi ("jab bhi id badle"). Lifecycle hooks ek timing abstraction hain; signals ek dependency abstraction. Jaise hi tumhara code "X badle toh Y karo" bolne lage, hooks galat tool hain. (6) Edge cases: initial undefined id (params se undefined return karo → loader skip); error retry (resource().reload()); aur SSR (resource ka server pe behavior + TransferState). (7) Testing: fake API se promise control karo; do params changes bhejo aur assert karo ki pehla abort hua (Ch8 S8).

S-03

ngOnChanges

Covers: SimpleChanges ka shape · firstChange · Reference equality · Object mutation trap · Signal inputs ke saath · Kyun ab obsolete hai

1. What is it?

ngOnChanges(changes: SimpleChanges) tab chalta hai jab component ke decorator inputs (@Input()) ki value badalti hai — reference equality ke hisaab se.

export class ChartComponent implements OnChanges {
  @Input() data!: Point[];
  @Input() color = 'blue';

  ngOnChanges(changes: SimpleChanges) {
    // changes = {
    //   data:  SimpleChange { previousValue: undefined, currentValue: [...], firstChange: true },
    //   color: SimpleChange { previousValue: undefined, currentValue: 'blue', firstChange: true },
    // }
    if (changes['data']) this.redraw();
  }
}

interface SimpleChange {
  previousValue: any;
  currentValue: any;
  firstChange: boolean;
  isFirstChange(): boolean;
}
type SimpleChanges = { [propName: string]: SimpleChange };   // stringly-typed!

2. Why do we need it?

Pre-signals duniya mein, "input badla toh kuch karo" ka koi declarative tareeka nahi tha. ngOnChanges wahi gap bharta tha: chart redraw karo, subscription reset karo, derived value recompute karo.

Aaj: computed() derived values ke liye, effect() side effects ke liye, resource() refetch ke liye. ngOnChanges naye code mein practically zaroori nahi hai — par legacy code mein har jagah hai, aur uske trap samajhna zaroori hai.

3. What problem does it solve?

  • Input-driven side effects → redraw, refetch, reset.
  • Previous value tak access → previousValue se diff.
  • First vs subsequent → firstChange.

Galtiyan

12. Best Practices

Production rules
  • Naye code mein ngOnChanges mat likho. computed(), effect(), resource(), afterRenderEffect().
  • Legacy code mein: immutable inputs (naya reference) — warna hook chalega hi nahi.
  • readonly T[] types se mutation compile-time pe block karo (Ch2 S7).
  • Refetch ke liye resource() — race condition structurally gayab.
  • Agar ngOnChanges rakhna hi hai: typed SimpleChanges wrapper aur isFirstChange() guards.
  • Input transform (Ch4 S5) se aksar ngOnChanges ki zaroorat khatam ho jaati hai.

13. Performance

ngOnChanges ka cost teen hisso mein: (1) δδNgOnChangesFeature input setters ko wrap karta hai — ek indirection per input; (2) har change pe ek SimpleChanges object + har badle input ke liye ek SimpleChange object — allocation per change per component; (3) hook call.

1,000 rows × 3 inputs jo har cycle badalte hain = 1,000 SimpleChanges + 3,000 SimpleChange objects per cycle → GC pressure (Ch1 S8).

computed() mein ye zero hai: koi wrapper, koi allocation, aur evaluation lazy + memoized. Isliye migration sirf ergonomics nahi, performance bhi hai.

14. Edge Cases

  • ngOnChanges pehli baar ngOnInit se pehle chalta hai.
  • Agar component ka koi input nahi hai, ngOnChanges kabhi nahi chalega (bina input ke change nahi hota).
  • Static attribute (<app-x value="5">) bhi ek input set hai → ngOnChanges ek baar chalega (firstChange: true).
  • model() (two-way, Ch5 S6) ke saath: parent-driven update ngOnChanges trigger karta hai; child ka apna .set() nahi.
  • Signal inputs v17.1+ mein ngOnChanges trigger karti hain (compat bridge) — par uspe naya code mat likho.
  • Input transform ke saath: currentValue transform ke baad ki value hai.
  • Directives mein bhi ngOnChanges hota hai (unke apne inputs pe).

EASY Array mein push karne pe child ka ngOnChanges kyun nahi chalta?

Kyunki comparison reference equality hai. push() se array mutate hota hai par reference wahi rehta hai → Object.is(old, new) true → input set hi nahi hota → hook nahi chalta. Fix: immutable update (this.points = [...this.points, p]), aur behtar: type ko readonly T[] banao taaki push() compile hi na ho (Ch2 S7).

MEDIUM ngOnChanges ki teen problems batao.

(1) Stringly-typed: changes['userId'] — input rename karo, compiler kuch nahi bolega, hook silently kaam karna band kar dega. (2) Reference equality: mutation pe nahi chalta, aur naya reference har baar banane pe zaroorat se zyada chalta hai. (3) Imperative aur allocation-heavy: har change pe SimpleChanges + SimpleChange objects; aur "kuch badla, ab main kya karoon" wala model derived state ke liye galat hai — computed() declarative hai, memoized hai, aur kabhi stale nahi hota.

MEDIUM Signal inputs ke saath ngOnChanges kaam karta hai?

Haan — v17.1+ mein signal inputs bhi ngOnChanges trigger karti hain, backward compatibility ke liye. Par ye ek compatibility bridge hai, design nahi. Sahi mental model: signal input ek dependency hai; uspe react karne ka tareeka computed() (derived value), effect() (side effect), ya resource() (refetch) hai. ngOnChanges timing-based hai; signals dependency-based hain.

HARD ExpressionChangedAfterItHasBeenCheckedError kya hai aur wo lifecycle se kaise juda hai?

Dev mode mein Angular har CD cycle ke baad ek verification pass chalata hai: bindings dobara evaluate karke compare karta hai ki koi value badli toh nahi. Agar badli hai, matlab CD ke dauran kisi ne state change ki — aur us cycle mein DOM us naye value ko reflect nahi karega. Error yahi batata hai. Lifecycle se rishta: ngAfterViewInit/ngAfterViewChecked CD cycle ke andar chalte hain (S1) — parent ke view hooks tak uske template ki bindings evaluate ho chuki hoti hain. Toh wahan state badalna exactly wahi violation hai. ngOnInit/ngOnChanges mein state badalna safe hai (wo bindings se pehle hain). Kyun ye ek feature hai, bug nahi: Angular ka model unidirectional hai — ek pass mein data upar se neeche behta hai, aur us pass mein state stable rehni chahiye. Error tumhe bata raha hai ki UI ek cycle ke liye stale rahega. Production mein verification pass nahi chalta → error nahi, par stale UI ka bug rehta hai. Fixes (sahi se galat tak): (1) state ko ngOnInit mein set karo; (2) DOM-dependent kaam afterNextRender/afterRenderEffect mein le jao — wo CD ke bahar hai (S9); (3) signals use karo — unka propagation glitch-free hai, toh ek hi cycle mein consistent value milti hai; (4) ❌ setTimeout — ek nayi CD cycle, jank; (5) ❌ cdr.detectChanges() — symptom chhupa diya, extra pass. Aakhri do "kaam karte hain" aur isliye khatarnaak hain.

Covers: Exact guarantee · Async initialization · Embedded views mein · resource() se replacement · Kab ab bhi chahiye · Testing

1. What is it?

ngOnInit() — ek baar chalta hai, pehli CD cycle mein, ngOnChanges ke baad. Uska ek hi guarantee hai:

  ngOnInit ka WAADA:
    ✅ Saare inputs set ho chuke hain (initial values)
    ✅ Component instance poora bana hua hai

  ngOnInit ka NON-WAADA:
    ❌ View render nahi hui (viewChild undefined hai)
    ❌ Content project nahi hua (contentChild undefined hai)
    ❌ DOM mein element nahi hai (browser ne paint nahi kiya)
    ❌ Children ke ngOnInit abhi nahi chale

  ► Ek line mein: "inputs ready hain, DOM nahi."

2. Why do we need it?

Constructor mein inputs nahi hote (S2). Toh input-dependent initialization ke liye ek jagah chahiye thi jo "inputs ke baad par view se pehle" ho. ngOnInit wahi hai.

Aur ek design reason: constructor ko pure wiring rakhna. Ek class jiska constructor sirf dependencies leta hai, wo test mein new karna safe hai (Ch8 S8).

3. What problem does it solve?

  • Input-dependent setup → fetch, subscribe, validate.
  • Constructor ko saaf rakhna → side effects yahan.
  • Deterministic timing → "ab sab ready hai".

10. Code Explanation — Line by Line

  • L1–L4: ngOnInit parent ki CD cycle mein chalta hai, child ke template process hone se pehle (S1 ka master diagram). Toh jab child ka template {{ doubled() }} evaluate karta hai, use already-updated value milti hai. Verification pass mein koi mismatch nahi.
  • L5: ngAfterViewInit child ke template ke baad chalta hai. Bindings already evaluate ho chuki hain. Ab state badalna matlab DOM stale hai → ExpressionChangedAfterItHasBeenChecked (S3/S7).
  • Yaad rakhne ka rule: "Pre-order hooks (ngOnChanges, ngOnInit, ngDoCheck) mein state badalna safe hai. Post-order hooks (ngAfterContent*, ngAfterView*) mein nahi." Kyunki pre-order hooks bindings se pehle hain.
  • L6: Aur asli jawab ye hai ki doubled ek derived value hai — use manually set karne ki zaroorat hi nahi. computed() mein wo value() ka function hai, memoized, lazy, aur kabhi stale nahi. Poora sawaal hi gayab ho gaya.
  • Meta-lesson: zyada tar lifecycle-timing sawaal state ownership ke sawaal hain jo galat shakl mein poochhe gaye hain. Jab tum poochh rahe ho "is state ko kab set karoon", toh aksar sahi jawab hai "isko set mat karo, derive karo".

13. Performance

ngOnInit ek baar chalta hai per instance. Uska cost tumhare code ka cost hai. Par do jagah wo chhupke mehenga ho jaata hai:

  • Pehli CD cycle mein hai → usme synchronous heavy work first paint ko block karega. 100 rows × ek 2ms ngOnInit = 200ms blocked.
  • Embedded views mein repeat hota hai → @if toggle karo, @for galat track se recreate ho → ngOnInit dobara → HTTP calls dobara. Ye ek asli "kyun 50 requests gayi" bug hai.

resource()/computed() lazy hain: declare karne ka cost ek object allocation, evaluation tab jab koi padhe (aur resource ka loader tab jab params valid hon).

14. Edge Cases

  • ngOnInit ngOnChanges ke baad chalta hai (pehli cycle mein dono).
  • Agar component ka koi input nahi hai, ngOnInit phir bhi chalta hai (par ngOnChanges nahi).
  • Embedded view (@if/@for) recreate → naya instance → ngOnInit dobara.
  • Dynamic components (createComponent): ngOnInit pehli CD pe chalta hai — toh setInput() uske pehle karo.
  • ngOnInit SSR pe bhi chalta hai (CD server pe chalti hai) — DOM code mat daalo.
  • Async ngOnInit (async ngOnInit()) legal hai par Angular uska promise await nahi karta.
  • Base class ka ngOnInit override karo toh super.ngOnInit() tumhara kaam hai.

EASY ngOnInit ka exact guarantee kya hai?

MEDIUM ngOnInit mein state badalna safe hai par ngAfterViewInit mein nahi — kyun?

Toh jab bindings evaluate hoti hain, unhe already-updated value milti hai. ngAfterViewInit post-order hai — template process ho chuka hai, bindings evaluate ho chuki hain. MEDIUM Signals ke baad ngOnInit ki zaroorat hai?

HARD ngOnInit mein HTTP call karne ki teen problems batao aur unka modern solution.

Aur deep point: ngOnInit ek timing abstraction hai ("ab"), resource() ek dependency abstraction ("jab bhi ye badle"). Jaise hi requirement "X badle toh Y" ban jaaye, timing hooks galat tool hain.

11. Common Mistakes

Galtiyan
  • Apne code ki mutation ko ngDoCheck se patch karna — immutable updates use karo.
  • ngDoCheck mein bhaari kaam (deep compare, JSON.stringify) — har CD cycle, har instance.
  • ngDoCheck mein state badalna jo bind hai — ye safe hai (pre-order), par usme markForCheck() loop bana sakte ho.
  • Ye maanna ki OnPush se ngDoCheck band ho jaayega — wo OnPush check se pehle chalta hai.
  • Khaali ngDoCheck chhod dena — call ka cost phir bhi hai.
  • ngDoCheck aur ngOnChanges ko alternatives samajhna — dono saath chalte hain.
  • ngDoCheck mein HTTP call — 60 requests/sec.

12. Best Practices

Production rules
  • Pehle poochho: mutation kahan ho rahi hai? Usko theek karo, ngDoCheck mat likho.
  • Immutable updates + readonly T[] types (Ch2 S7) — compile-time protection.
  • Third-party mutable objects ke liye ek adapter layer jo unhe signal mein map kare.
  • Agar ngDoCheck zaroori hai: cheapest possible check (ek number compare), aur early return.
  • Khaali ngDoCheck delete karo.
  • Differs (IterableDiffers) naye code mein nahi — wo ek legacy abstraction hai.
  • Dev-only instrumentation ko ngDevMode guard karo.

13. Performance

Poora math (S1 se aage):

  • 1,000 instances × 1 ngDoCheck = 1,000 calls per CD cycle.
  • Zone app + mousemove (60 cycles/sec) = 60,000 calls/sec.
  • Har call mein differ.diff() ho (O(n) items) = 60,000 × n operations/sec.
  • Plus 1,000 differ objects memory mein, plus unke internal change records (Ch6 S2).

Aur ek chhupa hua cost: ngDoCheck child ke OnPush check se pehle chalta hai, toh OnPush ka poora point (skip karna) partially defeat ho jaata hai. Tumhara template skip hota hai, par hook nahi.

Signals mein equivalent cost zero hai: koi polling nahi, sirf badalne pe notification.

14. Edge Cases

  • ngDoCheck ngOnChanges ke baad chalta hai (dono implement karo toh dono chalenge).
  • OnPush child ka ngDoCheck chalta hai chahe uska template skip ho.
  • Parent skip ho (OnPush + not dirty) toh child ka ngDoCheck nahi chalega.
  • Zoneless mein CD cycles bahut kam hain → ngDoCheck bhi kam chalega (par uspe rely mat karo).
  • ngDoCheck mein markForCheck() call karke tum apne aap ko har cycle dirty kar sakte ho — infinite CD.
  • Directives mein bhi ngDoCheck hota hai (NgClass yahi use karta tha).
  • Dev mode ka verification pass ngDoCheck ko dobara chalata hai — toh usme side effects double dikhenge.

15. Interview Questions

EASY ngDoCheck kab chalta hai?

Har change detection cycle mein — chahe kuch badla ho ya na ho. Wo ngOnChanges ke baad chalta hai, aur dono saath implement kiye ja sakte hain (ek doosre ko replace nahi karte). Uska maqsad: custom change detection likhna, jab reference equality kaafi na ho (mutation).

EASY ngDoCheck aur ngOnChanges mein farak?

ngOnChanges tabhi chalta hai jab koi input reference se badle, aur use SimpleChanges milta hai. ngDoCheck har CD cycle mein chalta hai, koi arguments nahi, aur tumhe khud check karna padta hai ki kya badla. Mutation (array.push()) pe ngOnChanges nahi chalta par ngDoCheck chalta hai — yahi uska classic use case tha.

MEDIUM OnPush component ka ngDoCheck chalta hai?

Haan — aur ye surprise karta hai. ngDoCheck ek pre-order hook hai jo parent ki CD cycle mein chalta hai, child ke OnPush check se pehle. Toh child ka template skip ho sakta hai, par uska ngDoCheck phir bhi chalega. Ulta bhi sach hai: agar parent OnPush hai aur dirty nahi, toh child ka ngDoCheck nahi chalega — chahe child ka apna signal badla ho. Isliye ngDoCheck ki reliability OnPush trees mein confusing hai.

MEDIUM ngDoCheck ki zaroorat kab padti hai, aur wo ek smell kyun hai?

Zaroorat tab padti hai jab input object mutate ho raha hai (reference wahi) — toh Angular ko change dikhta hi nahi. Ye smell isliye hai kyunki asli problem upstream hai: kisi ne array.push() kiya. Sahi fix immutable updates hain (naya reference), aur readonly T[] types se mutation compile-time pe block karna. ngDoCheck ek polling workaround hai jo har CD cycle mein cost leta hai aur aksar incomplete bhi hota hai (length check se item ki internal change nahi pakadti).

HARD ngDoCheck ka cost calculate karo aur bataao ki signals ne use kaise obsolete kiya.

Cost: 1,000 instances × 1 hook = 1,000 function calls per CD cycle. Zone-based app mein mousemove pe 60 cycles/sec = 60,000 calls/sec — khaali method ke liye bhi. Agar hook ke andar differ.diff() hai (O(n) collection traversal), toh 60,000 × n operations/sec, plus 1,000 differ objects aur unke internal IterableChangeRecord linked lists (GC pressure, Ch1 S8). Aur ek chhupa hua cost: ngDoCheck child ke OnPush check se pehle chalta hai, toh OnPush ka poora point (skip karna) partially defeat ho jaata hai. Signals ne kya badla: ngDoCheck ek polling mechanism hai — "har cycle poochho ki kuch badla kya". Signals push hain — "badla toh main consumers ko bataunga". Signal write → exact consumers invalidate → sirf wahi views dirty. Koi polling, koi diffing, koi per-cycle call. Aur type level pe readonly T[] mutation ko compile hi nahi hone deta, toh wo problem utpann hi nahi hoti jiske liye ngDoCheck chahiye tha. Bada pattern: yahi transition poore Angular mein hai (Ch6 S2) — ngClass + KeyValueDiffer → δδclassProp; *ngFor + IterableDiffer → δδrepeater ka keyed reconciler; Zone.js ka "kuch badla hoga" guessing → signal graph ki exact dependencies. Generic runtime polling ko exact, compile-time-known dependencies se replace karo. ngDoCheck us purani duniya ka aakhri bacha hua nishaan hai.

SCENARIO Legacy code: ek <app-grid [rows]="rows"> jiska ngDoCheck mein JSON.stringify(rows) !== this.lastJson check hai. 2,000 rows, page janky hai. Parent code tum badal sakte ho. Refactoring plan?

(1) Cost quantify karo pehle: JSON.stringify 2,000 rows ka ≈ 1-3ms. Har CD cycle. Zone app mein mousemove pe 60×/sec = 60-180ms/sec sirf serialization mein — poora frame budget kha gaya. Ye number team ko dikhao; "JSON.stringify slow hai" abstract hai, "180ms per second" nahi. (2) Root cause: ngDoCheck + stringify ka matlab hai ki parent rows ko mutate kar raha hai (warna ngOnChanges kaafi hota). Parent code dhundho: rows.push(), rows[i].status = 'x', ya rows.sort() (in-place!). (3) Fix, sahi order: (a) Parent ko immutable banao: rows = signal<readonly Row[]>([]); update(l => [...l, newRow]); nested updates ke liye l.map(r => r.id === id ? {...r, status} : r). Type readonly Row[] se push/sort compile hi nahi honge (Ch2 S7) — yahi asli guarantee hai, discipline nahi. (b) Child se ngDoCheck delete karo: rows = input.required<readonly Row[]>(), aur jo bhi derived tha wo computed(). Ab redraw sirf tab jab rows badle. (c) Agar chart/canvas hai: afterRenderEffect(() => chart.update(this.rows())) — CD ke bahar, batched (S9). (4) Migration safety: pehle characterization tests likho (current behavior lock, Ch8 S9). Phir parent ke mutation sites ko ek-ek karke immutable karo — har PR mein ek. Type change (readonly) sabse aakhir mein, kyunki wo compile errors ki baarish karega (aur wahi tumhe saare mutation sites dikhayega — use ek discovery tool ki tarah use karo: type badlo, errors list karo, revert karo, phir plan banao). (5) Aur bada sawaal: 2,000 rows DOM mein kyun hain? Virtualization (cdk-virtual-scroll) se ~30 rows — tab ngDoCheck ka cost bhi 30× ho jaata (par phir bhi hatao). (6) Verify: DevTools profiler mein CD time; console.count se ngDoCheck/redraw calls; aur ek long-task check mousemove ke dauran. (7) Punchline: ngDoCheck hamesha ek upstream mutation ka symptom hota hai. Symptom ka ilaaj mat karo — mutation ko compile-time pe impossible bana do.

1. What is it?

Content hooks tab chalte hain jab projected content (<ng-content> se aaya markup, Ch5 S9) initialize ho chuka hota hai.

})

2. Why do we need it?

Kyunki projected content parent ke template mein likha jaata hai, par child use coordinate karta hai (tabs, accordion, form). Child ko ek moment chahiye jab wo keh sake: "ab mere content children maujood hain, unhe register/activate karo."

Ch5 S9 se yaad karo: projected content parent ke LView mein banta hai. Isliye wo child ke apne view se pehle ready hota hai — aur isliye content hooks view hooks se pehle chalte hain.

3. What problem does it solve?

  • contentChild resolution → kab wo undefined nahi hai.
  • Container coordination → tabs, accordion, stepper.
  • Content ka observation → ngAfterContentChecked se change track.

  • Deep point: queries DOM structure pe depend karti hain; DI injector hierarchy pe. Nesting ke saath DOM structure ambiguous ho jaati hai, par injector hierarchy hamesha sahi hai. Isliye container-child coordination ke liye registration via DI queries se structurally superior hai.
  • Aur ek fayda: DI registration mein content hooks ki zaroorat hi nahi — panel constructor mein register hota hai, jo accordion ke ngAfterContentInit se bhi pehle hai (content pehle banta hai, S4 diagram).

11. Common Mistakes

Galtiyan
  • ngOnInit mein contentChild use karna — abhi undefined.
  • ngAfterContentChecked mein bhaari kaam — har CD cycle.
  • QueryList.changes subscribe karke unsubscribe bhoolna.
  • contentChildren ka descendants: true default — nested containers mein galat results.
  • Content hooks ko view hooks ke baad samajhna — content pehle aata hai.
  • Projected content ka DI parent se resolve hota hai — viewProviders use nahi dikhte (Ch7 S6).
  • Content queries ko dynamic content (@for) ke saath static maanna.

12. Best Practices

Production rules
  • Signal queries (contentChild(), contentChildren()) — timing hooks ki zaroorat khatam.
  • Derived values computed() se, ngAfterContentChecked se nahi.
  • Container-child coordination: registration via DI > content queries (nesting-safe).
  • ngAfterContentChecked almost kabhi nahi — agar chahiye toh cheapest check + early return.
  • Dynamic content (@for mein tabs) ke liye signal queries reactive hain — QueryList.changes ki zaroorat nahi.
  • Library components: content ka contract document karo (kaunse child components expect kiye jaate hain).

13. Performance

ngAfterContentInit ek baar — cost tumhare code ka. ngAfterContentChecked har CD cycle mein — wahi math jo ngDoCheck ka hai (S5): instances × cycles.

Content queries ka cost: har query ek TQuery entry hai TView mein, aur creation ke waqt matching nodes ko traverse karke collect kiya jaata hai. Dynamic content (@for ke andar) mein query har view insert/remove pe update hoti hai.

Signal queries ka faayda: contentChildren() ek signal deta hai → uspe computed() memoized hai → content na badle toh derivation dobara nahi chalti. ngAfterContentChecked version mein wo har cycle chalti.

14. Edge Cases

  • Content hooks tabhi chalte hain jab component ka <ng-content> ho ya na ho — wo hamesha chalte hain (content khaali ho toh bhi).
  • <ng-content> ke bahar bhi content queries kaam karti hain — wo host element ke children ko dekhti hain.
  • Projected content parent ke LView mein banta hai → uska CD parent ke saath (Ch5 S9).
  • contentChild static option ({static: true}) — ngOnInit mein available, par sirf tab jab content conditional na ho.
  • Signal contentChildren() reactive hai — @for se tabs add/remove hon toh signal update hota hai.
  • ngAfterContentInit view hooks se pehle, aur children ke ngOnInit ke baad.
  • Directives ke paas content hooks nahi hain (unka apna content nahi).

15. Interview Questions

EASY ngAfterContentInit kab chalta hai?

Jab projected content (<ng-content> se aaya markup) initialize ho chuka ho aur content queries (@ContentChild/@ContentChildren) resolve ho gayi hon. Ye ngDoCheck ke baad aur ngAfterViewInit se pehle chalta hai — kyunki projected content parent ke LView mein pehle banta hai, component ka apna view baad mein.

EASY Content hooks view hooks se pehle kyun chalte hain?

Kyunki projected content parent ke template mein likha jaata hai aur parent ke LView mein banta hai (Ch5 S9) — toh wo component ke apne view se pehle exist karta hai. Component ka apna template baad mein process hota hai. Isliye order: ngAfterContentInit → (apna template) → ngAfterViewInit.

MEDIUM ngOnInit mein contentChild undefined kyun hai?

ngOnInit ek pre-order hook hai jo parent ki CD cycle mein chalta hai, content queries resolve hone se pehle. Content queries ngAfterContentInit ke bilkul pehle bharti hain. Fix: ngAfterContentInit use karo, ya (behtar) signal query contentChild() jo ek Signal<T | undefined> deti hai — use kabhi bhi padho, aur computed()/effect() mein padhne pe resolve hone pe automatically re-run hota hai. Timing ka sawaal hi khatam.

MEDIUM ngAfterContentChecked ka use case aur khatra?

Use case: projected content har CD cycle mein badal sakta hai (dynamic tabs) aur tumhe react karna hai. Khatra: wo har CD cycle mein chalta hai — 1,000 instances × 60 cycles/sec = 60,000 calls/sec. Aur usme derived values compute karna galat hai (computed() memoized hai). Modern jawab: contentChildren() signal query + computed() — content badle tabhi derivation chalti hai.

HARD Nested containers (accordion in accordion) mein content queries kyun fail karti hain, aur sahi solution kya hai?

contentChildren ka default descendants: true hai (v17+) — toh outer accordion ko nested accordion ke panels bhi mil jaate hain. Silent bug, sirf nesting ke saath. descendants: false "fix" karta hai par wo DOM structure pe depend karta hai: agar consumer panel ko ek wrapper div mein daal de, wo miss ho jaayega. Yani query DOM shape pe coupled hai, jo ek fragile contract hai. Sahi solution: registration via DI. Panel inject(AccordionComponent) kare — ElementInjector chain "nearest wins" deti hai (Ch7 S6), toh nested panel apne nearest accordion mein register hoga, chahe kitne wrappers hon. Panel constructor mein accordion.register(this), aur DestroyRef.onDestroy(() => accordion.unregister(this)). Deep point: queries DOM structure pe depend karti hain; DI injector hierarchy pe. Nesting mein DOM structure ambiguous ho jaati hai, injector hierarchy nahi. Isliye container-child coordination ke liye DI registration structurally superior hai — aur Angular Material yahi karta hai. Bonus: DI registration constructor mein hota hai, jo ngAfterContentInit se bhi pehle hai (content pehle banta hai) — toh coordination logic ke liye content hook ki zaroorat hi nahi bachti. Aur signals ke saath registry ko signal<Panel[]> rakho toh open/close state bhi reactive ho jaati hai.

SCENARIO Ek <app-tabs> component hai. Tabs @for se dynamically aati hain, ek tab remove hone pe active tab galat ho jaata hai, aur nested tabs (tab ke andar tabs) mein outer tabs ke tab-bar mein inner tabs bhi dikhte hain. Redesign karo.

Do alag bugs, do alag mechanisms. (A) Dynamic tabs + stale active: @ContentChildren + ngAfterContentInit wala purana design ek baar activate karta hai; tabs.changes subscribe karke re-activate karna padta hai (aur unsubscribe bhi). Aur active tab ko reference se rakhne pe wo tab remove hone pe dangling ho jaata hai. Fix: (1) readonly tabs = contentChildren(TabComponent) — ek reactive signal; @for se add/remove hone pe wo update hota hai, koi subscription nahi. (2) Active ko id se track karo, reference se nahi: private _activeId = signal<string|null>(null). (3) readonly activeTab = computed(() => { const all = this.tabs(); return all.find(t => t.id() === this._activeId()) ?? all[0] ?? null; }) — fallback built-in. Ab tab remove hone pe activeTab automatically pehle tab pe gir jaata hai. Zero hooks, zero subscriptions. Ye "derived state kabhi stale nahi hota" ka concrete faayda hai (Ch8 S5). (B) Nested tabs leak: contentChildren ka descendants: true default — inner tabs outer ko mil rahi hain. descendants: false ek band-aid hai jo DOM structure pe coupled hai. Sahi fix: registration via DI — TabComponent apne constructor mein inject(TabsComponent) kare (nearest wins, Ch7 S6) aur usme register kare; DestroyRef se unregister. Inner tab apne inner <app-tabs> mein register hogi. Registry ek signal<Tab[]> ho taaki sab derived state reactive rahe. (C) Aur ek design sawaal: active tab navigable hai kya? Agar user link share karega ya back button dabayega, toh wo URL mein honi chahiye (Ch4 S9) — tab component ek activeId input le aur activeIdChange output de (model(), Ch5 S6), aur route use URL se drive kare. (D) A11y: role="tablist"/role="tab"/aria-selected, arrow-key navigation, aur aria-controls. Ye tabs component ka aadha kaam hai aur aksar bhoola jaata hai. Punchline: content hooks aur queries "kab" ka jawab dete hain; DI aur signals "kaun" aur "kya" ka. Container components ke liye doosre do behtar hain.

Covers: ngAfterViewInit / ngAfterViewChecked · viewChild resolution · ExpressionChangedAfterItHasBeenChecked ka poora mechanism · DOM access ka trap · SSR

1. What is it?

View hooks tab chalte hain jab component ka apna template render ho chuka hota hai — uske viewChild queries resolve ho chuki hain, aur uske children ke ngAfterViewInit bhi chal chuke hain.

export class ChartComponent implements AfterViewInit {
  @ViewChild('canvas') canvas!: ElementRef<HTMLCanvasElement>;

  ngAfterViewInit() {
    const ctx = this.canvas.nativeElement.getContext('2d');   // ab available
    // ⚠ Par ye SSR pe crash karega, aur CD ke ANDAR hai! (S9)
  }
}

Ye chapter ka sabse "khatarnaak" hook hai: wo lagta hai DOM access ki jagah, par actually nahi hai.

2. Why do we need it?

Historically: viewChild se DOM element chahiye tha, aur wo ngOnInit mein undefined hota tha. ngAfterViewInit ne wo gap bhara.

Aaj: signal queries (viewChild()) timing sawaal khatam karti hain, aur afterNextRender() DOM access ke liye structurally sahi jagah hai (S9). ngAfterViewInit ka legitimate use bahut chhota reh gaya hai.

3. What problem does it solve?

  • viewChild resolution → kab wo undefined nahi.
  • Child components ready → unke public APIs call kar sakte ho.
  • (Historically) DOM access → ab afterNextRender better hai.

4. Internal Working — ExpressionChangedAfterItHasBeenChecked

  DEV MODE mein Angular har CD cycle ke baad ek VERIFICATION PASS chalata hai:

  ┌──────────────────────────────────────────────────────────┐
  │  PASS 1 (normal)                                         │
  │    bindings evaluate karo, DOM update karo               │
  │    hooks chalao (ngOnInit, ngAfterViewInit, ...)         │
  └──────────────────────────────────────────────────────────┘
                          │
                          ▼
  ┌──────────────────────────────────────────────────────────┐
  │  PASS 2 (verification, SIRF dev mode)                    │
  │    bindings DOBARA evaluate karo                         │
  │    par DOM update MAT karo                               │
  │    agar koi value BADLI --> throw                        │
  │    ExpressionChangedAfterItHasBeenCheckedError           │
  └──────────────────────────────────────────────────────────┘

  ► Error ka matlab: "CD ke DAURAN kisi ne state badli.
    Is cycle mein DOM us naye value ko reflect nahi karega."

  ► Ye ek DEVELOPER AID hai, ek arbitrary rule nahi.
    Wo unidirectional data flow ki guarantee ko protect karta hai:
    "ek pass mein data upar se neeche behta hai, aur us pass mein state stable rehti hai."

  ► PRODUCTION mein pass 2 chalta hi nahi --> error nahi aata,
    par bug rehta hai: UI ek cycle ke liye STALE.

5. Execution Flow — error ka concrete example

@Component({
  selector: 'app-child',
  template: `<p>{{ status() }}</p>`,       // L1: binding
})
export class ChildComponent implements AfterViewInit {
  protected status = signal('loading');

  ngAfterViewInit() {
    this.status.set('ready');              // L2
  }
}

// Timeline:
//   Pass 1: {{ status() }} evaluate --> 'loading' --> DOM mein 'loading'
//           ngAfterViewInit()       --> status = 'ready'
//   Pass 2: {{ status() }} evaluate --> 'ready'
//           'loading' !== 'ready'   --> ERROR                          // L3
//
//   Message: "Expression has changed after it was checked.
//             Previous value: 'loading'. Current value: 'ready'."

// ✅ FIX 1: pre-order hook (bindings se pehle)
ngOnInit() { this.status.set('ready'); }                       // L4

// ✅ FIX 2: derive karo, set mat karo
protected status = computed(() => this.data() ? 'ready' : 'loading');  // L5

// ✅ FIX 3: DOM-dependent hai toh CD ke bahar
constructor() { afterNextRender(() => this.status.set('ready')); }      // L6

// ❌ ANTI-FIX 1: setTimeout — ek nayi CD cycle, jank, aur flash
ngAfterViewInit() { setTimeout(() => this.status.set('ready')); }       // L7

// ❌ ANTI-FIX 2: cdr.detectChanges() — ek extra sync pass
ngAfterViewInit() { this.status.set('ready'); this.cdr.detectChanges(); }  // L8

7. Real Project Example — ngAfterViewInit ke teen problems

export class MapComponent implements AfterViewInit {
  @ViewChild('mapDiv') mapDiv!: ElementRef;

  ngAfterViewInit() {
    this.map = new LeafletMap(this.mapDiv.nativeElement);   // L1
    const h = this.mapDiv.nativeElement.offsetHeight;         // L2
    this.ready.set(true);                                    // L3
  }
}

// PROBLEM 1 (L1): SSR pe crash
//   ngAfterViewInit server pe BHI chalta hai (CD server pe chalti hai).
//   nativeElement wahan browser ka HTMLElement nahi hai. LeafletMap crash.

// PROBLEM 2 (L2): timing
//   ngAfterViewInit CD cycle ke ANDAR hai, browser ke PAINT se pehle.
//   offsetHeight padhna --> forced synchronous layout (Ch6 S6).
//   Aur wo value abhi final nahi ho sakti (CSS abhi apply nahi hui).

// PROBLEM 3 (L3): ExpressionChangedAfterItHasBeenChecked
//   ready() agar template mein bound hai toh dev mode error.

// ✅ SAB FIX: afterNextRender (S9)
constructor() {
  const el = viewChild.required<ElementRef>('mapDiv');
  afterNextRender({
    write: () => { this.map = new LeafletMap(el().nativeElement); },
    read: () => { const h = el().nativeElement.offsetHeight; },
  });
}
// Server pe register hi nahi hota. Paint ke baad chalta hai. CD ke bahar.

@Component({

10. Code Explanation — Line by Line

  • L1–L3: Yahan koi "clearly wrong" line nahi hai. Child ngAfterViewInit mein ek output emit karta hai (bilkul normal), parent uspe apni state badalta hai (bilkul normal). Par output emission synchronous hai (Ch4 S6) — wo turant parent ka handler chalata hai, usi CD pass ke andar.
  • L4: Ab parent ki binding [label] stale hai. Verification pass use pakad leta hai. Aur error message child ke template ki taraf ishara karta hai ({{ label }}), jabki galti parent-child ke coupling mein hai. Isliye ye error itna confusing hota hai.
  • Debugging ka tareeka: error message mein "Previous value" aur "Current value" dono hote hain. Us binding ko dhundho, phir poochho: "is cycle mein kisne ye value badli?" Aksar jawab ek ngAfterViewInit, ek synchronous output emit, ya ek subscribe callback hota hai jo sync fire hua.
  • Signals kyun better hain: signal graph glitch-free hai — ek update propagation mein har consumer ko ek hi consistent value dikhti hai, aur intermediate states kabhi observe nahi hoti. Par ye tab bhi help karta hai jab tum derive kar rahe ho (computed). Agar tum abhi bhi ngAfterViewInit mein signal.set() kar rahe ho, wahi problem hai — error aayegi.
  • Asli lesson: ye error "state ko CD ke andar mat badlo" bol raha hai. Aur uske do sahi jawab hain: derive karo (computed) ya CD ke bahar chalo (afterNextRender). setTimeout aur detectChanges() dono "kaam" karte hain kyunki wo ya toh nayi cycle banate hain ya extra pass chalate hain — dono ne problem ko fix nahi kiya, chhupaya hai (aur jank add ki).

11. Common Mistakes

Galtiyan
  • ngAfterViewInit mein bound state badalna → ExpressionChangedAfterItHasBeenChecked.
  • setTimeout/cdr.detectChanges() se error "fix" karna — symptom chhupa, jank add.
  • ngAfterViewInit mein DOM access — SSR crash, aur paint se pehle (stale measurements).
  • ngAfterViewChecked mein DOM writes — har CD cycle, forced layout.
  • @ViewChild({static: true}) @if ke andar wale element pe — undefined.
  • ngAfterViewInit mein synchronous output emit — parent ki state badal sakta hai.
  • Ye maanna ki ngAfterViewInit "DOM ready" ka matlab hai — wo paint se pehle hai.

12. Best Practices

Production rules
  • DOM access = afterNextRender() / afterRenderEffect() (S9), ngAfterViewInit nahi.
  • Signal queries (viewChild()) — timing sawaal khatam, @if ke saath reactive.
  • State ko derive karo (computed()), view hooks mein set mat karo.
  • ExpressionChanged error aaye toh: kaunsi binding? kisne badli? — setTimeout se bhago mat.
  • ngAfterViewChecked almost kabhi nahi. Chahiye toh afterRenderEffect.
  • Output emit ngOnInit mein, view hooks mein nahi (sync emission ka trap).
  • ngAfterViewInit ka legitimate use: child component ke public API call karna (jo state na badle).

13. Performance

ngAfterViewChecked: har CD cycle, har instance — wahi math (S1/S5). Aur usme DOM reads (scrollHeight, offsetHeight) ho toh forced synchronous layout per cycle (Ch6 S6): browser ko pending styles flush karke layout compute karna padta hai. 60 cycles/sec × layout = jank.

Verification pass ka cost: dev mode mein har CD cycle do baar chalti hai (bindings ka evaluation). Isliye dev build hamesha slow lagti hai — ye normal hai, prod mein wo pass nahi hota.

afterRenderEffect ka faayda: wo CD ke bahar hai, paint ke baad, aur uske phases (earlyRead → write → mixedReadWrite → read) saare components ke DOM reads/writes ko batch karte hain — layout thrashing structurally rok deta hai (S9).

14. Edge Cases

  • ngAfterViewInit SSR pe chalta hai — DOM APIs wahan crash karenge.
  • @ViewChild({static: true}) ngOnInit mein available, par conditional elements pe undefined.
  • Signal viewChild() @if toggle pe automatically update hoti hai; decorator wali nahi (QueryList.changes chahiye).
  • ngAfterViewChecked verification pass mein dobara chalta hai (dev mode) — side effects double dikhenge.
  • Child ka ngAfterViewInit parent ke ngAfterViewInit se pehle (S1).
  • Directives ke paas view hooks nahi hain.
  • Production mein ExpressionChanged error nahi aata, par stale UI ka bug rehta hai.

15. Interview Questions

EASY ngAfterViewInit kab chalta hai?

Jab component ka apna template render ho chuka ho aur uske viewChild queries resolve ho gayi hon — aur uske saare children ke ngAfterViewInit bhi chal chuke hon (child pehle, parent baad mein). Par yaad rakho: ye CD cycle ke andar hai, browser paint se pehle, aur SSR pe bhi chalta hai.

EASY ExpressionChangedAfterItHasBeenCheckedError kab aata hai?

Jab CD cycle ke dauran koi bound value badal jaaye. Dev mode mein Angular ek verification pass chalata hai (bindings dobara evaluate karke compare) — mismatch mile toh error. Sabse common cause: ngAfterViewInit/ngAfterViewChecked mein bound state badalna, ya un hooks se ek synchronous output emit karna jo parent ki state badal de.

MEDIUM ngOnInit mein state badalna safe hai par ngAfterViewInit mein nahi — mechanism batao.

ngOnInit ek pre-order hook hai — wo parent ki CD cycle mein chalta hai, component ke template ki bindings evaluate hone se pehle. Toh bindings ko already-updated value milti hai; verification pass mein koi mismatch nahi. ngAfterViewInit post-order hai — template ki bindings evaluate ho chuki hain, DOM update ho chuka hai. Ab state badalne se DOM stale ho jaata hai, aur verification pass use pakad leta hai. Rule: pre-order hooks (ngOnChanges, ngOnInit, ngDoCheck) mein state change safe; post-order (ngAfterContent*, ngAfterView*) mein nahi.

MEDIUM setTimeout se ExpressionChanged error fix karna galat kyun hai?

Kyunki wo problem fix nahi karta, chhupata hai. setTimeout callback ek nayi CD cycle mein chalta hai (zone app mein wo cycle trigger bhi karta hai), toh verification pass ko mismatch nahi dikhta. Par: (1) ek extra CD cycle = extra work; (2) UI ek frame ke liye purani value dikhata hai → flash/flicker; (3) zoneless mein wo CD trigger hi nahi karega. Sahi fixes: state ko pre-order hook mein badlo, ya computed() se derive karo, ya DOM-dependent hai toh afterNextRender() (CD ke bahar). cdr.detectChanges() bhi wahi — ek extra synchronous pass, symptom chhupa.

HARD ngAfterViewInit mein DOM access ki teen problems batao.

(1) SSR: ngAfterViewInit server pe bhi chalta hai (change detection server pe chalti hai), aur wahan nativeElement browser ka HTMLElement nahi hai. canvas.getContext(), focus(), new LeafletMap(el) — sab crash. Isliye log isPlatformBrowser guards likhte hain, jo boilerplate hai. (2) Timing: wo CD cycle ke andar hai, browser ke paint se pehle. Toh layout measurements (offsetHeight, getBoundingClientRect) stale ya incomplete ho sakti hain (CSS abhi apply nahi hui, fonts load nahi hue), aur unhe padhna ek forced synchronous layout trigger karta hai (Ch6 S6). (3) ExpressionChanged: DOM measure karke state set karna (this.height.set(el.offsetHeight)) — agar wo bound hai toh dev mode error. Sahi jawab: afterNextRender()/afterRenderEffect() (S9). Wo (a) server pe register hi nahi hote — koi platform check nahi chahiye; (b) paint ke baad chalte hain — measurements final; (c) CD ke bahar hain — state change safe; (d) unke phases (earlyRead, write, mixedReadWrite, read) saare components ke reads/writes batch karte hain, toh layout thrashing structurally rok jaati hai. Yani ek API ne chaaron problems ek saath solve ki. Isliye modern Angular mein ngAfterViewInit ka legitimate use bahut chhota hai: sirf child component ke public API call karna jo state na badle.

ngOnDestroy & DestroyRef

Covers: Kab chalta hai, kab nahi · Root services ka jaal · DestroyRef · takeUntilDestroyed · Cleanup checklist · SSR

1. What is it?

ngOnDestroy() — component/directive/pipe/service destroy hone se pehle chalta hai. Cleanup ka aakhri mauka.

export class ChatComponent implements OnDestroy {
  private sub = this.ws.messages$.subscribe(...);
  private timer = setInterval(() => this.poll(), 5000);

  ngOnDestroy() {
    this.sub.unsubscribe();
    clearInterval(this.timer);
  }
}

// Modern: DestroyRef (functional, injection-context based)
export class ChatComponent {
  constructor() {
    const timer = setInterval(() => this.poll(), 5000);
    inject(DestroyRef).onDestroy(() => clearInterval(timer));   // cleanup setup ke SAATH
  }
}

2. Why do we need it?

Kyunki JavaScript ka garbage collector reachability dekhta hai (Ch1 S8). Ek destroy hua component agar kisi zinda cheez se reference ho — ek subscription, ek DOM listener, ek timer, ek Map entry — toh wo memory mein rehta hai, aur uska poora subtree bhi.

ngOnDestroy wahi jagah hai jahan tum wo references todte ho. Bhoolne pe: memory leak + zombie code (destroy hua component ka callback chalta rehta hai).

3. What problem does it solve?

  • Memory leaks → Big Five (Ch1 S8): subscriptions, listeners, timers, closures, detached DOM.
  • Zombie callbacks → destroy ke baad state update.
  • External resources → chart instances, maps, WebSockets, workers.

4. Internal Working — kab chalta hai (aur kab NAHI)

  COMPONENT / DIRECTIVE:
    ✅ Parent view destroy hui
    ✅ @if false hua, @for se item nikla (Ch5 S8/S10)
    ✅ Route change (component destroy)
    ✅ ViewContainerRef.remove() / clear()
    ✅ ComponentRef.destroy()
    ❌ Page reload / tab close  ← process hi mar gaya, koi hook nahi

  SERVICE:
    ✅ Component-scoped: component destroy hone pe
    ✅ Route-scoped: route ka injector destroy hone pe
    ✅ createEnvironmentInjector(...).destroy() manually
    ❌ providedIn: 'root' (browser mein)  ← root injector kabhi destroy nahi hota!
    ✅ providedIn: 'root' (SSR)           ← har request ka injector destroy hota hai

  ► YE SABSE ZYADA MISUNDERSTOOD BAAT HAI:
    Root service ka ngOnDestroy browser mein KABHI nahi chalta.
    Aur SSR mein CHALTA hai. Isliye wahan cleanup zaroori hai.

  DESTROY ORDER (S1 ka recap):

    Parent LView destroy

10. Code Explanation — Line by Line

  • L1–L3: Missing unsubscribe ka nateeja sirf "memory leak" nahi hai. Ye ek correctness bug hai.
  • L4: Analytics event fire ho gaya ek order ke liye jo user ne dekha hi nahi. Ab tumhara funnel data galat hai, aur ye bug kabhi kisi error log mein nahi dikhega. Yahi "zombie callback" ka asli khatra hai — silent data corruption.
  • L5: takeUntilDestroyed() subscription ko destroy pe complete kar deta hai — callback chalta hi nahi. Injection context mein destroyRef automatically milta hai; bahar (kisi method mein) explicitly pass karna padta hai.
  • L6: resource() ek kadam aur aage: wo AbortSignal deta hai, aur fetch/HttpClient use samajhte hain → request khud cancel ho jaati hai. Server ka kaam bhi bacha, network bhi. Ye "cancellation" hai (Ch1 S7), sirf "ignore karna" nahi.
  • Teen levels ka farak: (1) unsubscribe — callback nahi chalega, par request chal chuki hai; (2) takeUntilDestroyed — same, par declarative; (3) resource/abortSignal — request hi cancel. Teesra structurally sabse behtar hai.

11. Common Mistakes

Galtiyan
  • Root service ke ngOnDestroy pe bharosa — browser mein kabhi nahi chalta.
  • Subscriptions ka cleanup bhoolna → leak + zombie callbacks (galat analytics!).
  • Cleanup ko ngOnDestroy mein 40 lines door likhna — setup se drift ho jaata hai. DestroyRef use karo.
  • ngOnDestroy mein doosre components/services ke state pe depend karna — destroy top-down hai (S1).
  • Third-party instances (chart, map) destroy na karna — unke apne listeners/timers.
  • Page unload ke liye ngOnDestroy expect karna — pagehide/visibilitychange listener chahiye.
  • takeUntilDestroyed() injection context ke bahar bina destroyRef ke.

12. Best Practices

Production rules
  • DestroyRef.onDestroy() setup ke bilkul saath likho — locality se bhoolna mushkil.
  • RxJS: takeUntilDestroyed() (context mein) ya toSignal() (auto-cleanup).
  • Best: aisi APIs use karo jinme cleanup automatic ho — effect(), toSignal(), resource(), host listeners.
  • Root services: ngOnDestroy likho SSR ke liye, par browser cleanup ke liye service ko scope karo.
  • Cleanup order-independent rakho (destroy top-down hai).
  • Third-party lifecycle: create in afterNextRender, destroy in DestroyRef (Ch6 S6).
  • Leak test: route ko 20 baar navigate karo, heap snapshot compare karo.

13. Performance

ngOnDestroy ek baar chalta hai — uska direct cost ignorable hai. Uska na chalna mehenga hai:

  • Memory: ek leaked component apne poore subtree, uske signals, uske DOM nodes (detached), aur uske closures ko hold karta hai. Ek SPA mein 50 navigations = 50 leaked page components = sau MB.
  • CPU: leaked subscriptions aur timers chalte rehte hain. 20 leaked components × ek 5-second poll = 4 requests/sec background mein.
  • Correctness: zombie callbacks galat analytics, galat state updates.

Leak detection: Chrome DevTools Memory tab → do heap snapshots (navigate away and back × 20) → "Detached HTMLElement" aur component class names dhundho.

14. Edge Cases

  • Root service ka ngOnDestroy: browser mein kabhi nahi, SSR mein har request pe.
  • Page reload/tab close pe ngOnDestroy nahi chalta — process marta hai.
  • DestroyRef.onDestroy() ek unregister function return karta hai (agar cleanup pehle karni ho).
  • takeUntilDestroyed() bina argument ke sirf injection context mein.
  • Destroy order top-down — parent pehle (S1).
  • Directives aur pipes ka bhi ngOnDestroy hota hai.
  • ComponentRef.destroy() manually call karne pe hooks chalte hain; ViewContainerRef.detach() pe nahi (view zinda rehti hai).
  • Ek destroyed injector se inject() karna → error.

MEDIUM providedIn: 'root' service ka ngOnDestroy kab chalta hai?

Injection context mein (constructor/field) destroyRef automatically inject hota hai; bahar (kisi method mein) explicitly pass karna padta hai: takeUntilDestroyed(this.destroyRef). 10. Code Explanation — Line by Line

  • L1: afterRender ka naam confusing hai — log sochte hain "render ke baad" matlab "jab UI badle". Nahi: wo har render pass ke baad chalta hai, chahe tumhare component se koi lena-dena na ho. Ek unrelated component ka signal update → CD → render → tumhara callback. Iska legitimate use debugging aur metrics hai, business logic nahi.
  • L2: afterRenderEffect ek reactive effect hai jo render ke baad chalta hai: uske andar padhe gaye signals uski dependencies ban jaate hain, aur wo tabhi dobara chalta hai jab unme se koi badle.
  • L3–L4: Classic trap. this.rawData ek plain class field hai, signal nahi. Uske andar koi signal read nahi hua, toh effect ki koi dependency nahi — wo ek baar chalega aur phir kabhi nahi. Aur ye silent hai. Rule: afterRenderEffect ke andar jo bhi reactive hona chahiye, wo signal se padha jaaye.
  • effect() vs afterRenderEffect(): dono signals track karte hain, par timing alag hai. effect() change detection ke dauran chalta hai (state ke liye theek), afterRenderEffect() render ke baad (DOM ke liye theek, phases ke saath). Agar tum effect() mein DOM chhoote ho, tumhe wahi teen problems milengi jo ngAfterViewInit mein thi.
  • Yaad rakho: effect = state ke liye. afterRenderEffect = DOM ke liye. afterNextRender = ek-baar ka DOM setup. afterRender = sirf debugging.

11. Common Mistakes

Galtiyan
  • afterRender ko afterRenderEffect samajhna — wo har render pe chalta hai.
  • afterRenderEffect mein plain field padhna — koi dependency, ek baar chalega.
  • Simple callback form use karna (afterNextRender(() => ...)) jab phases ka faayda mil sakta tha.
  • effect() mein DOM chhoona — wo CD ke andar hai.
  • Render hooks ko method mein call karna — injection context chahiye (Ch7 S2).
  • Cleanup bhoolna — afterNextRender mein banaya chart DestroyRef se destroy karo (S8).
  • Ye maanna ki afterNextRender SSR pe chalega — wo register hi nahi hota.

12. Best Practices

Production rules
  • DOM setup → afterNextRender. DOM sync → afterRenderEffect. State → effect.
  • afterRender sirf debugging/metrics ke liye (aur ngDevMode guard karo).
  • Phases use karo: reads earlyRead/read mein, writes write mein. mixedReadWrite se bacho.
  • Constructor/field mein register (injection context).
  • Third-party: create in afterNextRender, update in afterRenderEffect, destroy in DestroyRef.
  • SSR guards (isPlatformBrowser) ki zaroorat nahi — hooks server pe register hi nahi hote.
  • afterRenderEffect ke andar signals padho, plain fields nahi.

13. Performance

Phases ka asli faayda: layout thrashing framework level pe rok di jaati hai. 20 tooltips jo alag-alag ngAfterViewChecked mein read→write→read karte hain = 40 forced synchronous layouts per cycle. Phases ke saath: saare earlyRead (1 layout), saare write, saare read (1 layout) — total 2.

afterRender ka cost: har render pass. Zone app mein mousemove pe 60/sec. Ek afterRender mein bhaari kaam = jank. afterRenderEffect sirf dependencies badalne pe.

SSR: callbacks register hi nahi hote → server render mein zero cost. Ye isPlatformBrowser guards se behtar hai (wahan callback register hota hai aur andar check hota hai).

14. Edge Cases

  • Render hooks ko injection context chahiye — constructor/field, ya {injector} option.
  • afterNextRender ek AfterRenderRef deta hai jiska .destroy() use pehle cancel kar sakta hai.
  • Phase ka return value agli phase ko argument milta hai (chaining).
  • afterRenderEffect ke phases mein signal reads ho sakte hain; dependencies un sabse banti hain.
  • Hydration ke saath: server-rendered DOM reuse hota hai, toh afterNextRender mein element wahi hai jo server ne bheja tha (Ch3 S8).
  • ngSkipHydration un elements pe jinhe third-party lib mutate karti hai (chart canvas).
  • Zoneless mein render hooks bilkul same kaam karte hain.
  • afterRender/afterRenderEffect tabhi chalte hain jab koi CD cycle chale — koi change nahi toh koi render nahi.

15. Interview Questions

EASY afterNextRender aur ngAfterViewInit mein farak?

ngAfterViewInit CD cycle ke andar chalta hai, browser paint se pehle, aur SSR pe bhi chalta hai. afterNextRender CD ke bahar, paint ke baad, aur server pe register hi nahi hota. Isliye DOM access, third-party library init, aur measurements ke liye afterNextRender sahi jagah hai — koi isPlatformBrowser guard nahi chahiye, koi ExpressionChanged error nahi.

EASY afterRender aur afterRenderEffect mein farak?

afterRender har render pass ke baad chalta hai — chahe tumhare component se koi lena-dena ho ya nahi. Uska use sirf debugging/metrics hai. afterRenderEffect ek reactive effect hai: uske andar padhe gaye signals uski dependencies banti hain, aur wo tabhi dobara chalta hai jab unme se koi badle. Chart update, scroll sync — ye afterRenderEffect ka kaam hai.

MEDIUM Chaar phases kaunse hain aur kyun hain?

earlyRead → write → mixedReadWrite → read. Kyun: browser geometry (offsetHeight, getBoundingClientRect) padhne pe pending styles flush karke layout compute karta hai. Agar components read→write→read karte hain, har read ek forced synchronous layout hai (Ch6 S6). Phases ke saath Angular saare components ke reads pehle chalata hai (ek layout), phir saare writes, phir saare final reads (ek layout). 20 components ke 40 layouts se 2 ho jaate hain. Ye framework-level batching hai — tumhare component ko doosron ke baare mein sochna nahi padta.

MEDIUM afterRenderEffect mein ek plain field padhne se kya hota hai?

Koi dependency nahi banti, toh wo effect ek baar chalega aur phir kabhi nahi — silently. afterRenderEffect reactivity signals se aati hai: jo signal uske andar padha jaata hai wahi uski dependency banta hai. Toh this.chart.update(this.rawData) (plain field) kabhi re-run nahi hoga; this.chart.update(this.data()) (signal) hoga. Ye ek common aur debug karne mein mushkil bug hai.

HARD afterNextRender ne ngAfterViewInit ki kaunsi teen problems solve ki? Mechanism ke saath.

(1) SSR: ngAfterViewInit server pe bhi chalta hai (change detection server pe chalti hai), toh nativeElement.focus(), getContext('2d'), new Map(el) sab crash karte hain — aur log isPlatformBrowser guards likhte hain. afterNextRender ka callback server pe register hi nahi hota: zero cost, zero crash, zero guard. Ye API design ka faayda hai — galat cheez likhna mushkil bana do. (2) ExpressionChangedAfterItHasBeenChecked: ngAfterViewInit CD cycle ke andar hai, template ki bindings evaluate hone ke baad — wahan state badalna dev-mode error deta hai (S7). afterNextRender CD ke bahar chalta hai, toh usme state change bilkul safe hai (wo agli CD cycle trigger karega). (3) Timing: ngAfterViewInit browser paint se pehle hai, toh layout measurements stale ya incomplete ho sakti hain (CSS/fonts abhi apply nahi hue), aur unhe padhna forced synchronous layout trigger karta hai. afterNextRender paint ke baad chalta hai — measurements final. (4 bonus) Layout thrashing: uske chaar phases (earlyRead, write, mixedReadWrite, read) saare components ke DOM reads aur writes ko batch karte hain — 20 components ke 40 forced layouts se 2. Yahi wo cheez hai jo tum apne component mein akele kabhi achieve nahi kar sakte the. Design lesson: Angular ne ek hook ko "thoda behtar" nahi banaya — unhone lifecycle ko do alag concerns mein tod diya: "component ka state kab ready hai" (lifecycle hooks) aur "DOM kab safe hai" (render hooks). Ye separation hi asli innovation hai.

SCENARIO Ek dashboard mein 20 chart components hain (chart.js). Symptoms: SSR pe build crash, dev mein ExpressionChanged errors, scroll pe jank aur "Forced reflow" warnings, aur route change ke baad memory badhti hai. Poora redesign karo.

Chaar symptoms, aur chaaron ek hi root cause ke hain: DOM ka kaam lifecycle hooks mein ho raha hai. (1) SSR crash: ngAfterViewInit mein new Chart(canvas.nativeElement) — hook server pe chalta hai, wahan canvas nahi hai. Fix: afterNextRender({write: () => this.chart = new Chart(...)}) — server pe register hi nahi hota. isPlatformBrowser guard bhi kaam karta, par wo boilerplate hai aur baaki teen problems solve nahi karta. (2) ExpressionChanged: chart create karke this.ready.set(true) (bound signal) — CD ke andar state change (S7). afterNextRender CD ke bahar hai, toh ye apne aap fix ho jaata hai. (3) Jank + forced reflow: har chart ngAfterViewChecked mein canvas size measure karke redraw karta hai — 20 components × read/write = 40 forced synchronous layouts per CD cycle, aur scroll pe 60 cycles/sec. Fix: afterRenderEffect({earlyRead: () => canvas.getBoundingClientRect(), write: (rect) => chart.resize(rect)}) — ab Angular saare 20 components ke reads pehle chalata hai (1 layout), phir writes. 40 layouts se 2. Aur afterRenderEffect sirf dependencies badalne pe chalta hai, har cycle nahi. (4) Memory leak: chart.destroy() kahin nahi. chart.js apne resize listeners aur animation frames rakhti hai. Fix: destroyRef.onDestroy(() => this.chart?.destroy()), aur wo afterNextRender ke andar likho — setup ke saath (S8). (5) Data updates: ngOnChanges mein chart.update(this.data) → afterRenderEffect({write: () => chart.setData(this.data())}). Signal read se dependency banti hai — sirf data badle tab update. Trap: agar this.rawData plain field hai toh koi dependency nahi banegi aur effect ek baar chalke chup ho jaayega. (6) Hydration: canvas ko chart.js mutate karti hai → ngSkipHydration us element pe, warna hydration mismatch (Ch3 S8). (7) Verify: SSR build pass; dev mein zero ExpressionChanged; Performance tab mein "Forced reflow" count (40×60 se ~2×60); heap snapshot 20 navigations ke baad flat. (8) Poora pattern (yaad rakhne layak): create in afterNextRender, update in afterRenderEffect, destroy in DestroyRef. Ye teen lines har third-party DOM library ka sahi lifecycle hain.

1. What is it?

Ye section poore chapter ka payoff hai. Aath classic hooks padhne ke baad, ab wo jawab jo zyada tar devs ke paas nahi hai:

2. Why do we need it (ye samajhna)?

Kyunki lifecycle hooks ek timing abstraction hain: "ab kuch karo". Signals ek dependency abstraction hain: "jab ye badle tab kuch karo".

Zyada tar UI code ka natural roop dependency hai, timing nahi. "Total = items ka sum" ek dependency hai. "Chart data ke saath sync rahe" ek dependency hai. Jab tum use timing mein express karte ho (ngOnChanges, ngDoCheck), tum ek derived problem solve kar rahe ho: "kab pata chalega ki badla?"

3. What problem does it solve?

  • Decision paralysis → ek clear decision tree.
  • Legacy migration → hook-by-hook table.
  • Testing → kaunsa hook kaise trigger karein.
  • Edge contexts → embedded views, dynamic components, directives.

  // ── Derived (koi ngAfterContentChecked) ──
  protected title = computed(() => titleFor(this.metricId()));       // L3

  private chart?: Chart;

  constructor() {
    const destroyRef = inject(DestroyRef);

    // ── DOM init (koi ngAfterViewInit) ──
    afterNextRender({                                            // L4
      write: () => {
        this.chart = new Chart(this.canvas().nativeElement);
        destroyRef.onDestroy(() => this.chart?.destroy());        // L5: cleanup
      },
    });

    // ── DOM sync (koi ngAfterViewChecked) ──
    afterRenderEffect({                                          // L6
      write: () => {
        const d = this.chartData.value();                        // signal dependency
        if (d) this.chart?.setData(d);
      },
    });
  }
}
// ZERO lifecycle hooks. Aur ye poora component SSR-safe hai.

10. Code Explanation — Line by Line

  • L1–L2: metricId ya range badle → params signal badla → resource refetch karta hai aur purani request abort. Ye ngOnInit + ngOnChanges + manual race handling + ngOnDestroy unsubscribe — chaaron ki jagah leta hai.
  • L3: computed() memoized hai — metricId() na badle toh titleFor() dobara nahi chalti. ngOnChanges mein ye har change pe chalti.
  • L4–L5: Chart creation browser-only, paint ke baad, aur cleanup usi jagah register hui — setup se 40 lines door nahi. Ye locality hi wo cheez hai jo cleanup bhoolne se bachati hai.
  • L6: Chart update reactive hai: chartData.value() ek signal read hai, toh wo dependency ban gayi. Data badle tabhi chart update hoga — har render pe nahi (afterRender hota toh 60/sec).
  • Zero hooks, aur har guarantee stronger: SSR pe crash nahi (render hooks register nahi hote), race condition nahi (resource abort), leak nahi (DestroyRef), ExpressionChanged nahi (CD ke bahar), aur derived state kabhi stale nahi (computed). Ye ergonomics ka upgrade nahi hai — ye correctness ka upgrade hai.
  • Aur ek baat imaandaari se: legacy codebase mein purane hooks kaam kar rahe hain, aur unhe bina wajah migrate karna galat hai. Migrate tab karo jab tum us component ko waise bhi touch kar rahe ho, ya jab uske saath ek asli bug juda ho (SSR, race, leak, thrashing).

11. Common Mistakes

Galtiyan
  • Saare 8 hooks implement karna "completeness" ke liye — har "Checked" hook per-cycle tax hai.
  • @if ke andar wale component ka state expect karna — wo destroy/recreate hota hai.
  • @for mein track $index — reorder pe component state galat item ke saath.
  • createComponent mein setInput attachView ke baad — pehli CD mein input undefined.
  • attachView bhoolna — CD chalti hi nahi, ngOnInit kabhi nahi.
  • Test mein fixture.detectChanges() bina ngOnInit expect karna.
  • effect()/afterRenderEffect() test karte waqt TestBed.tick() bhoolna.
  • Legacy code ko bina wajah migrate karna — working code ko touch karne ka cost hai.

12. Best Practices

Production rules
  • Naya code: constructor (wiring) + DestroyRef (cleanup) + kabhi-kabhi ngOnInit. Bas.
  • Decision tree follow karo: derived → computed; fetch → resource; DOM → render hooks; cleanup → DestroyRef.
  • Third-party pattern: create in afterNextRender, update in afterRenderEffect, destroy in DestroyRef.
  • State jo view ke jeevan se lambi hai, wo component mein nahi (scoped store, Ch8 S6).
  • @for mein hamesha stable track — correctness ka sawaal hai.
  • Migrate tab karo jab component waise bhi touch ho raha ho, ya ek asli bug ho.
  • Khaali hooks delete karo — lint rule se pakdo.

13. Performance

Poora chapter ek performance kahani bhi hai:

  • ngDoCheck/ngAfterContentChecked/ngAfterViewChecked: instances × CD cycles function calls (S1/S5).
  • ngOnChanges: SimpleChanges allocation per change per component (S3).
  • ngAfterViewChecked mein DOM reads: forced synchronous layout per cycle (S7).
  • Modern equivalents: computed memoized+lazy, afterRenderEffect dependency-driven + phase-batched, resource abort-driven.

Ek 5,000-row grid mein teen "Checked" hooks hataane se CD time ka 40% bach sakta hai (S1 ka scenario) — aur wo sirf pehla kadam hai; asli fix virtualization hai.

14. Edge Cases

  • Directives: sirf ngOnChanges, ngOnInit, ngDoCheck, ngOnDestroy (view/content hooks nahi).
  • Pipes: ngOnDestroy (impure pipes ke liye useful).
  • Services: ngOnDestroy (scoped injectors mein, S8).
  • createComponent: setInput() pehle, attachView() baad mein, phir pehli CD pe hooks.
  • @defer: hooks tab chalte hain jab chunk load hoke view bane.
  • Test: fixture.detectChanges() se ngOnInit; TestBed.tick() se effects + render callbacks; fixture.destroy() se cleanup.
  • Inherited hooks: base ka chalta hai agar derived override na kare; override karo toh super tumhara kaam.
  • Hook names string-matched — implements lagao (S1).

15. Interview Questions

EASY Naye Angular code mein kaunse lifecycle hooks chahiye?

Practically teen cheezein: constructor (wiring — inject, signals, effect/afterNextRender registration), DestroyRef.onDestroy() (cleanup), aur kabhi-kabhi ngOnInit (input validation, ek-baar registration, analytics). Baaki paanch hooks ke modern replacements hain: computed(), resource(), signal queries, afterNextRender(), afterRenderEffect().

EASY @if toggle karne pe component ka kya hota hai?

View destroy aur recreate hoti hai — hide/show nahi. Toh: purane instance ka ngOnDestroy, phir naya instance (constructor, ngOnInit, ngAfterViewInit). Uski poori state chali jaati hai (form values, scroll, animation), aur ngOnInit ki HTTP call dobara jaati hai. State preserve karni ho toh CSS hiding ya ek scoped store use karo.

MEDIUM Test mein ngOnInit kaise trigger karte hain?

TestBed.createComponent() se sirf constructor chalta hai. Inputs set karo (fixture.componentRef.setInput('id', '123')), phir fixture.detectChanges() — tab pehli CD cycle chalti hai aur ngOnChanges/ngOnInit/ngAfterViewInit chalte hain. fixture.destroy() se ngOnDestroy + DestroyRef callbacks. Aur effect()/afterRenderEffect() ke liye TestBed.tick() chahiye (signals sync hain par effects nahi).

MEDIUM Dynamic component (createComponent) mein lifecycle ka order kya hai?

createComponent() → constructor chalta hai (ngOnInit nahi). Phir ref.setInput(...) — ye attachView se pehle karo. Phir appRef.attachView(ref.hostView) — ab component CD tree mein hai, aur pehli CD cycle pe ngOnChanges/ngOnInit/ngAfterViewInit chalenge. Cleanup: detachView() + ref.destroy() (jo ngOnDestroy chalata hai). Traps: attachView bhoolna → CD kabhi nahi, ngOnInit kabhi nahi; destroy bhoolna → leak.

HARD "Lifecycle hooks ek timing abstraction hain, signals ek dependency abstraction" — explain karo.

Lifecycle hook bolta hai "ab ye code chalao" — tumhe moment milta hai (inputs set hue, view bani, destroy ho raha hai). Signal bolta hai "jab ye value badle tab ye code chalao" — tumhe dependency milti hai. Zyada tar UI code ka natural roop dependency hai: "total = items ka sum", "chart data ke saath sync rahe", "user badle toh refetch". Jab tum inhe timing mein express karte ho, tum ek derived problem solve kar rahe ho: "kab pata chalega ki badla?" — aur uske jawab hain ngOnChanges (reference equality, mutation miss karta hai), ngDoCheck (polling, har cycle), ngAfterViewChecked (polling + thrashing). Teeno mechanism hain us sawaal ke, na ki tumhare problem ke. Signals mein wo sawaal hi khatam ho jaata hai: dependency graph exact hai, propagation glitch-free hai, aur derivation memoized hai. Isliye computed() ne ngOnChanges+ngDoCheck ko replace kiya, aur afterRenderEffect() ne ngAfterViewChecked ko. Jo hooks bache hain, wo wahi hain jo sach mein timing ke baare mein hain: constructor ("main ban raha hoon"), ngOnInit ("inputs ready hain"), DestroyRef ("main mar raha hoon"), afterNextRender ("DOM ban chuka"). Ye moments dependencies se express nahi ho sakte — isliye wo zinda hain. Aur ek bada pattern: yahi transition poore Angular mein hai (Ch6 S2) — ngClass+differ → classProp, *ngFor+IterableDiffer → δδrepeater, Zone ka guessing → signal graph. Har jagah: generic runtime polling ko exact, declared dependencies se replace karo.

SCENARIO Tum ek 4-saal purani Angular app ke tech lead ho. 400 components, har jagah ngOnInit+ngOnDestroy+ngOnChanges, kuch ngDoCheck, aur SSR add karna hai. Lifecycle migration ka plan banao.

(1) Blanket migration mat karo. 400 components migrate karna 6 mahine hai aur business value zero. Working code ko touch karne ka apna cost hai (regression risk). Plan risk aur value se drive ho, ideology se nahi. (2) SSR pehle — kyunki wo ek hard requirement hai, aur wo tumhe migration ka natural filter deta hai. SSR enable karo aur crash list banao. Har crash ek ngAfterViewInit/ngOnInit hai jo DOM chhoo raha hai. Sirf unhe migrate karo → afterNextRender(). Ye ek focused, measurable, bug-driven migration hai. (3) Leak audit: 20-navigation heap test (S8). Jo components leak kar rahe hain, unke ngOnDestroy ko DestroyRef mein le jao (locality se bhoolna mushkil), aur subscriptions ko takeUntilDestroyed()/toSignal(). Zombie-callback wale analytics bugs bhi yahin nikalenge. (4) Performance profile: DevTools profiler se CD time. Khaali "Checked" hooks (ngDoCheck, ngAfterViewChecked) delete karo — zero-risk, turant win (S1 scenario). Jo ngDoCheck zaroori lagte hain, unke upstream mutation dhundho aur immutable + readonly T[] se fix karo (S5). (5) Race conditions: ngOnInit+ngOnChanges wale fetch patterns dhundho. Jinme user tez interaction karta hai (search, filters), wahan resource() pe migrate karo — race structurally gayab. Ye ek bug-driven migration hai, cosmetic nahi. (6) Naye code ke liye rules (ratchet): ek lint rule + PR checklist — "naya component: constructor + DestroyRef + optional ngOnInit. DOM → render hooks. Derived → computed." CI mein ngDoCheck/ngAfterViewChecked pe warning. Ye sabse important step hai — warna tum saaf karte rahoge aur team purana pattern likhti rahegi. (7) Opportunistic migration: jab koi component waise bhi touch ho, tab uske hooks modernize karo. 6 mahine mein 60% components apne aap ho jaayenge. (8) Kya nahi migrate karna: stable, 2 saal se na chhue gaye components jinme SSR crash, leak, race, ya jank nahi hai. Unhe chhod do. (9) Metrics jo report karoge: SSR crash count (→ 0), heap growth after 20 navigations, CD time on heaviest page, aur race-condition bugs. Har number ek business argument hai. (10) Punchline: migration ka justification "purane hooks bure hain" nahi hai — wo "in chaar bug classes ko ye APIs structurally impossible bana deti hain" hai: SSR crash, memory leak, race condition, layout thrashing. Bug-driven migration hamesha ideology-driven se tez chalti hai.
